### B. 알고리즘 요약

1. 대상은 SDR 멀티프로세서 플랫폼(PU들이 SIMD 슬라이스+로컬메모리를 가지며 상호연결망/글로벌메모리로 통신)에 매핑되는 QC 구조 irregular LDPC(rate 3/4, 코드워드 6176bit, submatrix 193x193).
2. 문제의식은 표준 코드 설계(FER 최적화)만으로는 실제 멀티프로세서 HW의 로컬메모리 부족, 큰 row weight로 인한 낮은 throughput, 글로벌메모리-PU 간 라우팅 충돌을 해결하지 못한다는 것.
3. 핵심 절차(Fig.1): 시스템요구(FER, rate, length)와 아키텍처 특성(PU 수, SIMD 폭, arithmetic complexity)을 "code feature"로 추출 -> 블록행렬 레벨/서브행렬 레벨 2단계로 코드 구성 -> 제약 충족 검증(반복 가능).
4. 사례1(수평분할, 로컬메모리 부족): parity-check 행렬을 supercode로 수평분할해 PU가 순차 처리(inner/outer iteration 2단 구조, Algorithm 2). degree-1 variable node를 최소화하는 제약(exchange step)을 부여해 supercode decoding의 수렴 저해 요인을 제거.
5. 사례2(수직분할, row weight 큼): 행렬을 수직분할하여 PU별 partial CP(check node processing) 후 partial sum을 교환하는 cooperative decoding(Algorithm 3). 파티션 간 row weight를 균등화하는 unbalance factor(UF) 최소화 제약을 부여.
6. 사례3(MIN 기반, 로컬메모리 매우 작음): butterfly MIN(multistage interconnection network)에서 동시 라우팅 가능한 노드 그룹을 exclusive node set(ENS)으로 정의, 같은 ENS 그룹 내 원소가 한 행에 2개 이상 배치되지 않도록 코드 구성 제약을 부여해 라우팅 충돌 원천 차단.
7. 부수 트릭: 코드 재구성은 오프라인(설계 단계)에서만 수행되어 런타임 성능에 영향 없음; block row/column을 교환해도 부호 자체는 불변(디코딩 후 원래 순서 복원).
8. 최적화 절차는 별도 학습이 아닌 조합적 탐색(무작위 배치 → degree distribution 만족하도록 1의 위치 조정 → 쌍 교환으로 추가 제약 충족, Section II-C 단계 i~vi).
9. 결과: 수평분할 최적화 코드가 비최적화 대비 반복횟수 절반(수렴 2배 빠름)으로 동일 FER 달성, 메모리 트래픽 2배 감소, MIN 최적화 코드는 원 코드 대비 47.5% 슬로우다운을 완전 제거(라우팅 충돌 0), SODA 4-PU 구성시 throughput 최대 42.8Mbps.
10. 한계: 대상 하드웨어가 범용 멀티프로세서 SDR(SIMD+MIN)이며 특정 ASIC 파이프라인 아님, FER 성능은 원 코드 대비 개선이 아니라 "저하 없음"이 목표(정정능력 자체 향상은 없음), butterfly MIN 등 특정 인터커넥트 토폴로지 가정.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| supercode 기반 수평분할 디코딩(로컬메모리 절약) | `decoder.cpp` `LDPC_Decoder()` | Prime ECC는 z=32 고정 lifting과 6-stage pipeline으로 이미 메모리 제약을 해결한 구조라 supercode 개념 자체의 이식 필요성은 낮음 |
| 수직분할 cooperative decoding(부하 분산) | `decoder.cpp` `CNU_Update_New_Mag()` | 병렬 CN 연산 부하 분산 아이디어는 참고 가능하나 Prime ECC는 이미 row 단위 병렬(z_sb=32) 구조 보유 |
| MIN 라우팅 충돌 회피 코드 구성(ENS 제약) | `ecc_top.cpp` `Load_PCM()` | H-matrix/코드 구조 설계 영역이라 이식 난이도 "높음"에 해당, 특정 QC-LDPC 고정된 Prime ECC에 재설계 부담 큼 |
| degree-1 노드 최소화 등 코드 구성 제약 | `ecc_top.cpp` `Load_PCM()` | 부호설계 레이어로 이식 난이도 높음, GT(Graph Thinning)와는 목적(HW MUX edge 축소 vs 멀티프로세서 라우팅 충돌)이 달라 직접 대응 없음 |
| 메모리 트래픽/글로벌-로컬 메모리 분석 | 대응 없음 | Prime ECC는 온칩 고정 HW 파이프라인으로 SDR 멀티프로세서의 global/local memory 구조 자체가 없음 |

적용 가치: **낮음** — 정정능력 향상이 없고(대상=both이나 C섹션 개선영역은 N/A), 대상 하드웨어가 SDR 멀티프로세서(SIMD+MIN)로 Prime ECC의 고정 ASIC 파이프라인 구조와 근본적으로 달라 코드 구성 제약을 그대로 이식하기 어려움. 다만 "아키텍처 제약을 코드 설계 단계 제약으로 변환"하는 방법론적 관점은 참고 가치 있음.
