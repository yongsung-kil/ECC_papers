### B. 알고리즘 요약

1. 대상: (1057,813)·(4161,3431) PG-LDPC(projective geometry, cyclic·regular, (y,p)=(33,33)/(65,65)) 및 (1944,972) 802.11n QC-LDPC(irregular), BPSK+AWGN.
2. 문제: LDPC BER 시뮬이 수일 소요, GPGPU가 유망하나 긴 부호/고weight면 H가 GPU on-chip 메모리(constant cache 64KB)를 초과해 느린 global memory 접근으로 성능 저하.
3. 핵심 기법: cyclic 부호는 첫 행/열 인덱스 `r[y]`만 저장하고 `(r[k]+n) mod N`으로 전체 H 재구성 → 필요 인덱스가 `2Nγ`에서 `p+γ`로 급감, constant cache에 상주.
4. QC 확장: H가 `δ×δ` permutation(순환 shift) submatrix로 구성 → submatrix당 2 인덱스면 충분, 메모리 `2δpγ→2pγ`로 δ배 절감.
5. 복호: 표준 SPA(식(1)-(3), `2·tanh⁻¹(∏tanh(|z|/2))`)와 normalized MSA(식(4)-(5), min 근사 + 밀도진화로 정한 정규화 계수 α).
6. CUDA 매핑: frame-by-frame + node별 병렬(check update M threads, bit update N threads), Z/L 메시지는 shared/global memory, 프레임 LLR·H는 constant cache broadcast.
7. multi-frame 대신 frame-by-frame 선택 이유: 분기·load imbalance·프레임 간 iteration 불균형 대기 회피.
8. warp(32 thread) 정렬 접근: 순환 인덱스 산술로 warp 내 동일 주소 접근 → constant memory 읽기가 register만큼 빨라짐(비순환이면 직렬화).
9. 결과: GTX 285에서 CPU 대비 SPA 30~40배, MSA 23~25배 speedup; 순환압축 미사용 대비 SPA 2배+(>200%), QC 1.5배(150%); (1057,813) FER 10⁻⁶ 추정에 CPU 60일→GPU 2일.
10. 한계: 부동소수점 시뮬 가속 툴로 BER는 CPU와 동일, 복호 성능·부호설계·ECC HW 개선 없음, normalized MSA는 SPA보다 iteration 더 많음((4161,3431) 2964 vs 1491).

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| SPA/normalized MSA 복호 | [decoder.cpp](../../Prime_ECC_3.1_Claude) `CNU_Update_New_Mag()`, `C2V_Cal_New_Sgn()` | 표준 min-sum 계열, Prime ECC가 이미 보유(중복) |
| cyclic/QC H 순환 인덱스 압축 저장 | [ecc_data.h](../../Prime_ECC_3.1_Claude) `PCM_b`(base+shift), [ecc_top.cpp](../../Prime_ECC_3.1_Claude) `Load_PCM()` | Prime ECC는 이미 base+shift(circulant)로 QC를 압축 보유 → 새 이득 없음 |
| GPU thread/메모리(constant cache·warp) 병렬화 | 대응 없음 | Prime ECC는 bit-exact HW 클록모델(C++), GPGPU 시뮬 최적화와 무관 |
| BER 시뮬 가속(수일→수시간) | 대응 없음 | Prime ECC는 HW 정합 시뮬 목적, Monte-Carlo GPU 가속 툴과 목적 상이 |

적용 가치: **낮음** — GPGPU 부동소수점 시뮬 가속이 핵심으로 Prime ECC(NAND binary QC-LDPC의 bit-exact HW 시뮬)와 목적·플랫폼이 다르고, QC 순환압축은 이미 base+shift로 보유해 이식할 신규 기법이 사실상 없음.
