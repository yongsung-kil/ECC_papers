### B. 알고리즘 요약

1. 대상은 rate 1/2 convolutional code(CC1=(7,5)8, CC2=(45,73)8)이며, turbo code에서 쓰이는 SISO 복호를 BCJR 대신 sum-product algorithm(SPA)으로 대체하려는 시도.
2. 문제: convolutional code의 parity check equation을 그대로 Tanner graph로 변환해 SPA를 적용하면 4-cycle이 존재하여 성능이 나쁨.
3. 채널 모델은 AWGN, 정보블록 1024비트, 최대 반복 100회로 시뮬레이션.
4. 방법 1(RmFC): 보조 비트열 `A(D) = X(D) + P(D)`와 보조 체크식 `H1 = X+P+A`, `H2 = DP + (D²+1)A`을 추가해 4-cycle을 제거, girth를 8로 늘림.
5. RmFC는 CC1에는 효과가 있었으나(BCJR 대비 0.1dB 열화) CC2에는 개선 효과 없음 — 4-cycle이 성능 저하의 유일 원인이 아님을 시사.
6. 방법 2(HOPCE): 원래 parity check equation `H(X,P)`에 임의 다항식 `M(D)`를 곱해 `H'=M(D)H(X,P)`를 만들어 차수를 높인 등가 check equation을 사용, 4-cycle 개수(nfc)가 최소인 M(D)를 휴리스틱하게 탐색.
7. HOPCE 적용 결과 CC1은 BCJR 대비 0.1dB, CC2는 0.3dB 이내로 성능 개선(기존 SPA 대비 각각 0.9dB, 1.3dB 개선).
8. 최적 M(D) 탐색은 사전 정의된 차수 범위에서 전수 조사 후 nfc 최소인 것을 선택하는 휴리스틱(별도 최적화 이론 없음).
9. 복잡도 비교: SPA 연산 수는 CC1에서 BCJR과 비슷(105.3 vs 101), CC2에서는 BCJR의 0.14배로 더 적음(평균 iteration 1.4~2.7).
10. 한계: HW 구현 없이 시뮬레이션뿐이며, convolutional code 전용 기법으로 LDPC 부호 설계나 QC-LDPC 구조와 무관.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| SPA(sum-product) 기반 convolutional code 복호 | 대응 없음 | Prime ECC는 QC-LDPC 전용이며 convolutional code 복호 대상 아님 |
| 4-cycle 제거(RmFC) | 대응 없음 | LDPC H-matrix가 아닌 convolutional code polynomial 구조에 특화된 기법 |
| 고차 parity check equation(HOPCE) | 대응 없음 | convolutional code parity equation 변형 기법으로 QC-LDPC 부호 설계와 무관 |
```

적용 가치: 낮음 — 대상 부호가 LDPC가 아닌 convolutional code이며, 제안 기법 모두 convolutional code의 polynomial 구조에 특화되어 있어 Prime ECC(QC-LDPC)에 이식할 요소가 없음.
