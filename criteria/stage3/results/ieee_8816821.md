### B. 알고리즘 요약

1. 시스템: 3GPP 5G NR QC-LDPC 디코더, Base Graph 1(46x68)·2(42x52), lifting size `Zc` up to 384, QPSK, `R=1/3`(BG1)·`R=1/5`(BG2), 최대 NR 블록 26,112 bit.
2. 문제: LDPC를 HW(ASIC/FPGA) 대신 GPU SDR로 가상화하려 하나 GPU는 불규칙 메모리 접근으로 latency가 큼 - 저지연 요구 충족이 목표.
3. 부호 구조: PCM은 compact `[h]` 행렬의 각 원소를 `Zc x Zc` circulant 순환치환/영행렬로 lifting해 생성(TS 38.212), 82개 H 행렬 존재.
4. 기저 알고리즘: binary LDPC의 Sum-Product Algorithm(SPA), Tanner graph의 info/parity 노드 간 LLR 메시지 전달.
5. 디코더 개선: 최적 가중치+고정소수점 확률계산, 벡터화된 message passing으로 중복 제거, PCM 압축 저장, GPU용 fully-independent thread rotator.
6. PCM 압축: 표준 28kbit -> 3.5kbit(BG1 기준), 82개 uncompressed 64kB -> compressed 9kB로 shared local memory 절감(핵심 기여).
7. GPU 병렬화: `Zc` 도메인당 최대 384 병렬 프로세스, H 행 내 원소 병렬, 저상관 행 결합으로 추가 병렬화 계획.
8. FPGA 구현: Xilinx Zynq UltraScale+ xczu15eg, 245.76MHz, HW 자원 <50%, iteration당 BG1 4.25us / BG2 3.14us(Zc=384).
9. 결과: 고정소수점 FPGA vs 32-bit float GPU BLER 차이 0.5dB 이내, GPU는 단일 multiprocessor 사용 시 목표 대비 약 32배 큰 latency(BG1 520us/iter), 평균전력 FPGA 7.5W / GPU ~5W*, PM_FPGA=42.75 > PM_GPU=11.11.
10. 한계: 정정능력 개선 아님(플랫폼 비교), GPU latency는 향후 개선 과제, throughput/gatecount 수치 본문 미기재, NR 전용 부호.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| QC-LDPC PCM(circulant lifting) 로드 | ecc_top.cpp Load_PCM() | 개념 동일(base+shift) - 단 NR BG는 우리 고정 H와 다름 |
| PCM compact 압축 저장 | ecc_data.h PCM_b (base+shift 표현) | 우리도 base+shift로 이미 컴팩트 저장 - 신규성 낮음 |
| SPA(full tanh) CN 연산 | 대응 없음 | 우리는 min-sum 사용, full-SPA는 미보유이나 HW 면적상 채택 안함 |
| 고정소수점 LLR bit-width 설계 | decoder.cpp VN_Cal_HD(), ecc_data.h PARAM_LLR | 양자화 관점만 약한 대응 |
| GPU thread rotator / 병렬화 | 대응 없음 | Prime ECC는 z=32 HW pipeline, GPU 아키텍처 무관 |
```

적용 가치: 낮음 - QC-LDPC이라 부호 형식은 겹치나 5G NR 전용 부호·SPA·GPU/FPGA 플랫폼 비교라 NAND HW 디코더 개선 기여가 없고, PCM 압축은 이미 base+shift로 보유.
