### B. 알고리즘 요약

1. 대상 코드: IEEE 802.11e/802.11ad QC-LDPC (C1~C6), `N=576~2304`, rate `1/2~13/16`, `Z=24~96`, circulant identity 서브블록 구조.
2. 문제: GPU/inter-frame 병렬 SW 디코더는 throughput은 높지만 latency가 수백µs~ms — 5G/SDR는 저지연 요구.
3. 접근: 단일 프레임 내부(intra-frame)를 병렬화해 프레임 버퍼링 제거, latency를 µs급으로 단축.
4. 스케줄: horizontal layered OMS(offset min-sum) — flooding 대비 iteration 절반, 메모리 ~30% 절감.
5. 도전1: 프로세서 SIMD 폭 `Q`(8-bit 값 병렬) vs. 코드의 `Z` CN 구조 불일치 → `Z/Q` 서브이터레이션 분해, 비정수 시 padding으로 효율 저하(예 Z=42, Q=64 → 66%).
6. 도전2: gather/scatter 부재 → VN rotation 접근이 난제. HW barrel shifter 동작을 SW로 재현.
7. 핵심 트릭: `rid` 회전값에 대해 twin aligned memory read + mask/AND/OR 로 VN 세트를 정렬 재구성 (generic 8b gather의 24~42 cycle을 5~10 cycle로 단축).
8. 구현: 코드길이·`Z`·`rid`가 컴파일 타임 상수 → C++11 template 파라미터화로 디코더 소스 자동생성, 루프 언롤·레지스터 배치 (LLVM 4.0, -Ofast).
9. 결과: 실측(Core-i7, ~3GHz) throughput 102~385 Mbps, latency <10µs. inter-frame[7] 대비 throughput 16~60%↓ 이지만 latency ~×22 개선. 양자화 8b VN/6b msg, OMS로 C6는 float 대비 0.1dB 열화.
10. 한계: 순수 소프트웨어(x86 AVX2) 구현으로 HW 설계·gatecount 없음, 정정능력 개선 없음, 최적화가 CPU 캐시/SIMD에 강결합.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| layered OMS 디코딩 루프 | [decoder.cpp](../Prime_ECC_3.1_Claude/) `LDPC_Decoder()` | 알고리즘은 우리 min-sum과 유사(중복), 논문 기여는 SW 매핑이라 알고리즘 이식가치 낮음 |
| CN min-sum 커널 | [decoder.cpp](../Prime_ECC_3.1_Claude/) `CNU_Update_New_Mag()` | 표준 min-sum, 새 기여 없음 |
| VN rotation twin-read 트릭 | 대응 없음 | x86 SIMD 메모리접근 특화, 우리는 HW barrel-shifter(z=32) 모델 — SW 트릭 이식 불가 |
| 조기종료(syndrome check) | [partial_CRC.cpp](../Prime_ECC_3.1_Claude/) `partial_crc_HW_T4()` | 우리는 CRC 기반, 논문은 syndrome 기반 — 개념만 유사 |
| 양자화(8b VN/6b msg) | [channel.cpp](../Prime_ECC_3.1_Claude/) LLR 테이블 | 우리는 hard/2SD/3SD, 논문 SW 8b와 직접 대응 안 됨 |

적용 가치: **낮음** — 알고리즘은 표준 layered OMS이고 핵심 최적화가 전부 x86 SIMD/캐시 특화 소프트웨어여서 NAND HW binary QC-LDPC 디코더에 떼어 쓸 요소가 사실상 없음.
