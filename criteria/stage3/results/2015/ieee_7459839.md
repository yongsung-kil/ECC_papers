### B. 알고리즘 요약

1. Protograph 기반 LDPC 코드(base matrix `B`, lifting factor `z`)에서 edge-wise 디코딩 스케줄을 다룸.
2. 기존 M2I2 알고리즘은 모든 protograph edge(`E_B`)에 대해 고차 예측(`P`) 시 트리 탐색 복잡도가 `O(E_B^P)`로 지수적으로 증가하는 문제가 있음.
3. C2V 메시지의 mutual information 증가량 `r(i,j) = I_Ep,C(i,j) - I_E,C(i,j)`을 기준으로 다음 업데이트할 edge를 결정한다는 M2I2의 기본 가정을 그대로 채택.
4. 핵심 기법: list decoding 개념을 빌려 각 단계에서 상위 `E_L`개 edge만 후보 리스트로 남기는 LM2I2 알고리즘 제안 (`E_L < E_B`).
5. 고차 증분 `rΔ(i,j) = r(i,j) + Δ(i,j)`은 order-1 증분과 depth-2 이후 최대 증분 `Δ(i,j)`의 합으로 재귀적으로 계산됨 (Algorithm 1).
6. `E_L = E_B`이면 LM2I2는 원래 M2I2와 동일해지므로, M2I2는 LM2I2의 특수 case로 볼 수 있음.
7. 리스트 크기를 줄이는 대신 예측 차수 `P`를 늘려, 동일 계산자원으로 더 빠른 수렴 스케줄을 얻을 수 있음.
8. 별도 학습/최적화 절차는 없으며, 순수 탐색 알고리즘(list 기반 가지치기)만 사용.
9. 결과: (2640,1320) Margulis 코드에서 `E_L=90, P=3` 조합이 `E_L=720, P=2`(원 M2I2) 대비 후보수는 늘었지만 더 빠른 수렴을 보였고, G.hn 코드에서는 `E_L=40`이 `E_L=76`(전체) 대비 예측 연산량을 1/10로 줄이면서 동일 성능 확보.
10. 한계: 시뮬레이션(FER/BER, ICMI 곡선)에 국한되며 HW 구현·정정능력(error-floor) 개선은 다루지 않음. 예측 차수와 리스트 크기의 최적 조합은 경험적 탐색에 의존.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 디코딩 스케줄 배치(LM2I2) | `decoder.cpp` `LDPC_Decoder()` | 이터레이션 루프 내 메시지 업데이트 순서 결정에 아이디어 참고 가능 |
| C2V mutual information 기반 edge 선택 | `decoder.cpp` `C2V_Cal()` | 현재 min-sum 기반 C2V 계산과는 별개의 스케줄링 계층으로, 순서 결정 로직에 적용 시도 가능 |
| protograph/QC-LDPC 구조 | `ecc_top.cpp` `Load_PCM()` | base matrix 개념은 QC-LDPC PCM_b와 유사하나 직접 대응 없음 |
| Dual-Update(교대 스케줄) 대비 개선 | `decoder.cpp` Dual-Update 옵션 | 기존 고정 스케줄과 비교 대상이 될 수 있으나 코드상 직접 연동 없음 |
```

적용 가치: 낮음 | 오프라인 트리 탐색 기반 스케줄 최적화로, 이미 Dual-Update/적응형 LLR 스케줄을 보유한 Prime ECC HW 파이프라인 구조에 새 스케줄을 이식하려면 재검증 부담이 크고, 정정능력 개선이 없어 실익이 제한적.
