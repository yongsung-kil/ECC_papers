### B. 알고리즘 요약

1. 4G 모바일 통신 환경에서 Gallager 구성법으로 만든 regular LDPC 코드(총 코드워드 길이 153600, 실험상 768000/1536000까지 확장)를 대상으로 함.
2. 기존 순차(serial) 인코딩(O(n²))과 BP 디코딩이 느려 모바일 단말(Tegra K1, 4-core CPU + 192-core GPU) 실시간 처리에 부적합하다는 문제를 다룸.
3. 채널/노이즈 모델에 대한 별도 정의 없음 - 표준 이진 대칭 채널 기반 BP만 다룸.
4. 인코딩: 비부호화 정보 `M`을 `G`(q*s 행렬)에 맞춰 `q`개 행 벡터로 분할, GPU 스레드 `q`개에 할당해 `ck=mk*G` 병렬 계산 후 병합.
5. 디코딩: variable-to-check `q_ji(b)`, check-to-variable `r_ij(b)` 갱신식(식 2~8)은 표준 BP(sum-product) 그대로이며, variable node/check node 각각 독립 GPU 스레드에 매핑해 병렬 계산.
6. 확장 없음 - 단일 CUDA 병렬화 기법만 적용.
7. CPU 스케줄링 + GPU global memory 데이터 공유 구조로 host-device 메모리 복사를 최소화하는 구현 디테일 언급.
8. 별도 학습/최적화(DE, GA, NN) 절차 없음.
9. 결과: 데이터 블록 100개(코드워드 총 길이 1536000) 기준 스레드 64→128 증가 시 성능 약 50% 개선; 병렬 인코딩이 순차 대비 처리시간 유의미하게 단축(구체 수치는 그림에만 존재해 본문 수치 없음).
10. 한계: HW(RTL) 설계 없이 GPU 소프트웨어 구현·실측 실험뿐이며, 부호설계/디코딩 알고리즘 자체의 정정성능 개선은 다루지 않음.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| BP(sum-product) 디코딩 알고리즘 | 대응 없음 | Prime ECC는 min-sum 기반이며 논문은 표준 full BP 사용, 알고리즘 자체 개선 없음 |
| GPU 스레드 병렬화 | 대응 없음 | 소프트웨어(CUDA) 병렬화로 HW(RTL) 병렬 아키텍처(z=32, HCU 등)와 무관 |
| Gallager 구성 H-matrix | `ecc_top.cpp` `Load_PCM()` | 규칙적 LDPC 구성법 자체는 이미 확립된 방식으로 이식 가치 낮음 |
```

적용 가치: 낮음 — 표준 BP 알고리즘을 GPU에서 소프트웨어 병렬화한 것으로, 알고리즘/부호설계/HW 아키텍처 어느 쪽으로도 Prime ECC(고정 QC-LDPC, HW RTL 시뮬레이터)에 이식할 요소가 없음.
