### B. 알고리즘 요약

1. 대상은 주기적 시변(periodically time-varying) LDPC 컨볼루션 코드(LDPC-CC)이며, CCSDS AR4JA 코드를 unwrapping해 구성한 두 코드(6144×10240, 3072×7168)로 실험.
2. 기존 GPU 기반 디코더는 LDPC block code(LDPC-BC) 위주였고, LDPC-CC는 블록 길이가 훨씬 커서 초기 디코딩 지연이 크므로 그대로 적용 불가.
3. 파이프라인 디코딩 방식을 채택: syndrome former memory `ms` 기준 `ms+1` 시간단위 이상 떨어진 variable-node는 독립적으로 처리 가능해 `I`개의 독립 프로세서(CUDA 커널)로 `I` iteration을 동시 수행.
4. Fine-grained 최적화 3단계: (1) Instruction-Level Parallelism - row/column block별 전용 커널과 loop unrolling으로 레지스터 활용 극대화, (2) Thread-Level Parallelism - `Q×P×Nbl ≤ Nreg`, `P×Nbl ≥ Nth`, `P≡0(mod c)` 조건으로 최적 TLP `P` 선택, (3) Function-Level Parallelism - grid size `I×M`이 `Nsm×Nbl`의 배수가 되도록 설계해 tail effect 방지.
5. Coarse-grained 최적화: H2D(host-to-device) 전송을 CUDA stream 단위로 배치(batching)해 PCI-E 대역폭 활용을 28%→66%로 개선하고, 다중 CUDA stream(Hyper-Q)으로 커널 실행과 데이터 전송을 중첩해 지연시간 은닉.
6. 디코딩 지연시간 근사식 `Tl = c×(ms+1)/Throughput × Sc×Ss × I`로 코드 스트림 배치 수(`Sc`, `Ss`)와 iteration 수 관계를 모델링.
7. TLP=256이 두 코드 모두에 적합함을 실험적으로 확인(64/128은 latency 은닉 부족, 512는 register spilling 발생).
8. 파라미터 탐색은 전수조사 대신 위 3개 조건식 기반으로 후보를 좁히는 방식(설계 criteria)으로 수행.
9. GTX460/GTX580에서 기존 GPU 기반 LDPC-CC 디코더([10]) 대비 약 3배 throughput 향상, H2D 대역폭 활용 28%→66% 개선을 수치로 제시.
10. FPGA 대비(예: 4-bit quantized LDPC-CC 2.0Gb/s on Stratix IV)로는 여전히 throughput이 낮으며, HW(ASIC/FPGA) 설계 없이 GPU 소프트웨어 구현·실측에 국한.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| LDPC-CC 파이프라인 디코딩 알고리즘 전체 | 대응 없음 | Prime ECC는 QC-LDPC block code 전용이며 convolutional code 구조·타임슬롯 파이프라이닝 개념이 없음 |
| GPU CUDA 커널/스레드 병렬화(ILP/TLP/FLP) | 대응 없음 | Prime ECC는 bit-exact HW(RTL) 시뮬레이터로 GPU 소프트웨어 병렬화 개념과 무관 |
| flooding schedule BP 디코딩 | `decoder.cpp` `LDPC_Decoder()` | 메시지 전달 반복 구조 자체는 유사하나 스케줄링(flooding vs dual-update)과 플랫폼(GPU vs HW)이 달라 직접 이식 불가 |

적용 가치: 낮음 — GPU 소프트웨어 구현, LDPC convolutional code 대상, binary QC-LDPC HW ECC인 Prime ECC와 부호구조·플랫폼 모두 상이하여 이식 요소 없음.
