### B. 알고리즘 요약

1. 대상 시스템: 2bit/cell NAND, 128GB SSD(8채널×8칩×4plane), LDPC 기반 ECC로 soft-read(다중 sensing level) 지원.
2. 문제: 공정편차(PV)로 블록별 RBER 편차가 10배 이상 → 신뢰도 낮은 블록은 더 많은 sensing level 필요 → read latency 증가.
3. 모델: 블록 read latency = 디코딩에 사용된 sensing level 수의 함수(Table I: 1~7 level, 25~175μs). PV 분포는 Gaussian 가정(선행연구 [12]).
4. 핵심 기법 1단 - 블록 그루핑: 블록의 첫 페이지 read latency(=사용 sensing level 수)로 신뢰도 측정, M개 그룹(group1=고신뢰 ~ groupM=저신뢰)으로 온라인 분류.
5. 핵심 기법 2단 - read-hot 데이터 배치: 임계값 `T`(=3) 이상 읽힌 read-hot 페이지를 상위 신뢰 그룹(group i-1)으로 마이그레이션.
6. 부수 트릭 - read-cold 축출: 고신뢰 그룹에 공간 부족 시 CLOCK 알고리즘(reference bit)으로 read-cold 블록 선택해 하위 그룹으로 이동 후 erase.
7. 오버헤드: 128GB 기준 grouping 정보 128KB + read counter ~800KB + CLOCK ~1MB ≈ 총 2MB, 비영속.
8. 평가: Disksim + SSD 모델, 5가지 grouping scheme(1/2/3/4/7-group) 비교, 8개 트레이스.
9. 결과: PV-unaware 대비 평균 read response time 60% 개선, 마이그레이션(추가 write) 오버헤드 평균 2%.
10. 한계: 시뮬레이션(Disksim)만, 부호·디코더 알고리즘 자체는 미변경, HW 설계 없음, 특정 sensing-level latency 모델 가정.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| PV 블록 그루핑 / 데이터 배치(FTL) | 대응 없음 | Prime ECC는 디코더/인코더 bit-exact 시뮬로 FTL 배치 계층이 없음 |
| sensing-level 수 → read latency | [channel.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `Set_R_Offset()`, `Set_LLR_Th()` | soft-read threshold 설정과 개념적 접점뿐, 알고리즘 이식 아님 |
| CLOCK read-cold 축출 | 대응 없음 | 컨트롤러 펌웨어 정책, ECC 엔진 무관 |

적용 가치: **낮음** - 부호설계·디코더·HW 어느 레이어도 건드리지 않는 FTL 데이터 배치 정책이라 Prime ECC 3.1에 이식할 요소가 없음.
