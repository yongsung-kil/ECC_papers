### B. 알고리즘 요약

1. 시스템: LLR BP(Jacobi log 근사) 복호 시뮬, 대상 부호는 Margulis `(2640,1320) R=1/2`, QC `(22200,5550) R=3/4`, QC `(22200,7400) R=2/3`.
2. 문제: error-floor level·Eb/N0 추정을 Monte Carlo로 하면 고SNR에서 오류 이벤트가 희소해 수 주~수개월 소요.
3. 핵심 원리: importance sampling으로 잡음 벡터 분포를 바꿔 특정 trapping set이 자주 발생하도록 유도, likelihood ratio로 보정. `µ = Eq[f(X)p(X)/q(X)]`(식3).
4. 잡음 모델: 잡음 벡터 X는 평균 a·분산 σ 정규분포(식4,5). trapping set 유발 위해 W개 원소 평균을 `Emin`만큼 증가시킨 수정 밀도 `q(xi)`(식6,7).
5. 보정: likelihood ratio `p(x)/q(x)=W(x)`(식8), 단일 TS 오류확률 `µTS=(1/M)Σ f(xk)W(xk)`(식9), 전체는 발견된 모든 TS 합산.
6. 소프트웨어 킷(5단계): OpenCL 멀티스레드 시뮬 기반. (1)미정정 error vector 저장("Bad Codewords") (2)TS Analyser가 그래프의 code node w·odd-degree check u·cyclicity e 산출 (3)e>=0 구조를 PostgreSQL trapping-set DB로 구축 (4)각 TS에 대해 Emin(디코더 실패 최소 impulse 레벨) 탐색 (5)importance sampling으로 FER 추정.
7. Emin 탐색: 초기 E0에서 성공하는 레벨부터 ∆E씩 증가시켜 복호 실패하는 순간을 Emin으로.
8. 검증: MC 결과와 비교. FER 값은 약 1 order 차이나 error-floor 발생 Eb/N0는 동일 → floor 검출 용도로는 유효.
9. 결과: Margulis, Eb/N0=2.2dB에서 `tIS=103+7+23+378=511min`, MC `2023min` → 3.96배 단축, 단 정확도는 약 2 order 저하(`2.24e-7` vs MC `1.60e-5`).
10. 한계: FER 직접 추정엔 부정확, heuristic 보정은 length/rate/type/degree 의존이라 일반화 불가, HW 미설계.

### C. Prime ECC 관련 모듈 핀포인트

대상이 other(시뮬/추정 툴)이므로 핀포인트 대응이 제한적이다.

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| error-floor 추정 / trapping-set 시뮬 워크플로 | 대응 없음 | Prime ECC는 bit-exact HW 시뮬로 error-floor 추정 전용 툴 없음 |
| 잡음/impulse 주입 (Emin 탐색) | [channel.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `Set_R_Offset()` | fixed-error 패턴 주입과 개념 유사하나 IS 방법론은 미보유 |
| BP 복호 실행부 | [decoder.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `LDPC_Decoder()` | 추정 대상 복호기일 뿐, 논문이 decoder를 개선하지 않음 |

적용 가치: 낮음 — 디코더·부호·HW를 개선하지 않는 error-floor 추정 시뮬 도구이며 정확도도 제한적이라 Prime ECC 이식 대상이 아니다(검증 워크플로 참고 정도).
