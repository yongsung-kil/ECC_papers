### B. 알고리즘 요약

1. 대상은 QC-LDPC min-sum 디코더로 base matrix + Z×Z circulant(expansion factor) 구조, 802.16e rate 1/2·확장인자 64 등 다양한 rate(1/2~7/8) 지원.
2. 문제: HLS는 개발생산성이 높으나 저수준 기술 부족으로 sub-optimal RTL을 내어 유연성과 throughput을 동시에 얻기 어렵다.
3. 가정: flood 스케줄 min-sum, LLR 반복 갱신, HBM 기반 FPGA(Xilinx U50), off-chip 메모리 접근 지연(~70cycle) 존재.
4. 핵심 기법 1단: dataflow 중심 multi-unit framework - 각 Decoding Core(DC)는 I/O 모듈+다수 Decoding Unit(DU)로 구성, HBM 폭에 맞춰 DU 수를 유연 조정(코드 파라미터를 FPGA에 비고정).
5. 의미: HBM-aware data-frame으로 서로 다른 블록의 동일 offset LLR을 묶어 critical DU 내부의 재정렬 버퍼·주소지정을 제거해 효율↑.
6. 핵심 기법 2단: fine-grained task-level pipeline의 interleaved-update - 두 codeword를 교차 갱신해 flood의 VNU↔CNU 순환 의존 stall 제거(반복수를 `K*N+N-1`로 확장, K=2).
7. 부수 트릭: ping-pong update(MA를 2배 depth로 두어 CN/VN 주소범위 비중첩), indirect vectorization(HLS 오탐 의존성 회피), two-stage addressing(intra-block 병렬 시 barrel shifter로 II=1 달성).
8. 학습/최적화: 학습 없음. HLS pragma(DATAFLOW/ARRAY_PARTITION/UNROLL)와 DC의 SLR 배치·HBM 채널 할당으로 선형 확장.
9. 결과: Xilinx U50에서 동일 파라미터 SOTA 대비 4x~84x, 실측 최대 116.51Gbps, DC 선형 확장(11.8x), DU 병렬4에서 처리량 포화.
10. 한계: HLS/FPGA 개발·throughput 최적화로 부호설계·정정능력·min-sum 알고리즘에는 기여 없음. HBM/FPGA 전제라 ASIC 컨트롤러와 다름.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| min-sum QC-LDPC 복호 이터레이션(flood) | [decoder.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `LDPC_Decoder()`, `C2V_Cal()` | 동일 min-sum 복호 대상이나 본 논문 기여는 HLS 파이프라인 최적화라 알고리즘 대응은 없음 |
| interleaved-update / ping-pong 파이프라인 | 대응 없음 | Prime ECC는 bit-exact HW pipeline 6-stage 자체 모델을 가지며 HLS/FPGA 스케줄 기법은 직접 대응 모듈 없음 |
| CN/VN 메모리(MA)·블록 순환 shift 주소지정 | 대응 없음 | Prime ECC는 z=32 고정 HW 구조로 HLS two-stage addressing/HBM data-frame과 정합 안 됨 |

적용 가치: **낮음** - min-sum 부호·정정성능이 아닌 HLS/FPGA 개발생산성·throughput 최적화 기법이라 ASIC 지향 bit-exact Prime ECC 시뮬레이터에 떼어낼 알고리즘 요소가 없다.
