> 총평: NAND read 성능을 위한 FTL 레벨 차등 코드rate 정책 논문으로, 디코더/부호설계 이식성이 없어 Prime ECC 관점에서 가치 낮음.

### B. 알고리즘 요약

1. 시스템: 3D TLC NAND(16GB, 8채널/2칩/2플레인, 페이지 16KB), QC-LDPC ECC, 데이터 크기 16KB 기준.
2. 문제: 고rate LDPC는 정정능력이 약해 read retry가 잦아 read latency가 지수적으로 증가; 저rate LDPC는 강하지만 parity/전력 비용 과다.
3. 핵심 아이디어(DECC): 접근 특성별로 서로 다른 code rate LDPC를 차등 적용 — 핫 리드+retry 데이터는 강한 `ECC_S`(rate 0.8), 그 외는 `ECC_N`(rate 0.9).
4. DECC 테이블: `ECC_S`(0.8, 핫 리드용)와 `ECC_N`(0.9, 일반용)을 상태-of-art AR4JA 코드에서 합성한 code rate 시퀀스로 구성(단일 회로로 다중 rate 지원, ref [22]).
5. 핫 리드 식별: read count 기반, 업데이트 사이 read 횟수가 임계값 `T_R`(=4) 초과 시 핫으로 판정; 업데이트 시 카운터 리셋("sluggish" 식별).
6. Re-coding: 핫+retry 데이터를 `ECC_N`으로 디코딩 후 `ECC_S`로 재인코딩해 write back; GC/WL 배경 작업 또는 retry 시점에 트리거.
7. 구현 오버헤드: 페이지당 ECC 강도 구분 1bit + read counter 3bit(매핑 테이블 예약영역), parity 추가공간 (0.9-0.8)*100%=10%.
8. 평가: FEMU 에뮬레이터, FIO+Zipfian 스큐 워크로드, 확률 기반 retry 모델(x/y/z), retry 최대 3회로 제한.
9. 결과: IOPS를 세 retry 모델에서 41.7%/15.7%/16.2% 개선, UECC 대비 전력 32% 감소, re-coding 오버헤드 0.5~1%.
10. 한계: 디코더/부호설계 알고리즘 신규 없음(코드rate 선택 정책+re-coding); HW 미설계, 에뮬레이터 시뮬만; 3D QLC 검증은 future work.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 데이터별 code rate 선택(DECC 테이블) | 대응 없음 | Prime ECC는 특정 QC-LDPC H-matrix 고정, FTL 레벨 rate 스위칭 개념 없음 |
| 핫 리드 식별 + re-coding | 대응 없음 | FTL/컨트롤러 정책 레이어로 디코더/인코더 코드와 무관 |
| AR4JA 기반 다중 rate 부호 | [ecc_top.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `Load_PCM()` | 부호구조 로드 지점이나, 다중 rate 합성 자체는 이식 부담 높음 |

적용 가치: 낮음 — 부호설계/디코더 알고리즘이 아니라 FTL 차등 정책이라 Prime ECC 디코더/인코더 레이어에 떼어낼 요소가 없다.
