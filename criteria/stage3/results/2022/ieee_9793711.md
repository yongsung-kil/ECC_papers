### B. 알고리즘 요약

1. 시스템: IoT 엣지 디바이스, edge intelligence 목표. 통신 IC의 디코더가 평시 50%+ 유휴라는 점을 이용.
2. 문제: 엣지에서 NN 추론용 전용 HW는 전력·면적·비용 과다, 클라우드 오프로딩은 통신 부하·프라이버시 문제.
3. 관찰: convolutional 디코더(Viterbi)의 trellis 구조가 NN과 아키텍처적으로 유사(ACS의 min-sum vs 뉴런의 MAC).
4. 핵심기법: Viterbi stage의 BMU에 곱셈기를 추가, XOR `x_i⊕c_k,j`를 벡터곱 `x_i·w_i,j`로 치환하고 기대코드워드 `c_k,j`를 학습가중치 `w_i,j`로 대체 → 디코더 datapath를 뉴런으로 재사용, TBU는 미사용.
5. 핵심식: 뉴런 경로 = 첫 stage부터 마지막 stage까지 연결된 node/edge 집합, 식 (5)로 feature/weight batching 적용.
6. 확장: MNP(최대 비중첩 뉴런경로, 보통 4~128)와 constraint length `K`(3~15)로 큰 NN을 batching(weight/feature) 처리, link tagging으로 MNP 확대.
7. 구현: 8-bit 정수 양자화(정확도 손실 0~0.2%), pipeline 구조 + 3개 FSM(feature/weight/accumulator 제어), Altera 28nm Cyclone V FPGA에 Verilog 합성.
8. 학습: Python quantization-aware training으로 8-bit weight 학습.
9. 결과: MNIST(81 feature) 97 cycle에 96.2% 정확도, 11.25 MACs/cycle, baseline 대비 power overhead 7.72%(K=3)/12.47%(K=9), logic overhead 53.88%/80.54%, 전용 NN HW 대비 최대 19배 전력·7.7배 logic 절감.
10. 한계: convolutional Viterbi 디코더 대상(LDPC 아님), K=27/81은 곱셈기 증가로 FPGA 미합성, 면적/전력 오버헤드 큼.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Viterbi trellis/ACS를 NN MAC로 재활용 | 대응 없음 | NN/딥러닝 디코더 계열, Prime ECC min-sum LDPC와 무관 |
| convolutional(Viterbi) 디코더 HW | 대응 없음 | Prime ECC는 QC-LDPC message-passing 전용, convolutional code 미대상 |
| FPGA pipeline/FSM 재사용 아키텍처 | 대응 없음 | LDPC pipeline(z=32/HCU/GT)과 구조 상이, 재활용 대상은 ECC가 아닌 NN |

적용 가치: **낮음** — 대상이 유휴 Viterbi 디코더의 NN 가속 재활용으로 NAND binary QC-LDPC 부호설계/디코더/HW 어디에도 접점이 없다.
