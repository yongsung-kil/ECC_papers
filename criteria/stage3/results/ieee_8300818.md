### B. 알고리즘 요약

1. 대상은 regular `(3,6)` binary LDPC (열가중치 `wc=3`, 행가중치 `wr=6`), rate `1/2`, BPSK+AWGN, LLR 입력.
2. 문제: min-sum은 SP 대비 구현 단순하나 성능 열화 + LDPC HW의 메모리·라우팅 자원이 병목.
3. 핵심 기법: CN 메시지는 각 CN에 연결된 6개 VN 중 최소·차최소값만 존재 → `(min, submin, index)`만 저장하는 공용집합(common set)으로 압축, 메모리 절감.
4. CNU 구조: 6-input MSM(min/sub-min) 모듈을 2-input 비교기+MUX 트리(`⌈log2(N-1)⌉=3`층, 2-input 5개)로 구성, 부호비트는 XOR로 계산.
5. 성능 보상: degree-matched 근사(식 6) — 입력 LLR에서 CN degree 의존 보정계수 `β`를 빼서 min-sum 과대추정 완화, `(3,6)` 부호는 `β=0.15`.
6. VNU 구조: index로 min/submin 선택 후 재구성(식 7), 4bit sign-magnitude를 6bit 2's complement로 변환해 덧셈 오버플로(최대 28) 처리 후 다시 4bit로 환원.
7. 양자화: 4bit(sign 1 + magnitude 3), ripple-carry adder로 합산, 합의 부호가 hard decision.
8. 메모리: multiplexed memory + LUT로 H의 열정보를 (m,n) 형태로 저장, RAM R/W 로직이 주소 생성 → slice 자원 절감.
9. 결과: Spartan-6 합성, LDPC decoder 2496 slice LUT / 826 slice register, 최대 87MHz(11.54ns), 10 iteration 수렴, CNU 53 LUT / VNU 90 LUT.
10. 한계: CN·VN 각 1개뿐인 직렬 구조라 처리속도 희생, 부호length·BER 곡선·throughput 수치 없음, 성능은 SP 대비 여전히 열화.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| min-sum CN 연산 (min/submin, sign XOR) | `decoder.cpp` `CNU_Update_New_Mag()`, `C2V_Cal_New_Sgn()` | 우리도 min1/min2+sign XOR 기보유 — 신규성 낮음 |
| degree-matched 보정계수 β | `decoder.cpp` `CNU_Update_New_Mag()` | scaling/보정 factor 삽입 지점, 우리 magnitude 테이블과 접점이나 β값은 재튜닝 필요 |
| (min,submin) 압축 저장 | `ecc_data.h` `PCM_b`, `decoder.h` `Get_VNU_*` | 메시지 압축 아이디어이나 우리 pipeline/GT가 이미 edge 축소 담당 |
| VNU 4bit 양자화·오버플로 처리 | `decoder.cpp` `VN_Cal_HD()` 등, `ecc_data.h` `PARAM_LLR` | 양자화폭·2's complement 변환은 우리 LLR 테이블 설계와 접점 |
| H 열정보 LUT 저장(multiplexed mem) | `GT.cpp` `Make_GT_HW()` | MUX 기반 edge/메모리 접근과 유사, 우리 GT가 상위호환 |

적용 가치: **낮음** — min-sum·(min,submin)·보정계수·4bit 양자화·MUX 메모리 모두 Prime ECC(§3)가 이미 보유, (3,6) 직렬 소형 구현이라 고rate QC-LDPC 파이프라인에 새로 이식할 실질 요소가 없음.
