### B. 알고리즘 요약

1. 대상: concatenated zigzag code(triplet 파라미터, 예시 rate-1/2), 이를 bipartite graph로 표현해 LDPC로 간주. 시뮬 부호는 `(16384,4,4)`.
2. 문제: turbo(BCJR/serial)는 수렴 빠르나 직렬, LDPC sum-product는 완전병렬이나 수렴 느림 → 병렬성 유지하면서 수렴속도를 높이는 스케줄이 필요.
3. 부호 관점: concatenated zigzag를 check node degree와 degree(`I`,2) 두 종류 variable node를 갖는 LDPC로 봄. degree-1 마지막 parity는 N→∞에서 비율 0으로 무시.
4. 임계값: density evolution으로 rate-1/2 concatenated zigzag의 threshold `0.906 dB`(Shannon limit에서 약 0.7 dB)로 계산.
5. 복호 스킴 정의: T-I(turbo/직렬), T-II(표준 sum-product, LDPC 스케줄). T-II는 병렬·저전력이나 zigzag trellis 구조를 못 살림.
6. 핵심 기법: zigzag는 본질적으로 accumulate code → 그 message-passing이 병렬 실현 가능(성능 손실 미미). check node에서 "check operation" `⊞`로 인접 정보노드 prior + 채널 LLR을 결합, forward/backward pass `(4)(5)`를 병렬 갱신.
7. 개선 스킴: (4)(5) 병렬 갱신을 적용해 MT-II(완전병렬) 도출. sub-decoder는 병렬 갱신하되 sub-code 간 successive 갱신을 유지하면 MT-I(모듈 1개 공유로 HW비용/시간 trade-off).
8. MT-I 트릭: sub-code 간 successive 갱신 시 최신 정보비트 LLR을 즉시 사용하도록 식(3)을 (6)으로 갱신 → 성능 개선.
9. 결과: `(16384,4,4)` 부호 시뮬(Fig.3)에서 수렴속도 차이 확인, T-II·MT-II·MT-I·T-I 순. parity node 메시지가 check node 도달 전 전역 확산되어 MT계열이 원 sum-product보다 빠르게 수렴. BER 자체 이득은 주장 안 함.
10. 한계: HW 미설계, Table I 복잡도·Fig.3 iteration별 수치 본문 서술 부재(추출 손상). zigzag/RA 특수 구조 전용, 일반 LDPC 스케줄 확장은 open problem으로 남김.
