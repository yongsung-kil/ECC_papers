### B. 알고리즘 요약

1. 시스템: (1024,3,6) regular LDPC(rate-1/2) 디코더 ASIC, 완전병렬 배선복잡도를 피하려는 메모리 기반 arch.
2. 문제: 완전병렬(Blanksby, 52.5mm2)은 배선 폭증, 완전직렬(Yeo)은 iter당 9612 cycle·거대 MUX. 병렬성과 배선단순화를 동시에 원함.
3. 핵심 관찰: sum-product의 vertical/horizontal step은 "같은 row/col 원소 전체 합 - 자기값 뺄셈"으로 분해 가능.
4. Loosely-coupled 이중 데이터플로우: summation을 제외한 처리유닛을 이중화, 각 path는 row 합 또는 col 합만 계산해 상대 path에 **합값만** 전달(모든 메시지 교환 X) → 배선 대폭 감소.
5. write-back 이점: 중간값을 읽은 동일 메모리 주소로 되쓸 수 있어 별도 인덱스 생성 불필요.
6. 양자화: LLR robustness로 `ǻ,ȁ`를 5bit(정수2+소수3)로 표현 충분, 같은 row/col 메시지를 한 word로 concat해 1회 read/write로 처리.
7. Segmentation: 메모리를 분할해 read/write 대기사이클 제거, 병렬 처리될 row/col을 서로 다른 segment에 배치.
8. 스케줄링: column을 공통 row 많은 순으로 그룹화 → 필요 row 최소 그룹부터 배치, 삽입된 그룹 row 제거 반복(휴리스틱); 그룹수로 면적-처리시간 trade-off.
9. 구현: log(tanh(x)), atanh(exp(x)) LUT + adder, 64 병렬 처리; (1024,3,6)에서 853Mbps/4.75mm2@0.25um(Blanksby 1Gbps/52.5mm2 대비 대폭 면적↓), col 합의 부호비트로 hard-decision 출력.
10. 한계: regular·rate-1/2 부호, SP기반 LUT(min-sum 아님), gatecount 미제공, NAND 무관, iteration은 8로 임의 선택.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 이중 데이터플로우·row/col summation 디코더 루프 | [decoder.cpp](../Prime_ECC_3.1_Claude/) `LDPC_Decoder()`, `C2V_Cal()` | CN/VN 교대 스케줄과 개념 연결, 단 Prime ECC는 Dual-Update(parity even/odd) 방식 |
| LLR 5bit 양자화 concat | [ecc_data.h](../Prime_ECC_3.1_Claude/) `PARAM_LLR`, [decoder.cpp](../Prime_ECC_3.1_Claude/) `VN_Cal_HD()` 등 | 양자화 테이블 개념 유사, Prime ECC는 magnitude 양자화 테이블 |
| segmented memory·병렬 스케줄링 | 디코더 HW 모델 (z=32) | 메모리 배치·병렬화 아이디어는 z=32 row 병렬과 정합 검토 필요 |
| log(tanh)/atanh(exp) LUT (Sum-Product) | 대응 없음 | Prime ECC는 min-sum(min1/min2) 사용, full-tanh SP는 미채택 |

적용 가치: **낮음** — 완전병렬 대비 면적을 줄인 메모리 기반 SP 디코더 arch이나, Prime ECC는 이미 min-sum·양자화 테이블·병렬 스케줄을 보유하고 부호도 고rate QC로 상이하다.
