### B. 알고리즘 요약

1. 대상은 일반 sparse `M×N` parity-check matrix로 정의된 binary LDPC 부호, 채널은 BSC(오류확률 `p`).
2. 문제: BP 복호는 H-matrix 크기에 선형 비례하는 시간복잡도를 가져 큰 부호에서 CPU 복호가 느림.
3. 핵심 기법: NVIDIA CUDA 프로그래밍 모델로 GPU에서 message-passing을 병렬화 — bit-node/check-node 각각에 1 thread 할당해 모든 노드를 동시 갱신.
4. BP 절차: check-node에서 `r_ij(b)`, bit-node에서 `q_ji(b)` 확률을 반복 교환하고 `Q_j(b)=K_j·P_j·∏r_ij` 로 추정 비트 `x̂_j` 결정.
5. 곱셈을 덧셈으로 대체하는 log-domain 최적화 적용(`r_ij` 갱신을 로그영역으로).
6. CUDA 워크플로우: host(CPU)→device(GPU) global memory 복사, 확률 초기화·codeword 추정·`r_ij`/`q_ji` 계산·비트 갱신을 각 커널로 병렬 실행 후 결과 회수.
7. 구현 디테일: shared memory로 다수 곱셈 가속, mapping array로 각 노드의 접근 위치 인덱싱, global memory latency는 thread scheduler로 은닉.
8. (학습/최적화 절차 없음)
9. 결과: Intel Core Duo 2GHz + NVIDIA GT GPU 환경에서 H-matrix가 커질수록 GPU 실행시간은 완만한 선형 증가, CPU는 급증 → 대형 부호에서 GPU가 수배 빠름(정확 배수는 표 손상으로 미상). 작은 부호에서는 GPU≈CPU.
10. 한계: 소프트웨어 GPU 가속뿐이며 HW(RTL/ASIC) 설계 없음, 표준 BP라 정정성능·iteration 개선 없음, 부호 rate/length·speedup 수치가 추출 손상.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |

| log-domain sum-product 연산 | 대응 없음 | Prime ECC는 min-sum(min1/min2+sign) 근사 사용, full-tanh/log BP 미채용 |
| GPU thread-per-node 병렬화 | 대응 없음 | Prime ECC는 HW z=32 row 병렬(HCU/GT), GPU 플랫폼과 무관 |

적용 가치: **낮음** — 표준 BP의 GPU 가속으로 복호 알고리즘·HW 아키텍처 기여가 없고, Prime ECC의 min-sum HW 디코더에 이식할 요소가 없다.
