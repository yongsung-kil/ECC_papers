### B. 알고리즘 요약

1. QC-LDPC 프로토그래프(base bipartite graph, lifting factor `z`)에서 girth/trapping set 제어를 위해 route(경로) 열거 소프트웨어를 개발.
2. 문제: lifted 그래프(수천 배 큰 그래프)에서 직접 사이클을 찾는 것은 계산 비용이 크므로, 리프팅 전 프로토그래프 수준에서 route와 lifted 사이클의 대응 관계[6]를 이용해 효율화.
3. 핵심 가정: 길이 `g`의 route는 base equation(BE)이 특정 조건을 만족하는 edge 시퀀스이며, route 하나가 lifted 그래프의 `q`개 사이클에 대응.
4. 1단계 기법: 길이 8/10 route를 길이 4/6 짧은 사이클 쌍의 조합으로 표현해 계산량을 줄이는 방식([7],[8] 기반) — BE 덧셈/뺄셈 1회로 조건 확인 가능.
5. 이 표현이 필요한 이유: 특정 타입의 route(예: girth-8 trapping set 유발 route)만 선택적으로 제거하면서 다른 사이클은 유지하는 세밀한 코드 설계가 가능해짐.
6. 2단계 확장: 길이 12 이상에서는 짧은 사이클 조합으로 표현 불가하므로 트리 기반 route 탐색 알고리즘[13] 도입, 단 동일 route가 여러 branch에서 중복 검출되는 문제 발생.
7. 구현 디테일: route를 BE의 양(+)/음(-) 계수 부분으로 분리 후 각각 오름차순 정렬해 중복(및 "mirror" route) 판별 최적화 — 완전연결 프로토그래프(dv=2,dc=3) 예제에서 route 수를 길이 8/10/12 각각 11.8%/42.5%/69.1% 감축.
8. 최적화 절차는 학습이 아닌 알고리즘/자료구조 개선(정렬 기반 중복 제거)이며 GA/DE/NN 없음.
9. 결과: dv=6, dc∈[9,11], g∈[12,14] 앙상블에서 트리 기반 방식이 짧은사이클-조합 방식 대비 14배 이상 빠름(Intel i7-8550U, 16GB RAM).
10. 한계: 순수 소프트웨어/그래프 이론 도구로 디코더 성능(BER, error-floor) 실측 없음, 부호 rate/length 구체적 사례 미제시, parallel edge가 있는 프로토그래프는 미지원.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 프로토그래프 route/사이클 열거, trapping set 유발 구조 검출 | `ecc_top.cpp` `Load_PCM()` | H-matrix 설계·검증 단계에 해당하나 Prime ECC는 특정 QC-LDPC 고정 사용, 새 H-matrix 설계는 고난도 이식 |
| girth/trapping set 튜닝 파라미터 조절 | 대응 없음 | 코드 자체가 아닌 오프라인 설계 툴이라 실행 코드에 대응 모듈 없음 |
| 사이클 길이별 error-floor 영향 분석 | 대응 없음 | 본 논문은 BER/error-floor 실측을 제공하지 않아 정량적 대응 불가 |

적용 가치: **중간** — 디코더/HW 개선은 없으나, elementary trapping set 제거를 통한 error-floor 개선 설계 방법론으로서 향후 Prime ECC H-matrix 재설계 시 참고할 수 있음.
