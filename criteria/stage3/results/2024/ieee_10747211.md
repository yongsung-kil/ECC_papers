### B. 알고리즘 요약

1. 대상은 DVB-S2(x) 위성통신 표준의 QC-LDPC 코드(`N=16200~64800`), CPU(Intel/ARM) SIMD 멀티코어에서 소프트웨어로 디코딩.
2. 기존 CPU/GPU LDPC 소프트웨어 구현은 DVB 특유의 double~quadruple diagonal 서브행렬 때문에 flooding 또는 inter-frame layered 방식만 써서 latency·메모리 대역폭 문제가 컸음.
3. 채널/노이즈 모델은 별도 정의 없이 기존 표준 통신 채널 가정, 핵심은 알고리즘이 아니라 소프트웨어 구현 전략.
4. Offset Min-Sum + horizontal-layered scheduling(HLS)을 채택, `Kernel 4.1`(min1/min2, cn_sign 계산) → `Kernel 4.2`(Lmn 메시지 생성·En 갱신) 구조.
5. VN 데이터를 `Z/F × F` 블록으로 재배치 저장해 회전(rotation) 시 SIMD load/store 오버헤드를 줄임(`blendv` 마스킹 로드 방식).
6. DVB 특유의 2~4중 대각 서브행렬은 짝/홀 iteration마다 다른 CN→VN 메시지를 갱신·동결(freeze)하는 `H1(odd)/H1(even)` 분리 기법으로 처리.
7. 펑처링된 서브행렬은 대상 VN을 `-∞`로 중립화해 병렬 처리 유지, unrolling 대신 함수 포인터 기반 branchless 커널 선택으로 명령어 캐시 미스 최소화.
8. 별도 학습/최적화 절차 없음(고정 알고리즘의 구현 최적화).
9. 결과: 단일코어 최대 500Mbps, 멀티코어(Intel Xeon P3) 최대 11.48Gbps, latency 최소 31μs; 기존 CPU/GPU 대비 최대 5배 처리량, 최대 64배 latency 개선.
10. 한계: 부호설계·복호 알고리즘 자체 개선 없음(순수 SW 구현), FPGA/ASIC 대비 여전히 처리량 열위이며 하드웨어 구조 설계는 다루지 않음.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Offset Min-Sum 알고리즘 | `decoder.cpp` `LDPC_Decoder()`, `CNU_Update_New_Mag()` | Prime ECC도 min-sum 계열 보유(이미 구현), 신규 기여 없음 |
| Horizontal-layered scheduling | `decoder.cpp` `LDPC_Decoder()` | Prime ECC도 layered/dual-update 방식 이미 보유 |
| CPU SIMD 메모리 매핑/병렬화 | 대응 없음 | Prime ECC는 RTL/ASIC bit-exact 시뮬레이터로 CPU SIMD 소프트웨어 아키텍처와 레이어가 다름 |
| DVB 특유 다중 대각 서브행렬 처리 | 대응 없음 | Prime ECC 부호는 QC-LDPC 단일 identity 기반(HCU만 특수), DVB 구조와 직접 대응 없음 |

> 적용 가치: 낮음 — CPU 소프트웨어 구현 최적화 논문으로 알고리즘·정정성능·HW arch 기여가 없어 binary NAND LDPC ECC(HW) 이식 관점에서 참고할 요소가 거의 없음.
