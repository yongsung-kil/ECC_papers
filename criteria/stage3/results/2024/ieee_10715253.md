### B. 알고리즘 요약 (10줄 내외)

1. 대상 부호는 CCSDS-231 규격의 binary regular LDPC 3종: (128,64), (256,128), (512,256), rate 0.5, BSC(binary symmetric channel) 채널 가정.
2. 풀려는 문제는 데이터 이동 병목(memory wall)으로, 프로세서-메모리 간 데이터 전송이 병렬 처리 성능의 제약 요인이 되는 것.
3. 핵심 가정은 UPMEM PiM 아키텍처의 저복잡도 ALU(정수 연산만 지원)가 Min-Sum보다 bit-flipping(BF) 알고리즘에 더 적합하다는 것.
4. 핵심 기법은 parallel multi-BF(Gallager BF) 디코딩: 각 CN에서 syndrome `s`를 modulo-2 덧셈으로 계산, 각 VN에서 unsatisfied check 수 `f_n`을 합산.
5. 핵심 식은 `f_n > t`일 때 해당 비트 `z_n`을 flip하는 규칙이며, threshold `t`는 최대 VN degree `max(d_v)`로 초기화 후 flip 없는 iteration에서 감소시키는 단일 적응형 threshold.
6. 확장 기법 없음(단일 BF 알고리즘), 다만 threshold를 적응형 단일값으로 단순화해 PiM ALU의 저복잡도 특성에 맞춤.
7. 구현 디테일은 그래프 연결정보와 변수를 64KB WRAM(SRAM 스크래치패드)에 저장(64MB MRAM 대비 4배 성능↑), DPU당 16 threads로 syndrome/unsatisfied-check/bit-flip 단계를 병렬 처리하고 threshold 계산만 단일 thread.
8. 별도 학습/최적화 절차 없음(고정 알고리즘, HW 매핑 최적화가 핵심).
9. 결과는 단일 디코더 기준 WRAM+16thread가 MRAM+1thread 대비 최대 17배 향상(1.099 vs 0.030 Mbps @(512,256)), 멀티코드워드 병렬(2557 DPU)로 2.814 Gbps 달성 — 동일 PiM 상의 MS 디코더(0.743 Gbps) 대비 약 4배 높은 처리량.
10. 한계는 hard-decision 전용(soft-info 미지원)으로 정정성능 개선은 다루지 않으며, UPMEM 클록(350MHz)이 낮아 단일 디코더 성능은 GPU 대비 열세(고성능은 대량 병렬화로만 달성).

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Bit-flipping 디코딩 알고리즘(syndrome/unsatisfied-check/flip) | `decoder.cpp` `LDPC_Decoder()` | Prime ECC는 Min-Sum 기반이라 BF는 별개 알고리즘 계열, 대체재로서 참고만 가능(중복 아님이나 hard-1bit로 정정성능은 열세) |
| 적응형 단일 threshold | `ecc_data.h` `PARAM_LLR`, `decoder.cpp` `Get_VNU_Table_Idx()` | 개념적으로 iteration별 LLR threshold 테이블과 유사하나 BF는 정수 카운트 기반이라 직접 대응 아님 |
| WRAM/MRAM 메모리 계층 분리(그래프 데이터 SRAM 배치) | 디코더 HW 모델(z=32, HCU, GT) | 메모리 라우팅/배치 전략 아이디어로 참고 가능하나 DRAM PiM 특유 구조라 NAND controller ASIC에 직접 이식은 어려움 |
| 멀티스레드 파이프라인 병렬화(16 threads, 동기화 barrier) | 디코더 HW 모델(pipeline 6-stage) | 병렬화 스케줄링 아이디어 참고 수준, PiM 전용 아키텍처 종속 |
```

적용 가치: **중간** — hard-decision BF 자체는 Prime ECC의 min-sum과 다른 알고리즘 계열이고 정정성능 기여가 없어 디코더 알고리즘 이식 가치는 낮지만, 메모리 계층 활용·병렬 스케줄링 아이디어는 HW 아키텍처 관점에서 참고할 만함.
