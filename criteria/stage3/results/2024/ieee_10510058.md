### B. 알고리즘 요약

1. 대상은 임의의 LDPC 코드 팩터그래프(Tanner graph); 예시로 Margulis code(2640,1320), Margulis code(4896,2474, rate~0.505), Mackay code(408,204, rate=0.5), QC-LDPC(2560,2048) 등 다양한 rate/length에서 검증.
2. 풀려는 문제: BP 디코딩 실패의 원인인 trapping set `TS(a,b)`(a개 변수노드 중 b개가 홀수차수 체크노드에 연결된 부분그래프)를 찾는 것은 NP-hard이며, 기존 완전탐색은 `C(n,a)` 조합 폭발, 기존 ILP 방법(Velazquez-Subramani)도 지수적 복잡도.
3. 핵심 가정: 최단 사이클(shortest cycle)에 속하는 변수노드 리스트(`cyclevns`, EMD가 작은 노드)를 사전 지정하면 mixed-ILP 솔버(presolver)가 불가능 조건을 조기에 대량 제거할 수 있다는 것.
4. 핵심 기법: 목적함수 `min(f^T·x)`로 `TS(a,b)`의 `a`를 최소화하는 mixed-integer linear programming 정식화, `x_i∈{0,1}`는 변수노드가 trapping set에 속하는지 여부, 등식 제약 `Aeq·x=beq` (식 2~4)에 패리티체크 행렬 `H`와 사전 리스트 `cyclevns`의 위치 마스크(`Neq`)를 반영.
5. 식(3)의 `Aeq = [2Im, Im, -H; 0, 0, I]` 구조는 체크노드의 짝/홀 차수 판정(2m개 정수변수)과 변수노드 선택(n개 이진변수)을 동시에 만족하는 해를 강제 — 이 제약이 없으면 exponential search가 필요.
6. cyclevns 선정에 두 가지 전략: (1) Fast strategy — lollipop cycle count(인접행렬 거듭제곱)로 ACE(Approximated EMD) 최소 노드 선정, (2) Accurate strategy — EMD Spectrum 방법으로 EMD 최소 노드 선정.
7. 구현: Python + IBM CPLEX(Community/Commercial), GitHub 공개, 16코어 AMD Ryzen 3950X·128GB RAM·32스레드 환경에서 실행.
8. 별도 학습 절차 없음 (조합최적화/ILP 솔버 기반, 알고리즘 자체는 정확해 탐색).
9. 결과: Margulis(2640,1320)에서 TS(6,6)을 0.29초에 발견(Velazquez-Subramani 대비 15082배 가속), Margulis(4896,2474)에서 TS(62,16)·TS(52,14)를 최초 발견(기존 대비 2266배 가속), Mackay(408,204, rate=0.5)에서 TS(102,2)·TS(108,4) 발견(완전탐색 시 `C(408,108)≈1.13×10^101` 필요).
10. 한계: 순수 이론/알고리즘 논문(HW 미설계, 시뮬 결과 없음), 발견한 trapping set을 실제로 제거하는 부호 재설계 절차는 제시하지 않고 탐색 자체가 목적, 상용 CPLEX(Commercial Edition) 의존 및 대형 코드에서의 확장성은 별도 검증 필요.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| trapping set 탐색(부호구조 검증) | `ecc_top.cpp` `Load_PCM()` | 로드된 H-matrix(`PCM_b`)에 대해 오프라인 분석 도구로 적용 가능 — trapping set 사전 검출로 error-floor 취약점을 부호 검증 단계에서 파악 |
| 짧은 사이클/EMD 기반 변수노드 선정 | 대응 없음 | 코드 구성(construction) 단계 보조 도구로, 현재 코드베이스에 해당 사이클 카운트/EMD 계산 모듈 없음 |
| trapping set 제거/회피 부호 재설계 | 대응 없음 (논문이 다루지 않음) | 이 논문은 탐색만 하고 재설계는 하지 않아 코드 대응 지점 자체가 없음 |
| corner-case/수렴실패 추적 | `corner.cpp` | 실제 디코딩 시뮬레이션 중 수렴 실패 case 수집 로직과 목적(error-floor 원인 식별) 유사하나, 논문은 시뮬레이션이 아닌 정적 그래프 분석 |

> 적용 가치: 중간 — 실제 디코더/HW 로직에 바로 이식되진 않지만, Prime ECC가 사용 중인 QC-LDPC H-matrix의 trapping set을 오프라인으로 사전 검증해 error-floor 리스크를 부호설계/검증 단계에서 조기 발견하는 보조 도구로 활용 가치 있음.
