### B. 알고리즘 요약

1. 대상: rate-1/3, 8-state tailbiting LTE turbo code, BPSK/AWGN 채널, 50~2048비트 프레임 길이.
2. 문제: 기존 Stochastic Turbo Decoder(STD, [36])는 근사 오류정정 성능은 좋으나 처리 latency가 매우 크고(최대 25만 DC), 전원전압 변동에 의한 timing error/metastability에 취약.
3. 채널/오류 모델: 전원전압을 평균 `μ`, 표준편차 `σ`인 가우시안으로 모델링(`3σ/μ`=3~7%), 클록주기 초과 시 랜덤 비트값이 해당 DFF에 클록인되는 timing-error 모델.
4. 핵심 기법 1단(timing-error tolerant STD): output synchronizer(2-DFF)로 metastability 전파 차단, 2개 프레임 동시 디코딩으로 처리량 손실 보상, TFM(Tracking Forecast Memory) 기반 Edge Memory 도입, δ모듈에 파이프라인 스테이지 추가.
5. 핵심 식 `P(t+1) = (1-φ)P(t) + φJ(t)`(TFM 갱신), 이동평균으로 확률을 추적해 SR(shift-register) 기반 Edge Memory보다 latching 문제에 강함, φ=2^-4일 때 9-DFF로 구현(SR 대비 32→9 DFF 절감).
6. 핵심 기법 2단(Reduced-Latency STD, RLSTD): OR게이트 기반 근사 stochastic 가산기(MUX 가산기 대비 착출 BS 길이 단축), Reduced-Complexity TFM(φ=2^-1로 고정해 shift 연산만으로 구현).
7. 부수 트릭: Noise-Dependent Scaling(η=1, ψ=2)으로 채널 확률 재조정해 latching 완화, Clock Gating으로 동적 에너지 절감, 4비트 saturated up/down counter를 RLSTD에서 단일 DFF로 축소.
8. 학습/최적화 절차 없음(고정 파라미터 φ, η, ψ 수동 설정 및 실험적 튜닝).
9. 결과: (TFM-5, 1.20V, 2.2ns, 3σ/μ=7%)가 기존 SR-1(1.20V, 4.0ns, no variation) 대비 처리량 2.42배, 에너지 0.25배 달성. RLSTD는 SR-1 대비 DC수 0.035배, latency 0.015배, throughput 65배(200비트 프레임은 78배), chip area 0.23배.
10. 한계: 실측 아님(post-layout gate-level 시뮬레이션, TSMC 90nm), turbo code 전용(LDPC 아님), stochastic computing 패러다임 전체가 Prime ECC의 fixed-point min-sum 구조와 근본적으로 다름.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Stochastic computing 기반 디코딩 전체 | 대응 없음 | Prime ECC는 fixed-point min-sum 디코더이며 stochastic bitstream 연산 방식 자체를 채택하지 않음 |
| TFM 기반 Edge Memory (latching 방지) | 대응 없음 | min-sum 구조에는 latching 문제가 없어 대응 개념 없음 |
| timing-error tolerance (synchronizer/전원변동 내성) | decoder.cpp LDPC_Decoder() | 개념적으로만 유사(전원변동에 강건한 HW 설계 아이디어), 실제 회로 기법은 stochastic 전용이라 직접 이식 불가 |
| 파이프라이닝 | decoder.cpp pipeline 6-stage(col_M1/Z/P1~P4) | Prime ECC도 자체 6-stage pipeline 보유, 다만 이 논문의 파이프라이닝은 stochastic APP 계산 전용 |
```

> 적용 가치: **낮음** — turbo code + stochastic computing 조합으로 대상 부호·연산 패러다임 모두 Prime ECC의 binary QC-LDPC min-sum 구조와 이질적이며, 코드 자체의 정정능력 개선도 없음(timing-error 내성/latency 개선만 다룸).
