### B. 알고리즘 요약

1. 시스템: enterprise NVM 스토리지, 8 SSD x 16 channel x 4 chip, raw 32TB, CW 8KB 고정, 64k bit codeword 실험 참조.
2. 문제: 채널마다 LDPC 디코더를 두면 채널 수에 비례해 HW 복잡도가 폭증, 특히 variable-rate 지원 시 심각.
3. 모델: PE cycle(노화)에 따라 raw BER이 증가하고 목표 UBER `10^-14`를 유지하려면 강한(저rate) 부호가 필요하다는 비선형 관계.
4. 핵심 기법1(적응 rate): 6개 rate(5/6, 3/4, 2/3, 1/2, 1/3, 1/4)를 노화에 따라 전환, iteration이 임계 초과 시 다음(강한) rate로 스위칭.
5. 지표: `LTC = (Endurance x User Capacity)/WAF` (WAF=1 가정), fixed rate 대비 lifetime capacity 향상 계산.
6. 핵심 기법2(디코더 pool): 채널별 전용 디코더 대신 N개 디코더 pool을 PCIe 스위치로 모든 SSD가 공유(N << M·R), 오류 검출 시 idle 디코더 할당.
7. 데이터 매핑: UP를 512B DB로 쪼개 rate별 DW/CW 구성, M1(UP를 CW로만 분할)과 M2(NVM page 내 CW 혼합 허용) 두 방식 비교.
8. 최적화: iteration 상한(10/20/30/50)별 LTC와 개선 factor 표로 분석, 상한이 LTC에 미치는 영향은 10-15%로 작음.
9. 결과: adaptive가 fixed 대비 LTC 개선 factor 1.6~2.8배, lifetime 2.30년, normalized data rate 8.4 GBps.
10. 한계: transaction/capacity 시뮬 분석뿐, 디코더는 FPGA/GPU HW accelerator로 blackbox 처리하고 실제 HW·부호 구조 미설계.

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 노화 적응 rate 전환 정책 | 대응 없음 | Prime ECC는 단일 고정 rate QC-LDPC 엔진, multi-rate 전환 상위 정책 없음 |
| PCIe 디코더 pool 아키텍처 | 대응 없음 | 채널 간 디코더 공유는 컨트롤러 SoC 레벨, 디코더 코어 밖 |
| variable-rate QC-LDPC 부호 | ecc_top.cpp Load_PCM() | rate별 H-matrix 로드 개념뿐, 실제 다중 rate 구성법은 본문 부재 |
```

적용 가치: 낮음 - 스토리지 시스템 아키텍처·LTC 분석 논문이라 binary QC-LDPC 디코더 코어나 부호설계에 이식할 알고리즘 실체가 없음.
