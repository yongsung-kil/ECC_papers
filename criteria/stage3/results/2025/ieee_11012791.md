### B. 알고리즘 요약

1. Static PPM은 PPM 압축의 빈도표를 고정(static)해 블록 단위 독립 복호(부분 복호)를 가능케 한 방식이며, 압축 데이터는 중복성이 낮아 오류에 취약하다.
2. 기존 방법(convolutional code + Viterbi)은 다중 후보를 생성해 복호 후 압축해제한 빈도표와 헤더의 빈도표를 비교, 불일치 시 차순위 후보로 재시도하나 Viterbi는 연산·메모리 비용이 크다.
3. 본 논문은 채널/노이즈 모델을 별도 정의하지 않고, Static PPM 블록을 LDPC 부호화 후 오류가 발생한 상황을 가정한다.
4. 핵심 기법: 인코딩은 (1) Static PPM으로 압축 (2) 블록을 LDPC로 부호화하는 2단계.
5. 디코딩은 Sum-Product 디코딩을 고정 iteration 수행 후 수렴 실패 시 "중간 결과 후보들"을 계산량(반복 횟수) 순으로 저장해두는 것이 핵심 - 특정 수식 제시 없이 절차로만 기술.
6. 확장 단계: 최다 계산 후보부터 압축해제 → 빈도표 재구성 → 헤더 빈도표와 비교, 불일치 시 차순위 후보로 교체 반복(Step 3~4 루프).
7. 부수 트릭: LDPC의 sparse parity-check 행렬과 병렬 가능한 Sum-Product 특성을 이용해 Viterbi 대비 연산시간 단축을 기대한다는 서술뿐, 정량적 HW/시간 측정 없음.
8. 학습/최적화 절차 없음.
9. 결과(수치): 압축률 비교만 제시 - English/DNA/Sources 3개 소스파일에서 제안 방법 압축률이 Static PPM 대비 약 1.5배(기존 convolutional 방법은 약 2배)로, LDPC 부가 오버헤드가 기존 방법보다 낮음을 시사. 계산시간/메모리는 정성적 기대만 서술(정량 데이터 없음).
10. 한계: LDPC 부호 자체의 rate/length/구조/정정능력이 본문에 전혀 명시되지 않고, 계산시간·메모리 절감도 실측치 없이 정성적 주장에 그침. HW 설계 없음.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Sum-Product(표준 BP) 디코딩 채용 | `decoder.cpp` `LDPC_Decoder()` | Prime ECC는 min-sum 기반이라 표준 BP(tanh) 자체는 이미 대체된 대상, 이 논문은 알고리즘 개선 없음 |
| 수렴 실패 시 다중 후보 저장·재선택(응용계층 CRC 대체) | `partial_CRC.cpp` `partial_crc_HW_T4()` | 개념적으로 조기종료/실패판정과 유사하나 여기선 압축헤더 빈도표 비교로 대체되어 NAND ECC decoder 로직과 무관 |
| LDPC 부호 구조 (rate/length 미기재) | `ecc_top.cpp` `Load_PCM()` | 본문에 H-matrix 정보 없어 대응 판단 불가 |

> 적용 가치: 낮음 - LDPC를 표준 Sum-Product 그대로 사용한 데이터압축 응용 논문으로 디코더/부호설계 개선이 없어 Prime ECC 이식 대상이 아님.
