### B. 알고리즘 요약

1. 5G NR LDPC(QC-LDPC, BG2 base graph, lifting size `Zc`, circulant permutation matrix 기반), 짧은 블록 길이(URLLC/mMTC向) 데이터 채널 대상.
2. 문제: BG 설계는 이상적 디코더(무제한 iteration) 가정 하 DE/EXIT 분석으로 최적화되었으나, 실제로는 유한 블록 길이·제한된 iteration 수 하에서 정보비트 펑처링 효과가 이상적 가정과 달라질 수 있음.
3. 채널/모델: AWGN 채널, BG matrix의 첫 2열(가장 높은 열 가중치, A/B/C/I 영역 구조)에 대한 고정 펑처링 전략 적용.
4. 핵심 기법: BG의 처음 두 열(고weight column)을 펑처링하여 Tanner 그래프의 short cycle 발생 빈도를 억제 — short cycle은 반복 복호 중 메시지가 제한된 노드 집합 내에서 순환해 outdated 정보를 반복 전파시켜 성능을 저하시킴.
5. 핵심 관찰: 펑처 비트 수가 데이터 길이 대비 낮은 비율(전형적으로 `Zc`, `2Zc` 펑처)일 때, 펑처링은 유효 코드율을 낮추는 효과를 겸해 non-punctured 대비 BER/BLER를 개선하나, 펑처된 노드는 초기 LLR이 없어(0으로 시작) 수렴이 느려짐.
6. 확장 비교: `Zc` 펑처 vs `2Zc` 펑처 두 케이스 비교 — 저 SNR에서는 `2Zc` 펑처가, 고 SNR에서는 `Zc` 펑처가 우세(코드길이 N=132/(문서 손상으로 정확 수치 불명, "N=1"로 표기된 부분)에 따라 경향 상이).
7. 부수 관찰: 코드 길이가 길어질수록 최소 해밍 거리 증가로 정정능력이 향상되고 short cycle 영향이 상대적으로 약화되어 LLR 수렴이 빨라짐 — 이 경우 펑처링에 의한 iteration 증가 부담이 완화됨.
8. 별도 학습/최적화 절차는 없음(순수 시뮬레이션 기반 정성·정량 비교).
9. 결과: 짧은 데이터 길이·제한된 최대 iteration 조건에서 `Zc` 펑처가 `2Zc` 펑처보다 우수한 BLER을 보이는 것으로 요약(구체 BER/BLER 수치는 본문 텍스트 손상으로 확인 불가, Fig.2/Fig.3 참조 필요).
10. 한계: HW 미설계·순수 시뮬레이션, 5G NR BG2 표준 구조에 국한, 본 텍스트 추출본이 폰트 인코딩 손상으로 정량 수치 다수(N값, iteration 수, SNR 범위 등) 및 저자 정보가 소실됨.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 정보비트 펑처링(첫 2열, Zc 단위) 정책 | `ecc_top.cpp` `Load_PCM()` | Prime ECC도 `punct_col_num = 2`로 이미 유사한 펑처 컬럼 구조를 채택하고 있어 펑처 비트 수(Zc vs 2Zc) 정책 비교가 직접 참고 가능 |
| short cycle 억제를 위한 고weight column 펑처 | `ecc_top.cpp` `Load_PCM()` (H-matrix 구조) | H-matrix 구조 설계 단계 참고용이나 Prime ECC는 특정 QC-LDPC 고정이라 재검증 부담 큼 |
| 펑처 노드 초기 LLR 부재로 인한 수렴 지연 | `decoder.cpp` `Get_VNU_Table_Idx()`, `ecc_data.h` `PARAM_LLR` | iteration별 LLR 테이블 설계 시 펑처 컬럼(col 127/128 HCU)의 초기값 처리 정책과 연관 |
| 유한 iteration 하 short-block 코드율 트레이드오프 분석 | 대응 없음 | 분석/방법론 참고용이며 특정 코드 모듈에 대응되지 않음 |

적용 가치: `중간` — Prime ECC가 이미 punct_col_num=2 구조를 갖고 있어 본 논문의 펑처 비트 수(Zc/2Zc) 대비 성능 트레이드오프 분석 결과는 Prime ECC의 펑처 컬럼 초기 LLR·iteration 정책 튜닝에 직접적 참고가 되나, 부호설계(H-matrix) 자체의 이식은 불가.
