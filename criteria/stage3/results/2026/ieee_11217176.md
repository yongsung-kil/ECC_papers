### B. 알고리즘 요약

1. 시스템: QC-LDPC 디코더, 802.16e 표준 기반 code rate 0.5~0.9, codeword 1536 symbol(GF(2)), block size 64(24블록), min-sum 알고리즘, flooding scheduling.
2. 문제: FPGA 기반 LDPC 디코더는 최악 채널조건 대응을 위해 고정된 큰 반복횟수를 쓰거나, 조기종료(early termination)를 적용하면 동적 실행시간 때문에 확장 가능한 병렬 구조를 만들기 어려움(HLS는 결정론적 실행에 유리, 동적 실행엔 취약).
3. 핵심 가정: HLS(`DATAFLOW`, `hls::stream`, `UNROLL`, `PIPELINE` 등 pragma)의 제약(제어로직 미지원, 큐 블로킹) 안에서 가변 반복 디코딩 유닛(DU)을 구현해야 함.
4. 핵심 기법 1단: 3계층 아키텍처 - (i) 저수준 VNU/CNU를 HLS 함수로 추상화해 다양한 LDPC 파라미터·비트폭에 적응, (ii) 중간수준 DU는 8단계 파이프라인(read→VNU→write→CNU→write→TC 판정→출력)에 termination-check(TC) 스테이지 통합.
5. 핵심 식: CNU는 `sign XOR`+최소값 선택(min-sum), VNU는 `output[i+1] = sum - input[i+1]`(saturation 포함) - TC를 파이프라인 안에 넣어 조기종료 검사 지연을 숨김.
6. 핵심 기법 2단: 고수준에서 FIFO 큐(`dataFIFO`,`indexFIFO`)로 input-DU-output을 loosely coupled 시켜 다중 독립 디코딩 스트림 구성, DU별 가변 반복시간에도 순서보장(in-order output)·블로킹 회피(free-blocking streams).
7. 부수 트릭: VNU-TC 간 메모리 접근 충돌 해결을 위한 duplicate OMB(outward memory block array) + `ARRAY_PARTITION`으로 read/write 포트 증설; DU 내 멀티 파이프라인(DU-ET-N)으로 코드워드 병렬 디코딩.
8. 학습/최적화 절차: 없음(알고리즘 학습 아님, HW 아키텍처/HLS pragma 설계).
9. 결과: Xilinx U55C 구현, 최대 49.5Gb/s 처리량(HF-LDPC 45.6Gb/s 대비 우세), 기존 조기종료 SOTA 대비 최대 5배 처리량, 802.16e rate 0.5/length 1536 기준 4/6/8-bit 양자화에서 BER 성능 유지, DU-pipeline 수는 정정성능에 거의 영향 없음.
10. 한계: min-sum 알고리즘·조기종료 판정 자체는 기존 기법 재사용(신규 정정능력 개선 아님), NAND는 서론에서 배경으로만 언급되고 실제 실험은 802.16e/5G 표준 부호 기준(FPGA 시뮬레이션 데이터), ASIC 합성/실측 없음.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| min-sum CNU/VNU 연산(sign XOR + 최소값 선택) | `decoder.cpp` `CNU_Update_New_Mag()`, `C2V_Cal_New_Sgn()` | 동일한 min-sum 근사 방식을 사용하나 이미 보유 기법이라 신규 기여 아님 |
| 조기종료(termination-check, TC) 스테이지 | `partial_CRC.cpp` `partial_crc_HW_T4()`, `full_CRC.cpp` | Prime ECC는 CRC 기반 조기종료를 이미 보유, 본 논문의 TC는 syndrome(parity-check) 기반이라 판정 메커니즘 상이하지만 "이터레이션 중 조기종료"라는 목적은 동일 |
| DU 파이프라인/멀티 디코딩 스트림(병렬 코드워드 처리) | `decoder.cpp` `LDPC_Decoder()` 이터레이션 루프, HW pipeline(6-stage) 구조 | 우리 6-stage pipeline과 유사한 파이프라이닝 개념이나, 멀티 코드워드 병렬 스트림 구조는 Prime ECC 문서상 미확인(단일 코드워드 위주) |
| VNU-TC 메모리 접근 충돌 회피(duplicate OMB) | 대응 없음 | HLS 특유의 컴파일러 제약 회피 기법으로 수기 RTL bit-exact 시뮬레이터 구조와 직접 대응되지 않음 |
| flood scheduling(전체 VN→전체 CN 갱신) | `decoder.cpp` `LDPC_Decoder()` | Dual-Update(even/odd 교대)와는 다른 스케줄링 방식, 참고 가능하나 우리는 이미 Dual-Update 보유 |

적용 가치: `중간` — min-sum·조기종료 자체는 이미 보유한 기법이라 알고리즘 이식 가치는 낮으나, 멀티 디코딩 스트림·파이프라인 내 TC 통합·out-of-order 처리 등 HW 아키텍처 설계 아이디어는 스루풋·유연성 개선 관점에서 참고 가치가 있음.
