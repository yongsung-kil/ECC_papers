### B. 알고리즘 요약

1. 대상: Polar 코드의 Belief-Propagation(BP) 디코더 HW. 팩터그래프는 `log2 N` stage, 각 stage에 `N/2` PE. 부호길이는 짧은~긴 Polar 코드 비교(구체 N 본문 텍스트 손상으로 미상).
2. 문제: 하드웨어용 Scaled-Min-Sum(SMS)은 부호길이가 길어질수록 `g(x,y)` 근사오차 누적으로 성능 저하, 또 PE의 계산경로(critical path)가 길어 지연 큼.
3. BP 갱신: 좌우 전파 메시지 `L_ij`, `R_ij`를 `g(x,y)=ln[(1+e^{x+y})/(e^x+e^y)]` 기반으로 갱신, frozen 위치는 `R`을 큰 상수/0으로 설정.
4. SMS 근사: `|g(x,y)| ≈ s·min(|x|,|y|)` (scaling factor `s`로 min-sum 근사오차 보정). 그러나 min 대체로 여전히 상당한 오차.
5. 계산경로 단축: `a=b⊞c`에서 `|a|`의 가능값이 `|b|,|c|` 중 하나 뿐임을 이용, 세 후보를 첫 Tadder에서 병렬 계산 후 부호 판정으로 선택 → SC/CS 변환 유닛 제거, PE 경로를 `4Tadder→3Tadder`로 단축(Type-I/II 블록).
6. LUT 알고리즘: `LUT(x,y)=min(|x|,|y|)+f(|x|+|y|)-f(||x|-|y||)`, `f(z)=ln(1+e^{-|z|})` (식 `(15)(16)`), SMS 근사오차를 상쇄. `f(z)` 값테이블을 축약(Table I→II)해 자원 절감.
7. LUT-SMS: 부호길이별로 LUT와 SMS를 조합(Table III) - 짧은 부호는 SMS, 긴 부호는 LUT 위주로 waterfall 영역 성능 유지.
8. 근사 우월성: `ERROR_LUT(x,y) < ERROR_SMS(x,y)`인 영역(흰색)이 반대(검정)보다 훨씬 넓음을 오차맵으로 제시.
9. 결과: 긴 Polar 부호에서 LUT-SMS가 SMS 대비 추가 복호이득(수치 텍스트 손상으로 정확값 미상, "extra dB gain" 서술). 수정 PE 아키텍처는 계산경로 단축으로 clock↑, 어떤 Polar 디코더에서 평균 throughput 향상(정확 % 손상). ASIC 파라미터(nm/V, Ling adder Tadder=ns) 언급.
10. 한계: Polar 코드 BP 전용(LDPC 아님), 계산유닛이 전체 디코더 면적의 작은 비중이라 긴 부호에선 메모리 read/write가 지연 지배. 텍스트 인코딩 손상으로 다수 수치 확인 불가.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Polar BP 디코더 전체 | 대응 없음 | Prime은 binary QC-LDPC min-sum 디코더, Polar BP는 부호·팩터그래프 구조 자체가 상이 |
| SMS min-sum 근사 + scaling | [decoder.cpp](../Prime_ECC_3.1_Claude/paper_screening_profile.md) `CNU_Update_New_Mag()` | 개념적으로만 min-sum 유사 - Polar의 g(x,y) 갱신식은 LDPC CN 연산과 다름 |
| f(z) 보정 LUT | 대응 없음 | LDPC Min-Sum의 offset/normalized 보정과 발상은 유사하나 Polar 전용 테이블이라 직접 대응부 없음 |
| PE 계산경로 단축(HW) | 대응 없음 | Polar PE 아키텍처 특화, Prime의 z=32 QC-LDPC pipeline과 무관 |

적용 가치: **낮음** - Polar 코드 BP 디코더 전용이라 binary QC-LDPC인 Prime ECC와 부호 구조가 근본적으로 다르며, LUT 보정·PE 경로 단축 모두 Polar 특화로 직접 이식 불가.
