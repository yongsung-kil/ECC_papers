### B. 알고리즘 요약

1. 시스템은 regular `(n, d_v, d_c)` LDPC 부호(GF(2)) 기반, 시뮬 예시는 QC-LDPC `(3744,3,6)` rate 1/2와 `(3744,3,9)` rate 1/3.
2. 문제의식: sum-product 반복복호는 대부분 블록을 정확히 복호하지만, 실패하는 잔여 블록은 오류 비트 수가 적지 않아 BER이 0으로 점근하지 않는 error-floor 문제가 있음.
3. 채널모델: AWGN + BPSK, sum-product 알고리즘, 최대 반복 20회.
4. 핵심 기법: 원본 블록 `s`개를 한 그룹으로 묶어 `A=[a_0 H, a_1 H, ..., a_{s-1} H]`를 정의하고, 벡터 `Γ=[1,1,...,1]`로 sum-check 블록 `a_s = A·Γ^T`(즉 s개 블록의 XOR)을 계산해 매 s개 원본 블록마다 1개씩 추가 전송.
5. 핵심 식: 수신측에서 그룹 내 정확히 1개 블록(`a_j`)만 오류로 검출되면, `â_j = (Â·Γ^T) XOR â_s`(식 5)로 해당 블록을 복구 — sum-check 블록이 나머지 정상 블록들의 XOR합이라는 성질을 이용한 오류정정.
6. 확장 없음 — 단일 outer-code 기법(2단 알고리즘 아님), 그룹 내 오류블록이 0개 또는 2개 이상이면 sum-check 복호를 생략.
7. 부수 구현 디테일: 송신측은 피드백 시프트레지스터+XOR게이트로 sum-check 블록 생성(카운터 a/b/c로 모드 전환); 수신측은 ping-pong 버퍼(Buffer-0/1, structure-0/1 대칭 구조)로 그룹 단위 스트리밍 처리 및 오류블록 인덱스를 주소 레지스터에 저장.
8. 최적화 절차 없음 — 수학적 확률 분석(식 7~12, Mackay의 undetected error bound 이론 적용)으로 성능을 해석적으로 예측.
9. 결과: 블록오류확률(BEP)이 원래 LDPC 대비 대폭 감소(표 1, 예: q=1e-4, s=10일 때 BEP≈9.996e-8); 새 rate는 `R'=s/(s+1)·R`로 저하; s가 작을수록 성능이 좋으나 rate 손실은 커짐.
10. 한계: 시뮬레이션(AWGN)만 검증, "HW 회로"는 개념도 수준(XOR게이트/시프트레지스터/버퍼) 블록다이어그램뿐 게이트카운트·타이밍·합성 결과 없음; 그룹 내 2개 이상 블록오류 시 무력.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| sum-check(XOR) outer 블록 추가로 error-floor 개선 | decoder.cpp LDPC_Decoder() | Prime ECC는 codeword 단위 min-sum 복호이며 다중 블록 그룹 개념이 없어 대응 없음 |
| 조기 종료/오류 검출(그룹 내 오류블록 수 판정) | partial_CRC.cpp partial_crc_HW_T4(), full_CRC.cpp | 오류블록 존재여부 판정이라는 역할은 CRC 조기종료와 유사하나 메커니즘(XOR 복구 vs CRC 검증)은 상이 |
| 송수신 XOR/시프트레지스터/버퍼 회로 | (대응 없음) | Prime ECC 코드베이스에 대응하는 다중블록 그룹 버퍼링 모듈 없음 |
```

> 적용 가치: 낮음 — outer XOR 코드로 error-floor를 낮추는 아이디어 자체는 단순하지만 단일 codeword 기반의 Prime ECC 구조와 전제가 다르고(다중 블록 그룹 필요), rate 손실 대비 이득도 불명확.
