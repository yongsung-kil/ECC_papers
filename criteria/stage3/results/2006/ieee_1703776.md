### B. 알고리즘 요약

1. 대상은 LDPC convolutional code(LDPC-CC) — 무한하지만 주기적인 parity-check 행렬 `H[0,∞]`로 정의되며, rate 1/2, `(128,3,6)` 코드(row weight 3, column weight 6, memory `M=128`)를 실증 대상으로 사용.
2. 문제: LDPC-CC는 블록코드와 달리 스트리밍 데이터를 임의 길이로 인코딩 가능하지만, 인코더를 all-zero state로 되돌리는 termination sequence가 필요하며 이를 실시간 VLSI 회로로 효율적으로 생성하는 방법이 없었음(기존엔 linear system solver로 전체 시퀀스를 병렬로 한 번에 계산할 뿐).
3. 인코더를 FSM으로 모델링: 상태 `s(t)`는 최근 `M`개의 정보 비트와 코드 비트로 구성, 각 시점 phase `φ(t)`에 따라 주기적으로 동작.
4. 핵심 기법: termination bit는 인코더 상태에 대한 XOR 연산의 함수이므로, all-zero state와 각 single-one state(비트 하나만 1인 상태)에 대해 개별적으로 termination sequence를 구해(선형대수로 `T(s,φ)` 계산) 그 결과들을 중첩(superposition, XOR)해 임의 상태에 대한 termination 회로를 구성.
5. 핵심 식: `s(t)·H_termination = 0`(all-zero 상태로 귀결되는 조건, 식 12)을 만족하는 `M+1`비트 종료 시퀀스를 선형대수로 계산 — 이 결과를 회로화(Algorithm 1)하는 것이 논문의 기여.
6. 확장 없음(단일 기법), 대신 최적화 단계에서 XOR 항 축소를 위한 backward/partial substitution 기법 적용.
7. HW 구현 디테일: 원 인코더(Fig.2, FIFO 지연선 + XOR 게이트)에 추가 delay element, XOR 게이트, mux를 덧붙여 termination 회로 구성(Fig.6); ROM 기반 설계(Fig.7)로 mux 대신 온칩 메모리 활용해 FPGA 자원 절감.
8. 최적화 절차: (a) 마지막 몇 비트는 XOR 입력 대부분이 0이 되는 성질을 이용한 완전 치환, (b) 나머지 항에 대한 후방대입(back-substitution) 부분 치환 — 총 27% 게이트 절감.
9. 결과: termination 회로가 인코더 크기를 FPGA 기준 7배, ASIC 기준 8배 증가시키나 20-processor 디코더 대비는 절반 이하 크기; 0.18μm CMOS에서 최대 189MHz 클럭 동작 가능; ROM 기반 설계는 최적화 회로 대비 LE(logic element) 절반 이하로 축소.
10. 한계: 부호설계나 디코더 성능(BER/error-floor)과 무관한 순수 인코더 종료(termination) 회로 문제이며, 이 문제 자체가 block-based QC-LDPC에는 존재하지 않음(블록 경계가 명확해 termination 불필요).

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| LDPC-CC 인코더 termination 회로 | 대응 없음 | Prime ECC는 block-based QC-LDPC로 dual-diagonal 인코딩(`encoder.cpp` `LDPC_encoder_4KB()`) 사용, termination 개념 자체가 없음 |
| FSM 기반 인코더 구조 | `encoder.cpp` `Generate_PCM_encoding()` | 개념적으로 인코더 회로라는 점만 공유, LDPC-CC 특유의 FIFO delay-line 구조는 대응 없음 |
```

> 적용 가치: 낮음 — LDPC convolutional code(SC-LDPC 계열) 고유의 인코더 종료 문제로, Prime ECC가 다루는 block-based QC-LDPC 인코더 구조(`Column_Reordering_Dual_Update()` 등)와 근본적으로 다른 부호 클래스를 대상으로 함.
