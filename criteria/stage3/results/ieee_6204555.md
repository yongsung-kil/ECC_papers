> 총평: LDPC 코드의 error-floor 원인인 최소 trapping set과 critical number를 CPLEX IP로 정확히 찾아내는 부호 분석·설계 도구로, 디코더/HW 자체를 개선하는 논문은 아니며 코드 설계 단계에서 trapping set 제거 대상을 식별하는 오프라인 분석 기법.

### B. 알고리즘 요약

1. 대상: 일반 (n,k) LDPC 코드의 Tanner 그래프, 특히 사례 연구로 Tanner (155,64) quasi-cyclic 코드 사용.
2. 문제: error floor를 유발하는 최소 크기 trapping set 탐색은 NP-hard이며, trapping set은 거의 모든 parity check를 만족시켜 탐색이 어려움.
3. 핵심 가정/모델: 하드결정(hard-decision) 메시지 패싱(Gallager-B류) 디코더를 기준으로 trapping set을 정의하며, Richardson의 결과에 따라 AWGN 소프트결정 디코더도 동일 trapping set에 걸린다고 가정.
4. 핵심 기법: `vi`(수신 벡터 비트), `ci`(check node 오류 여부) 이진 변수와 `ki`(정수 보조변수, mod-2 표현용)를 도입해 최소 trapping set을 정수계획법으로 정식화: 목적함수 `min Σvi`, 제약 `vH^T + c = 2k`, `cH ≤ y/2`, `Σci ≥ 1`.
5. 식의 의미: `Σci ≥ 1` 제약으로 유효 코드워드(0 trapping set)를 배제하고, `cH ≤ y/2` 제약으로 각 비트 노드의 이웃 check node 과반수가 정상(짝수) 상태가 되도록 강제해 실제 trapping set 조건(디코더가 갇히는 조건)을 구현.
6. 확장 기법(critical number 탐색): 최소 trapping set을 찾은 후 Algorithm 1로 해당 trapping set 내에서 디코더를 진동시키는 최소 오류 비트 그룹(critical number)을 Hamming weight 순으로 탐색, 코드 테이블(과거 방문 벡터 기록)로 중복 계산 방지.
7. 구현 디테일: mod-2 비선형 제약을 정수변수 `k`(2배 곱)로 선형화해 표준 IP 솔버(CPLEX)로 풀 수 있게 정식화.
8. 최적화 절차: CPLEX 12.3으로 IP 모델을 풀어 최소 trapping set 크기(예: Tanner 155,64 코드에서 5)를 탐색, 이후 Algorithm 1로 해당 trapping set의 critical number(예: 2) 산출.
9. 결과: 순열행렬 기반 (2,2) trapping set 포함 다양한 크기 H-matrix에서 모두 성공적으로 최소 trapping set 발견, 코드 길이 증가에도 실행시간이 실용적 수준(Figure 3). Tanner(155,64) 코드에서 최소 trapping set 크기 5, critical number 2로 확인.
10. 한계: trapping set 탐색·분석 도구일 뿐 실제 코드 개선(제거)은 별도 covering 방법([4] 참조)에 의존하며, 실시간 디코딩과 무관한 오프라인 코드 설계 단계 전용 기법. HW/성능 검증 없음.
