### B. 알고리즘 요약 (10줄 내외)

1. 시스템: regular (3,6) LDPC 코드, `n=1008` (일부 실험 `n=204~2640`), rate 0.5, BPSK+AWGN, full-precision belief-propagation(BP) 복호.
2. 문제: 기존 flooding/serial(Serial-V, Serial-C) 스케줄은 variable/check node 단위 고정 순서로만 메시지를 갱신해 성능·수렴 여지가 제한적임(Forney의 미해결 질문: 스케줄로 반복복호 개선 가능한가).
3. 핵심 아이디어: edge `e_cv`를 스케줄링의 "원자 단위"로 삼는 일반화된 Serial-E 스케줄 제안. `Serial-V`, `Serial-C`는 Serial-E의 특수 사례.
4. 핵심 기법: 매 iteration 초기에 edge 리스트를 정렬하고, `k_sort`개 메시지 처리마다 남은 fresh edge를 재정렬(재정렬 주기 `k_sort`가 성능-복잡도 트레이드오프 제어).
5. 핵심 식: `m_v^(l) = m_v + Σ m_{c'v}^(l-1)` 갱신에서 edge `e_cv`가 가져오는 증분 `Δm_{v/c}^(l)`를 정렬 기준으로 사용(식 (5)) — 이 값이 클수록 해당 edge를 먼저 처리해야 LLR이 빨리 개선됨.
6. 이론적 상한: genie-aided 스케줄(전송 codeword를 알고 `Δm` 부호가 맞는 edge를 우선 처리) 도입, 실현 불가능하지만 스케줄링 이득의 tight upper bound로 사용.
7. 실용 스케줄: 확률적 정렬(메시지 정확 확률 기반)은 불안정해 실패, 대안으로 differential schedule(`Δm_{v/c}^(l-1)` 절대값 기준 정렬)을 제안해 실용성 확보.
8. 분석 기법: 유한장 코드에 맞춘 Gaussian Approximation(GA) 일반화 — 한 iteration을 `n·d_v`개의 edge-step으로 쪼개 단계별 LLR 기대값(Lemma 2)과 확률벡터를 결합해 첫 iteration 성능을 해석적으로 예측.
9. 결과: genie-aided 스케줄은 단일 iteration에서 BER=10^-3 기준 3dB 이상, 100 iteration 시 serial 대비 0.75dB 이득. 실용 differential schedule은 단일 iteration 기준 약 1.85dB 이득이나 100 iteration에서는 이득 소멸(단, 필요 iteration 수는 serial보다 적음). 짧은 코드일수록 스케줄링 이득이 큼.
10. 한계: 순수 시뮬레이션(HW 미설계), full-precision tanh 기반 BP로 min-sum/양자화 영향 미검토, differential schedule의 실질적 이득은 iteration 수가 커질수록 감소.

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| 메시지 스케줄링(flooding→edge 단위 동적순서) | `decoder.cpp` `LDPC_Decoder()`, `C2V_Cal()` | 현재 Dual-Update(even/odd 교대) 스케줄과 다른 축의 스케줄링 아이디어로 이터레이션 루프 구조 변경 필요 |
| CN/VN 메시지 갱신(식 (1)~(4)) | `decoder.cpp` `CNU_Update_New_Mag()`, `C2V_Cal_New_Sgn()` | full-precision tanh BP이며 Prime ECC는 min-sum이라 메시지 정의는 다르나 스케줄 순서 개념은 이식 가능 |
| edge 재정렬(differential schedule) 로직 | 대응 없음 | 매 스텝 edge 우선순위 재계산·정렬은 현재 pipeline HW(z=32 row 병렬)와 구조적으로 상충, 별도 하드웨어 큐/정렬 로직 필요 |
| iteration별 LLR 테이블 | `ecc_data.h` `PARAM_LLR`, `decoder.cpp` `Get_VNU_Table_Idx()` | 스케줄 도입 시 iteration 정의가 달라져(edge-step 단위) 기존 LLR threshold 테이블 재설계 필요 |
```

적용 가치: 중간 — 스케줄링을 통한 iteration 감소/waterfall 개선 아이디어는 흥미로우나, edge 단위 동적 재정렬은 Prime ECC의 병렬 pipeline(HCU, GT) 구조와 정합이 어렵고 실용 스케줄의 실이득도 제한적.
