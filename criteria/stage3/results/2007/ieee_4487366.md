> 총평: FPGA semi-parallel LDPC 디코더의 메모리 뱅크 충돌을 부호 설계 단계(block-regular 제약)에서 해결한 2007년 초기 연구로, QC-LDPC 기반 Prime ECC와는 부호 구조가 다르지만 메모리 뱅킹/주소생성 아이디어는 참고할 만함.

### B. 알고리즘 요약

1. Semi-parallel LDPC 디코더는 FPGA 자원 제약으로 P개 페이지로 나눠 VNP/CNP를 시분할 재사용하며, 각 엣지 메시지는 P개 메모리 뱅크에 저장된다.
2. 문제: H-matrix의 엣지를 임의로 라벨링하면 한 페이지 처리 시 여러 메시지가 같은 뱅크에 몰려 충돌(collision)이 생기고, 이를 피하려면 clock cycle이 추가로 필요해 throughput이 떨어진다.
3. 핵심 가정: 코드는 regular LDPC이며, H-matrix를 `(n-k)/P × n/P` 크기의 서브블록(intersection) `P×P`개로 분할할 수 있다.
4. 핵심 기법: 각 서브블록 내 1(edge)의 개수가 동일하도록 구성한 "block-regular" 코드를 설계하면, `Ccirc` 순환치환 행렬을 이용해 엣지 라벨을 `1..E`로 배정할 때 모든 행 집합·열 집합이 항상 완전한 라벨 집합 `{1,...,E}`를 갖게 되어 충돌이 원천적으로 없다.
5. 라벨 하한/상한 식: `lower = (j-1)E + Ccirc(i,j)(EP)+1`, `upper = (j-1)E + Ccirc(i,j)(EP)+(EP)` — 서브블록 (i,j)에 속한 엣지들의 라벨 범위를 결정.
6. 부수 트릭: block-regularity 덕분에 주소생성기가 단순한 P개 카운터 집합으로 구현 가능 (복잡한 주소 계산 로직 불필요).
7. 학습/최적화 절차 없음 (구성적 설계).
8. 결과: length-9984, rate-1/2 코드에서 메시지 저장에 look-up table 9360개 필요 (Virtex4-xc4vlx200의 약 5%). block-regular와 block-irregular 코드의 BER/FER 성능 차이는 미미함 (length-1008, (6,3) regular, 50 iteration min-sum 비교).
9. 한계: HW 합성/실측 없이 시뮬레이션과 자원 추정에 그침. QC-LDPC가 아닌 일반 regular LDPC 대상이며 P=16 고정 사례 중심.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| block-regular 서브블록 제약 부호 구성 | `ecc_top.cpp` `Load_PCM()` | Prime ECC는 QC-LDPC(base+circulant)로 이미 구조적 정합성을 갖춰 이 구성법은 재설계 시 재검증 부담이 큼 |
| 메모리 뱅크 충돌 회피용 엣지 라벨링/주소생성기 | 대응 없음 | Prime ECC는 z_sb=32 row 단위 병렬 구조로 이미 뱅크 충돌 문제를 다른 방식(QC circulant 정렬)으로 해결 |
| VNP/CNP 페이징 구조 | `decoder.cpp` `LDPC_Decoder()` | 개념적으로 유사하나 논문은 semi-parallel 일반 아키텍처, Prime ECC는 고정 z=32 병렬 구조 |

> 적용 가치: 낮음 — Prime ECC는 이미 QC-LDPC circulant 구조로 메모리 정렬 문제를 해결했고, 논문의 block-regular 구성법은 non-QC regular 코드 대상이라 이식 시 H-matrix 전면 재설계가 필요함.
