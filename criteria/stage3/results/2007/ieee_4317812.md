### B. 알고리즘 요약

1. 대상은 일반 regular LDPC 코드(H 행렬 기반), row degree `dr`, column degree `dc` 고정. 예시로 고rate(8/9, 자기저장장치용) 코드도 다룸.
2. 문제: fully-parallel 아키텍처는 처리량은 높지만 배선 복잡도가 매우 커서 면적 오버헤드가 큼(예: Blanksby 디코더 면적의 절반이 배선). Serial 아키텍처가 대안이나 기존 check-node-centric 방식은 메모리 요구량이 큼.
3. 채널/노이즈 모델은 특정하지 않고 표준 sum-product 알고리즘(`Q`, `R` 메시지, φ함수 기반 tanh 근사 없는 정확 sum-product) 기준으로 설계.
4. 핵심 기법: bit-node-centric 아키텍처. 표준 BNU(bit node processing unit)는 순차적으로 `Q_{i,j}` 메시지를 계산하고, check node 연산은 `P_j^k = Σρ_{l,j}^k`, `S_j^k = Πσ_{l,j}^k`로 분해해 매 사이클마다 부분적으로(on the fly) 갱신.
5. 핵심 식: `|R_{j,i}^k| = φ(P_j^k - ρ_{i,j}^k)`, `sign(R_{j,i}^k) = S_j^k × σ_{i,j}^k` — check node의 합/곱 연산을 미리 누적해두면 최종 메시지 계산 시 감산/XOR 한 번으로 끝나 별도 check node 유닛이 불필요.
6. 확장 없음(단일 기법), 대신 기존 check-node-centric(Hocevar, Mansour) 아키텍처와 메모리/사이클 비교.
7. 구현 디테일: check node 연산은 sign-magnitude, bit node 연산은 2's-complement 포맷 사용, 두 FIFO 버퍼(magnitude/sign 각각)로 이전 반복값 유지.
8. 학습/최적화 절차 없음(수학적 메모리량 유도).
9. 결과: 메모리 요구량 `(2m+ndc)q` bit로, check-node-centric `(2n+mdr)q` 대비 rate 8/9, dc=3 예시에서 29mq vs 45mq(약 35~50% 절감). Throughput은 사이클 수만 다르고 유사(짧은 사이클 n개 vs 긴 사이클 m개).
10. 한계: full sum-product(φ 함수, tanh 기반)만 다루며 min-sum 근사 미적용, 게이트카운트/공정/FPGA·ASIC 검증 없이 이론적 메모리 비트수 계산에 그침, regular code만 다룸.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| check node 연산 on-the-fly 분해 (P_j, S_j 누적) | `decoder.cpp` `CNU_Update_New_Mag()`, `C2V_Cal_New_Sgn()` | min-sum 근사가 아닌 정확 sum-product(φ함수) 기반이라 CN 연산 대체는 불가하나 메모리 절감 스케줄링 아이디어는 참고 가능 |
| bit-node-centric serial 스케줄 및 메모리 요구량 비교 | `decoder.cpp` `LDPC_Decoder()` | z=32 row 병렬 구조와 다른 완전 serial 구조라 직접 대응 어려움 |
| sign-magnitude ↔ 2's-complement 포맷 변환 | 대응 없음 | Prime ECC는 magnitude 양자화 테이블 기반으로 별도 포맷 변환 불필요 |

> 적용 가치: 낮음 — full sum-product 기반 메모리 최적화로 이미 min-sum + z=32 병렬 구조를 가진 Prime ECC와 알고리즘 layer가 달라 직접 이식 난이도가 높음.
