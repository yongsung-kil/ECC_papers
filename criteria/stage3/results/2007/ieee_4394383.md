### B. 알고리즘 요약

1. IEEE 802.16e/802.11n 등 다중 rate를 지원해야 하는 시스템向 구조적 LDPC 코드 및 유연한 multi-rate 디코더 설계 문제를 다룸(N=16320bit, rate 1/2~7/8).
2. 문제: 기존 multi-rate 디코더는 (a) rate별 별도 프로세서 설계 또는 (b) 최악 케이스(최대 check node degree) 기준 설계로 복잡도·자원낭비가 큼.
3. 코드 구조를 `H=[H1 H2]`(H2는 staircase, IRA 코드의 systematic 버전)로 정의, outer(SPC)-inner(2-state accumulator trellis) turbo-like 분해로 디코딩.
4. 핵심 기법: parity check matrix를 `Heq=[H1eq H2eq]`로 변환(hidden/punctured 중간비트 `eik` 도입), H1의 row당 1의 개수를 `nJ0`에서 고정 `J0`로 낮춰 SPC 프로세서를 단일 크기로 통일.
5. 핵심 식: `g(a,b)=sign(a)sign(b)min(|a|,|b|)` (Turbo-like Min-Sum), FBA(BCJR-like) forward/backward `Fj`, `Bj` 재귀로 accumulator(inner code) 복호.
6. 확장: sliding window(SW) decoding으로 forward/backward 메트릭 버퍼를 윈도우 크기로 한정, 메모리 요구량을 코드 rate와 무관하게 유지.
7. 부수 트릭: parity 비트에 해당하는 VN은 turbo-like scheduling에서 갱신 불필요(degree-1 취급)해 BP 대비 갱신·메모리 부담 절감.
8. 코드 설계 시 J0=4가 중간 블록길이(100~20000bit)에서 IEEE 802.16/DVB-S2급 평균 check node degree와 유사한 절충안으로 선정(최적화 절차는 경험적 탐색).
9. 결과: 4bit 채널 양자화(0.1dB 손실), FPGA(Altera EP1S80)에서 logic cell 15.3%/memory 19.5% 사용, 50MHz@8iteration 시 12.46Mbit/s, N=16320 다양한 rate에서 BER 1e-10까지 early error floor 없음.
10. 한계: 단일 프로세서 기준 처리량이 낮고(듀플리케이션 필요 언급), turbo-like/trellis 기반 디코딩은 표준 BP/min-sum 대비 근본적으로 다른 아키텍처이며 IRA 특유 구조(H2 staircase)에 의존.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Turbo-like(BCJR/FBA) 디코딩 알고리즘 전체 | 대응 없음 | Prime ECC는 min-sum 기반 BP만 지원, turbo/BCJR 계열 미보유 |
| equivalent parity check matrix 변환(multi-rate) | `ecc_top.cpp` `Load_PCM()` | H-matrix 로드/구조 변경 관점에서만 참고, 실제 변환 로직은 코드 구조 자체를 재설계해야 함 |
| Turbo-like Min-Sum 근사식 `g(a,b)` | `decoder.cpp` `CNU_Update_New_Mag()` | min 근사 아이디어는 유사하나 트렐리스 기반 스케줄과는 별개 |
| Sliding window decoding(메모리 절감) | `decoder.cpp` `LDPC_Decoder()` | 파이프라인/HCU 메모리 관리와 목적은 유사하나 구현 방식 상이 |
| IRA 코드 H-matrix 구성(SPC+accumulator) | 대응 없음 | Prime ECC는 QC-LDPC 고정 구조, IRA 계열 코드설계 미보유 |

> 적용 가치: 낮음 | 이유: IRA 계열 코드에 특화된 turbo-like(BCJR) 디코딩 패러다임으로 Prime ECC의 QC-LDPC min-sum 아키텍처와 구조적으로 상이하며, NAND 타깃도 아님.
