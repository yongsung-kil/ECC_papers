> 총평: syndrome check(정상종료)로는 놓치는 저SNR/트랩 상태를 최근 반복들의 다수결 코드워드와 비교해 조기 포기(early give-up)하는 보완적 조기종료 기법으로, SS/SCR 대비 최대 43% iteration 감소를 보였으며 Prime ECC의 partial-CRC 조기종료(정상종료 전용)와 상호보완적으로 결합 검토할 가치가 있음.

### B. 알고리즘 요약

1. 대상은 짧은 binary LDPC block code (128,64)/(192,96)/(256,128), rate 1/2, CCSDS/Wimax-type 부호, AWGN+BPSK 채널, SPA(sum-product) 디코딩, 최대 iteration 20.
2. 저SNR에서 syndrome check(정상종료)는 codeword가 parity check를 만족하지 못해 조기종료가 거의 발생하지 않아 불필요한 iteration이 낭비되는 문제를 해결.
3. 기존 SCR(sign change ratio)는 MSA 전용이라 SPA에서 무력하고, SS(syndrome stagnancy)는 수천-비트 장문 코드에는 유효하나 수백-비트 짧은 코드에는 부적합하다는 한계를 지적.
4. 핵심 기법: 관측 윈도우 크기 `w`로 과거 디코딩 비트 `c_i^l` (l-w≤l≤l-1)의 비트별 다수결로 `c_MV,i^l`(majority-voted bit)을 계산.
5. 현재 iteration의 디코딩 비트 `c_i^l`과 `c_MV,i^l`이 다른 비트 개수 `nc = num(i | c_MV,i^l ≠ c_i^l)`를 세어, 이것이 "정정 가능성(POC)"이 있는 비트 수를 의미함 — `nc`가 threshold `δ_th` 이하이면 조기 포기.
6. 조기종료 방식 분류: EC(early confirm, 기존 syndrome check)와 EGU(early give-up, 본 논문 제안) 중 EGU에 집중 — 저SNR에서 다수 codeword가 오디코딩될 확률이 높다는 점을 활용.
7. 파라미터: (128,64)는 `δ_th=2`, (192,96)/(256,128)는 `δ_th=1`로 설정(코드 길이별 상이).
8. 별도 학습/최적화 절차는 없고 Monte-Carlo 시뮬레이션으로 임계값을 실험적으로 결정.
9. 결과: SS 대비 최대 43% iteration 감소((128,64) 코드, 1.0dB), (192,96)에서 0dB 약 30% 감소, (256,128)에서 1.0~1.5dB 구간 32~34% 감소, BLER 성능 저하 없음.
10. 한계: HW 미설계·시뮬레이션뿐, SPA(tanh 기반) 기준으로만 검증되어 min-sum 알고리즘에서의 효과는 불명, 매우 낮은 SNR(BLER≈1) 구간에서는 SS가 더 우수함을 저자도 인정.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| majority-vote 기반 조기포기(early give-up) 로직 | [decoder.cpp](Prime_ECC_3.1_Claude) `LDPC_Decoder()` 이터레이션 루프 | Prime ECC는 partial-CRC 기반 정상종료(EC)만 보유, 저SNR 트랩 상태를 잡는 EGU 기법은 대응 없어 보완적으로 추가 검토 가능 |
| 관측 윈도우(w) 기반 과거 디코딩 이력 비교 | [ecc_data.h](Prime_ECC_3.1_Claude) `PARAM_DEC` (반복별 상태 저장 구조 확장 필요) | 현재 프로파일에 과거 iteration 비트 이력 저장 관련 언급 없음 — 신규 저장 구조 필요 |
| 조기종료(CRC) 대비 정정가능비트수(nc) 카운트 방식 | [partial_CRC.cpp](Prime_ECC_3.1_Claude) `partial_crc_HW_T4()` | 기존 CRC 조기종료(성공 codeword 확인용)와 목적이 다름(실패 예상 시 조기 포기) — 병행 사용 시 추가 실패-경로 조기종료로 latency 개선 여지 |

> 적용 가치: 중간. SPA 전제(Prime ECC는 min-sum 사용)라 그대로 이식은 어렵지만, "저SNR 트랩 상태 조기포기"라는 아이디어 자체는 Prime ECC의 기존 정상종료(성공 codeword only)와 상호보완적이며 avg iteration/latency 개선에 기여 가능해 min-sum 버전으로 재검증할 가치가 있음.
