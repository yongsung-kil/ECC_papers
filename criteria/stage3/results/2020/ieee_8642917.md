### B. 알고리즘 요약 (10줄 내외)

1. 시스템: LDPC 기반 NAND flash memory(MLC, 128GB, 8채널 SSD)에서 flash block 간 공정편차(process variation, PV)로 인한 신뢰도(RBER) 차이가 존재.
2. 문제: LDPC는 RBER이 높을수록 더 많은 reference voltage(sensing level, 최대 7단계)로 soft decoding해야 하므로 read latency가 증가하며, 기존 PV-aware wear-leveling(write-hot 데이터를 고신뢰 블록에 배치)은 오히려 read-hot 데이터가 저신뢰 블록에 몰려 read 성능을 저하시킴(PV-unaware 대비 70% 저하).
3. 핵심 가정: 블록 간 신뢰도는 Gaussian 분포를 따르며, 각 블록의 신뢰도는 LDPC 디코딩에 필요한 sensing level(reference voltage 수)로 온라인 측정 가능.
4. 핵심 기법 1: PV-aware block grouping - 블록을 LDPC 디코딩 성공에 필요한 sensing level 수(최대 7단계, 실험은 최대 7-group)에 따라 M개 그룹으로 분류, group ID는 로그 M 비트로 저장.
5. 핵심 기법 2: read-hotness 기반 데이터 배치 - 페이지가 threshold `T`(기본값 3, 2비트 카운터)회 읽히면 read-hot으로 식별해 고신뢰 그룹으로 마이그레이션, CLOCK 알고리즘 기반으로 read-cold 데이터를 저신뢰 그룹으로 축출.
6. 확장: group partition scheme - 고신뢰 그룹을 WLG(wear-leveling용, 다수)와 RPG(read performance용, 소수 5~30%)로 분할해 기존 PV-aware wear-leveling과의 충돌을 최소화.
7. 부수 트릭: read latency 모델 `RC(N) = α·N + β·log(N+1)` (N=sensing level 수), 조건은 `RBER < CBER_LDPC(N)`; 구현 오버헤드는 스토리지 약 1MB(그룹 ID 2bit/block + read counter 2bit/page + CLOCK 구조).
8. 학습/최적화 절차: 없음(휴리스틱 threshold 및 그룹 수 파라미터를 실험적으로 튜닝, threshold=3, 3-group을 기본값으로 채택).
9. 결과: DiskSim+SSD 모델·MSR Cambridge 워크로드 실측에서 PV-unaware 대비 read response time 평균 60%+ 개선, RPG 5%만으로 큰 성능향상 달성, migration cost는 전체 요청의 평균 2% 이하, lifetime 저하는 7% 미만.
10. 한계: LDPC 자체(부호설계/디코더 알고리즘)는 전혀 수정하지 않고 blackbox로 취급, 순수 시스템/펌웨어 레벨 시뮬레이션(DiskSim)이며 HW(RTL) 설계 없음, 3D NAND로의 확장은 future work로 남김.

### C. Prime ECC 관련 모듈 핀포인트

| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| LDPC sensing-level(soft-read) 빈도 감소 기법 | 대응 없음 | flash controller의 데이터 배치·wear-leveling 정책이며 Prime ECC(디코더/인코더 알고리즘·HW) 범위 밖 |
| soft decoding sensing level(1~7단계) | `channel.cpp` `Set_R_Offset()`, `Set_LLR_Th()` | 참조되는 soft-read 개념 자체는 채널/양자화 모듈과 유사하나, 논문은 이 양자화 로직을 개선하지 않고 발생빈도만 시스템적으로 줄임 |
| LDPC vs BCH 비교 (부호율 8/9, 4KB) | 대응 없음 | 배경 설명용 표일 뿐 부호구성이나 디코더 기법 제안 없음 |

> 적용 가치: 낮음 - LDPC 디코더/부호설계를 개선하는 논문이 아니라 flash controller의 데이터 배치·wear-leveling 정책으로 soft decoding 발생빈도를 줄이는 시스템 논문이라 Prime ECC 코드베이스(디코더/인코더/HW)와 직접적 접점이 없음.
