### B. 알고리즘 요약 (10줄 내외)

1. 시스템: NAND flash 컨트롤러 내 QC-LDPC ECC, base PCHK code rate 0.8(4parity-20data), 확장 시 0.67(8parity-24data)→0.57(12parity-28data)로 단계적 저하.
2. 문제: 고정 code rate LDPC는 P/E cycle 증가에 따른 RBER 급증에 대응 못함 — 초기엔 parity 과잉(공간/관리 오버헤드), 후기엔 parity 부족(정정력 저하).
3. 핵심 가정: P/E cycle 임계값(예 2000, 4000)마다 단계적으로 PCHK를 확장, 각 블록의 P/E cycle을 FTL이 추적해 해당 단계 PCHK 적용.
4. 핵심 기법: base PCHK(A: m×m non-singular QC, B: m×k QC)를 `O, I, D_en, E_en` QC 서브행렬 추가로 확장(extended PCHK) — 기존 RC-LDPC 대비 넓은 rate 폭과 덜 편향된 1-element 분포로 확장.
5. 핵심 식 의미: 확장된 PCHK로 만든 codeword는 base parity(P)와 확장 parity(추가 e bit)로 완전히 분리되어, 디코딩을 base→확장 순으로 단계적(stepwise) 시도 가능.
6. 확장 2단: 초과(excessive) parity는 page 내 저장 불가 시 별도 ECC 블록에 모아 저장, FTL 매핑 테이블에 offset 정보 추가 관리.
7. 부수 트릭: DRAM을 data cache/ECC cache로 분리(LRU), 낮은 단계 PCHK로 디코딩 성공 시 상위 단계 초과 parity 읽기 불필요 → read 오버헤드 절감.
8. 최적화 절차 없음(GA/DE 등 미사용) — PCHK 확장은 구조적 규칙(non-singular, 비편향 QC 서브행렬)으로 설계.
9. 결과: 3-step extended PCHK가 legacy 고정 LDPC 대비 UBER를 낮추고(P/E cycle별), read amplification을 SCORE 대비 약 1.3~1.8배로 낮춤(캐시 32MB 시 legacy 대비 25~35% 개선).
10. 한계: LDPC 인코딩/디코딩 자체 latency 미측정(시스템 IO 레벨 시뮬만), HW 구현 없음, retention/read-disturb 미고려(저자 명시 future work).

### C. Prime ECC 관련 모듈 핀포인트

```
| 논문 요소 | Prime ECC 모듈 | 관련성 |
|---|---|---|
| Extended PCHK (rate-compatible QC-LDPC 확장) | ecc_top.cpp Load_PCM() | H-matrix/부호구조 로드 레이어에 대응하나 Prime ECC는 고정 QC-LDPC이며 단계적 rate 확장 개념 없음 - 이식 난이도 높음 |
| 초과 parity FTL 관리/ECC 캐시 | 대응 없음 | Prime ECC는 시뮬레이터 레벨 ECC 엔진이며 FTL/캐시 시스템은 범위 밖 |
| stepwise 디코딩(base→확장 순 시도) | decoder.cpp LDPC_Decoder() | 조기종료(partial_CRC.cpp)와 유사한 "필요한 만큼만 시도" 철학이나, 이 논문은 부호 자체를 단계별로 바꾸는 것으로 다른 개념 |
| 인코딩(base parity/확장 parity 분리 생성) | encoder.cpp LDPC_encoder_4KB(), Generate_PCM_encoding() | 부호 PCM 생성 로직과 유사하나 dual-diagonal 고정 구조와 다른 확장형 구조 - 이식 난이도 높음 |
```

적용 가치: 중간 — NAND 직접 타깃의 rate-compatible QC-LDPC 아이디어는 흥미로우나 Prime ECC는 특정 고정 H-matrix 기반이라 부호설계 이식 난이도가 높고, 시스템(FTL/캐시) 요소는 코드베이스 범위 밖.
