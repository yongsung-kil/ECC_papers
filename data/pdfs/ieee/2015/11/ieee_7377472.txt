Efficient Architectures for Parity Check Matrix Generation

Stefan Scholl, Norbert Wehn Microelectronic Systems Design Research Group
TU Kaiserslautern 67663 Kaiserslautern, Germany Email: {scholl, wehn}@eit.uni-kl.de

Abstract—New advanced decoding algorithms for error correction systems can improve the frame error rate over commonly used systems. However, they also pose new challenges for their hardware implementations. One part, which consumes a considerable amount of hardware resources, is the storage of the parity check matrix. This problem has not been addressed in literature since it is not yet required by today’s typically applied decoding algorithms. In this paper we investigate sophisticated hardware architectures for storing and generating parity check matrices of BCH, Reed-Solomon and LDPC codes, that exploit the matrix structures. It is shown that these new architectures largely simplify the hardware and require up to five times less area compared to state-of-the-art implementations.
I. INTRODUCTION
New soft decision decoding algorithms for error correction in communications lead to large improvements of frame error rate (FER) in comparison to currently adopted decoding methods. Especially for dense codes like BCH and Reed-Solomon (RS), but also for low-density parity-check (LDPC) codes, a large number of different advanced algorithms have been proposed, which are not yet well investigated with respect to their hardware implementation. They exhibit a huge difference in complexity and FER performance. Some interesting algorithms are: adaptive belief propagation [1], ordered statistics decoding [2]–[4], box and match [5] and methods based on linear programming [6]–[8], to mention just a few.
The motivation for this paper is that many of the afore mentioned advanced decoding heuristics need to store the full parity check matrix [1]–[8]. This issue has not been addressed in detail in literature, because storage of the parity check matrix has not been necessary for traditional algebraic hard decision decoding of BCH and RS codes or for iterative LDPC and Turbo code heuristics. Decoder implementations in literature [9], [10] rely on the straightforward approach of simply storing the matrix in large memories. In [11] the storage of the parity check matrix is even considered as an insurmountable obstacle for hardware implementations. For many practically relevant codes, such as the RS(255,x), the matrix is indeed of considerable size and it is worth to investigate new hardware architectures.
In this paper, we propose a new hardware building block, that generates parity check matrices. We compare it with alternative implementations as RAM and as look-up table (LUT) regarding their complexity for FPGA and ASIC implementations.

II. PRELIMINARIES
A. Architectural Design Choices
For the storage of parity check matrices three different options are available for hardware implementations, that are considered in this paper:
• Storage in a Memory: The straightforward method is to store the matrix in a RAM or ROM.
• Look-Up Table: The matrix is implemented as a large LUT, which can be reduced by logic optimization of the design tools.
• Matrix Generator Architecture: If the matrix has some structure (as most practically relevant codes do) it can be exploited, i.e. the matrix entries are generated instead of stored. In this paper, we propose sophisticated architectures for matrix generation.
In the following, it is assumed, that the parity check matrix is accessed column wise, i.e. one column per clock cycle. Access is granted randomly, so the input will be a column number and the output the corresponding column of the parity check matrix. This is a typical access pattern which is permitted by many algorithms.
B. Implementation
The hardware implementations have been done for a Virtex 7 FPGA (XC7VX690t-3) with Vivado 2014.2. including placement and routing (P&R) and for a 65 nm low-power ASIC technology (results after synthesis with Synopsis Design Compiler under worst case PVT) to investigate their complexity. The achieved clock frequencies are for all designs well above 300 MHz for FPGA and 650 MHz for ASIC.
C. Notation for Galois fields
RS and BCH codes are defined using a Galois field GF (q = 2m) with the primitive polynomial p(α) = a0 + a1α + ... + αm and the primitive element denoted by α. The elements of the GF can be represented in different ways. One is the representation as powers of the primitive element, i.e. GF (q) = 0, 1, α, α2, ..., αq−2 . Another representation is the polynomial representation a0 + a1α + a2α2 + ... + am−1αm−1. The polynomials are generated by taking αi mod p(α). The proposed architecture to generate the columns of a BCH parity check matrix is shown in Fig. 1 and is divided into two parts:
1) Vandermonde Generator: Calculates the exponent of the corresponding GF elements in the column
2) Binary Transformation: Transform the exponents into binary column vectors
For the exponent generation the well-defined structure of the Vandermonde matrix is exploited. In an arbitrary column c = 0, 1..N − 1 the exponent of the GF element in the first row simply equals c. The exponents below the first row are calculated by adding c (mod 2m-1) to the exponent above it. Since this is not an addition modulo a power of 2, the use of a basic hardware adder with overflow is not possible. Therefore we use the architecture shown in Fig. 2 for exponent addition.
For conversion of the exponents to the binary vector representation a small LUT is utilized, that enables quick conversion in a single clock cycle. Alternatively, a shift register could be used if large latencies can be tolerated.

IV. MATRIX GENERATION FOR RS CODES
A t-symbol-correcting (N, K) RS code is a non-binary code defined over GF (q). Its non-binary (N − K) × N matrix is the same Vandermonde matrix as for BCH codes in Eq. (1). Many soft decoding algorithms for RS codes do not utilize this non-binary matrix, but its equivalent (N − K)m × N m binary image matrix (e.g. in [1], [4]).

C. Generation Architecture
Since the (non-binary) parity check matrix of BCH and RS codes are equal, the Vandermonde Generator for exponent generation can be reused. Only the addressing scheme for RS codes needs to be split up into symbol and bit address.
A straightforward implementation considering the ordinary binary image form of Section IV-A is not very useful. This is because the look-up step, transforming the exponent to its binary form, requires to store q m × m matrices. This GF look-up table is required N − K times leading to a total size, which even exceeds the dimensions of the binary parity check matrix in explicit form.
The solution to this problem is the structure we have disclosed in Section IV-B, leading to a much more efficient architecture. The new structure requires only LUTs for the binary column representation, thus largely reducing hardware resources. The exponent generation needs to combine the symbol and bit addresses in order to calculate the exponents in a column. In comparison to BCH generation the slightly modified Vandermonde generator of Fig. 3 is used.
Table II and Table III show the implementation results for the proposed RS column generator for different RS codes. For the implementation of small matrices on FPGAs the use of BRAMs may be considered. For all other cases the generator architecture requires smaller area or less hardware resources. It clearly outperforms the memory and look-up table architectures for the practically very relevant RS(255,x) codes.

VI. CONCLUSION We have proposed new hardware architectures, that implement the generation of parity check matrices of BCH,

RS and LDPC codes by exploiting the matrix structures. Except for small matrices where the implementation as LUT is advantageous, the dedicated matrix generation architectures consume much less hardware resources. This is especially the case for RS codes of high practical interest, like the RS(255,x), where an area saving of up to a factor of five could be achieved.
