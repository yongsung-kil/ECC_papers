Template-Based QC-LDPC Decoder Architecture Generation

Oana Boncalo, Petru Florin Mihancea, Alexandru Amaricai University Politehnica Timisoara Timisoara, Romania
{oana.boncalo, petru.mihancea, alexandru.amaricai} @cs.upt.ro

AbstractвАФ This paper presents an automated template-based approach for layered QC-LDPC architectures. The underlying idea is to be able to generate any number of fairly optimized hardware designs with only a minimum effort required by tuning high level parameters (parity check matrix, number of AP-LLR messages being processed at a time, etc.). Having chosen a partially parallel architecture, template design effort has been concentrated in two directions: distributed control across the LDPC decoder for modularity and versatility, and datapath design in order to support any given parallelism at processing node level. All user inputs are propagated by the proposed automated flow for all design phases: (i) Verilog HDL LDPC architecture (ii) test-benches and verification environments (iii) evaluation step (BER/FER, average iterations, cost, throughput) of the LDPC decoder architecture. We present the results for the architectures generated for аҐКаҐЬ аµМ аЂЬ«° аҐКаҐЬ аµМ аЂЭ regular QC-LDPC code, as well as for the WiMAX (1152,2304) irregular code, with different levels of parallelism at processing node level.
KeywordsвАФtemplate; layered QC-LDPC; design exploration;
I. INTRODUCTION
Time, space and energy optimization of LDPC decoders is of main concern for today hardware industry [5]. However, this kind of investigation requires the usage of a wide range of distinct tools for hardware description and architectural variation specification, testing, simulation, synthesis, etc. [1][2][3][4][5]. In this paper, we present an environment to support design parameter exploration for LDPC decoders. It includes a tool for automatic generation of customized decoder architectures, as well as the integration of well-known industrial tools for verification and implementation (Modelsim, Xilinx ISE). As pressure is ever-increasing for design productivity, people are reluctant to try-out new languages/methodologies. We have opted for a template based design with some smart tags for specifying information/tuning architecture parameters that would shorten the learning curve in an already established process, but, on the other hand, would support the extra flexibility needed. Hence, the template approach tries to act as extension of the existing Verilog HDL constructions for modeling variability.
The emphasis when building the QC-LDPC template architecture has been: (i) IP reuse by versatile building blocks вАУ design has been intended as modular as possible in a bid to extend and/or reuse some of the building blocks for new

templates; furthermore, the goal is to have fairly optimized and parametrized sub-design modules that target good throughput and high hardware (HW) utilization efficiency (HUE), while taking into account the characteristics of FPGA technology; (ii) veri¬њcation environment reuse - a key ingredient that drives the acceptance of the overall environment is the System Verilog verification environment that co-simulates C++ models of decoders with their HDL architectures; (iii) automatized flow that integrates both academic and commercial tools covering all the steps from generation to actually running the processes (e.g. simulation, implementation).
The contributions of this paper are: (i) automated flow with centralized parameter tuning (one configuration file) that propagates user options throughout the entire process; (ii) modular architecture with distributed control; a key ingredient is datapath parallelism p, which states the number of parallel APLLR messages processed at one time; (iii) efficient memory utilization (multi-codeword approach, packed AP-LLR messages, serialization of check-node message accesses). Although the generated architecture is fairly optimized, the focus of this work is to provide an efficient way to generate HW designs, which can be validated through co-simulation with a higher level model for LDPC codes. By using the proposed methodology, weeks of design and verification time can be saved by not manually redoing the work needed to customize: the Verilog HDL design files, the testbench files, the scripts for simulation, synthesis, and implementation, the higher level model inputs.
This paper is organized as follows. Section II summarizes related approaches. Section III offers a brief introduction of QCLDPC codes. It then presents the proposed decoder template architecture with its sources of variability. Section IV discusses the build flow for the design exploration environment, the process and the customization tool. Results for both regular and irregular codes are presented in Section V, while Section VI is dedicated to concluding remarks.

III. LAYERED QC-LDPC DECODERS
A. Layered LDPC decoding
LDPC codes [6] are linear codes defined by sparse paritycheck matrices. Two types of processing nodes are required: variable-nodes and check-nodes, corresponding to the columns and the rows of the parity check matrix вАЂ№™вАђ. Message inputs for these nodes are dictated by the non-zero entries of the paritycheck matrix. Quasi-Cyclic (QC) LDPC codes are built using a base matrix вАЂ№§вАђ, where each non-negative entry is replaced by a circulant square permutation matrix of size ЁЙ аµИ ЁЙ[8]. The resulting matrix H has a layered structure, where each layer is composed by ЁЙ consecutive rows. LDPC codes can be iteratively decoded using message-passing algorithms, and the exchange of messages takes place in several rounds or iterations.
Based on the order the messages are being processed, two types of scheduling exist: flooding, where all check nodes are processed followed by the processing of all variable nodes [9]. A different approach is layered scheduling, where horizontal layers of the matrix вАЂ №™вАђare processed sequentially [10]. The processing within a layer is performed in parallel; similar to the flooding scheduling. The layers are processed serially.
Regarding decoding algorithms, Min-Sum (MS) and its variants are widely used due to their suitability for HW implementation. Designing, verifying and implementing an LDPC decoder represents a complex task. Even if the same architecture is used, modifying few parameters requires the entire design-verificationimplementation cycle to be redone. To address this problem, we have built a tool that generates the entire framework вАУ RTL source code, verification framework and implementation scripts вАУ for LDPC decoder, based on the set of input/customization parameters.

IV. DESIGN EXPLORATION FLOW

A. Overview
Fig. 2 summaries the main actions performed automatically by the proposed environment. It consists of 5 steps, of which steps 2 вАУ 5 correspond to a usual circuit design flow. They are achieved by invoking well-known commercial tools such as ModelSim and Xilinx ISE. However, these tools operate on an HDL description and thus, when exploring architectural alternatives, the tools will usually work with many similar but distinct decoder specifications. Moreover, all these tools must be properly configured with (possibly common) information such as test-bench and configuration data, synthesis and implementation options, etc.

The most important step in the proposed flow is represented

by the customization. This action is performed by a dedicated

tool that we developed. It takes as inputs the design and code

parameters given by the user

(described in Section II.c)

and the HDL decoder templates, and it generates the RTL

Verilog source code for the desired LDPC decoder. The decoder

templates capture the general architecture of the decoder and of

the internal components, but also specify possible architectural

variations (e.g., parallelization degree) and the way the decoder

must be built in the presence of such a variation. Consequently,

processing the template and using the user arguments (e.g., the

desired parallelization degree), the customization tool will

output the concrete Verilog decoder description with the

particularities / variations desired by the user.

Templates have been developed for different abstraction layers within the LDPC decoder hierarchy: (i) LDPC decoder template - it describes the common layered LDPC architecture, the building blocks and connections within (ii) LDPC decoder components templates вАУ they describe the architecture of the main modules of the LDPC decoder (e.g. gamma memory read interface, beta module, processing units array with local control unit, see Section II.b) (iii) basic block templates such as barrel shifter, comparators, etc.

Fig.2. Customization, verification and implementation flow
The user parameters are passed through the hierarchy in a topdown approach. From the description perspective, these are (i) parametrized Verilog HDL blocks (using generate constructs, module parameters), fully custom templates (e.g. control ROMs, which are fully re-generated for each set of user inputs), hybrid descriptions (a mix of Verilog HDL and smart tags for the parts that require more advanced customization features than those supported by Verilog HDL language вАУ e.g. tree comparator used in check node processing with optional pipeline levels).
For each set of parameters, an instantiation of this environment and the concrete Verilog HDL design files for the LDPC decoder is generated. To re-create all this steps manually requires a great deal of effort and copy-paste operations that are error prone. Furthermore, after setting the inputs from the process in place, any number of instantiations of this environment can be made.
B. Verification Methodology
Verification has been carried out at several levels in the design: block level (barrel shifter, comparator in the check node processing), sub-module level (for the AP-LLR memory subsystems, the processing nodes, and the check node memory subsystem), as well as the top-level LDPC decoder. For block level verification, Verilog test-benches have been used. For submodules and top level LDPC architecture, SystemVerilog environments (SVE) have been developed. The validation of the result is through co-simulation with a C++ simulator of the transmission chain. It includes: random frame generator, encoder, modulation, channel noise, automatic gain control, demodulation, quantization, decoding, and result processing. The RTL LDPC decoder architecture can be verified in two ways: (i) the output of the RTL implementation is compared with the output of the decoder in the C++ simulator, (ii) the output of the RTL decoder is verified with the output of the encoder in the C++ simulator. The latter is used in order to evaluate the frame error rate and the

V. RESULTS
We have generated several LDPC decoders for the following LDPC codes: WiMAX rate Ќ≥ќ§Ќі irregular code, ЁАвАЂ ЁТвАђаµМ Ќµ, rate Ќ≥ќ§Ќі regular code, ЁАвАЂ ЁТвАђаµМ Ќґ rate Ќ≥ќ§Ќі regular code and ЁАвАЂ ЁТвАђаµМ Ќґ, rate Ќµќ§Ќґ [11]. The number of coded bits is 2304 for WiMAX and 1296 for the regular codes. The circulant sizes are: 96 for WiMAX, 54 for rate Ќ≥ќ§Ќі regular codes and 27 rate Ќµќ§Ќґ regular codes. The message quantization is (4,6) for all cases. The synthesis has been performed for the Xilinx Virtex-7 XC7VX485T FPGA device. Results, depicted in Table I, show that processing node level parallelism increase will lead both to better throughput, but also to increase cost (BRAM and slices). Furthemore, increased parallelism will also lead to a decrease in the maximum frequency. For both regular codes and WiMAX irregular code, the increase in throughput, as well as cost, is proportional to the parallelism degree.
Custom designed layered LDPC decoders for FPGA have been proposed in [12][13]. The decoder in [12] has been implemented on Zynq platform (7-series as Virtex-7) for WiMAX rate Ќ≥ќ§Ќі LDPC code; it has a cost of up to 3732 slices and 24 BRAM and throghput of up to 410 Mbps for coded bits (throughput computed for 10 iterations) for the highest rate ЌЈќ§ЌЄ. The generated decoder has approximativelly the same resource usage/throughput ration. The decoder in [13] has been implemented on Xilinx Virtes-6 devices for a 2048 length rate Ќ≥ќ§Ќі LDPC code. It has a cost of more than 6200 slices and 36 BRAM and a throughput of 150 Mbps (coded bits). We can observe that the proposed decoder provides better or comparable throughput and better BRAM usage for most parallelization degrees for the WiMAX rate Ќ≥ќ§Ќі LDPC code, even though the proposed solutions are generated.
VI. CONCLUSIONS
The contributions of this work are as follows: (i) tool for customization of templates of layered QC-LDPC decoders (ii) automated build flow: customization of verification, synthesis, and implementation scripts (iii) FPGA technology specific knowledge embedded in the template (multi-code-word approach to compensate pipeline delay and improve decoding performance and convergence; barrel shifter pipeline delay is a вАЬclever wayвАЭ of buying вАЬtimeвАЭ and relax port constraints for the check message memory; automated pipeline insertion for the datapath check-node unit implementation when varying the number of AP-LLRвАЩs processed in one clock cycle). The developed layered LDPC generation flow outputs layered LDPC decoders with higher throughput than hand-made LDPC decoder solutions in [12][13] and comparable cost.
