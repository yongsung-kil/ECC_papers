A New Reliability-Based Incremental Redundancy Hybrid ARQ Scheme Using LDPC Codes

Hamid Saber and Ian Marsland Department of System and Computer Engineering,
Carleton University, Ottawa, ON, Canada hamid.saber@carleton.ca, ianm@sce.carleton.ca

Abstract—We present a new reliability-based hybrid automatic repeat request (RB-HARQ) scheme based on low density parity check (LDPC) codes. With the proposed RB-HARQ, which uses a rate-compatible LDPC code with puncturing and extending, the longest codeword is divided into clusters of code bits. Unlike previous works, in the event of a decoding failure, the receiver measures the reliability of received clusters, instead of code bits, and determines which cluster would be most beneficial for retransmission. Several metrics to determine the best cluster candidates for retransmission are derived analytically. We show that one of the new metrics outperforms the previous metrics. We also show that even with the feedback overhead taken into account, our RB-HARQ can still result in significant gain over the previous works, provided that the cluster size is appropriately selected.
Keywords—Hybrid ARQ, Rate-compatible LDPC codes, Reliability-based hybrid ARQ.
I. INTRODUCTION
Rate-compatible (RC) codes are a sequence of codes that have been widely used to provide the rate flexibility requirement to address the channel variations in wireless digital communication [1]. The rate flexibility feature of RC codes makes them well-suited for automatic repeat request (ARQ) protocols and also provides efficient encoding complexity. RC codes are used for a special type of ARQ scheme called incremental redundancy hybrid ARQ, where the parity check bits of the next lower rate code are transmitted if the decoding of the previous higher rate code fails. Puncturing is the most common way to produce RC codes, and puncturing of low density parity check (LDPC) codes was studied in [2] and [3] and references therein.
Reliability-based hybrid ARQ (RB-HARQ) was introduced by Shea in [4]. The RB-HARQ scheme is a type of selective ARQ which, based on information extracted from the received codeword reflecting the reliability of the received code bits, requests retransmission of only those code bits which are weakly received at the receiver. RB-HARQ has been used with convolutional codes [5] and LDPC codes [6]. Substantial performance improvements over the traditional hybrid ARQ schemes were reported. However, these improvements come at the expense of some complexity issues that prevent them from being used in practice. In particular, the decoder needs to send information specifying which code bits most need to be retransmitted. For example in [4] and [6] the receiver sorts the received code bits from the least to the most reliable and asks for retransmission of a number of the least reliable ones. This not only introduces complexity to the system but

also affects the actual throughput due to the time needed for communication over the feedback channel, since each index requires log2N bits to specify, where N is the block length. In [7], the authors proposed a new RB-HARQ scheme for LDPC codes that reduces the feedback overhead in two ways. Firstly, the feedback channel is only used at times when it is more beneficial to do so (i.e. only for the extended codes). Secondly, instead of requesting retransmission of individual code bits, the receiver requests clusters of adjacent bits, and therefore only needs to specify the index of the cluster that needs to be retransmitted.
In this paper, we extend and improve the work of [7]. In particular, we analytically derive three different reliability metrics, including the metric used in [7] which was suggested heuristically without justification. We also show that one of the new reliability metrics proposed in this paper which is based on relative entropy function, outperforms the other two including the one used in [7]. Furthermore, unlike [4]-[7], we take into account the feedback overhead and show that significant throughput gain can still be realized, even after compensating for the feedback overhead, provided that the cluster size is appropriately selected. We also provide a more precise description of the code structure and the RB-HARQ used in [7]. In the following section we explain the code structure. In Section III we derive different metrics for choosing the best cluster candidates for retransmission. Simulation results are presented in Section IV. Section V concludes the paper.
II. CODE STRUCTURE
We use the same RC-LDPC code structure introduced in [3] depicted in Fig. 1, but we allow the extension matrix, HE, to be of arbitrary size W × W . Also we let the mother code be of arbitrary rate of the form a/b with the parity check matrix HM of size of (b − a)W × bW for arbitrary integers a and b, a < b. The W × W identity matrices I are to provide correlation between the code bits of the mother code and the parity bits of the extended codes. Both HM and HE are constructed according to an optimized degree distribution (for more details on this code structure see [3]). Although the mother code could theoretically be extended indefinitely, it is more practical to limit the maximum number of extension to some finite value, NE, so the maximum codeword length is N = (b + NE)W . In [3], the parameters W = 128, a = 5, b = 13 and NE = 6 were used. To get higher rate codes, puncturing is done on the mother code. In the puncturing the same approach as in [3] is used, i.e. code bits are transmitted according to the puncturing pattern in clusters of S bits. This process is repeated until all the clusters of the mother code have been transmitted. If decoding still fails the next phase starts. In this phase reliability metrics are used to transmit the code bits of the extended codes. The first cluster of the first extended code is transmitted and decoding is attempted on the Tanner graph of the first extended code. If decoding fails, the receiver calculates a “reliability metric” for each cluster in the extended code. These reliability metrics are based on the code bit log-likelihood ratios LLRs after decoding, and are calculated according to one of the algorithms described in Section III. The least reliable cluster is selected for transmission. This cluster may have been previously transmitted, in which case maximum ratio combining is used to combine the multiple received samples, or it may be a new cluster. After transmission of the requested cluster, decoding on the graph of the extended code is again attempted, and this process is repeated until decoding succeeds, or W/S clusters have been transmitted. If decoding still fails, the transmission of the parity bits of the next lower rate code commences, decoding is attempted on the Tanner graph of the corresponding extended code and the same procedure as the previous extended code is taken. In case of decoding failure when the longest code has been used, all decodings are attempted on the full Tanner graph.

Fig. 2 shows the throughput of the system for each of the three different reliability metrics, using a cluster size of S = 48, along with the throughput of a traditional system that does not use reliability feedback (labelled “Non-RB”). The non-RB scheme uses the same mother code as in [3], punctured with Ha’s algorithm [2], but allows for up to NE = 50 extensions to avoid problems when the fully extended code is still not long enough for successful decoding. As can be seen, use of reliability feedback with any of the metrics provides better performance than the non-RB scheme, particularly at low signal-to-noise rations (SNRs). As also shown in Fig. 2, the metric C outperforms the other two, including the metric A used in [7]. We will therefore focus on this metric for the remainder of this paper.
One important parameter of the feedback scheme is the cluster size, S. By using a smaller cluster size it is possible to more precisely specify which code bits should be transmitted, thereby reducing the number of unnecessary code bit retransmissions, which in turn improves throughput, as shown in Fig. 3. We can see the advantage of using a very small (S = 1) cluster size over small (S = 8), intermediate (S = 48), and large (S = 128) cluster sizes. However, there is a penalty with using small sizes because more bits are needed to specify the cluster, thereby increasing the amount of the feedback required. The number of bits needed to specify a cluster of size S within a codeword of length N is log2(N/S), which should be kept small.
It is possible to adjust the throughput, taking into account the feedback. For each cluster which has been transmitted in the feedback mode of the system, the actual number of transmitted bits is S + log2(N/S), while for all the other clusters this quantity is equal to S. Fig. 4 shows the throughput results of the proposed method with different cluster sizes, taking into account the feedback overhead according to the above. As it can be seen even with feedback overhead taken into account, there is still up to 2 dB gain over the nonfeedback method, provided that the cluster size is selected

V. CONCLUSIONS
The design of RB-HARQ with LDPC codes based on feedback is considered. The proposed scheme measures the reliability of clusters of code bits instead of individual code bits. By requesting retransmission of whole clusters, along with partial use of the feedback channel in the extended code mode, the proposed scheme is significantly less complex for practical systems compared to the previous RB-HARQ systems. The analytical derivation of different reliability metrics to choose the best candidate cluster are presented. Our simulation results show that an efficient reliability metric can result in significant gain over the previous RB-HARQ schemes as well as schemes which do not use reliability metrics.
