ASIC Design of a Gbit/s LDPC Decoder for Iterative MIMO Systems

Christina Gimmler, Frank Kienle, Christian Weis, Norbert Wehn Microelectronic Systems Design Research Group University of Kaiserslautern
Erwin-Schroedinger-Strasse, 67663 Kaiserslautern, Germany Email: gimmler@eit.uni-kl.de

Matthias Alles Creonic GmbH 67663 Kaiserslautern, Germany Email: matthias.alles@creonic.com

Abstract—The design of double-iterative systems like turbo synchronization or iterative demapping and decoding will be one of the big challenges in the next years. The components of such systems need to fulfill stringent conditions on throughput and flexibility thus making a reuse of available standard components difficult.
We present a high throughput ASIC design for LDPC decoding in iterative MIMO systems with a high flexibility on block sizes (3720 to 14880 bits, granularity 186 bits) and code rates (1/2 to 4/5). After P&R, the ASIC design has an area of 4.588 mm2 and consumes 1271 mW at a clock frequency of 275 MHz. The LDPC decoder has a throughput of 3.5 Gbit/s (@5 iterations) while the resulting iterative MIMO system (4 feedback loops with 5 LDPC iterations each) runs at 275 Mbit/s.
I. INTRODUCTION
Double-iterative systems like e.g. turbo synchronization [1] or iterative demapping and channel decoding [2] [3] attract more and more attention due to their high spectral efficiency. In these systems the synchronization or demapping unit works iteratively with the channel decoder which can be iterative itself, e.g. for LDPC or turbo codes. The advantage of those systems is clearly the gain in communications performance (up to several dB). However, the double-iterative scheme poses high demands on the throughput of the component decoders even for relatively low system throughputs. LDPC decoders are well suited for this scenario as they can be easily parallelized to support highest throughputs. Due to OFDM framing structure, decoders have to support long block sizes (up to 15000 bits). Changing channel conditions require different modulation schemes and antenna constellations as well as flexible channel coding rates.
Recent high-throughput LDPC decoder architecture publications [4] [5] are limited with respect to flexibility and maximum block length. E.g., [4] achieves a throughput of 16 Gbit/s but the decoder supports a single LDPC code only. On the other hand, [5] achieves a throughput of 6.7 Gbit/s and supports different code rates, but the maximum block length is limited to 672 bits. These high-throughput decoders are register-based architectures, which make the usage of longer blocks (e.g., 15.000 bits) prohibitive. Instead, memorybased architectures become mandatory. For instance, DVB-S2 defines block lengths of 16200 and 64800 bits for various code rates. Decoder architectures for these long blocks are memory-

A. New Contributions
This paper has three main contributions. We analyze the channel decoder as a component of a double-iterative MIMOBICM decoder. We show that even for a moderate system throughput of 200M bit/s the decoder has to work in the Gbit/s range.
In contrast to the decoder implementations mentioned before, we present a high throughput LDPC decoder architecture for long blocks up to 14880 bits. We show that a system approach including joint code construction and architecture design is mandatory to fulfill all requirements. The decoder supports different block lengths and code rates and the throughput is independent of the block length.
Additionally, we present implementation results after P&R and a detailed power analysis which includes the LDPC decoder, the channel interleaver and deinterleaver and the interface to the MIMO detector.

III. LOW DENSITY PARITY CHECK CODES
LDPC codes [9] are linear block codes defined by a sparse parity check matrix H of size M × N , see Fig. 2. For binary LDPC codes a valid codeword x has to satisfy HxT = 0 in a modulo-2 arithmetic. A column in H is associated to a codeword bit, a row corresponds to a parity check. Typically, 2% or less of the elements of H are non-zero.
The code structure can be visualized by a Tanner graph. Each variable node (VN) represents one code bit, and each check node (CN) represents one parity check, respectively. Edges connect variable and check nodes according to the nonzero elements of the parity check matrix. Fig. 2 shows a small Tanner graph with N = 10 variable nodes and M = 5 check nodes.
LDPC decoding is an iterative process in which soft information is exchanged between variable and check nodes. Due to the high implementation complexity of the optimal belief propagation algorithm, we use the suboptimal normalized Min-Sum algorithm [10]. Another major advantage of this

C. Joint code construction and architecture design
In the previous sections we have derived three important constraints for the LDPC decoder in the context of an iterative MIMO system.
• Flexibility (II): the decoder architecture has to support block sizes ranging from N = 3720 to N = 14880 while supporting various code rates ranging from RC = 1/2 to RC = 4/5.
• Throughput (IV-B): due to the MIMO feedback loop the decoder has to provide a throughput within Gbit/s range requiring a minimum parallelism of P = 256.
• Layered decoding (III): the decoder has to support layered decoding to obtain the best communications performance result at each iteration.
All three constraints heavily influence the design of the parity check matrix of the LDPC code and the final decoder architecture. Fulfilling the three constraints at the same time is only possible by joint channel code construction and architecture design.
Typically, numerous parallel check nodes are instantiated in an LDPC decoder to increase its throughput. The memory access patterns to supply all check nodes with data are crucial for the code design to avoid access conflicts. Communications standards define LDPC codes with structured parity check matrices [18]. These check matrices are composed of submatrices of size IxI which are either 0 or cyclically shifted identity matrices, see e.g. Fig. 4. This structure avoids memory

Most state-of-the-art LDPC decoders which support long block lengths process up to I parity checks in parallel which belong to the same submatrix, thus P ≤ I, e.g. [6] [7] [19]. In this case, the check nodes work serially, i.e. on one edge per clock cycle. Meeting the target throughput in the Gbit/s range on a standard decoder architecture would require a very high I. However, this contradicts with the given constraints on flexibility and layered decoding: For a small block length like the given N = 3720, a large I results in submatrices with multiple diagonals. These multi-diagonals lead to severe memory access problems for layered decoding architectures which are well-known from literature [20] [8]. Thus, increasing the submatrix size I in order to reach the required parallelism on a standard architecture is infeasible. Instead, we need to process several submatrices in parallel.
In IV-B, we derived a minimum degree of parallelism of P = 256. Constructing a structured LDPC code for layered decoding and a minimum block length of N = 3720 allows to choose a submatrix size of I = 93. Thus, 3 submatrices have to be processed in parallel, resulting in P = 3 · 93 = 279. The resulting architecture is based on the concepts presented in [15] and shown in Fig. 5. Each set of I Variable Nodes is attributed to one of the three slots. One slot processes one submatrix per clock cycle. In contrast to standard architectures, each of the 93 check node blocks (CNB) reads and writes three edges in parallel, with each edge coming from a different slot. The minimum search within the CFU is performed according to [19]. The variable node operations according to Equation (5) and Equation (6) are processed within the CNBs and do not require an explicit functional unit. This architecture fulfills the constraints on flexibility, throughput and layered decoding.

The challenge of the code design lies in the placement of nonzero submatrices. Continuous processing has to be ensured which is not delayed by blocked variable node columns, i.e. there always has to be a row of submatrices available which does not contain any of the currently blocked variable node columns.
For example, consider Fig. 4: We start by processing the first group of 93 check nodes (the first 93 rows of the parity check matrix). In the first clock cycle the three submatrices labeled with ’1’ are processed in parallel. Next, the submatrices ’2’ are processed and all values for the first 93 parity checks have been read. But only 3 clock cycles later the results of the sub-matrices ’1’ will be written back. Thus, all variable node columns containing ’1’ are blocked until clock cycle 6, all columns containing ’2’ until clock cycle 7, see (9). When we select the next group of check nodes, we have to find a row of sub-matrices which does not contain any blocked columns. In the next two clock cycles, the first possible row of check nodes is processed, submatrices labeled ’3’ and ’4’. Now, 11 variable node columns are blocked. The next non-blocked row has to be located and processed and so on. In clock cycle 6, the columns ’1’ are released again making other rows accessible.
The layered constraint mainly influences the placement of identity matrices for short block sizes. Thus, the maximum variable node degree for the shortest blocks is restricted to V Nmax = 5. Starting from a block size of 7440 bits a maximum VN degree of V Nmax = 15 can be supported. The LDPC codes can then be matched to the MIMO system [14] for closed loop iterations. Different antenna systems (2x2,4x4) and modulation schemes (16-QAM,64-QAM) require different degree distributions of the LDPC code to obtain a good communications performance.
The permutation of each identity matrix is determined by the progressive edge growth technique [21]. The resulting decoder architecture supports block sizes between 3720 and 14880 bits with a granularity of 186 bits and code rates between 1/2 and 4/5. Due to space limitations we cannot show any simulation results of the communications performance of the constructed codes. However, many publications have shown that iterative demapping and decoding achieves a gain of approximately 3 dB compared to open loop decoding.

VI. CONCLUSIONS
We presented an ASIC design of a high throughput LDPC decoder for iterative MIMO systems. Even for moderate system throughputs of a few hundred Mbit/s, the components of an iterative MIMO system have to work in the Gbit/s range. We derived a high throughput architecture with flexible block sizes between 3720 and 14880 bits and code rates from RC = 1/2 to RC = 4/5. The ASIC design after P&R has an area of 4.588 mm2 and consumes 1271 mW at a clock frequency of 275 MHz. The LDPC decoder has a throughput of 3.5 Gbit/s (@5 iterations) while the resulting iterative MIMO system (4 feedback loops with 5 LDPC iterations each) runs only at 275 Mbit/s.
