Parallel Decoding of LDPC Convolutional Codes Using OpenMP and GPU
Chi H. Chan and Francis C. M. Lau Department of Electronic and Information Engineering, The Hong Kong Polytechnic University, Hong Kong
Email: andyjv5433@gmail.com & encmlau@polyu.edu.hk

Abstract—Recently, there have been different applications, namely 10GBase-T Ethernet, video broadcasting and satellite communication, utilizing low-density parity-check (LDPC) codes as the forward-error-correction codes. The main reason is that the error performance of LDPC codes can be very close to the Shannon limit. LDPC codes can be further categorized into LDPC block codes (LDPC-BCs) and LDPC convolutional codes (LDPC-CCs). It has also been discovered that LDPC-CCs usually outperform LDPC-BCs. Simulation of LDPC-BCs and LDPCCCs can take a lot of time because the decoding algorithms are relatively complex. Fortunately, the decoding steps can be performed in parallel. In this paper, we create three different platforms for simulating the error performance of LDPC-CCs. The first two platforms are run on a Central Processing Unit (CPU) while the third one involves the use of a Graphics Processing Unit (GPU). We show that using GPU can improve the simulation speed substantially.
Index Terms—CPU, error performance, GPU, LDPC code, LDPC convolutional code.
I. INTRODUCTION
Low-density parity-check (LDPC) codes were invented in 1963 [1]. The challenge on their implementation at that time lied in the requirement to decompose a large iterative decoding algorithm into multiple computations. Hence LDPC codes had remained un-noticed for a few decades compared to other channel coding methods. With the advance of the Very Large Scale Integration (VLSI) technology, it has become more feasible to implement algorithms involving intensive computations. In 1996, LDPC codes were rediscovered [2] and were shown to achieve error-correction performance very close to the Shannon limit [3]. Subsequently, LDPC codes have been adopted by several standards such as DVB-S2, WiMAX(IEEE 802.16e) and 10G Ethernet (IEEE 802.3an).
LDPC codes can be further categorized into LDPC block code (LDPC-BC) [3]–[5] and LDPC convolutional code (LDPC-CC) [6]. LDPC-BCs have a fixed code length while LDPC-CCs have variable code lengths. LDPC-CCs are found to have lower error floors compared with LDPC-BCs. In this paper, we simulate the sum-product algorithm (SPA) [1] and the min-sum algorithm (MSA) [7] for decoding LDPC-CCs based on the OpenMP and CUDA programming platforms. The algorithms are executed by a multi-core CPU and a GPU. We further record and compare the simulation times.
The work described in this paper was partially supported by a grant from the RGC of the Hong Kong SAR, China (Project No. PolyU 521809).

Compared with using a single-core CPU, a significant increase in computation speed can be observed by using a multi-core CPU or a GPU.

III. IMPLEMENTATION LDPC-CC DECODERS
An LDPC-CC decoder usually consists of a number of processors in concatenation. The channel messages are input to the first processor, and the outputs of the first processor are input to the second processor, and so on. For the last processor, hard decisions are made on its outputs to form the decoded codeword. Each processor can therefore be treated as performing one decoding iteration on the LDPC-CC.

For a single-core CPU, all the instructions are executed sequentially. For a multi-core CPU, we make use of the OpenMP application programming interface (API) to carry out each of the instructions (b)–(f) across different cores at the same time. When a GPU is available, we use the CUDA platform to carry out each of the instructions (b)–(f) in parallel using the cores in the GPU. Note that data have to copied between the CPU and GPU when the GPU is used.
IV. SIMULATION RESULTS
We perform the simulations using a PC with a quad-core AMD CPU operating at a clock frequency of 3.0 GHz. Further, the GPU we use is a Nvidia GTX 460 which possesses 336 cores. The parameters used in the LDPC-CC are ms = 3, J = 4, K = 24. Moreover, the sizes of the block HTi (t) are 2532× 422, 4608 × 768 and 6144 × 1024.

When we compare the bit-error-rate (BER) results produced by the three different platforms, we find that the BERs are very similar regardless of the platform being used. Finally, we use the GPU platform to evaluate the BER of different LDPCCCs when 10000 encoded blocks are sent. In each scenario, we find the error rate when 10 and 20 decoding iterations are, respectively, performed. In Figs. 4 to 6, the BERs are plotted. As we have expected, lower BERs are obtained (i) when 20 iterations are performed instead of 10 iterations and/or (ii) when SPA is used instead of MSA.

V. CONCLUSION
Programs based on the OpenMP and CUDA platforms have been written to simulate LDPC-CC decoders. Compared with using a single-core CPU, using a multi-core CPU (OpenMP

platform) or a GPU (CUDA platform) can improve the simulation speed. In most cases, the GPU-based decoder outperforms the multi-core CPU-based decoder. Moreover, all simulation platforms produce very similar BER results.
