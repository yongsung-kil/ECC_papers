Abstract
In this paper, we present a domain-programmable (code-independent) parallel architecture for efficiently implementing iterative probabilistic decoding of LDPC codes. The architecture is based on distributed computing and message passing. The exploited parallelism wus found to be communication limited. To increase the utilization of the computational resources we separate the routing process and state management functionalities pelformed by physical nodes from computation functionalities pelformed by function units that can be shared by multiple physical nodes. Simulation results show that the proposed architecture leads to improvements in FU utilization by 251%, 116%, and 209% compared to a hypothetical fully parallel custom implementation, a fully sequential implementation, and a proprietary FPCA custom implementation, respectively, that all use the same core FU design. Compared to an implementation on a shared-memory general-purpose parallel machine, the proposed architecture exhibits 75.6% improvement in scalability. In this paper, we also
introduce a new low cost store-and-forward routing
algorithmfor deadlock avoidance in torus networks.
1. Introduction
As communication technology continues to advance and increasing number of communication standards continue to emerge all over the world, there is a need to implement different communication algorithms on programmable or reconfigurable hardware [ 11. The challenge is to reduce cost and power consumption while meeting the required performance target, especially in wireless applications. Some of these algorithms like iterative decoding of Low Density Parity Check Codes (LDPCC) have a large amount of inherent parallelism. Efficiently exploiting this parallelism can lead to significant power savings while attaining the same level of performance, because it allows the system clock to run at a slower rate [ 2 ] . What makes this even more attractive

is the recent advances in digital VLSI technology that have allowed higher integration densities and lower cost per transistor.
Low-density parity check (LDPC) codes proposed by Gallager [3, 41, appear as a class of codes which yield very good performance on the binary symmetric channel (BSC) as well as on the additive white Gaussian noise (AWGN) channel. A regular ( M , N , t,., t,) LDPCC is defined by a M x N parity check matrix H that has 0 entries everywhere except for t, 1’s in each column and t, 1’s in each row, where the numbers t, and t, are small relative to the matrix dimensions. Irregular LDPC codes, with parity check matrices that have varying number of non-zero entries in each column and row, can yield better performance than the regular constructions if designed properly [5-71. Recently, it was shown that the theory of factor graphs and the message-passing algorithm provides a powerful tool for understanding many iterative decoding algorithms. Iterative decoding techniques, in general, have received significant attention recently and various results have been reported [7-121. LDPC code can be

The aim of this paper is develop a domainprogrammable parallel architecture for efficiently implementing the message-passing algorithm, and in particular for iterative LDPCC decoding. The architecture is based on distributed computing and message passing and is composed of physical nodes connected to each other using a direct interconnection network. The design of the interconnection network is optimized to take advantage of the special characteristics of factor graphs so as to increase the communication throughput and improve scalability by dividing the direct interconnection network into multiple sub-networks in which messages are routed in parallel. A comparison based on cost and performance was done to select the most appropriate 2D sub-network topology and routing algorithm for the proposed architecture. The exploited parallelism was found to be communication limited. The architecture increases the utilization of the computational resources by separating the routing and state-management functionalities performed by physical nodes from computational functionalities performed by function units that can be shared by multiple physical nodes. This results in a decoupling between the exploited communication and computation parallelism, and allows the optimization of each of them individually to increase the overall efficiency of the architecture. The optimal function unit sharing-factor was found to be between 2 and 4, regardless of the number of processes mapped to a physical node. Therefore, more than 50% reduction in the cost of computational resources can be achieved without a significant degradation in performance leading to a much higher FU utilization and computational efficiency.
2. The proposed architecture
2.1. Description
Function and variable nodes of the factor graph of the code are referred to as logical function and variable nodes to differentiate them from physical processing nodes of the implementation architecture. The 1 subscript will be used to indicate that. The proposed architecture is composed of physical nodes that communicate with each other by passing messages. Since different processing is needed for logical variable and function nodes, physical nodes are also divided into two types: variable physical nodes, and function physical nodes. Multiple logical
nodes are mapped to a physical node of the same type.
The architecture divides the whole network into three similar 2D sub-networks of the same topology and of

equal size =NP nodes: one sub-network of physical function nodes and two sub-networks of physical variable nodes. Each physical function node pJ in the function node sub-network is physically connected via bidirectional links to the two physical variable nodes at the corresponding positions at the two variable node subnetworks: p v ; and p v , + ~ p .This facilitates exploiting more communication parallelism between nodes of different types. 2.1.1. Synchronization. In iterative parallel programs, like LDPCC decoding, execution proceeds as a sequence of sequential iterations, where at each iteration all parallel processes corresponding to logical function and variable nodes can execute independently, but each logical node then needs to communicate values computed during that iteration with logical nodes it is connected to in the factor graph, before it can commence its next iteration. Given a mapping of logical nodes to physical nodes, over a number of iterations, the slowest executing logical node will control the overall execution time. As shown in Figure 2, the control flow is done in such a way that every iteration, a logical node sends data to its logical neighbors and then wait until it receives messages back from all of them before it can start to process and send results of the next iteration to any of these neighbors. Therefore, no overflow or iteration overlap can be caused by a node continuously sending out data that is not timely absorbed by other nodes in the system, which makes the system self-synchronized. No further attempt is made to synchronize the state of different logical nodes. There is minimal correlation between nodes located physically close to each other in the network. This would minimize contention in the network and other shared resources by more uniformly distributing the load on the time scale, while at the same time it reduces the implementation cost.

2.1.2. Isolating control from processing and sharing
FUs. One feature of the proposed architecture is isolating logical nodes control and state-management functionalities and routing functionalities from the computational functionalities to introduce one more degree of freedom in balancing the communication and computation resources. This also results in a more modular architecture. Physical nodes shown in Figure 1 only perform the routing functions and maintain the state and local storage of the logical nodes mapped to them. Actual function and variable node processing takes place in custom function units (FUs) scattered in the function and variable sub-networks. These function units are shared between multiple physical nodes of the same type. Since the proposed parallel implementation of the LDPCC decoding algorithm is based on passing a large number of short messages back and forth between variable and function nodes over a limited set of communication resources, logical nodes spend a significant portion of the iteration cycle waiting for input messages from their neighbors before they can commence processing. This causes the extracted parallelism to be limited by the communication not computation latency. Computational resources, i.e., number of function units, can therefore be reduced without decreasing the available communication parallelism, and hence without significantly degrading the overall performance, as will be shown in the simulation results. The function unit sharing-factor is a design parameter that can be used to optimize the cost-performance tradeoff of the architecture.

2.1.3. Physical node structure. A simplified illustration of the physical and logical node structures is shown in Figure 3. The physical node has in addition to the logical nodes mapped to it, a router, a FU interface server, and a send server that are shared by these logical nodes. The router performs the routing and switching functions from input ports to output ports or local storage if one of the local logical nodes is the destination of the message. Information that indicates the number of logical nodes mapped to a physical node, number of logical neighbors of each logical node, and the routing address to reach each of them, and other such static information is stored in the local memory of the physical node. This information is code dependent, and is set by the programmer at the initialization phase. 2.1.4. Function Unit (FU) Structure. The FU is composed of a control part for interfacing with the physical nodes sharing it, and a processing part, called the core, for doing the actual computations. There are two types of FUs: variable FUs for performing variable node processing, and function FUs for performing function node processing. The control part of the two types of FUs is the same, but they differ in the core processing part. 2.2. Sub-network topology and Mapping Optimization
There are many sub-network interconnection topologies that can be used, However, since the proposed architecture is meant to be implemented on a single chip, we have limited our consideration to two-dimensional direct interconnection topologies to reduce complexity. We did a comparison between the following 2D direct interconnection network topologies in terms of cost and performance: unidirectional and bi-directional kxk torus, kxk mesh, fat tree, and star network. 2D unidirectional torus topology was chosen, because it offers the best costperformance tradeoff when combined with code mapping optimization. Optimizing code mapping results in a much more localized communication pattern, and hence significantly alleviates the traffic congestion problem of the unidirectional torus topology. Figure 8 shows the routing distance distribution with and without mapping optimization. Some heuristic techniques for optimizing the mapping of LDPC codes on parallel decoding architectures are presented in [ 181. Each cluster of logical nodes is allocated to a bundle of physical nodes in the

proposed architecture. As illustrated in Figure 1, a bundle in defined as the set of physical function node pJ and the two physical variable nodes, pv, and p v , + ~ pc,onnected to it. Logical function nodes of a cluster are allocated to the physical function node of the bundle, and logical variable nodes are divided equally between the two physical variable nodes of the bundle. When the mapping approach presented in [I81 is used, sharing a FU by multiple physical nodes results in more processing aggregation, and hence in less computational load imbalance across FUs in irregular LDPC codes.

2.3. Routing algorithm
Store-and-forward packet switching with short fixedlength messages was used. The torus topology, and in unidirectional case in particular, can have dependency cycles, and hence is prone to deadlocks. Therefore, the routing algorithm must take special provisions to prevent deadlocks from occurring, or detect and resolve them if they are allowed to occur. We used a new flow control algorithm that offers a better cost-performance trade-off compared to traditional techniques that are based on using virtual channels [19], or on deadlock detection and recovery [20]. The new algorithm is used in conjunction with minimal dimension-ordered routing, in which case it guarantees deadlock freedom. Relative addressing, in terms of direction and offset is used to simplify the routing operation. "Hot nodes" are used for avoiding potential deadlocks. These nodes are chosen along the
diagonal of the 2D torus network, as shown in Figure 9, so that every ring in the torus topology is touched by at
least one hot node. Messages can have either one of two priority states: normal or urgent, with normal being the default state. If there is a message at the input port of a hot node that is to be routed along the same dimension,

An urgent message is routed by overwriting the normal message at its destination port if it is not changing dimension, or is converted back to normal state and routed normally if its destination port is free. A single additional output buffer is used at each output port of each node to store the latest normal message routed through that output port and allow later rerouting of overwritten messages. Urgent messages do not change dimension while still in urgent state. This limits the injection of urgent messages in vertical and horizontal rings of the torus to ihose generated by hot nodes. To compare the performance and scalability of the proposed architecture with that of general-purpose parallel machines for implementing LDPCC decoding, the same fixed-point iterative probabilistic decoding of the (510, 1020, 2-3-7, 7) LDPC code with 25 iterations was implemented on a Silicon Graphics (SGI) shared-memory parallel machine. The machine is composed of 16 processors that each has 16KB L1 D-Cache, 16KB LI I-Cache, IMB L2-Unified Cache with 128-byte lines, and runs at a system clock rate of 100MHz. It has 512MB of 2-way interleaved shared memory. Processes, which are logical variable and function nodes in this case, are distributed uniformly among n processors, where n ranges from I to 16, so that each processor is allocated -N/n logical variable nodes and -M/n logical function nodes. Two global arrays are shared by all processor: msgs-v2f, which is used to store messages from logical variable nodes to logical function nodes, and m s g s s v , which is used to store messages from logical function nodes to logical variable nodes. Each array has a maximum size equal to the number of edges in the factor graph of the code times the number of bits per message. These arrays need to be stored in shared memory, because they are used to communicate messages between logical nodes mapped to different processors. Data that do not need to be communicated between different processors are stored in processors local memories. The iteration computation is divided into two phases with barrier synchronization in between. In phase I, all processors perform function node processing of the logical function nodes mapped to them by reading inputs from the shared array msgs-v2f and writing the results into the shared array msgsf2v. On phase 11, they all perform variable node processing of the logical variable nodes mapped to them, by reading inputs from msgsJ2v

4. Conclusion
In this paper, we have presented a new domainprogrammable parallel architecture for efficiently implementing iterative LDPCC decoding. A chip based on this architecture can be programmed to decode different LDPC codes. The design of the interconnection network is optimized to take advantage of the special characteristics of factor graphs by dividing the direct interconnection network into multiple sub-networks of the same size and topology to reduce the effective diameter of the network and increase communication parallelism. 2D unidirectional kxk torus was adopted as a sub-network topology for the proposed architecture, because it offered the best performance-cost tradeoff when combined with mapping optimization compared to other 2D direct subnetwork topologies considered in this work. We introduced a new low cost routing algorithm for deadlock avoidance in 2D torus networks. Since in LDPC decoding the extracted parallelism is communicationlimited, it was shown that by sharing FUs between physical nodes it is possible to decouple the exploited communication and computation parallelism, and cut the cost of computational resources by at least 50% without noticeably degrading the overall performance, which leads to a significant improvement in computational efficiency. Compared to a programmable implementation on a shared-memory general-purpose machine, the proposed architecture not only offers orders of magnitude improvement in cost and performance, but also enjoys 75.6% better scalability.
Even though this paper focuses mainly on the implementation of iterative LDPCC decoding, most of the concepts proposed here can be applied for developing programmable implementations of other iterative
