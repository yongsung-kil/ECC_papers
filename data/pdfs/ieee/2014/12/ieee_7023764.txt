Auto-configurable LDPC Codes For Distributed Storage

Yongmei WEI, Yong Wee FOO, Khai Cher LIM and Fengmin CHEN
School of Engineering Nanyang Polytechnic
Singapore Sophia_wei@nyp.edu.sg

Abstract — The current distributed storage systems mainly rely on data replication to ensure certain level of data availability and reliability. A recent trend is to introduce erasure codes into the distributed storage. Inspired by the RAID system, early attempts have been focused on designing Reed-Solomon (RS) based solutions and other block codes including Low Density Parity Check (LDPC) codes. This paper investigates in details about the usage of the system resources when different configurations of LDPC codes are applied to distributed storage systems. Auto-configurable LDPC-based method is proposed and integrated with the Hadoop system with various configurations of LDPC codes. Simulations show great improvement in terms of encoding and repairing latencies compared with Reed-Solomon. Simulations also show that trade-off in terms of different system resources is achieved through different configurations.
Keywords—Distributed storage, erasure codes, LDPC codes, Reed-solomon codes, auto-configurable
I. INTRODUCTION
Big data storage and processing usually demands large clusters with distributed storage systems. For example, Facebook’s production cluster has more than 3000 nodes, 30 PB of logical data storage. In order to provide enterprise-ready services, reliability of the data stored in the distributed storage system is one of the most important concerns. In the current practice, redundancy usually is added through adding replicas to ensure certain level of reliability. For example, Hadoop uses 3 replicas as the default value which introduces 200% spatial overhead. There is a surging interest to implement erasure codes such as Reed-Solomon (R-S) codes used in RAID 6, Low Density Parity Check (LDPC) codes, network codes and self-repairing codes in distributed storage system [1-5, 16] to reduce the spatial overhead introduced in replication-based methods. The idea of applying erasure codes to add reliability is as follows, the file/data to be stored is sliced into chunks. If an (n, m) erasure code is applied, every m chunks of the data to be stored are encoded to generate n-m parity chunks by the erasure code. Total n chunks are stored in the distributed system. In case of nodes’ failure, decoding process is started to gather required chunks from different nodes to retrieve back the lost chunks. For the Maximum Distance Separable (MDS) codes such as R-S codes, it can tolerate any m chunks’ loss. Researchers from Carnegie Mellon University have developed Reed-Solomon code overlaid on Hadoop [1]. The system realizes R-S codes asynchronously from the critical path. Another cross-object coding [17] is proposed to encode cross

different objects/files and implemented over Hadoop based on RAID-4 like codes. To enhance the repair efficiency for the RS codes, Locally Repairable Codes (LRC) [4] have been proposed to reduce half of the repair bandwidth and disk I/O for single node erasure through adding a few additional local parity chunks. Similarly, another Local Reconstruction Codes (LRC) which also derived from the Reed-Solomon codes have been developed to be used in Windows Azure Storage as well [5]. In [5], a few parities generated from local repair group have been added together with partial parities generated through Reed-Solomon encoding to reduce the repair traffic for single erasure.
It is well known that the range of n and m of the erasure codes such as LDPC can be from tens and easily up to a few hundreds [3]. Furthermore, research [2, 3, 20] has also shown that the space overhead and the reliability of the file also vary with the configurations erasure coding. In another word, different level of trade-offs can be achieved by re-configuring the specific erasure codes applied. For example, high code rate such as 5/6 leads to lower spatial redundancy as 1.2 at the cost of relative high computational complexity and network traffic. Furthermore, in a big data processing platform such as Hadoop, the distributed storage system HDFS share the same system resources with the map-reduce jobs or other types of processing jobs. The system resources include CPU, RAM, disk space, network bandwidth and etc. Depending on the nature of the workload handled by the platform, the system resource is expected to be consumed in different manners. For example, compute intensive workload consumes more CPU power while storage heavy workload consumes more disk I-O and network bandwidth. This poses challenges not only on the initial hardware selection and setup, but also on the final system capacity. This is because it is extremely difficult to predict exact how the system resources will be utilized especially for the platform designed for the general purpose where the workload includes both compute intensive and storage heavy workloads. Thus, it is highly possible the resources are consumed in an unbalanced way ending up with reaching the maximum system capacity with the overutilization of one system resource and under-utilization of the other system resource. To address this challenge, the paper proposes a novel technique to explore the flexibility of the trade-off provided by auto-configuration of the erasure codes so that the system resources can be consumed in a balanced

II. LDPC CODES FOR DISTRIBUTED STORAGE
A. Erasure codes for distributed storage
Redundancy is a must to provide reliability for distributed storage. Traditional way to add redundancy is replication-based method realized by making multiple replicas. For the erasure code based method, redundancy is added by additionally storing parity chunks generated from encoding of the original data. For an (n, m) erasure code, n-m parity chunks are generated through encoding m chunks as shown in Figure 1 (a) where m=6 and n=9. Each data chunks and parity chunks are stored in different nodes of the distributed storage system. In the event of a node failure, decoding process is engaged to gather all the required chunks to decode the lost chunk. Figure 1 (b) shows the repair process for the same erasure code that is used in Figure 1 (a). Node 4 storing data chunk D4 is down leading to the loss of D4. Another working node (Node 10) starts the repairing process by gathering all the chunks required to decode and recover the lost D4.
Figure 2 shows the comparison between the replicationbased method and the erasure code-based method. The erasure code-based method has two advantages compared with replication-based method. Firstly, higher reliability can be achieved by the erasure code-based method compared with that of the replication-base method for the same redundancy rates. Secondly, to increase the reliability, the replication-based method has to sacrifice the spatial efficiency by increasing the number of replicas. On the other hand, the erasure code-based method has an additional way besides sacrificing the spatial efficiency. The alternative way, instead of increasing the spatial redundancy, is to increase the number of chunks m to be encoded or to increase n of the erasure codes.
B. An example of LDPC for distributed storage
Consider a normal block code with parity check matrix H with the size (n-m) x n. LDPC codes refer to the block code with low density. Normally, systematic codes instead of nonsystematic codes are used for distributed storage application because retrieving the original blocks does not require decoding for systematic codes. III. RELIABILITY AND CONSUMPTION OF SYSTEM
RESOURCES
In this Section, reliability of the LDPC with different configurations is discussed, together with the.
A. Reliability and Consumption of system resources – Storage resources
When (n, m) LDPC is applied, (n-m) additional parity chunks are stored besides the m data chunks. This leads to the redundancy rate as n/m. As a result, storage resources such as disk space will be consumed in a different way if different ratio of n/m is used. The reliability of the (n,m) LDPC is not only influenced by n and m, but also determined by the design of the LDPC codes. Researchers have spent years’ work to design LDPC codes to approach the optimal bound. Generally speaking, the studies [2, 3] have already shown that the reliability is increased with redundancy rate n/m. This makes sense that the reliability can be enhanced with the sacrifice of storage more space. Thus, if the storage resources are more precious, lower redundancy rate should be applied to save the storage space. Furthermore, the studies also show that the reliability is increased with n when the redundancy rate n/m remains constant.
B. Consumption of the system resources - Computing and network bandwidth resources
The computing resources such as CPUs and memory and network bandwidth are consumed during the decoding/encoding processes of the erasure codes. For replication-based method, there is no computing required when chunks are lost. Simple copying operations are conducted to recover the missing blocs and the traffic incurred is roughly the same as the amount of the lost data. For erasure code-based method, the consumption of these two types of system resources are more complicated. The computing resources are consumed because of the mathematical operations. For LDPC codes, the operations involved are major XOR operations and it is much simpler than R-S code which requires complicated Galois Field operations. The network bandwidth resources are consumed more than that of replication-based method. This is because of the traffic incurred during the encoding and decoding processes as seen from Figure 1. In the following the encoding and decoding complexities are explained in details.

IV. HADOOP APPLICATION AND SIMULATION RESULTS
Our work is built upon the Xorbas [16]. The Xorbas is built upon Facebook’s Hadoop system running HDFS-RAID [15]. Along the process of integration and testing, a new refined reconstruction process is implemented to further increase the efficiency of the reconstruction. Detailed description of the refined reconstruction process is explained in [14]. The Hadoop [17] cluster is set up on one rack. The cluster comprises of one name node, one secondary name node and forty-eight data nodes. The name node and secondary name node are installed on two bare metal servers. Each server is a product of HP ProLiant DL360P Gen8 with two processors Intel(R) Xeon(R) CPU E5-2667 0 @ 2.90GHz and 64 GB memory. The data nodes are installed as virtual machines on four servers, each with twelve virtual machines. Each server is a product of HP ProLiant DL380P Gen8 with two processors Intel(R) Xeon(R) CPU E5-2640 0 @ 2.50GHz and 48 GB memory. The data nodes are configured with 4 GB memory each. All the machines are installed with CentOS 6.4. The total capacity for this cluster is 18.15 TB. The Hadoop version used for the purpose of this experiment is 0.20.1, built on from the release of HDFS-Xorbas [11]. In HDFS-RAID file system [10], each file is divided into stripes. Each stripe contains the data blocks representing the original file and parity blocks representing the file after it is encoded by the erasure code. For each erasure code, the stripe length represents the number of data blocks and the parity length represents the number of parity blocks for each stripe. Thus, the stripe length equals to m of the erasure codes applied. It is not necessary to divide the file to exact m slices as a file can consist of multiple stripes. E.g. The total number of strips is 12 for the tested file if (14, 10) R-S code is engaged. For each erasure code, there are a maximum number of blocks that can be missing in each stripe. B. Encoding tests
Figure 5 shows the encoding latency in minutes for encoding 10 files with total 1200 blocks and the size of the block is 64MB. The total size of data encoded is 75GB. It can be seen that the encoding latency for all three LDPC codes are better than R-S code and LRC codes. It can be seen from the results that the (42, 30) LDPC code has the shortest encoding latencies although the column weightage of the H is not the smallest among all three LDPC codes. This is because the encoding latency is also influenced by the total amount of parity blocks generated. The selected (21, 8) has the shortest encoding latency to generate one single parity block because the column weightage of its G is in the range of 3 to 7. Since its redundancy rate is 21/8=2.65. It has almost twice of the parity blocks to be generated compared with the (42, 30) LDPC code. By employing map-only processes in Hadoop (1 mapper per file), the encoding latencies for three LDPC codes are further enhanced because each file engages one mapper to do the encoding in parallel. All three LDPC codes achieve fewer than 14 minutes for encoding 75GB of data. The encoding speed is in the range of 1.4 Gbps for (42, 30) LDPC code to 0.74Gbps for (21, 8) LDPC code.
C. Reconstruction tests (decoding)
As mentioned before, 120 blocks have been removed for all the reconstruction tests. The decoding latency to recover all the 120 blocks have been shown in Figure 6. The total size of the repaired blocks is 7.5GB. It can be easily observed that LDPC codes again show a significant improvement in terms of the decoding efficiency. Even for the (180, 120) LDPC code, the decoding time is reduced by more than half compared with using R-S codes. The improvement is due to two aspects. Firstly, LDPC decoding process only requires XOR operation. Secondly, LDPC decoding process, not like that of R-S code, does not require all the blocks to be gathered for decoding. With multiple mappers being employed, the decoding speed reaches 0.51Gbps for the selected (21, 8) LDPC code. D. Auto-configuration of the LDPC codes
Table 4 shows the LDPC code configurations for different use profiles. Use Profile 1 represents the type of system which is prone to be more computing-intensive as computing-intensive operations lead to high consumption of the computing resources such as the CPU load and memory. In this profile, the consumption of disk space may not be intensive. LDPC1 is applied because it has been shown in Figure 4, Table 2 and Table 3 that LDPC1 introduced lowest encoding and decoding traffic, computing complexities. Use Profile 2 and 3 represents the system which has more storage-heavy workloads. In these two use profiles, the consumption of the disk space is high. The LDPC2 and LDPC3 are configured with different reliabilities. This is because that LDPC2 and LDPC3 only has the redundancy rates 1.4 and 1.5, respectively. This autoconfigurable LDPC-based method is integrated with the Hadoop system as well and Figure 8 shows the interface of the proposed system. The consumption of the system resources are monitored by Ganglia Monitoring System [19] in realtime. Depending on the current status of the consumption, the corresponding LDPC code is selected to make sure that the relatively more precious resources are consumed in a more conservative way.

V. CONCLUSION AND FUTURE WORK
The paper presents an auto-configurable LDPC-based framework to add redundancy to the distributed storage system with configurable LDPC codes. It has been discovered that different configurations of LDPC codes has different characteristics in terms of consumption of system resources. The framework is applied to Hadoop distributed file system. Benchmarking results show significant improvement in terms
