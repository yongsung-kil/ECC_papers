ABSTRACT
This paper considers the LDPC code designed for the IEEE 802.11ad WLAN standard and analyzes different architectural options for the hardware decoder. Three decoder architectures are studied that provide different tradeoffs between the throughput and the required amount of the resources for the FPGA implementation. The standard fully parallel decoder is demonstrated to have the maximum throughput of above 6.5 Gbit/s and also the maximum hardware efficiency (a ratio of the decoder throughput to the amount of the consumed hardware logic). The layered decoder architecture allows reusing the hardware check nodes blocks between the subiterations to achieve 46% lower hardware resources utilization relative to the fully parallel decoder though at the expense of the 42% lower hardware efficiency and providing the throughput of 2.1 Gbit/s. The serial-parallel decoder architecture exploits the layered and quasi-cyclic properties of the code matrix and occupies only 11% of the hardware resources needed by the fully parallel decoder but with the throughput of only 159 Mbit/s.
Index terms — LDPC, hardware decoder architecture, IEEE 802.11ad.
1. INTRODUCTION
Low Density Parity Check (LDPC) forward error correction codes have gained a wide popularity and acceptance among the modern wireless communications systems because of their excellent error correction capabilities, moderate decoding computational complexity, and suitability for high speed implementation with a significant level of hardware parallelism. A relative drawback of the LDPC codes is that the decoder architecture is limited to a number of the predefined options, and, in a general case, does not allow to flexibly select an option for the required performance (decoder throughput) or the amount of resource utilization (hardware area). In particular, only the fully parallel and fully serial architectures providing the maximum and minimum extremes of the decoder performance and hardware resources utilization are available for a general LDPC code. However, a different level of the decoder performance and, correspondingly, the logic consumption may be required for the system design.
To overcome this limitation, special regularized LDPC codes have been proposed with their structure defined such as

to allow additional options for decoder realization. Such class of LDPC codes is generally known as architecture-aware (AA) codes. The known effective modifications within this class that are analyzed in this work include LDPC codes with nonoverlapping layers [3] (to support the layered decoding algorithms) and quasi-cyclic LDPC codes [5].
This paper considers the LDPC code designed for the IEEE 802.11ad specification of 60 GHz Wireless Local Area Networks [1]. As the 60 GHz WLAN systems are used to provide multi-Gbps throughput in mobile and handsets devices, the codes are optimized to enable multi-Gbps decoding and simultaneously allow effective hardware implementation. The LDPC code with the code rate ½ of the family of the IEEE 802.11ad codes is analyzed. Different options for implementation of the decoder are considered and the results for the throughputs and occupied hardware resources of the FPGA implementation are presented.

The considered parity check matrix has the property that its rows can be divided into four groups (layers) (see Fig. 3) with each column within a layer having the weight of no more than one. That property allows implementation of the layered decoding algorithm (as described in Section II.C below) that has a twofold impact on the decoder architecture and characteristics. First, the layered algorithm has better convergence properties and requires on average less iterations than for the fully parallel decoder. Second, a new architectural option of the decoder with reuse of some hardware resources between sub-iterations becomes possible (as further discussed in Section III.B).

3. DECODER HARDWARE ARCHITECTURE OPTIONS
3.1. Fully Parallel Architecture
The fully parallel architecture does not rely on any structural properties of the code parity matrix and is available for all LDPC codes. The architecture realizes the BP algorithm directly with every check node and variable node of the parity check matrix H having corresponding hardware instances at the decoder implementation. The top structure of fully parallel decoder is presented in Fig. 7. All messages are computed in parallel so that one iteration of the decoding algorithm is computed in one clock cycle. The fully parallel architecture allows achieving the highest throughput and the lowest latency. However, implementing each algorithm unit with a hardware block and a significant amount of wiring used to carry algorithm messages require a large hardware area to realize the decoder. For the considered code, the fully parallel decoder uses 672 hardware variable nodes and 336 hardware check nodes that perform the processing as described in Section II. As a consequence, the goal of the alternative architectures is to look for a lower level of hardware resources utilization even at the expense of the reduced decoder throughput since the maximum throughput cannot be always utilized by the application.

3.2. Layered Decoder Architecture
The layered decoder architecture uses the layered property of the parity check matrix and implements the LBP algorithm to improve the convergence and achieve some level of hardware reuse between the decoding sub-iterations. According to the algorithm description provided in Section 2.4, for each subiteration the hardware decoder has to update only check and variable nodes, related to the layer being updated. Since the rows of a layer have no overlapping and the maximum column weight is one, sub-iteration can be completed in one clock cycle without any scheduling conflicts for n(p) value calculation. As such, the check nodes hardware can be reused between the sub-iterations and the number of hardware check nodes is decreased to 84 instances that is 4 times lower in comparison with the fully parallel architecture. The layered decoder includes variable nodes, check nodes, direct and inverse interconnections between the two types of the nodes, and delay registers for messages from the check to variable nodes. Every variable node is associated with only one check node that allows simplifying the hardware structure. However, interconnections between the nodes are more complicated in comparison with the fully parallel decoder as they are required to switch the nodes connections every clock according to the current layer being processed. The message registers are used for delaying messages from the check to

variable nodes according to the requirements of the LBP algorithm. It can be seen that the layered decoder architecture is conceptually more complicated in comparison with the fully parallel one, but the total number of the occupied hardware resources is lower as confirmed further by the synthesis results.

3.3. Fully Serial Decoder
The fully serial architecture is the minimum extreme of decoding algorithm implementation (as opposed to the maximum extreme provided by the fully parallel architecture). This architecture assumes sequential calculation of all the algorithm messages so that all the operations could be performed using one computational node and, hence, requiring the least amount of hardware resources. Serial decoders have the lowest throughput and therefore are predominantly used in software decoder realizations. The fully serial architecture is not considered in details in this work.

3.4. Serial-Parallel Decoder
The serial-parallel architecture is an intermediate approach between the fully parallel and fully serial architectures that becomes feasible due to the QC structure of the analyzed LDPC code.

The architecture of the serial-parallel decoder shown in Fig. 9 is proposed in this work. The LBP decoding algorithm is applied and one sub-iteration of the LBP algorithm (corresponding to an update of one layer) is calculated over 2×Nne(p) clock cycles, where Nne(p) is the number of nonempty elements of the base matrix for the p-th layer. The Nne(p) cycles are used for the check nodes updates and then another Nne(p) cycles to update the variable nodes. However, orthogonality of the two rows forming one layer of the base matrix allows performing check nodes updates and variable nodes updates in parallel thus reducing the effective time for the update of one layer to Nne(p) clock cycles.

The elementary matrix size Z of the QC-LDPC codes determines the degree of algorithm parallelism, achieved by the decoder. Increasing its value for the fixed codeword length allows reducing the number of the clock cycles needed to calculate the layer update that increases the throughput, but also raises the hardware resources consumption. Reducing Z has an opposite effect and, thus, optimizing Z allows achieving the tradeoff between the decoder performance and resource utilization. For the considered code Z is equal 42, and the serial-parallel decoder uses 42 hardware variable nodes and 42 hardware check nodes, that is 16 times less in comparison with the fully parallel architecture.

In this section the three considered decoder architectures (fully parallel, layered, and serially parallel) are compared in terms of the achievable throughput and the required hardware resources.

With Niter being the maximum number of iterations, Eq. (7) gives an estimate of the minimum decoder throughput. However, early termination is supported by the decoder algorithm and implementation, and, as a result, the average number of the iterations is less or equal to their maximum number and that should be accounted in the throughput estimate of Eq. (7). As the average number of iterations depends on the operating signal-to-noise ratio (SNR), the SNR corresponding to 1% block error rate (BLER) was selected for the throughput calculation. With the help of numerical simulations, the average number of iterations for the analyzed LDPC code was estimated as 5.1 for the BP algorithm (used by the fully parallel decoder) and 4.1 for the LBP algorithm (applied in the layered and serial-parallel decoders).

Expression (7) does not account for the time required for decoder initialization and termination that is identical for all the architectures and can be optimized to match the basic decoder throughput as provided by (7).

Hardware resource utilization directly depends not only on the chosen architecture but also on the parameters of the decoder fixed point arithmetic. Higher precisions of the signals improve the accuracy of a fixed point decoder, but require more logic for the implementation. For the considered decoders the fixed point arithmetic parameters were unified and optimized so that the Eb/N0 degradation relative to the floating point model is below 0.1 dB. To achieve that the bit widths for

the input LLRs and internal messages were taken equal to 8 and 5 bits respectively.
All the three considered hardware decoder architectures were implemented for the Xilinx Kintex7 XC7K325T (speed grade -2) FPGA device. The clock frequency for all the architectures was taken equal to 100 MHz.
The basic hardware resources of FPGAs are look-up-tables (LUTs) and registers. Thus the hardware resource utilization is presented as the number of the used slice LUTs and the slice registers. The total available hardware resources of the considered FPGA are 203800 LUTs and 407600 registers.

Analysis of the results shows that, as expected, the fully parallel architecture provides the highest throughput of 6.5 Gbps and is also the most hardware efficient realization of the LDPC decoder requiring the least amount of the hardware resources per 1 Mbit/s of the throughput. However, the drawback of this architecture is its inability to scale down for a lower throughput. Hence, the fully parallel decoder can be recommended for the case when the maximum throughput or the minimum latency are sought.
The layered decoder requires 4 clocks per one algorithm iteration, thus lowering the throughput in 4 times. However, better convergence properties of the LBP algorithm lead to decreasing the average number of iterations by 20% that gives the overall decoder throughput of 2.06 Gbit/s. Low resource utilization of the serial-parallel architecture allows recommending it as an LDPC decoder platform for communication systems with relatively low throughput and essential hardware constraints.
5. CONCLUSION
LDPC codes are one of the most prospective error control schemes that feature both high error correction capabilities and suitability for efficient hardware implementation. One of the limitations for LDPC codes is a reduced choice of the hardware architecture options that potentially cannot meet the requirements of a particular application. To remove this limitation, the LDPC code structure should include special provisions to allow additional architectural options for the decoder.
This paper presents the fully parallel, layered, and serialparallel hardware decoder architectures for the LDPC code of the IEEE 802.11ad specification. The three decoder architectures are analyzed by comparing their throughput and the required amount of the resources for the FPGA implementation. An additional metric of the relative hardware efficiency is introduced a ratio of the decoder throughput to the amount of the consumed hardware resources.
The fully parallel decoder not relying on any particular code structure properties is demonstrated to have the maximum throughput of above 6.5 Gbit/s and the also the maximum hardware efficiency. The second layered decoder architecture exploits the layered property of the code parity check matrix

that allows reusing check nodes hardware to achieve 46% lower hardware resources utilization relative to the fully parallel decoder though at the expense of the 42% lower hardware efficiency with provision of the maximum throughput of above 2 Gbit/s. The third serial-parallel decoder architecture also uses the layered property of the parity check matrix and, in addition, relies on the quasi-cyclic structure of the code. That allows realizing an architecture occupying only 11% of the hardware resources needed by the fully parallel decoder though with the corresponding relative throughput of only 2.4% or 159 Mbit/s. The recommendations on using different decoder architectures are given that suggest selecting the architecture that meets the throughput requirements of the application while minimizing the amount of the consumed hardware resources.
