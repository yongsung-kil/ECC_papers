High-throughput DOCSIS Upstream QC-LDPC Decoder
Michael Wu1,2, Bei Yin1, Eric Miller1, Chris Dick2, and Joseph R. Cavallaro1
1Department of ECE, Rice University, Houston, TX; e-mail: {mbw2, by2, ewm1, cavallar}@rice.edu 2Xilinx Inc., San Jose, CA, USA; e-mail: chrisd@xilinx.com

Abstract—The newly released Data Over Cable Service Interface Specification (DOCSIS) 3.1 standard enables gigabit-speed broadband connection over cable television network. The main challenge at the upstream receiver is a computationally intensive LDPC decoder which need to achieve the 1 Gbps throughput with low block error rates. In this paper, we present an implementation of the upstream LDPC decoder on Xilinx FPGA using the VIVADO high level synthesis tool. We show that the resulting design can achieve 1.43 Gbps throughput on a Xilinx Kintex7 XC7K480T FPGA at 204.8 MHz clock frequency with 24% device usage. To our knowledge, this paper presents the first hardware implementation of DOCSIS 3.1 upstream LDPC decoder on FPGA.
I. INTRODUCTION
LDPC codes offer capacity-approaching error correction performance and are widely used in many wireless communication standards [1] such as IEEE 802.16e WiMAX and IEEE 802.11n. Recently, the newly released Data Over Cable Service Interface Specification (DOCSIS) 3.1 standard [2] combines wideband OFDM, 4K QAM modulation, with quasi-cyclic low-density parity-check (QC-LDPC) code to exceed the 1 Gbps upstream transmission data rate with low block error rates.
To perform error correction, the receiver requires an LDPC decoder, which is extremely computationally intensive. The layered scaled min-sum LDPC decoding algorithm is broadly adopted to reduce hardware complexity while ensuring fast error-correction convergence [3], [4]. However, due to the data dependency between adjacent layers, the layered decoding algorithm updates soft bit-level reliability values by processing the parity check matrix sequentially layer by layer which limits the achievable decoder throughput. In addition, compared to the 600 Mbps throughput requirement in 802.11n [5], the DOCSIS 3.1 standard dictates a throughput requirement in excess of 1 Gbps. Furthermore, the block length of the QCLDPC code in the DOCSIS 3.1 standard is also much longer (16200 bits compared to 1944 bits for 802.11n), which increases computational complexity. As a result, meeting the performance requirement of the DOCSIS 3.1 standard poses significant challenge compared to the 802.11n standard.
The key to maximizing decoder throughput is a good decoding schedule that enables concurrent processing by reducing the data dependency between adjacent layers. In a typical decoder design, an efficient implementation is enabled through hand-coded RTL. As an alternative, the Xilinx Vivado high level synthesis (HLS) tool uses C/C++/SystemC as input

languages and outputs register-transfer level (RTL) designs. This tool can provide quality of result (QoR) similiar to handcoded RTL, while dramatically reducing the design time [6]. In this paper, we implemented a layered min-sum LDPC decoder with a good decoding schedule using the Xilinx Vivado HLS tool that exceeds the DOCSIS 3.1 standard performance requirement. We show that this design achieves 1.43 Gbps throughput on a mid-range Xilinx Kintex 7 XC7K480T FPGA running at 204.8 MHz with only 24% device usage.
II. DOCSIS 3.1 UPSTREAM FEC
A. DOCSIS 3.1 Upstream FEC Codes
The newly released DOCSIS 3.1 standard [2] enables 1 Gbps upstream transmission data rate with 192 MHz OFDM channel and 4K QAM modulation. To achieve good errorcorrection performance, the new standard adopts three QCLDPC [7] codes for forward error correction (FEC) with rates 0.75, 0.85 and 0.89. The codes differ in length (L) and the number of information bits (K). The standard specifies L ∈ {1120, 5940, 16200} and K ∈ {840, 5040, 14400}, respectively, for rate 0.75, 0.85 and 0.89 codes.
The standard employs different mother matrices for rate 0.75, 0.85 and 0.89 codes which are used to construct the parity check matrices. The mother matrix for the longest QCLDPC code, the rate 0.89 code, is shown in Fig. 1. Each number in the mother matrix represents a Z by Z cyclicshifted identity matrix, where Z = 360 for the rate 0.89 code. A number in the mother matrix represents a shift value. For example, the number 93 in the above matrix indicates a Z × Z identity matrix is cyclic-shifted to the right by 93 positions. The symbol ‘−’ in the matrix indicates a Z × Z zero matrix. The key difference among rate 0.75, 0.85 and 0.89 codes is that the standard specifies a different Z value for each code, where Z ∈ {56, 180, 360} respectively for rate 0.75, 0.85 and 0.89 codes.
B. Layered Scaled Min-Sum Decoding Algorithm
To decode a QC-LDPC code, the layered scaled min-sum decoding algorithm is widely used [3], [4], [8], [9]. The decoding algorithm processes the mother matrix layer by layer, where each layer corresponds to a row of the mother matrix. To reduce complexity, the computation at the check node can be simplified using the two-min algorithm [8], [9] which is summarized in Alg. 1. The two-min algorithm takes advantage of the fact that (1) can be computed easily if the two smallest |Qm,k| values and the product of the signs of all VTC messages, sm = k∈Nm sign(Qm,k), are known. To this end, the twomin algorithm decomposes the decoding algorithm into two phases [8]. The first phase, the MIN phase, computes the VTC messages, Qm,k∀k, and finds the two smallest |Qm,k| values, min1 and min2. The second phase, the SEL phase, computes the CTV messages, Rm,k∀k, using the first and second smallest |Qm,k| values and sm. The second phase then updates the decision LLRs.
Due to the data dependencies between these two phases, the SEL phase of the current layer needs to update the decision LLRs before the MIN phase for the next layer can start. As a result, data dependencies between these two phases limit the LDPC decoder throughput as steps of the algorithm must occur in a sequential manner, as shown in Fig. 2. The main challenge

is that we need to break the data dependencies between these two phases to achieve the required throughput.
III. QC-LDPC DECODER IMPLEMENTATION
We used the Vivado HLS tool to implement our DOCSIS 3.1 decoder. Fig. 3 is a high-level overview of the decoder. The DOCSIS 3.1 decoder consists of two main functional units, a MIN unit and a SEL unit, that exchanges data through memory. In the following section, we first describe the MIN and SEL units. Since the DOCSIS 3.1 standard specifies three different code profiles, we then show the modifications required to enable reconfigurability in order to support these code profiles. Finally, we describe the quantization scheme used to maintain good performance while reducing overall memory requirement.

We leverage the fact that the computational steps at check nodes are completely data parallel, where each node performs the same set of computations on different sets of data. Consequently, we decompose the computational steps into 360 lane wide vector operations, where each lane performs the computational steps required by a check node. Fig. 4 is an overview of the MIN unit.
Using Vivado HLS, we built a number of vector operators which we used to construct the MIN unit, as shown in the right side of Fig. 4. The vector lane-wise addition (vadd), vector lane-wise absolution value (vabs), and vector lane-wise compare and select (vcmpsel) perform operations in Line 3,5 of Alg. 1. In Line 4 of Alg. 1, we map sign value +1 to 0 and sign value -1 to 1. The product of the signs can then be computed efficiently with bitwise XOR.
However, each check node processes a different set of nodes and thus requires a different set of input data. While each VTC message Qm,k and CVT message Rm,k are unique to each node, the decision LLRs, Lk, are shared among the check nodes. To this end, the MIN unit reads 360 consecutive decision LLRs as we process a layer of the mother matrix. Since each element of the mother matrix is a shifted identity matrix, the decision LLRs, Lk, can be distributed in the right order across lanes using a cyclic-shifter. The shift required is specified by the mother matrix. For example, for the first layer, the shift value for the first 360 consecutive decision LLRs is 93 for the rate 0.89 code.
B. SEL Unit
We use a similar approach to the SEL unit. We decompose the computation steps outlined in Line 8-14 of Alg. 1 into

360 lane wide vector operations. We use the vector lanewise absolute value (vabs), vector lane-wise compare and select (vcmpsel) and the vector lane-wise select (vsel) to performs the steps outlined in Line 8–12 of Alg. 1. We use an XOR to perform the step in Line 13 of Alg. 1. Finally, we use the second vsel to compute the CTV messages and the vector addition to update the decision LLRs in Line 14 and 15 of Alg. 1.
Since we used a cyclic-shifter in the MIN unit to perform a cyclic shift on the input decision LLRs, we use a cyclic-shifter to perform the inverse cyclic shift to update these decision LLRs.
C. Scheduling
With the previous schedule, although we have dedicated units for the MIN phase and the SEL phase, the recursion cannot be pipelined efficiently. In order to increase the throughput of the LDPC decoder, we need to overlap the MIN phase and the SEL phase. To this end, we break the data dependencies between the MIN phase and SEL phase by using the following schedule. We overlap the SEL phase of the current layer with the MIN phase of the next layer. After a small fixed delay, the MIN phase for the next layer can begin after the SEL phase begins for the current layer. We introduce a small delay between the beginning of these two phases to avoid the memory conflict between updating decision LLRs in the current layer during the SEL phase and reading decision LLRs in the next layer’s MIN phase. In fact, an updated decision LLR obtained in the SEL phase is forwarded to the MIN phase for the next layer and is immediately consumed. By concurrently scheduling the MIN phase and SEL phase, we can improve the LDPC decoder throughput by approximately twofold.

D. Reconfiguration
The decoder needs to support the three code-rates specified by the DOCSIS 3.1 standard. First, different code profiles require different mother matrices, which requires a different set of shifts. We store the three sets of required shifts in a LUT table. To switch code profiles, we simply switch the starting address of the LUT table. Second, different code profiles have different Z values. Since the vector operators in MIN and SEL units have 360 lanes, to support smaller Z values, we simply disable lanes for smaller values. However, the cyclic-shifter needs to be reconfigurable to support smaller Z values. We can construct a cyclic-shifter that supports any Z ≤ 360 wide shifts. We used the design presented in [8], where the authors presented a reconfigurable Z ≤ 360 wide cyclic shifter that uses two 360-wide cyclic shifters and an array of muxes. As shown in Fig. 7, to cyclic shift the input by c places with a Z wide cyclic-shifter, the 360-wide left shifter cyclic-shifts the input by 360−Z +c and the 360-wide right shifter cyclic-shifts the input by c. The array of muxes can then be used to obtain the correct output.
E. Quantization

The largest memory requirements for the DOCSIS decoder is CVT messages. We stored these messages in block memory on the FPGA. Since the memory requirement for the VTC messages and the decision LLRs is much smaller, we stored these values within look up tables on the FPGA.
IV. RESULTS
We implemented the proposed QC-LDPC decoding algorithm using a fully parallel architecture with Vivado HLS 2013, a high level synthesis tool for Xilinx FPGA. As described in Section III, the design can be parameterized to support all three LDPC code rates. Our resulting design is efficient enough to fit into a mid-end Xilinx Kintex7 XC7K480T FPGA.

To reduce the memory requirement of the design, we investigated the minimum bitwidth required to achieve good performance. We evaluated the performance of the decoder in an additive white Gaussian noise (AWGN) channel. We found that the internal computational width needs to be 8 bit and the input channel LLR width needs to be 6 bit. This allowed the DOCSIS decoder to achieve < 0.05 dB performance loss at BER performance of 10−8. Although we can use an input channel LLR width of 5 bit and internal computational width of 7 bit, the corresponding performance loss at BER performance of 10−8 is approximately 0.2 dB. To achieve BER performance close to that of the floating point reference, we kept the internal computational width to 8 bit and the input channel LLR width to 6 bit. V. CONCLUSION
In this paper, we presented the first implementation of the DOCSIS 3.1 uplink LDPC decoder on FPGA. The implementation is able to exceed the data throughput requirement specified by the standard with only 24% device utilization on a Xilinx Kintex7 XC7K480T FPGA. There are many avenues for future work. For example, there are alternative algorithms such as the self-corrected Min-Sum algorithm [10] which may reduce the bit-width requirement while maintain good BER performance.
