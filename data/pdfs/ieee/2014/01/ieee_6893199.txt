Memory Sharing Techniques for Multi-standard High-throughput FEC Decoder

Beijing Institute of Technology Beijing, 100081, China

Abstract—Nowadays multi-standard wireless baseband, Convolutional Code (CC), Turbo code and LDPC code are widely applied and need to be integrated within one FEC module. Since memory occupies half or even more area of the decoder, memory sharing techniques for area saving purpose is valuable to consider. In this work, several memory merging techniques are proposed. A non-conflict access technique for merged path metric buffer is proposed. The results show that 41% of total memory bits are saved when integrating three different decoding schemes including CC (802.11a/g/n), LDPC (802.11n and 802.16e) and Turbo (3GPP-LTE). Synthesis result with 65nm process shows that the merged memory blocks consume merely 1.06mm2 of the chip area.
I. INTRODUCTION
Forward Error Correction (FEC) codes including Convolutional Code (CC), Turbo code and LDPC code are widely adopted in many wireless standards, especially high throughput standards such as 3GPP-LTE, 802.11n and 802.16e. An efficient FEC decoder supporting all these codes is in demand for today’s multi-standard baseband systems. Therefore interalgorithm architectural merging has received more attention in recent years. Sun [1] proposed a Turbo/LDPC duo-mode decoder which shows 80% area reduction for the kernel computation unit inside the data path. TU Dresden [2] researches on the area consumption for tri-mode decoder and yields 16% of saving in data path and 5% of saving in inter-connection network. Condo [3] proposed a NoC based Turbo/LDPC decoder with a special memory arrangement pattern. Multi-standard FEC [4] discussed memory saving in general but results in negative conclusion. In conclusion, most of the papers are focusing mainly on the data path, and less investigation has been done on the memory system. Considering that the memory in a FEC decoder consumes 50%-70% of chip area, reported in [5] [6] [7] [8], it is desirable to reduce the area consumption by merging and sharing the memory for these algorithms. In this paper, the unified trimode FEC memory system is introduced. The memory block arrangements, sharing techniques for global memory system and Processing Element (PE) buffers are introduced. The area saving in terms of bits is discussed in detail.

II. TOP-LEVEL ARCHITECTURE OF TRI-MODE FEC
DECODER
Stemming from the single mode decoders including Turbo decoder [9], LDPC decoder [10], and also from multi-mode designs [11] [2] [1], we proposed a unified tri-mode FEC decoder. In our proposal, Quasi-Cyclic LDPC (QC-LDPC) is considered targeting on 802.11n and 802.16e, with maximum parallel degree Z = 96. A turbo-decoding message-passing (TDMP) also referred as Parallel Layered Decoding algorithm is applied as the decoding algorithm. Turbo decoding applied in 3GPP-LTE(A) and HSPA is considered. Other Turbo codes with code length less than 6144 and trellis states equal to 8 are supported as well. Bahl-Cocke-Jelinek-Raviv (BCJR) algorithm for MAP decoding is utilized. Convolutional codes with constraint length less than 9 are considered. Soft-input-hardoutput Viterbi algorithm is applied. All these algorithms can be treated as forward-backward recursion, therefore merging into a unified architecture is admissible.
The Top-level architecture is shown in Fig 1. The memory subsystems arrangement and their functions are listed in Table I. A-posteriori messages (for LDPC) or extrinsic messages (for Turbo), channel messages and decoded codes are stored in global memory. It connects with PEs through a permutation network. Assuming that the number of PEs is P , there are 5P 8-bit memory banks. Temporary data buffers, including Input Buffer (IPB), Alpha Buffer (ALB) and Inner Buffer (INB), are located in the PEs.
Because the parallel degree of LDPC is 96, when 8 layers are processed within one PE, the total amount of PEs should be not less than 12. Turbo decoding processes 8 states in each PE, and a parallel degree of 12 is necessary for high throughput scenarios like 3GPP-LTE. Therefore P = 12 is selected in our approach. A similar decision can be found in Sun’s work [1]. Other values of P may lead to differences in the numerical memory sharing results, while the architecture is similar.
Table I shows that all these memory subsystems can be utilized for performing more than one algorithm, which enables saving memory area. The data arrangement pattern, access control as well as saving result for each memory subsystem are discussed in the following sections.

III. GLOBAL MEMORY SUBSYSTEM
The global memory subsystem shared by all PEs stores a-posteriori or extrinsic messages and supplies message exchange among PEs. For Turbo decoding, it supplies the channel data being decoded and extrinsic messages transferred between half iterations. For LDPC decoding, it supplies aposteriori messages with Z parallel degree and is updated after the PE finished the processing of one sub-matrix. For CC decoding, it stores the channel data as well as the final decoded bits.
The global memory subsystem composes of a linear address generator, which supplies the forward-backward recursion step. An address transform module converts the linear address into interleaved addresses for Turbo or non-zero matrix locations for LDPC decoding. The address of each memory bank is specified in the Memory Bank Address Mapping module. For each memory bank, the read/write address as well as write enable signal is independently configurable and the memory banks offer the capacity of reading five 8bit messages in parallel and writing another five messages back simultaneously. The data is reordered by a Permutation Network based on the permutation route. The sequence of the permuted data corresponds to the PEs. The data from the PEs is permuted by the inverse reorder operation and written back to the memory banks. A simplified diagram is shown in Figure 2.

In the first half of Turbo MAP decoding (MAP1), the channel data r0, r1 as well as extrinsic a-priori message La are required simultaneously to compute α metrics. The extrinsic a-posteriori message Le is written back to the memory banks. For MAP2 mode, r0, r2 and Le (respect to MAP1) is read and La (respect to MAP1) is written back. The difference is that interleaving is required for r2 and Le, and inverse interleaving is required for La. Assuming that the memory architecture is homogeneous, at least five memory banks are required for r0, r1, r2, Le, La respectively. If heterogeneous memory is available, r0, r1, r2 can be combined into one bank with three times longer data width. The entire code of length N is split into P segments and decoded in parallel and therefore totally 5P memory banks are required. The depth of memory is D = N/P . For a satisfying decoding performance, the data is quantized to at least 8-bit. For the case of 3GPPLTE Turbo, N = 6144, and we set P = 12 with D = 512, the total amount of banks is 60.
Regarding to CC(Z, 1, M ), the code length being supported is flexible. Because there is no iteration operation in the Viterbi algorithm, the long convolutional code can be split into frames and decoded in a time division manner. The maximum code length supported for each frame is N . The branch metric update in each clock cycle requires Z channel messages to be read simultaneously. In our proposal, Z is 2 to 4 decided by the standard specification. Therefore in each time up to four messages are read from the memory banks. With the quantization of three bits for channel messages, and another bit representing the puncturing flag, totally 4Z bits bandwidth is necessary. When merging with Turbo, the remaining four bits can be used for another frame of data. The decoded bits are written back to global memory as well, which requires an extra bank. Similar to Turbo decoding, the code of length N is further split into P segments which dispatch to P processing elements. Therefore totally Z · P of messages are demanded. For adapting to the same parallel scale with Turbo, totally 60 banks are required for Z = 4 and 30 banks for Z = 8. When totally 60 banks are employed for adapting Turbo decoding, extra banks can be used for storing extra frames as well. For each bank, the memory depth D is counted by the sum of the data length (N/P ) and the traceback length T B (T B > 5M ). Take CC(2,1,9) with N = 5600 for example, it can be mapped into 24 512-depth 4-bit memories, or 12 512-depth 8bit memories. Extra 12 memories are required for the written back bits.
The memory bank assignment scheme for LDPC decoding is decided by the parallel degree of Z defined by the LDPC matrix. In each time, Z data is fetched using the same address. Therefore it requires at least one memory bank with 5Z bits data width. For the case of a read/write operation being performed simultaneously, the memory should be designed as dual port. B. Multi-algorithm Merged Memory Subsystem
A unified memory subsystem with 5P 8-bit memory banks is proposed for adapting Turbo, LDPC and CC. Each five memory banks are in a group. One group can supply 40 bits at one time. The 40 bits correspond to five 8-bit data for Turbo, also correspond to eight 5-bit data for LDPC. Because the address for accessing the eight messages in LDPC mode is the same, so that no addressing mismatch happens. It overcomes the discrepancy that Turbo requires a larger data width whereas LDPC requires a larger parallel degree. Otherwise 8P memory banks instead of 5P are demanded. CC with Z ≤ 4 requires no more bandwidth than Turbo. In summary, all memory banks are re-used for all algorithms, and the global memory system is highly re-used.
Although LDPC requires dual port memories, it is possible to raise the frequency to double only for the memory system. Because each memory banks are in small size, the latency is acceptable (less than 2 ns supplied by the memory block specifications). The logical equivalent model is still dual port RAMs, which requires no modification to other parts of the decoder.
Considering all these techniques, the global memory system is shown in Figure 3. For simply show, P = 4 is demonstrated here.

In the case of merging 802.11n, 802.16e and 3GPP-LTE, the code size is 1944 (LDPC), 2304 (LDPC) and 6144 (Turbo). The memory usage is at least 6144·8bit·5banks = 245760bits even supporting Turbo merely. The amount of memory for LDPC codes 2304 · 5bit = 11520bits can be saved. CC decoding consumes no more memory resource. LDPC in these two modes consumes less resource compared to Turbo. In another occasion when DVB-S LDPC is considered, 64800 · 5 = 324000bits is required to be stored, and 245kbits are saved. In conclusion, more than 50% of bits are possible to be shared. The memory pattern being adopted finally is 512word 8-bit single port.
IV. PE LOCAL BUFFERS
Processing Element (PE) is a multi-algorithm data path for Soft-In-Soft-Output (SISO) processing. The diagram is shown in Figure 4. It consists of several common arithmetic kernel modules including α update (including branch metric calculation), β update (including branch metric calculation). It also includes Λ update for LDPC decoding, LLR update for Turbo decoding, and TB for Viterbi traceback. Several local buffers are introduced mainly including Input Buffer, Alpha Buffer, and Inner Buffer.
A. Alpha Buffer
For Turbo and LDPC decoding, Alpha Buffer builds up a bridge between the forward recursion and the backward recursion. The α for each time step in the forward recursion within the sliding window is recorded. For both algorithms, the forward and the backward recursion can be performed simultaneously, which leads to the buffer to be dual port. Eight data with a quantization of 9 bit are stored. Considering a sliding window of 32 (For Turbo), the buffer should be at least 72 bits in width and 32 bits in depth.
It is also applied for storing the path metrics in CC decoding. Because there are maximum 256 states within a CC trellis stage and only 8 of them are processed at one time, a depth of 32 and a bandwidth of 8 · 9bits are required. The memory patterns for these schemes are depicted in Table III.

C. Input Buffer
In forward-backward recursion, the incoming data are demanded for both forward path and backward path. A stack RAM can be applied for buffering the incoming data within the sliding window. All the posteriori messages needed for backward path will be obtained from it other than from the global memory. It is odd/even-bank banks organized. In Turbo decoding, the incoming data is stored. The depth of the Input

Buffer equals the width of the sliding window and the width is 3 × 8 bits. For LDPC, the width is 8 × 5 bits. The apriori message rather than the incoming a-posteriori is stored. The depth of the buffer equals the total number of nonzero elements within each layer (less than 32 for 802.11n and 802.16e). Because Viterbi decoding performs traceback instead of backward recursion, no input data buffering is needed. For a unified architecture, two buffers with 40 bits in width and 32 in depth are required, shown in Table IV.
D. Inner Buffer
When decoding LDPC codes, the extrinsic messages for all non-zero matrices are required storing, which equals less than 90 messages among all modes in 802.11n and 802.16e. Each time 8 layers 8-bit extrinsic messages are stored. For Viterbi, Inner Buffer is re-used as traceback memory, which records the path transition information. For each state inside one trellis step, one bit path transition information is required. A compression storage scheme is proposed that combining the survival path bits within each 8 states into an 8-bit data. Since there are totally (up to) 256 states, which will be dispatched into 8 banks in order to reuse with LDPC mode, and the data width of RAM is 64, thus there are 4 cells in necessary for recording 256 states. For a sliding window of 64 (For CC), the deep of RAM is at least 256. The memory sharing rate is shown in Table V. It is revealed that 35% of bits are saved due to the merging scheme.

VI. CONCLUSION
In this paper, the memory sharing schemes for multialgorithm FEC decoding are proposed. RAM blocks for storing a-posteriori messages and forward-backward recursion messages are all re-used for the area saving purpose. Global memory compressing for LDPC 5-bit a-posteriori messages is proposed, which saves 1/3 of memory banks. Conflict-free network for path metric access is provided for arbitrary trellis size. As a result, more than 40% of bits saving is obtained. The proposed schemes are adopted in our multi-standard high throughput FEC processor, and it shows great area efficiency to other tri-mode or independent FEC implementations.
