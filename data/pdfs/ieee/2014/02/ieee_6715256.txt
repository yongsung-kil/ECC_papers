High Throughput LDPC Decoder on GPU
Yong Lin and Wensheng Niu

Abstract—The available Lower Density Parity Check (LDPC) decoders on Graphics Processing Unit (GPU) do not simultaneously read and write contiguous data blocks in memory because of the random nature of LDPC codes. One of these two operations has to be performed using noncontiguous accesses, resulting in long access time. To overcome this issue, we designed a multi-codeword parallel decoder with fully coalesced memory access. To test the performance of the method, we applied the method using an 8-bit compact data. The experimental results demonstrated that the method achieved more than 550Mbps throughput on Compute Unified Device Architecture (CUDA) enabled GPU.
Index Terms—CUDA, decoding, GPU, parallel processing, LDPC code.

L OW Density Parity Check (LDPC) codes, developed by Gallager in 1960 [1] and rediscovered in 1996 [2], are a class of the error-correction codes whose performance is close to the Shannon limit [3]. A commonly used LDPC decoding algorithm is Belief Propagating (BP) algorithm which iteratively exchanges extrinsic messages between variable and check nodes. It is suitable to parallel implementation.
Nowadays, Graphics Processing Unit (GPU) has been developed into a general purpose multi-core processor. New GPUs can be used not only in traditional graphic processing but also in general data processing. As a parallel processing tool, GPU has been employed in several approaches to accelerate decoding of LDPC codes. These decoders on GPU are less costly but more flexible and scalable than hardware decoders [4-9]. Unfortunately, most of these implementations cannot simultaneously read and write contiguous data in global memory; one of these two operations has to be performed using non-coalesced memory accesses [5-9]. When these noncontiguous memory locations do not fall in a cache line, additional memory transactions are needed and memory access latency is increased. Since arithmetic instructions issued by BP are not sufficient enough to hide global memory latency, the GPU capacity is not fully utilized. Hence, there still is large amount of room to further improve the performance of GPU based decoders by developing a new mapping algorithm with less memory access time.

II. SUM PRODUCTION ALGORITHM

LDPC code can be defined by the equation H · xT = 0, where x is a codeword, and H is the parity-check matrix, which is a sparse matrix with a few of ones in each column and row. Intuitively, the LDPC is also modeled by its Tanner graph [10] in which edges connect the variable and check nodes according whether their corresponding elements in the check matrix are ones. If the number of ones in each row/column is constant, it is called a regular LDPC code, otherwise, an irregular code.
The most common decoding algorithm for LDPC codes is BP, also known as Sum Product Algorithm (SPA) [10]. SPA iteratively passes the messages between check nodes (CNs) and variable nodes (VNs) along the edges connecting them. We decode BPSK modulated LDPC codes transmitted through Additive White Gaussian Noise (AWGN) channel with zero mean and variance δ2. To hide global memory latency, there should be sufficient independent arithmetic instructions issued when waiting for the global memory access to complete. Hence, our approach improves decoding speed in two ways. One is to reduce the global memory access time via a novel mapping method, and the other is to issue more arithmetic instructions for each memory access by using 8-bit data.

All decoding processes of the 32 codewords will be carried out by threads within a single block. In a block, threads in the same warp process the same sections from different codewords. Since update messages for threads in same warp are stored in contiguous addresses, which will be considered in the next section, all global memory accesses are fully coalesced.

Our strategy ensures that threads in a warp execute the same functions on the same sections. There is no divergence among them and the thread workloads are balanced for regular LDPC codes. Another advantage of this strategy is that two message passing steps can be integrated into one kernel and coordinated by synchronization. As a result, there is no call overhead during execution, and decoding is further accelerated.

D. Compacted Data Transfer
The L2 cache line is 128 bytes in the device of CC3.0. If the message data is 32-bit, only one codeword data can be loaded to a thread per transaction. In our work, we use an 8-bit fixed-point number to represent a message. When a warp loads 128 bytes into cache, every thread gets 4-byte data from four codewords. One thread can process sections from four codewords simultaneously without memory latency increasing. The number of codewords for a block would no longer be 32 but 128.

In Algorithm 1, the message qnm and rmn are huge in size and can only be stored in global memory. To ensure that all threads in a warp access contiguous message data, qmn and rmn must be permuted. The permutation method is determined by the mapping strategy. For example, permutation in [7] saves the messages with same index belonging to consecutive nodes of a codeword to contiguous addresses. We pack update data into char4 CUDA built-in vector type. Four 8-bit components in char4 data represent four update data from different codewords.
