From multicore LDPC decoder implementations to FPGA decoder architectures: a case study
Yann Delomier, Bertrand Le Gal, Jeremie Crenne and Christophe Jego IMS Laboratory, University of Bordeaux, Bordeaux INP, France.

Abstract—Last generation high-level synthesis (HLS) tools enables hardware generation of software descriptions speeding up hardware development times. LDPC codes are involved in many digital standards even 5G, generating high quality architecture from software description is needed for Software Defined Radio (SDR) or Cloud-RAN (C-RAN) systems. Previous works in the field focused on reusing many-core software description of LDPC decoding algorithm for hardware generation purpose. In this article we demonstrate that multicore oriented ones once transformed for HLS are more suitable for hardware decoder generation. Indeed, throughputs and hardware complexities observed are closer to handmade LDPC decoders in comparison with related works. Presented results open new perspective for fast integration or prototyping of ECC architectures.
I. INTRODUCTION
Low-Density Parity-Check codes [1] are one of the most popular error correction codes (ECC) because they achieve performances close to the Gaussian channel capacity. Consequently, this code family was adopted in many digital communication standards such as Wifi, WiMAX and future 3GGP 5G. Handmade ASIC or FPGA LDPC decoders [2], [3] were considered as the only way to provide real-time decoding feature. However, the advent of multi/many-core architectures transformed this state of affairs. The processing power provided by such architectures, made possible to reach throughput performance compatible with softwaredefined radio (SdR) or Cloud-RAN applications [4]. This kind of implementation provide the lacking flexibility required by researchers and industrials. To map on CPU or GPU targets efficiently, different parallelization approaches for LDPC decoding algorithms were proposed in the literature [5], [4]. Depending on the selected algorithmic formulation and the applied parallelization model chosen, different throughput and latency trade-offs are possible: throughputs up to Gbps were reported in on a GPU device while latencies of some µs were reported in [5]. Independently of these disparities, these works have demonstrated the viability of software approaches for software simulation and even in the field of real-time LDPC decoding.
However, the high-performances reached on multi/manycore architectures goes hand in hand with high power consumption. Indeed, software architectures have to operate at high processing frequencies to compete with hardware architectures. Consequently, in the field of cloud computing, a number of actors are currently migrating from GPU or CPU solutions to configurable FPGA ones [6], [7]. To response to time-to-market pressure, automatic methodologies for generating ASIC or FPGA hardware architectures based

on parallel software descriptions are investigated [8], [9]. Indeed, the current maturity of architectural synthesis tools allows designer to contemplate this path, which has been awaited for several years [10], [8], [9]. Recently, works were undertaken on the automatic generation of hardware LDPC decoders implementation thanks to high level synthesis tools [11], [12], [13], [14]. For this purpose, software descriptions from earlier works in the field of GPU-based implementation [15], [16] written in CUDA or OpenCL languages were exploited. Automatically generated architectures reached throughputs roughly comparable to handmade architectures and rivalled with software implementations on GPUs [12], [13], [14], [17]. However, the hardware complexity of such architecture was important.
The software description provided in the HLS tools drives and constrains the architectural choices. In this article, a different way of describing the LDPC decoding process is presented. SIMD programming model oriented for multicore architectures has been chosen for this. It enables to better guide the architecture generation than manycore oriented programming language e.g. OpenCL. Two parallelization schemes used with success for multicore implementation of LDPC decoders are thus investigated. The implementation results demonstrate the relevance of the proposed approach.
The remainder of the paper is organized as follows. Section 2 introduces LDPC codes. Then, in section 3 the SIMD parallelization of the LDPC decoding process are presented such as their probable implementation architectures. Finally, section 4 summarizes the experimental results.
II. LDPC DECODING ALGORITHMS
LDPC codes are linear error correcting codes defined by a sparse matrix called H. The H matrix is composed of N columns and M = N −K rows [1]. To transmit K information bits, a codeword of N bit is generated at the emitter side. At the decoder side, the M parity equations are used to correct the N information received from the channel [4].
The most efficient decoding algorithm is the Sum-Product one. However, for implementation purpose more friendly formulations are preferred eg. Min-Sum based algorithms [18]. Different scheduling are available to order the decoding computation. The flooding based scheduling used in the overall previous work offers the highest computation parallelism but has a slow convergence speed. It ease software and hardware implementation of LDPC decoder thanks to a two phase computations in which no data dependency exist. III. LDPC DECODING PROCESS PARALLELIZATION
A. Inter-frame parallelization
A first LDPC decoder implementation is based on a interframe parallelized decoder description that is also applied for x86 and ARM processor implementations [21], [20]. This strategy also applied for GPU-based decoder implementation is the easiest way to parallelize the layered LDPC decoding algorithm. The complete decoding algorithm (Algorithm 1) is processed sequentially from the frame point of view. Consequently, VN data dependencies between CN computations are always respected. This serial execution of the decoding process is quite slow. But to improve the throughput, a set of Q frames can be decoded in parallel. Indeed, the processing regularity between decoded frames, enables to have a single

and low-cost controller and a highly parallel decoding behavior. Consequently, the throughput performance should scale linearly with Q. The management of the H matrix is done thanks to a ROM memory that stores the memory addresses of the VN elements that are involved in CN computations. The serialization on the computation discard the requirement of hardware interleavers such as in intra-frame decoders.
An overview of the hardware architectures that should be generated according to this LDPC decoding description is provided in Figure 1. In figure 1, Q frames are processed in parallel, thus Q memory banks and Q processing units are required. This architecture can be scaled independently of code structure. Moreover, it should provide a constant decoding latency with an interesting trade-off between hardware complexity and the decoding throughput according to Q value.

B. Intra-frame parallelization
Second, an intra-frame parallelization approach was experimented. It is based on x86 software decoder implementations that are detailed in [5]. This SIMD parallelization scheme targets only QC LDPC codes. Indeed, the QC structure ensures that a set of Z independent CN elements exist. This set can be evaluated in parallel without data dependency. However, to handle the H matrix structure, an interleaving step is necessary after the VN data uploading from memories. The opposite operation is also necessary at the end of the CN computation.
A model of architecture that should be obtained according to this LDPC decoding description is provided in Figure 2. It is similar to most handmade hardware architectures [?]. It is composed of Z memory banks and Z processing units. The parallelization factor is constraint by H structure. Thus it provides variable decoding latencies and throughputs depending on the N and Z values.

Note that a single operation is almost performed by line and that function declaration and blank line are also taken into account.
The quantization format applied in the final LDPC decoder description is 8 bits for the VN elements and 6 bits for CN to VN messages. It provides performance closed to those of floating point ones. Note that it is possible to still reduce more the selected quantization format to improve architecture frequency and lowering the hardware complexity.

The first set is focused on intra-frame parallelized decoders and the second one is focused on inter-frame parallelized decoders. LDPC codes from 802.16e and 802.11n standards were selected for benchmarking.

The generated architectures have throughputs in range from 26 to 124 Mbps when 10 layered iterations are executed. The decoded throughput performance mainly depends on the frame length. Indeed, the number of processing units set according to Z value increases from 27 when N = 648 up to 81 when N = 1944. For this reason, the number of clock cycles required to decode a frame seems independent of N value. The hardware complexity of the decoder also mostly scale accordingly to Z except for LUT consumption. For instance, when Z is multiplied by 2, the cost of the data interleaver that is implemented thanks to a barrel shifter increases by a factor higher than 2. The working frequency is the same for all the experiments because the critical path is located in min computation that is independent of N and Z values.
Second, the implementation results for the inter-frame parallelized decoder are provided in Table II. The throughput of the generated architectures increases linearly with the Q factor. Also the hardware complexity of those architectures does not scale accordingly to Q. Indeed, when the number of parallelized frames increases the resources are factored for the computation controller as well as for memories. However, for an equivalent throughput inter-frame architectures are more expensive in terms of memory used than intra-frame ones.
In this section we showed that it was possible to generate a multitude of hardware architectures for decoding LDPC codes from 2 generic software descriptions. The algorithmic choices associated with the parallelization model makes it possible to achieve interesting performance levels with limited hardware complexity. However, it is important to notice that the interframe parallelization strategy that is interesting for software based implementation where it uses the resource available in the fixed processor architecture is much less interesting when resources are allocated according to the real needs.
We will continue to compare the performance of the architectures generated with that offered in the literature, mainly developed by hand or using dedicated automated tools.

A comparison with CPU and GPU targets demonstrate also the interest of the approach. A CPU-based implementation was proposed in [21] on a ARM Cortex-A9 and Cortex-A15. For N = 2304 the the proposed solution reach a throughput equivalent to Cortex-A15 with only 1 core instantiated in the FPGA and is twice as fast as Cortex-A9. As for GPU implementation on GTX Titan as in [19], the proposed FPGA architecture has the same throughput when 2 LDPC decoder cores are instantiated in the FPGA device (at least 10 cores can be instantiated in parallel in a Virtex-7 FPGA). Proposed automatically generated LDPC decoder architectures are thus serious competitive for SdR and C-RAN domain where flexibility and power efficiency are key features.

VI. CONCLUSION
In this article two ways to generate efficient LDPC decoder architectures using HLS tool were proposed. Intraframe parallelized description provides efficient architectures closed to handmade designs in terms of throughput and hardware complexity. Inter-frame parallelized architectures are disappointing except for low-throughput oriented applications. However, these both experimentation demonstrate that HLS methodology are effective for ECC domain. Moreover, it shows that multicore descriptions seems more suited for LDPC decoder design than manycore ones. Finally, performance levels in terms of throughput and latency reached by generated architectures make this solution attractive to replace x86, ARM or GPU based LDPC decoder implementations in SdR or CRAN systems.
