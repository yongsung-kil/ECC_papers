2018 IEEE 10th International Symposium on Turbo Codes & Iterative Information Processing

Model-based Design of Efficient LDPC Decoder Architectures
Yann Delomier, Bertrand Le Gal, Jeremie Crenne, Christophe Jego Bordeaux-INP, CNRS UMR 5218, Universite´ de Bordeaux, France

Abstract—LDPC codes are finding increasing use in digital communication. The fast-paced evolution of new communication standards and ad-hoc applications requires rapid design iteration. Hand-coded RTL architectures generally provides high performance levels but slower the path to IP creation. In this paper we present a hardware LDPC decoder architecture generated with Vivado HLS from a generic SystemC behavioral model. We evaluate the performance of our approach and assess efficiency versus competing approaches. Throughput speedups between 1.1× and 16.7× are shown with an area improvements of 1.6× to 23.8×. A single LDPC decoder reaches up to 150 Mbps on a Xilinx Virtex-7 device for ten layered decoding iterations.
Index Terms—LDPC, model-based design, HLS, FPGA, DSE
I. INTRODUCTION
Low-density parity-check (LDPC) [1] are popular linear error correcting codes (ECC) as they are channel capacityapproaching. To date, most research has focused on multi and many-core architecture implementations of the decoding process. Software efficiency is achieved with the use of highlytuned libraries to provide high computational performance required in software defined radio (SDR) [2]–[4]. However, software implementations exhibit critical energy degradation over existing approaches.
The use of hand-coded LDPC decoder designs to achieve energy efficient and real-time decoding performances for both application-specific integrated circuit (ASIC) and fieldprogrammable gate array (FPGA) has also been an on-going effort for decades [5], [6]. As part of this effort, a specific methodology to automatically generate LDPC decoder architectures has emerged in [7] and an application specific instruction set processor (ASIP) was described in [8]. Although useful for reducing development times, these architectures are generated from fixed templated descriptions and tend to limit the theoretical achievable performances.
FPGA provide effective targets for LDPC decoders implementations as evidenced by recent migration from central processing unit (CPU) and graphic processing unit (GPU) [9]–[11]. To benefit from intrinsic parallelism and pipeline structure of FPGA devices, specialized knowledge is required in many cases. Integrating processing elements, memory interfaces and interconnects together is challenging. Optimum performance cannot be reached without tailoring specifically to the targeted architecture [12]–[14]. Hardware development is facilitated by the use of model-based design and high-level synthesis (HLS) [15], [16] to incorporate hardware accelerated LDPC decoders into end-user applications [13], [14], [17], [18]. II. LDPC CODES
LDPC codes [1] are defined by a sparse matrix H. The H matrix is composed of N columns and M = N − K rows with K information bits. M parity equations (check nodes CN) are used to correct N received informations (variable nodes VN) [2]. To reduce computational complexity most implementations are based on min-sum algorithms [19]. The LDPC decoding algorithm was originally based on a flooding schedule where all messages from VN to CN (Lnm) are evaluated before messages from CN to VN (Lmn) are updated. The process iterates Imax times to decode the entire frame. This scheduling exhibits a massive computation parallelism and makes the flooding formulation attractive for both pure

III. DECODING ALGORITHM PARALLELIZATION
Model-based HLS tools make it easier for designers to generate hardware architectures from a high-level behavioral model. A number of approaches exist to adapt LDPC decoders descriptions from many-core software implementations using Open Computing Language (OpenCL) and Compute Unified Device Architecture (CUDA) parallel computing languages. High decoding throughput performances are achieved but it still cannot compete with RTL hand-coded decoders [13].
To improve the description of the layered based decoding process we use a programming model based on SIMD parallelization. The model-based generation is then guided to fit placement constraints and to map efficiently to device hardware resources. The LDPC decoding algorithm is similar to [4] for real-time decoder implementations on multi-core devices but without any processor architectural constraints.

A. Parallelization techniques
Two approaches to parallelize the LDPC decoding process at model-based levels are known. The inter-frame [3] and the intra-frame [4] parallelization. Inter-frame parallelization consists of decoding Q independent frames in parallel. This strategy has two benefits: 1) it increases the throughput linearly with Q and 2) it reduces the hardware complexity with control element merging. However, it increases the memory footprint by Q. The intra-frame parallelization resolves this issue by taking advantage of possible computation parallelism in a frame. This strategy applied in hand-coded RTL decoder architectures minimizes the latency and hardware complexities. The Z parameter in QC-LDPC codes specifies the minimal CN computations in parallel and defines the parallelization factor.

IV. IMPLEMENTATION OF THE DECODING ALGORITHM
A. Behavioural description for HLS
We use Xilinx Vivado HLS tool [16] to generate RTL architectures from our behavioral model. Vivado takes C, C++ languages and SystemC as design entry points. For this experiment, we use the x86 source codes from [4]. To generate synthetizable and high quality RTL architectures from this implementation important code transformations are required:
1) The dynamic memory allocation of the x86 source codes are replaced by static memory allocations. To precisely define LDPC decoder inputs and outputs (IOs) behaviors we use SystemC. The LDPC decoder source code is encapsulated in a single SystemC process sc cthread and IOs are declared as sc fifo in and sc fifo out ports.
2) A custom and generic SIMD library for LDPC decoding is developed. The x86 LDPC decoder implementation uses Intel Advanced Vector Extensions 2 (AVX2) instruction set and are not available for model-based synthesis on ASIC or FPGA.

The software description is constrained by the internal bitwidth of the x86 SIMD datapath. A set of Z CN computations can require many rounds (Z ≥ 32) and can badly map with the SIMD units (Z ≤ 32) depending on QC H matrix. The sc bigint<> SystemC data type enables to describe bit accurate models even for large bit sets. A lowlayer scalar function library is needed. The scalar functions implement basic computations required by the LDPC decoding task (saturated addition, minimum, XOR, sign extraction). The scalar functions layer is bitwidth parametrizabled to support variable node or message quantization format modifications. This bitwidth flexibility is achieved using sc int<T> and sc uint<T> data types. T fixes the manipulated data width and is a function template parameter. Data specification and width is defined and can be modified at model level. On top of this library, a vector library is built. A set of Zways SIMD functions to process data vectors of Z elements is implemented. The number of scalar elements in a vector is flexible where AVX2 instruction set is not and is set to 32. It allows to update the computation parallelism at synthesis according to the H matrix specific properties. The complete LDPC decoder model is built on the top of the vector library. It implements the layered min-sum decoding algorithm (Algorithm 1). The flexibility of scalar and vector layers provides highly configurable levels to our behavioral description with both architectural (VN width, message width, internal saturation levels) and application parameters (N , K, M , Z). To optimize the HLS process, pipeline directives (pragmas) are added to the LDPC decoder model as shown in Algorithm 1. We also apply unroll and inline pragmas in scalar and vector libraries implementations.
The description of the LDPC decoding algorithm is fully generic and only one source code is necessary. A header parameters file sets N , M , Z factor, H matrix features and the quantization format. Any QC LDPC decoder architecture can be generated with the appropriate parameters file. The quantization format applied is 8 bits by default for VN elements and 6 bits for messages. Performances are comparable to floating point LDPC decoding algorithms. A performance degradation of only 0.05 dB in comparison with floating point software simulations are observed. Compared to RTL handcoded architectures the behavioral description of the LDPC decoder and libraries are shorter (respectively 200 and 300 lines of code)1.

With δki being the latency associated to Kernel i. The PU architecture is pipelined as specified by pragmas in Kernels 3.1 and 3.2. Their processing latencies are equal to δk3.i = degcn + 3.i where degcn is the degree of the executed CN and 3.i is the latency penalties introduced by pipeline flushes between two consecutive CN computations. These 3.i penalties ranged from 1 to 12 and are related to targeted clock frequencies and FPGA parts.
V. EXPERIMENTATIONS
A. Absolute performances
To understand and evaluate the flexibility and performance benefits of our approach over existing solutions, we used the LDPC codes from 802.16e, 802.11n and 3rd generation partnership project (3GPP) 5G standards. RTL architectures generations from our behavioral model were experimented with Vivado HLS 2017.1. We targeted a Virtex-7 (xc7vx485tffg17612) FPGA part and two clock frequencies were applied: 100 megahertz (MHz) and 400 MHz to obtain different architecture trade-offs. Throughputs Γ between 31 to 147 megabit per second (Mbps) are reported in Table II. These performances are predominantly influenced by Z (dc value has an minimal effect). Pipeline penalties involved at the end of each CN computation are smaller as dc values are high (high code rates). Throughput improvements are shown when increasing clock frequency from 100 MHz to 400 MHz. However, it does not scale with operating frequency. The CN processing unit is splited with additional pipeline stages and increases flushes penalties. For example, 3,550 cycles and 5,470 cycles are respectively required for 100 MHz and 400 MHz architectures when N=1024.
Generated decoders occupy 0.8% to 4% of the total Virtex7 FPGA available logic resources. Excepted for look-up table (LUT) assignation, resources usage also increases with Z. When Z is multiplied by 2, the data interleaver implemented with a barrel shifter increases by a higher factor.
B. Relative performances
To provide a fair comparison we evaluated a broad range of solutions. This is challenging as previous works use custom quantization formats and specific optimizations and implementations are not always reproducible or publically available. When compared to similar works, results reported in Table III show the attractiveness of the proposed model for LDPC decoders generation. Hardware complexity and throughput are improved for every implementations studied in this paper. This work exhibits significant area improvements (2× to 20×) over results reported in [13], [18] and throughput benefits (up to 20×) compared to described decoders in [14].
The comparison analysis is more touchy for hand-coded RTL architectures [23]–[26]. This work achieves similar performances but uses approximately 2× more resources than [23]. Instantiating 35 decoders reaches a throughput of 3,360 Mbps and increases hardware complexity to 120,610 LUTs. The proposed approach achieved better performances both in terms of hardware complexity and throughput in comparison to [26]. Our proposal is either close or comparable to handcoded decoder architectures.
This work also analyses CPU and GPU implementations. Both lower and higher throughputs for a single LDPC generated decoder are presented when respectively compared to [4], [21] and [20], [27]. 20 LDPC decoders can be instantiated on a Virtex-7 FPGA. This system is able to sustain a 2.5 gibabit per second (Gbps) peak throughput compared to [4] and more than 5 Gbps throughput compared to [21]. It demonstrates the competitive nature of model-based FPGA decoder implementations versus software based decoders.
VI. CONCLUSION
A generic behavioral model to generate efficient layered hardware LDPC decoder architectures has been presented in

this paper. It takes advantage of intra-frame parallelization to
demonstrates the relevance of SIMD-enabled descriptions for
HLS implementations. This work has been developed and fully
integrated into the Xilinx Vivado HLS tool suite. It is open
source and available at [28]. Generated architectures provide
interesting throughput versus area trade-offs. Performances
achieved with a single LDPC decoder on a Virtex-7 FPGA
are competitive with CPU and GPU implementations. The
solution is suitable for implementations of ECC features in
software-defined radio (SDR) systems or cloud radio access
networks (C-RAN). For future work, we aim at increasing
throughputs by improving the model description and providing
architectural flexibilities at run-time.
