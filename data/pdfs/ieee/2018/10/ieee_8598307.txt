An Efficient Implementation of LDPC Decoders on ARM Processors
Bing Liu, Rongke Liu, Zhanxian Liu and Ling Zhao School of Electrical and Information Engineering BeiHang University, Beijing, China
Email: {1302liubing, rongke liu, liuzhanxian, zhaoling}@buaa.edu.cn

Abstract—In this paper, we present an ARM based decoder for Low Density Parity Check (LDPC) codes. To maximize the efficiency of the parallel execution and fully utilize the ARM processors’ capacity, instruction-level parallelism optimizations are applied. Compression storage and efficient data prefetching optimizing methods are used to enhance the memory cache efficiency. The experiments are carried out on NVIDIA Jetson K1 platform. Our decoder can achieve 1.85 times speed up compared with the related work. And the decoder has better energy efficiency comparing to the decoders on x86-CPU and GPU.
I. INTRODUCTION
Forward error correction (FEC) technology is widely used in digital communication systems. Among FEC technologies, LDPC codes have excellent performance and are widely used in various wireless standards. Decoders with high throughput and low latency can be implemented on the Field Programmable Gate Array (FPGA) and Application Specific Integrated Circuit (ASIC) platforms. The difficulty of transplanting and modifying limits them to satisfying the flexible requirements of cognitive radio (CR) and software defined radio (SDR). In recent years, researchers focused on multi-core devices (e.g.GPU and CPU) [2]–[5], [9], [10] to seek for better flexibility. Eventually, those decoders achieved excellent throughput, which can meet the wireless standard requirements. Nevertheless, their high power consumption can’t satisfy the requirement of embedded systems in low power environment. Thus, a more efficient software decoder architecture based on the ARM platform is proposed to solve this issue [1]. However, after carefully analyzing the result, the ARM processors’ processing capacity is not fully released. Several optimization techniques can be applied to maximized the throughput of the decoder.
This letter is organized as follows: Section II briefly introduces decoding algorithms of LDPC codes. Then we analyze the theoretical performance of the decoder. And three optimizing methods on the ARM processor platform to fully release ARM processors’ computing capacity are described in Section III. Finally, the performance analysis and comparison with other decoders on ARM, x86-CPU and GPU are displayed in Section IV.
II. DECODING ALGORITHM
Among soft decision algorithms, Belief Propagation (BP) algorithm can achieve excellent error code performance. How-

ever, it is too difficult for ARM processors to compute large number of complex operations directly in the check node update processing. Min Sum (MS) algorithm removes these complex functions and only uses comparing and adding operations in the check node update processing, which extremely reduces the computing complexity. In order to compensate the performance loss from the approximation, Normalized Min Sum (NMS) and Offset Min Sum (OMS) was proposed. From the perspective of decoding schedule, usually, most works were based on the Two Phase Message Passing (TPMP) structure. In fact, an increasing number of works started to use Turbo Decoding Message Passing (TDMP) schedule, which is also called horizontal layered-based decoding schedule. This layered schedule reduces the computational complexity and provides some memory footprint saving.
In this work, layered decoding algorithm is associated to the OMS algorithm, which improves the correction performances compared with the MS algorithm and has lower complexity compared with NMS algorithm [2]. Data is represented on an 8-bit format. The decoding architecture with horizontal layered-based decoding schedule refers to the architecture in [1]. The fixed-point format management is based on results given in [7], which is also as same as the related work [1].
III. ANALYSIS AND OPTIMIZATION
ARM Cortex-A series of multi-core processors support SIMD (Single Instruction, Multiple Data) instruction set called NEON and SIMT (Single Instruction, Multiple Thread) to achieve multiple threads computing. To completely take advantage of NEON properties, an efficient approach consists in processing 16 different frames simultaneously while using 128-bit width registers. To maximize access bandwidth, VN values from different frames have to be rearranged contiguously in memory before starting decoding. This process can be described as a matrix transposition [1]. Similarly, there is an another inverse operation of matrix transposition at the end of decoding. For multi-threading programming, we apply OpenMP directives considering the convenience of creating threads and barriers.

B. Instruction level parallelism optimization
Except Cortex-A5, all of ARM Cortex-A series are superscalar processors, which can issue 2 or 3 instructions in one clock cycle. As for those processors designed for high performance like A15, A57 and A72, their micro-architectures are as similar as the demonstration shown in Fig 1. For example, the Cortex-A15 processor is a superscalar processor with out-of-order execution, which can decode 3 instructions simultaneously and issue them to the 8 execution clusters in one clock cycle. The number of blocks in one execution clusters indicates the number of pipeline stages.
After fully optimizing the decoder, codes are the simplest without any redundant processing. However, most NEON instructions are depended on previous NEON instructions. The strong flow and data dependency reduces the efficiency of the decoder in two aspects.
Firstly, though the A15 processor has the potential ability to rearrange instructions order using out-of-order execution and then issue 2 NEON instructions into 2 independent NEON pipelines in 1 cycle, such huge flow dependency may stall one NEON pipeline and make it idle.

Secondly, when an instruction is issued into a NEON pipeline, it has to go through longer execution pipeline and takes longer clock cycles until be available to some particular related instructions by bypassing function compared to ARM instructions. Then it takes another several cycles for results to be written back to registers file. At this time, the result can be accessed by any other instruction. Take the vector add instruction called VADD in NEON as an example. According to the official manual [8], the VADD in A9 costs 3 execution cycles when the result of the operation is ready. At this time, the result might be ready as source operands by bypassing function using forwarding paths for other instructions. Then, the VADD takes 6 execution cycles to write back the result to the register file. Thus, a dependent instruction after the VADD may be blocked 4 cycles or more than 10 cycles. Normally, maximizing,comparing and shifting operations usually take 4 or 5 cycles to go through the execution pipeline. For the simple multiplying operation, the delay is up to 7 cycles. If instructions are strongly dependent on each others, more bubbles will be created in pipeline and the efficiency of the decoder will be reduced.
To break the dependency among instructions without bringing any redundant processing, loop unrolling is the best choice. Loop unrolling can reduce the possibility of the pipeline being interrupted by branches and increase the number of uncorrelated instructions. From this aspect, we could rearrange the uncorrelated instructions order to help compiler better optimize codes.
Considering from the view of processors, more independent instructions help the processor issue them continually into 2 NEON pipelines. At the same time, because of the out-oforder execution, more instructions without any dependency can be emitted and filled into execution pipelines instead of waiting for the previous relevant instruction to be executed. This could promote the instructions per cycle (IPC) of the decoder and reduce execution time. Thus, 4 times or more loop unrolling may have better performance. However, loop unrolling is not limitless, excessive loop unrolling increases register pressure and cause more memory access, which could bring more latency to the program. Excessive loop unrolling also increases code size, and causes an increment of instruction cache misses. The result of loop unrolling will be presented in Section IV.

C. Compression storage optimization
Tmemory is related to the total memory footprint. The total memory footprint can be expressed as the equation 6.

Fcore = ((n − m) × c + n) × q + Fp

(6)

Because of the structure of the LDPC matrix, the memory access of obtaining variable nodes is discontinuous in memory. We can precompute all the positions to be accessed and store in Fp [1], [2]. For those codes whose check node s number is (m-n) and the row weight is c, (m-n)×c×4 byte memory space is needed to save all the positions. For QuasiCyclic LDPC (QC-LDPC) codes, their parity check matrixes have stepped structure, which means that memory access is continuous within a certain range. This features could be used to only record M head nodes and the length of each stepped structure. Thus, this strategy only costs M×c×(4+1) byte. Usually, M is much smaller than (m-n). For unstructured codes, the strategy is degraded to normal strategy.
For the code with length of 2304 the number of head nodes is calculated as 63. Using the above compression strategy, the Fp is reduced to 1.95KB, which Fp is used to be 28.5KB with normal strategy. And for the code whose length is 7168 containing 4096-bit message information, this method can reduce the Fp to 1.8KB, which used to be 92KB. The method can greatly reduce the memory footprint of Fp. Less memory footprint means less memory access and less access latency. Especially, for long codes whose memory footprint are large or close to cache size, this strategy can ease the cache pressure, reduce the possibility of swapping useful variables out and improve cache hit ratio when using 4 cores to decode.

D. Efficient data prefetching
Since date is transferred between memory and cache in blocks of fixed size, called cache line, there is always a cache miss taking place when accessing the data in a cache line at the first time. For the continuous memory access of CNs to VNs, we can prefetch next 2 cache lines manually to hide the latency of cache miss. However, for the noncontinuous memory access to VNs, manual data prefetching seems ineffective in the test. Similarly, lots of manual data prefetching is a burden for cache. This will decrease the throughput of decoder significantly.

The SoCs is composed of 4 processor cores with 2 MB of L2 cache memory using RAND strategy shared by its 4 cores. The decoder is described in C++ language and then compiled by GCC 4.8. The air throughput was evaluated on board. To obtain the average performance, the decoding time is averaged over 2 minutes.

A. Performance evaluation on ARM processors
To compare with related works, codes from 802.11a, 802.11e, CMMB and DVB standards with code lengths from 576 to 16200 are selected. These codes include QC-LDPC codes and unstructured LDPC codes. To have a better view of the promotion from manual loop unrolling, considering most codes’ row weight in the experiment are 6 or 7, 6 times or more loop unrolling is meaningless and reduces portability. So 5 times and following conditions are measured and presented in Table I.
Its clear to find out that 4 times loop unrolling has the best throughput, because both pipeline overlap and register pressure are taken into account. However, compiling with −funrollloops dosen’t get a significant improvement comparing with no loop unrolling. The reason is that GCC compiler will choose which loop will be unrolled entirely. Those loop with complex loop control or uncertain loop condition may not be unrolled or unrolled with radical jump instructions when there are branch in the loop, especially. By disassembling the codes of related work, we find that there are too many jump instructions through the codes. Branch prediction failure will significantly bother execution pipeline and be a burden on program speed. And temporal locality, spatial locality and order locality of the instruction cache is broken by the branch. Instructions may be loaded to the L2 cache, which means that potentially useful data may be swapped out from the cache, and may increase cache misses. On the other hand, there are fewer jump instructions using manual loop unrolling. Thus, manual loop unrolling may be the best choice to speed up the program.

V. CONCLUSION
In this article, a more efficient software LDPC decoder based on ARM processors is presented. By using the ILP optimization, this approach could make full use of computing resource. And compression storage and efficient data prefetching are used to make sure that the decoder could maximize the use of storage bandwidth. Experimental results demonstrate that the proposed decoder is more efficient in throughput and power consumption.
