A Unified Reconfigurable Datapath for 5G
Compatible LDPC Decoding
Ting Lin, Shan Cao, Shunqing Zhang, Shugong Xu
Shanghai Institute for Advanced Communication and Data Science
Key laboratory of Specialty Fiber Optics and Optical Access Networks
Joint International Research Laboratory of Specialty Fiber Optics and Advanced Communication
Shanghai University, Shanghai, China
{zjzslt, cshan, shunqing, shugong}@shu.edu.cn
Abstract—With low-density parity-check (LDPC) codes se-
lected as the standard codes for the fifth generation (5G)
enhanced Mobile Broad Band scenario (eMBB) and the raising
requirement for 5G network deployment, a LDPC decoder that
supports all LDPC coding structures according to Release 15
(R15) 5G standard becomes necessary. In this paper, we propose
a LDPC decoding unit with unified reconfigurable datapath to
adapt LDPC code structures of 5G data channel. The proposed
decoding datapath can be configured into several separate ones
for unified check node update and the corresponding variable
node updates. The LDPC decoding unit is implemented on Altera
Stratix IV FPGA with 333.33 MHz frequency and around 26
Mbps throughput per unit.
Index Terms—LDPC code, reconfigurable architectures, belief
propagation
I. INTRODUCTION
Low-density parity-check (LDPC) [1] codes are linear block
error-correction codes, and have been proven to offer reliable
performance and high parallelism in hardware. To meet the
high throughput demands for the fifth generation (5G) wireless
systems [2], a solution to apply LDPC code for data plane
transmissions in 5G enhanced Mobile Broad Band scenario
(eMBB) scenario was agreed in the Third Generation Partner-
ship Project (3GPP) radio access network meeting [3]. And
novel structural features are introduced in code design to meet
the broad requirements of 5G New Radio (NR) data channel
[4].
LDPC codes can commonly be decoded with belief propa-
gation (BP) algorithms, information is updated and exchanged
iteratively between the variable nodes and check nodes. Wire-
less standards, including 3GPP Release 15 (R15) standard
[5], support more than one kind of quasi-cyclic LDPC codes
with different characteristics. Thus the previous hardware
implementations of LDPC BP decoding algorithms are often
composed of check nodes processing units (CNUs) and vari-
able node processing units (VNUs), which are consume signif-
icant hardware resources and require complicated connecting
network to route information between processing units through
shift operations [6]. Compared to the wireless standards that
LDPC codes have already been adopted into, such as IEEE
802.11n [7], IEEE 802.16e [8], digital video broadcast (DVB)
[9], 5G NR proposed new requirements to LDPC decoding,
such as larger codeword length, wide range of block sizes
and code rates. In this case, the previous LDPC decoding
architecture requires even higher complexity and latency in
the interconnection network to connecting CNU and VNU
efficiently.
Therefore in this paper, we proposed a unified LDPC
decoding unit with reconfigurable datapath for LDPC de-
coding, which merged CNU, VNU and the interconnection
network. The proposed LDPC decoding unit, compatible with
different code lengths and coding structures according to the
5G standard decoding requirements, can process messages of
one or two layers simultaneously. We implement the proposed
decoding unit with Field-Programmable-Gate-Array (FPGA)
by a pipelined hardware architecture. The implementation
results show that the proposed decoding unit can support 26.65
Mbps throughput, less than 36 µs delay and 21.80 Mbps
throughput, less than 24 µs delay for two kinds of base LDPC
coding structures.
The rest of this paper is organized as follows. Section II pro-
vides necessary background information about LDPC codes.
The proposed unified reconfigurable decoding architecture is
discussed in Section III. In Section IV, we provide the simulate
and implementation results. Finally, we conclude this paper in
Section V.
II. PRELIMINARIES
The decoding algorithm basics for LDPC codes are intro-
duced in this section.

matrix. Consider a (N, K) LDPC code, which generates N

b = [b1, b2, . . . , bK] according to [1]. A (N, K) LDPC code
has a (N−K)×N parity check matrix H which determines the
connectivity of check nodes and variable nodes. The process
of BP decoding for LDPC codes can be shown based on a
graphical description known as a Tanner graph [10], which
mirrors the parity check matrix as illustrated in Fig. 1. The
corresponding messages are exchanged between two kinds of
nodes along the edges and iteratively updated through the
following equations until the stopping criterion is satisfied.

III. RECONFIGURABLE DECODER
In this section, we propose a reconfigurable decoder for
LDPC codes. The architecture of the proposed reconfigurable
architecture is shown in Fig. 2, composed of several merged
reconfigurable decoding units (MRDUs), a control unit and a
memory controller.
Accordingly, 3GPP R15 [5] has agreed to consider two
kinds of rate-compatible base graphs, base graph 1 (BG1) and
base graph 2 (BG2), for the data channel coding and BG1 and
BG2 have similar structure. BG1 is for larger block lengths
and higher rates, while BG2 for smaller block lengths and
lower rates. The H matrix is composed of 46×68 and 42×52
elements for BG1 and BG2 respectively according to [5]. Each
element in the H matrix represents a Z × Z submatrix, and
the sets of LDPC lifting size Z is same for BG1 and BG2.

The layered decoding requires sequential schedule due to
data dependencies. According to the principle of layered
decoding, check nodes with no shared variable nodes can be
updated simultaneously. Therefore, we reorder the check nodes
and regroup them so that several check nodes can be processed
in parallel according to the following principles.
• Check nodes cannot be updated simultaneously with
variable nodes that have connections with check nodes
in the same block, to avoid memory access conflicts.
• The difference in the number of cyclic shifts between two
adjacent running layers differs by no more than Z −7.
Since one update process of the LDPC decoding requires
5 clocks for calculation and 2 clocks for reading and
storing messages. And the L(t)
vi cannot advance to the
next update until the last update is completed. Therefore,
it is necessary to ensure that at least 7 clocks are taken
before the same L(t)
vi message enters to the next update.
• The number of type conversions of LDPC decoding is
as small as possible. Therefore, the iterative update of
the LDPC decoding in the same combination mode is
updated in the adjacent order.

IV. SIMULATION AND IMPLEMENTATION
The proposed architecture is implemented and evaluated by
Altera Stratix IV FPGA [13]. In this section, we simulate and
implement the proposed reconfigurable decoding scheme and
simulate the floating-point and fixed-point bit error rate (BER)
versus signal-to-noise ratio (SNR) performance using different
data widths and data precision. In what follows, we provide the
hardware implement results and performance of the proposed
design.
A. Effect of Data Width and Data Precision
In the FPGA implementation, data width and data precision
are often used to trade off the decoding performance against
the hardware complexity. The hardware resource increases in
proportional with the data width. As a baseline performance,
we assume the 16-bit floating-point parameters are applied
and the overall BER versus SNR curve is evaluated as the
performance metric. The decoding performance for different
data widths and data precisions are illustrated in Fig. 4.
Based on the simulated results, we show that 6-bit data
width provides sufficient solution for LDPC codes in terms of
BER versus SNR performance, and requires relatively smaller
amount of hardware resources in the meantime.

B. Hardware Implementation Results
In Table II, the FPGA implementation results with 6-
bit fixed-point LLRs are summarized, where the utilization
of different types of hardware resources is considered. We
can see that the MDRU takes a small amount of hardware
resources compared to the controlling units. The decoder can
be therefore easily expanded by integrating more MRDUs in
a single-instruction-multiple-data (SIMD) manner.
Several examples are given by configuring the MRDU for
two kinds of base graphs and different codeword lengths,
the results of which are given in Table III and Table IV
respectively. The results show that the latency to decode one
codeword increases almost proportionally with the length of
the codeword. The throughput however remains almost the
same despite the increase of codeword lengths due to the full
pipelined architecture. Therefore, the proposed architecture
can be used to a wide range of codeword lengths.
V. CONCLUSION AND FUTURE WORK
In this paper, a reconfigurable datapath for LDPC decod-
ing unit is proposed, and a merged LDPC decoding unit
compatible with 3GPP R15 standard with pipelined structure
is implemented accordingly. Based on the proposed LDPC
decoding unit, a reconfigurable LDPC decoder is implemented
to support all code structures of LDPC codes in 3GPP R15
standard [5]. To further improve the throughput of the decoder,
a clustered architecture composed of more MRDUs and its
implementation by Application Specific Integrated Circuit
(ASIC) is one of our future work.
