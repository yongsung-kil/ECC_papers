DVB-S2

2310 E. El Segundo Blvd. El Segundo, CA 90245 david.s.kun@aero.org

Abstract—Previous studies that used Graphics Processing Units (GPUs) to decode Low Density Parity Check (LDPC) codes for DVB-S2 employed inter-codeword parallelism and/or Turbo Decoding Message Passing (TDMP) to achieve high throughput. By converting the LDPC parity check matrix into a quasi-cyclic structure, we show that LDPC encoding can be efficiently implemented on a GPU, and a different approach to LDPC decoding with intra-codeword parallelism and early termination that can achieve approximately 100 Mbps increase in throughput per 0.1 dB increase in signal-to-noise ratio and, for some cases, achieve 1 Gbps or greater overall throughput.♦

1. INTRODUCTION
Decoding Low Density Parity Check (LDPC) codes using Graphics Processing Units (GPU) has been an active research topic in which high throughput is attained by decoding multiple codewords simultaneously for a fixed number of iterations [1], [2]. The primary disadvantage of these approaches is decoding continues regardless if a valid codeword has been reached. This reduces throughput with the GPU by performing additional unnecessary calculations. On the contrary, decoding LDPC in hardware, such as Field Programmable Gate Arrays (FPGA), employ intra-codeword parallelism and/or early termination to increase decoder throughput [3].
This paper adopts the decoding approach that is similar to hardware and takes advantage of the GPU architecture to efficiently encode and decode all the LDPC codes described in the DVB-S2 specification. Simulation results using NVIDIA Titan Xp reveal that encoding LDPC can achieve a throughput of multi-Gigabits per second and decoding can

reach Gigabits per second throughput for low bit error rates of less than 1e-9 with early termination.

The rest of the paper is organized as follows. Section 2 discusses how to restructure the LDPC parity check matrix for intra-codeword parallelism encoding and decoding. Sections 3 and 4 covers the GPU kernel designs of the LDPC encoder and decoder, respectively. Finally, section 5 presents simulation results of these GPU kernels using the NVIDIA Titan Xp graphics card.

When mapping this algorithm to a GPU kernel, the following key design choices were implemented to attain high GPU occupancy.

1. Thread and grid block dimensions are , 1,1 and @ A, 1,1 , respectively.
2. For each thread block, allocate shared memory as
(maximum row degree of )*2 bytes. Shared memory stores λmn mp as half precision (16 bits); however, all computations for min-sum algorithm are performed as single precision (32 bits). The half precision intrinsics, __float2half_rn() and __half2float(), are used. 3. In step 4 of the decoding algorithm, Lm is double-counted for the superimposed circulant, so it needs to be retrieved from global memory to subtract out its contribution. 4. By storing the degree of each circulant row in each 32-bit entry in constant memory, different row degrees in the 16,200 code sizes of DVB-S2 are readily supported. 5. When computing the two minimum magnitudes for the min-sum algorithm, branchless version of the ‘if’ statement was implemented to eliminate thread divergence within a warp. 6. For early termination, the early termination flag and checksum are declared as volatile __shared__. This forces the compiler to not move shared memory to registers and allows these values to be used a communication vehicle among threads in a block.

Except for the code size of 16,200 and rate 9/10, the limiting factor to achieving maximum usage of the Titan Xp is the number of registers per thread block. For NVIDIA GPUs, the number of threads per block (TPB) is a multiple of 32, so TPB is rounded up to 384 for 360. Thus, each thread block uses 12,288 out of the possible 65,536 registers. So only five thread blocks can run simultaneously, which results in 1920 active threads out of the possible 2048. Thus, we achieve a high occupancy of 93.75% using the NVIDIA visual profile and CUDA occupancy calculator.
5. RESULTS
Simulations were ran on the NVIDIA Titan Xp with a compute capability of 6.1. The source code was C++ and was compiled with the NVIDIA CUDA compiler (nvcc) version 8.0. The GPUs kernels were grouped by functionality and

separated into different CUDA C files with a .cu extension and corresponding header files with .cuh extension. Since the GPU kernels for encoding, kernel_computeGamma(), and decoding, kernel_LDPCDVBS2DecoderStagMsg(), refer to the same variable in constant memory, they needed to be in the same .cu file. During kernel compilation, the – dc flag is required for separate .cu file compilation and during linking, the –dlink option must be present to combine all the device objects into one object.

6. CONCLUSIONS

In this paper, we have shown that converting the LDPC parity check matrix for DVB-S2 into a quasi-cyclic structure allows for intra-codeword parallelism that achieves throughputs for (a) an encoder greater than 1 Gigabit/sec and (b) an early terminating decoder from 500 Mbps for rates < 1/2 to 1 Gigabit/second for code rates > 1/2 at bit error rates < 1e-9. Furthermore, with each 0.1 dB increase in •Z⁄@*, the decoder yields approximately 100 Mbps increase in throughput.

and decoding using low-cost GPUs versus existing intercodeword approaches. As newer GPU architectures are released by NVIDIA such as Volta, it is expected that the throughputs will increase with the benefit of not having to rewrite CUDA C++ code.
Areas of future work are to design an efficient GPU implementation of (a) a soft-input hard-output BCH decoder and (b) a fast BCH encoder that does not use the conventional shift-register approach. These are required to eliminate the inflection point in the error rate performance as •Z/@* increases. Coupled with an early terminating LDPC decoder, the overall throughput of decoding BCH and LDPC should not decrease significantly at low BERs.
