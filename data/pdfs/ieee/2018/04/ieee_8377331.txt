ADMM Hardware Decoder for Regular LDPC Codes Using a NISC-Based Architecture
Imen DEBBABI1, Bertrand LE GAL2, Nadia KHOUJA1, Fethi TLILI1 and Christophe JEGO2 1. GRESCOM Laboratory, High School of Communications, Carthage University, Tunisia.
2. Bordeaux INP, Enseirb-Matmeca, CNRS IMS, UMR 5218, University of Bordeaux, Talence, France.

Abstract—The Alternate Direction Method of Multipliers (ADMM) approach is an original method for LDPC decoding based on the linear programming (LP) technique. It introduces a novelty at the error correction performance level. Nevertheless, this method can be toughly implemented due to its high computational complexity. In this paper, an implementation of the ADMM LP decoding algorithm on an FPGA target is presented. Its hardware resource cost is evaluated and compared with the state of the art LDPC decoders using the belief propagation (BP) decoding approach. The preliminary logic synthesis results show that an LP based hardware decoder for LDPC codes should be viable for applications with tough error correction requirements. However, additional research works are required to reach equivalent hardware complexity and throughput performances that are similar to traditional BP based LDPC decoders.
I. INTRODUCTION
Linear programming (LP) technique was proposed by Feldman in [1] as an alternative to belief propagation (BP) decoding algorithm for LDPC codes. It improves the frame error rate (FER) performance in comparison with the usual BP decoding based on the sum-product (SPA) algorithm and its simplified formulations: Min-Sum (MS), Normalized MS (NMS), etc. [2]. With the recent optimizations of the LP decoding thanks to the Alternate Direction Method of Multipliers (ADMM) technique [3], LP LDPC decoding has become viable for practical implementations.
In [4], a first analysis of the ADMM decoding algorithm complexity compared with the commonly-used SPA and MS decoding techniques is presented. It showed that the ADMM approach is likely to be implemented in hardware unlike the previously proposed LP decoding techniques based on solvers. Although the algorithm complexity seems comparable to BP SPA decoding approaches, its implementation remains a challenging task due to the complicated check node computation which needs an intricate Euclidean projection operation. A recent work given in [5] provides a reduced complexity Euclidean projection algorithm. As such, the hardware implementation of the ADMM approach has become feasible. The work in [6] describes the first FPGA implementation of an ADMM decoder for quasi-cyclic LDPC codes using the Euclidean projection in [5] and a partially parallel architecture.
This paper aims to describe the hardware design of the ADMM l2 decoder based on the horizontal layered scheduling [7] which helps improving the decoding convergence speed as for the BP algorithms [8]. It builds upon the LDPC

decoder described in [9] which was initially designed for the MS decoding algorithm using a non-symmetric NISC-based architecture (No Instruction Set Computing) [10]. First, we describe the partially parallel architecture. Then, we focus on the design of the check node processor unit (CNU) for the ADMM l2 decoder for regular codes. We describe the different quantization formats and the storage elements organization. Furthermore, the hardware implementation results on an FPGA device and the throughput performance are further discussed while a comparison with equivalent NMS decoders on FPGAs is detailed. Conclusions and future works are finally provided.

II. LINEAR PROGRAMMING DECODING OF LDPC CODES
An LDPC code C of block length N is described by a parity check matrix H. It is composed of M check equations with (M ≤ N ). The code rate of C is defined by r = (N − M )/N . The matrix H can also be represented by its Tanner graph where the variable nodes (VN) are indexed by I = {1, ..., N } and the check nodes (CN) are indexed by J = {1, ..., M }. Each one in H at the position (i,j) corresponds to an edge in the Tanner graph connecting VNi to CNj. Let dvi and dcj be the VNi and CNj respective degrees, being regular or irregular depending on H. To facilitate the algorithmic formulation of the decoding process, let Nv(i) be the index of the set of neighbors of vi and Nc(j) be the index of the set of neighbors of cj.
The check polytope Pdcj is the convex hull of all binary vectors of length dcj with an even number of ones. III. ADMM l2 DECODING ALGORITHM
The first ADMMl2 algorithmic formulation [12] is based on a flooding scheduling computation. To reduce the computational complexity of the ADMM l2 decoding algorithm that is essential for an implementation, a layered scheduling has been proposed in [7]. Similar to BP-based decoding algorithm, it approximatively halves the computation complexity of the decoding process while reducing the memory requirement by 30%. This horizontal layered formulation of the ADMM l2 algorithm looks similar to the usual BP-based algorithm in CN and VN computation scheduling. The horizontal layered scheduling offers interesting features for a hardware based implementation. Consequently most of the recent BP LDPC decoder implementations [8], [9], [14] have focused on this CN centric scheduling. However, it is important to notice that the computations involved in VN and CN for ADMM l2 decoding differ, to a great extent, from BP ones.
A formalized description of the ADMM l2 algorithm based on the horizontal layered scheduling is provided in Algorithm 1. It is composed of three main steps: variable to check message processing, Euclidean projection and parity check to variable message processing. In this paper, we consider a simplified version of the Euclidean projection algorithm provided in [5]. Compared with the one proposed in [3], it minimizes the control dependencies as well as the computational complexity which facilitates its parallelization and thus its implementation.

IV. NISC-BASED ADMM l2 LDPC DECODER The design of LDPC decoders is an arduous task. Different techniques were proposed to reduce the design cycle of such architectures. One way is to generate the control part and the memory mapping of LDPC decoder from the parity check matrix on a generic architecture thanks to an automated flow [9]. In such case, the hardware architecture is partially parallel. Therefore, only a subset of nodes in the Tanner graph is instantiated as check node functional units (CNU). An overview of the architecture proposed in [9] is provided in Figure 1. To facilitate the design of hardware ADMM l2 decoders and to make fair complexity and throughput comparisons with NMS decoders, we decided to reuse this generic architecture and its associated design methodology. However, as detailed below, the CNU design of the ADMM decoder is very different from the NMS CNU in [9]. The variable to check message processing is more complex since the high computational complexity Euclidean projection block is introduced. Moreover, additional memory elements are necessary for the messages storage. In fact, the global decoder architecture looks similar. It includes Mu memory modules for storing the VNs (Ti) and P CNUs that process the CN computation in parallel and store the z and λ messages as well. Besides, two Benes networks [15] are dedicated to interleave and deinterleave functions to route VN values both at the beginning and at the end of the CNUs processing according to the H matrix structure. A controller unit is in charge of the control signal generation and also of the address generation for data-path and memories. PUs and MUs numbers may vary. Indeed, the number of memory units can be higher than the number of processing units in order to help the memory access conflict resolution. Finally, a system interface monitors the data exchanges with the rest of the system.

B. Quantization formats
Like in all communication systems design flows, the ADMM l2 decoder’s design started at a behavioural level where all numeric formats in the system model are in a floating-point format. In a hardware implementation, fixed-point numeric formats are used exclusively since they offer significant savings

in terms of silicon area cost and processing latency. Hence, conversion from floating-point to fixed-point numeric formats is an essential step. To this end, we consider the range and the precision of each data in the design as well as the overflow and the induced errors. Thus, a trade-off between performance and hardware complexity is optimized through finding the lowest implementation cost which gives an acceptable level of performance.
The fixed-point conversion of the CNU is split into two steps. First, the number of bits for the integer part is fixed after a dynamic range evaluation. This number of bits is minimized while ensuring that no overflow nor a limited number of overflow occurs. Second, the number of bits for the fractional part is determined. The limited number of bits used to represent the data generates an error between the infinite and finite precision values. This error must be precisely estimated and controlled. Indeed, the ADMM l2 decoding includes multiplication and division operations that are not used in Min-Sum based decoding algorithm.
According to the top level data path of the CNU depicted in Figure 2, three data widths are mandatory for the CNU block:
• wv: the width of the variable to check message used to get the MU data size.
• wλ and wz: the widths of the check to variable messages which are used to get the message RAM data size wm.
In the remainder of this paper, the following notation is considered: Ql.fr format refers to a quantization format on l bits, where f r bits are used for the fractional part.

V. EXPERIMENTAL RESULTS
The architectures were synthesized using Xilinx ISE 14.7 tool targeting a Xilinx Virtex6 FPGA device (xc6vlx240t3ff1156). NISC-based ADMM l2 decoders dedicated to the decoding of the LDPC codes in Table I were synthesized. Quantization format presented in previous section were used.
One can notice that the three decoders for LDPC codes C1, C2 and C4 have comparable logic and registers costs as they seem to have almost the same configuration P |Mu in terms of processing units (P ) and LLR memories (Mu). The two decoders having the same P |Mu configuration need the same memory amount (C1 and C2).
Concerning the longest considered code C3, the corresponding decoder has an increased hardware resources requirement. This can be due to the fact that the number of processor units is different from the number of memory units. An other point is about its configuration that has 30% more memory units than the other considered long codes. In addition, all the synthesised decoders assign DSP blocks. They are to be likely dedicated to the division operations used in the Euclidean projection inputs which increase with the number of CNU configured.
Regarding the frequency, only the short LDPC code C0 succeeds to reach 100 MHz. The other selected LDPC codes are below the target frequency. The path is slowed not only because of the complex Euclidean projection block, but because of the large data widths too.

All the NMS decoders succeed to reach around 150 MHz as frequency and their throughputs are ×1.5 to ×2 higher than those of the ADMM l2 decoders. This difference is mainly due to the decoders’ frequency. Indeed, a difference of dc cycles per iteration is detected between the two decoders. For the five synthesized decoders, the ADMM l2 decoder consumes between 6 and 7.5 times more LUTs and 5.5 times more flip flops than the NMS decoder. Meanwhile, the ADMM l2 decoder consumes between ×1.2 and ×1.7 more block RAMs than the NMS decoder. In addition, the NMS hardware decoders do not use any DSP modules since the operations in the NMS algorithm have a low complexity. The hardware complexity increase of the ADMM l2 decoder, when compared with the NMS is due to many factors. First, the quantization format used for ADMM l2 decoding is higher than the NMS decoder. In fact, the quantization scheme in the NMS decoder is 8 bits for Ti and 6 bits for check messages. These data lengths are a key factor to determine the memory footprint of the decoder. Also, Ti’s length impacts the hardware complexity. For instance, the adder complexity doubles in accordance with Ti length. Second, the computational complexity of the CN data-path is high due to the Euclidean projection implementation. Finally, as stated previously, the ADMM l2 decoding process requires the storage of two (z and λ) message elements which doubles the message memory cost in comparison with the NMS decoder.
To show the high impact of all these factors, we synthesised the ADMM l2 CNU. Hardware complexity and timing results are provided in Table III which includes also the synthesis results of the NMS processing unit. Additional synthesis results show that the Euclidean projection consumes 40% of the logic resources of the ADMM l2 CNU block and 10% of its registers. The critical path is between the output of the stage φ1 and the output of the Euclidean projection in the second stage φ2. Additional pipeline stages are necessary to solve this limitation. To sum up, these synthesis results show that the hardware cost of the ADMM l2 architecture is higher compared with the NMS one. However, it is counterbalanced by the peerless error correction performance as shown in Figure 7.
VI. CONCLUSION
In this paper, an ADMM l2 decoder based on a NISC architecture is described. The proposed decoder is based on a partially parallel architecture. It implements a horizontal layered decoding algorithm. It can be used for regular LDPC

codes. Implementation results demonstrate the applicability of the ADMM l2 algorithm for a hardware FPGA target though the implementation cost is found to be higher than the NMS decoder with the same NISC-based architecture. This cost can be further reduced if we succeed to decrease the data widths in the decoder’s data-path without impacting the error correction performance. In addition, the timing results can be enhanced by modifying the data flows of the NISC. The results demonstrate that LP based hardware LDPC decoders are realizable, while still requiring a great deal of optimizations in order to reduce the decoder hardware complexity.
