A Low-Power Termination Criterion for Iterative LDPC Code Decoders

G. Glikiotis and V. Paliouras
Electrical and Computer Eng. Dept., University of Patras, 26500 Greece

Abstract—This paper introduces a novel criterion for the termination of iterations in iterative LDPC Code decoders. The proposed criterion is amenable for VLSI implementation, and it is here shown that it can enhance previously reported LDPC Code decoder architectures substantially, by reducing the corresponding power dissipation. The concept of the proposed criterion is the detection of cycles in the sequences of soft words. The soft-word cycles occur in some cases of low signalto-noise ratios and indicate that the decoder is unable to decide on a codeword, which in turn results in unnecessary power consumption due to iterations that do not improve the bit error rate. The proposed architecture terminates the decoding process when a soft-word cycle occurs, allowing for substantial power savings at a minimal performance penalty. The proposed criterion is applied to Hardware-Sharing and Parallel Decoder architectures.
I. INTRODUCTION
Error correcting codes are used for achieving reliable data transmission over noisy channels [1]. LDPC codes are powerful error correcting codes [2], originally introduced by Gallager in 1961 [1]. There is much interest in LDPC codes, because they offer bit error rate performance, which approaches asymptotically the Shannon limit [3] and they are endowed with a parallel, linear time-complexity decoding algorithm [4].
In some cases, the decoder is trapped in a continuous repetition, without concluding in a codeword; i.e., after a certain number of iterations, the soft words start to repeat. This happens mainly for small channel SNR values. In such cases, the decoding process continues without improving the final result.
In this paper, two LDPC architectures are presented, which allow the decoding process to terminate whenever a cycle occurs. In this way, unnecessary iterations are avoided and power consumption decreases without imposing a substantial performance loss.
The remainder of the paper is organized as follows: In Section II, the LDPC codes and the message passing algorithm are reviewed. In Section III, the performance of a 1920-bit, rate 1/3, soft decision LDPC decoder is presented, for various word length message values. The simulated

number of cycles detected for different SNR values is also presented. In Section IV, two different types of LDPC decoder architectures are presented and in Section V, these architectures are modified so as to allow cycle detection. Finally, in Section VI power consumption measurements are presented, compared, and discussed for the proposed architectures.

II. LDPC CODES

A. Matrix Representations of LDPC Codes
LDPC codes [7] are linear block codes, whose paritycheck matrix H is a sparse binary matrix with regard to set elements. As in all linear block codes, it holds that

HxT = 0,

∀x ∈ C

where x is a codeword and C is the set of all code words.

Each row of H corresponds to a parity check and each set element in position (i, j), shows that the jth data symbol is included in ith parity check. Within a block of n symbols or bits, there exist m parity symbols and the code rate r is

r = (n − m) / n,
where n is the block length and m is the number of parity checks.

The set row and column elements of H are chosen to satisfy a desired row and column profile. Row and column weight profiles are defined as the number of set elements in a particular row or column respectively [8].

B. Graph Representation of LDPC Codes
LDPC codes can be represented using a bipartite graph [9], as the one shown in Fig. 1. The graph of Fig. 1 refers to a regular LDPC code of length 9, column weight 2, and row height 3. One set of nodes represents the parity check constraints and the other set represents the symbols. The graph edges represent the set elements of matrix H. For every set element in position (i, j), check node i is connected to bit node j.

C. The Message Passing Algorithm
LDPC codes are decoded iteratively using the message passing algorithm [5]. The algorithm calculates extrinsic probability values associated with each bit node using disjoint parity-check equations. Each iteration consists of two calculation parts. In part 1, all updates of all bit nodes are done by sending messages to neighboring check nodes. In part 2, all updates of check nodes are performed by sending messages to neighboring bit nodes.

III. LDPC CODE SELECTION AND PERFORMANCE
A. LDPC Code Selection
The proposed technique is demonstrated by its application to an irregular LDPC code with a block size of 1920 bits and a code rate of 1/3. Other LDPC codes were also tested. Finally, the selected code was preferred for its sufficiently good performance in relation to its block size which allows for an implementation and use in real applications. The test codes were taken from [6]. The particular soft decision LDPC code is based on a graph with an average column weight of 2.66 and an average row weight of 4. The graph is shown in Fig. 2 and it consists of 1280 bit nodes of degree 3, 640 bit nodes of degree 2, and 1280 check nodes of degree 4.
B. LDPC Code Performance and Comparison
The 1920-bit rate-1/3 LDPC code was tested for several different message word lengths, as well as for double floating point precision. Fig. 3 depicts the coding gain of the LDPC code decoder exchanging 7-b and 8-b messages compared to the coding gain using double-precision messages. The performance was obtained using a bit accurate software implementation of the algorithm described in Section II. It is clear that the BER for 7-b, 8-b and double precision messages remains in the same order of magnitude. Using messages of less than 7-b precision, results to BER values of lower order of magnitude, so they are not shown in Fig. 3. In all three cases, simulations were done having a maximum limit of 100 iterations. In Table I, values of BER and the average number of iterations for values of SNR 0, 1, and 2 dB and accuracy of 7-b, 8-b, and double-precision are presented. We can see that as the value of SNR increases, the average number of iterations decreases. This is due to the fact that as the value of SNR increases, fewer cycles occur. For an SNR value of 0dB, the average number of iterations is 100, meaning that a cycle occurs in every packet.

B. Parallel Decoder Architecture
Parallel decoder architecture follows by mapping the LDPC code graph (Fig. 2) to hardware [7]. In order to complete one iteration, bit nodes and check nodes are used only once and the messages are exchanged through routed wires. This architecture allows for a high throughput and does not demand complex control. The drawback is the relatively increased area.

VIII. CONCLUSION
In this paper, a termination criterion for iterative LDPC decoding and its proper impact on the implementation was studied. The proposed criterion reduces the number of iterations of the decoding process, each time the decoder is trapped in a continuous repetition without concluding in a codeword. This is due to the soft-word cycles that occur in certain cases of low SNR values. To estimate the reduction of the energy dissipation by using the proposed termination criterion, a 1920-bit, rate-1/3 LDPC code decoder was selected, described in VHDL and synthesized in 0.18-µm 1.8V 1P6M logic process for 8 bit messages. The wordlength of the messages was defined due to bit-accurate software simulation. The energy consumption comparison reveals that for values of SNR 0 and 1 dB, the reduction of the energy consumption is up to 78% for the Parallel Decoder architecture. For the Hardware-Sharing Decoder architecture, the maximum energy saving is 18%, for 1 dB SNR. For value of SNR 2 dB, the energy consumption is increased by 73%. That is because in the case of the Hardware-Sharing architecture, the energy consumption of the CAM is 3.23 times the energy consumption of the decoder. For both architectures, a minimal performance penalty occurs, as shown in Table III.
The methodology followed in this paper, can be used in the design of other LDPC code decoders, in order to reduce the power consumption of the design, without a substantial loss in performance.
