Diagonal Low-Density Parity-Check Code for Simplified Routing in Decoder

Euncheol Kim
Electrical Engineering Texas A&M University
Bryan, Tx USA eckim@ee.tamu.edu

Abstract—We propose a novel Low-Density Parity-Check (LDPC) decoder design methodology by introducing a special code named Diagonal-LDPC (DLDPC) code. An LPDC code, defined by a parity check matrix H, can be represented by a bipartite graph. To address the complex routing problem in the LDPC decoder implementation, a partitioned bipartitegraph code is proposed and generalized to a DLDPC code having constraint of positioning 1’s near the diagonal area. This structured code simplifies the routing problem [1] [2] [3] and enables cell-based highly regular fully-parallel decoder design without compromising the code performance.
I. INTRODUCTION
LDPC code is well known for its remarkable errorcorrecting capability. The application of this code has been studied to implement its hardware efficiently while maintaining its outstanding code performance. The difficulties of hardware implementation arise mainly from the decoder implementation specifically in its complex wire interconnections of functional circuits. In order to address such a problem, the relationship between the properties of parity check matrix H and the associated bipartite graph and physical hardware implementation is thoroughly investigated. To obtain localized and simple routing solution, a graph partitioning technique is explored. The result of graph partitioning technique is further generalized to a construction of an H matrix which has restriction on positioning 1’s near the diagonal area. This approach yields code construction methodology for cell-based highly-regular decoder design. This design method provides much simpler routing solution and stackable decoder design without compromising the code performance. The approach used in this study can also easily be extended to irregular code [4]. An example DLDPC code construction method and its decoder implementation is carried out for a (2048, 3, 6), rate 1/2 code. The Bit Error Rate (BER) performance of this example code shows 2.56 * 10-6 BER at 2.3dB Signal to Noise Ratio (SNR). This is comparable to that of optimal LDPC code. Advantage of this approach is that, due to the regularity and locality in routing resulting from the structured code, the DLDPC decoder implementation leads to a cell-based regular array architecture. A die size of ~59

mm2 physical layout is realized with TSMC 0.25um technology for (2048, 3, 6), rate 1/2, regular DLDPC decoder.
This paper is organized as follows: In Section II, the background of LDPC code is described briefly together with decoding algorithm. In Section III, a check-node partitioned graph code is presented for fully parallel decoder implementation. The resulting parity-check matrix is analyzed with different design constraints. Bit error rate simulation is performed and the results are shown. Section IV describes generalization of graph partitioned code and DLDPC code construction. An example DLDPC code construction and its decoder implementation is demonstrated. Section V concludes the paper.
II. BACKGROUND This section describes the background of LDPC codes and the physical design issues in decoder implementation. In digital communication, the channel coding technique is widely used to maximally utilize the channel bandwidth and to reduce the data transmission power. It is a ForwardError-Correction (FEC) technique which uses data redundancy to correct errors at the receiving end. However, even with channel coding there is a limitation of data rate that can be transmitted over a physical channel. This boundary is so called Shannon limit. Shannon showed that given the bandwidth and the Signal to Noise Ratio (SNR) of the channel, there is upper bound of data rate so called Channel Capacity. Therefore one of the important objectives of channel coding is to find good error-correcting codes which can approach as close to the Shannon limit as possible. Turbo codes are the first codes which showed near Shannon limit performance. After the success of turbo codes in achieving remarkable error-correcting performance, the iterative decoding of turbo-like codes inspired the researchers to revisit the LDPC code and the possibility of implementing its hardware system using the latest advanced VLSI technology. As a result of these efforts, much of work is reported regarding the characterization of code performance and the code construction methodologies which can yield high coding gain [5][6]. Several research studies have been presented about the interconnect issues in LDPC decoder implementation. A prototype parallel LDPC decoder with a code-length of 1024-b is implemented in [7]. A special placement and routing software is developed to insert and place buffers by enforcing Manhattan routing geometry. With increase of code-length, this approach poses difficult implementation challenges due to the complex interconnects resulting from the random connectivity between the functional circuit units. A partly parallel decoder [8] [9] implementation tries to reduce the routing complexity by considering design of structured LDPC codes having regularity in its physical features. The routing complexity is reduced and the scalability is improved at the cost of reducing the degree of parallelism. However, implementation solutions for fully parallel LDPC decoder designs that promise a very high throughput have not yet emerged due to the difficulty in addressing the routing requirements of its massive number of data-flow structures.
The constraints of LDPC code construction and the decoding steps are briefly introduced to help explain detailed discussions about hardware implementation in the following sections. LDPC codes belong to a class of linear errorcorrecting block codes. As the name suggests the codes are defined by sparse parity check matrix that consists mostly of 0’s. The coding gain of this code is known to approach very close to the Shannon Limit as the code-length increases [5][6]. First introduced by Gallager in 1962 [10], the parity check matrix H of (n, j, p) regular LDPC code has the property such that each column contains a small fixed number j ≥ 3 of 1’s and each row contains a small fixed number p > j of 1’s. The code-length of this code is n and the H matrix consists of n columns. Suppose that the number of data bits before the channel encoding is k, then the number of rows of H matrix is n – k. The rate of this code is defined as R = k/n ≥ 1 – j / p. The equality holds when the rank of H matrix is equal to k. The set of codeword, x ∈ C, consists of all one-dimensional row vectors that spans the null space of a parity check matrix H; x · HT = 0, ∀ x ∈ C.

LDPC codes can also be represented by a bipartite graph shown in Figure 1. In this bipartite graph, one set of nodes (check nodes) corresponds to the parity check equations of H matrix and the other set of nodes (variable nodes) corresponds to the symbol inputs to the parity check equations.
The decoding of LDPC codes is based on the iterative Message-Passing Sum-Product (MPSP) algorithm, also known as Belief Propagation (BP) algorithm [6][11][12]. It is a probabilistic decoding which iteratively computes extrinsic probability values associated with each variable node using disjoint parity-check equations that the variable participates in. The decoding procedure at each iteration stage includes two distinct probability computation steps. One is a Check Node Update (CNU) and the other is Variable node Update (VNU). In the CNU, each row of the H matrix is checked to verify that the parity check constraints are satisfied. In the VNU, the probability will be updated by summing up the other extrinsic probabilities computed from the rest of the rows and the intrinsic probability from the channel output [10].
Even though this efficient iterative decoding algorithm yields very good coding gain like the turbo codes [13], practical implementation and use of LDPC codes has been limited due to the excessive hardware resource requirements and complex routing problems. In this study, we focus on fully parallel LDPC decoder that is a direct implementation of the bipartite graph shown in Figure 1. The direct implementation is to construct decoder hardware by instantiating variable nodes and check nodes in the graph with appropriate circuit components. Although mapping of the circuit components to the nodes may be straightforward, physical implementation is still difficult because of the complex randomized connections between the nodes.
III. PROPERTY OF H MATRIX In this section, we describe the property of parity check matrix H mentioned earlier together with its associated bipartite graph and the physical layout. The matrix is analyzed with respect to the connectivity issues that arise when an optimized routing solution at the layout-level implementation is sought. We propose a scheme wherein the bipartite graph is partitioned and the resulting LDPC code is scrutinized through BER simulation to study the impact on the code performance.
A. Partitioning Bipartite Graph The objective of this graph partitioning is to simplify the
connectivity requirements between the check nodes and variable nodes in the direct implementation. A design approach that explores this problem is presented in [7]. The problem is again pointed out in [8] where the lengths of interconnect wires and the area of the decoder are lower bounded and proportionally increase to n1/2, when the decoder layout is planned with uniform distribution of circuit components.

The direct implementation is preferable since not only it is natural to use the graph structure in a circuit implementation but also the resulting decoder performs block parallel decoding. However, the routing complexity of direct implementation increases as the code-length increases. The number of circuit components and interconnections is linearly proportional to the code-length. Due to the dense interconnection wire network, it is very difficult to find routing solution for long code-length decoder with limited number of metal layers. Furthermore, the random construction of the code will make it more complicated to connect wires. With arbitrarily positioned ‘1’s in the parity matrix, the resulting layout architecture should provide flexible interconnection between CNUs and VNUs.
There have been many efforts to reduce the complexity of interconnection in the LDPC decoder. However, not so many of them are related to the direct implementation of LDPC decoders. In the direct implementation, so far the effort was to relocate the position of circuit components to minimize the routing complexity. In this approach, the parity matrix is first designed to achieve the best error correction performance and the layout will be followed after that. However, with increase of code-length, this approach will give much more difficulty in finding reasonable routing solution. To overcome this problem, layout oriented code construction can be considered.

In the parity check matrix, the position of ‘1’s determines which CNU and VNU would be connected. Therefore, a localized ‘1’s in a parity matrix can lead to a localized routing in the decoder layout. However, localized ‘1’s in a matrix usually tends to create many short cycles and aggravates code performance. So it is desirable code construction approach to minimize the performance degradation and maximize the localization of ‘1’s in a matrix.
A simple example is shown in Figure 3 to examine the effect of localizing the position of 1’s in the H matrix. In Figure 3 (a), the rows are divided into two subgroups of C1, C2 and the columns into three subgroups of Va, Vb, Vc. The shaded area in the matrix denotes the area that can be filled with ‘1’s. The blank area is filled with only ‘0’s. With this constraint imposed on the H matrix, the locations of ‘1’s are localized. All the 1’s in the column subgroup Va only appears at the row C1. The 1’s in the column subgroup Vb appears both at C1 and C2. Vc column subgroup has entry of 1’s only at C2. Figure 3 (b) shows the partitioned bipartite graph resulted from the H matrix in Figure 3 (a). The graph shows that only the variable nodes in Vb have edges from both C1 and C2. In Figure 3 (c), an example of physical layout that can easily be obtained from the partitioned graph is shown. In this layout floorplan, Va-C1 area and Vc-C2 area is divided by Vb VNU circuit area. Assuming that the code-length is equal to the direct implementation in Figure 2 (c), much simpler localized routing solution can be achieved. The worst-case routing distance is almost halved and average routing distance is shortened.

B. DLDPC Code Construction In order to construct an H matrix that has most 1’s are
placed along the diagonal, it’s possible to use random construction method. In this method, first the boundary will be set and then the position of 1’s are randomly picked. However, this method has several drawbacks in physical layout and code performance. Since the construction method is random, the resulting matrix will have random connection between the check and variable nodes. Although the range of interconnection wire length can be limited by the predefined diagonal boundary, the global routing should be performed manually. And second, if the diagonal width W is not big enough, the possibility of creating many short cycles will be increased. The short cycles will degrade the code performance. [16]
One other way of generating DLDPC code is to repeat a structured sub-matrix pattern and eliminate short cycles. In Figure 8, one example code construction of structured H matrix is illustrated. Figure 8 (b) is a part of H matrix shown in Figure 8 (a). In Figure 8 (b), the small squares are identical sub-matrix pattern that repeats every 16 columns. And each neighboring adjacent sub-matrix differs by 8 rows in its relative row position. The size of sub-matrix is 260*16. In the sub-matrix, each column consists of three 1’s. In the diagram above, CNU gets six Log-likelihoodratio (LLR) values from VNUs corresponding to each variable node. To compute next iteration LLR, five ψ transformed LLR will be added except for the LLR of itself. And then the sum is inversely transformed to get the new LLR. The block computational units are synthesized and the physical layout is generated using the Cadence Silicon Ensemble. The technology is TSMC 0.25u process. The resulting block layout size for CNU is 280u x 160u = 4.48x104 u2 and VNU is 80u x 80u = 6400 u2. Using the DLDPC code, the layout is simply stacking array of cells. Each cell consists of one CNU and two VNUs. The estimated die size is ~59 mm2 for 2048 code-length decoder. To compare the die size with the state-of-the-art design, the die size is estimated for 0.16u technology and the result is ~ 24 mm2. When compared to the die size of [7], which is ~52.5 mm2 for 1024 length code, this is only half the size. So the equivalent area reduction is ~75%. This reduced chip area will turn into reduction of power consumption and enhancement of operation frequency.
V. CONCLUSION An LDPC code construction approach is presented to obtain simplified routing architecture. The approach uses code design to minimize the wire routing distance in a parallel design, and thereby reducing the average parasitic loading associated with interconnects in the layout. The effect of partitioning geometry in the parity matrix is analyzed through the BER simulation. With block size of

2040, the connectivity between the partitioned sub-matrices can go down to 0.2 while achieving BER performance comparable to that of direct implementation. The worst-case routing distance is reduced by a factor of 1/2. This approach can be extended further to achieve hierarchically localized routing architecture. To show the effect of localized routing LDPC code construction, an example of structured LDPC code construction methodology is developed. The resulting LDPC decoder implementation is cell-based regular array architecture and the routing congestion is effectively reduced. The die size of proposed architecture is estimated and compared with state-of-the-art LDPC decoder design. The chip size reduction is ~75% and this reduced chip size will lead to reduced power consumption and speed enhancement. This novel methodology will lead to a high throughput parallel LDPC decoder without routing congestion.
