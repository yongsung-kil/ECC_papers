Design and Implementation of LDPC Codes for
DVB-S2
Manoj K. Yadav & Keshab K. Parhi Department of Electrical and Computer Engineering
University of Minnesota Twin Cities, MN 55455 manoj,parhi@ece.umn.edu

Abstract --In this paper, we present the design and FPGA implementation of 11 LDPC codes with code rates 1/4, 1/3, 2/5, 1/2, 3/5, 2/3, 3/4, 4/5, 5/6, 8/9 and 9/10 for normal frame length of 64800 bits as used in DVB-S2. Out of these 11 codes, 7 are regular and 4 are irregular. All of them have been synthesized into Xilinx Virtex-II XC2V8000 FPGA.

I.

INTRODUCTION

Low density parity check (LDPC) codes have been proposed as the inner code for forward error correction in DVB-S2 (Digital Video Broadcast –satellite) [1]. LDPC code is a linear block code described with a binary sparse M x N parity check matrix H. Each row of the matrix H corresponds to a parity check and each column represents a demodulated symbol. The number of non-zero elements in a row or column is defined as the row weight dc or the column weight dv. If all rows and all columns are of uniform weight, the LDPC code is called a regular code. Otherwise it is an irregular code. The Tanner graph has been introduced to represent LDPC codes. Tanner graph is a bipartite graph with variable nodes on one side and constraint or check nodes on the other side. In the graph, each variable node corresponds to a received symbol, each check node corresponds to a set of parity check equations, and each edge corresponds to a non-zero entry in the parity check matrix. For example, a regular LDPC H matrix, dc = 2 and dv = 1, and its corresponding Tanner graph is shown in Fig. 1.

The DVB-S2 standard requires LDPC codes for 11 different code rates namely 1/4, 1/3, 2/5, 1/2, 3/5, 2/3, 3/4, 4/5, 5/6, 8/9 and 9/10 for normal frame length of 64800. It also requires these codes to perform within 1 dB of the Shannon limit. Although it is possible to have random LDPC codes that satisfy these constraints, it might not be feasible to implement them. In fact the standard itself provides all these codes but it is very difficult to implement them unless we know how these codes were constructed. Unfortunately, the standard doesn’t give any detail about this.
We have designed all these codes in such a way that using the same decoder architecture they can be synthesized into a Xilinx Virtex-II XC2V8000 FPGA. Out of these 11 codes, 7 are regular and 4 are irregular. This paper describes the details of the code construction approaches and their FPGA implementations. We first describe the decoding algorithm and decoder architecture that we have used and then code designs, followed by conclusions.

DECODING ALGORITHM

We have used modified min-sign algorithm as proposed in [3] since it provides an excellent trade-off between performance and hardware complexity. Due to lack of space, we are not providing the details here. Interested readers are advised to refer [3.]

III.

DECODER ARCHITECTURE

The decoder structure for irregular parity check matrix is very complex. But if H is composed of Identity matrices and their cyclic-shifted matrices, then decoder structure can be greatly simplified.
E.g., H is a 2 x 4 array of 5 x 5 circulants where each circulant is either Identity matrix or its cyclic shifted version. It has 20 bit nodes {vi}(i=1.. 20) and 10 check nodes {cj} (j= 1.. 10). Each check sum involves 4 bits and each bit is checked by 2 checks. The decoder corresponding to above H matrix is shown in Fig. 2. The decoding process can be described as follows:

1. Initialization: Flush received messages to both 5 x 4 receive data memories m_vi (i= 1,2 3,4) and the corresponding 5 x 6 message memories Mj_k (j = 1, 2 and k = 1,2 3,4). Here 4 are the length of received data and 6 are the length of messages. The data are stored column-wise in the both type of memories. Since our received symbols are of length 4, we sign extend them by 2 bits before saving them in message memories. 2. Check Node update: In each subsequent iteration, for all the Check-Node-Computation-Units (CUs) read the updated variable-to-check message from the message memories in the same block row and then after Check node computation write the updated check-to-variable message back to the same address as read operation in the order of offset, offset+1…4,0….offset-1. The value of offset is different for each message memory and it is basically the location of single 1 in the first row of each permutation matrix Pi,j of H. Note that here indexing start from 0 instead of 1. 3. Variable Node update: Similarly in the same iteration, for all the Variable-Node –Computation-Units (VUs) read the updated check-to-variable message from the message memories and the intrinsic messages from the received data memories in the same block column, and then after Variable node computation write the updated variable-to-check message back to the same address in the order of 0 ……4. At the same time, the decoded code bits are written to the

5 x 1 decoded code word memories m_ci (i = 1,2 ,3 4).

The decoding process terminates when the pre-assigned maximum number of iterations are reached.

IV.

DESIGN OF CODES

In this section, we describe how we designed the 11 LDPC

codes for DVB-S2 with code rates 1/4, 1/3, 2/5, 1/2, 3/5, 2/3,

3/4, 4/5, 5/6, 8/9, 9/10 and code word length = 64800. 7 of

these codes, code rate 3/5 to 9/10, are regular and rest 4, code

rate 1/4 to 1/2 are irregular. Our approach has been to first try

to find a regular code for a particular code rate. Only when we

failed to do so then we tried for an irregular one. The reason

for this preference is the ease of implementation of regular

codes. Our goal has been to design codes that can use the

decoder architecture described in section III and can be

synthesized into a reconfigurable device like Xilinx Virtex-II

XC2V8000 FPGA.

The major constraint in the hardware implementation of

B. Irregular LDPC Codes We first describe our method in general and then we provide
details for each of the 4 codes. To construct an LDPC code with an M x N parity check matrix, where M = p*Ms and N = p*Ns.

1) Use a heuristic algorithm (such as bit-filling) to construct a group of base parity check matrices of girth 6 with the size of Ms x Ns. 2) Randomly expand each base matrix into a p* Ms x p* Ns parity check matrix. By expand we mean that replace each ‘1’ in the base matrix by a permutation matrix of size p x p and each ‘0’ by a zero matrix of size p x p. The position of single ‘1’ in each of these permutation matrices is found randomly. An example is shown in Fig. 11. The advantage of using these kinds of structured codes is that we can still use our decoder architecture of Fig 2. Though it is not exactly the same but for small base matrices it is not very difficult to use same kind of architecture.

We would like to point that it is possible to generate very good irregular codes but their implementation may not be feasible. Our irregular codes have been designed by keeping in mind the hardware resources available to us in XC2V8000.

V.

CONCLUSIONS

In this paper, we have showed the construction of LDPC codes with code word length of 64800 and 11 different code rates as desired for DVB-S2. All our codes have been synthesized into Xilinx Virtex-II XC2V8000 though not all at the same time. It is possible to have all at the same time if we have more memory resources at our disposal. The minimum and maximum frequencies are 21.58 MHz and 46.46 MHz respectively. One noticeable thing is the BER performance for rate 1/3 and 1/4 codes. Again the reason is lack of enough memory. We know it for a fact that better irregular structured codes that have performance within 1 dB can be generated. But as we have stressed from the beginning itself that we have tried to generate codes that can be synthesized into our target device. It also shows the good low rate LDPC codes esp. with code rate less than 3/5 are difficult to construct.
