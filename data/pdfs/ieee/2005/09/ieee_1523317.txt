Structured eIRA Codes with Low Floors
Yifei Zhang, William E. Ryan, and Yan Li ECE Dept., University of Arizona, Tucson, AZ 85721 Email: [yifeiz,ryan,yanli]@ece.arizona.edu

Abstract— In a recent paper, Yang et al. presented the class of extended irregular repeat-accumulate (eIRA) codes which are efficiently encodable LDPC codes possessing very low errorrate floors and are appropriate for code rates 1/2 or greater. While efficiently encodable, the left-most (n-k)-by-k submatrix of an eIRA code parity-check matrix is random in nature, making efficient decoder implementation problematic. In the present paper, we present structured eIRA codes. We first present some ensemble results for the general class of eIRA codes, after which the subclass of structured eIRA(SeIRA) codes is defined. Software- and hardware-based performance results for SeIRA code designs (rates 0.5 to 0.9) are then presented.
I. INTRODUCTION
Random or pseudo-random LDPC code constructions result in an unstructured bipartite graph topology, which makes efficient LDPC encoder and decoder implementations very difficult. Parallel and partially parallel encoder and decoder implementations demand a structured LDPC code parity-check matrix and, hence, a structured bipartite graph topology. In this paper, we propose a design scheme for constructing a class of SeIRA LDPC codes that not only exhibit very good error-control performance, but also allow efficient decoder parallelization.
In a previous paper [1], we introduced the class of pseudorandomly constructed eIRA codes which admit high rates, low encoder complexity, and have very low error-rate floors. The eIRA codes are a family of low-density parity-check (LDPC) codes which have the special parity-check matrix form

II. PROPERTIES OF RANDOM EIRA CODE ENSEMBLES
It was recognized in [1] that, unless H1 is properly designed, weight-1 and weight-2 encoder inputs give rise to the lowest-weight codewords. Proper design of H1 means arranging the ones in the columns of H1 so that: (1) they are maximally separated, thus maximizing the parity weight corresponding to weight-1 encoder inputs and (2) the ones in the sums of column-pairs of H1 are maximally separated, thus maximizing the parity weight corresponding to weight-2 encoder inputs. In this section, we will examine the performance of random eIRA code ensembles to support the above design guidelines.

C. Numerical Results
We consider the (4161, 3430) eIRA code ensemble defined in this paper and the codes presented in [1]. The Pb conditioned on inputs of weights from 1 to 4 for the ensembles with wc = 3 and 5 were calculated using the above analytical results and are shown in Fig. 2. We observe that the lowweight inputs, i.e., w = 1 and 2 are dominant, even though
their multiplicities are much smaller than those of the larger-
weight inputs. Note that the conditional performance of the wc = 5 ensemble is better than that of the wc = 3 ensemble. The simulation results of the three (4161, 3430) eIRA codes in [1] with different column weights wc = 3, 4, and 5 are also plotted in Fig. 2. Among these three simulated eIRA codes, the code with wc = 5 has no error floor down to Pb = 10−9 and is only about 0.2 dB inferior to the wc = 3 code at Pb = 10−6, whereas, both wc = 3 and wc = 4 codes have error floors above Pb = 10−9. For this reason, we favor wc = 5 in our designs hereafter.

IV. CODE DESIGNS AND PERFORMANCE
While the algebraic constructions such as those in [2] and [3] generally have limited flexibility in the choice of length and rate, construction of SeIRA codes is much more flexible. Given code length n, rate R, and girth g (in this paper, we refer it to the minimum cycle including both information and parity variable nodes), the first step is to determine the values of L, J, Q, and the initial form of the base matrix P defined in (11). If masking is used, we set the corresponding bl,j’s to ∞. The masking technique used in this paper is cyclic masking [9]. We randomly selected the masking pattern for the first column of the base matrix P. Then the masking patterns for the following L − 1 columns are generated by cyclically shifting the pattern for the first column. It is obvious that the masking pattern will repeat after shifting L times. Hence, we usually change the masking pattern after masking L columns, that is, randomly select a masking pattern which is different from any of the previous patterns (including the

shifted ones). This yields the initial base matrix. According to it, we randomly generate integers to replace the unmasked bl,j’s until certain girth conditions are satisfied.
As noted above and in [1], encoding is easily performed via the H matrix given the form of (13) and (2). The H matrix in this form may also be used at the decoder. However, the pixelated image of H (in which a pixel in an m × n array of pixels is darkened for each 1 in H) fails to display much structure. Thus, in Example 1 below, we present the images of the alternative H matrix form
H = Π−T · H = [P Π−T H2] = [P ΠH2], (19)
where Π is an orthogonal permutation matrix. Because H possesses more apparent structure than H, it is more obvious that it would lead to a lower-complexity encoder/decoder than its unstructured counterpart. Because it is merely a row-permuted version of H, an H -based decoder will have performance equivalent to that of an H-based decoder.
Example 1: For n ≈ 1050 and R ≈ 0.77, we have constructed four SeIRA codes with various parameters for error-rate performance comparisons. The first two eIRA codes have n = 1053, k = 810, L = 3, J = 10, Q = 81 (note nonprime), and one of these has g = 6 and the other 8. Another code has n = 1062, k = 826, L = 4, J = 14, Q = 59 , and g = 6. The fourth code has n = 1056, k = 816, L = 5, J = 17, Q = 48 (nonprime), and g = 6. Note that the parameter L of our SeIRA codes is also the column weight of H1 and masking was not used for any of these codes. The parity-check matrix images for the form H of (19) for three of the above eIRA codes are compared with that of a MacKay code [11] with wc = 4 and wr ∈ {17, 18} in Fig. 3. The computer-simulated frame-error rate (FER) results are shown in Fig. 4. All of the computer-simulations in this paper have a maximum of 50 decoder iterations. We note that performance is improved by increasing L from 3 to 5. For L = 3, the girth-8 eIRA code has almost the same performance as the girth-6 eIRA code, and both of them have lower error floors than MacKay’s code. MacKay’s code performs a little better than our SeIRA code with L = 4. The L = 5 SeIRA is clearly the best code and has no floor down to an FER of 10−6.
Example 2: Consider a rate-1/2 (2048, 1024) eIRA code with L = 16, J = 16, Q = 64. Cyclic masking is used for both codes to set the column weight of H1 to 5. The hardware [12] and software simulation results are shown in Fig. 5. The hardware simulator counted errors in both parity and information parts of the codeword, whereas we count them only in the information part for both BER and FER. The hardware simulator also employed 200 decoder iterations whereas we employed 50. Observe that the hardware curve reaches a near-zero-slope for at BER=10−9. We attribute this to trapping sets [13] rather than a small dmin.
Example 3: In this example, two groups of eIRA codes are presented. They both have information bit length k about 4096, but the rates are 0.9 and 0.8. For the rate 0.9 SeIRA code, we choose L = 5, J = 46, Q = 91 and obtain a (4551,

4096) code by deleting J · Q − k = 46× 91 − 4096 = 90 columns of ΠT P from the right. The (4550, 4095) eIRA code was constructed pseudo-randomly by the method in [1]. The parity-check matrices of these two rate-0.9 codes have the same weight distribution. For the rate 0.8 SeIRA code with k = 4096 and n = 5120, we choose L = 8, J = 32, Q = 128 and use the cyclic masking technique to change the column weight of H1 to 5. A pseudo-random eIRA code with the same k, n, and weight distribution was also constructed by the method in [1]. Only the structured (5120, 4096) eIRA code was simulated by both software and hardware. The hardware simulation has a maximum of 200 decoder iterations. All of the simulation results are shown in Fig. 6. From the results, we notice that the SeIRA codes perform as well as the pseudorandom eIRA codes in the simulated region. For all of the above codes, there is no error floor down to BER ≈ 10−8. We remark that, for the hardware simulation in Fig. 6, only one codeword error occurred in approximately 730 million simulated codewords at an Eb/N0 of 3.67dB.
