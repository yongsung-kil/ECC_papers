VLSI Design of a High-Throughput Multi-Rate Decoder for Structured LDPC Codes

Massimo Rovini, Nicola E. L’Insalata, Francesco Rossi and Luca Fanucci Department of Information Engineering - University of Pisa Via G. Caruso, I-56122 Pisa - Italy
{massimo.rovini, nicola.linsalata, francesco.rossi, luca.fanucci}@iet.unipi.it

Abstract— Despite recent advances in the microelectronics technology, the implementation of high-throughput decoders for LDPC codes remains a challenging task. This paper aims at summarising the top-down design flow of a decoder for a structured LDPC code compliant with the WWiSE proposal for WLAN. Starting from the system performance analysis with finite-precision arithmetic, a high-throughput architecture is presented as an enhancement of the state-of-the-art solutions, and its VLSI design detailed. The envisaged architecture is also very flexible as it supports several code rates with no significant hardware overhead. The overall decoder, synthesised on 0.18 µm standard cells CMOS technology, showed remarkable performances: small implementation loss (0.2 dB down to BER = 10−8), low latency (less than 6.0 µs), high useful throughput (up to 940 Mbps) and low complexity (about 375 Kgates).
I. INTRODUCTION
Low-Density Parity-Check (LDPC) codes were first introduced by Gallager in 1963 [1], but soon forgotten by the scientific world. Indeed, their implementation complexity was far exceeding the capabilities of the microelectronics technology of the time, so that this class of codes was not considered for practical applications. However, the recent need of the modern communication systems of operating close to the Shannon limit of channel capacity [2], renewed the interest in LDPC codes [3].
Despite the recent advances in microelectronics technology, the hardware design process is not straightforward neither trivial, the main challenge to the success of LDPC codes remaining the possibility of defining low-complexity high throughput decoders. It has largely been demonstrated that decoder implementations based on parallel architectures show large chip areas even with short block sizes [4], mainly caused by routing congestion; resorting to turbo-like encoded codes partially relaxes the place and route phase [5], but the related decoder still suffers from poor flexibility and reconfigurability.
Many studies have been accomplished to conceive families of LDPC codes that simplify the VLSI implementation of the related decoder: these are called “architecture-aware” [6] or “VLSI oriented” [7] LDPC codes. Typically, the parity check matrix of these codes is blockwise partitioned, which allows the definition of efficient time-multiplexed architectures. Forthcoming communication standards, such as DVBS2 [8], WLAN (802.11n) [9] and WMAN (802.16e) [10], have adopted or are considering the use of such codes.
Modern communications are extremely demanding in terms

of transmission throughput (up to 1 Gbps for multimedia broadcasting), power consumption (for mobile terminals), chip complexity and architecture flexibility (support of different code rates and modulations).
In this context, this paper presents the top-down design flow and implementation of a decoder for a structured LDPC code considered as optional Forward Error Correction (FEC) scheme in IEEE 802.11n, for WLAN applications. After a brief introduction to the LDPC decoding problem and a concise description of the proposed codes, some considerations about fixed-point decoding will be reported along with an overview on high-throughput decoder architectures. Then, the VLSI design will be detailed and the results of this analysis in terms of implementation loss, throughput and chip complexity will be shown.
II. LDPC DECODING ALGORITHM
LDPC codes are linear block codes characterized by a sparse binary parity check matrix H counting M rows and N columns. A powerful representation tool of an LDPC code is the Tanner graph [11], which is a bipartite graph made up of two entities, variable nodes (VN) and check nodes (CN), connected each other through a set of edges (see Fig. 1). Check nodes represent a (even) parity check equation, while variable nodes are representative of the elements in the codeword. An edge links the check node i to the variable node j if and only if the element Hi,j of the parity check matrix is non-null.

IV. FIXED-POINT LDPC DECODING

III. LDPC CODES DESCRIPTION
The WWiSE proposal for IEEE 802.11n [9] features a short block LDPC code as optional FEC scheme for high throughput applications. Four code rates are supported for backward compatibility, namely 1/2, 2/3, 3/4 and (optionally) 5/6, along with high order quadrature amplitude modulations (16-QAM and 64-QAM).
The proposed LDPC codes are architecture-aware, as they are structured and blockwise partitioned into squared subblocks with size P = 27. In particular, to allow the definition of high-throughput parallel decoding, these sub-blocks, when non-null, are either the diagonal identity or a rotation of the diagonal identity matrix.
To speed up the encoding time an IRA-like structure has been adopted [17] and the dimensioning of the elementary processing units and interconnections has been relaxed thanks to the quasi-regular VN degrees distribution.
Independently of the code rate, the codeword length is constant to N = 1944, as a good trade-off between FEC capabilities and hardware implementation complexity.

Due to the high hardware complexity required by LDPC decoding, the finite precision analysis of the algorithm is a must, in order to address the desired trade-off between implementation complexity with dedicated circuits and performance loss, compatible with the selected application.

B. CN processing
As far as check node processing is concerned, as mentioned in Sect. II, we focused our attention on the modified-MS approximation [15], which proved to behave very close to the true BP algorithm, but admits a simpler VLSI implementation.
Independently from the approximation, the CNP operates on homogeneous inputs, thus no representation alignment is required. In addition, the output check-to-variable message belongs to the same category of the input variable-to-check messages, with no change in representation, again.
In particular, under the modified-MS approximation, the CNP computes its output messages by recursively using the binary operator M-min*(|µa|, |µb|) =. min(|µa|, |µb|) − log(ed/(1 + ed), with µa, µb two generic variable-to-check messages, and d = ||µa| − |µb||. It is easy to prove that this update rule holds the granularity and dynamic range of the inputs; thus, its straightforward fixed-point implementation is

V. STRUCTURED LDPC DECODER ARCHITECTURES
Structured LDPC codes allow the use of efficient highthroughput decoder architectures. The main requirement for this is the blockwise partitioned structure of the parity check matrix, which lends itself to the implementation of partially parallel structures [6], [7]. Furthermore, when the non-null sub-blocks are either the diagonal unit matrix, or its rotations, as in WLAN or DVB-S2 [8], customised architectures are viable [18]. Independently from the practical solution, they all parallelise the decoding process, by taking advantage of ‘ad-hoc’ structures in the code parity check matrix. The main difference among those is in the number and the nature of processing units.

B. Vectorised Decoding
In a Vectorised LDPC decoder [18] decoding proceeds blockwise serially. A vector of P processors (V/CNPs) update a macro column/row a time, in parallel; to allow collision-free parallel memory access, the memory is vectorised as well, a memory row storing all messages related to a non-null subblock. Within the single macro column/row, the vector of P processors reads, updates and write back in memory its messages in sequence. This calls for serial architectures of VNPs and CNPs. To implement the message rotation within sub-blocks, the vector of P messages is passed through a programmable switch-bar with size P before being written back in memory. Fig. 4 summarises the architecture of a vectorised LDPC decoder.

C. Parallel Vectorised Decoding
Although Partially Parallel decoder architectures assure remarkable performances in terms of throughput (greater than 500 Mbps), unsolved points remain the high level of memory fragmentation (for long codes and small page sizes P ) and the chip routing, which can reduce the expected, theoretical figures in an unpredictable way. On the other side, when P is small to medium, the Vectorised decoder does not perform as good as the Partially Parallel one, but its complexity is lower (fewer processors, serial processors are simpler). However, its memory organisation is very efficient, and re-configuration to support multi-rates is easily achieved.
To boost the throughput of a Vectorised architecture, a further degree of parallelism has been introduced. In this way, not a single vector of processors, rather, G vectors are used in parallel. Each vector operates in the same way described in Sect. V-B, but only on a sub-set of consecutive macrorows/columns; to allow parallel memory access, memory is not only vectorised, but also organised in G × G separate pages. C. Decoding of higher rate codes
The support of higher rates has to cope with high CN degrees, as reported in Tab. II. Dimensioning the system on the worst-case scenario of code rate 5/6 would demand a huge complexity of the elementary CNP (dC,max = 21) and so of the whole decoder. Furthermore, with this approach, there is hardware resource wasting when lower rates are considered.
The problem is solved by exploiting the peculiar way higher code rates are defined (see Sect. III). For instance, to derive the rate 5/6 parity matrix, rows in the prototype matrix are grouped in three consecutive subsets and mod-2 summed. This operation preserves the blockwise structure of the parity check matrix.
An efficient hardware implementation could then compute the parity checks of higher rate codes as the (even) parity check of the (up to three) component rows. To do that, two auxiliary, slave, processors must accompany the master CNP of Fig. 7, which is now dimensioned for dC,max = 7 of the prototype matrix. These auxiliary units are in charge of computing the un-marginalised M-min* over the whole set of messages belonging to the summed rows. This metric, defined µ∗, is then used by the master CNP to initialise the accumulation register of the M-min* accumulator on the βbranch. As the β-recursion starts with a delay of dC clock

cycles with respect to the first message arrival time, the computation of µ∗ from the auxiliary unit may be performed in parallel to the α-recursion of the master CNP, with no extra latency in the overall decoding time.
The architecture of this auxiliary unit, generally quite simple, under the modified-MS approximation becomes a trivial M-min* operator with feedback register, already sketched in Fig. 8. An output register triggers the accumulated metric every dC clock cycles. A similar architecture with xor accumulation is used for signs.
D. Message Vector Switch
As highlighted in Fig 4, the message vector must be rotated after the VN/CN phase and before writing in memory. This is performed by a programmable switch, capable of performing P = 27 possible rotations of the input message vector.
An efficient network that implement this function has been found in a Banyan switch [19]. Particularly, the switch size is extended to L = 64: the input message vector is first duplicated and padded to L with L − 2P = 10 null messages. Note that the modular nature of the Banyan structure easily allows the introduction of pipeline into the network, not to spoil the decoder performance (throughput and latency).

The overall synthesis of a decoder for WWiSE has been performed using the vectorised architecture with memory paging. To keep the decoding latency below 6 µs, compulsory for WLAN applications, decoding is run up to Nit = 12 iterations. Accordingly, the paging factor G = 6 has been chosen and the memory split in G2 = 36 vectorised pages.
Table IV reports the logic synthesis breakdown of the whole decoder with the clock frequency of 180 MHz. Complexity is measured in equivalent Kgates for IPs and in bits for memories. An overall complexity of 375.14 Kgates is measured, plus 88452 bits of RAM and 10872 bits of ROM.

Under this assumptions, the overall decoding latency of equation (8) is evaluated for E = 6804 and Lit = 6, Llast = 11 clock cycles. With the clock frequency of 180 MHz and the overall parallelism factor of G · P = 162, expression (7) yields 5.73 µs, which meets the aforementioned constraint of 6 µs. Note that our architectural design assures a decoding latency, which is independent of the channel code rate.
With this latency the useful throughput, measured in information bits per second, ranges in 565.1 Mbps with code rate 1/2 to 941.8 Mbps with rate 5/6. This also corresponds to the constant channel throughput, measured in coded bits per second, of 1.13 Gbps; the related channel baud rate is 282.6 Mbaud with 16-QAM or 188.4 Mbaud with 64-QAM.

The design space has been explored at many levels: first, BER performances have been analysed for many of the most significant code configurations, and the decoder implementation loss studied as a function of the quantisation parameters. Then, after conceiving a flexible, high throughput VLSI architecture as an enhancement of the state-of-the-art solutions, the impact of quantisation on the implementation complexity has been appreciated, and the VLSI design detailed in its peculiarities. Particularly, the design presents input LLRs and extrinsic messages quantised on 5 bits, which is roughly paid with only 0.2 dB of loss in the BER curves.
The proposed architecture features the paging of the message memory to boost the throughput by parallelising decoding; however, its management is kept efficient as in a Vectorised decoder. With the paging factor of 6 the decoder reaches useful throughput as high as 941.8 Mbps, which corresponds to channel baud rates ranging in 282.6 to 188.4 Mbaud.
The logic synthesis with 0.18 µm standard cells CMOS technology showed that the overall decoder only counts 375 Kgates plus 86.4 kbit of RAM. Despite this is only RTL complexity, no strong overheads are expected from the back-end place and route thanks to the vectorised approach. Moreover, flexibility in terms of code rates is assured by our architecture with no extra impact on latency.
The described design-flow is very general, and different design drivers might have been adopted, as not to spoil performances or to further reduce the final decoder complexity. Although this activity focused on the WWiSE LDPC codes proposed for WLAN, results are easily extendible to other structured codes of the same family, like DVB-S2.
