STRUCTURED LOW-DENSITY PARITY-CHECK CODE DESIGN FOR NEXT GENERATION DIGITAL VIDEO BROADCAST
Mustafa Eroz and Lin-Nan Lee Hughes Network Systems Germantown, MD

ABSTRACT
We describe a novel design for low-density parity-check (LDPC) codes that eliminates the routing problem associated with LDPC decoder implementation resulting in a small silicon area in application specific integrated circuits (ASICs). Unlike previous solutions that require a large number ofsmall and hence area inefficient memory blocks, this design allows collision-free memory access on a single memory block by imposing certain restrictions on the parity check matrix ofthe codes. Therefore, it leads to substantially smaller circuits. Furthermore the resulting codes have outstanding error rate performance very close to Shannon limitfor a wide range ofthroughputs from 0.5 bitslsymbol up to 4 bitslsymbol. As a result LDPC codes designed with this method have been standardizedfor the next generation digital video broadcasting (DVB).
INTRODUCTION
One of the main reasons that prevented LDPC codes [1][3] from widespread practical use before turbo codes was their decoding routing complexity. Unlike turbo codes, LDPC codes have an easily parallelizable decoding algorithm. But the major challenge in LDPC decoder implementation is how to establish the communication between "randomly" distributed bit nodes and check nodes which has emerged as the subject of many research papers [6][11]. There are two basic approaches: The first one is a fully parallel architecture where there are as many physical nodes in the circuit as the logical nodes of the code [8]. Even though this approach exploits all of the parallelism inherent in the decoder and helps achieve very high data rates, it becomes rapidly infeasible with increasing block size due to the complexity of interconnections between the nodes. The second approach, which we follow in this paper, uses a partially parallel architecture, where a small number of physical nodes (i.e. parallel engines) are re-used to serve all of the logical nodes avoiding the complexity of fully parallel approach especially for long block sizes [6]. But the challenge here is to access relevant information for the logical nodes in process (i.e. contents of adjacent edges of the nodes in process), fast enough from memory in high speed applications. All of the previous papers on this topic attacked this problem by dividing the memory block into

as many small pieces as the number of parallel engines and by designing LDPC codes that allow simultaneous access to small memory blocks without any collisions. But, implementation of a large number of small memory blocks in modern Very Large Scale Integrated (VLSI) circuits tends to be very inefficient in area. Furthermore, Tarable et al.
[12] showed recently that as long as small, multiple mem-
ory blocks are available, it is possible to obtain collisionfree memory access for any code. In this paper, we describe a novel way of designing LDPC codes by imposing certain restriction on the parity check matrix so that a single, area-efficient memory block can be accessed in parallel without any collision. Therefore, the multiple memory block assumption of [12] does not apply to our design.

Considering that numerous iterations are required to achieve good performance and several read/writes of edge contents are needed per iteration, it becomes clear that high speed applications cannot be supported due to memory access speed limitations. DESCRIPTION OF THE PARITY CHECK MATRIX DESIGN
The key to structured access while achieving good performance lies in the specification of the parity check matrix. In general a parity check matrix can be described by specifying which check nodes are connected to the bit nodes. In our method, we specify the adjacent check codes for a group of M bit nodes at a time, as described next. The number M has certain physical meanings. Firstly, it denotes the number of parallel computation engines for bit node and check node processing. Secondly it denotes the number of columns in the RAM matrix that holds the contents of bipartite graph edges. Relation of these two meanings will become clearer in an example given later in this section. We place the following main restriction on the parity check matrix:

ARRANGEMENT OF THE EDGE RAM MATRIX
Before processing a group ofM check nodes or M bit nodes, the contents of all the adjacent edges have to be read from RAM and after processing, the updated values have to be written back. But in order to efficiently perform this task, the required edge contents have to be placed "together" in RAM for all the groups of M nodes so that multiple edges can always be fetched in one clock cycle. Of course, while achieving this structure some level of randomness must be maintained in order not to sacrifice performance.

One would rightfully argue that the above favorable placement of edges adjacent to groups of M check nodes can be achieved for any random LDPC code by suitably labeling the edges. The innovation of the design lies in the fact that adjacent edges for groups of M bit nodes are simultaneously placed together in some rows of edge RAM's which follows directly from the previously described parity check matrix restriction. On the other hand unlike check nodes, adjacent edges of the bit nodes are not stored in consecutive top RAM rows. For a group of M bit
nodes, all with degree dv > 2, the indices for the dv rows
that carry all the adjacent edges are random and must be stored in ROM. Furthermore within a specific row, the edge adjacent to the first bit node can be anywhere in that row. Therefore the column number for that edge must also be stored in ROM. The column indices of the edges adjacent to the other bit nodes follow the first column index in
a "wrapped around" fashion. For example, if the jth edge
in the row belongs to the first bit node, then the (j + 1)'t

edge belongs to the second bit node, (j + 2)nd edge belongs to the third bit node,., and (j - 1)'t edge belongs
to the Mth bit node. In this example, only the num-
ber j would be stored in ROM.

HOW TO CHOOSE M
From the previous discussion, it is evident that the most important design parameter is M, the number of parallel computation engines for bit node and check node processing. Obviously choosing a large Mwould enable high speed decoders at the expense of increased area, and vice versa. More quantitatively, we have
