Abstract
In this paper we propose a constant-time parallel algorithm for implementing the message-passing decoder of LDPC codes on a two dimensional RMesh, trying to keep the number of processors small. The R-Mesh provides dynamic reconfiguration, hardware reuse, and flexibility to problem changes. To decode a different code, we may simply set up the required connections between the bit-nodes and check-nodes by modifying the initialization phase of the R-Mesh algorithm. No extra wiring or hardware changes are required, as compared to other existing approaches. Moreover, the same hardware can implement the decoder in both probability and logarithm domains. We illustrate that the R-Mesh is an efficient model for parallel implementation of the decoder in terms of time complexity, flexibility to problem changes and simplicity of routing messages.
1. Introduction
Low-Density Parity-Check (LDPC) codes, introduced by Gallager [1, 2] are error control codes defined by sparse parity-check matrices. These efficient block codes have attracted a lot of attention due to their remarkable bit error rate and signal to noise ratio (SNR) performance and elegant decoding scheme. LDPC codes have been accepted as part of the new Digital Video Broadcasting (DVB) standard and are the leading candidates in the next generation Wireless Local

Area Network (WLAN) designs with data rates over 100 Mbps. The message-passing decoder is a competent iterative algorithm for decoding LDPC codes [1-3]. This decoder is based on groups of nodes communicating with each other by computing and sending messages. The computations may be in probability or logarithm domain, where the latter domain is more practical. The time complexity of the decoder grows linearly with the code length. This is considerable, as efficient codes have long codewords. However, the parallel architecture of the decoder allows reducing its time complexity significantly.
There are a number of parallel implementations of the LDPC decoder in the literature that can implement the decoder in constant time. Blanksby and Howland [4] discussed the issues involved in parallel versus hardware-sharing architectures, and designed a parallel architecture for a 1024-b, rate1/2 decoder, using a 1.5-V CMOS process. AlRawi et al. [5] proposed a pipelined parallel architecture and tried to optimize the node mapping and scheduling (NP-hard) to achieve minimum conflicts in memory access. Levine et al. [6] implemented the decoder using a commercial Field Programmable Gate Array (FPGA) with lookup tables whose number increased with the number of nodes. These parallel architectures can implement the decoder in constant time. However, they do not exploit hardware reuse. Each node is implemented using extra hardware, while practically a bit-node and a check-node act in serial and they may use the same hardware without increasing the time complexity. They also employ fixed connections between a bit-node and the parity checks that include it. Thus, decoding a different code (defined

Reconfiguration enables hardware reuse, scalability, and performance improvement. It also accommodates algorithms that are adaptive to data and problem changes. The most important feature of the reconfigurable bus architecture is its ability to create different interconnection topologies allowing exchanging information efficiently. These architectures use their buses as a computational resource, allowing the execution of fast parallel algorithms [8-12]. Reconfigurable models have been shown to be computationally more powerful than non-reconfigurable models [7].
In this paper we use the 2-dimensional reconfigurable mesh (R-Mesh), which is a simple and efficient reconfigurable bus-based model [7], for parallel implementation of the message-passing decoder in logarithm domain. To design the decoder in probability domain, we may use the same mesh with some minor computational modifications in the R-Mesh algorithm.
The proposed R-Mesh decoder runs in constant time and provides hardware reuse. The same hardware implements both a bit-node and a checknode in each iteration of the algorithm. Moreover, the same hardware may be used to implement decoders for different LDPC codes defined by different parity-check matrices. To setup the required connections between the bit-nodes and check-nodes as specified by the parity-check matrix, we only need to modify the initialization phase of the R-Mesh algorithm. Thus, designing a decoder for different LDPC codes involves no hardware modifications, as compared to other existing methods.

2. The R-Mesh Model
The Reconfigurable Mesh (R-Mesh) is a parallel model of computation that utilizes dynamic reconfiguration. An M×N R-Mesh is a two dimensional array of processors with M rows and N

columns. Processor P (i, j) is the processor on row i and column j of the array.
Each processor has four ports: N, S, E, and W. These ports connect the processor to its north, south, east, and west neighboring processors through a set of fixed external connections. Each processor can also establish 15 different internal connections among its ports. These connections are the different partitions of the set of ports, and can change locally and independently at each step of an algorithm [7]. 3. Low-Density Parity-Check (LDPC) Codes
LDPC codes are linear binary block codes defined by sparse parity-check matrices [1, 2]. An M u N parity-check matrix HM u N defines an LDPC code of length N, where each codeword c = (c1 c2 c3 … cN) satisfies M parity-check constraints. In other words, the LDPC code is the set of all vectors c satisfying c u HT = 0. The sparseness of the parity-check matrix enables near optimal decoding and the randomness ensures an efficient code probabilistically [1-3].
Definition 1: A sparse parity-check matrix HMu N, with exactly k ones in each row, and exactly j < k ones in each column defines a regular (j, k) LDPC code. Matrix H is binary and sparse. In other words, only a small fraction of the entries of H are one and the rest are zero. Therefore, j and k are both small constants compared to N [1-3].

4. The Message-Passing Algorithm

The message-passing algorithm is an efficient, iterative algorithm used for decoding LDPC codes

[1-3]. The algorithm is based on passing messages from the bit-nodes to the check-nodes of the Tanner graph, and vice versa per iteration. Each node operates like a processor. It first collects the incoming messages, and then computes and sends out its own message. Figures 3 and 4 show the algorithm notations and the algorithm in log domain, respectively.
The algorithm has two stages: initialize and iterate. Let c = (c1 c2 … cN) be the codeword transmitted on the channel (channel input), R = [r1 r2 … rN] be the received signal (noisy channel output), and G2 be the variance of channel noise. Consider the Tanner graph of the parity-check matrix that defines the code. During initialization, each bit-node 1 d j d N computes its message vj = (2 / G2) u rj and sends it down to Mj, the set of all check-nodes connected to bit-node j. Check-node update. This phase updates the upward messages for each edge of the Tanner graph (refer to Figure 2). Assume that an edge connects check-node 1 d i d M and bit-node 1 d j d N. The upward message corresponding to this edge (ui,j) is a probability measure that indicates which value of cj (cj = 1 or cj = 0) satisfies parity-check i. This probability measure is computed based on the messages received from all the bit-nodes connected to check-node i, excluding bit-node j. Thus, during the check-node update, each check-node i computes its message using its received data, and sends the message up to Ni, the set of all the bitnodes connected to check-node i. Figure 2 marks the edges connected to Ni = {1, 3, 4, 6} for the first check-node (i = 1) of the Tanner graph of H in Relation 2.
Bit-node update. This phase updates the downward messages for each edge of the Tanner graph. The downward message corresponding to the edge that connects check-node 1 d i d M, and bit-node 1 d j d N, is denoted by vi,j. The downward message vi,j is a probability measure indicating if cj is one or zero. This measure is computed based on the messages received from all the check-nodes connected to bit-node j, excluding check-node i. Thus, during the bit-node update, each bit-node j employs its received data to update its own message and then sends it down to all the check-nodes of the Tanner graph that are connected to it (Mj).
These messages iterate between bit-nodes and check-nodes to finally compute the a posteriori log likelihood ratio (LLR), On = log (Pr (cn = 1 | R) / Pr (cn = 0 | R)) for each bit-node n. The number of iterations is denoted with lmax in Figure 4. The algorithm converges when there are no considerable changes in the output values, or we are within the desired degree of result accuracy. Practically, this happens after a constant number of iterations (usually no more than 10 or so). Therefore, lmax is a constant [1-3]. After convergence, we use On to decode the sequence. Positive On indicates that Pr (cn = 1 | R)  Pr (cn = 0 | R), and so cn = 1. Similarly, negative On means cn is zero. For details and derivations refer to [1-3].
5. Overview of the Proposed R-Mesh Algorithm
We can employ multiple processors to compute multiple messages in parallel, and reduce the running time of the message-passing decoder significantly. The proposed algorithm works on the following basis. In each iteration, the processors

first compute all the upward messages, and then all the downward messages in parallel. However, each processor needs messages from some other processors for updating its own messages. Thus, we should route to each processor the data it requires. However, routing each required upward or downward message to the corresponding processors makes routing complicated, and increases the hardware size.
Instead of routing individual messages, the proposed algorithm computes a key value P, for each check-node (using the downward messages of the edges connected to the check-node) and broadcasts it to the processors that compute the upward messages for these edges. It also computes a key value S, for each bit-node (using the upward messages of the edges connected to the bit-node) and broadcasts it to the processors that compute the downward messages for these edges. We design the proposed algorithm based on the following principle.
Principle 1: Any processor that is computing the upward (downward) message for a given edge holds the downward (upward) message corresponding to this edge locally, at the time of computation. The processors might have computed this local data or received it from other processors.
In the R-Mesh implementation each processor computes this local data itself at the preceding check-node or bit-node update. Section 6.2 provides the details.
The proposed algorithm updates the messages as follows. Consider the update for a given checknode m. The processors computing the upward message um,j for 1 d j d N hold vm,j locally (Principle 1). These processors team up and compute the key value P by Equation 3.

6. The Proposed Implementation

We explain the proposed implementation in the following sections. We explain the implementation model, the inputs and the output of the algorithm. We also present the proposed algorithm in detail, and clarify it by an example. We complete the section with the algorithm time complexity analysis.

6.1 The Implementation Model Consider the LDPC code defined by a given
parity-check matrix HM×N. We use an N×M (the same size as HT) R-Mesh to implement the decoder for this code. The hardware has one row for each bit-node and one column for each check-node of the Tanner graph of H. Note that we number the rows and columns of the R-Mesh from 1 to N and 1 to M, respectively.
Input: The algorithm inputs are, 1. R , the channel output. 2. G2, the variance of channel noise. 3. Mn, the set of the parity-checks that include bit-node 1nN. Mn = {i:H(i, n) =1} for 1didM.
Output: The algorithm output is the a posteriori log likelihood ratio On = log (Pr (cn = 1|R) / Pr (cn =

0|R)), for each bit-node n. The output On is a probability measure indicating if the bit-node is zero or one based on the channel observation.

This stage initializes the local variables for each of the processors, maps the Tanner graph to the RMesh by marking the processors as required, makes lists of the marked processors, and determines the head of each list. The processors that contribute in updating messages set their local variable flag to one, and are referred to as marked processors. Note that there is a one to one relation between the location of ones in HT and the location of marked processors in the R-Mesh. The unmarked processors mainly set up connections among the marked processors as required. After marking the processors, neighbor localization [7] is used to make lists of the marked processors on each row (row-list) and also the marked processors on each column (col-list). The idea is to establish a bus between any two successive marked processors on a row or a column, by fusing the ports of all the unmarked processors between them. For example, to make lists on a row, each marked processor writes its index to its west port and reads from its east port to identify the first marked neighbor to its right.
The initialize stage of the algorithm, also determines the head of each list. If the marked processors write their indices to their east ports and read from west ports then the row list is reversed. Then, the head of the list can be determined as the only marked processor without a neighbor to its left [7]. A similar method finds the head of a column list.
During the second stage of the algorithm (iterate), the processors update the messages until convergence. Figure 6 shows the iterate stage of the proposed algorithm.
The marked processors of each column list 1djdN team up to compute the key value Pj by Equation 3. The processors compute Pj using their downward messages and tree structure multiplication [7]. The head of each list holds the result and broadcasts to every processor in the list. Each marked processor P picks the product up, and computes its upward message uP by Equation 4 using its local data vP, and the received product P.
Next, the marked processors of each row list 1didN team up to compute the key value S by Equation 5, using their upward messages and tree structure summation [7]. Each marked processors P picks the summation up, and computes its downward message vP by Equation 6, using its local upward message uP and the received summation S. Thus, the iteration is complete.

6.4 Time Analysis The following theorem analyses the time
complexity of the proposed decoder.
Theorem 1: The message-passing decoder of the LDPC code defined by a regular (j, k) sparse parity-check matrix HM×N, runs on an NuM R-Mesh in constant time.
Proof: The initialize stage of the proposed algorithm (refer to Figure 5) mainly includes constant-time instructions for reading and broadcasting the input values. This stage also includes neighbor localization on columns and rows of the R-Mesh, in that order, to make onedimensional column and row lists, followed by finding the head of each list. A one-dimensional RMesh can perform both in O(1) time, as well [7].
The message-passing algorithm converges after a constant number of iterations, denoted with lmax [1-

3]. Thus, the main loop of the iterate stage (refer to

Figure 6) requires a constant number of iterations.

Each iteration includes computing the key values

P and S. For a regular (j, k) code, the key values P

and S are the product of at most k and sum of at

most j < k numbers. Using tree structure

operations, the R-Mesh can compute the key values

in at most O(log k) time [7], where k is a constant

by the definition of regular LDPC codes in Section

3. After computing the key values, the R-Mesh can

broadcast these values to the required processors,

and update the messages in O(1) time. Thus, using

the R-Mesh and the proposed algorithm, the

decoder runs in constant time.

7. Conclusions

We illustrated that an NuM R-Mesh implements the message-passing decoder of the LDPC code defined by a sparse parity-check matrix HM×N, in constant time. Let E be the number of edges in the Tanner graph of H or the number of ones in H. Among these NuM processors, E processors are marked and contribute in message computations. Each marked processor computes a pair of messages per iteration. The rest are mainly used to simplify routing messages. Instead of routing each and every required message, two key values are computed and broadcasted to the corresponding processors. In each of the iterations, the processors use their local data and these key values to update their messages. This approach helps reduce routing and hardware complexities significantly.
The R-Mesh is an efficient model for implementing the message-passing decoder. It provides dynamic reconfiguration, hardware reuse, and scalability. It enables decoding LDPC codes of any length, in constant time. Decoding a different code involves no architecture changes, extra wiring, or reordering hardware blocks, as compared to other existing methods. Moreover, the same hardware implements the decoder in both logarithm and probability domains.
