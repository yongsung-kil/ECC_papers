A Scalable Decoder Architecture for Linear Congruential LDPC Codes
Abhiram Prabhakar and Krishna Narayanan Department of Electrical Engineering, Texas A&M University, College Station, TX-77843 Email: {pabhiram,krn}@ee.tamu.edu

Maximal length linear congruential sequence (MLLCS) based LDPC codes have the advantage that the LDPC code graph can be generated at the receiver without having to explicity store the graph. Hence, these codes are advantageous when the same hardware needs to be used for different sets of rates and lengths. In this paper, we reveal an inherent structure in these codes that facilitates parallel implementation of the decoding algorithm. Based on this, we present an architecture for the MLLCS-LDPC decoder that facilitates parallel scalable implementation and joint code-decoder design.
Keywords-LDPC decoder, MLLCS, Parity check matrix, SPA, bipartite graph.
I. INTRODUCTION
Low Density Parity check codes [2] have been the focus of intense research in recent years. The parity check matrix of a regular (N, K) LDPC code is sparse with λ ones in each column and ρ = λ/(1 − K/N ) ones in each row. A bipartite graph can be associated with the LDPC code and the decoder is a message passing decoder which passes extrinsic messages along the edges in this graph. While the bit error rate performance of LDPC codes is impressive, the application of LDPC codes for real world error correction systems is limited by issues related to hardware implementation of the decoder. Serial architectures have high latency while parallel architectures have high hardware complexity that prohibits practical implementations. Partly parallel implementations that explore the trade off between throughput and hardware complexity have been proposed as the solution.
The parity check matrix (connections in the graph) of an LDPC code is typically constructed at random with constraints to avoid short cycles, especially those of length 4. While the performance of such constrained random LDPC codes are good, they lack any special properties that can be exploited for efficient implementation of the decoder. Also the parity check matrix for such codes need to be stored at the decoder. For a communication system operating with codes of various lengths and rates, the parity check matrix of each code has to be stored at the decoder and this requires huge quantity of memory. Recently, a number of LDPC decoder architectures have been explored [4]- [9]. Some of these [4], [6], [8], present a joint code-decoder design. For such designs the parity check

matrix can be generated at the decoder and need not be pre stored. These designs propose structured LDPC codes that can be exploited for the decoder design. In our previous work [1] we have proposed regular MLLCS-LDPC codes with λ = 3 and girth 6. Such codes require only a set of 3 values to generate the parity check matrix of the LDPC code at the decoder and still perform as well as random constructions with explicit constraints to remove loops of length 4. In [3] further research on [1] has been done and techniques for constructing both regular and irregular MLLCS-LDPC codes of girth 6 and higher have been proposed.
In this paper, we show that MLLCS based LDPC codes posses additional structure which makes these codes suitable for partly parallel scalable implementation. We show one such design which works directly on the edges of the graph structure associated with parity check matrix and can be used to obtain varying degrees of parallelism.
Section II explains MLLCS codes in brief and the corresponding bi-partite graph structure. Section III explains how such codes can be used for partly parallel decoder implementation with an example. Section IV explains the message-passing decoding algorithm briefly. Section V explains the proposed decoder.
II. GRAPH STRUCTURE AND MLLCS-LDPC CODES
Low density parity-check codes are well represented by bipartite graphs. One set of nodes, the variable nodes or bit nodes, corresponds to elements of the codeword and the other set of nodes, viz. check nodes, corresponds to the set of paritycheck constraints satisfied by each codeword. A (λ, ρ)-regular low density parity check code has a graphical representation in which each variable node or bit node is connected to λ edges and each check node to ρ edges. We use the following representation to ease the exposition in the paper. We first associate with each check node ρ connections called check connections and with each bit node λ connections called bit connections as shown in Fig. 1 for λ = 3 and ρ = 6. Therefore, there are totally M = N λ bit and check connections. An edge in the graph connects a bit connection on the left to a check connection on the right. The edge between the pth bit connection and Apth check connection represents an edge between p/λ th bit node and Ap/ρ th check node.

III. INHERENT STRUCTURE IN MLLCS-LDPC CODES
The key for implementing a partly parallel LDPC decoder is to divide the check connections of the graph into subgroups. This allows the extrinsic messages along the edges of the graph in each subgroup to be stored in different memory banks and therefore simultaneous read/write operations for several extrinsic messages is possible. V. MLLCS-LDPC DECODER ARCHITECTURE
The block diagram of the decoder architecture is shown in Fig. 2. The key aspect to this structure is the memory implementation for storing the extrinsic messages made possible due to the subgrouping of check connections. We shall number the extrinsic messages at the check connections and since each check connection is connected to only one edge, both the extrinsic message number and check connection number refer to the same value. The memory for storing the extrinsic messages is divided into an array of smaller memory blocks. There are M 2 rows with M 1 number of memory blocks in each row. Here, M 1 is the number of simultaneous extrinsic messages processed at the bit node update(BTU) unit and M 2 × M 1/ρ is the number of simultaneous messages processed at the check node update(CNU) unit. The logical mapping of check connections to memory locations is shown in Fig. 3. The example with parameters M = 40, M 1 = 4 and M 2 = 2 is only for illustration purposes and does not satisfy the necessary conditions for a MLLCS-LDPC code. The first check connection is mapped to the first memory location on (row1,column1) memory block. Subsequent check connections are mapped from left to right to the first memory location of memory blocks on the same row. Then the mapping is continued to memory location 2 of each memory block and so on.
The extrinsic messages for a check node update are read from the same memory location of ρ consecutive memory blocks in a row. Hence the M 1 extrinsic messages read in parallel from each row can be used for M 1/ρ check node updates. Since there are a total of M 2 rows, M 1 × M 2/ρ check node updates can be processed in parallel. We have shown in section III that M 1 consecutive bit connections are connected to check connections belonging to M 1 different subgroups. In the proposed architecture, the memory blocks in each column represent a subgroup. Hence the M 1 columns represent the M 1 subgroups of check connections. Since M 1 extrinsic values can be read in parallel M 1/3 bit node updates

can be processed in parallel. Address generation to identify the exact memory location and memory block to which each bit connection is connected to, can be easily computed by knowing the check connection to which it is connected as defined by the MLLSC sequence, M 1, M 2 and the size of each memory block which in effect is determined by M .
The channel output values are stored in M 1/3 memory blocks for parallel access by the bit node units.The purpose of the router is to channel the extrinsic message from each memory column to the correct bit node update unit and viceversa. It should be noted that for a given code, the routing is constant over time. This is because the subgroup sequence repeats itself in the same order. If the decoder is used to decode only one LDPC code the router can be replaced by hard wiring each memory column to the appropriate bit node update unit. This is an advantage when compared to partly parallel design in [5] where we need a switch because the routing changes with each clock cycle. Also, for a given code one can choose different acceptable M 1 and M 2 values and hence various levels of parallelization can be achieved using the same LDPC code giving scalability to the architecture. Since the same parity check matrix can be used for various levels of parallelization there will be no difference in bit error performance of the code between the implementations. For the sake of completeness of this paper, Fig. 4 taken from [1] compares the bit error rate performance of a rate 1/2 code of length 3072 with a randomly constructed code without cycles of length 4. It can be seen that the MLLCS code has no noticeable loss compared to the random code. In the architecture in [5] each level of parallelization needs a new parity check matrix. If the memory addresses are to be generated on the fly the address generation module must be run at a much higher clock frequency compared to the decoder since M 1 addresses need to be generated for each clock cycle using the MLLCS equation. In general, for a decoder operating with codes of different lengths and rates the addresses for the selected code can be generated before the decoding starts and stored in a memory. The check node update units can be built in a flexible way to support codes of different rate.
A. Decoding sequence
The sequence of operations for the LDPC decoder is summarized below.
1) Using a counter to generate addresses, read M 2 × M 1 extrinsic messages in parallel and complete M 2×M 1/ρ check node updates and write back the check to bit messages into the extrinsic memory. This process takes a total of M/(M 1 × M 2) clock cycles.
2) Read M 1 extrinsic messages one from each column using the address generated from the MLLCS sequence. Compute the M 1/3 bit node updates and write back the bit to check messages into the extrinsic memory. This process takes a total of N/(M 1/3) clock cycles. The total time for a decoding iteration is (M/M 1 × M 2) + (N/(M 1/3)) clock cycles.

VI. CONCLUSION
Joint code-decoder design and partly parallel implementation are increasingly used in the state of art LDPC decoder implementations. In this paper we have shown that our previous work namely MLLCS-LDPC codes is suitable for joint code-decoder design. We have also proved that there is a method to divide the extrinsic messages into subgroups to facilitate a partly parallel decoder implementation. Various levels of parallelization can be achieved giving scalability to the architecture. Since the same parity check matrix is used for various levels of parallelization there is no difference in bit error performance.
