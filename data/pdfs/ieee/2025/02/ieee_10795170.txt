Soft-Output Guessing Codeword Decoding

Ken R. Duffy , Senior Member, IEEE, Peihong Yuan , Member, IEEE, Joseph Griffin , Student Member, IEEE, and Muriel Médard , Fellow, IEEE

Abstract— We establish that it is possible to extract accurate blockwise and bitwise soft output (SO) from Guessing Codeword Decoding (GCD) with minimal additional computational complexity by considering it through the lens of Guessing Random Additive Noise Decoding (GRAND). Blockwise SO can be used to control decoding misdetection rate, while bitwise SO results in a soft-input soft-output (SISO) decoder that can be used for efficient iterative decoding of long, high redundancy codes.
Index Terms— GCD, GRAND, soft output.

I. INTRODUCTION

F OR any soft input (SI) forward error correction decoder, it would be highly desirable if it could produce accurate blockwise and bitwise soft output (SO). Blockwise SO, the a posteriori likelihood that the decoding is correct, can be used to inform retransmission requests or to label untrustworthy blocks for an erasure correction code to rectify [1], while bitwise SO can be used for efficient, iterative soft input soft output (SISO) decoding of long, high redundancy codes such as product codes [2], LDPC codes [3], and GLDPC codes [4]. Most error correction decoding algorithms, however, cannot generate accurate SO for reasons articulated in seminal work of Forney [5].
Consider a codebook, C, consisting of 2k binary codewords of length n, where transmitted codewords are selected uniformly at random. Guessing Codeword Decoding (GCD) [6], [7], which can decode any binary linear code, generates a list of candidate codewords and their likelihoods as part of its execution. It may be tempting to use Forney’s approximation to generate SO from that list, but we shall demonstrate that it is inaccurate.
In the context of SI Guessing Random Additive Noise Decoding (GRAND) algorithms, e.g. [8], [9], [10], and [11], we have developed a method for extracting accurate SO that obviates the need for a decoding list and can, distinctively, provide SO with a single decoding [12], [13]. Central to the approach is an estimate of the term in eq. (2) that is assumed to be 0 in the Forney and Pyndiah approximations. This estimate is evaluated during GRAND’s execution with minimal additional computation. Building on the approach, we have established that accurate SO can also be produced by all successive cancellation (SC)-based decoders of polar-like codes [14].
By considering GCD [6], [7] through the lens of GRAND, we establish that it is possible to extract accurate blockwise and bitwise SO with minimal additional computation. In Section II we introduce SI GRAND algorithms and how SO is extracted from them. In III we show how SO can be generated during GCD’s execution. Section IV establishes the accuracy of the blockwise and bitwise SO, while Section V provide SISO performance results as an illustration of the method’s utility.

III. GCD AND SO
While GRAND algorithms can decode any code of any structure, including non-linear ones created with length constraints and cryptographic functions [20], [21], [22], proposals have been made to exploit the structure of binary linear codes to influence the query order and reduce total query numbers, e.g. [23], [24], and [25]. For example, for even linear block codes [1], which includes eBCH codes, polar codes, and many others, the parity of the noise effect pattern being sought is the same as the parity of the hard demodulated signal. Armed with a noise effect generator that can skip given Hamming weights,

While not originally described in those terms, GCD [6], [7] can be thought of as a way to use the structure of the codebook to make 2n−k noise effect guesses at a time. Not all of the queries will have high likelihood, but one will necessarily result in a codeword and 2n−k − 1 of them will not. If a GRAND algorithm does not avail of codebook structure to reduce query numbers for a binary linear code, it is proven in [6] that the maximum likelihood decoding is identified in fewer queries by GCD. When, for example, decoding an even binary linear code with a GRAND variant that avails of that structure that is not necessarily the case.
GCD is most readily understood for systematic binary linear codes where the first k bits are information bits, and yk denotes the systematic demodulated bits. Noise effects are sequentially queried from most likely to least likely on the systematic bits, zk,1, zk,2, . . . alone and then extended via the generator matrix, G, to create a codeword (yk ⊕ zk,i)G = cn,i. No guarantees are made about the likelihoods of the individual cn,i, but, as originally observed in [27], the sequence of noise effect queries can cease when

With an (n, k) component code, k2 information bits are written into a k × k square matrix, each of the k rows are encoded, and each of the resulting n columns are encoded, resulting in an (n2, k2) code.
In 1998, Pyndiah proposed iterative SISO decoding of product codes, also known as turbo product codes, [2]. For the original decoder, the component code was required to have an efficient hard detection decoder. Recently, the performance of product codes with polar component codes have been considered with successive cancellation list decoding using Pyndiah’s soft update [33], [34] or GRAND-like SO [14], which gives better performance. SI GRAND algorithms have also been applied with Pyndiah’s soft update solely as true list decoders [18], [19] or, more recently, with GRAND SO [13], which results in significantly enhanced performance. As GRAND can decode any moderate redundancy component code, they open up a broad palette of product code constructions.
Here we consider block turbo decoding with SO-GCDORB. As with GRAND, GCD can decode any linear component code and, armed with accurate SO from considering it through the lens of GRAND, it can be used for iterative decoding. Block turbo decoding of product codes works as follows:
0 Channel LLRs are stored in an n × n matrix LCh. The a priori LLRs of coded bits are initialized to zero, LA = 0.
1 Each row of LCh + LA is processed by a SISO decoder, and the resulting APP LLRs and extrinsic LLRs are stored in the rows of LAPP and LE, respectively. A hard decision is from LAPP. If all rows and columns of the decision correspond to valid codewords, the block turbo decoder returns the hard output. Otherwise, LA is set to αLE, for some α > 0, and the decoder proceeds to the column update.

Fig. 5. SO-predicted BER vs. empirical BER: eBCH codes, SO-GCD-ORB, Eb/N0 = 3 dB.
2 Each column of LCh + LA is decoded and the columns of LAPP and LE are updated as in step 1. A hard decision is performed using LAPP. If the obtained binary matrix is valid, decoding success is declared. If the maximum iteration count is reached, a decoding failure is returned. Otherwise, LA = αLE and proceed to the next iteration (i.e., return to step 1).
In simulation, we set α = 0.5, and use SO-GCD-ORB as the SISO decoder with eq. (8). In practice, rows are decodable in parallel, as are columns, resulting in desirable low-latency.

VI. DISCUSSION
SI error correction decoders that generate accurate blockwise or bitwise SO provide significant extra functionality. Blockwise SO can be used to accurately trigger ARQ requests or flag data for erasure. Bitwise SO can be used in SISO decoders to efficiently decode long, high redundancy codes.

By considering GCD as a sophisticated version of GRAND that makes parallelized queries, we establish that SO can be produced with it too. The accuracy of that SO is confirmed, as well as a demonstration of the powerful applications that result.
