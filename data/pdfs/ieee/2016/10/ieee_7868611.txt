Abstract—Low-Density Parity-Check (LDPC) codes belong to a class of linear block codes that can approach the Shannon limit. However, due to the high decoding complexity, LDPC codes with extremely long block lengths will result in large decoding latency in practical applications. Recently, Cloud Radio Access Networks (C-RANs) have attracted much attention because of their innovative architecture, which involves embedding cloud computing capability into wireless networks. In this context, C-RANs provide an ideal platform for breaking the decoding bottleneck of LDPC codes with extremely long block lengths. However, given that there are currently as yet no specific cloud computing techniques proposed for C-RAN, the data processing architecture for LDPC decoding is still an open problem. In this paper, based on the Spark cloud computing platform, a data processing architecture is proposed for realtime decoding of extremely long LDPC codes. Based on the principle behind the belief propagation (BP) decoding algorithm, a novel distributed BP decoding algorithm is proposed and implemented on a Spark based cloud computing platform. Extensive experimentation illustrates that the proposed architecture enjoys a significant performance gain in decoding latency. Therefore, this work can be seen as a promising example of applying powerful cloud computing techniques to enhance the performance of wireless networks.
I. INTRODUCTION
Shannon proved that there exist block codes that achieve arbitrarily small error probability at any rate up to the capacity of a given memoryless channel [1]. Low-Density Parity-Check (LDPC) codes, first proposed in 1963 by Gallager [2], are a class of block codes that can approach Shannon capacity. Since MacKay [3] and Sipser [4] rediscovered LDPC codes, they have attracted much attention from both industry and academia. In particular, it was shown that the LDPC codes can achieve Shannon capacity within 0.04 dB at a bit error rate of 10−6 using a block length of 107 [5].
Serving as efficient error correction codes, LDPC codes have been adopted in DVB-S2 [6], WiMax, and Wi-Fi standards [7]. Moreover, LDPC codes are also a popular candidate for 5G systems [8]. The belief propagation (BP) [9][10] algorithm, which works on a factor graph representation of LDPC codes, is a well-known soft decision decoding method. Recently, the Cloud Radio Access Network (C-RAN) architecture has drawn much attention because of its innovative approach of embedding cloud computing capability into wireless networks [12][13]. This architecture makes it possible for realtime decoding of LDPC codes with extremely long block lengths. However, there lacks an efficient cloud computing architecture for decoding tasks in the C-RAN. In the past decade, as one of the most popular cloud computing platforms, Apache Hadoop [14] and Spark [15] have emerged to manage massive data and run analytical jobs on large clusters with many worker nodes. The Hadoop platform has been a leading open source big data processing framework for many years. But recently the more advanced Spark platform, which is built on the Hadoop Distributed File System (HDFS), runs 100 times faster than the Hadoop platform from memory and 10 times faster on disk [16].
In this paper, we propose a data processing architecture for realtime decoding of extremely long LDPC codes, referred to as the Spark-CRAN, by combining the advantages of the Spark platform in computation with those of the C-RAN in communication. In order to verify the decoding efficiency of the Spark-CRAN, a novel distributed BP decoding algorithm is proposed and implemented on our Spark based cloud computing platform. Results obtained from extensive experimentation show that the running time of the proposed algorithm on our platform is reduced by 47% compared to traditional methods when the block length n is equal to 90, 000. The proposed Spark-CRAN architecture is also scalable. Therefore, the proposed Spark-CRAN is a prototype for applying powerful cloud computing techniques to enhance the performance of wireless networks.

A. The BP Decoding Algorithm for LDPC Codes
There are three typical methods for decoding LDPC codes, namely soft decision, hard decision, and hybrid decision decoding. The BP algorithm, based on soft decision, is one of the most powerful decoding methods for LDPC codes. The basic BP decoding algorithm for LDPC codes is listed in Algorithm 1.
LDPC codes can be described by a sparse parity-check matrix Fm×n. The corresponding factor graph G consists of the variable node set v = {v1, . . . , vi, . . . , vn} and the factor node set f = {f1, . . . , fj, . . . , fm}. The edges in the adjacency list E = {(fj , vi)}i∈{1,...,n},j∈{1,...,m} connect the two sets of nodes. The number of edges is denoted by e. Let the transmitted code vector be c = [c1, . . . , cn], and the received code vector be y = [y1, . . . , yn]. The posterior probability of c is denoted by P = [P1, . . . , Pn], where Pi = Pr{ci = 1|yi}. The state of ci is denoted by s, where s ∈ {0, 1}. The algorithm is an alternately iterative process. To calculate the message passed from fj to vi, the factor node fj processes the messages that are received from its neighboring variable nodes vj, and then produces a new message rji(s). Similarly, to calculate the message passed from vi to fj, the variable node vi processes the messages that are received from its neighboring factor nodes fi, and then combines its own value Pi to produce a new message qij(s). This iterative process continues until the messages converge. Then the belief Qi(s) of vi can be obtained, and we get the estimated code vector ˆc, where cˆ = [cˆ1, . . . , cˆn]. In Algorithm 1, the normalized parameters Kij and Ki are chosen to ensure that qij(0) + qij(1) = 1 and Qi(0) + Qi(1) = 1, respectively. The current iteration number is denoted by t, and the maximum number of iterations is denoted by tmax.
B. Spark-CRAN
As shown in Fig. 1, the C-RAN architecture consists of remote radio heads (RRHs), baseband units (BBUs), and the optical frounthaul/backhaul network. An RRH, consisting of a co-operative radio and distributed antennas, provides radio access capability. A BBU pool, consisting of numerous high performance computation units, is applied to carry out cloud computing on all of the baseband data. The optical transport network provides high bandwidth and low latency. The objective of the whole system is to reduce power consumption and improve infrastructure utilization by coordinating the three components.
It can be seen that the cloud computing capability in the BBU pool is key to the performance of a C-RAN. The big data processing platform of Spark provides a potential solution for the BBU pool. The Spark platform uses the HDFS to store large-scale data. The HDFS, storing and managing big files or other large datasets, provides a highly fault-tolerant and scalable storage mechanism by distributing a replicate file to multiple worker nodes in a cluster. There are three ways to create an RDD, namely created from a file, created by the parallelize function of the Spark platform, and created by transforming an existing RDD. An RDD, a collection of partitioned data across the worker nodes, can be rebuilt when a partition is lost. This rebuilding mechanism guarantees fault-tolerance. Two classes of operations on the RDDs are transformations and actions. The operations of transformations record how the RDDs transform and where they are derived from. These operations do not execute immediately until the operations of actions are carried out. The fact that a machine needs to store and handle some partitions of the RDD in itself, ensures the parallel nature of computation. Two shared variables, namely the broadcast variables and the accumulators, are provided to reduce communication overhead by copying data to each machine and caching them in each memory unit. The Spark platform runs in two modes, namely a local mode running tasks locally and a cluster mode running tasks in a distributed cluster.
By embedding the Spark platform into the BBU pool, we propose a data processing architecture for realtime decoding of extremely long LDPC codes, referred to as Spark-CRAN. Combining the high spectral efficiency of the RRHs in communication and high performance of the Spark platform in computation, a large multiplier effect will be achieved.

III. A DISTRIBUTED BP DECODING ALGORITHM FOR EXTREMELY LONG LDPC CODES IN SPARK-CRAN
In this section, we investigate the distributed BP decoding algorithm for extremely long LDPC codes. The implementation issues in the Spark based cloud computing platform are also considered.
In this paper, we are interested in the reducing the LDPC decoding latency. The main computational overhead in each iteration comes from the following two calculations. One is the calculation of messages from factor nodes to variable nodes in step 4 of Algorithm 1. The other one is the calculation of messages from variable nodes to factor nodes in step 7 of Algorithm 1. The key to designing distributed decoding algorithms is that there should be no relation between any two calculations within for loops. The flow diagram of the distributed algorithm is presented in Fig. 2. We convert the data input into RDDs and get the initialized messages qij(s). To simplify the discussion, we demonstrate the calculations of messages passed from factor nodes to variable nodes. In each iteration, the cluster manager distributes the intermediate messages qij(s) to all worker nodes. Then the parallel computations are executed in all worker nodes. Finally, the cluster manager collects all the intermediate messages rji(s) to obtain the global information over all partitions.
We characterize LDPC codes by an adjacency list E due to the fact that the LDPC parity-check matrix F is very sparse in practice. To reduce the communication and computation overhead, the messages of state 0 in step 4 of Algorithm 1 are stored and computed in the distributed algorithm. To calculate rji(0), the messages from vk ∈ vj \ {vi} to fj are needed.

However, it is worth pointing out that in the Spark platform each operation corresponds to one element of an RDD with map or one partition of an RDD datasets with mapPartition, where map and mapPartition are transformations in the Spark platform. Therefore, the information of fj and vj should be together as an element of the RDD. We transform E = {(fj , vi)}i∈{1,...,n},j∈{1,...,m} to {(fj , vj )}j∈{1,...,m} as an RDD.
In a Spark cluster, since the number of worker nodes is generally far smaller than the number of edges, parallel computation can be realized in two modes. In the fullparallelization mode, a calculating unit corresponds to one edge. While in the half-parallelization mode, a calculating unit corresponds to one factor node. In each calculating unit one intermediate variable is created. Since the number of factor nodes m and variable nodes n is far smaller than the number of edges e, to improve the efficiency we choose the halfparallelization mode in our distributed algorithm by reducing the intermediate variables. Though the parallelization of the Spark platform relies on the mechanism of data serialized in an RDD, the Spark platform does not allow any single operation to gather information from two different RDDs. IV. EXPERIMENTS AND ANALYSIS
In this section, we test the performance of the Spark based architecture for decoding LDPC codes with extremely long block lengths. A photo of the hardware platform is shown in Fig. 3, where the configurations of the hardware platform are shown in Table I. The Spark cluster is connected by a 100/1000Mbps switch, and the operating system of each node is Ubuntu 14.10. The Algorithms 2-6 are implemented in Python due to its popularity in the scientific community with a clean and simple syntax. The server is set as the Spark cluster manager. The number of partitions is set as 14. The datasets of the LDPC codes are shown in Table II. For convenience, we use the abbreviations TBPSM, TBPSV, DBPSL, and DBPSC to denote the traditional BP decoding algorithm in the single machine, the traditional BP decoding algorithm in the server, the distributed BP decoding algorithm in the Spark local mode, and the distributed BP decoding algorithm in the Spark cluster mode, respectively.
To verify the high performance of the proposed architecture, the running time is tested among TBPSM, TBPSV, DBPSL, and DBPSC with different block lengths of LDPC codes. In Fig. 4, we compare the running time of TBPSM and TBPSV. It can be seen that TBPSV outperforms TBPSM for any block length. The reason is that the hardware configuration of the server is higher than the single machine in CPU and memory resources. For example, the running time of the TBPSV is

reduced by 84% compared to the TBPSM with the block length n equal to 36, 000.
In Fig. 5, we compare the running time of TBPSV and DBPSL. DBPSL outperforms TBPSV in the identical server as the block length of LDPC codes increases. For example, the running time of DBPSL is reduced by 47% compared to TBPSV when the block length n is equal to 90, 000. However, TBPSV is better than DBPSL when the block length n is less than 40, 000. The reason is that the Spark platform uses multiple cores to enhance the computing performance, while the traditional algorithm uses a single CPU core. But the multicore communications are not enough to compensate for the calculation upgrade when the block length is short.
