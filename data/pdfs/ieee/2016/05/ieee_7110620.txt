High-Throughput Multi-Core LDPC Decoders Based on x86 Processor

Bertrand Le Gal and Christophe Jego

Abstract—Low-Density Parity-Check (LDPC) codes are an efficient way to correct transmission errors in digital communication systems. Although initially targeting strictly to ASICs due to computation complexity, LDPC decoders have been recently ported to multicore and many-core systems. Most works focused on taking advantage of GPU devices. In this paper, we propose an alternative solution based on a layered OMS/NMS LDPC decoding algorithm that can be efficiently implemented on a multi-core device using Single Instruction Multiple Data (SIMD) and Single Program Multiple Data (SPMD) programming models. Several experimentations were performed on a x86 processor target. Throughputs up to 170 Mbps were achieved on a single core of an INTEL Core i7 processor when executing 20 layered-based decoding iterations. Throughputs reaches up to 560 Mbps on four INTEL Core-i7 cores. Experimentation results show that the proposed implementations achieved similar BER correction performance than previous works. Moreover, much higher throughputs have been achieved by comparison with all previous GPU and CPU works. They range from x1.4 to x8 by comparison with recent GPU works.
Index Terms—LDPC decoding, software decoders, SIMD architecture, x86 processor, GPU, throughput, software define radio (SDR)

1 INTRODUCTION

LOW-DENSITY Parity-Check codes [1] have become one of the most popular forward error correction code family that can achieve performance close to the channel capacity on a variety of transmission channel models. They were adopted in digital communication standards: WiFi (IEEE 802.11n), WiMax (IEEE 802.16e) and Digital Video Broadcasting (e.g. DVB-SX2 released in 2014). However, the drawback of high correcting efficiency comes from its decoding computation complexity. However, in recent years, many researchers have proposed some be implemented of LDPC decoders in software on multi-core or many-core devices to meet for instance, Software Defined Radio (SDR) flexibility requirements.
Indeed, during the last decade, a modification of the computation models of software programmed devices was observed. Progress in terms of performance previously obtained by increasing clock frequencies has reached a limit. To hold this increase in performance, other techniques based on parallel processing were investigated e.g. Single Instruction Multiple Data (SIMD), Single Program Multiple Data (SPMD) and Single Instruction Multiple Threads (SIMT). These techniques are widely integrated in current processor and GPU devices. Mapping a real application onto these targets is challenging and constraints the algorithm selection.
Pure software decoders for LDPC codes was widely studied during the last five years on multi/many-core devices due to its inherent parallel computations. However,

researchers have first mainly focused on a flooding-based decoding algorithm that is not the most efficient one from the decoding performance point of view. Second, most efforts were spend on GPU devices that is not the most efficient target for LDPC decoding.
The main contribution of our work is the following: we propose a way to efficiently implement a software LDPC decoder using both SIMD and multi-core programming models. Proposed SSE/AVX2 [2] decoders enable the support of any kind of LDPC codes and provides high throughput performance. A large set of experiments were performed to characterize the LDPC decoders and compare them to previous works. Speedup factors reach from Â5 up to Â12 were obtained compared to recent x86 CPU decoders and speedup factors from Â1:4 up to Â8 were achieved by comparison with to latests GPU works. High performance comes from the decoding algorithm choice, the H matrix transformations, the algorithmic source code optimization and the highly efficient implementation of CN computation kernels.
The rest of the paper is organized as follow. Section 2 details the different algorithms and schedules proposed in the literature to decode LDPC codes. Section 3 introduces an exhaustive list of the related works. The mapping of the layered-based algorithm and the proposed optimizations are detailed in Section 4. Finally, the decoder efficiency is validated in Section 5.

Many authors provide interesting information concerning the performance bottlenecks. For example, Wang et al. [15] has demonstrated that the bottleneck of the LDPC decoder on GPU is due to the device slow memory accesses. Thus, many authors used LDPC complex decoding variants (e.g. SPA) to hide memory access latencies. However from a functional point of view the correction performance is similar.
The previous works have only focused on mapping and optimizing the flooding-based scheduling on multi-core or many-core devices. Throughputs achieved by the proposed approaches are under real-time requirements for a sufficient set of decoding iterations (!30 iterations). Moreover, depending on the structure of the LDPC code (regular or irregular, QC or unstructured), throughputs achievable are highly impacted.
For these reasons, we have considered the Horizontal TDMP algorithm. Proposed approach that explores interframe parallelism provides flexibility and quite constant throughput level independently of the LDPC code. These claims and the decoder high-efficiencies will be discussed in the experiment section.

Many recent works have focused on mapping LDPC decoders onto multi-core and many-core devices. Work objectives were to achieve high or real-time decoding throughputs for different kinds of LDPC codes: short or long frame lengths, regular or irregular codes, etc. However, for some of those, high throughput performance was obtained on expensive GPU devices such as NVIDIA Tesla C2050 [11], NVIDIA Titan [21] or ATI HD5870 [18]. To achieve high-throughput performance, related works have focused on improving the mapping of such decoding algorithm onto devices using different optimization schemes such as data packing methods, memory coalescing techniques [12] or decoder stopping criteria [19].

Frames are composed of n LLR informations coded using a 8-bit fixed-point representation. SSE instructions such as load and store require 128-bit data width. 8-bit data from the 16 codewords are interleaved to produce a 128-bit aligned memory structure. The 16 first values of each frame are first stored, followed by the 16 next values and so on. Efficient data access during the decoding process is thus obtained. However, after the decoding, decisions have to be also reordered thanks to an inverse transformation. The impact of both transpositions is limited due to the fact that (i) they take advantage of SIMD instructions and (ii) the reordering times are independent of the number of decoding iterations. If many iterations are performed, then the reordering durations become negligible.

5 EXPERIMENTS
A complete evaluation of the proposed parallel processing LDPC decoders1 was done. This section that details the experiments is composed of two main parts. The first one presents the x86 target experiments performed under various configurations (#iterations, #processor cores, SIMD configuration). The second one gives some performance comparisons between the proposed x86 LDPC decoders and ones from related works (CPU and GPU devices).
The evaluation platform employed was a MacBook Pro computer that runs OS X 10.10. It is composed of an INTEL Haswell Core-i7 4960HQ CPU. This processor is running at 2:6 GHz, with 6 MB of L3 cache memory and 16 GB of DDR3 running at 1,600 MHz. It is composed of 4 cores that contains each 256 KB of unified L2 cache. L3 cache memory is shared between the four cores. Turbo-boost technology was switched on. Thus the working clock frequency of the processor reaches 3.6 GHz when a single processor core was used and 3.4 GHz when four cores were switched on.
For all the experiments, a custom framework was used to generate the optimized C source codes (e.g. the constant definition files) according to the corresponding H matrix. These codes were then compiled with the MacOS version of INTEL C++ compiler 14 toolchain. The following compilation flags were provided to the toolchain: -march=native -fast -fopenmp -opt-prefetch -unroll-aggres-
sive -xCORE-AVX2 -fma. To reduce as far as possible the OS impact during the performance measurement, we first disabled and closed all possible programs and services. Moreover, the x86 LDPC decoders were launched as administrator with the highest task priority (nice -n -20).
The air throughputs of the decoder called G provided in this section are in Mbps. G corresponds to the throughput of the decoder when it executes the complete decoding pro-

set of decoded frames in order to avoid spurious peak measures. Time measurements used to compute G were first obtained with the get_clocktime API function and confirmed by using the Chrono API from Boost library.
Prior to these G performance measurements, the behaviour of the LDPC decoders were evaluated. Similar to what was implemented in [7], [33], random information were generated (MKL API), mapped (BPSK) and added to an emulated AWGN channel. BER performance curves for different LDPC codes are provided in Fig. 5. These curves compare performance of our decoders with ones reported in the related works (for ones that provides Eb/N0 curves with BPSK modulation). The number of decoding iterations is divided by two for our decoder when compared to works using a flooding scheduling, identical otherwise. Concerning the decoding variant, we used alternatively the NMS/ OMS decoder releases depending on the BER performance. G of both OMS and NMS decoders are provided in the follow section. Finally, in our case, all LDPC decoders were fixed-point implementations, quantization format is 6 bits for channel values and 8 bits inside the LDPC decoder. The fractional part is stored on 2 bits. Data format were not optimized or tweaked using approach like [32]. All BER points were obtained with at least 100 frame errors.
This BER curves demonstrate that the implemented x86 LDPC decoder using 8 b fixed-point arithmetic provides performance often better than related works if an equivalent amount of decoding iterations is performed. This efficient performance of fixed-point decoder compared to some floating-point ones mainly comes from the fact that OMS/NMS approaches were preferred to MS one. In conclusion, one can be seen that proposed decoders: (i) efficiently decode LDPC frames; (ii) provides performance similar to related works independently of the decoding algorithm and the data format.
