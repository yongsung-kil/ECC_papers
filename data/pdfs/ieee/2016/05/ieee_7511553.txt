Fully Parallel Window Decoder Architecture
for Spatially-Coupled LDPC Codes
Najeeb Ul Hassan, Martin Schl¨uter, and Gerhard P. Fettweis
Vodafone Chair Mobile Communications Systems, Dresden University of Technology (TU Dresden), Dresden, Germany,

Abstract—Spatially-coupled
low-density
parity-check
(SC-
LDPC) codes have been shown to be superior in performance
than LDPC block codes. In order to comply with the practical
constraints on latency, SC-LDPC codes are decoded using a
window decoder that reduces the decoder latency and complexity
compared to traditional block-wise decoding. However, so far
the literature only considers the structural decoding latency of
window decoder, ignoring the processing latency. Note that the
processing latency directly impacts the decoder’s throughput and
is an important parameter in any modern communication system.
The throughput of an iterative decoder is directly influenced by
the number of iterations and hence in this paper we propose a
fully parallel window decoder architecture for SC-LDPC codes
where the decoding iterations are performed in parallel. This
guarantees a high throughout while fulfilling the low latency
requirements. The overall decoding latency (structural and pro-
cessing latency) of the proposed window decoder architecture is
compared with the classical window decoder.
I. INTRODUCTION
Gallager introduced a class of linear codes which are
described by a sparse parity-check matrix [1] referred as low-
density parity-check (LDPC) codes. Due to the sparsity of the
parity-check matrices, these can be decoded using a simple
and practical iterative decoder. Hence as a result, in general,
the decoding complexity per decoded bit is independent of the
length of the code when the number of decoding iterations
is fixed. Thanks to this property of LDPC codes, various
standards that deal with high data rate, e.g., DVB, WiMAX,
WiGig, propose LDPC codes which allow them to operate
close to the Shannon limit using long block lengths.
Spatially-coupled LDPC (SC-LDPC) codes were first in-
vented by Jimenez Felstr¨om and Zigangirov in [2]. Since
they are characterized by a semi-infinite diagonal type parity-
check matrix, they were referred to as LDPC convolutional
codes. Like LDPC block codes they are defined by a sparse
parity-check matrix, which allows decoding with belief prop-
agation algorithms. The difference between SC-LDPC and
LDPC block codes is the memory. For LDPC block codes,
codewords of different time instants are encoded and decoded
independently of each other. A codeword (v[1,L]) of a ter-
minated SC-LDPC code consists of L coupled code blocks
vt, t = 1, . . . , L. The original block-wise decoding approach
waits until all L blocks are received and then performs belief
propagation. In [2] and [3] a pipelined block-wise decoder

is considered. For a large L, SC-LDPC codes are known to
achieve superior bit error rate (BER) performance compared to
LDPC block codes. However, the drawback of having a large
L is the increase in latency and decoding complexity [4] [5].
Hence, the pipeline decoders presented in [2] [3] have large
initial delay, e.g., equal to the length of the coupled code L.
Another approach is a window decoder of size W. The
window decoder operates on W ≪L code blocks, and the
resulting decoding latency and decoding complexity becomes
independent of L. It has been shown in [4] and [6] that binary
and non-binary SC-LDPC codes decoded with the window
decoder outperform their counterpart LDPC block codes under
the same latency, respectively. However, the comparison was
solely based on the structural latency of the codes. The
structural latency depends on the structure of the code and
reflects the fact that the mapping operation at the decoder
can be done once a certain number of symbols (or bits) are
available at the input [7]. Hence decoder processing latency
for LDPC block codes and SC-LDPC codes decoded using the
window decoder was assumed to be equal. This assumption
was based on the following argument; since the number of
symbols and the iterative message passing algorithm in two
decoders is the same, the processing latency of the LDPC
block decoder and window decoder is equal. Note that this
argument inherently assumes that the implementation of LDPC
block decoder and the window decoder can be done in the
same way.
In an iterative decoder, the processing latency depends on
the number of iterations. In case of block codes, i.e, when
blocks are encoded and decoded independently, the processing
latency can be reduced by parallelizing the iterations or also
known as iteration unrolling. In [8], an ultra-high speed
LDPC decoder architecture has been presented achieving a
throughput up to 160 Gbit/s based on this principle. Note
that the pipeline decoder for SC-LDPC codes proposed in [3],
reduces to [8] when applied to block codes. However, due
to the memory read access required in the window decoder,
these architectures are not suitable and up to now, no such
architecture has been proposed for the window decoder. We
propose a way to adapt the high throughput architecture from
[8] and design a modified window decoder in Section IV
that performs all iterations in parallel and approaches the
performance of the classical window decoder. We show the
challenges in unrolling the iterations of the window decoder.
The BER results are shown in Section V.

A. Block-wise Decoding
One of the benefits of having low density of ones in the
parity-check matrix is that LDPC codes can be decoded using
an iterative message passing decoder. The decoding complex-
ity increases with the density of ones in H, hence a low density
is desirable. In case of AWGN channel, messages in the form
of log-likelihood ratios (LLR) are exchanged between the
connected check and variable nodes in every iteration. Since
the absolute value of an LLR represents the strength (belief)
of the message, this message passing algorithm is known as
belief propagation (BP).
The decoding for an SC-LDPC code can be performed by
running BP algorithm over the complete chain of L coupled
blocks v[1,L]. Since the coupled code is treated in this case
as a block code, we refer to such a decoder as a block-wise
decoder. However, running BP algorithm over the complete
chain of L coupled blocks v[1,L] is not optimum in terms of
decoding latency and complexity.
B. Windowed Decoding
The convolutional structure of SC-LDPC code allows to
define a latency constrained window decoder of size W [9].
Consider the blocks vt and vt′ in the chain of coupled
code, where t′ ≥t + mcc + 1 and t, t′ ∈[1 L]. Since
t′ ≥t + mcc + 1, vt and vt′ do not share any parity-check
equation (check node). Window decoder exploits this property
of the convolutional parity-check matrix of SC-LDPC code to
define a decoder consisting of W received blocks y[t,t+W −1]
such that W ≥mcc + 1. In order to decode symbols in yt,
the window consists of W received blocks, y[t,t+W −1] and
mcc previously decoded blocks in memory of decoder. This is
shown in Fig. 1 as a shift register of size W code blocks for
window that holds the channel values of y[t,t+W −1]. Similarly
a shift register of size mcc code blocks holds the output LLRs
corresponding to the decoded code blocks y[t−mcc,t−1] in the
memory.
In terms of parity-check matrix, the window decoder of size
W operates on a section of MW = W · (c −b) rows (check
nodes) and NW = W ·c columns (variable nodes) of the matrix
H[1,L], corresponding to W coupled blocks.

The window decoder provides a way to increase the termi-
nation length L arbitrarily large and is suitable for continuous
decoding of input blocks. Furthermore, since message passing
algorithm is applied only to W received blocks, the memory
and hardware requirements for such a decoder are far less
compared to block-wise decoding of SC-LDPC codes. In
general, the storage requirements for the decoder reduces by
a factor of L/W compared to the block-wise decoder.
IV. FULLY PARALLEL WINDOW DECODER
In the decoder architecture presented in Fig. 1, a new
code block enters from right only after Imax iterations are
performed. Assuming that Tp seconds are required to perform
a single decoding iteration (updating all NW variable and
MW check nodes in a window), then the decoder generates
an output every TpImax seconds. Note that the update of all
NW variable and MW check nodes are done in parallel. In
terms of processing latency, the critical path1 of the decoder
depends on the number of iterations Imax. One way to reduce
the decoding processing latency or consequently increasing
the decoding throughput is to parallelize these Imax iterations.
This will reduce the critical path to 1 iteration.
Considering block codes, i.e., each code block is encoded
and decoded independently, these Imax iterations can be un-
rolled (pipelined) to obtain a fully parallel decoder. Hence
a code block enters and leaves the BP decoder every Tp
seconds2. This increases the decoder throughput by a factor
of ≈Imax. Such a decoder architecture to decode LDPC
block codes was proposed in [8]. The architecture, however,
requires that all the code blocks are encoded and decoded
independently and does not require a read access to the
memory. In the following we propose an unrolled fully parallel
window decoder architecture.
A. Unrolling the Iterations of Window Decoder
Now considering the window decoder presented in Sec-
tion III, we aim at proposing an unrolled architecture for the
window decoder in order to parallelize the decoding iterations.
As a result, the decoder generates an output code block every
Tp seconds.

[8] is the memory of the SC-LDPC code. As described in

B. Circular Pipeline Window Decoder
As explained in Remark 2, the newly computed results are
not used in the following windows when the iterations are
unrolled. To explain this further, let us take an example of
a window decoder of size W = 6 performing Imax = 3
iterations on a SC-LDPC code with mcc = 2.
Figure 5 shows a snapshot of the code blocks being pro-
cessed within all pipeline stages at time t and t + Tp. In the
first pipeline stage I1, symbols in yt and yt+1 are the target
symbols at time t and t + Tp, respectively. The symbols in
yt are also the target symbols in the second pipeline stage at
time t + Tp, which takes all the computed messages for the
nodes within the window at the beginning of time t+Tp from
the first stage, i.e., Iin
2 = Iout
1
. Now at time t, Imax iterations
are processed on the code blocks y[t−2,t+3]. The output LLRs
corresponding to yt−2 are stored in the memory and the results
computed for code blocks y[t−1,t+3] are discarded. However,
in this example the results for code blocks y[t+1,t+3] can be
fed back to the first pipeline stage, making it a circular pipeline
window decoder.
As a result, the variable nodes in code blocks y[t+1,t+3] in
the first pipeline stage at time t + Tp are not initialized with
the channel values. And only the variable nodes in code blocks
y[t+4,t+6] are initialized with the channel values.

In this section, we compare the BER curves for the window
decoder from Section III and the two pipeline window decoder
architectures presented in Section IV. The time-invariant SC-
LDPC code has been generated following the method from
[10]. Note that in this work, we focus on the decoder archi-
tecture and do not aim to optimize the SC-LDPC code. A rate
1/2 SC-LDPC code with memory mcc = 23 is decoded with a
window size of W = 35. The window decoder has NW = 840
variable and MW = 420 check nodes. The decoder performs
Imax = 10 iterations.
Figure 6 shows the BER curves for the three window
decoder architectures. It is evident here that feeding back the
calculated results from last pipeline stage to the first pipeline
stage helps reduce the gap between the pipeline window
decoder and classical window decoder. The gap is reduced
to 0.1 dB at high SNR’s but for low SNR values, there is no
significant improvement in performance for circular pipeline
window decoder compared to the pipeline window decoder.
This is because at low SNR, the code blocks further away
from the target block show very little improvement in terms of
LLR values with iterations. This has been previously observed
in [5]. Hence the last stage is not able to provide much new
information to the first stage at low SNR.
VI. CONCLUSION
We propose a fully parallel window decoder architecture
for window decoder by pipelining (unrolling) the iterations.
Due to the memory read access required in the window
decoder, the already proposed fully parallel architectures are
not applicable. Compared to the classical window decoder,
negligible loss in performance was observed which is due to
utilizing the unfinished (intermediate) values in the memory.
The connections between the pipeline stages (input-output and
memory management) are also detailed. To the best of authors
knowledge, no such architecture for window decoder has been
proposed in the literature. We further compare the overall
decoding latency of the window decoder. With this architecture
now the assumption regarding the decoding processing latency,
detailed in Section I, is valid. Since, for both block decoder
and window decoder, iterations can be unrolled to increase the
decoding throughput by a factor of Imax. The results are also
verified using Verilog simulation, with FPGA implementation
left as a future work.
