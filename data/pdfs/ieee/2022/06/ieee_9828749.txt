Abstract—Since the low-density parity-check (LDPC) code is widely used in today’s wireless communication standards, its decoder architecture has been discussed in many works. To achieve high universality in hardware implementation, blockparallel is applied in the decoder for quasi-cyclic (QC) LDPC code. Configurability can be easily achieved by designing the operation instructions for blocks. The pipeline is also introduced as a traditional scheme for higher throughput. However, because of the data dependence in the decoding process, the pipeline would cause read-write conflicts, and some techniques should be utilized to solve them. Thanks to the block-parallel architecture, re-scheduling of the blocks can reduce the conflicts, but cannot guarantee the total avoidance of them. Inserting no-operation instructions (NOPs) is another solution that can remove all potential conflicts at the cost of lower throughput. In this paper, we combine these two techniques and propose an optimal algorithm for the instruction design aiming at the pipelined blockparallel decoder. It is proved that our algorithm can generate decoding instructions with no read-write conflict by inserting the minimum number of NOPs.
Index Terms—channel coding, low-density parity-check (LDPC) code, block-parallel, pipeline, block schedule.
I. INTRODUCTION
Due to its good performance, the low-density parity-check (LDPC) code has been widely used in many wireless communication standards such as the Fifth-Generation New Radio (5G NR) [1], IEEE 802.11 [2], and IEEE 802.16 [3]. The LDPC codes in these standards are all featured with the quasicyclic (QC) structure, which is beneficial to code description and codec design. The parity check matrix (PCM) of a QCLDPC code can be simply represented by a base graph. Each element in the base graph corresponds to a z × z circulant permutation matrix or a z × z all-zero matrix in the PCM, which is called a block.
Compared to the encoding part, the decoding part is more critical for performance and should be well-designed in both the decoding algorithm and decoder architecture. Layeredscheduling [4] is recommended for the decoding algorithm

which facilitates the extrinsic information transfer and improves the convergence speed compared to the traditional flooding-scheduling. As for the architecture, block-parallel design is preferred in recent works [5]- [7], especially when facing the LDPC code with highly flexible base graphs in 5G NR. It decouples the decoder architecture with the base graph, leading to higher universality in hardware implementation.
To achieve higher throughput, the pipeline is utilized but introduces a risk of read-write conflict. Thanks to the blockparallel architecture, the schedule of blocks can be designed to reduce conflicts. However, this method can not solve the problem completely and may fail in some extreme cases. A more simple solution is inserting no-operation instructions (NOPs), which is equivalent to stalling. The conflicts can be totally removed by inserting enough NOPs at the cost of lower throughput. Obviously, we want to design a set of decoding instructions that causes no conflict and has a small number of NOPs by combining these two techniques.
Although these techniques have been used in [5] and [7], neither a particular algorithm for designing the schedule nor a discussion on how to reduce the number of NOPs is presented. In [6], the conflict problem is solved by the modification of the decoding algorithm and decoder architecture. No NOP is needed in this modified design, but it suffers from performance degradation. Thus, in this paper, we focus on the original pipelined block-parallel decoder in [5] and [7], and propose an optimized algorithm to design the block schedule. The generated schedule is proved to be an optimal one, which indicates that it has the minimum number of NOPs under the premise of causing no conflict.
The rest of the paper is organized as follows. Section II gives a clearer description of the problem model. Under the model, the algorithm is proposed with the proof of its optimality in Section III. Section IV presents some results when applying our algorithm. Finally, Section V concludes the paper.

A. Graphical Description
The waveform graph is used in the previous works to give a more intuitive description of the scheduling problem [6] [7], as shown in Fig. 1. Each colored box represents a block with its column index written in it, and the blank box represents no operation. Note that the two-thread design is presented in Fig. 1, which is an essential feature of the pipelined block-parallel decoder [6]. Thread 2 always lags behind Thread 1 for each row.
For the block at Row i Column j, a read operation and a write operation of a-posteriori message Aj should be performed in the first stage of Thread 1 and the last stage of Thread 2, respectively. Thus, there is a risk of read-write conflict between different rows if they have the blocks with the same j. In Fig. 1, the conflicts occur on A1 and A4.
As mentioned before, designing the schedule of blocks, inserting NOPs, and the combination of these two methods can be used to solve the conflicts, as shown in Fig. 2. In Fig. 2, (c) avoids all conflicts, which is not achieved in (a), and has fewer inserted NOPs compared to (b). Thus, the combination of methods is preferred, and we tend to design a schedule causing no conflict with the minimum number of inserted NOPs.
The row schedule can also be re-designed and would influence the number of NOPs. However, it is proved that different row schedules can also result in different convergence speeds of the decoding algorithm [8]. A more complicated criterion combining the number of NOPs and the number of iterations should be utilized, which is beyond the scope of this paper. Thus, our discussion is under a natural or predetermined schedule of rows.

In general, the variables related to Thread 1 and 2 are notated with X and Y , respectively. Take Thread 1 as an example. IV. APPLICATION RESULTS
The proposed algorithm is universal for different LDPC codes, different row schedules, and different pipeline designs of the decoder, and gives an optimal design. Thus, it can provide a new evaluation indicator to code, schedule, and decoder design. The criterion can be the total number of instructions or the ratio of NOPs. Some results given by the algorithm with a variety of inputs are presented in the following.
Firstly, to show the validity of the algorithm, it is applied to a typical case. We use the whole Base Graph 1 (BG1) of 5G NR LDPC code with the code rate of 1/3. The row schedule is set from Row 1 to Row 46 of BG1. PX = 4, PY = 2, which is same as the reference design of the block-parallel decoder in [7]. The total number of instructions is 354, and the NOP ratio is 0.1073. A part of the schedule is shown in Fig. 5 by waveform graph. Compared to the most trivial case in which blocks are processed in the natural order with directly NOPinserting, as illustrated in Fig. 2 (b), our algorithm reduces the number of NOPs and the number of instructions by 68.6% and 19.0%, respectively. When PX = 5, PY = 3, the reductions come to 71.2% and 25.6%.
Considering different code rates, BG1 is cut to sub-matrices. We retain 22 information variable nodes but reduce the number of check nodes and extended variable nodes, leading to higher code rates. The instruction numbers and NOP ratios under different code rates are shown in Fig. 6. In general, when BG1 is cut, the number of instructions decreases along with the increment of NOP ratio. A reason for this result is the specially designed rows in BG1 featured with quasi-orthogonal. The risk of conflict is decreased by the design. Thus, the more complete these rows are, the smaller NOP ratio is achieved. A similar result can be obtained when facing Base Graph 2 (BG2) and is illustrated in Fig. 7.
To show how the change of stage number influences the results, we use the whole BG1 with a row schedule from Row 1 to Row 46 and set both PX and PY from 1 to 8. To make the results more meaningful, we assume that the frequency of the system clock is proportional to PX + PY . Thus, a normalized throughput can be obtained by (PX + PY )/L, where L is the instruction number. The results are shown in Fig. 8. In general, the throughput increases when more stages are introduced. However, the slowdown of growth when PX + PY is large should also be noticed. It indicates that a careful balance between throughput and resource occupation is needed in practical implementation.
