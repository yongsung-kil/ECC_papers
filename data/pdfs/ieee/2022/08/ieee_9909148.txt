Abstract—Quasi-cyclic low-density parity-check (QC-LDPC) codes are widely used owning to its good performance and easy hardware implementation. However, most classical LDPC encoder designs do not support changing the LDPC code types or parameters according to different communication services and channel conditions. In this paper, we propose an encoder architecture with configurable code types, input information block lengths, coding rates and degrees of parallelism, which can realize "hot-switching" of configuration parameters. A new scheme that makes full use of the characteristics of the generation matrix is used in the core of parallel coding. The implementation on field programmable gate array (FPGA) shows that the throughput of the encoder can reach 1.6 Gbps under the condition of 7/8 coding rate (8176, 7154) and the degree of parallelism set to 8.
Keywords—QC-LDPC, encoder, configurable, parallel coding
I. INTRODUCTION
Low-density parity-check (LDPC) is a kind of channel coding, and its long code performance has been proved to be close to Shannon limit. Compared with Turbo codes, LDPC also has lower linear decoding complexity and structural characteristics for parallel decoding [1]. Among them, quasicyclic LDPC codes with quasi-cyclic properties in structure have received a lot of attention and research due to their outstanding realizability [2]-[4].
With the development of society and the improvement of communication requirements, such as the quality of service (QoS) of different mobile multimedia services, and the requirements of bit error rate in complex channel environment, the communication link is required to have a certain adaptive ability. The existing LDPC coding schemes are mostly aimed at a certain code type and coding rate [5]-[7]. QC-LDPC coding for different code types and parameters requires the joint use of multiple encoders, resulting in a waste of resources.
This paper proposes a universal QC-LDPC encoder architecture, which can dynamically adjust the code types, information block lengths, coding rates and degrees of parallelism according to the input coding control instructions, and also provides a design of a codeword operation structure suitable for high-speed parallel coding to approach the configurable coding parameters and high throughput. The proposed LDPC coding architecture is suitable for adaptive communication links and joint encoding schemes [8].

II. QC-LDPC CODES OVERVIEW
LDPC codes belong to a subset of linear block codes，an LDPC code can be specified indirectly by a m × n parity check matrix H consisting of n linearly independent rows. coded sequence of k bits must lie in the n - m dimensional dual space of H, so, all of them must satisfy all m parity check equations corresponding to the v rows of H. In other words, the code can be described through a (n – m) × n generator matrix G.
A. Generation of LDPC Codes
Typical LDPC code generation methods mainly include Mackey method [1], Davey method [9], and PEG method [10]. In order to obtain QC-LDPC codes with quasi-cyclic characteristics that are conducive to hardware implementation, a large number of structural construction methods have been proposed [11]-[13]. By designing the check part of the check matrix into an approximate lower triangular structure, the iterative coding can be directly carried out through the check matrix in the coding process, thus effectively reducing the coding complexity [14]. Design of protograph-based LDPC codes is more flexible in parameter selection and bit rate design, It can effectively avoid short loops and has better error correction performance [15]. Take the construction of QCLDPC codes parity check matrix of AR4JA family recommended in consultative committee for space data systems (CCSDS) 131.0-b-4 standard as an example [16]. B. Performance of QC-LDPC Codes
In order to illustrate the influence of information block length and coding rates on LDPC code performance, binary state-shift keying (BPSK) modulation, additive Gaussian white noise channel and normalized min-sum algorithm are used to simulate the performance of information bit length 256 bits with 1/2, 2/3, 255 bits with 3/4 coding rate and 7154 bits with 7/8 coding rate under different signal-to-noise ratio (SNR). As shown in Fig. 1, under the condition of low Eb/N0, the bit-error-rate (BER) performance of LDPC codes with four coding rates is similar, while with the improvement of Eb/N0, LDPC codes with 1/2 coding rate have the highest BER reduction rate. Fig. 2 shows that when the coding rate is close, and Eb/N0 is high, a small input information block length (255 bits) can also achieve relatively good BER performance. Therefore, when the channel conditions are good, it is necessary to trade-off different code types and coding rates to obtain the most appropriate performance.

A. Efficient Codeword Computing Unit
By analyzing the principle of LDPC coding, the only thing the codeword computing unit needs to do is to calculate (5). All operations are performed over GF(2), so the multiplication between two elements can be replaced by logical operation AND, and the addition can be replaced by logical operation XOR. Further, the value of each bit in the information vector k can be regarded as the magnitudes of component of the corresponding row in the G matrix in the final codeword, so an calculate can be omitted if it is detected that the current information bit is 0. Fig. 3 shows the improved codeword calculation iteration structure. The width of the register groups depends on the degree of parallelism. For serial coding, of course, only one is required for all components.

D. Encoder Architecture
The entire encoder architecture is mainly divided into SCHEDULER, CONFIGURATOR, INDICATOR and CODE_CORE modules according to their functions. The CONFIGURATOR completes the handshake, storage and decode of the input instructions, and provides the required data to the INDICATOR to complete initialization of configuration information. Then, the system starts to perform the encoding task by the CODE_CORE. The collaboration between modules is controlled by SCHEDULER, inside of which is a finite state machine (FSM).
In order to realize dynamically configurable LDPC coding, there are four states defined in SCHEDULER’s finite state machine, which are: idle (IDLE), instruction fetch (IF), configure (CONFIG) and encode (ENCODE), and the operation of the module is scheduled according to different states (e.g., Instruction fetching can only be carried out in the IF state). After completing one encoding, the system will return to the IDLE or IF state according to the coding enable signal EN_CODE. When the state of the system jumps from CONFIG to ENCODE, it must ensure that the valid signal of configuration VALID_ CONFIG had asserted. Fig. 6 shows the state transition graph of the encoder and some key interaction signals.
INDICATOR will continuously monitor the working process of encode and give coding control signals according to the coding progress, including controlling the boot-area data stored in the G matrix initialization ROM loading into the circular shift register, commanding the output of parity check bits, asserting signals related to the end of coding, etc. There are five states defined in INDICATOR finite state machine, which are: idle (IDLE), guide (GUID), convert (CNVT), precomplete (PRE_CMPL) and complete (CMPL). The GUID state is the calculation guide state, which needs to read the data from the control information ROM and control the output of the G matrix initialization Rom. this process is realized by the address generator. For systematic Codes, the information bits remain unchanged in the encoded codeword, CNVT state means that the output mode of CODE_CORE needs to be converted from input follow to parity check bits output. PRE_CMPL and CMPL are both single cycle states. The main reason why "complete" is divided into two states is that the value of the last byte’s mask must be obtained before the last byte of the codeword is output (when the degree of parallelism to be implementation ranges from 1 to 8, the bit width of input and output should be 8 bits of bytes. Because the significant bits contained in the last byte of codeword are related to the degree of parallelism and the length of the codeword, the mask of this byte needs to be given by the control information ROM).

IV. IMPLEMENTATION RESULT
In order to verify the feasibility of the LDPC encoder architecture proposed in this paper, it is implemented on xc7z100ffg900 FPGA of Xilinx company. See Table Ⅲ for the resource occupation. This implementation option supports the information block length of 1024 bits, 1/2, 2/3 and 4/5 coding rate of AR4JA family and 7154 bits, 7/8 coding rate of C2 family according to CCSDS standard, The above two sets of LDPC codes have been proposed because the former is optimized for very low SNR and the latter for high rates, corresponding to deep space communication and near earth communication respectively. Their control instructions can be found in Table Ⅱ.
It should be noted that the consumption of ROM resources will increase with the number of different LDPC codes types, because different code types needs different circulant submatrix information. When the encoder is configured to 7/8 coding rate (8176, 7154) and Parallelism is set to 8, i.e., coding instruction equals to 8’b100_111_11, the system can work at the clock frequency of 220 MHz and the data throughput is about 1.6 Gbps.
