Rapid Design and Prototyping of a Reconfigurable Decoder Architecture for QC-LDPC Codes
Purushotham Murugappa∗, Vianney Lapotre†∗, Amer Baghdadi∗, Michel Jezequel∗ ∗Institut Mines-Telecom; Telecom Bretagne; CNRS UMR 6285 Lab-STICC, Brest, France
†Univ. Bretagne Sud; CNRS UMR 6285 Lab-STICC, Lorient, France Email: {purushotham.murugappa,amer.baghdadi,michel.jezequel}@telecom-bretagne.eu {vianney.lapotre}@univ-ubs.fr

Abstract—Many modern and emerging designs require having efficient dynamically reconfigurable and reprogrammable processors. However, when the implemented design needs an upgrade, newly added features have to be quickly supported and validated. This is clearly noticed in modern receivers of recent wireless communication standards that feature continuously different frame lengths and code rates for the channel decoder. This paper explores with an example the possibility of realizing a flexible channel decoder to implement and validate new/incremental algorithm changes with fast turnaround time in design. An application specific instruction-set processor (ASIP) is proposed as flexible core that can decode low-density parity-check (LDPC) codes with the various block sizes and code rates as specified in WiFi and WiMAX standards. Furthermore, the proposed architecture enables quick support of other Quasi-Cyclic LDPC (QCLDPC) codes, e.g. DVB-S2, with simple incremental hardware changes at design time.
I. INTRODUCTION Many efforts are being made since the past decade towards the proposal of efficient dynamically reconfigurable and reprogrammable systems and components optimized for specific applications [1]. In this context ASIP design approach [2] is being widely explored to preserve implementation efficiency while allowing limited flexibility. Such flexibility becomes a major requirement of many emerging applications to support at run time different execution modes, data formats or even complete standards. Nevertheless, flexibility at design-time becomes equally crucial as new features are usually proposed as an extension to an existing design. Sometimes these features need to be targeted, customized or even optimized for specific design metric catering to an application domain or market sector. To that end there is a need to have run-time and designtime flexibility considered in the process of architecture specification, such that new features can be adopted incrementally as needs arise. In this paper we illustrate how this need can be satisfied using an ASIP design flow with a design example of a flexible LDPC decoder. In fact, digital communication domain is very representative of such requirement where many flexible designs have been recently proposed as solutions for LDPC decoding [3][4]. For this application, there are large variety of coding options specified in existing and future digital communication standards, besides the increasing throughput requirement (Table I). Several architectures have been proposed targeting to meet the throughput requirements specified in these standards, among which the notable ones are [5][6][7]. The architectures

On the other hand, the architecture proposed in [7] produce dedicated cores to decode a specific code rate of a block size of either WiMAX or WiFi, thus giving optimized design time reconfigurable architecture solutions.
The LDPC codes specified in the wireless digital video broadcasting standards (DVB-S2) are characterized by two main differences with respect to those specified in WiMAX and WiFi: (1) very large frame sizes of 16200 bits and 64800 bits and (2) existence of double diagonal permutation matrices in the definition of check matrix. These parameters impact considerably the decoder architecture, memory requirements, and error rate performances. Several dedicated architectures have been proposed [8][9][10]. As an example, the architecture proposed in [8] processes 360 check nodes in parallel achieving a throughput of 91 Mbps (at 300 MHz for rate 1/2) and occupying an area of 11 mm2 in 90nm CMOS technology. [9] solves the well known problem of super-positions of permutation matrices. The enhanced convergence speed of Gauss-Seidel decoding is used to reduce area and power consumption. Furthermore, it proposes a modified version of the lambda-Min algorithm which allows to further decrease the memory requirements of the decoder by compressing the extrinsic information. However inspite of these efforts, studies proposing architecture templates for rapid design for LDPC decoding targeting WiFi and WiMAX standards with possible extensions to DVB-S2 standard are still unexplored.
The contribution of the presented work in this paper is twofold: (1) investigate the maximum attainable architecture efficiency for ASIP-based LDPC decoding and (2) explore the possibility of realizing a flexible decoder that allows to implement and validate new/incremental algorithm changes with fast turnaround time in design. II. BACKGROUND AND ALGORITHM
Fig. 1a shows the typical format of the check matrix Hbase specified to define Quasi-Cyclic LDPC (QC-LDPC) check matrix. Most wireless communication standards specify this popular class of structured LDPC codes as it is hardware aware; facilitating decoder implementation. However, each standard specifies different code rates and block lengths usually expressed through different expansion factors Z ( Table I). The check matrix Hbase (e.g. Fig. 1a) consists of N block columns and M block rows wherein each of the non-negative values is replaced by a cyclic permutation matrix of size Z×Z. The negative values are replaced by zero square matrix of size Z. The expanded LDPC check matrix Hexpanded thus obtained is as shown in Fig. 1b. The LDPC check matrix can also be represented in a graphical form with a bipartite tanner graph (Fig. 1c) where each column and row in Hexpanded represents a variable node and a check node, respectively. In this figure, CN Gx ∀ x = 0..(M − 1) are groups of check nodes of size Z connected to V N Gy ∀ y = 0..(N − 1) groups of variable nodes. The lines connecting the variable and check node groups are representative of the ones in the check matrix Hexpanded (Fig. 1b).
We utilize hardware efficient version of the belief propagation algorithm [14] called the Normalized 2-Min-Sum LDPC decoding algorithm [15] with layered decoding schedule, which consists of iterative exchange of messages (soft values) between the check and variable nodes as described below.

III. PROPOSED RECONFIGURABLE DECODER
ARCHITECTURE
Table I shows various specifications of Z for different standards namely 24 to 96 for WiMAX standard and 27, 54 or 81 for WiFi standard. On the other hand high throughputs are imposed. In this context, the design of high throughput decoder requires an implementation strategy that enables the re-configurability at run time to support WiFi and WiMAX, and at design time for future support of other QC-LDPC codes (e.g. DVB-S2 with Z = 360). These expansion factors are indicative of the possible parallelism degree in processing of check nodes allowed by the standard without degradation of error rate performance. Supporting the maximum best case parallelism degree of 360 leads to inefficient use of hardware when lower parallelism degrees are permitted (e.g. Z=24). Further, the VN update and CN update steps of the decoding algorithm can be parallelized and pipelined as long as there are no conflicts in simultaneous updates of the variable nodes. Moreover, simulation studies show that maximum number of iterations required for decoding is in the range of 20 (for WiMAX and WiFi) to 25 (for DVB-S2).
Considering the above factors and the throughput requirements, we present a flexible LDPC ASIP decoder architecture (Fig. 2) with maximum parallelism degree PDMeagx=48, i.e. maximum of 48 check nodes are processed in parallel by the ASIP. Two flexible barrel shifters of size 48 are used to implement the cyclic shifts of any permutation (related to the standards): one for reading and one for writing channel LLRs in the input memory banks. With this parallelism degree, decoding LDPC frames with Z > 48 are supported by dividing the layer of check nodes into two sub-layers (refer Fig. 1b). Each sublayer is scheduled for processing in two phases by applying the layered decoding schedule as explained in previous section. In fact, all expansion factors for Z > 48 are multiple of ”2” except for Z = 81 in WiFi standard. For this particular case, the layer is split into two sub-layers containing 40 and 41 check nodes. In this context, barrel shifter flexibility covers all specified expansion factors Z < 48: 24, 26,28,30,32...48, besides the support of Z=27 and 41.
The proposed ASIP architecture has 8 pipeline stages and only 4 instructions. The configuration parameters regarding check matrix definition and the decoding instructions are stored in a single 17 bit wide instruction/config memory. The first part of this memory stores the configuration parameters specifying the key characteristics of the check matrix namely the number of parallelism degree, the depth of the instruction memory, number of sub-layers, number of iterations and the lookup table containing the scaled 4-bit CN to VN message

This field can have one of the four values namely: CNupdt, VNupdt, CombinedVNnCNupt or NOP corresponding to the execution of CN update, VN update, combined VN and CN updates or idle cycle respectively.
CNupdt instruction initiates the variable to check node messages calculated according to equation (2) by using the output of the barrel shifter and the check node to variable node message of the previous iteration Lie−xt1(m, n) fetched from the extrinsic memory banks. The 8 bit intermediate values namely Li(n, m) are stored in a FIFO memory along with Permutation, Hcol and SubLayer values to be used to generate addresses and barrel shift values during update step. The MinFind pipeline stage (Fig. 2) serially calculates the 2 least absolute minimums min(n , m)0,1 and sgnm of the variable to check node messages corresponding to a check node of the sub-layer under processing.
Once the variable node group corresponding to the last column of the sub-layer/layer connected to check node group is read, the Update and InvBarrelShift stages of the pipeline is activated. This done by CNupdt or CombinedVNnCNupt instructions depending on if one or more sub-layers are present per layer. Li(n, m) is read from the FIFO and aposteriori LLRs are calculated according to equations (6) and (7). The InvBarrelShift pipeline stage (in Fig. 2) shifts the newly calculated aposteriori LLRs to the original order and writes them back to the channel memories. The check node to variable node update messages Liext(m, n) are stored in the extrinsic memory banks.
When more than one sub-layers are present, the CombinedVNnCNupt instruction is used to execute the VN update step for the current sub-layer while the Update and Inverse Barrel Shifter pipeline stages execute the VN update for the previous sub-layer. This implies the use of dual-port memories for input and extrinsic, however it allows a perfect usage of all pipeline stages.

IV. PROTOTYPE AND INCREMENTAL FEATURE ADDITION The above described architecture is prototyped on an FPGA platform and controlled via graphical user interface on a host computer. A hardware-in-the-loop strategy is used to validate the design under test, with LDPC encoder and hardware equivalent decoder as shown in Fig. 4. Such an approach is simpler and faster than building a full hardware prototype of the communication system including a flexible hardware emitter and an accurate hardware channel model [16]. Furthermore, as the barrel shifter description is also included in the LISA modeling, the generated VHDL needs only addition of memory modules and input interface to realize a complete decoder system. The input configuration parameters and the channel memories are written through the USB interface from the host computer running the C based simulation program and can be modified at run time.

An interesting point in the above expression of the AE concerns the normalization of the throughput achieved with respect to the considered clock frequency (fclk) which increases the fairness when comparisons are done between different decoding architectures running at different clock frequencies. Published results in this context consider either the maximum achievable clock frequency by the proposed architecture or a lower operational clock frequency which is sufficient to achieve the target throughput. Thus, normalizing the presented throughput by the considered clock frequency enables to better exhibit the efficiency of the proposed architectural choices. Towards the same objective, the above expression of the AE normalizes the throughput by the considered number of decoding iterations (Niter) as the published results can use slightly different values which impact the overall throughput. In most of these works, the same low complexity decoding algorithms, with identical convergence speed, are used. Similarly, the AE expression uses a normalized area measure (AreaNorm) as the published decoders are often based on different technology nodes (e.g. 180nm, 130nm, 65nm, etc.). In addition, when the published design area is given post-place and route a downscaling factor of 2 is applied to obtain a reasonable estimate of the post-synthesis area. This factor is not very accurate as it depends to many parameters (technology node, CAD tools, operating conditions, etc.), but it gives a reasonable idea as it corresponds to the usually (or even worst case) observed ratio.

However, there still room for further optimizations in order to reach the maximum achievable AE of 36.3 obtained in [7]. The architectures presented in [5] and [6] support only WiMAX LDPC codes which have better organization for parallelism than those specified in WiFi standard.
Similarly, synthesis on Virtex-5 (xc5vlx330-1ff1760) FPGA resulted in a design utilization of 3991 slice registers, 15807 LUTs and 60 BRAMs and occupying about 10% of the FPGA logic resources with maximum clock frequency of 90MHz. We can notice that the decoding latency is approximately twice for frames with Z > 48 than when compared to Z < 48 due to the sub-group decoding strategy. Yet the maximum throughput that can be achieved is more than 470Mbps for both WiFi and WiMAX standards. The negligible loss in error rate performance of the implemented design compared with the C-simulation performance is illustrated by Fig. 6.
Furthermore, the expected throughput for DVB-S2 standard with 25 iterations is given in Table VI with a maximum reaching 257 Mbps. As only slight incremental changes are expected during design time in the logic structure of the hardware, the memories are the major contributor for area increase.
VI. CONCLUSION
In this paper we have presented a flexible and optimized ASIP-based architecture for multi-standard LDPC decoding. The proposed architecture was designed to maximize the architecture efficiency, which was achieved by increasing the parallelism degree and the memory bandwidth with flexible barrel shifters. It exploits a maximum parallelism degree of

Fig. 6. Hardware measured and C-simulated (floating point) BER curves for WiMAX LDPC code with Z=96 and code rate=1/2
48, i.e. 48 check nodes are processed in parallel, achieving a maximum throughput of 475 Mbps and an architecture efficiency of 13.6 bit/cycle/iteration/mm2. It supports all LDPC codes and related parameters specified in WiFi and WiMAX standards. The other main contribution of this work concerns the design time flexibility feature which allows to prototype new/incremental algorithm changes with fast turnaround time in design. The proposed architecture enables through incremental changes the future support of other QC-LDPC codes, such as those specified in DVB-S2 with high expansion factor Z = 360. This illustrates how the ASIP-based design approach can be used as a way to achieve both design-time and runtime flexibility features. Finally, the proposed architecture has been fully validated through an FPGA prototype based on a hardware-in-the-loop approach.
