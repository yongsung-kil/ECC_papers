2010 IEEE 21st International Symposium on Personal Indoor and Mobile Radio Communications

Encoding of Certain LDPC Codes with Decoding Resources
Z. H. Cai, P. S. Chin, J. Hao, C. M. Pang, and S. Sun
Institute for Infocomm Research #21-01, 1 Fusionopolis Way Singapore 138632

Abstract- This paper addresses the issue of encoding of LDPC codes for certain transmission standards that employ LDPC as coding scheme. It is shown that for these LDPC codes, the encoding can be done simply by reusing decoding resources with negligible overhead. Compared to conventional encoding designs, our approach requires much less resources specified for encoder.
I. INTRODUCTION
Low density parity check (LDPC) codes are one of the most powerful error correcting codes available today. First introduced by R. G. Gallager decades ago and rediscovered by Spielman, Mackay and many others, can achieve very good performance when decoded with the belief-propagation (BP) or the sum-product algorithm [1-2]. The remarkable performance of LDPC codes has positioned themselves as strong candidate for forward-error-correction (FEC) codes in many digital communication systems, such as IEEE 802.11n [3], WiMedia [4], IEEE 802.15.3c [5], and WiMax [6]. LDPC decoding requires huge amount of messages exchanged in an iterative manner. The LDPC codec implementation becomes very hardware efficient when employing an architecture-aware code design [7-8].
Compared with decoding, the encoding of LDPC codes is much simpler, but it is by no means a trivial task. By taking advantage of certain properties that exist in LDPC codes, encoder design can be very straightforward and hardware efficient [9-11]. By exploiting the dual-diagonal structure of the standard H matrices, with little modification of Hs, a lowcomplexity correction-based efficient encoding scheme that requires no designated hardware is reported in [12], i.e., the encoding can be done with the hardware resources for decoding. And the overall computational complexity of the proposed encoding scheme is lower than the well-known Richardson’s efficient encoding scheme [9].
However, modifications on standardized matrices are not always feasible; this will raise compliance issues. In this paper we show that, without any modification on the standard H matrices, for LDPC codes defined in WiMedia 1.5 [4], WiMax [6] and IEEE 802.11n [3], and the emerging IEEE 802.15.3c [5] (mmWave), efficient encoding can be done by reusing decoding resources such as buffer memory, parity check matrices ROM, routing/shuffling network between check node processors (CNPs) and variable node processors (VNPs). Only negligible overhead is required.

The rest of the paper is organized as follows. Section II presents efficient encoding process for some carefully designed LDPC codes. Algorithms/architectures for LDPC decoding are discussed in Section III. Fitting encoding into decoding architecture is detailed in Section IV and the summary is given in Section V.

The architecture in Figure 6 is with serialized Rcv/Qv input to CNP and is able to process one block (z edges/check nodes) per clock cycle. For some applications this z-edge processing per cycle capability is not enough. A design that allows for 3z edges per clock cycle is reported in [15] for WiMedia 1.5 [4], which instantiates 3 slots/banks of Qv RAM and each CNP has 3 Qv/Rcv input pairs. In this way decoding latency of less than 1 μs can be achieved. The price is more hardware resources are needed, such as barrel shifters and bigger check node processors, and memory control logics.
The architecture in Figure 6 is for general block-LDPC codes, i.e., any H with the form expressed in (1) can be implemented in this way. This architecture, and its variants, is able to support the codes defined in [3-4] and [6]. To have higher throughput/lower decoding latency, such as, to support 5.28 Giga-bps data throughput [5], i.e., to decode a codeword

of length 672-bit within just 672/5280 = 0.1273 μs, more parallelism of check node processing is necessary, i.e., depending on the processing clock frequency, the CNP needs to read/write more Qv/Rcv messages, such as 8, 16, or even 32 messages, within one decoding clock cycle. This requires more specific features of the LDPC matrices defined in [5] to be explored:
(1) The parity check matrices have the periodically cyclic structure of length 4. For example, assuming the first 4 elements of the first row of rate 7/8 H are [r1 r2 r3 r4], and the second, third and fourth are [r4 r1 r2 r3], [r3 r4 r1 r2] and [r2 r3 r4 r1], respectively. For code with rate ¾, the cyclic structure is per two-row. And for code with rate ½, the cyclic structure is per 4-row.
(2) The code ¾ and ½ can be expanded from the base code with rate 7/8, e.g., every two rows of the code with rate ¾ are expanded from one single row of the base code with empty entries filled with all -1.
Based on these features, we have implemented a synthesizable IP core for IEEE 802.15.3c draft compliant LDPC codec with multi-Giga bps throughput support. Only 4 cycles are necessary for one iteration. All three coding rates are supported with the same architecture.
We have discussed efficient encoding schemes for transmission standards [3-6] and the architecture design for block-LDPC codes. Next we show that for LDPC codes with staircase form and lower triangular form of H, the encoder can be fit into decoder nicely. The resources illustrated in Figure 5, i.e., input/output buffer, CNP circuit and the routing network between CNPs and Qvs, can be reused with only minor hardware overhead. No modifications on Hs are needed.

The box with dotted-line is for the computation of syndrome of the current layer. It accumulates the XOR of the sign bit of the variable node message Qv for current layer and gets reset for each layer. The syndrome is used to test whether the received word has been successfully decoded so that the decoding process can be terminated. This is the part needed for encoding process. The D register here can be used to store the accumulated XOR result to compute p0 without reset between layers, and subsequently, to compute all other parity check sub-vectors with minor hardware changes. The procedure is summarized as follows:
(1) Store the input bit to the sign of the Qv message and clear the parity check part of Qv memory. Disable the Rcv memory and NMS module.
(2) During the first iteration, accumulate the XOR result without reset of the D register. At the end of the first iteration, the first parity sub-vector p0 is generated.
(3) Route the D register output to the sign of the Qv message at the output of the check node processor so that the p0 is stored to Qv memory. Reset the D register.
(4) Within 2nd iteration, the D register is reset per layer as in normal decoding process. Back-substitution is executed for each parity check sub-vector. At the end of each layer, new parity check bits are generated and stored to Qv memory.
It can be seen from above procedure that the encoding takes 2 iterations to complete. Unlike decoding process where NMS takes several cycles to finish, the encoding XOR is much simpler and it takes only one cycle to finish. The hardware overhead is the modification of Qv memory control, the D reset control, and a multiplexer added to route D output to the sign bit of Qv message.
The encoding process discussed above is for the CNP design with 1 Qv/Rcv pair, but it is applicable to the CNP with multiple/parallel Qv/Rcv pairs as well, such as, the design in [15]. It can be used for parity check codes with staircase form of parity check matrices, such as, the codes in [3], [4] and [6], without any change of the parity check matrices.

For encoding purpose, the CNP is changed to XOR and ‘dummy’ straight-through process. Actually the process is even simpler than the encoding process for LDPC codes with dual-diagonal structured Hs:
(1) Store the input bit to the sign of the Qv message and clear the parity check part of Qv memory. Disable the Rcv memory and NMS module.
(2) The XOR result is output to port 29 of the CNP, at the first encoding cycle the first parity check sub-vector is generated and cyclically shifted to right position of Qv memory by the routing network between Qv memory and CNP arrays.
(3) The other 3 parity check sub-vectors are generated in sequence during the next 3 clock cycles by back-substitution.
In this way only 4 cycles are needed to encode the rate 7/8 LDPC code defined in [5]. Encoding for other rates can be done in a similar way. The overhead is just the minor modification of the CNP controller and some multiplexers added to the CNP array.

V. SUMMARY
In this paper we have proposed to encode the LDPC codes using decoding resources for LDPC codes defined in several transmission standards, such as WiMedia, IEEE 802.11n, WiMax, and IEEE802.15.3c, as long as the parity check matrices are with the staircase form or lower triangular form. The idea is applicable to other block-LDPC codes as well. The encoding process is straightforward and no modifications on H matrices are necessary. Only minor overhead is needed.
