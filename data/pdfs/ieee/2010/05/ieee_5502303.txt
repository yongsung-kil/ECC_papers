Abstract—For binary linear codes with short and medium block length ML decoding can be achieved by solving the associated integer programming (IP) problem with a general purpose solver. IP also offers algorithms for computing the minimum distance. In this article, we present several IP formulations and computationally compare them on various LDPC and BCH codes. Most of these formulations are obtained by forcing integrality on linear programming (LP) decoding formulations proposed in the literature.
I. INTRODUCTION
Integer programming provides powerful means for modeling several real-world problems. The problem of maximum likelihood (ML) decoding has been first modeled by Breitbach et al. [3] in the late 1990s. Several other formulations can be derived from LP models which have been proposed in the LP decoding literature. Some of them are applicable to general binary linear codes while others are designed for some specific subclasses. Integer programming models are usually solved by commercial solver packages. Together with these general purpose solvers, these models allow practitioners to study codes closely. For example, information about ML decoding curves or about code features like the minimum distance can be obtained quickly for a large variety of codes without the need of implementing a specially designed decoding algorithm.
Since several different models have been formulated in the literature, the question of the capabilities of these different models for maximum likelihood decoding arises when using a general purpose solver. In this article we use the software package CPLEX [4] to compare the existing models numerically. First, we recall the different models by explaining their objective functions and sets of constraints. Then, we analyze the performance of these models with respect to number of variables, number of constraints, average number of simplex iterations, average number of nodes in the branch and bound tree, average number of cuts, and accumulated running time. Finally, we give some concluding remarks explaining the numerical results obtained.
LP decoding is improved by branch and bound approaches (see [6], [7], [16]) and cut generation algorithms (see [9], [12], [13]). In this context, our results may also provide some insights into the behavior and eligibility of various LP

formulations.

III. IP MODELS
Breitbach et al. [3] introduce an IP formulation we denote by IPD1. It can easily be verified that x ∈ {0, 1}n is a codeword if and only if Hx = 2z where H is an m × n parity check matrix of the underlying code and z is an integer vector of dimension m. Formally, the IPD1 can be written as
min λT x (IPD1)
s.t.Hx − 2z = 0 x ∈ {0, 1}n
z ≥ 0, integer.

An alternative formulation can be obtained by using error vectors. Let y¯ be the received hard decision vector. Comparing y¯ ∈ {0, 1}n with a codeword x ∈ C results in an error vector e ∈ {0, 1}n, i.e., e = (y¯ + x)mod2. Let s = Hy¯. IPD1 can be reformulated as
min λT e (IPD2)
s.t.He − 2z = s e ∈ {0, 1}n
z ≥ 0, integer.

An IP formulation, for those codes which can be defined by a finite state machine is given by Feldman [8]. Particularly, this formulation is applied to turbo codes. Any turbo code can be represented by a directed tree T = (V, E) such that the nodes v ∈ V model trellis graphs. In a trellis graph edge labels are binary bit strings which are the outputs of the encoder for information bits. Paths from the starting node to the end node correspond to codewords. The cost of a codeword is derived from the received word and the edge labels on the path associated with this codeword. The IP formulation for turbo codes with numerical comparisons are given in the long version [14] of this article.

IV. NUMERICAL EXPERIMENTS
In this section we compare the aforementioned formulations numerically using CPLEX [4]. CPLEX IP solver employs an LP based branch and cut procedure to solve IP problems. In the following we briefly explain a generic branch and cut algorithm. For details [11] is referred. Initially, i.e., in the root node of the branch and bound tree, the integrality constraints of the given IP problem are relaxed and the resulting LP is solved, e.g., by a variant of the simplex method [11]. If an integer solution is found, then the solver outputs the LP optimum as the solution. Note that in this case, LP optimum is the IP optimum as well. If the optimal solution is nonintegral, then the solver tightens the polytope of feasible solutions by generic cut generation techniques, e.g., mixed integer rounding cuts or Gomory fractional cuts (see [4]). Cuts found are added to the LP formulation and the resulting LP problem is solved again. If the LP optimum remains nonintegral and no further cuts can be found, then the solver branches on a variable, say xi, having a non-integral value, say v, in the current solution. Branching is achieved by adding constraints on this variable, thus producing two new subproblems referred to as child nodes. For example, one subproblem is obtained by adding the constraint xi ≤ v to the original formulation while the second is obtained by adding xi ≤ v . By this technique, the search space is implicitly enumerated. Several fathoming rules based on lower and upper bounds of subproblems are established for pruning branches in the search tree, thus speeding up the computation. Repeating this procedure iteratively to all child nodes of the search tree yields the IP optimum eventually.
We compare the performances of IPD formulations as ML decoders on several LDPC, BCH codes in our numerical experiments. Here we present only (204,102) [10] LDPC code and (63,39) BCH code as representatives due to the space restrictions. We refer to the longer version of this article for the complete set of codes. The BIAWGN channel is assumed to be the transmission channel and the zero codeword is transmitted. Computations are executed on a compute server equipped with a Dual AMD Opteron 242 1.60GHz and 2 GB RAM, running Linux Kernel 2.6.16 SMP x86 64. Running time is measured after building the model and therefore only the computation time consumed by the solver is considered. For each SNR value, 100 received words are simulated. The performance measures we use to compare IP formulations are the number of variables, number of constraints, average number of simplex iterations, average number of cuts found by the IP solver, average number of bounds in the branch and bound tree and the accumulated CPU time for solving 100 instances in seconds.
The number of variables in IPD3 and the number of constraints in IPD4 grow exponentially in δmax. In our experiments these numbers are comparatively large (see columns Var., Const.). In IPD5, the number of variables and constraints grow polynomially but in O(n3) which also results in a fairly large number especially if block length increases. For LDPC codes IPD3 has many variables but the number of constraints is not large. Conversely, IPD4 has many constraints and only few variables. On the other hand, formulation IPD5 has large number of variables and constraints. Generally, it is observed that the running time of the IP solver increases with the number of variables and constraints of IPD formulations for ML decoding. This is due to the increasing size of the search space and to the increasing computational effort required to solve LP problems. A large number of variables and constraints increases the size of the simplex tableau. In our tests, the running times of IPD3 and IPD4 are roughly comparable while solving IPD5 takes more time. The motivation for IPD5 is the observation that the size of the formulations IPD3 and IPD4 depends on the maximum check node degree which becomes prohibitively large for dense parity check matrices. In contrast, IPD5 has the advantage that it can theoretically also be applied to any binary linear code independent of the check node degree. IPD1, IPD2 and IPD5 share the same advantage. Moreover, the number of constraints and variables grow linearly in block length in these latter formulations.

As a consequence nonintegral optimal vertices of the relaxed polytope are cut off and the chance that an integral optimum is found in the subsequent iterations increases. The IP solver adds significantly (by one to three orders of magnitude - depending on the SNR value, block length, and the density of the parity check matrix) less cuts in formulations IPD3 to IPD6 than in formulations IPD1 or IPD2. This implies that formulations IPD3 to IPD6 have tighter LP relaxations. For all codes and SNR values, formulations IPD1 and IPD2 have the largest average number of cuts. However, for high density codes no significant differences in the number of cuts can be observed. Thus, formulations IPD5 and IPD6 do not provide a tight LP relaxation for high density codes.
Simplex algorithm iteratively goes through the vertices of the underlying polytope until the vertex corresponding to the optimal solution is reached. If there exists a vertex for which the objective function can be improved in the current step, the simplex algorithm moves to this vertex. Otherwise the algorithm stops. The procedure of moving from one vertex to an other is a simplex iteration. The average number of simplex iterations are reported in column Iterations. In the formulations IPD3, IPD5, and IPD6 the number of variables is larger than in formulations IPD1 and IPD2. Similarly, the number of constraints is larger in IPD4 than it is in IPD1 and IPD2. A large number of variables or constraints seem to lead to polytopes with a large number of vertices and facets in our experiments. As a consequence, more simplex iterations are needed in these formulations. The relaxed polytopes of IPD1 and IPD2 have either a lower dimension or less number of constraints. Less simplex iterations are needed on the initial LP. As the block length increases, the number of cuts added in IPD1 and IPD2 increases as well. During the process of generating cuts and resolving LPs, a larger number of LPs are solved and, consequently, the number of simplex iterations also increases. For high density codes, far less (factor 100 or more) iterations are needed in IPD1 and IPD2.
To sum up, the IP formulations can be grouped into two categories. IPD3 to IPD6 have tight LP relaxations with relatively large numbers of variables and constraints. This leads to larger Simplex tableaus and more complicated polytopes and only a few number of cuts have to be added to tighten the LP relaxation. Each simplex iteration needs considerable effort. The search space increases with the number of variables.
Finally, we analyze the accumulated running times of the IP solver for each formulation. The CPU times in seconds to solve 100 instances are listed in column Time. The formulations IPD1 and IPD2 are constructed with less variables and less constraints. Their LP relaxations are not as tight and cuts need to be generated when traversing the search tree. For LDPC codes - except for the (204,102) LDPC code - the running time of the solver for IPD1 and IPD2 is less than for IPD3 to IPD6. For LDPC code (204,102) there is an increase in the running time of IPD2 compared to IPD1. For high density codes, the running times of IPD1 and IPD2 are significantly lower than those of IPD5 and IPD6. As the block length or the number of test instances increase, the running times of IPD5

and IPD6 becomes prohibitively large. IPD3 and IPD4 could not be tested on high density codes due to excessive size of the model. Based on our tests we conjecture that it is favorable for ML decoding via IP models to start with a formulation having few variables and constraints and iteratively generate cuts in regions of the relaxed polytope lying in the minimization direction.
