Abstract—Rate-Compatible/Quasi-Cyclic LDPC codes are gaining importance because of their tradeoff between performance and simplicity. For these reasons, RC/QC LDPC codes have been included in several recent standards such as 802.20 and 3GPP2. General proposals for hardware encoding of this kind of LDPC codes either consume a lot of area to increase the throughput or are slow-encoding schemes to reduce the area consumption. Some others have to pre-compute a non-sparse generator matrix and store it in memory in order to encode. In contrast, by exploiting the number of null matrices in the mother parity check matrix defined in these standards, this proposal defines a low-cost encoder with high input packet throughput (little time between input frames). Moreover, as the encoding is performed using the original parity check matrix, there is no need to pre-compute or store the dense generation matrix in memory.
Index Terms—3GPP2, RC/QC- LDPC, 802.20, VLSI.
I. INTRODUCTION
Rate-less codes, unlike traditional fixed-rate fixed-length block codes, are characterized by a continuous stream of code bits, all generated from a single fixed-length message word [1]. Depending on the channel state, a variable number of parity bits are appended to the systematic word. A promising rate-less coding technique involves ratecompatible low-density parity-check (RC-LDPC) codes [2]. A RC-LDPC code is defined in terms of an ever-expanding parity check matrix. First, a fixed-rate LDPC code is chosen as the mother code. Then, the parity check matrix of the mother code is extended to achieve lower rates and puncture this matrix to achieve higher rates. To ensure linear encoding complexity, each parity check matrix, H, of a family of RCLDPC parity check matrices is designed with lower-triangular form. One important type of the RC-LDPC codes is the ratecompatible quasi-cyclic LDPC (RC/QC-LDPC) codes. These codes show a good tradeoff between performance and simplicity. They are currently proposed as candidates for forward error correction in the 3GPP2 [3] and 802.20 MBWA [4] standards. They have also been proposed to be included in the 802.16m standard [5].

The mother parity check matrix of the RC/QC-LDPC codes, Hmth, is constructed by substituting circulant matrices [11] into a base matrix. This mother parity check matrix is expanded to obtain lower rate codes.
Some flexible hardware encoding architectures for RC/QCLDPC codes have recently been proposed in [6] and [7]. These works establish a RC/QC-LDPC scheme and define three possible architectures to encode these kinds of codes. The encoders proposed in [6,7] are fully compatible with 3GPP2 and 802.20 codes. However, the architectures proposed do not exploit the inherent number of null sub-matrices included in the parity check matrix. Moreover, the architectures in [6,7] require the non-sparse generator matrix to be pre-computed and stored in memory. As the generator matrix is non-sparse, much more memory is required to store it than for the sparse parity check matrix. In addition to the memory requirements, the complexity of the encoding using the non-sparse generator matrix is also greater, as discussed in [11]. Another advantage of using a sparse H matrix for encoding appears when decoders and encoders are in a single platform. In this situation both the encoding and decoding process can be developed using the parity check matrix, avoiding extra memory consumption.
Another way of encoding RC/QC-LDPC codes defined in the 3GPP2 and 802.20 standards is proposed in the 3GPP2 standard documentation [3]. The standard recommends obtaining the parity bits corresponding to the mother parity check matrix using the classical Richardson and Urbanke method [8], exploiting the lower triangular form of the parity check matrix of the mother code. For the rest of the parity bits, the standard defines single parity check, which is straightforward. The Richardson and Urbanke method has been applied in [9] in which an encoder architecture for the 802.16e standard, with reduced latency, has been proposed. This standard defines a QC-LDPC code whose parity check matrix is the same as the one defined for the mother codes in the 3GPP2 and 802.20 standards. The encoder we present in this paper is also suitable for encoding that standard. The encoder proposed in [9] is particularly suitable for a fair comparison with our proposal as well as the [6,7] one mentioned above, so results for these approaches will be compared with ours later.

We present an architecture aimed to present a good tradeoff between area and throughput.
To obtain a good throughput, the proposed architecture is designed to calculate pC in parallel with pA and pB. This is
possible by exploiting the [M 2 I ] structure in the H matrix.
Parity bits pC depend only on systematic bits, pA and pB because of the I matrix involved in the pC calculation. Therefore, we can calculate pC bits while receiving systematic bits and while extracting pA and pB bits.
To obtain a reduced area, we design the modules that compute pA, pB and pC to work serially. This reduces the hardware cost with respect to a parallel architecture. Moreover, it does not proportionally increment the time to encode due to the large number of null sub-matrices that make up H1.
Hereafter, the architecture proposed is described in detail.

B. Computation of pC
As previously mentioned, we calculate the pC parity bits at the same time as pA and pB. It can be done in such a way because the parity bits pC depend on the systematic bits and on

the pA and pB parity bits, but never on some other pC bits. When a group of systematic bits is received, it is used to

serially operate with the corresponding column of M1 for pA and pB calculation. At the same time it serially operates with the corresponding column of M2 for pC calculation. In Fig. 3 this process is carried out by a circuit named Circuit 4. These

parity bits will be stored in a dual port dedicated memory of

word size equal to Z. The dual port memory is used in order to read a group of Z bits in one cycle and update the previous

group in the same cycle.

The standards [3][4] define more non-null sub-matrices per

M2 column than per M1 column. This means that if we serially compute non-null matrices with a single circuit for computing

pC , the time necessary to operate over an H matrix column is
bounded by M2. Therefore, in order not to increase the number
of cycles used to compute pA and pB, the memory is split into NN memories in such a way that the number of pC words
stored in a memory dependent on a group of systematic bits is

equal to or less than the number of matrices in H1 part for the
column involved with these systematic bits. It also requires cloning NN times the cyclic shift register and the XOR

computation part of Circuit 4.

pC dependence on pA and pB bits has to be updated while pA and pB are extracted from circuit 3. pA and pB are taken out at a rate of a parity word per cycle. Therefore, following the same

reasoning as above for systematic bits, memory has to be

arranged in such a way that no more than one pC word is involved per memory per column. These can be easily

obtained because the part of M2 involved is very sparse. To calculate the time between input frames, we must

consider that the total time required by Circuit 4 to send pC to the output is equal to the total number of parity bits of pC

divided by Z, i.e. (mb − mb' ) cycles. The following frame

cannot be accepted until Circuit 4 is ready to operate with it.

This work presents 3 kinds of encoders: the first one, called serial encoder, receives the input data in blocks of Z bits and calculates the parity bits serially operating by rows. The second one, called parallel, receives the entire input sequence in parallel and performs the codification by columns. Finally, the one denoted S-Parallel, also receives the entire input data in parallel, but performs the codification over all the sub-matrix columns at the same time, reducing the encoding time to the dimension of one of the sub-matrices (Z). All three encoders presented in [6,7] require the generator matrix to be pre-computed and stored in memory. Although the H matrix is sparse, the G matrix is not necessarily sparse and therefore more memory is required to store it. An example can be seen in Figure 4 for a rate 0.5 mother code matrix defined in the 3GPP2 standard. Moreover, when the encoder and the decoder are to be implemented in the same platform, they cannot share the same matrix unlike what happens with our proposal.

The second work we use for comparison was presented in [9]. This work implements an encoder for the 802.16e standard, whose parity check matrix follows the same structure as the one defined for the mother parity check matrices in 3GPP2 and 802.20. The coding method presented in [9] is based on the Richardson and Urbanke method, but with some improvements to reduce the encoding latency. We have chosen this work for comparison because the 3GPP2 standard recommends the Richardson and Urbanke method for encoding the initial parity bits using the mother parity check matrix, which is the most complicated part of the encoding. Obviously, the results provided using this architecture will only represent the cost of implementing the encoding over the mother parity check matrix. However, the requirements for encoding using the expanded part of the final parity check matrix are far less demanding.

Summing it all up, we can see the proposal outperforms the recently published related works in most of the measures. Analyzing each category:
• The time between input frames in our proposal is the minimum among all the proposals (therefore the input packet throughput is the highest). Our result is almost three times better than the best case achievable in [6,7].
• The memory consumption is inferior to whichever architecture in [6,7] and only outperformed by the architecture in [9]. However, this architecture only performs the mother code encoding.
• The number of registers required by the architecture proposed is less than the ones proposed in [6,7] and [9]. Even when a parallel architecture [6,7] is three times more register-consuming than our proposal for the example described.
• The number of XORs required in our proposal is only higher than the number required in [9].
• Neither [9] nor our proposal requires AND gates, unlike the [6,7] proposals.
• Finally, the number of cyclic shifters required by our architecture is less than is required by any other proposal. It is half of the best case in [6,7].

V. CONCLUSIONS
In this paper a novel architecture for RC/QC-LDPC encoding, compatible with 3GPP2 and 802.20 standards, has

Moreover, the proposal reaches a high input packet throughput by exploiting the high number of null sub-matrices that make up the parity check matrix.. Finally, the proposal is flexible, being able to reduce the number of necessary logic resources by increasing the time between frames with minor modifications. In terms of experimental results, the proposed architecture has been compared with most recently presented works outperforming them in most aspects.
