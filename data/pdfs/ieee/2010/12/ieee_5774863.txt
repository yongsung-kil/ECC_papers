Abstract— LDPC coding has attracted much attention due to its high performance, and it has been widely used in telecommunication systems. This paper focuses on the decoder hardware architecture, especially on memory size reduction, which is an important part of the entire area cost. The design has been post-layout simulated using a UMC 0.18 micron technology at a clock speed of 74 MHz. Using the proposed 3level memory structure together with the described control logic, the required number of bits of memory can be reduced by up to 34.9% compared to prior approaches.
Keywords— LDPC, low density parity check code, decoder, layered decoding, hierarchical memory organization.
I. INTRODUCTION
LDPC codes are among the most powerful forward error correcting codes available due to their near Shannon limit performance. Thus, they have been widely adopted by several communication standards, including IEEE 802.11n, 802.16e, UWB and DVB-S. Layered decoding [1, 2] is typically implemented using a partially parallel architecture and gives a good balance between hardware complexity and decoding throughput. Generally speaking, LDPC codes with long code-length (N ³ 1Kbits) [3] can achieve excellent performance, but this requires a very large memory which dominates the decoder’s area. Therefore, memory size reduction becomes an important factor in achieving both low power and low area cost.
This paper proposes the following three techniques for optimizing the design of an LDPC decoder: 1) a layered decoding architecture with medium-grained functional units; 2) a three-level memory organization; 3) a mechanism for the overall signal routing controls. We abstract the basic processing elements to merge them into a new execute unit (EU) as part of a reconfigurable baseband platform [4]. The three-level memory structure along with the signal control

II. CODE CONSTRUCTION AND ALGORITHM OVERVIEW
The parity check matrix of an LDPC code in IEEE 802.11n and IEEE 802.16e is partitioned into several smaller sub-matrices, in which an extension factor zf determines the sub-matrix size, and a total of 22 modes are defined [5, 6]. The extension factor value ranges from 24 to 96, with a maximum code length of 2304 bits, and the supported code rates are from 1/2 to 5/6. Those sub-matrices are either the zero-matrix or a cyclically-shifted identity matrix.

The basic LDPC decoding algorithm, the sum-product algorithm (SPA) [7], is a bi-directional iterative message passing process between variable nodes and check nodes based on the Tanner graph. In order to reduce the required number of multiplications, the SPA is normally mapped into the log-domain [8] to achieve a more practical hardware implementation. Successive multiplications are replaced by accumulations, and the signal update calculations are

III. PROPOSED LDPC DECODER ARCHITECTURE In order to support the maximum code size parameters, we arrange the parity check matrix H into 12 sub-matrix rows and 24 sub-matrix columns. Each sub-matrix has 96 rows and columns; two sub-matrix columns are considered to be one processing column, while each sub-matrix row is one processing row, as shown in Figure 1. In order to reduce the initial idle time, we use the layered decoding method starting from sub-matrix column processing. Since the minimal coding rate is 1/2, the number of rows is no more than half of the number of columns.
Figure 1. Row/column processing terminology used in this paper.
A. BNP/CNP Structure A BNP (bit node processor) and CNP (check node
processor) handle variable node updates and check node updates, respectively. A BNP accumulates a new value for the current common column which operates row by row until the current column is completed. As shown in Figure 2, the basic element in the BNP is a (3, 2) compressor with a local register which holds the processed value. The input of the compressor is statically configured by the controller either from the local register or from an external input value. For maximal support of the required coding lengths in these two standards, 12 clock cycles are needed for a column constraint by one BNP.
One CNP compares the two new values in the same row with the previous result to select the minimum value during one clock cycle, and this operation is handled column by

column until the current row is completed. As shown in Figure 3, there are two basic elements for the absolute value calculation, multiplexer A with a subtraction result condition and multiplexer B with an equality condition. The 3-input XOR functions as the basic element for computing the signs, with any empty input positions filled by zeros. There are also local registers for the storage of the processed signs, and for the minimal and the second minimal values. The inputs of both of the multiplexers and the XORs come from controller, and 12 clock cycles are required for a row constraint by one CNP cell.

B. Memory Organization
A hierarchical memory organization is proposed which helps to optimize the data storage and improve the decoding throughput. The 3-level memory structure is composed of local registers in the processing cells (i.e., the BNP/CNP cells), a register file and a main memory. Figure 4 presents the proposed LDPC decoder architecture. We use a total of 2x96 BNP and 96 CNP cells in order to process two submatrices at once, which allows simultaneous row and column updates.
The register file facilitates data transfer between BNPs and CNPs, and holds data for the current processing column (i.e., two sub-matrix columns). We use 7 bits to represent each value in a sign-magnitude format, which was determined to be sufficient from Matlab modeling. Data in the register file will be updated immediately after the current processing column is processed by the BNPs (including variable node pre-processing and post-processing), and the updated values in the register file will then be used in the CNPs for check node pre-processing. Therefore, the data used by the BNPs and CNPs is largely provided by the register file.
The main memory stores the signs of the H matrix values (Si), the minimum (min) and its position (index) of each row, the 2nd minimum (min2), and a flag signal (0/1) to indicate a zero/non-zero sub-matrix ( fij ) , (i=0,…95; j=0,…95), as illustrated in Figure 5. Main memory access occurs when check node post-processing is enabled, that is after all the processing columns have been operated on by BNPs and CNPs within the same decoding iteration.

IV. PERFORMANCE AND COMPARISON
The proposed LDPC decoder has been synthesized and laid out using a UMC 0.18 micron CMOS technology; a clock frequency of 74 MHz has been achieved with a postlayout area of 21.16 mm2. (A reduction in this area could be achieved if SRAM macros were used.) With this architecture, both IEEE 802.11n and WiMax systems can be implemented, as the supported code lengths are from 576 to 2304 bits with a wide range of sub-matrix sizes (from 24 to 96 bits) under various coding rates (1/2, 2/3, 4/5, 5/6). V. CONCLUSIONS
This paper has presented a flexible LDPC decoder which provides support for the codes specified in both IEEE 802.11n and IEEE 802.16e. Using a hierarchical memory structure together with signal routing control, this architecture requires fewer memory bits than other approaches. Our future work will seek to expand the flexibility of the architecture by supporting an even wider range of parity check matrices.
