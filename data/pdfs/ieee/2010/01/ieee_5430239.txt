Common Architecture for Decoding Turbo and LDPC Codes
T. S. V. Gautham, Andrew Thangaraj, Devendra Jalihal Department of Electrical Engineering,
Indian Institute of Technology Madras, Chennai, India 600036. Email: gautham.thasari@gmail.com; andrew,dj@iitm.ac.in

Abstract—Turbo codes and Low Density Parity Check (LDPC) codes have been shown to be practical codes that can approach Shannon capacity in several communication systems. In terms of performance and implementation complexity, LDPC codes and turbo codes are highly comparable, especially at coding rates around 1/2. In many recent wireless standards such as 3GPP LTE and WiMax, both turbo and LDPC codes have been recommended at the encoder. However, the decoder for turbo codes involves trellises and the BCJR algorithm, while the decoder for LDPC codes uses sparse graphs and the message passing algorithm. Therefore, in several implementations, a designer is forced to implement either the turbo decoder or the LDPC decoder. The main idea behind this work is to enable the implementation of both decoders using a common architecture. We view the constituent convolutional code in a turbo code as a block code, and construct a sparse parity check matrix for it. Then, the sparse matrix and the associated bipartite graph are used for decoding the convolutional code by soft message passing algorithms. Simulation results show a manageable degradation in performance with a reduction in complexity.
I. INTRODUCTION
Turbo codes and Low Density Parity Check (LDPC) codes have been shown to be practical codes that can approach Shannon capacity in several communication systems [1][2]. Turbo codes are essentially interleaved concatenated convolutional codes, and their exceptional performance is because of the powerful ‘turbo’ decoder. Turbo decoding is based on trellis soft-decision decoding algorithms that are run iteratively with exchange of extrinsic information. LDPC codes are linear block codes with sparse parity check matrices. These matrices may be represented effectively by a bipartite graph, and LDPC codes are decoded on these graphs using soft message passing algorithms. The bipartite graph is analogous to the trellis of convolutional codes in the sense that it provides a complete representation of the code and aids in the description of the decoding algorithm.
In terms of performance and implementation complexity, LDPC codes and turbo codes are highly comparable, especially at coding rates around 1/2. In many recent wireless standards such as 3GPP LTE and WiMax, both turbo and LDPC codes have been recommended at the encoder [3]. However, the decoder for turbo codes involves trellises and the BCJR algorithm, while the decoder for LDPC codes uses sparse graphs and the message passing algorithm. Therefore, in several implementations, a designer is forced to implement either the turbo decoder or the LDPC decoder. This essentially

means one of the two codes have to be chosen. The main idea behind this work is to enable the implementation of both decoders using a common architecture. The concept in this work is different from LDPC Convolutional codes[4], which presents a class of convolutional codes defined by a low density parity-check matrix. In contrast, we decode regular turbo codes in standards using a LDPC like decoder.
In this paper, we are concerned with developing a common implementation architecture for both turbo codes and LDPC codes. We view the constituent convolutional code in a turbo code as a block code, and construct a sparse parity check matrix for it [5]. Then, the sparse matrix and the associated bipartite graph are used for decoding the convolutional code by soft message passing algorithms. In this manner, both turbo and LDPC decoders involve similar blocks such as bit node processors, check node processors and permuters. We design a common architecture for implementing both the decoders, and study the error-rate performance of turbo codes in this architecture.
For many tail-biting codes, it might be difficult or impossible to construct sparse parity check matrices that are wellsuited for message passing decoding. In cases where length-4 cycles are unavoidable, we propose the use of clustering to reduce their detrimental effect [6]. We show that the clustered decoder performs better in the case of the turbo code in the WiMax standard.
This paper is organized in the following fashion. Section II describes the convolutional Turbo code (CTC) used in the IEEE 802.16m (WiMax) standard, which uses a tail-biting convolutional code (TBCC) as a constituent of the turbo code. Section III describes TBCCs as block codes, discusses the link between TBCCs and quasi cyclic codes (QCC), and provides the construction for a sparse parity-check for a TBCC. The proposed decoder for turbo codes and the common architecture for decoding turbo and LDPC codes are presented in the section IV. Section V shows the simulated error-rate performance results. Section VI summarizes the results and concludes the paper.

In our search for low-weight dual codewords in the row space of H, we found one codeword of weight 4 and a few of weight 7 such that the positions of 1s are within the first 16 positions (the positions are numbered 1,2,3,· · · starting from 1). The weight-4 dual codeword has 1s in the locations {1, 4, 7, 8} and is denoted v4, while one of the weight-7 dual codewords has 1s in the locations {1, 2, 3, 7, 9, 10, 12} and is denoted v7. The length of both vectors is taken to be 4h assuming h > 3. Notice that both these vectors remain valid dual codewords of the TBCC for any value of h > 3.
By the quasi-cyclic property of the matrix H, all cyclic shifts of v4 and v7 by 4 positions are valid dual codewords as well. The h vectors formed by the h distinct cyclic 4-shifts are assembled in a h × 4h matrix denoted H4. It can be readily verified that no two rows of H4 overlap in more than one position, or the Tanner graph of H4 has no 4-cycles.
The matrix H4 provides only h rows, but the complete parity check matrix needs h more rows from the row space of H. We have not succeeded in finding a set of 2h dual codewords that will provide a sparse matrix with no length-4 cycles. Though

IV. DECODING TURBO CODES USING LDPC DECODERS
In the previous section, we saw that the best possible sparse parity check matrix of the TBCC suitable for message passing decoding will contain several length-4 cycles. So, to get improved performance from message-passing decoding, we employ the idea of clustering nodes as suggested in [6]. In this section, we briefly discuss the node-clustered version of the belief propagation algorithm and describe the common architecture to decode turbo and LDPC codes.

A. Clustered belief propagation algorithm
The Belief propagation (BP) decoding converges to MAP solutions for graphs that are trees. For graphs with cycles,

there is no such optimality. The performance of BP decoders depends on the structure of loops in a graph, especially small loops. The problem with small loops can be avoided by using the method of clustering i.e. any loops in a graphical model could be erased by the clustering method. This method collapses multiple nodes into a single node and performs joint processing at the clustered nodes. Enumeration and counting of loops [8] is required before clustering the nodes. In this article, we describe the clustered BP algorithm, where two check nodes in a cycle of length 4 are combined together as a joint check node. The decoders used in this work employ clustering of length-4 cycles only. We further require that the length-4 cycles that are to be clustered do not overlap.
After combining the check nodes involved in length-4 cycles, a resultant Joint Check (JC) [6] bipartite graph is generated. All edges incident on the original check nodes are incident on the joint check node. The Joint-Check Belief propagation (JCBP) decoder is discussed below, very briefly, for completeness.

The set of bits that participate in the m-th check is denoted L (m), and the set of checks in which bit l participates is denoted as M (l). B. Proposed Architecture
The proposed decoder for Turbo codes using LDPC decoders is shown in Fig. 2. This decoder has significant changes in comparison to the standard Turbo decoder. The constituent decoders are based on the JCBP algorithm. The cycles present in the parity check matrix are enumerated [8], and check nodes are clustered to reduce length-4 cycles.
For the turbo code in the WiMax standard with h = 128, we found that the 384 length-4 cycles overlap. So, a set of about 110 cycles were chosen for clustering in a greedy manner to avoid overlap. The notation p∗1 and p∗2 represent parity 1 and parity 2, which improve in reliability with iterations within the decoder. This is in contrast to regular Turbo decoding, where only the systematic bits are updated.
A common frame work to decode both LDPC codes and Turbo codes is presented in Figure 3. The selection of decoding either LDPC or Turbo codes is done using the switches S1 ans S2. The permuters are suitably activated and changed according to LDPC or Turbo decoding. Notice that the bit node and check node processors are reused completely in Turbo decoding through the JCBP algorithm.

In summary, the proposed turbo-JCBP decoder is about 1-2 dB poorer in performance, while using only 60% of the operations needed by a comparable standard turbo-BCJR decoder. Hence, the proposed common architecture provides a useful trade-off between performance and implementation complexity for decoding turbo codes. With further improvements to the performance of turbo-JCBP, either through a better parity check matrix or better decoders, a common architecture for decoding turbo and LDPC codes could become comparable in performance to the standard decoders.
