AbstractтАФ In this paper, we propose a design methodology of low-density parity check (LDPC) code for very high speed transmission. To support various code rates with reasonable hardware and, possibly, the type-II hybrid automatic repeat request (HARQ), a row-splitting/combining method is employed in conjunction with shortening. In row-splitting approach, a parity check matrix of a high rate mother LDPC code, saying rate 5/6, is designed first and then successively row-split the mother matrix to obtain lower rate codes. Shortening provides us another degree of freedom for rate and frame length flexibility for concatenation with MIMO system. The code performance is investigated in terms of frame error rate and compared with those of the LDPC code proposed in IEEE 802.16e standard.
I. INTRODUCTION
Since the first introduction of LDPC code [1] and rediscovery by McKay [2], there were a lot of works for LDPC code design for practical application, especially targeting wireless transmission systems (See [3]-[6] and the references therein). Compared with the turbo codes, two main problems encountered in practical code design were in frame length flexibility and rate compatibility to support type-II hybrid automatic repeat retransmission (HARQ). To provide frame length flexibility, the quasi-cyclic LDPC (QC-LDPC) code, also called as block LDPC (B-LDPC) has been proposed in [7], where the parity-check matrix is constructed by square sub-matrices, which is either zero-matrix or an identity matrix right column shifted. In this structure, it is quite simple to reconfigure the entire parity-check matrix by resizing and shifting the sub-block matrices, providing us a more flexible implementation methodology. To maximize the utilization of wireless channel, adaptive modulation and coding (AMC) is generally considered, for which puncturing is efficiently used for rate compatibility. When using convolutional turbo code, the degradation in performance, compared with that obtained from separate code optimization, is marginal. While, in LDPC code, however, the puncturing does not work well as much as in turbo code making it more desirable to use shortening or to design each rate code separately. In industry, the latter seems to be more preferable as the LDPC code in IEEE 802.11n [8] and 16e standards [9] have been designed

separately for each rate. In addition, the codeword length is fixed for all the code rates, making the information word length necessarily be variable. Such codes do not satisfy rate compatibility and type-II hybrid ARQ cannot be applied. In this paper, we employ a тАШrow-splittingтАЩ method in combination with shortening, to provide high flexibility in frame length and, possibly, to support type-II HARQ. We first design parity check matrix of a high rate mother LDPC code, saying rate 5/6, and then successively row-split and shorten the mother matrix to obtain lower rate codes. Row-splitting more and more, the row degree gets lower while column degree remains the same. The primary advantages of using this type of design strategy are first that the hardware implementation is easy especially when interleaving is implemented by wires and that it may efficiently provide rate compatibility to support type-II hybrid ARQ. The paper is organized as follows; in the next section, we briefly overview the description of QC-LDPC code and the design criteria. We introduce a design methodology based on a combination of row-splitting and shortening in Section III, where a specific design example is also provided. In Section IV, some simulation results are provided to verify the performance of the LDPC codes designed. The concluding remark is given in Section V.
II. DESIGN CRITERIA AND DESCRIPTIONS
A. Design Criteria
One issue that has to be considered in practical LDPC codec design would be throughput and latency. Due to the iterative nature of decoding algorithm, the decoder, for both LDPC and turbo codes, may have unacceptable latency, and sometime unacceptable throughput as well, especially when targeting very high speed transmission, e.g., multiple Giga bps throughput. Large decoding latency not only requires much larger buffer size at both transmitter and receiver but also may affect ARQ complexity and performance. One solution to increase throughput is to use pipelining, where multiple decoding units with low processing speed are used in frame by frame fashion, i.e., as soon as a codeword is received, it is assigned to an idle decoder and a multiple of received frames are processed simultaneously by a multiple decoding units. Unfortunately, however, pipelining cannot reduce the latency unless the processing speed of each decoding units

Most of all, one of the strength of LDPC over turbo codes would be that for LDPC code fully parallel decoder structure can be used. It is also possible for turbo code. But, as reported in [10], full parallelization in turbo code may increase the latency. In implementation point of view, however, parallel decoding algorithm still has a problem. Usually, a single port memory, or at most dual port memory, is used to store intermediate LLR values, in which case throughput is limited by memory accessing time, not by limited computing power. It is the case when targeting very high throughput at reasonable latency. And, according to our estimation, the bottle neck in Giga bps LDPC decoder is in memory access time. In such case, it would be more reasonable solution to use registers and direct connections, instead of memory and bus type connection. That is, we implement interleaving/deinterleaving by wiring directly the local registers that store LLR values. This wire implementation, however, raises another problem when supporting multiple code rates and designing each code separately since in such case decoding unit cannot be reused for different code rate, requiring much more hardware. Accounting this situation, we consider in this paper the row-splitting method, where we design the parity-check matrix of a mother code first and then successively derive other parity-check matrices for all other code rates from the mother code.

III. LDPC CODE DESIGN
When supporting multiple code rates, two possibilities can be considered, i.e., either the codeword length n is fixed or the information word length k is fixed. When n is fixed, the row size of the parity-check matrix H should be variable according to the code rate r. In such case, instead of designing each code of different rate separately, row combining/splitting can be used to support the codes of various code rates in onehardware. Or, at least, a large portion of decoding hardware can be reused for other code rates. This type of code design has been proposed in [11]. To support rate compatibility and incremental redundancy retransmission type (type-II) hybrid ARQ, however, we need to fix the information word size k in which case the size of parity bits n тИТ k should be changed according to the code rate and row-combining/splitting simply cannot be used, In some cases, however, the parity block of the parity-check matrix Hp has a well-defined structure, for example, given by a dual diagonal matrix or similar, so that the resizing of Hp in hardware architecture is much easier than Hs, in which case we may apply row-splitting/combining partially to the systematic block of the parity-check matrix. Difference between combining and splitting is in the order of code design. In row-combining, we design the lowest rate code first and then successively derive higher rate codes, while, in row-splitting we do in opposite order. In [11], the rowcombining has been used to obtain a set of LDPC codes of different rate but with fixed length codeword. In this paper, designing a set of LDPC code of different rate with fixed information word length and row-splitting approach is employed. Shortening technique will also be incorporated for additional frame length flexibility.

IV. SIMULATION RESULTS
Simulation has been performed to verify the effectiveness of the design methodology proposed in this paper. In the simulation, we consider the code of rate 2/3, of which the parity check matrix is given in Table 1, and of rate 1/2, which is derived from the 2/3 rate code according to the rule given in (6). As a benchmark, we compare the block error performance of the two codes with the same rate codes proposed in 802.16e [9]. The detailed code parameters for those proposed in this paper and those in [9] are summarized in Table 2. The BPSK encoded bits were sent through a binary-input AWGN channel. For each corrupted codeword, a maximum of 50 iterations were performed. Each simulation was stopped when 50 block errors were detected. The block error rate (BLER) results are plotted in Fig. 2. As shown in Fig. 2, both the codes of rate 2/3 and 1/2 perform comparably to those proposed in [9]. The result shows the effectiveness of the proposed design methodology.

V. REMARKS
In this paper, we have proposed a row-splitting/combining design methodology for Giga bps LDPC codec. For such high speed transmission, the bottle neck is the memory access time, not the limited computing power. In such situation, the message exchange should be done by direct connection between registers that store intermediate LLR values requiring more complex hardware, especially when supporting adaptive modulation and coding. One way to reduce this increase in hardware complexity would be to design the set of codes of different rates in such a way that the hardware for a code can be reused by other codes of different rate. To support various code rates with reasonable hardware and, possibly, the type-II HARQ, a row-splitting/combining method was used in conjunction with shortening. In row-splitting design, a parity check matrix of a high rate mother LDPC code is designed first and then successively row-split the mother matrix to obtain lower rate codes. Shortening provides us another degree of freedom for rate and frame length flexibility for concatenation with MIMO system. The block error rate performance of the code designed using the proposed methodology was shown to perform almost the same as that of the LDPC code proposed for 802.16e, where the codes of different rates were optimized separately.
