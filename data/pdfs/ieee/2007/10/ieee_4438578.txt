A Reliable Scheme for Cluster Storage System

Xu Li, Changsheng Xie, Qinqi Wei, Qiang Cao Huazhong University of Science & Technology, Wuhan National Laboratory for Optoelectronics, Wuhan, China 430074
lixu_cn@163.com

Abstract
An important issue in the design of large-scale cluster storage systems is their vulnerability to nodes crash and network failures. To explore the feasibility of implementing a reliable, high performance distributed cluster storage system, we adopt a reliable scheme that data are distributed across storage nodes using erasure coding with small Low-Density Parity-Check (LDPC) code. We implement our new scheme in RSC, a proof-of-concept reliable cluster storage system, along with two other well known RAID-1 and RAID-5 schemes. Performance test results show that such a reliable scheme can provide high reliability while keeping the performance overhead small.
Keywords: Erasure code, LDPC, Fault tolerance, Reliability, Cluster storage systems
1. Introduction
With the exponential growth of network bandwidth, storage capacity, and computational resources, distributed storage systems such as the PVFS, Lustre deliver high-bandwidth I/O to applications running on clusters. In these systems, data files are usually split into multiple data stripes across cluster nodes to improve the system scalability and I/O throughput. At the same time, their reliability is far worse than that of smaller storage systems due to the increased number of storage nodes.
Thus an important problem in the design of such striped storage systems is how to reduce their vulnerability due to components’ failures efficiently. The usual approach to building a reliable storage system out of unreliable components is importing redundancy scheme. Traditional method towards redundant storage is full replication, where every block is replicated in a mirror block on another one. The other method is using erasure coding schemes: each data object is divided into m fragments and recoded into n fragments which are stored in distributed nodes.
Low-Density Parity-Check code (LDPC) [1] provides a good solution to this problem because it

allows decoding original data using relatively few and simple XOR parity operations. In contrast to the ReedSolomon code, it does not code the data optimally which require f*m blocks to reconstruct the stored file, where f is an overhead factor that is greater than one, but approaches one as m approaches infinity[2].
In this paper our work focuses on design and implements a high performance fault-tolerant scheme based on small LDPC code. The rest of this paper is organized as follows: Section 2 provides a literature review of related works. In section 3 we present the brief knowledge of LDPC and present an efficient way to generate graphs with good guarantees on their redundancy requirement by utilizing Monte Carlo techniques [3]. The implementation of reliable scheme is shown in Section 4. Our evaluation and measurement results are present in Section 5. We summarize this paper and outline future work in Section 6.
2. Related Work
Distributed redundancy schemes have been extensively studied in a number of striped file system projects [4]. Many distributed storage systems rely only on replication, including PAST, CFS for WANs, and Tertiary disk, Google File System (GFS) for LANs. The RAID-x architecture [5] is a distributed RAID scheme that uses a mirroring technique. These applications need high, sustained bandwidth, replication scheme suffers from the limitation of mirroring.
Erasure codes have been used by several distributed storage systems. Such as Reed-Solomon code, it is widely used for storage systems in WAN and LAN. Since Reed-Solomon code has a significant storage overhead, mostly it is used for archival storage. LDPC code has not been studied as much for storage systems before. Recently there is also some work [6] that researches the suitability of LDPC code for Wide Area Network storage.
We rely solely on erasure coding by using LDPC code which is similar to Gaidioz [7]. In contrast to their work which based on a Peer-to-Peer topology that our

LDPC codes have m + n nodes and are based on bipartite graphs with two sets nodes--the left nodes, l1, l2, . . . , lm, and the right nodes, r1,r2, . . . , rn. Edges only connect left nodes to right nodes. A simple example graph is depicted in Figure 1. From m=4 data words l1, l2, l3, l4 (bits in the simplest case), n=3 coding words r1, r2, r3 are calculated by exclusive-or operation. For example r1 = l1+l3+l4. Encoding the redundant information in the coding words is done in a time growing linear with the number of edges in the graph.

The basic building blocks of RSC are two types of cluster nodes: client compute nodes (client) and storage nodes (server). Compute nodes are used to response the user request and computing tasks. 4.2 Storing and Retrieve Files Each file is split into several data stripes with the
same block size, and then evenly distributed on a set of hosts. Figure 3 shows how the data and redundancy blocks are distributed in the code scheme to prevent data loss in the case of no more than three nodes crashes. The data blocks are labeled L10, L11 etc. and the corresponding coding blocks are labeled R10, R11 etc. The contents of a coding block are parity for specific portions of the corresponding data blocks.
Figure 3. The LDPC code scheme
When reading a file, the client first tries to retrieve the data blocks. Unless some nodes are unreachable or some data blocks are not found, these m fragments are sufficient to reconstruct the file efficiently. In case of failure, the client uses the code graph to find the control blocks required and retrieves them instead.
When writing a file, the involved data blocks are changed, and the corresponding parity blocks should be updated simultaneously. Client checks the offset and size of a write to determine the number of partial stripe writes to be performed.
5. Performance Evaluation 5.1. Experimental Setup
We present an evaluation of the performance of the reliable scheme on RSC platform. We have implemented performance test on 1Gbps Ethernet, the test is carried out on twenty-six PCs that are jointed through IP switch, 16 client nodes and 16 server nodes. The hardware configurations of these PCs are the same: Intel Xeon 2.8GHZ CPU; 1GB RAM; Maxtor 300GB

ATA hard disk. The benchmark tool is the latest version of XDD 6.5.
We compare their read and write performance with other two configurations: RSC with RAID-1 scheme and RAID-5 scheme.

performance are very close when clients number less than 8, but when clients number continue increase the RAID-5 outperform other two schemes. The reason is that RAID-5 has more data servers to support larger read bandwidth than LDPC scheme and RAID-1 scheme when clients’ number added.

However their write performances are in diversity. For example in Fig.4(b), when client number is 6, the aggregate write performance of LDPC, RAID-1 and RAID-5 are 165MB/s, 127MB/s, 257MB/s respectively. The write performance of LDPC scheme is lower than RAID-1, but higher than RAID-5 mode. The LDPC scheme’s aggregate read and write performance increase as the client node number increase to 10, then keep near-constant. It shows that when up to 10 more clients, the aggregate performance is limited as network has reached its utmost bandwidth.

6. Conclusion and Future Work

We have implemented a new reliable scheme for a high performance storage system on a commodity computing cluster. The redundancy of storage files are based on erasure coding with small Low-Density Parity-Check code. The contrastive experiments verify this scheme can provide higher reliability with a low storage and performance overhead than other popular redundancy schemes.

This work presented here is ongoing work and many interesting future research are under further study. (1) Currently our scheme doesn’t support the dynamic scalability. We need improve our scheme to keep working consistent when storage nodes number is changed. (2)We intent to design a more flexible code scheme that can generate different code graph according to the variable reliability requirement. (3)We will implement an efficient cache mechanism to improve the write and reconstruct performance.
