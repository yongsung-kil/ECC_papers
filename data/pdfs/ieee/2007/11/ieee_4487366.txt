A solution for memory collision in semi-parallel FPGA-based LDPC decoder design

Radivoje Zarubica, Stephen G. Wilson Charles E. Brown Department of Electrical and Computer Engineering
University of Virginia, Charlottesville, VA 22904
Research supported by L-3 Communications West, Salt Lake City, UT 84116

Abstract- Low Density Parity Check (LDPC) decoders implementing long blocklength codes require semi-parallel design. One challenge when implementing these codes on Field Programmable Gate Arrays (FPGAs) is efficiently storing messages needed in the iteration process. To meet this challenge, a new class of LDPC codes is presented. They combine regularity of implementation with reduction of time needed for decoding process, and do not suffer of any significant performance loss due to their structure.
I. INTRODUCTION
Low density parity check (LDPC) codes have garnered a great deal of attention in the error control coding field recently, due to the ability to implement soft-decision decoders that perform very near the Shannon bound (for long blocklengths). The decoding algorithm is iterative, using message passing among nodes in a Tanner graph representation of the code, and allows parallel, or semi-parallel processing in the computation to speed decoding.
Already LDPC codes have become part of important standards, including lOGigabit Ethernet over copper pairs, NASA's Deep Space Coding standard, digital video broadcasting (DVB-S2), MIMO-WLAN 802.11n, 802.12, 802.20, and long-haul optical communication systems. Many investigators have studied the design of high-speed LDPC decoders either for custom ASIC implementation [1][2][3][4] or in generalpurpose FPGA's [5][6][7][8]. In this paper the focus is on problems arising in FPGA implementation. Since FPGAs have strictly-determined structure, implementing an LDPC decoder becomes much more efficient if that structure is taken into account. Implementing LDPC codes of longer blocklength in FPGAs requires a semi-parallel design because of limited resources. Semi-parallel designs have issues with memory access which complicates their structure and reduces speed. Here we introduce the special class of codes that avoid these problems, which leads to a faster, less complex decoder.
In the following sections, we give a brief overview of LDPC codes and the message passing algorithm, then proceed to report the problems with the memory access in a semi-parallel design and the ways to overcome them, and at the end we give simulation results.
II. BACKGROUND
Given k message bits, an LDPC encoder constructs a codeword of length n (n > k) code bits, yielding a code with
rate=k/n. The coding scheme relies on n -k overlapping

subsets of the code bits such that the parity of each subset in any codeword is even. These subsets are described by a
(n- k) x n matrix H = [hij], such that hij = 1 iff the i-
th subset has bit j in it. The H matrix is called the parity check matrix of the code and it is very sparse, hence the name "low density parity check" codes. An LDPC code is called regular if the weights of rows and columns in its parity check matrix are equal and is called irregular if not. The parity
check matrix can be visualized as a bipartite graph, referred
to as the Tanner graph of the code. One such graph of a short (16, 8) code is shown in Figure 1. Check nodes correspond to n -k parity checks and variable nodes correspond to n bits of the codeword that is transmitted. The Hamming weights of rows and columns in this representation are called the degrees of check and variable nodes, respectively. A variable node is connected to a check node if the bit corresponding to it participates in the parity check operation of the check node. In the H matrix representation this means that a variable node
j is connected to a check node i iff hij = 1.

III. FPGA MEMORY STRUCTURE
In a semi-parallel design the number of Check Node Processors (CNP's) and Variable Node Processors (VNP's) is smaller than the number of check and variable nodes in a Tanner graph of the implemented code. The ratio of the number of nodes in the graph to the number of implemented processors determines the number of "pages", where the page is considered to be a part of the Tanner graph that is implemented in hardware. The number of nodes implemented in hardware determines the page size. In a semi-parallel design processors operate on a part of the Tanner graph and all the outgoing messages from processors need to be stored in memory. Since our objective is a semi-parallel design suited for an FPGA implementation, the structure of different memories, available on FPGAs, needs to be considered. In an FPGA, messages can be stored in three different ways: block RAM, register memory, and distributed RAM. For example, the block RAM in the Virtex4 FPGA, made by Xilinx, is organized in blocks of 18K bits. The number of such blocks varies, and is 336 for the top of the family (Virtex4-xc4vlx200). This means that if a VNP is of degree three, only outgoing messages from 112 variable nodes could be stored in memory at the same time. This impacts performance since the page size is limited to a small number and implementation of a code of with large blocklength requires a large number of small pages, which ultimately leads to the lower throughput of a decoder. The second way for storing messages is register memory. The resources of register memory are limited and implementation is feasible only if blocklength of a code does not exceed a couple of thousand. The third way of storing messages is distributed memory. The distributed memory is implemented as an array of 4-input look-up tables (Xilinx's Virtex4-family). A look-up table has depth of 16 = 24 (4 is the number of inputs to a look-up table), which means that 16 different one-bit messages could be stored into one look-up table.

FPGA with constrained resources can implement decoders for longer blocklengths, through paging.
In a paged architecture the total message storage is allo-
cated among nd,/P memory banks, each holding P message
locations. Obviously, each such bank can allow only one read (or write) operation per clock cycle. For maximum speed, we wish to be able to read (or write) all the messages associated with a given page simultaneously from the memory banks. This requires careful assignment of edges in the graph to the memory banks, such that memory access collisions do not occur. This will be illustrated using the (16,8) example code whose whose Tanner graph and H matrix are shown in Figures 1 and 2.

IV. MEMORY COLLISION
The message passing algorithm must store up/down messages attached to each edge of the Tanner graph as iterations
progress. The total number of messages is nd, where d, is
the variable node degree. In a fully-parallel decoder, FPGA resources are allocated to each variable and check node and messages are merely latched once per iteration cycle to synchronize the computation, e.g. [8]. In a semi-parallel architecture however, the complete variable node and check node computation is time-shared using P pages, with the variable (check) node processor having resources to process
n/P and (n- k)/P variable (check) nodes respectively. This
time sharing implies a slowdown penalty of at least P, relative to a fully-parallel decoder for the same code. However, a given

Figure 11 helps to understand our method used to design collision-free memory scheme which is also address generator efficient. The matrix in Figure 11 is derived from the matrix in Figure 7. The process starts with the first four columns of the matrix. These columns belong to the first page of the decoder and their labeling is unchanged from from the previous labeling. In the second set of columns, belonging to the second page, the numbers are decreased by 8, the next set of columns gets decreased by 2 x 8, etc. Now all the sets of columns are labeled with numbers from 1 to 8. The trick that actually frees memory of collision is that now all the sets of rows have also labels from 1 to 8. In order to design a collision-free memory scheme for a semiparallel decoder design, the columns in the H matrix first need to be split into P sets of rows and P sets of columns in the H matrix, where P represents the number of pages. Than all the edges in those sets need to be labeled with the numbers from ( = {1, 2, 3, .., E}, where E is the number of edges in one page of the Tanner graph (this design assumes that all the pages have the same number of edges in each page of the Tanner graph). When labeling these sets of edges it needs to be ensured that each set of columns and each set of rows is labeled with the full set (.

The processing of the first page for VNP's requires that all the memory banks send messages from the first address; in the next page, the memory banks send messages from the second address, etc. The processing of the first page for
CNP's requires that the first 1/16 of memory banks send messages from the first address, the next 1/16 of memory
banks send messages from the second address, etc. This is possible because of the block-regularity and significantly reduces the complexity of the address generator, which can be implemented as a set of 16 counters. In the case of the 9984-blocklength code in Figure 12, the number of messages that need to be stored is 3 x 9984 = 29952. If each message is represented with 5 bits, the design would require storage for 149760 bits. Since, 16 of them could be stored in one look-
up table, the entire design would need 149760/16 = 9360
look-up tables for message storage. The number of required look-up tables required for storing the up and down messages in the length-9984 code takes only about 5% of the Virtex4xc4vlx200 FPGA.
