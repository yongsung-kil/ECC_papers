Implementation of a Flexible Encoder for Structured Low-Density Parity-Check Codes
Sunitha Kopparthi and Don M. Gruenbacher
Department of Electrical & Computer Engineering Kansas State University, Manhattan, KS 66506–5204
Email: {sunithak, grue}@ksu.edu

Abstract— The hardware implementation of an encoder for randomly generated low-density parity-check (LDPC) codes requires large area. Using structured LDPC codes decreases the encoding complexity and also provides design flexibility. In this paper, an architecture for implementing an encoder is presented that adheres to the structured LDPC codes defined in the IEEE 802.16e standard. The design methodology is flexible in terms of both the code rate and code length. Results are provided for an implementation on a Stratix FPGA for codes with rates 1/2, 2/3, 3/4 and 5/6, and block lengths ranging from 576-2304. The number of logic elements, clock speed, and throughput of the encoder for the different code lengths are presented. The design achieves encoding rates in excess of 400 Mbps.

Low-density parity-check (LDPC) codes were invented by Gallager [1], [2]. LDPC codes have recently received much attention because of their efficient decoding algorithm, excellent error correcting capability and their performance close to the Shannon limit for large block lengths [3]. They are widely considered as next-generation error correcting codes for many real applications such as telecommunications and magnetic storage. LDPC codes are proposed as an optional code in many IEEE standards. In [4], Europe’s DVB standards group have selected LDPC codes because of their superior performance over turbo codes for next generation digital satellite broadcasting. In [5], it was shown that properly constructed irregular codes can approach the channel capacity more closely than regular codes.
A variety of encoder architectures have been presented in the past. Zhang [6] proposed a systematic efficient encoding scheme by effectively exploiting the sparseness of its (3, k)regular LDPC codes. In [7] and [8], it is suggested that the encoding complexity can be significantly reduced by using an approximate upper triangular parity check matrix to construct LDPC codes. Using this idea, Zhang presented an efficient encoding scheme for generalized LDPC codes in [9]. In [10][12], a hardware design of an efficient LDPC encoder based on Richardson and Urbanke method [8] was described. Recently, an architecture for a structured LDPC encoder/decoder has been presented that supports IEEE 802.11n [13]. This paper provides the implementation details and results of a similar encoder architecture that is also based on structured LDPC codes.

II. BACKGROUND AND OVERVIEW
LDPC codes are linear block codes specified by a parity check matrix, H, which is sparse. There are two types of LDPC codes: regular and irregular. Regular LDPC codes are defined by parity check matrices that have a fixed number of nonzero entries in each row and column; whereas irregular LDPC codes are defined by parity check matrices having a variable number of non-zero entries in each row and column. In this paper, H matrices defined in IEEE 802.16e are used for the implementation of a hardware encoder for LDPC codes [14]. The structure of LDPC codes defined in other standards such as IEEE 802.11n and IEEE 802.22 are similar to that of IEEE 802.16e.
The standard IEEE 802.16e defines LDPC codes as a set of one or more fundamental LDPC codes. Each of the fundamental codes supports code lengths from 576 to 2304 with code rates of 1/2, 2/3, 3/4 and 5/6. The parity check matrix, H, is of size m × n, where m is the number of parity check bits in the code and n is the length of the code. The matrix H is expanded from a base matrix, Hb. The size of Hb is mb × nb where mb = m/z, nb = n/z = 24 and z an integer. A parity check matrix is obtained by replacing each -1 in Hb with a z × z zero matrix, each 0 with an z × z identity matrix, and any element greater than zero is replaced by a z × z permutation matrix. The permutation matrix is an identity matrix that has been circularly right shifted by the associated value specified in Hb. The base matrix Hb can be partitioned into two matrices, Hb1 and Hb2. III. ENCODER IMPLEMENTATION

The encoder implementation is performed in two steps. In the first step, the product of matrices Hb1 and S is computed and is denoted by V . In the second step, the parity bits are computed by solving Eq. 5. These two steps can be performed in a pipeline fashion as shown in figure 1. Initially, V is computed for the first set of information bits. Then the parity bits, P, are computed. During the computation of P for the first set of information bits, V is computed for the second set of information bits simultaneously. This pipeline implementation of the encoder increases the encoder throughput.

The work in this paper is based on parity check matrices defined by the IEEE802.16e standard with code rates of 1/2, 2/3, 3/4, and 5/6 and code lengths which vary from 576 to 2304. A hardware implementation was performed on an Altera Stratix EP1S80F1508C5 FPGA using Quartus II for synthesis. The results are shown in Table I. Variable, z, in Table I represents the size of the sub-matrix in the base matrix, Hb, and is equal to n/24. Column LE in Table I denotes the number of logic elements required for the implementation of the encoder on the FPGA, while CPC represents the number of clock cycles required per codeword for encoding. CPC is equal to the maximum number of clock cycles required for computation of V and P . The computation of V requires z/2 + 3 clock cycles in which z/2 clock cycles are required to compute ep, and three clock cycles are required for loading and processing the data and computing v. Computation of P requires z/2 + 3 clock cycles, in which one clock cycle is used for computing p0, z/2 clock cycles are required to compute c(0,0)p0, and the remaining two clock cycles are required for loading the data and computing the remaining parity bits. Hence this method requires z/2 + 3 clock cycles.
Column Clke in Table I represents the encoder clock. From Table I, it is observed that increasing the code length increases LE’s and CPC. The information throughput, (IT), is equal to m × Clke/CP C. For a particular code rate, throughput is approximately equal to a constant value. For a given code length, an increase in the code rate will also increase the information throughput. These calculations are based on the internal encoder design and not on any special I/O limitations. The encoder implementation assumes that all input data bits are available for encoding, so serialization factors are not included in the results. In order to consider the encoder implementation under serialization, a shift register needs to be added. The throughput thus becomes limited by the speed at which the shift register can run, Clks, which is 422.12 MHz. The latency of the encoder considering I/O serialization, is the maximum value of [m/Clks, CP C/Clke] which is m/Clks. The information throughput of the encoder considering I/O serialization is equal to Clks. Thus, the throughput value is constant for different code lengths and code rates and is equal to 422 Mbps. This value is significantly high when compared to a throughput of 2.6 Mbps obtained when a code length of 2304 with code rate 1/2 is used in [12]. The design methodology accommodates different code lengths and code rates. The encoder design presented can easily fit on FPGA’s and has a significantly high throughput of 422 Mbps.
V. CONCLUSION
This paper provides a general design methodology for the implementation of an LDPC encoder which is flexible in terms of code rate and code length. Parity check matrices defined in the IEEE 802.16e standard are used for the encoder implementation. By using structured LDPC codes, the encoder design is relatively simple and straightforward compared to other complex designs. The design flexibility in code rate and code lengths can be utilized in a real time implementation of LDPC codes in a new technology such as cognitive radio which needs physical reconfigurability.
