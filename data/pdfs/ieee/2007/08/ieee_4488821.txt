Partially-Parallel Irregular LDPC Decoder based on Improved Message Passing Schedule

Xing LI, Kazunori SHIMIZU, Zhen QIU, Takeshi IKENAGA, Satoshi GOTO
Graduate School of Information, Production and Systems, Waseda University 2-7 Hibikino, Wakamatsu-ku, Kitakyushu, Fukuoka 808-0135 Japan Email: xing-li@asagi.waseda.jp

AbstractŌĆö In this paper, we propose a new efficient messagepassing schedule for irregular LPDC code. Our approach is based on the schedule designed for regular LDPC code in Ref.[8]. We have modified the original schedule for regular LDPC code and improved it particularly for the irregular LDPC coder realization. The experimental results show that our method could achieve better performance than conventional one, and improve the converging rate as well.
I. INTRODUCTION
Proposed by Gallager in the early 1960ŌĆÖs, LDPC code was re-discovered by Mackay and Neal [1] in 1996. LDPC code has been recognized as the best error correction code in the near future. The irregular LDPC code (code rate 1/2) proposed by Chung[2] demonstrates only 0.04dB close to Shannon limit at 10ŌłÆ6 BER. Recently, the inherent parallelism of the well-known message-passing algorithm that can realize row and column operations independently for LDPC codes makes it very suitable for LDPC decoder hardware implementation.
There are two kind of parallel LDPC decoders for message-passing algorithm, one is Full-Parallel decoder [3],[4] and the other one is Partial-Parallel decoder. The former one not only occupies less hardware area but also shows better correcting performance than Full-Parallel decoder [5],[6],[7]. For irregular LDPC codes, the schedule proposed in Ref.[8] could not be utilized directly. If we continue to use this schedule originally designed for regular LDPC, there will exist lots of redundant computations in the irregular LDPC decoding. Because the row and column weights in irregular LDPC are not fixed, it is very likely to appear that column operations perform over frequently within some local column blocks of parity check matrix, and most messages are already covered before use as well. The unnecessary computation of column operations for LDPC decoder definitely induces much larger hardware area and more power consumption. In this paper, we will propose a new efficient message-passing schedule for irregular LPDC code. Our approach is based on the schedule designed for regular LDPC code in Ref.[8]. IV. HARDWARE ARCHITECTURE
As shown in Fig.1, the row weights wr of parity check
matrix we use are 7 and 8 respectively. Theoretically the row operation time will be different if the row weight changes.

However the unfixed row operation time will increase the difficulty of control circuit design. It is indeed necessary to unify the row operation execution time. Our decoder utilizes the
pipeline processing (see Fig.2), which is executed every wr +’╝æ
clocks. We use the maximum row weight to unify the row operation execution time. The row operation module will read
messages form beta memory every wrMAX +’╝æclocks. After the
operations row operation module will write the messages into
alpha memory. In Fig.2, we could find that wrMAX column
operations could be accomplished within one row operation time when adjusting the Timing of row and column operation schedules. The row operation will execute one row operation on each row of every sub-block. So there will be twelve CFUs(Check note Function Unit) for the parity check matrix in Fig.1.

The configuration of BFUs(Bit note Function Unit) is also another important issue. In Ref.[8] the BFUs is specifically configured for each column block. From section three we could know that, for irregular LDPC code, the column operations induced by the row operations in each column block differ largely. If we still use the similar configuration with that of Ref.[8], the overload in some local BFUs will be caused while some other BFUs will be too idle. For BFUs, the input ports and computation complexity will differ when the column weights change. The BFUs have fixed hardware architecture for specific column weight. Hence, the BFU could be shared for those column operations with the same column weight. We could realize the column operations on the corresponding array of the BFUs. The BFUs in the array have the same hardware architecture. Using this style, we could guarantee that the amount of BFUs in array is minimal when finishing all the column operations in the given time. The number of BFUs in column operation module depends on the conditions below. As mentioned in section three, several column operations will be executed while each row operation occurs.
