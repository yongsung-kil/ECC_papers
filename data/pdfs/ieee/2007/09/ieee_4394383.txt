ABSTRACT
In this paper we address the problem of flexibility of LDPC decoders in terms of coding rates. The paper introduces a code construction based on structured LDPC codes. The design of the code is performed in close relation with a targeted architecture in order to facilitate implementation at later stage. The proposed solution is based on the transformation of the parity check matrix resulting in a generic representation. This new representation can be used for different codes with distinct coding rates. This concept is illustrated through an example: a multiple-rate decoder is implemented on an hardware prototype and performance results are provided.
I. INTRODUCTION
Low-Density Parity Check (LDPC) codes have raised a significant interest in the scientific community for the past few years and are now increasingly considered for introduction into communication systems and standards (IEEE 802.16e and 802.11n for example). This evolution towards industrialization motivates the design of LDPC codes leading to simple and flexible implementation. In many cases, the system has to support different coding rates, block sizes and modulations.
Despite many progress in the design of LDPC decoders, the implementation of these decoders is not straightforward. One of the main challenges is the possibility of defining lowcomplexity multiple-rate decoder. A good state of the art regarding the architecture of LDPC decoders is summarized in [1]. Belief Propagation decoders are usually defined with two processors: A variable node processor (VNP) and a check node processor (CNP). Parallel architectures [2] are based on parallel VNP and CNP. The computing complexity is highly located in the CNP, since a non linear function intervene. The support of multiple-rate implies a large range of check nodes degrees. Furthermore, supporting larger coding rates comes at the expense of larger check node degrees. A first possibility for the design of multiple-rate decoders consists in designing as much processors as the number of different connection degrees. This method is not appropriate when large ranges of coding rates are required. Another possible method for the design of multiple-rate decoders, consists in dimensioning the processor for the worst case scenario. This method would induce a large complexity for the elementary check node processor, and consequently for the whole decoder. Moreover the hardware ressources are not, with this approach, optimally used when lower rates are considered. The solution presented in [1] and [3] proposes a serial architecture for CNP. A forward recursion computes intermediate results which are combined with the backward calculations to elaborate output messages. As

shown in [1], buffers should be properly dimensioned for the worst case, which may result in a high complexity. The problem can be solved if the design of the code and the associated architecture have been considered jointly. For instance, in [1], both serial CNP and auxiliary processors are employed for the decoding of high rate codes.
In this paper, we propose a method for the design of multiple-rate decoders. First the overall structure of the code is defined. A particular decoding algorithm, which aims at fitting with the code structure, is then investigated. Through a simple concept based on parity check matrix transformations, the problem of a multiple-rate decoder is addressed. Eventually, an hardware implementation of a multi rate decoder is illustrated with complexity and performance evaluations.

II. DEFINITION OF MULTIPLE-RATE STRUCTURED LDPC
CODES
A. Structured LDPC codes
The design of ”hardware efficient” LDPC codes requires some constraints on the code structure. The design of a LDPC code should take into account practical issues related to realization, in order to define a code construction already suited for implementation. Deterministic constructions are potentially good candidates [4]. For long code lengths, random constructions or pseudo-random constructions of irregular LDPC codes have been shown to closely approach the theoretical limits for the Additive White Gaussian Noise (AWGN) channel [5]. On the other hand, in the case of medium LDPC codes size, deterministic constructions can outperform random ones [6]. When finite lengths are considered, some design rules should be respected. One of the most important ones consists in avoiding short cycles in the code graph. A particular attention should be brought to the short cycles involving only degree two variable nodes [7]. In many cases, variable nodes of degree two are connected in a staircase structure. Moreover, degree two variable nodes should be affected to the parity portion of the codeword [7]. Indeed low degree variable nodes converge slower than larger degree variable nodes [8]. It was also demonstrated in [7], the presence of degree two variable nodes is necessary to optimize the irregularity of the code. Intuitively, low degree variable nodes must be present to balance out the presence of the low degree check nodes which have less opportunities to fail as fewer bits are involved in the parity equation.
Therefore a class of structured LDPC codes can be defined using the parity check matrix of size (N − K) × (N ) by:

B. Multiple-rate decoder
Supporting higher coding rates implies higher check nodes degrees (see for exemple Table 1). Dimensioning a system based on the worst case scenario would require a large complexity for the elementary check node processor and consequently for the whole decoder. The problem can be solved by exploiting the concept of equivalent parity check matrix combined with the proposed decoding method. We assume the code of the smallest coding rate has a constant check node degree equal to J0 + 2. J0 denotes the number of ones in matrix H1. An elementary check node processor is thus implemented for this case. In order to decode higher rate codes with the same decoder, the check node degrees of equivalent parity check matrix H1eq should be equal to J0. Therefore a code of higher rate, which is designed with a constant check node degree equal to nJ0 + 2, can be decoded by the same decoder. Indeed the parity check matrix H can be transformed into an equivalent parity check matrix Heq with J0 + 2 ones per row. We have investigated how to decode these different codes (n can take different values) with a single decoder, able to decode one type of outer code (a SPC code with J0 inputs). In other words, the objective was to design a generic multiple-rate decoder.
The parity check matrix transformation allows the conversion of the original concatenation, which can be viewed as a concatenation of a code defined by a trellis (two states trellis) and an outer code (a SPC code with nJ0 inputs), into a new one: an outer code (a SPC code with J0 inputs) and a modified trellis. Thanks to puncturing, the trellis is expanded. The expansion of the trellis is equivalent to the insertion of intermediate bits into the equivalent parity check matrix. These intermediate bits are not transmitted, and can therefore be viewed as punctured bits, or hidden variables. In our case, the proposed method enables to benefit from the inherent flexibility offered by trellis decoding.
Now considering the decoding of this equivalent parity check matrix, it appears that turbo like scheduling is particularly appropriate in this case. If SPC processor are designed for J0 inputs, then all codes with check node degrees equal to nJ0 + 2 can be decoded using the concept of equivalent parity check matrix. The design of a such a multiple-rate decoder requires only one elementary SPC inward/outward processor with J0 inputs and a Forward/Backward processor (see Figure 3). Neutral values (0 if the decoder works on LLRs) have to be inserted into the channel observation frame between each observations ykp. Such an architecture enables the decoding of a large scale of coding rates if the code is well designed (i.e with nJ0 ones per row of H1).
The direct implementation of this principle using a BP algorithm is not very efficient. First the decoding of the equivalent matrix introduces an increase of memory required for the storage of edge messages. The number of these messages is correlated with the number of ones in the parity check matrix to be decoded: memories are dominated by the storage of this values. However the decoding of the equivalent parity check matrix is very useful when a turbo like scheduling is used. First the decoding is realized by windows. IV. IMPLEMENTATION EXAMPLE
A. Multiple-rate code design
The major constraint on the code design is the number J of ones per row in matrix H1. If SPC inward/outward processor has J0 inputs, then for each rate considered, the number of ones per row of H1 should be equal to nJ0. In the case of finite length block size the optimization of the code parameters is not very easy. Moreover it was demonstrated that a code asymptotically optimized is not in all cases better than a non-optimized one, especially when finite length block sizes are considered. Indeed for medium size, an important parameter is the average number of one per row d˜c. For a context with medium block sizes (from 100 to 20 000 bits) our investigations have demonstrated a parameter J0 equals to 4 is a good trade-off. The resulting d˜c is similar to the ones proposed for IEEE 802.16 or DVB-S2 LDPC codes. As an exemple Table 1 denotes parameters d˜c for various well known LDPC codes. The choice of parameter n for each coding rate is proposed and the resulting average check node degree is compared. The choice of a parameter J0 equal to 4, results in a very flexible design for considered block sizes.

V. CONCLUSION
In this paper, we have introduced a class of LDPC codes designed in close relation with a targeted architecture suitable for implementation. This class of code is based on structured LDPC codes which are increasingly used into systems and standards. We have proposed a method to transform a parity check matrix through the introduction of hidden variables. This new representation enables to view the code as a structure amenable to generic decoding of multiple code rates. The interest of this approach has been confirmed by performance evaluations and integration into an hardware platform. The proposed solution solves two aspects often considered as drawbacks of LDPC codes: it enables an efficient implementation and, above all, it introduces a high coding rate flexibility.
