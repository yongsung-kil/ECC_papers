I. INTRODUCTION
Low-density parity-check(LDPC) codes are defined by a very sparse parity check matrices. LDPC codes were first
discovered by Gallager in 1962[11 and were rediscovered and generalized in 1996[2]. These codeS have been shown to perform even closer to the Shannon limit than the well
known Turbo codes[3]. Since their rediscovery there have been attempts to provide efficient hardware implementations [4J[5J[6J[7]. Fully parallel implementations have high throughput, however they are inflexible and unscalable. Serial architectures require less hardware but have low throughput to meet demands of most applications. Partly or semi-parallel designs reuse hardware to reduce cost, but they still have to deal with the issue of memory conflicts and complex control and addressing. Some architectures use structured code design to reduce hardware complexity at the expense of decoding performance. This paper presents a communication conflict-free architecture that permits a wide range of tradeoffs between latency, hardware compleXity and cost, code flexibility and performance. Though communication cost is high, node mapping, addressing, routing and control require minimal hardware and software.

II. LDPC CODES AND MESSAG£ PASSING ALGORITHM
A LDPC code is a linear block code defined by a sparse MxN parity check matrix H[I). It contains j Is per column and k I s per row, called column and row weight respectively. A (j, k)-regular LDPC code has row and column weights of uniform weight, otherwise the code is called irregular. A parity check code can be represented by a bipartite or Tanner graph. The bipartite graph has two sets of nodes called bit and check nodes connected by edges. The M check nodes correspond to the parity constraints and the N bit nodes represents the data symbols of the codeword. An edge in the graph corresponds to a I in the parity check matrix.
A. LDPC Encoding and Decoding
To encode, data is multiplied by the encoding matrix, G,to get the codeword. The matrix G is derived from the parity
check matrix H.
LDPC decoding uses message passing(sum-product) algo­ rithm. T he algorithm estimates codewords by iteratively updat­ ing and exchanging messages between connected bit and check nodes of the bipartite graph. The codeword estimation and message computations are b ased on the initial values/recieved information and received messages from adjacent nodes. Each bit node sends a message to each check node it is connected to, and vice versa. The message passing algorithm is usually implemented in the log domain to simplify computations. Mul­ tiplication operations are converted to additions and division to subtraction in the log domain. A summary of the steps and computations of the algorithm in the log domain is presented in the next subsection.

III. MESSAGE PASSING IMPLEMENTATION
It has been suggested in [2] that the decoder implementation
is straight forward, due to the inherent parallelism of the message passing algorithm. This is true if the hardware directly matches the graph. Each variable and check node is instantiated in hardware and the connection is hard-wired which leads to a fully parallel decoder. Though efficient this approach is fixed and not scalable. The main issues faced when implementing a LOPC architecture are communication complexity, flexibiIity and scalability ,and latency.

2) Flexibility and Scalability: Despite the irregular communication patterns in LOPC it is important to have flexible and scalable architectures. The flexibility (programmability) makes it easier to change some check matrix parameters such as row and column weights and the code rate which may improve the decoding performance. The more flexible the check matrix design is, the better the performance[I][8]. Restricting the flexibility of the check matrix due to communication complexity compromises the potential of the coding scheme. Scalability makes it easier to extend the architecture for larger block codes. Larger blocks codes also result in better decoding performance. Besides efficient hardware implementation, an acceptable level of bit error rate (BER) performance should be maintained.
3} Latency: The above constraints should be optimized to get a cost effective implementation that could produce required applications throughput. LDPC decoding could be used in wireless, magnetic recording systems and optical communications systems. Wireless applications require rates at a few hundreds of Kbls to several Mb/s, storage applications reqnire data rates of about IGb/s, while optical communication rates can be above IOGb/s. Latency is often reduced by parallel processing. The merits and demerits of parallel and serial LDPC implementations are outlined in the next subsections.
A. Parallel Implementation Architecture
A fully parallel decoder architecture directly maps the nodes in the bipartite graph onto message computation units and the edges of the graph onto a network of interconnect. This approach leads to higher throughput and lower power dissipation because of the inherent parallelism. It is however, not scalable to codes with larger block sizes. Area of im­ plementation and interconnect wiring are the major issues in parallel implementations. A hardwired fully parallel design
was shown implemented in [5J using N = 1024, R = 1/2
irregular code with a resulting throughput of I Gbit/s. The design has unique routes for each message which is double the number of graph edges. The wiring occupied 50% of the chip area.

B. Serial Implementation Architecture
Serial implementations require less hardware, support mul­
tiple block sizes and code rates. However, the throughput is limited by the number of processing elements used and
the multiple accesses to memory per decoder iteration results
in high power consumption. In a serial implementation the processing element has to read and write operations for each message in each iteration. The number of messages is double
the number of edges.
The common compromise between serial and parallel im­
plementations is to use semi-parallel implementations, which multiplex computations to a number of processing elements to
reduce hardware and increase speed. Some implementations
go further to reduce communication complexity and cost by
designing architectures for structured codes. Structured codes
put constraints on the placement of 1'5 in the code matrix
to produce a regular or predictable communication pattern between nodes. In the following section we present an edge­ partitioned architecture model that can be varied from fixed
to full flexibility versus hardware cost and allows flexibility in structured code design compared to previous architecture models.
IV. THE PROPOSED ARCHITECTURE
In the proposed arcbitecture nodes edges are partitioned and connected using edge-coloring which ensures that tbere is no
communication conflicts.
A. Avoiding Communication Conflicts
Apart from the random nature of LOPC communication
pattern, there is a problem of communication conflict when
accessing nodes messages. This is because nodes may try to access the same memory block or use the same connection
at the same time. In this architecture model, communication conflicts are avoided by making sure that edges which share a node are in different memory blocks or multiplexors.
An edge coloring of a bipartite graph is a way of coloring all edges in the graph such that no p air of edges with the
same color share a vertex. [n 1916 Konig proved in [9)

that it is possible to properly edge color any bipartite graph with k colors, where k is the highest vertex degree of the graph. Translated to LDPC graph, this means we can divide the graph edges into k groups, where edges in the same group do not share the same destination or source. Each node has at most one edge or connection to each group. This partitioning or grouping ensures that we can never have a node reading/writing more than one message fromlto the same memory block at the same time. The block diagram in Fig. I shows messages/edges grouped in memory blocks numbered I
to k by color. Bit nodes have one edge from each bit-memory
block and one to each check-memory block. Check nodes also have one edge from each check-memory block and one to each
bit-memory block. With many bit and check nodes this model
requires multi-port memories blocks which are complex and
slow. Fig. 2 shows a modified model that uses multiplexors to
provide the same flexibility and functionality anhe memory­ block model. Each node has k mu ltiplexors, one from each
color group. The multiplexors have all the inputs from edges
of the same color, thus allowing a node to read a message
from any node. Reading is done by selecting the appropriate multiplexer input according to the coloring and numbering of edges. Without the edge-coloring partitioning mUltiplexors may have more tbat one message going to a single node resulting in a conflict.
B. Edge-Coloring Partitioning and Mapping of Nodes
An edge-coloring algorithm is first run on the code matrix.
Then edges of the same color are numbered (I to maximum in group). The numbering provides the addresses of the messages within a memory block or selection number to the multiplexer. Mapping larger codes to the architecture is straight forward using the model in Fig. I. Bit nodes execute and store mes­ sages in check-memories, then check nodes read and produce
messages to be stored in bit-memories using edge numbers as
address locations. In the Fig. 2 model, the message memories are inside the
nodes. Bit nodes execute the first fraction of the code and

The edges in each group of 8 are numbered I to 8 in their color group.Thus with 32 nodes, in each color group we have 4 groups with edges numbered I to 8.
C. Architecture Throughput
The amount of parallelism is equal to the number of physical nodes available. Each iteration takes 2 cycles. One cycle for check nodes and one for bit nodes. Taking a parallelization factor of 32, a clock cycle of 10 ns and 20 iterations, the throughput is 80 Mbps. Increasing the parallelization factor to 512 nodes increases the throughput to 1.28 Gbps at the expense of increase in hardware cost. Large paralIelization factors can be combined with structured coding to reduce costs for latency sensitive applications such as magnetic recording systems.

V. CONCLUSIONS
Semi-parallel and scalable LDPC architectures impose con­ straints on the code matrix to extract some parallelism and scalability and avoid communication conflicts. These archi­ tectures trade off error correcting perfonnance for reduced implementation complexity through structured code construc­ tion. The proposed architecture provides the framework that allows the designer to trade off code flexibility, hardware costs and area. The architecture is easily reconfigured from fixed to full f1exibility.!t also offers flexibility in structured code design, easy execution control and node mapping. The ease of scaling combined with flexibility in structured coding allows easy modification from low throughput to high throughput applications or from low power,area sensitive applications to less power, area sensitive applications. Though the commu­ nication hardware is high, the hardware or software required for avoiding conflicts, routing and addressing is minimal. The amount of parallelism depends on the number of physical nodes we can afford to implement which can be up to the code size(N).

D. Hardware Implications
The proposed architecture is very costly in the communica­ tion hardware. Each physical node needs k multiplexors. The size of the multiplexer is determined by the parallelization factor. With S bit nodes and T check nodes, each bit-node
multiplexer (see Fig. 2) has to have T inputs, and each check­
node multiplexer has to have S inputs. The total number of multiplexors is Sk + Tk.
E. Reducing Hardware Costs with Structured Codes
The proposed architecture could be modified to reduce hardware costs through the use of structured codes. Instead of connecting all the node edges to the multiplexer, we can restrict the number of inputs/edges to the multiplexer. This reduces the size of the multiplexer, however, it also reduces the flexibility of the code. The placement of I's in the code matrix is restricted by the physical connections of the nodes. Different connection patterns and connection sizes could be implemented for different edges.
