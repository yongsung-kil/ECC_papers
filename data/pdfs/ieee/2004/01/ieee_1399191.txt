ABSTRACT
This paper studies load imbalance problem in the two stages of belief propagation decoding algorithm for LDPC codes and re-distributes computational load between two stages. To further reduce the critical path delay, new look-up-tables (LUT) are developed to replace both conventional LUTs and data format transformation blocks. The adder trees are also reorganized for speed. This novel approach can reduce the critical path delay by 41.0% with negligible increase in the logic core size. This paper also exploits the similarity between these two stages and derives an area efficient design that re-maps the functional units for these two stages onto the same hardware, which can reduce the logic core size by 10.2% and reduce the critical path delay by 16.2%.
1. INTRODUCTION
Regular quasi-cyclic LDPC codes [1] have been proposed that can achieve comparable and even better performance to random codes [2] recently, as opposed to random construction of LDPC codes. Furthermore, the hardware implementations of LDPC codes lead to significantly simpler memory address generation and wire interconnections. A similar code construction method has also been extended to irregular quasi-cyclic LDPC codes to further improve the decoding performance [3].
Belief propagation (BP) algorithm [4] is usually used for the LDPC codes decoding, where two types of messages are exchanged iteratively along the corresponding Tanner graph, namely, the variable-to-check messages updated by the variable node functional units (VNFU) and the checkto-variable messages updated by the check node functional units (CNFU). In terms of the complexity of the decoder architectures, partly parallel decoder [5] is more practical compared to fully parallel design [6] as it significantly reduces the decoder complexity. In both approaches, the two types of functional units have been implemented with separate hardware.

In this paper, the computational load between the CNFUs and VNFUs is re-distributed. New LUTs are used to replace both the conventional LUTs and the data format transformation blocks between two’s complement and signmagnitude format. The adder trees within each stage are also optimized so that the number of additions in the critical path is reduced from four to two. For the considered example (155, 64, 31) code, this approach reduces the length of the critical path by 41.0% with negligible increase in the logic core size. Also, by observing the similarity between the CNFUs and VNFUs, the CNFUs and VNFUs for the partly parallel decoder are re-mapped onto the same hardware to reduce the overall complexity. Using this remapping approach, the area for the same considered example code is decreased approximately 10.2% while still reducing the length of the critical path by 16.2%.

According to the above decoding process, in one iteration both check node update and variable node update operations have to be performed, one after another. This leads to the merely 50% hardware utilization efficiency (HUE) of the logic core in Fig. 3 because all the VNFUs are idle when CNFUs are busy during the check node update and vice versa during the variable node update. To improve the HUE of logic core, the VNFUs and CNFUs can be re-mapped into the same hardware by making use of similarity between the two stages of BP algorithm. In this way, a smaller area design can be achieved.

5. CONCLUSIONS
In this paper, the computational load between the CNFUs and VNFUs is re-distributed. New look-up-tables are used to replace both the old look-up-tables and the data format transformation blocks (between 2’s complement and SM format). The adder trees in the architectures of both CNFUs and VNFUs are reorganized. For the considered example (155, 64, 31) code, these three critical path reduction techniques reduce the length of the critical path by 41.0% with negligible increase in the logic core size.
Also, by observing the similarity between the two stages of the decoding algorithms, the corresponding processing elements at each stage for the partly parallel decoder are remapped onto the same hardware to reduce the overall complexity. Through this re-mapping technique and the above critical path reduction techniques, the logic core size can be reduced by 10.2% while the length of the critical path can be reduced by 16.2% for the same considered example code.
