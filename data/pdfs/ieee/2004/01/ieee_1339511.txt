Abstract
Low Density Parity Checks (LDPC) are a method of error detection and correction that are able to achieve near Shannon-limit channel communication. LDPC decoders involve a series of computations between two units, the check node and the bit node. In this paper we propose the use of an approximation unit to perform the check node operation. Additionally, we propose a ROM based look-up table (LUT) as a function approximation technique, to be used with an LDPC decoder. The paper shows that a ROM based LUT achieves better performance than using a piecewise linear approximation method to approximate the LDPC computation function. Furthermore, this paper shows that the ROM LUT method can gradually take over as the selected function approximation technique for computationally intensive demanding VLSI designs as the technology shifts to the nanometer era.
1. Introduction
Low Density Parity Check (LDPC) codes are a form of error correction codes similar to Turbo codes, but much more computationally intensive with the advantage that they can achieve near Shannon-limit communication channel capacity [1]. They offer excellent decoding performance and good block error performance. There is almost no error floor and they are decodable in linear time. The most notable advantage of LDPC codes is their suitability for parallel hardware implementation.
An LDPC code is a linear message encoding technique, defined by a very sparse parity check matrix. The message to be sent is encoded using the sparse parity check matrix. When it reaches its destination, it is decoded using another sparse matrix (H-Matrix – Figure 1). The LDPC decoding algorithm consists of a series of intensive computations derived from a message-passing iterative bipartite graph, as shown in Figure 1.

Message passing iterations are performed by the two computation units - the bit node and the check node [2]. Each type of node interacts with a number of other nodes, all of the opposite type, to decode a word. The number of check nodes and bit nodes, as well as how they communicate, is defined by the parity check matrix H, shown also in Figure 1 [7]. The idea is to measure the probability of an encoded bit y having a logic state of 0 or 1, based on the probable values of the other bits in the same word. A sequence of iterations is performed to either converge the probability value towards zero or towards one. The initial probability measurements can be obtained straight through the physical layer (i.e. a continuous analog signal) [7]. Since the probability of a bit being 0 is linearly related to the probability of a bit being 1, a more distinctive term has to be used to converge asymptotically to 0 or to infinity in order to make the distinction clearer. Therefore, the term ‘likelihood ratio’ (lr), a ratio of the probability of a bit in a word being 1, to the probability of it being 0 (i.e. lr = P(data==1) / P(data==0) ) is introduced [2]. It is derived by the well known Bayes’ Rule [2] that gives an explicit relationship between the two probabilities. The idea is that the likelihood ratio can converge asymptotically and non-linearly towards zero or infinity by virtue of the bilinear transform [2].

The check node therefore receives the set of likelihood ratios for bits participating in a check, and takes the bilinear transform of the old ratio to obtain the new ratio (lr_new = (1-lr_old) / (1+ lr_old)). It then converts this back to the new likelihood ratio (lr_new) by applying the bilinear transform again on it and sends the value back to the bit node. The bit node in turn, for the given bit x, assumes that it does not know what check it was and computes an assumed check based on the other checks that bit x participates [2]. It starts with an original likelihood ratio that then it multiplies with the new ones that it receives from each of the check nodes participating in that check, sending it back to the check node. This operation is iterative and eventually after a series of iterations the likelihood ratio converges to either infinity or zero. Bit x (and all other bits participating in a check) are determined, giving a valid word which should be close to the received data [2]. The check node unit is the bottleneck in the LDPC decoding operation in the idea that it has to perform a set of divisions per iteration, complicating the entire process. The bit node on the contrary deals mostly with controlling the iteration count and involves only multiplication and accumulation of the ratios that are simpler to implement. The check node computation unit has to compute a complicated function, and it is desirable to improve this computation. It makes sense therefore, to use appropriate approximation techniques to avoid division in such a way that while it improves speed, it maintains the accuracy of the operation. Two common function approximation techniques can be used effectively with the check node computation; ROM based LUT and piecewise linear approximation. This paper illustrates both implementations and compares and contrasts between the two. It highlights possible benefits for using each technique and shows the disadvantages of using each method as well.
There have been various related efforts in designing hardware for LDPC decoders mainly because of the wide range of applications in which it is employed. LDPC are used in applications such as wireless communications [5, 6, and 7], Error Correction Codes (ECC) for memories [7, 8, 9 and 10], storage devices data exchange [7, 8, 9 and 10] and other communication and interconnect applications. Also, applications where fault tolerance is desired benefit from LDPC codes [6 and 7]. In addition, error control is becoming more and more critical in vital applications, therefore stressing the need for fast, reliable and noise-tolerant error correction codes, properties that LDPC codes achieve. Prior hardware implementations have primarily focused on exploiting the inherent parallelism in operations in LDPC [5, 6, 7 and 12] and providing desired throughput. These architectures address issues such as communication between the nodes, memory efficiency and power efficiency of the implementations [6, 7, 8, 9, 10 and 11]. Other efforts focus [14, 17, 18 and 19] on constructing the decoding sparse matrix in ways to ease the computation. Most of the existing work focuses only on system-level issues rather than component issues. Special attention is focused on achieving better interconnection and communication between the check node and the bit node [5, 7, 8, 9, 11 and 13], improving the speed of the LDPC algorithm, with each work assuming standard hardware implementation for the mathematical operations involved in each unit. However, there is still a need to evaluate appropriate implementation choices of the two primitive operations (check and bit node computations) to enhance performance while keeping the area within a reasonable value. It is equally important to approach the LDPC implementation bottom up as well, exploring issues that occur at the component level instead of only the system level. The FPGA implementation [7] is efficient in dealing with component architecture, but is limited to existing FPGA technology and standard logic cells in order to implement the decoding functions of the bit node and the check node. While standard hardware is fine for

performing the bit node operation however, the check node operation requires specialized hardware in the sense that the operation performed is more complex.
This paper shows that the most relevant part of the check node unit in achieving better performance both in terms of speed and accuracy, is the selection of the mathematical computation hardware. We explore the tradeoffs involved between two alternative ways of implementing the hardware to perform this computation. The paper illustrates the advantages of choosing a ROM based approximation method to implement the function computation performed in the check node, and also raises some interesting observations that can be used in similar designs.
The remainder of this paper is organized in the following manner: Section 2 will detail the check node operation showing why the approximation is important. Section 3 will illustrate the selected implementations, explaining the architecture of the two alternative approximation methods demonstrating the operation of each method and how it affects the computation. Section 4 will compare the hardware performance between the ROM based approximation and a piecewise linear approximation method. Section 5 continues discussing the importance of the results and the implications of using ROMs as the basis for function approximation methods, and Section 6 concludes the report and proposes suggested future work and considerations.

3.2.1 Piecewise Linear Approximation (PWL). PWL approximates a continuous function linearly, using line segments to separate the function in the range required. This allows each particular segment to be approached using a linear equation rather than the function’s actual higher order equation. Given the function B(x), plotting software can be used to graphically segment the function as shown in Figure 3. The objective is to estimate B(x) around x by treating the function section around x as a straight line instead. We divide the function in a number of segments knowing that smaller segments yield better results. Each function segment is approximated using the tangents at the two points that construct each segment using simple geometric triangle properties (Figure 3 – magnified region).
As shown in Figure 3, B(x) becomes linear for each segment and can then be represented by B(x) = ax +b. We only need to store the values of a and b to perform the approximation in hardware. We can use a small LUT to store and retrieve these values. The remaining hardware operations involve the addition and the multiplication in the linear equation. Choosing the tangent gradients as powers of two eliminates the need for multiplication and makes the operation a left shift simplifying the implementation even more. Piecewise linear approximation has its advantages and disadvantages. It is very simple to build in hardware requiring only an adder and a shifter per unit. Based on the number of segments, it needs only a relatively small table to hold the values of a and b. Determining the segment in which x lies can be done upon arrival of x using parallel comparators. The accuracy of the operation is not optimal however. There are only a certain number of distinct values given by the PWL approximation unit, limited by the number of segments. The number of segments is constrained by the range of the operand as well, as we are limited by the available powers of two represented in the operand. In some regions, this translates into noticeable error; in other regions, it translates into little to no error. Since the check node operation is iterative, this affects results emphasizing the need for as many bits of accuracy as possible. In addition, the unit is still relatively slow, as it has to do a comparison, a shift, and an addition, where in a pipelined design this will be a bottleneck. Figure 4 shows the hardware block diagram for the PWL operation. As soon as x arrives the segment that lies is determined and the values of A and B are retrieved. These values are then used to calculate the approximated result as shown in Figure 4.

The operation can potentially happen within a single clock cycle, thus being optimal for a pipelined operation.

3.2.2 ROM-Based Look-Up Table. The alternative implementation is a ROM based LUT. The algorithm is simple – calculate the expected values of B(x) in software using as much precision as desired, and store the results in a ROM with bits from x as the address bits of the ROM. This is expected to provide improved accuracy compared to the PWL implementation, as within the representation range, almost every word is mapped to its respective B(x) calculated value. In theory, the accuracy or a ROM implementation can be 100% but given the limitations of the number representation and the result of the data compression, lower accuracy is expected. ROMs are large and potentially require complicated address decoding logic with the word length becoming a major issue. Larger bit width means larger ROM, and consequently more complicated address decoding logic. In order to resolve these issues, methods such as data compression and shortening the word length were used.

4. ROM vs. PWL comparison

Tables 1 and 2 show relevant comparisons between both implementations and help us make very useful observations. The question of course remains on the area of the ROM. At 4096x15 bits, the ROM is still quite large compared to the PWL implementation

when using the same bit format in both implementations. However, the ROM area decreases exponentially every time we reduce the word length by a bit. Tables 1 and 2 summarize the area, speed, and accuracy comparisons of the PWL and the ROM implementations for all possible representations. The accuracy and the speed of the ROM are better than the PWL accuracy and speed, even as we decrease the number of words in the ROM. Interestingly enough, when both units have roughly the same area, the ROM is still more accurate and faster than the PWL implementation. Given that the major argument against the ROM would be the area difference as shown by tables 1 and 2, this illustrates that the ROM implementation is superior to the PWL. If we consider the same number of operand bit representation, the ROM implementation is up to approximately 17 times larger but upon compressing the ROM data and reducing the number of word bits to 11, we have the same area for the ROM implementation as we do for the PWL implementation with 11 bits. The ROM is still more accurate and faster than the PWL implementation, simply because of the number of iterations required. Tables 1 and 2 also show results for other word lengths as well indicating that the rate of decrease in the area due to the reduction of entries is more than the growth of error in the accuracy. This makes the ROM LUT method a more suitable design method for our approximation. A relevant question is as to why the number of segments in the PWL implementation cannot be increased. The answer to this is simple – increasing the number of segments to a number which is a non-power of two, would require multiplication incurring performance and area penalties. At the same time increasing the number of segments to the next integer power of 2 (32 segments) would require the operand bits to change as well. One can argue however, that as the number of bits per word increases, eventually the increase in the ROM area might become much more of a factor against the ROM implementation, with the PWL accuracy improving at the same time. This might make the ROM inappropriate for a larger number of bits in an operand. In order to obtain the bit error rates, both the ROM implementation and the PWL implementation were modeled in software [3], varying the decoding structure into which the check node could be used every time, and also changing the noise variation encoded in the test message in each case for each implementation.

5. Discussion and Observations
As shown in section 4, the ROM implementation is better than the PWL implementation, simply because when the ROM area is reduced to the same size that the PWL occupies, the performance of the ROM is better than the performance of the PWL approximation. The results also indicate that ROM approximation methods could soon become more appropriate for iterative applications like LDPC, and in particular to implement transcendental functions. Knowing that the function we need to approximate is transcendental, we can then look at ideal methods of approximation. Logarithmic arithmetic units often use ROM-based LUT for all or part of the computation, and clever optimization techniques can greatly reduce the actual ROM size in many cases [16]. Ideally, single-cycle ROM lookups would be implemented in place of iterative or computation-intensive operations like division or reciprocation. ROM tables are often fast, especially when considered with respect to the substituted operation. However, overuse of ROM tables creates unnecessarily large units.
An alternative to ROM LUT are piecewise linear approximations, which combine linear approximations with a very small lookup table. Accuracy with piecewise linear approximations is often acceptable, though not as accurate as the ROM-based LUT [15, 16]. The main theoretical advantage of a piecewise linear unit is its small area. It is easy to see the tradeoff between size and accuracy in a piecewise linear approximation unit; increasing the accuracy requires either an increased number of segments or a less convenient number base (requiring multiplications instead of shifts, for example), or both. [15].
6. Conclusions
It is obvious that a ROM LUT based arithmetic approximation unit has many advantages compared to a similar piecewise linear approximation unit. The bit error rates are comparatively smaller for smaller ROM units, and the area of the ROM decreases at a faster rate compared to the rate of loss of accuracy when we decrease the word length involved in each computation. Also, the ROM unit is faster and allows the check node as a whole to operate faster, resulting in larger overall throughput.
