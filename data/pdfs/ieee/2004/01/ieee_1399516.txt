Structured eIRA Codes∗
Yifei Zhang, William E. Ryan, and Yan Li ECE Dept., University of Arizona, Tucson, AZ 85712 email: [yifeiz,ryan,yanli]@ece.arizona.edu

Abstract— In a recent paper, Yang et al. presented the class of extended irregular repeat-accumulate (eIRA) codes which are efficiently encodable LDPC codes possessing very low error-rate floors and are appropriate for code rates 1/2 or greater. While efficiently encodable, the left-most (n − k)-by-k submatrix of an eIRA code parity-check matrix is random in nature, making efficient decoder implementation problematic. In the present paper, we present structured eIRA codes. We first present some ensemble results for the general class of eIRA codes, after which the subclass of structured eIRA codes is defined. Software- and hardware-based performance results for structured eIRA code designs (rates 0.5 to 0.9) are then presented. Results include a 0.5(2048, 1024) code with a BER floor at 10−9 and a 0.8(5120, 4096) code with a BER floor below 10−11.
I. INTRODUCTION
Random or pseudo-random LDPC code constructions result in an unstructured bipartite graph topology, which makes efficient LDPC encoder and decoder implementations very difficult. Parallel and partially parallel encoder and decoder implementations demand a structured LDPC code parity-check matrix and, hence, a structured bipartite graph topology. Structured LDPC codes include codes based on finite geometries [1], [2], difference sets [3], combinatorics [4], and circulant permutation matrices [5], [6]. In this paper, we propose a design scheme for constructing a class of structured extended irregular repeat-accumulate (eIRA) LDPC codes that not only exhibit very good error-control performance, but also allow an efficient decoder parallelization.
In a previous paper [7], we introduced the class of pseudorandomly constructed eIRA codes which admit high rates, low encoder complexity, and have very low error-rate floors. The eIRA codes are a family of low-density parity-check (LDPC) codes which have the special parity-check matrix form

II. PROPERTIES OF RANDOM EIRA CODE ENSEMBLES
It was recognized in [7] that, unless H1 is properly designed, weight-1 and weight-2 encoder inputs give rise to the lowest-weight codewords. Proper design of H1 means arranging the ones in the columns of H1 so that: (1) they are maximally separated, thus maximizing the parity weight corresponding to weight-1 encoder inputs and (2) the ones in the sums of column-pairs of H1 are maximally separated, thus maximizing the parity weight corresponding to weight-2 encoder inputs. In this section, we will examine the performance of random eIRA code ensembles to support the above design guidelines.
The eIRA code ensembles we consider are those for which H1 has a constant column weight wc (or A has a constant row weight wc) since we have found that such codes provide excellent performance (low floors and good decoding thresholds) at high rates [7]. We will find it convenient to consider the AΠ encoder form in Fig. 1 so that we may use some of the results in the literature which employ an outer encoder, an interleaver, and an accumulator (e.g., [8],[9]).

sum of two adjacent columns within the same column group,
then it is clear that the resulting accumulator output, p2, will
have low weight. In view of the above difficulty, we replace AT with P
instead of replacing H1 with P so that from (3)

Our goal in doing so is to achieve a larger minimum codeword
weight — at least for codewords corresponding to weight-1
and weight-2 inputs — via proper choice of the permutation matrix ΠT (interleaver Π). We would simultaneously like to impart an organized structure in H, which constrains Π to be
deterministic rather than (pseudo-)random. Thus, we choose Π so that rows 0, 1, ..., L − 1 of H1 = ΠT P are, respectively, rows 0, Q, ..., (L − 1)Q of P; rows L, L + 1, ..., 2L − 1 of H1 are, respectively, rows 1, Q + 1, ..., (L − 1)Q + 1 of P; rows 2L, 2L+1, ..., 3L−1 of H1 are rows 2, Q+2, ..., (L−1)Q+2 of P; and so on. In general, row qL + l in row group q of H1 is row lQ + q in row group l of P, where 0 ≤ l < L and 0 ≤ q < Q. Observe that this choice of interleaver has the effect of spreading the 1’s in the word p1 for weight-2 inputs.

IV. CODE DESIGNS AND PERFORMANCE
While the algebraic constructions such as those in [1] and [5] generally have limited flexibility in the choice of length and rate, construction of structured eIRA codes is much more flexible. Given code length n, rate R, and girth g, the first step is to determine the values of L, J, Q, and the initial form of the base matrix P defined in (11). If masking is used, then set the masked bl,j’s to ∞. The masking technique used in this paper is cyclic masking [12]. We randomly selected the masking pattern for the first column of the base matrix P. Then the masking patterns for the following L − 1 columns are generated by cyclically shifting the pattern for the first column. It is obvious that the masking pattern will repeat after shifting L times. Hence, we usually change the masking pattern after masking L columns, that is, randomly select a masking pattern which is different from any of the previous patterns (including the shifted ones). Then we get the initial base matrix. According to it, we randomly generate integers to replace the unmasked bl,j’s until certain girth conditions are satisfied.
As noted above and in [7], encoding is easily performed via the H matrix given the form of (13) and (2). The H matrix in this form may also be used at the decoder. However, the pixelated image of H (in which a pixel in an m × n array of pixels is darkened for each 1 in H) fails to display much structure. Example 2: Consider a rate-1/2 (2048, 1024) eIRA code with L = 16, J = 16, Q = 64. Cyclic masking is used for both codes to set the column weight of H1 to 5. The hardware [15] and software simulation results are shown in

and information parts of the codeword, whereas we count
them only in the information part for both BER and FER.
The hardware simulator also employed 200 decoder iterations
whereas we employed 50. Observe that the hardware curve reaches a near-zero-slope for at BER=10−9. We attribute this to trapping sets [16] rather than a small dmin.
Example 3: In this example, two groups of eIRA codes are presented. They both have information bit length k about 4096, but the rates are 0.9 and 0.8. For the rate 0.9 structured eIRA code, we choose L = 5, J = 46, Q = 91 and obtain a (4551, 4096) code by deleting J ·Q−k = 46× 91−4096 = 90 columns of ΠT P from the right. The (4550, 4095) eIRA code was constructed pseudo-randomly by the method in [7]. The
parity-check matrices of these two rate-0.9 codes have the
same weight distribution. For the rate 0.8 structured eIRA code with k = 4096 and n = 5120, we choose L = 8, J = 32, Q = 128 and use the cyclic masking technique to change the column weight of H1 to 5. A pseudo-random eIRA code with the same k, n, and weight distribution was also constructed by the method in [7]. Only the structured (5120, 4096) eIRA code
was simulated by both software and hardware. The hardware

simulation has a maximum of 200 decoder iterations. All of the simulation results are shown in Fig. 6. From the results, we notice that the structured eIRA codes perform as well as the pseudo-random eIRA codes in the simulated region. For all of the above codes, there is no error floor down to BER ≈ 10−8. We remark that, in addition to Fig. 6, only one codeword error occurred in approximately 730 million simulated codewords at an Eb/N0 of 3.67dB.
