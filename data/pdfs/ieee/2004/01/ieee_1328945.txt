POWER EFFICIENT ARCHITECTURE FOR (3,6)-REGULAR LOW-DENSITY PARITY-CHECK CODE DECODER
Yijun Li, Mahmoud Elassal and Magdy Bayoumi
The Center for Advanced Computer Studies University of Louisiana at Lafayette, Lafayette, LA, 70504, USA
{yxl4444,mhe1362,mab}@cacs.louisiana.edu

ABSTRACT Most of the current LDPC docoder VLSI architecture research are focusing on increasing system throughput or reducing hardware implementation complexity, but neglecting power consumption. In this paper, we analysize the power consumption of the (3,k)-regular LDPC decoder architecture [1]. Our analysis shows that 95% of the power consumption is consumed in accessing the memory. A new architecture is proposed which reduces memory access, hence power consumption, without sacrifying the performance. Experimental results show reduction in the power consumption by 14% and lower hardware complexity without sacrifying the Bit-Error-Ratio performance compared to that of [1].
1. INTRODUCTION
Recently, Low-Density Parity-Check (LDPC) codes[2] have widely been considered as a promising error-correcting coding scheme for many real applications in telecommunications and magnetic storage, because of their superior performance and suitability for hardware implementation. Most of the current LDPC decoder architecture research are focusing on increasing throughput or reducing implementation complexity, neglecting power analysis. In fact, power comsumption presents a critical issue in computing, particularly in portable and mobile platforms, because of battery life and power dissipation. Designing a practical architecture must consider the trade-off among throughput, power consumption and hardware complexity.
LDPC decoder architecture can be implemented in parallel message passing and/or serial message passing. In the parallel decoder architecture [3], the nodes in the bipartite graph are directly mapped into message computation units and edges of the graph are mapped into network of interconnects. The parallel architecture achieves high throughput at the cost of interconnect complexity. In the serialized architecture[4], a fully pipelined implementation with two memory buffers per stage, alternating between read/write, was introduced. In [5], hardware scaling and memory partitioning were used to achieve high throughput at the cost

In [6], a joint code decoder design approach was adapted to construct a class of (3,k)-regular LDPC code and a partly parallel decoder architecture was proposed to reduce the hardware complexity and achieve resonable throughput. In [7], less complex interconnection network was used to achieve a low power design.
In this paper, power analysis for (3,k)-regular LDPC decoder architecture is provided and analysis result shows that 95% of the power consumption is consumed in the read/write memory operations. Following the power analysis, a new low power architecture for (3,6)-Regular LDPC Decoder is proposed. The proposed architecture reduces the read/write memory operation, implements computation components more efficiently, and has less interconnect complexity. Our results show that the proposed architecture reduces the power consumption by 14%, and moreover have lower hardware complexity without sacrifying the performance compared to that of [1].
The rest of the paper is organized as follows. Section 2 briefly introduces (3,k)-regular LDPC code. In Section 3, power analysis for the architecture of [1] is provided and our architecture is proposed. Section 4 shows the experimental results.

Since every memory bank has 256 elements, it must take 256 cycles to guarantee that all messages in H are already updated by VNCs completely. For the check node side, each of the 6 Check Node Components (CNCs), takes 6 inputs from six memory banks that are all located in either H1 or H2 or H3. The CNC components perform computations as shown in Equation 2 and writes back outputs (i.e. updated messages) into the same locations of the six memory banks where inputs were taken from. Since 6 CNCs can only operate on either H1 or H2 or H3, the 18 CNCs are used to make sure that it take 256 cycles before all messages are updated by CNCs completely.
Using ASIC methodology and 0.35µm technology, the proposed architecture is implemented. The power consumption results are shown in Table 1. The results show that 95% of the power consumption is consumed in memory access. Hence, reducing the number of memory access may result into a significant power reduction. In [1], for every computation, either variable nodes or check nodes need to read inputs from memory and write outputs back into memory. From an overview of (3,k)-regular LDPC code given in Section 2, H1 is composed of identity matrix Ii, j and H1 shuffle network can be implemented as an identity shuffle network (i.e. without any shuffle). After the 36 VNCs finish their computation on the 36 messages from H1, the messages required by the 6 CNCs are updated. The 6 CNCs could start computation without waiting 256 cycles, because of the regularity of (3,6)-regular LDPC codes and identity matrix of H1. In other words, messages for H1 from memory can go through both variable nodes and check nodes before they are written back into memory. However, messages for either H2 or H3 from memory cannot go through both variable nodes and check nodes before written back into memory, because H2, H3 and their shuffle networks are not identity.

The proposed architecture’s work mode is explained as follows: In the first 256 cycles, each of the 36 VNCs takes three inputs from three memory banks that are distributed in H1, H2 and H3 seperately, i.e.RAMi,1, RAMi,2, RAMi,3. Each VNC component produces 3 outputs. Two outputs from the VNC are connected to H2 and H3 and writen back into RAMi,2 and RAMi,3. The third output is connected to CNC directly, instead of writing into RAMi,1 as [1], as shown in Fig. 4. The 6 CNCs compute for these 36 updated messages from the 36 VNCs. The outputs of the 6 CNC will be writen back into RAMi,1, i = 1, ..., 36, saving almost half of write and read access to RAMi,1. In the second 256 cycles, CNCs are only used to calculate the updated messages from H2 part and H3 part, i.e. RAMi,2 and RAMi,3, since H1 computation is already finished in the first 256 cycles. Therefore the same 6 CNCs can work at the first 256 cycles and the second 256 cycles without any conflict. The total number of CNCs needed in our architecture is 12 CNCs instead of 18 CNCs, with extra small MUX overhead, saving power cosumption and hardware area. Address control units determine memory block both VNCs and CNCs access. The shuffle networks perfom like in [1], as discribed in Section 2.
3.1. Variable Node and Check Node
The implementation of the variable node (VNC) is shown in Fig. 5, where the most significant bit (MSB) determines hard decision output and binary number conversion between sign magnitude (SM) and two’s complement (Two) is done by using SM-to-Two and Two-to-SM. By adding the Priori value first, we can reduce the number of adders in the VNC. In each VNC component, only 6 adders are used, instead of 8 adders in [1]. In order to obtain less area and make efficient utilization of hardware, we move the look-up-table (LUT) parts from VNC to CNC. Since some CNCs can be used in both the first 256 cycles and the second 256 cycles, moving LUTs into CNCs is more efficient.

5. CONCLUSION
In this paper, we analysize power consumption of (3,k)regular LDPC decoder architecture[1]. The analysis results motivate us to propose a new architecture, which reduces memory access and makes more efficient implementation. Experimental results show reduction in the power consumption by 14% and lower hardware complexity without sacrifying BER performance compared to that in [1].

4. EXPERIMENTAL RESULTS

The proposed architecture is prototyped following standard cell ASIC methodology. Verilog model was designed as a HDL for the complete architecture and synthesized using Synopsys tool. The architecture power consumption was estimated using 0.35µm technology. Architecture in [1] was also prototyped using the same flow. The power consumption results of the proposed architecture is compared against the results of the architecture in [1], as shown in Table 1. The proposed architecture’s 14% drop in power consumption could attribute to the following proposed enhancements: reducing the number of memory access; moving the LUT into CNCs from VNCs; using the parallel adder tree in CNCs.
