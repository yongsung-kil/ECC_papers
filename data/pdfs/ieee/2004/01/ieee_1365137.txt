Abstract — It is well-established “folk knowledge” that in order to be iteratively decodable, a code should have a sparse parity-check matrix. But exactly how sparse does it need to be and why? We provide some qualitative and quantitative answers to this question, for finite-length codes.

I. INTRODUCTION
There seems to be preciously little work in the existing literature related to the question in the abstract. It is shown in [1] that if a binary linear code is representable by Tanner graph without cycles, then it must have an extremely sparse and structured parity-check matrix. On the other hand, Sason and Urbanke [3] show that in order to approach channel capacity with vanishing bit error probability, ensembles of LDPC codes should not have too sparse parity-check matrices.
In this work, we show that the message-update equations in the sum-product algorithm can be readily interpreted to mean that parity-checks of large weight essentially do not affect any of the messages. We then proceed to analyze a version of Gallager’s decoding algorithm [2, p. 39], applied to finite-length linear block codes. We find that, for a given number of erroneous code positions, under this algorithm, parity-check equations can be classified according to their weights into those that contribute to error correction, and those that do not. Surprisingly, whether a given parity-check is useful in this way does not necessarily depend on its weight in a monotonic fashion.
