SCALEABLE CHECK NODE CENTRIC ARCHITECTURE FOR LDPC DECODER Rohit Singhal, Gwan S. Choi, Nathan Mickler and Prabhavati Koteeswaran Electrical Engineering, Texas A & M University, College Station, TX-77840

ABSTRACT
Low density parity check codes are a popular class of linear block codes for forward error correction in communication channels. Recent years have seen a lot of work towards the development of decoding architectures for these codes. The architectures range from completely parallel to completely serial. While the parallel architectures have a high throughput, they have a large hardware resource requirement. On the other hand, although the serial architectures are very efficient in terms of hardware requirement, they suffer from low throughput. This paper presents a novel scalable check node centric architecture with a 1.5Gbps throughput. The throughput may be further increased by using more readily scalable data-paths which have a individual throughput of 0.5Gbps.
1. INTRODUCTION AND BACKGROUND
The fidelity of transmission in most modern digital communication systems is maintained by using a variety of channel codes for forward error correction. Turbo codes [1] are perhaps the most popular code for error correction and come very close to the Shannon’s limit. Rivaling the performance of turbo codes, in 1962 Gallager [2-3] proposed a class of linear block codes called the Low Density Parity Check (LDPC) codes. An LDPC code can be defined uniquely by a parity check matrix H or by a bipartite graph. If k is the length of the input data block and if n is the size of the code word, then the rate of the code is R = k/n. Also, the size of H is then (n − k) × n. The H of a regular LDPC code can be constructed randomly using the following rules.
• Each column contains j ≥ 3 ones. • Each row contains ρ > j ones. The code is termed an (n, j, ρ) code. The n bit long code words are formed with the linear combination of the k input bits. Let C be the set of n bit long code words, then all vectors x ∈ C satisfy the relation H.xT = O, where O is a vector having n − k zeros. An H matrix can be seen as a bipartite graph. One side of the bipartite graph contains n Variable Nodes (VN) while the other side contains n − k Check Nodes (CN) representing the columns and rows of the H matrix respectively. A 1 on the rth row and the cth column of the H matrix shows the presence of a bidirectional edge between the rth VN and the cth CN. It follows that each VN is connected to exactly j CN, while each CN is connected to exactly ρ VN. The exchange of information, or message passing, along these edges in the graph is explained in the following paragraphs. Several papers showing the implementation of the LDPC decoder have been presented recently. The architectures range from a completely serial architecture to a fully parallel architecture [4-5].

The serial architecture has the advantage of lower hardware requirement. It however, suffers from low throughput. On the other hand, although the parallel architecture permits high throughput, the hardware resource requirements may be prohibitive. Several other approaches that combine advantages of both serial and parallel architectures have also been developed [6-9].
The above mentioned architectures employ an iterative message passing algorithm, where messages are passed sequentially form the VN to the CN, and then to the next VN and so on. The decoder achieves a better estimate of the transmitted bits after each iteration. In other words the decoding procedure consists of several VN stages interleaved with an equal number of CN stages as shown in fig. 1.

1.1. Motivation

Barring the exception of the fully parallel architecture, almost all other architectures split up the ρ components of the CN while keeping the VN intact. Some FIFO buffers are also inserted to maintain the scheduling sequence in the data-path. Also, in these

architectures, the processing sequence is based on the sequence of VNs. In contrast to this approach, this paper discusses an architecture based on splitting the VN instead of the CN. The motivation behind this is multifold.
• In any LDPC code there are more VN than CN. A timing cycle based on the sequence of CN will therefore be completed faster than that based on the arrangement of the VN. Also, the CN have more information content than the VN.
• This arrangement also moves the buffers from the center of the data path to the edge, resulting in simpler hardware.
The rest of this paper is organized as follows. Section 2 describes the novel data-path. Section 3 gives the complete architecture with timing and control sequences. Section 4 discusses the optimality of the design. Finally, section 5 states the conclusion.

2.2. Variable Node
This module has 3 4-bit words (2 Lcb and 1 Lch) as inputs and it generates a 4-bit output (Lbc) according to the equations given above. Even though, the VN is supposed to have 4 values as inputs, this architecture has only 3 inputs. This happens because we have split the VN into j = 3 parts, and for each part, only 3 inputs (2 Lcb and 1 Lch) are required. The control hardware therefore blocks the 1 irrelevant Lcb from reaching the VN. The VN module, like the CN, is a sign-magnitude saturation adder.
2.3. Combining the Small Parts
The ψ and ψ−1 circuits are two level combinatorial circuits. All these circuits are combined to form a data-path as shown in fig. 4. Each CN has values coming in from exactly 6 VN, through a ψ function module. The check node in turn passes back updated values to the same variable nodes through the ψ−1 block.
3. FLOW CONTROL AND PARALLELIZATION
We use a regular H matrix, with j = 3 and ρ = 6. As a result our H matrix can be broken down into 3 × 6 = 18 parts such that each of the small parts has 170 rows and 170 columns. Also, each row and each column in these smaller matrices have exactly one 1 in it, all the other elements are 0. Without further modifications on the matrix, simultaneous updates on 3 check nodes can therefore be performed instantly.

3.1. Input Control
The input control circuit is an array of registers followed by an array of transmission gates. These two parts working in tandem, take the timing signals from the counter, and completely control the flow of signals through the circuit. Since there are 510 CN, the counter counts in cycles of 170 for our 3-way parallel design, covering all the CN 3 at a time. The registers are designed to store the LLR values for each of the 1020 VN. Each VN has 4 LLR values associated with it, namely 3 Lcb and 1 Lch. These values have a 4-bit precision. Therefore there is a need for 1020×4×4 = 16230 memory cells.
This memory structure is followed by an array of transmission gates. Based on the timing signals, the circuit determines the CN being processed in each data-path and the corresponding 6 VN. The appropriate LLR corresponding to each VN are passed into the data-paths. Each of the 6 VN therefore has an incoming 16 bit long sequence. These 16 bit values are further trimmed into 12 bits, as discussed in sec. 2.2, depending on the data-path the values are transmitted into. For example, out of Lcb1, Lcb2, Lcb3,

3.2. Output Control
The output control is similar to the input control in its construction and complements it in the flow control functionality. It is an array of transmission gates preceding an array of registers. There are 6 LLR from each data-path. The transmission gates, activated by the timing signals, direct the incoming data into a pre-designated memory cell.

The delay in the data path from the VN to the output of ψ−1 circuit is estimated at about 3.8ns using the 0.25µm CMOS technology. This allows a clock period of 4ns. 5. CONCLUSION
We have presented a novel scalable check node centric data-path capable of producing a throughput of 0.5 Gbps. A complete architecture using 3 such data-paths is also presented resulting in a 1.5 Gbps throughput while using only 275740 switches and 569400 memory cells. Although the architecture is based on a (1020,3,6) code, the discussion can be easily extended to any other regular LDPC code.
