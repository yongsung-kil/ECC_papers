Abstract—This paper introduces a novel concept for exploiting processing in-memory (PiM) hardware decoding low-density parity-check (LDPC) codes. The PiM hardware can be used to mitigate the high cost of data movement in modern computing systems by placing computing units near memory where data resides. The memory hierarchy and the parallel nature of the limited arithmetic units allow the multithreaded-based parallelization of the LDPC decoder inside each data processing unit (DPU) and the simultaneous launch of multiple LDPC decoders. The experiments confirm a speedup of 1155×, compared to the baseline, a sequential high-end CPU, for the binary LDPC decoder of CCSDS-231 standard reaching 742 Mbps of global throughput performance in-memory.
I. INTRODUCTION
Low-density parity-check (LDPC) codes are incorporated in several applications such as communications [1], [2], storage [3], [4], and digital video broadcasting [5], [6]. However, developing these LDPC decoders poses a substantial challenge due to their high computational complexity. For several years, researchers have proposed parallel implementations of these decoders in central processing units (CPUs) [7], [8], graphics processing units (GPUs) [9], [10], field-programmable gate arrays (FPGAs) [11], [12] and application-specific integratedcircuits (ASICs) [13], [14] to increase throughput performance and comply with strict latency targets. However, the partialparallel nature of these decoders still relies upon the calculation of partial results, increasing data movement between computing units and memory.
In the past decades, the development of computing units has followed Moore’s law, where the number of transistors doubled every 18 months for the same area. However, memory technology has not kept the same rate of progress as processing units, causing a data movement bottleneck [15], [16], where a large portion of the execution time and energy is spent moving data between memory and processing units. A recent study reports that up to 62% of the system energy is used for data movement [17].
New challenges in dynamic random-access memory (DRAM) scaling (increasing density and performance while maintaining reliability, latency, and energy consumption) have

These new advances in memory architecture allowed a paradigm shift away from a processor-centric design to processing-in-memory (PiM) [20], [21], where memory chips are designed with processing capabilities in order to alleviate the data movement bottleneck [15], [16], [20].
The UPMEM PiM architecture is the first commercial realization of a PiM system implemented in hardware [22], [23]. UPMEM uses the more mature fabrication and design of 2D DRAM to avoid the limitations of current memory systems [22], [23]. Furthermore, the UPMEM architecture allows up to 24 independent threads to perform operations in each DRAM processing unit (DPU), which can be viewed as an independent processing unit with dedicated memory.
This paper proposes the following contributions:
• The first PiM-based LDPC decoder implementation, to the best of our knowledge.
• An efficient mapping of the LDPC decoder to the UPMEM architecture, exploiting the system’s advantages and avoiding its drawbacks and limitations.

II. LDPC CODES
The LDPC decoding process works by applying calculations over the probability of a received bit being correct. An LDPC code can be represented by a Tanner Graph [24] and is defined by a sparse parity check matrix H, which determines the number of check nodes (CNs) (M ) and bit nodes (BNs) (N ) and the connections between them (Sv(m) and Sc(n) defines the set of BNs/CNs connected to each CN/BN). For instance, (1) defines an LDPC code with six BNs and three CNs. The non-zero elements of (1) dictate the connections between CNs and BNs, as illustrated in Fig. 1. This code has a CN degree of four (dc = 4) and a BN degree of two (dv = 2), meaning that each CN connects to four BNs and each BN connects to two CNs.

III. UPMEM ARCHITECTURE
The general UPMEM architecture is described in Fig. 2. The system is controlled by a host CPU that uses DRAM memory and PiM-enabled memory through standard DDR4-2400 DIMM slots. The system features 20 PiM-enabled DIMMs, working at 350 MHz, with each DIMM containing 16 PiM chips. In turn, each PiM chip contains eight DPUs (represented in the right-hand side of Fig. 2), totaling 2560 DPUs in the whole system [22].
Each DPU has 64 MB of DRAM, or MRAM, a faster 64 KB scratchpad-based static random-access memory (SRAM) called WRAM, and 24 KB of instruction RAM. It also features a 14-stage processor pipeline, allowing up to 24 threads

IV. PIM-BASED LDPC DECODERS
The proposed decoder implementation exploits available memory of the faster SRAM-based WRAM to save the decoder Tanner graph, the intermediate variables and results, and the codewords as shown in Fig. 3. Different codewords are sent for each DPU WRAM from the host CPU. The codewords are processed following Algorithm 1, with the workload (BN and

CN processing, the a-posteriori information computation, and tentative decision) distributed over 16 threads. After the decoding process completes, the decoded codewords are returned to the host CPU. V. EXPERIMENTAL RESULTS
A. Experimental apparatus
The experimental results presented in this work use binary LDPC codes of sizes (128, 64), (256, 128), and (512, 256) recommended by CCSDS-231 [2] running five decoding iterations. This work is divided into two approaches: The single codeword decoder, where a CPU baseline is established and is compared to several UPMEM implementations with incremental optimizations, as depicted in Fig. 4 (the rightmost bar in Fig. 4 represents the multiple codeword decoder for comparison against the baseline). The second approach selects the best UPMEM single-decoder implementation, replicates it over the available DPUs, and compares it against lowpower and desktop GPUs, namely the Jetson Nano, Jetson TX2, Jetson AGX Xavier, Jetson AGX Orin, RTX 2080 Ti, RTX 3090, and the RTX 4090, with decoders implemented in CUDA, as depicted in Fig. 5.
B. Single-decoder results
Fig. 4 shows a baseline performance between 569 and 675 Kbps for a sequential Intel Xeon Silver 4110 CPU. The first implementation of the UPMEM decoder uses floating-point representation, MRAM, one thread, and one DPU, resulting in low performance due to using the slower DRAM-based MRAM and the UPMEM system not having native floatingpoint support and instead emulating it using integer hardware [23], only reaching 2 Kbps.

The following implementations use incremental optimizations where only one variable (from data representation, memory type, number of threads, code optimizations) is changed. The implementation was improved by using int32 representation, allowing the UPMEM system to use the integer hardware directly without emulation, reaching 6 Kbps.
Next, the variables and the Tanner graph of the decoder were saved in the faster SRAM-based WRAM scratchpad memory instead of MRAM DRAM, allowing the performance to reach up to 12 Kbps. The highest increase in performance was obtained by implementing multithreading. By dividing the workload between 16 threads, a performance between 88 and 116 Kbps was achieved.
The following three implementations use structural code changes to improve performance. The first unrolls the inner loops (edges connected to each BN/CN), increasing performance to 161 Kbps. The second change implements a custom abs() function [25] that performs better than the standard abs() function provided by UPMEM, improving the decoder performance to 166 Kbps. The third improvement makes use of the -O2 flag, reaching up to 183, 264, and 342 Kbps for the code sizes (128, 64), (256, 128), and (512, 256), respectively, and represents the best single-decoder performance.
C. Multiple decoders results
Compared to the sequential CPU baseline, the best singledecoder implementation achieves a speedup between 0.271 and 0.532, due to a slower clock frequency. However, the UPMEM’s level of native parallelism is higher than that of CPUs by providing more parallel computing cores. From the observed data, the UPMEM system is comparable to a Jetson Xavier or Orin in terms of performance. Furthermore, it is shown that the UPMEM system scales better for larger codes. At the same time, the performance in GPUs remains stable when the number of parallel decoders varies for different code sizes. This is explained by the scheduling mechanism incorporated in GPUs, which allows the workload to be automatically distributed among the available resources. Such mechanisms are not present in the UPMEM system, and that responsibility is transferred to the programmer.
VI. CONCLUSION
Although UPMEM’s individual DPU cores are limited in performance, the parallel nature of the architecture and the efficient mapping of the decoder allows for extracting performance orders of magnitude higher compared to high-grade CPUs. Although the performance of the UPMEM is not at the same level as high-end GPUs, the evidence shows that the UPMEM system performance can achieve the level of lowpower GPUs while executing fewer decoders in parallel, suggesting that performance is bound to the number of computing cores. Moreover, the evidence also suggests that performance could increase in the UPMEM implementation if scheduling mechanisms were incorporated into the hardware.
Lastly, this work demonstrated the feasibility of PiM-based LDPC decoding, and further work is needed to assess energy efficiency, which is one of the main advantages of PiM processing systems.
