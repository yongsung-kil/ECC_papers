Decoder Architecture for Array-Code-Based LDPC Codes

Sedat Ölçer
IBM Research, Zurich Research Laboratory 8803 Rüschlikon, Switzerland

Abstract—We describe a decoder architecture intended for decoding array-code-based low-density parity-check (LDPC) codes using the sum-product algorithm (SPA). The advantages of the proposed architecture as compared to the fully parallel implementation of the SPA are: reduced memory size, avoidance of complex signal interconnect patterns, and ease of programability to accommodate various code parameters. These advantages are derived from exploiting the well-defined structure of the parity-check matrix of array-code based LDPC codes. Sumproduct decoding with modified message-passing schedules are proposed to simplify the decoder implementation further.
I. INTRODUCTION
The “sum-product” algorithm (SPA) [1],[2] is the most commonly used algorithm for decoding low-density paritycheck (LDPC) codes. Its efficient implementation has become a topic of increasing interest, see, e.g., [3]-[5]. Implementations for very high data rates require fully parallel architectures where all symbol-node messages or all check-node messages are computed concurrently. Fully parallel implementations entail a large memory size and suffer from complicated interconnect patterns induced by the edges of the underlying graph. Programmability to accommodate various code parameters becomes an issue. Large codeword values exacerbate these difficulties.
Here we develop a sum-product decoder architecture suited for decoding a specific family of LDPC codes, the arraycode-based LDPC codes [6],[7]. These codes represent a family of regular, deterministically constructed LDPC codes with a parity-check matrix that is akin to that of array codes. They were shown to exhibit very good performance for binary as well as multilevel transmission over the additive white Gaussian noise (AWGN) channel and to lend themselves to efficient linear-time encoding [7]. They can easily be constructed for a wide range of code parameters.
An important aspect of our architecture is its programmability, allowing the decoder to easily accommodate different code parameters. Modifications to the regular decoding schedule permit to decrease the hardware complexity further and/or operate the decoder at higher speeds. Note that in [8] a staggered decoding schedule was proposed that decomposes each decoding iteration into a number of smaller steps during which a limited number of check nodes are processed. In contrast, for the “partial-update” schedule introduced here, all symbol and check nodes are processed at each iteration but a limited number of messages are computed.

II. IMPLEMENTATION OF THE SWITCH UNIT
Note that, for high-speed LDPC decoding, it is desirable that the switch be “nonblocking,” i.e. that any distinct input can be routed to any distinct output for the set of connections to be made. The crossbar topology shown in Fig. 4(a) leads to the conceptually simplest nonblocking switch unit, but is not efficient because it requires P2 switching elements for a P % P switch unit (P is assumed to be a power of 2). A possible alternative is the Benes-network-based switch unit shown in Fig. 4(b). When all input-to-output connections are known in advance, as is the case for LDPC decoding, the Benes switch is nonblocking. Each switch element can be set either in the BAR state or in the CROSS state, as depicted in Fig. 4(d).The Banyan switch in Fig. 4(c) represents another possible architecture. As the number of switching elements ( log2P stages each with P/2 2%2 switching elements) is smaller than for the Benes switch, it cannot be nonblocking in general. However, the following lemma can be shown to hold.

The architecture of Fig. 3 achieves this result through a combination of serial and parallel processing steps. It is possible to adopt different schedules with the objective of increasing processing speed and/or lowering implementation complexity.
The schedule illustrated in Fig. 5 consists in updating, for each p-block of symbol nodes, only those messages that are intended for a single p-block of check-nodes. In this “partialupdate” schedule, each p-block of symbol nodes sends messages to a single p-block of check-nodes in a round-robin fashion in successive iterations. At the first iteration, the topmost p-block of symbol nodes sends p messages to the topmost p-block of check-nodes only. At the second iteration, the same block of symbol nodes sends messages to the middle p-block of check-nodes only and at the third iteration to the lowest p-block of check-nodes only. In general, each symbolnode block will have sent one message to all check-node blocks after j iterations. This partial-update schedule implies that only one of the j register banks in Fig. 3 is updated during each iteration, which reduces the processing time for one iteration from 2k(1 + j)t to k(2+j)t. Processing speed can typically be increased by ~60%.

This result is useful for realizing the switch unit in our decoder architecture for array-code based LDPC codes. In
that case the permutation matrix α has dimensions p % p,
where p is a prime number. A Banyan switch realization is
still possible because matrix α can be embedded into a larger
P % P permutation matrix, with P > p and P a power of 2.
This only requires introducing a multiplexing stage at the switch output (or at its input).

IV. CONCLUSIONS Full parallelism in the SPA implementation is needed in some applications to meet requirements on very high throughputs. An example is magnetic recording, where data rates in excess of 1 Gbit/s are becoming a reality. In many other applications where extreme data rates are not needed, such as in wireless or wired transmission systems, some amount of parallelism can be traded against significant savings in hardware complexity and power consumption. The approach described here is intended for such applications. It is, for example, suited for digital subscriber line (DSL) transmission

at data rates on the order of 100 Mbit/s. Its flexibility to accommodate a wide range of code parameters is an advantage for DSL-like transmission, where different code parameters are selected in real time depending on actual channel attenuation and noise characteristics.
APPENDIX: PROOF OF THE LEMMA
Assume that the (P / 2) × (P / 2) switch unit is nonblocking. We will show that the P × P switch unit is then nonblocking. This requires showing that the last stage of the switch is nonblocking.
Let α m , m = 0, 1, ..., P −1, be the matrix defining the input-
output interconnections of the switch in the sense specified in the text, with α the P × P permutation matrix representing a single left (say) cyclic shift. Consider the n-th 2 × 2 switch element in the last stage of the switch unit, with n = 0, 1, ..., P / 2 −1. This switch element has output-port
labels l out,1 = n and l out,2 = n + P / 2 . Then the labels of the input ports connected to these two output ports are given by l in,1 = l out,1 − m (mod P) and l in,2 = l out,2 − m (mod P) .
Note now that the binary representations of l out,1 and l out,2 differ only in their most significant bits, denoted as msb(l out,1) and msb(l out,2 ) , respectively. We have that either msb(l in,1) = msb(l ) out,1 and msb(l in,2 ) = msb(l out,2 ) or msb(l in,1) ≠ msb(l ) out,1 and msb(l in,2 ) ≠ msb(l out,2 ) . In the former case, the 2 × 2 switch element is set in the BAR state, in the latter in the CROSS state. Thus no blocking occurs in the last stage of the switch. Finally, because the P = 2 case is trivially nonblocking, we have by induction that the switch is nonblocking for all P.
