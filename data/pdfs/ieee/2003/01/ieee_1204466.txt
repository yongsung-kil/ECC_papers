LDPC Code Construction with Flexible Hardware Implementation

Dale E. Hocevar
DSP Solutions R&D Center Texas Instruments — Dallas, TX

Abstract–This paper presents an LDPC code construction tech-
nique for irregular codes of various block sizes and code rates,
thus obtaining the performance benefit of irregular distributions. It represents an extension of the methods presented by Sridhara
et al. [7] for regular codes. More importantly, an efficient and
practical decoder architecture is also presented that achieves flexibility in block size and code rate for this broad family of
codes, a capability not present in other approaches. This decoder
can also achieve a high degree of parallelism, thus exploiting one of the benefits of belief propagation.
I. INTRODUCTION
LDPC codes are receiving an ever-increasing amount of attention due to their excellent performance and the simplicity of iterative decoding. They can provide performances that can be quite close to Shannon’s limit; but what may be more important is that the decoding algorithm can allow for a very large degree of parallelism that can be exploited in the decoder. This is difficult for turbo decoders. Also, as compared to turbo codes, the decoder implementations for each are quite different. Turbo codes are based upon trellis structures provided by the convolutional codes and this provides a simplifying organization for the implementation. This is not the case in general for LDPC codes as there are many ways to generate these codes, some completely random, others full of structure.
LDPC code decoders are not necessarily simple to implement in hardware though there is an efficient iterative decoding algorithm called belief propagation that can be used [1] & [2]. Some approaches apply constraints or structure to the codes to facilitate building the decoder. Two such works are important since they provide some useful approaches towards such implementations, [5] & [6]. Alternately, others have simply hardwired all connections of a particular code to achieve amazing levels of parallelism, [3]. However, the practical approaches thus far have been limited to a single code or a small selection of codes. What is more desirable is a set of codes for various block sizes and code rates together with a single flexible decoder that can decode any of these codes. This would include the capability of the decoder to be designed for various degrees of parallelism.
This paper presents one solution to this problem. First we will discuss a construction technique for irregular LDPC codes that have desirable properties as related to a flexible hardware implementation. These codes are based upon the SFT approach in [7] and exhibit very good error correction performance and can be designed for various block sizes and code rates. An overall concept for a flexible hardware decoder is given that solves many of the implementation problems with such decoders such as routing and addressing. Various concepts will be

discussed for achieving high levels of parallelism and for achieving flexibility in block sizes and code rates.
II. IPP CODES: IRREGULAR PARTITIONED PERMUTATION LDPC CODES
A recent construction technique has been given by Sridhara, Fuja and Tanner (SFT), [7]-[8], for regular LDPC codes. First we’ll give a quick overview of the SFT construction method for regular LDPC codes.
The SFT method accommodates a variety of code rates and block sizes and is based on simple permutation matrices. Their approach constructs a block structured parity check matrix H whereby each block is from a subclass of square permutation
matrices and all have the same dimensions, m × m, where m is
a prime number. Each permutation matrix is an identity matrix whose rows have been circularly shifted by a set amount. Assume H has j rows of blocks and k columns of blocks. Each block’s shift value is given by the expression bsat mod m where 0 ≤ s < j , 0 ≤ t < k , and (s, t) denotes the block’s location.
Generator a is chosen to have multiplicative order k, likewise b has order j.1 A quasi-cyclic, regular (j, k) LDPC code with rate
R ≥ 1-(j/k) and codeword size m⋅ k is thus constructed. Their
error performance results compare very favorably with the best regular LDPC codes.
We have devised a modified code construction technique that produces a block structured parity check matrix for irregular LDPC codes of nearly any code rate and that is based upon the method found in [7]. Our irregular LDPC code construction method uses optimized degree distributions [2].
To get the best performance, irregular LDPC codes with optimized degree distributions and with no, or very few, short cycles are necessary. The SFT codes have good girth properties, but they are regular codes, and as such their performance will not compare favorably to comparable irregular codes. However, one can construct irregular codes that perform very well by reworking the SFT approach. A very rough description of our approach follows. Choose the parameters needed that will work for your problem, optimize the degree distributions, construct a matrix of dimensions j×k that is populated largely with 0’s and sparsely with 1’s such that it closely matches the desired distributions,2 and lastly, replace each 0 with an m×m matrix of zeros and replace each 1 with the corresponding SFT circularly shifted permutation matrix for that position. The result is your parity check matrix. This method has a little similarity at the block level to the approach given in [10].

III. ITERATIVE DECODING ALGORITHM

LDPC codes typically use a form of iterative decoding starting with the soft-data received inputs from the channel. The algorithm is called belief propagation (BP), though it has other names such as message passing algorithm and sum-product algorithm. See [1] & [2].

The first of these 2D arrays is L(qmj) and the second is Rmj where m is the parity check equation or row index and j is the bit or column index. IV. DECODER ARCHITECTURE
A. Overview of Target Architecture
Some of the complications that arise when building an LDPC decoder for high data rates is the large amount of data I/O that is required assuming most algorithm values are stored in memories. Another is the indexing into the memories to get the necessary data at the correct time and also the routing of

A new decoder architecture is described next that is similar at the top level to that from Boutillon et al. [6], and that also uses some concepts from Zhang et al. [5]. Our architecture is significantly different in several aspects as will be discussed and it has been devised to exploit the structural properties of our IPP codes discussed previously, though it is not limited to just these codes.
Fig. 3 shows the top-level architecture. The R storage memory holds the Rij terms packed by rows. A single word holds all the elements for one row, or many rows, as we will expand upon later. At times this memory will simply be called the row memory and it can consist of a single or multiple memories. For this section, assume only single row processing.
This architecture computes both algorithm passes simultaneously. The operations for computing the new Rij terms occur in the parity check update block (PCUB). This block performs the operations as in eqn.s (1), (2) and (3), or as in (7).4 The results are stored in the row memory. The inputs for these operations, L(qij), come from the parallel adder block. Here the previous Rij terms are read from the row memory and subtracted from the L(qj) terms that come from the reverse router block. This is in effect the operations from eqn. (5).
In addition, these Rij terms are simultaneously fed into the router block which routes each single term to the appropriate bit update block. In these blocks the data is accumulated by bit as in (4) without the channel input term. Each block handles some predetermined subset of the codeword bits, though only one at a time, at least for now in the discussion. The memories within these bit update blocks will be called column sum memories. Notice that for this portion, data routing and addressing in the column sum memories are non-trivial operations in general. We will come back to this later.
It is also necessary to test the parity check equations on each iteration as one stopping criterion. This can easily be done in our architecture. The decisions are easily formed in the parallel adder, then fed into the PCUB where the checks can be done as they are a trivial form of that parity update process.
Fig. 4 details the structure of the bit update block. The two memories, A and B, operate in a ping-pong manner. One acts as part of the accumulator for new data while the other supplies last iteration’s data. The mux they feed is actually a crossswitch to facilitate their alternating operation. The adder on the right forms the rest of the accumulator and the following demux selects the receiving memory. While data flows out from last iteration, the channel data is added to this. In principal, only one set of addresses need be computed as the data is referenced by column number in all three memories and is used in the same sequence by both sides. In practice, address delay (or data delay) elements would be needed to align the processes in time.
A rough outline of just one approach for the parity check update block (PCUB) is depicted in Fig. 5. Assuming the input

is one single row, the data elements are sent to a parallel set of
LUT’s for the Ψ(⋅) function. Each input element consists of L(qmj) and a hard decision cn, whereas, there are three outputs from each LUT block: an unsigned function value from LUT, sign of L(qmj) and hard decision. These then flow through an augmented adder tree that sums the function values, multiplies the signs together (Exc.–Or), and forms the Exc.–Or of the hard decisions. The following parallel set of adders then subtracts the necessary terms to complete computation for (1). Another parallel bank of LUTs and appropriate sign adjustment completes eqn. (3). The results are now ready to be sent out of the block. Rows with fewer terms than the width of the above datapath pose no problem since the unused portions are easily nullified. Rows with too many terms require alteration to the PCUB as discussed later. The outline above is simply a straightforward implementation of the equations (1), (2) and (3); it is very similar to that given in [3].
Alternatively, these parity check updates can be implemented by using blocks that compute eqn. (7) and various approaches towards construction can be taken. One such approach is given in [6]. In addition, this approach can be modified to operate very similarly to the full sum then subtract arrangement in Fig. 5 by also defining a special LLR subtraction, [12].
To this point, the architecture we have been describing is very similar to that found in [6]. One main difference is that the R memory and parallel adder block are on the same side of the router as is the parity check update block in our approach. In [6] the parity check update block is on one side of the router and everything else is on the other side. They do this because they use every column group on each iteration and assume a straightforward 1:1 sequence order of terms from row data to column groups. Hence their router block does all the work in arranging the data. Our approach does not need to use all column groups and can achieve a wider variation in the number of terms per row equation. We do not need to use random address generators for the column sum memories (bit update blocks), nor a random permutation generator for the router, as they do.
So, the question could be asked, ‘What forms of LDPC codes will map onto the architecture we have described?’ Certain assumptions concerning this have been made. It is assumed that the codeword bits, or columns of H, have been partitioned into several groups, each one corresponding to a bit update block. In addition, for any single row of H, at most one entry exists in any column group. The matrix H would be reordered to match this column grouping.
What are the consequences of these assumptions and actions? First, on any cycle, a bit update block has at most one data item. This in itself simplifies the router.5 But in addition, due to the reordering, the router has other restrictions. Namely, the order of data elements that it is fed does not change when passing through the router.6 Since the number of data elements will be the same as, or less than, the number of update blocks, but never by much, the router is even more restricted in the number of paths it must be able to realize. B. Architectural Aspects Specifically for IPP Codes
It can be seen that the IPP codes presented earlier match the assumptions for our architecture just discussed. Each column in the macro matrix expands into a block column in the parity check matrix, likewise for the rows. So each block column can be mapped to its own bit update block with column sum storage (Fig.s 3 and 4). Another key advantage for IPP codes is that of column addressing.
The idea for simple column addressing in Fig. 4 lies in fact that our permutation matrices have a certain structure; namely, they are circularly shifted identity matrices. To utilize this fact note that since the rows are done in sequence, the columns will be as well, only there will be a starting offset and the sequence will wrap around modulo m. By storing the entry for each column linearly in its memory, addressing becomes a trivial counter. Note that Zhang et al. exploited the same property when they used circularly shifted identity matrices, [5].
C. Advancing Beyond the Basic Architecture
Many extensions beyond the basic architecture are known, but there is only space to briefly mention those most important.
1) Parallel Row Computation: Thus far, only parallelism within a single row has been exploited. Many row operations can be performed in parallel however, and for the IPP codes this can be done fairly simply. Again, properties due to the simple form of the permutation matrices can be exploited. Assume several rows in sequence within a row block are to be operated upon in parallel; let’s say w rows. Recall that their corresponding columns, considering just one macro matrix connection, are also sequential, modulo m. So, one can use wider words in the column sum memories so that each word holds w entries. For this macro connection, the row updates produce w terms and the column summing can handle w terms. However, it is not nearly this simple as there are alignment issues; but these can be handled with store and shifting hardware.
This multiple row computation requires that w rows be stored as a single word in the row memory. The computational portions of the parity check update block, parallel adder block, bit update blocks, and the data path of the router and reverse router would all have to be duplicated w times. But the decoding speed would increase by a factor of w.
2) Column Grouping & Simplified Routing: A further benefit of these codes is realized by forming sections of the column blocks. This reduces the number of bit update blocks, which more importantly reduces the number of individual memories. Notice in Fig. 1 the partitioning of the macro matrix into 9 segments. In any segment, at most there is one column connection per row. So, all column blocks for any section can be grouped into a single bit update unit containing just the three memories as in Fig. 4. Overall memory size in bits remains the same, however, a large decrease in the number of memories is
achieved. For Fig. 1, only 9×3 = 27 memories are now needed versus 24×3 = 72 without this grouping. This also simplifies the router as previously shown in Fig. 6.

3) Flexibility for Block Sizes and Code Rates: These codes can be designed for various block sizes, macro matrix definitions and code rates. The architecture also can be built in a flexible manner to accommodate these. Variations in the block size for a fixed macro matrix can be handled with addressing and alignment flexibility in the bit update blocks. Varying the macro matrix structure further requires more paths in the router blocks and memory allocation changes in the bit update blocks. Designing for multiple code rates is more difficult as the change in ratio of matrix dimension changes the ‘dimensions’ of the hardware blocks. This issue only becomes difficult when the range of desired code rates is large. In particular, for long rows, hardware sharing and flexible restructuring are needed in the row memory, PCUB and bit update blocks.
V. CONCLUSIONS
A construction technique for irregular LDPC codes that is a modified form of the SFT method [7] was presented. A practical architecture was shown that exploits the structure of these IPP codes. With this approach it is possible to attain the performance gain of irregular codes and the flexibility to handle various block sizes and code rates. It can also achieve a high degree of parallelism thus enabling very high data rate and flexible decoders. A structured code approach was adopted to help solve the difficult LDPC decoder implementation issues. Future work will entail further code design and evaluation methods and alterations to the basic IPP approach.
