FPGA based design and prototyping of efficient 5G QC-LDPC channel decoding
Jeremy Nadal1,2, Amer Baghdadi1 1IMT Atlantique, CNRS UMR 6285 Lab-STICC, Brest, France 2Department of Electrical Engineering, Polytechnique Montreal, QC H3T 1J4, Canada
e-mail: jeremy.nadal@polymtl.ca, amer.baghdadi@imt-atlantique.fr

Abstract—The Quasi-Cyclic (QC) Low-Density ParityCode (LDPC) is the key error correction code for the 5th Generation (5G) of cellular network technology. Designed to support several frame sizes and code rates, the 5G LDPC code structure allows high parallelism to deliver the high demanding data rate of 10 Gb/s. This impressive performance introduces challenging constraints on the hardware design. Particularly, allowing such high flexibility can introduce processing rate penalties on some configurations. In this context, a novel efficient and flexible hardware architecture for the 5G LDPC decoder is proposed, targeting Field Programmable Gate Array (FPGA) devices and supporting all 5G configurations. The architecture supports frame parallelism to maximize the utilization of the processing units, significantly improving the processing rate. Compared to a recent commercial 5G LDPC decoder, the proposed FPGA prototype achieves a higher processing rate for most configurations while having similar complexity.
Index Terms—LDPC, 5G, Parallelism, Throughput, FPGA
I. INTRODUCTION
Future communication systems are foreseen to support several services, each having different system parameters optimized for specific requirements, such as latency, data rate and energy consumption. To meet these challenging requirements, the Quasi-Cyclic (QC)-Low-Density Parity-Code (LDPC) [1] has been selected as the next error correction code for the upcoming 5G communication systems [2]. The 5G LDPC code design allows to support several frame sizes, at a granularity level of a few bits, for several code rate from 1/5 to 9/10. Furthermore, the basegraphs of the 5G LDPC code can be expanded by a lifting size of up to 384, allowing to support very high parallelism levels to guarantee the expected 10 Gb/s of data rate in 5G networks [3]. In term of performance, a FER of 10−5 without error floor is achievable for all frame and code rate configurations [4].

The 5G LDPC decoders can be integrated in Field Programmable Gate Arrays (FPGAs) at the base station. Such devices being reprogrammable, the physical layer can be re-configured accordingly to the future evolution of the upcoming standards, reducing costs for the telecommunication operators. Therefore, there is a high interest of designing 5G LDPC decoders for FPGA devices. However, all the 5G features put high constraints on the hardware design: the architecture must be carefully designed to support all the 5G configurations. Furthermore, the 5G LDPC code structure is well designed to support high parallelism, but only for long frames using the highest lifting sizes. For shorter frames, the processing units are under-used, and the processing rate is greatly reduced. For instance, only 2/384 = 0.52% of the processing units are exploited if a frame configured with a lifting size of 2 is being decoded. Finally, the permutation network is more complex than a barrel shifter as currently implemented in typical LDPC, causing complexity and routing issues. To address these issues, a novel hardware architecture for a 5G QC-LDPC decoder is proposed in this paper, targeting FPGA devices. The architecture features a high flexibility and a high processing speed for all frame sizes. For optimized hardware resources and processing power utilisation, frame parallelism is proposed to decode small frame sizes.
This paper is structured as follows. Section II-A summarizes the 5G LDPC code characteristics and presents the typical layered decoding algorithms. Then, the proposed 5G LDPC decoder architecture is detailed in III, along with the main processing units. Section IV presents and compares the synthesis results with the commercial Xilinx 5G decoder. The processing throughput is also analysed. Finally, section V concludes the paper.

Each non-null element of this BG is expanded by a Z-by-Z circularly shifted identity matrix, where Z corresponds to the lifting size. The circular shift (rotation) values depend on Z and the edge positions in the BG. This code design allows suitable hardware implementation and enables high parallelism level for high processing throughput. However, compared to existing code design in other standards, several lifting sizes Z and BGs have to be supported for the 5G QCLDPC encoder and decoder.
In fact, two BGs are defined in 5G, their characteristics being detailed in Table I. The number of columns, rows and edges of the BG number b are respectively denoted ncb, nrb and neb. The CN degree at row i and BG b is denoted dci,b, whereas the VN degree at column j and BG b is denoted dvi,b. It can be observed that the Variable Node (VN) and Check Node (CN) degrees are highly uneven at different rows and columns. Therefore, the 5G QC-LDPC code is irregular. Furthermore, row-byrow orthogonality is not ensured for all the rows, which can potentially lead to update conflicts in a practical implementation.
For these 2 BGs, 51 lifting sizes can be used. They are defined as Z =∆ B × 2z ≤ Zmax, with Zmax = 384, B ∈ {2, 3, 5, 7, 9, 11, 13, 15} and z ∈ 0, 7 [2]. This contributes to support a large number of frame sizes. At the output of the encoder, the first 2Z bits of the obtained sequence are punctured: this enables to improve the performance of the code [4]. To adapt the Code Rate (CR), the last Npunct bits are punctured too, with Npunct = Z × ncb − 2 − (ncb − nrb )/CR .

III. PROPOSED HARDWARE ARCHITECTURE
A. Proposed architecture
The proposed architecture, shown in Figure 1, contains an APP memory bank which can stores up to nc1 = 68 words of QAP P × Zmax = 3072 bits, where QAP P corresponds to the number of bits for the APP. Based on a quantization study, it is set to 8 bits in our architecture. An input/output interface unit drives and writes the channel LLR into the APP memory bank and reads the decoded bits when the last iteration is reached. The addresses of the memory bank correspond to the column indexes of the BG. A permutation network rotates Z outputs of the APP memory banks at each clock cycle. It is configured to support all rotation values from the 51 lifting sizes. The Z rotated APPs are then processed by up to Zmax = 384 Node-Processing Unit (NPU) in parallel, enabling high processing throughput. The NPU serially processes all the edges of the chosen BG. B. Frame parallelism
Finally, the updated APP at the outputs of the NPUs are written back to the APP memory banks.
The use of a second permutation network is avoided at the output of the NPU: there is no need to rotate the APP back to their original positions. Instead, the wellknow offset permutation technique [8] is employed. It consists of rotating the APPs at column i by ∆r(i) = rnew(i) − rold(i), with:
• rnew(i) is the expected rotation value at column i. • rold(i) corresponds to the previous rotation value at
column i: rnew(i), stored in a dedicated RAM (not shown in the figures) and becomes rold(i) for the next processing step.
The memory addresses, offset permutation values and the control signals are generated by the Global Controller Unit (GCU). Details about this unit are presented in Subsection III-D. The adopted permutation network in this work is the optimized one proposed in [9].
In typical decoder architectures, only a subset of the NPUs are operating, depending on the selected frame size. This results in a loss of the processing rate: the available hardware resources are not efficiently re-used. This hardware efficiency is proportional to the lifting size, as shown in Figure 2. This becomes a major issue when short frames must be decoded.
To overcome this problem, an original solution is proposed. It consists of re-using the NPUs by processing several frames in parallel. We assume that these parallel processed frames hold the same code rate, frame size

and BG configuration. It is then possible to process up to P = Zmax/Z frames in parallel, P being the parallelism level. The maximum lifting size being 384, frame parallelism can be supported if Z ≤ 192. The hardware efficiency, denoted η, is expressed as η = P × Z/Zmax and is shown in Figure 2. Averaged over all the available lifting sizes, the hardware efficiency is η = 0.92 with frame-level parallelism and η = 0.23 without. Therefore, our approach enables to multiply the hardware efficiency by almost 4 in average, leading to a processing throughput increase by the same magnitude. The processing throughput, denoted PT, is expressed as

PT = Fclk × K × P ,

nIT × ncc

where Fclk is the operating clock frequency, K is the frame size, nIT is the number of iterations and ncc ≥ neb is the number of clock cycles per iteration.

C. Node Processing Unit
The NPU, presented in Figure 3, corresponds to the main computational unit of the decoder. It takes as input the rotated APP associated to column i, row j. The rotated APP is then subtracted to the old C2V message at the same column/row position to generate the V2C message, through the block SUB in Figure 3. Beforehand, a Message Reconstruction (MR) block, referred as block MR1, re-combines the global components of the old C2V messages to form the original C2V message. The obtained V2C message is then transferred to the external V2C RAMs and an internal serial-sorting block,

based on two comparators, registers and multiplexers. This block serially processes the received V2C messages to find the global components of the new C2V messages, which are stored into dedicated registers and written into the C2V RAMs. Finally, the new C2V messages are reconstructed (block MR2) and added to the delayed V2C messages coming from the V2C RAMs (block ADD). The updated APPs are then obtained.
In addition, the processing is split into 2 main pipeline stages:
• The pipeline stage 0 serially processes the dcj+1 APPs of row j +1, computes the V2C messages and the new C2V messages (serial sorting block). This stage contains blocks SUB, MR1 and serial sorting.
• The pipeline stage 1 serially updates the dcj APPs, where dcj corresponds to the variable node degree of row j. It also writes the new C2V messages into the corresponding memories. This stage contains blocks MR2 and ADD.
This 2-stage pipeline architecture enables to efficiently exploit the available hardware resources and, by consequence, can double the processing rate. As the VN degree can differ between successive rows, one of the pipeline stages can be put into IDLE state until the second one finishes its processing. This introduces a slight throughput penalty.
D. Global Controller Unit
The GCU mainly includes a Finite State Machine (FSM) unit and two addresses generator units, one for each pipeline stage. The FSM controls and schedules the whole decoder architecture. All control actions and events are synchronized on the same clock cycle. This is made possible thanks to a pipeline interface unit within the GCU. It consists of a shift register bank delaying the addresses according to a specific latency value. The latency corresponds to the RAMs read operation and the internal pipeline registers of the permutation network and the NPUs.

B. Synthesis result
Post Place & Route (P&R) synthesis results, displayed in Table II, has been obtained for the Kintex 7 xc7k160t device, considering the Xilinx and the proposed LDPC decoder architectures. The proposed architecture requires 69% more LUTs than the Xilinx one, but requires 14% less registers. The number of Block RAMs (BRAMs) of the proposed architecture is almost doubled (92% increase) when compared to the xilinx architecture. These results can be easily explained by the fact that the Xilinx decoder uses 128 processing units in parallel, 3 times less than our proposed design. However, this complexity increase is acceptable considering the PT gain evaluated in Sub-section IV-A, where the PT of the proposed architecture is more than doubled when compared to the Xilinx one.
For the proposed design, the critical path after P&R is located between the output of the APP memories and the pipeline registers located at the 5th multiplexer stage of the permutation network. This constrains the operating clock frequency at 160 MHz, which is 27.3 % lower than the operating clock frequency of Xilinx architecture. However, the proposed architecture still has higher PT for most 5G configurations, as evaluated in Sub-section IV-A. It should be noted that the pipeline depth in the

V. CONCLUSION
A novel 5G LDPC decoder design for FPGA devices is proposed in this paper. The designed decoder features high processing throughput and high flexibility to support all 5G configurations. Thanks to the use of frame-level parallelism when short frame sizes are processed, most of the 384 parallel processing units can be re-used to increase the processing throughput. Post P&R synthesis results show that averaged processing throughput is doubled to tripled when compared to recent commercial Xilinx 5G LDPC decoder. The complexity in LUTs and BRAMs is only doubled, demonstrating the efficiency of the proposed FPGA based design.
