Abstract‚ÄĒRecently, type-II quasi-cyclic (QC) low-density parity-check (LDPC) codes have attracted increasing attention due to their compact structures and promising decoding performance. In this paper, the type-II QC-LDPC code standardized for use in near-earth application is implemented by FPGA. On the basis of analysis for the generator matrix and parity-check matrix of the code, the codec for the type-II LDPC code is designed. By using an XC4VLX200-FPGA, the maximum clock frequency of the encoder is 287MHz at the cost of 810 slices and 15 Blockrams, while the maximum clock frequency of the decoder is 244 MHz at the cost of 10481 slices and 74 Blockrams. The testing result for the codec performance shows that such a code can completely satisfy the requirement for on-board channel coding application. The codec developed in this paper has been successfully employed in many remote-sensing satellite missions in China.
Keywords-codec; low-density parity-check codes;quasi-cyclic; FPGA
I. INTRODUCTION
Low-density parity-check (LDPC) codes were originally invented by R. Gallager in 1961 [1] and are largely ignored for the following three decades. In the early 1960s, the decoding performance for LDPC codes with practical lengths was very difficult to realize due to the constraints of decoding algorithm and hardware. It is several decades of VLSI development and the discovery of an iterative decoding algorithm (called Belief Propagation decoding [2]) that finally enable LDPC codes to be implemented at a manageable complexity.
A linear block code can be represented by (n,k), where n and k are the codeword length and information-bit length, respectively. An LDPC code is a linear block code whose m hn parity-check matrix has a very small ratio of the total number of 1‚Äôs to the total number of 0‚Äôs, where m ńĪ n—łk. The k h n generator matrix used to encode a linear block code can be obtained from the parity-check matrix via linear operations.
Well-designed algebraic LDPC codes based on Euclidean geometries [3] have been shown to provide low error floors and fast iterative convergence. For near-earth applications,

since very high data rates and high reliability are the driving requirements, these codes match well with the near-earth applications. The near-earth LDPC code standardized by CCSDS is implemented in this paper. Firstly, the paritycheck matrix and generator matrix for the LDPC code are analyzed. Secondly, the coder and decoder for the LDPC code are designed based on FPGA. Finally, The codec performance for the LDPC code is tested.
II. PARITY-CHECK MATRIX AND GENERATOR MATRIX
The parity-check matrix for a quasi-cyclic (QC) LDPC code is an array of circulants with the same size [3], [4], [5]. Owing to such a structure, a QC-LDPC code possesses two advantages regarding hardware implementation: i) by utilizing shift registers, the encoding complexity can increase linearly with the code length or parity bits, and ii) the routing complexity for encoder and decoder can be reduced in terms of the interconnections of integrated circuits.
The LDPC code recommended in CCSDS standard [6], [7] for use in near-earth scenario is a special class of QCLDPC codes, termed as type-II QC-LDPC codes [8], [9], [10], [11], [12], [13], [14], [15]. The parity-check matrix for a type-II QC-LDPC code is an array of weight-2 circulants. A weight2 circulant of size P h P can be represented by a pair of nonnegative integers smaller than P. The two integers, say (a,b), are the two positions (indexed from zero) of √ĺ1√Ņ s in the first row of the circulant.
A. Parity-check matrix
The parity-check matrix for the code in CCSDS standard (near-earth case) is designed based on circulants derived from the 3-D Euclidean geometry [3].
The parity-check matrix is a 2 h 16 array of 511 h 511 circulants, yielding a parity-check matrix of dimension 1022 h 8176. The parity-check matrix is shown in Eq. 1.

III. IMPLEMENTATION OF LDPC (8176,7154) CODEC
A. Hardware Implementation of Encoder The encoder block diagram is depicted in Fig. 2. While

Since there are two dependent rows within the paritycheck matrix, the number of information bits for the corresponding LDPC code is 8176-1022 +2 =7156.
B. Generator Matrix Since there are two dependent rows within the parity-
check matrix, the generator matrix for the (8176,7156) code is composed of two parts. The first part is a 7154h8176 submatrix G‚Äô, while the second part consists of two independent rows. G‚Äô corresponds to an (8176,7154) code which is the focus of our subsequent discussion.
The submatrix G‚Äô can be represented by a 14h16 array of 511 h511 circulants, in a systematic circulant form, as shown in Eq. 2.
The array also includes two parts. The first part is a 14h 14 square array, in which the circulants in the diagonal are all identity submatrices (I‚Äôs) and the circulants elsewhere are all zero submatrices (0‚Äôs). The second part is a 14 h 2 array, which is composed of two columns of circulants Bi,j‚Äôs (1 ńį i ńį 14,1 ńį j ńį 2). The circulants, Bi,j‚Äôs, are specified in Table C-1 of CCSDS 131.0-B-2.
Each codeword in the code consists of 7154 information bits and 1022 parity-check bits; therefore, the rate of the code is 0.875. Besides, this code has a minimum distance at most 24, owing to Theorem 8 in [9].

Figure 2. Encoder block diagram for the LDPC (8176,7154) code.
In the encoder, two groups of shift registers work in parallel. In each clock period, each input bit is ANDed with 2*511 columns of circulant submatrices, and the resultant 2*511 bits as temporary information are XORed with the temporary information generated in the previous clock period. After that, 2*511 bits are generated. When the input bits are ended (K = 7154), two groups of 511 bits are generated in parallel, which are used as the parity-check information. Finally, the input bits and the obtained two groups of parity bits are continuously output in a serial manner, which completes the encoding process of the LDPC (8176,7154) code.
The LDPC (8176,7154) encoder implemented by FPGA is simple, fast, real-time and reliable. As the encoder can be easily implemented by And gate, XOR gate and shift registers in FPGA, the speed can reach the maximum processing speed of the FPGA chip. By using an XC4VLX200-FPGA, the maximum clock frequency of the encoder is 287 MHz at the cost of 810 slices and 15 Blockrams.
B. Hardware Implementation of Decoder
The min-sum (MS) algorithm is employed in the implementation of the LDPC decoder. This decoder is based on a 1022h8176 parity-check matrix, which is a 2h16 array of 511h511 circulants. The row weight of the parity check matrix is 32, and the column weight of the parity check matrix is 4. The positions of the ‚Äė1‚Äôs in each circulant are completely specified by two initial phases (IPs), i.e., two 1‚Äôs positions in 1st row of the circulant. As a result, only 64 IPs need to be stored when the decoder is implemented in practice.
The decoder block diagram is depicted in Fig. 3. The decoder comprises 6 modules, i.e., timing-control module, control-signal-generation module, input-sequence-storage module, vertical operation module, horizontal operation and decision module,and data-output module.Besides,the decoder also includes 8 dual-port blockrams (1022h16), 64 dualport blockrams (511h8) and 2 single-port blockrams (511h 16).

1) Timing-control module: The main function of the timing-control module is to yield necessary timing signals for decoding the continuous input-sequence (i.e., a codeword) in real time. Specifically, a total of 10 iterations should be completed during the time of a codeword. An iteration comprises of vertical operation, horizontal operation and decision, and the reading/writing operation associated with blockrams. The delay time from the input to the output of a codeword is two codewords‚Äô time. In the first codeword‚Äôs time, an input codeword is stored, while in the second codeword‚Äôs time, the stored codeword is decoded. In the beginning of third code‚Äôs time, the decoded codeword is output. The timing diagram of the decoder is shown in Fig. 4.
Figure 4. The timing diagram of the decoder.
2) Control-signal-generation module: In this module, the clock and synchronization signals output from the timingcontrol module are fed to generate other signals: (i) the reading and writing addresses and enable signals for the 72 dual-port blockrams and 2 single-port blockrams; and (ii) the control signals used by all other modules. At the same time, the decision signals output by horizontal operation and decision module are also fed in the control-signal-generation module to generate the number of iterations and the iteration termination signal. To be specific, clkin, syn1 and syn2 are input signals, and others are output signals, as shown in Fig. 5.

Syn1/syn2: For syn1 and syn2, the high level has a width of 8176+40 clocks for storing the codeword, and the low level has also a width of 8176+40 clocks for the iterative decoding. Syn1 and syn2 are both periodic signals.
Syn_v: The high level has a width of 530 clocks during which the vertical operation module works. The alternation of high level and low level for Syn_v occurs 10 times.
Syn_h: The high level has a width of 280 clocks during which the horizontal operation and decision module works. The alternation of high level and low level for Syn_h also occurs 10 times.
Flag_ judge: The high level indicates the end of the iteration, and the rising edge is ahead of the falling edge of syn_h by 18 clocks.
Cnt_iteration: It denotes the number of iterations of the decoding. When flag_judge is a rising edge, the number of decoding iterations is Cnt_iteration + 1.
72 dual_ram: It denotes the reading/writing addresses, reading/writing enable signals and initialization signals of 72 dual-port blockrams. Among them, 64 reading/writing addresses are based on the non-zero positions of the check matrix, and based on the reading/writing scheme of the horizontal operation and decision module as well as the vertical operation module. The reading/writing enable signals are based on the timing of the horizontal operation and decision module, and the vertical operation module. The initialization signal is based on the update of codewords.
2 ram: It represents the reading/writing addresses and enable signals of the two single-port blockrams. The blockrams (511*16) are used to store the decision sequence of the output from the vertical operation module. The reading/writing addresses and enable signals are generated based on the timing of the vertical operation module and the iteration termination signal (flag_judge).
3) Input-sequence-storage module: Its function is to store the input data into 8 dual-port blockrams, each with a size of 1022*16. Real-time storage is completed while data is input.
4) Vertical operation module: From 64 dual-port blockrams, read 4 data per column (16 columns in parallel), denoted by a1, a2, a3 and a4, respectively, and at the same time read one data from each column in 4 dual-port blockrams (16 columns in parallel), denoted by pl. The timing for vertical operation module is designed as follows. The check matrix has 8176 columns in total, which need to be processed by the vertical operation. By utilizing a pipeline operation, 16 columns can be processed with a clock in a parallel manner, and hence all columns can be completed in 511 clocks. In addition, considering that there are the delay time for reading blockrams and the computation time, the vertical operation module requires 530 clocks per iteration in total.
5) Horizontal operation and decision module: Given 32 data values in the nonzero positions of each row of the parity-check matrix, the horizontal operation and decision module performs the following operations. (i) Sign Operation is to perform XOR with respect to the 31 values (excluding the self value). (ii) Evaluation Operation is to first find the minimum and second minimum of the 32 values, and then to multiply the modified factor. The multiplication can be transformed into shifting & addition operation, while the task for obtaining the minimum and second minimum will need several clock periods. (iii) Decision Operation is to multiply a 1—Ĺ8176 vector by each row of the parity-check matrix (i.e., inner product). The output of the multiplication is a vector with length of 1022. If the vector is an all-zero vector, then the decoding process is terminated; otherwise, iteration process continues. For hardware implementation, the multiplication can be carried out by XORing the 1*8176 vector with each 1*8176 vector (within the row of paritycheck matrix). By this manner, executing 1022 times results in 1022 data values. These data values are then added together one by one to attain the final iterative data value. Due to the parallel processing, the horizontal operation and decision module can be completed in 280 clocks.
6) The data output module: After the iteration is finished, this module converts 16 signals, each with a width of 511 bits, into a serial output signal. Each codeword has a total of 8176 data, of which the first 7154 data are valid data.
Using the above method, we have implemented a decoder for LDPC (8176,7154) code. The speed of this decoder can reach the maximum speed available for blockrams in FPGA chip. By utilizing an XC4VLX200FPGA, the maximum clock frequency of the decoder is 244 MHz at the cost of 10481 slices and 74 Blockrams.
C. Coding-gain Test for the Codec
In our coding-gain test, the Bit Error Rate Tester (BERT) is used as the source to transmit data and the sink to receive data. The LDPC coded and decoded data with Gaussian white noise added to the channel are compared in real time. Then, the gain of the LDPC code is calculated by changing the level of the white Gaussian noise.

Fig. 6 depicts the result for coding-gain testing. We observe that at the bit error rate of 10-7, the coding gain for LDPC(8176,7154) codec is about 7.0 dB. At the BER of 10-8, our codec performs only about 1.56 dB away from the Shannon limit.

IV. CONCLUSION The LDPC (8176,7154) code recommended by CCSDS standard is a type-II QC-LDPC code, whose parity-check matrix is composed of weight-2 circulants. Owing to this feature, the hardware with both simple implementation and fast speed is applicable to remote-sensing satellite applications. The encoder and decoder designed in this paper, have a small occupation of FPGA, a high speed and a performance congruent with the CCSDS Standard. Therefore, the codec designed in this paper completely satisfies the requirement of the on-board channel coding applications.
