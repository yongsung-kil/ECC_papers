Design and Implementation of Quasi Cyclic Low Density Parity Check (QC-LDPC) Code on FPGA
Dharmesh J. Patel1 and Pinalkumar Engineer2
1Department of Electronics & Communication Engineering, Government Engineering College, Surat, India-395001 2Department of Electronics Engineering, Sardar Vallabhbhai National Institute of Technology, Surat, India-395007
Email: 1djpgecs@gmail.com 2pje@eced.svnit.ac.in

Abstract—Low density parity check (LDPC) code is a widely used error correcting code in various applications such as Wi-Fi, Wi-Max and Digital Video Broadcasting – Satellite – Second Generation (DVB-S2). Proposed work is focused on LDPC decoder design using soft decision iterative message passing for a code length 222 bits, 546 bits, 642 bits, 648 bits and 1152 bits that gives BER of 10−5 with coding gain of approximately 4 dB. Proposed work shows fully parallel design on isolated shifted identity matrices based on structured construction quasi cyclic low density parity check code (QC-LDPC) that give throughput of 2 Gbps.
I. INTRODUCTION
Low density parity check code is a popular forward error correcting code due to its performance near channel capacity given by Shannon in 1948. The code was introduced by R. Gallager in his Ph.D Thesis [1]. Due to its implementation complexity at that time it was forgotten and rediscoverd in mid 90’s by MacKay and Neal [2]. Construction of LDPC is from sparse parity check matrix having less number of nonzero entries in parity check matrix where the quasi cyclic (QC) is a sub type of LDPC code. It is a structured type of LDPC code used to construct a short length and small girth code due to its simplicity in construction. It is adopted in various wireless communication standard i.e. Wi-Fi (IEE 802.11n), Wi-Max (IEEE 802.16e) and also in satellite communication as a DVB-S2.
II. RELATED WORK
Several work has been done on code construction from structured methods either from graph based or design based approach [2]. Design based code constructions are either combinatorial or algebraic method of the code construction while hierarchical and high-girth code construction suggested by [3] is a graphical approach of code construction. The construction suggested by [4], [5] are the design based approaches. Proposed LDPC code is constructed based on design based approach.
Implementation of decoder can be done by three different approaches: Fully parallel, partial parallel and bit serial approach [6]. Proposed work is done using fully parallel design. Decoder design is based on hard decision or soft decision decoding algorithm. Proposed decoder is implemented using min-sum soft decision decoding algorithm and implemented

using fully parallel design. In [7], LDPC decoder is implemented on Virtex-7 FPGA for code length (1152, 2304) and obtained throughput of 300 Mbps for 20 iterations and 400 Mbps for 15 iterations. The design is done by using BRAM, which makes throughput slightly reduced with the gain of saving of logic slices. Quantization format of (6,4) is used in [7], which is flooded architecture with the requirements of Variable nodes and Check nodes depends upon the columns and rows of the parity check matrix H. Partially parallel design for different rates is proposed in [8], which gives throughput of 100 Mbps with 12 iterations. This design is based on offset BP algorithm. This architecture use BRAM for update of Check node process and Variable node process. This design used input and output buffer for storing incoming code and decoded code respectively. The throughput of 104 Mbps, 136 Mbps and 168 Mbps for the rate of 13/25, 17/25 and 21/25, is achieved in [8]. It uses slice registers, LUTs and block RAM on average of 20%. Reconfigurable layer decoding is suggested in [9] with throughput of 382-1852 Mbps for 5 iterations and 4-bits of quantization. It is tested on Stratix-IV for code length of 648, 1296 and 1944 for half rate.
III. CODE CONSTRUCTION OF PROPOSED LDPC CODE
A. Code Construction Methods
The LDPC code construction are of two types: (1) Random construction and (2) Structured Construction. Construction of both types might be regular or irregular type. The regular constructed LDPC code contains equal degree or weight of rows and columns as wr and wc respectively. While the irregular type construction having different weights of rows and columns. Structured construction LDPC code can be represented either graphically known as Tanner graph or by using parity check matrix.
1) Random Construction: Random constructed code gives good BER performance compared to structured design but implementation of random constructed code is more complex which requires more resources except that it having good girth and degree distribution. Progressive edge growth (PEG) and bit filling are two different types of algorithms, used to construct a code using random construction [2].

2) Structured Construction: This code construction is either based on combinatorial design or algebraic design. The construction based on row circulant permutation matrix is a type of combinatorial type construction while block circulant type code construction is a type of algebraic type code construction. Proposed design is using isolated shifted identity matrices is type of algebraic code construction. Structured construction requires less resources compared to random construction.

B. Encoding and Decoding
Encoding can be done either by linear time encoding or using Gauss-Jordan elimination method to create a generator matrix [2]. Proposed work is implemented using Gauss-Jordan elimination method. Decoding can be done either by hard decision decoding algorithm or soft decision decoding algorithm. Proposed decoder is implemented using min-sum message passing soft decision decoding algorithm which have less computational complexity with cost of less accuracy compared to other decoding algorithm such as sum product algorithm (SPA).

IV. MATLAB IMPLEMENTATION OF QC-LDPC CODE
MATLAB implementation is used to design the parity check matrix H of (3, 6) regular LDPC code using various permutation matrices. The size of permutation matrix is different for different code length. The parity check matrix is used to create generator matrix using Gauss-Jordan elimination method for encoding of the message. Encoded message are transmitted from transmitter and passed through the AWGN channel and received at the receiver for decoding. The proposed design use soft decision min sum decoding algorithm using 4-bit quantization of LLR using fixed point arithmatic. Each output

Soft decision decoding is a iterative message passing decoding algorithm in which messages are passed between Variable nodes and Check nodes for a given number of iterations or till the exit condition c · HT = 0 is satisfied. Proposed design, LDPC decoding converges well at around 30 iterations as shown in Fig. 1. Therefore, 30 iterations are considered for design of code length 222, 546, 642, 648 and 1152.
After completion of all iterations, it gives decoded output which is the accumlation of LLR’s of channel and connected Check nodes messages of the particular Variable nodes.
V. PARALLEL ARCHITECTURE OF QC LDPC DECODER
The general architecture of decoder design is shown in Fig. 2. Decoder have four different unit Inputnetwork, Outputnetwork, Variable nodes—Check nodes permutation network and Control unit. In which Control unit is the heart of the decoder that control Inputnetwork, Output network by enabling it at the start and end of decoding process respectively. While the Variable nodes and Check nodes of permutation network are enabled in alternate manner as per given number of iterations. Proposed design is implemented using fully parallel architecture. In which Variable nodes and Check nodes process takes single clock cycle only that will increase the throughput with cost of increasing hardware. Brief of all these blocks are described next:
A. Input Network (Inputnetwork)
Inputnetwork is made of multiple registers connected in pipeline. It takes quantized LLR value as a input and moves it into the permutation network of Variable nodes and Check nodes. Number of clocks to collect input data depends on folding factor (Foldfactor ) at the Inputnetwork. Therefore, it takes Foldfactor clock cycles to collect input data code for length of n bits, where (n/Foldfactor ) inputs are given to the decoder at a time. Foldfactor is decided according to available inputs pins on FPGA.

B. Output Network (Outputnetwork)
Similar to Inputnetwork, Outputnetwork is also made of multiple registers connected in pipeline. It takes decoded output from the permutation network of Variable nodes and Check nodes.

C. Control Unit
Control unit is responsible for generating control signals for various units such as Inputnetwork, Outputnetwork, Variable nodes—Check nodes for proper synchronization during decoding operation.
In proposed design, decoder takes one clock cycle for variable node and one clock cycle for check node in one iteration. For 30 iterations, (30 × 2) = 60 clock cycles are required for completion of decoding process. Decoded output takes one clock cycle to be made available for the output network in each design.

VII. FPGA IMPLEMENTATION OF DECODER
Fig. 2 shows interfacing of FPGA board with PC through UART. Proposed design tested on Virtex-5 FPGA board with board frequency of 100 MHz. Intrinsic LLR values (4-bit quantization) are transmitted from PC to the FPGA board using MATLAB serial object through UART. Decoding is performed inside the FPGA board and final decoded output is sent back to PC through UART. Proposed design uses register to store encoded messages, sent from PC then decode it. After decoding, it is stored in output register and send back to PC shown in the Fig. 2.
A. Comparision with Existing Work
Resource requirements for the decoder design of different code lengths (222, 546, 642, 648 and 1152) is shown in Table II.
1) Comparision of BER Plots: Observation in Table I shows that the coding gain of 648 bits length code is 1 dB more than the existing BER. Observation in Table II shows that comparison of resource utilization of our implementation with existing work.
2) Comparision of Fully Parallel Design with Partially Parallel Design: Observation in Table III shows that comparison of resource utilization of our implementation with existing work of partially parallel decoder. We can observe from the Table III that the throughput is less in partially parallel design

but it require less slice registers and LUTs compared to fully parallel architecture so, its a trade-off between cost and speed.
VIII. CONCLUSION
In all these designs, obtained BER of 10−5 is around 4 dB. Observation shows that as the length of code is increased, its BER is improved with increased throughput. Due to increase in code length the minimum distance also increased. From the comparision table of BER, we conclude that the proposed design performs well for code 648 with coding gain of 1 dB compared to existing work done by author [10]. Additioanlly, proposed design have throughput of 2 Gbps.
