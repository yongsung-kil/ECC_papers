Parallel LDPC Decoder Implementation on a Mobile GPU

Aleksei Kharin, Igor Volkov, Evgeny Mirokhin, Konstantin Terekhov, Aleksei Dryakhlov, Konstantin Zavertkin, Aleksei Ovinnikov, Evgeny Likhobabin and Vladimir Vityazev, Member, IEEE

Abstract—In this paper we explore acceleration of the belief propagation (BP) and BP-based algorithms for decoding LDPC codes, using parallel computation on graphics processing units (GPUs) for mobile hardware platform.
Different modifications of BP decoding algorithm are considered in search for best error correction performance in limited decoding time. Possible combinations of decoding algorithms, number precision and compute resources were found. Error correction performance for each found combination was measured.
Best combination of decoding algorithm, number precision and compute resource was found for data transmission system with LDPC coding for DVB-T2/S2 (16200, 7200) code.
Index Terms—ARM, Mali, LDPC, OpenCL, GPU, SDR.
I. INTRODUCTION
Modern digital telecommunication equipment, such as transmitters or receivers, are quite complex devices in terms of computational complexity. In fact, their computational costs force to use really powerful hardware implementations, that utilize application-specific integrated circuits (ASICs). Of course, they are superior in terms of power consumption and throughput, in comparison with general purpose processors, but they don’t have such a flexibility as a software implementation. Which allows to make modifications or fixes easy, even after deployment. Moreover, it allows to use the same hardware for different systems. Which in summary sufficiently reduces deploy and support costs, that is very attractive for industry. Technology, which allows to make software implementations of different telecommunication systems is called software defined radio (SDR).

But the main challenge for such projects is the real-time constraints, that should be satisfied. Therefore many SDR implementations required very powerful processing units. And vast majority of implementation used very powerful CPUs, GPUs or FPGAs.
Nowadays mobile CPUs provide sufficiently high performance processors, taking into account number of different types of co-processors, GPUs, DSPs and other accelerators. Such mobile platforms have referenced as systems-on-chips, and offered [3] a convenient conditions for implementation of SDR receivers, on the assumption of utilizing all accelerators.
Forward error correction (FEC) is one of the most complex procedures in every digital receiver. In this paper we were focused on software implementation of decoder for low density parity check (LDPC) codes. It is one of the most popular class of FEC codes, that can achieve very high error performance, but at the expense of high complexity.
There are some research works that devoted to implementation of different LDPC decoders on low-power ARM processors [3], [4]. In the present paper, we provide details about implementation of several floating point LDPC decoders on GPU of ARM processor. The aim of this paper is to explore the performance of the ARM processor’s GPU for different decoding algorithms, check if it satisfy real-time constraints for SDR, and choose the best one for this mobile platform.
As a test code DVB-T2 Lite [5] LDPC code was chosen, that require complex enough decoding procedure, and has not so high real-time constraints.

II. LDPC CODES
The low-density parity check codes were proposed by R. Gallager [6] in 1963 and later rediscovered by MacKay and Neal [7]. Like turbo codes, LDPC codes can achieve near the

Gallager also proposed algorithms for decoding LDPC codes. The most powerful of them is near optimum probabilistic belief-propagation algorithm, also known as sum-product algorithm (SPA) [6]. The main advantage of BP decoding algorithm is high performance, that allows to achieve near the Shannon limit, and the main drawback in it is complexity, that remains relevant nowadays. Therefore a lot of simplified modifications of this algorithm were proposed [9]-[10] for different applications.
Any LDPC code can be represented as a check matrix which contains mostly zeros and only a small number of ones. A regular (N, K) LDPC code is represented as the matrix H which includes M ≥ N − K rows and N columns. We refer to the M rows of H as check nodes, and to all the elements of a LDPC codeword as variable nodes.
Also define N (i) the set of ones in i-th column, M (j) the set of ones in j-th row of matrix H. N (i) − {j} and M (j) − {i} are the same sets, but without j-th and i-th positions respectively.

During the research several LDPC decoding algorithms were used. All of them are based on the logarithmic likelihood ratio (LLR) BP decoding algorithm [6]. Its formalization is presented in Algorithm 1. Further each used algorithm considered in details.
LLR BP decoding algorithm implementation based on the numerical approximation of box-plus operator with Jacobi logarithm is called LLR Jacobi BP decoding algorithm, where Jacobi logarithm approximation is represented as [10]:

In this way computation of Lsum is reduced to four addition operations, one compare operation and two correction term finding operations. Besides, the function g(L) can be calculated directly or by one of the simplified methods [10].

One of the most popular simplified methods of g(L) calculation is the table method called LLR Jacobi Look Up Table BP (LLR Jacobi LUT BP) decoding algorithm [10].
A better precision for g(L) calculation can be achieved by piecewise-linear approximation. With this modification, called LLR Jacobi PL BP decoding algorithm, the logarithm function is approximated by a finite set of lines.
A special case of this method in which only one line is used is called the LLR Jacobi Line BP algorithm [10], where the approximating line g(x) is defined as:

IV. SIMULATION ENVIRONMENT
For our simulations we have used simulation model from [11]. The main advantage of this model is a high flexibility, it allows to run the developed model on various platforms and OpenCL devices.
The OpenCL technology is an open framework for a software development for many types of heterogeneous systems [12]. Any OpenCL system can be represented as a pair of two main blocks: host and the OpenCL device. Each OpenCL device contains processing elements grouped in several compute units. Using GPU with OpenCL support for LDPC codes decoding reduces the time of one decoding iteration and significantly increase the throughput of the decoder. OpenCL implementation of the LLR BP is described in details in [11].
In this work we run the model on heterogeneous multiprocessing ARM computing platform Odroid XU4. Its main processing unit is Samsung Exynos 5422 Cortex-A15 2Ghz and Cortex-A7 Octa core CPUs. And it also includes MaliT628 MP6 mobile GPU. Which consists of 2 OpenCL devices with 2 and 4 compute units respectively. The model was run under Ubuntu Linux 16.04.02 operation system based on kernel version 4.9.30-41. GCC 5.4.0 compiler was used.

We chose DVB-T2 standard for our simulation, for this standard Lite extension was introduced recently. The extension targets mobile platforms, therefore it suites our hardware platform well. For coding in DVB-T2 Lite LDPC code of 16K length is used. With DVB-T2 Data-Rate Calculator [13] we estimated data-rate, which we should process on our target platform in real-time. With next parameters: Bandwidth 1.7 MHz, Pilot Pattern PP1, FFT 16k, Guard 1/4, Modulation QPSK, Code Rate 1/2, LDPC 16k, Data Symbols 22, FEC Blocks 34 - Brutto bit-rate is 2 281 877 Bit/s. To be able to operate at this bir-rate we have to process roughly 140 codewords per second. Thus, decoding latency for single LDPC codeword should be less than 7 milliseconds. This constraint will be taken into account later in this work.
