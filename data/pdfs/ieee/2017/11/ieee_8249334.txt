Flexible Parallel Implementation of LLR BP Decoding Simulation on Multicores Using OpenCL
Igor Volkov, Aleksei Kharin, Aleksei Dryakhlov, Evgeny Mirokhin, Konstantin Terekhov, Konstantin Zavertkin, Aleksei Ovinnikov, Evgeny Likhobabin and Vladimir Vityazev, Member, IEEE

Abstract—In this paper we explore acceleration of the logarithmic likelihood ratio (LLR) belief propagation (BP) algorithm for LDPC codes decoding simulation, using parallel computation on graphics processing units (GPUs) with OpenCL technology.
The proposed parallel implementation of the LLR BP algorithm includes parallel version of all decoding steps: decoder initialization, check and variable nodes update, a posteriori LLR calculation and decoding termination check.
Decoder throughput was measured by simulating data transmission system with LDPC coding for DVB-T2/S2 (64800, 32400) and PEG (1008, 504) codes.
Index Terms—belief propagation, LDPC, OpenCL, GPU.
I. INTRODUCTION
Low-density parity check codes were proposed by R.Gallager [1] in 1963. But they were forgotten for almost 40 years due to the high complexity of their iterative decoding algorithm. Interest for this codes increased with the discovery of turbo codes [2]. Nowadays LDPC codes are widely used in many modern communication standards like DVB-T2, DVBS2, DVB-C2, WiFi, WiMax, IEEE 802.15.3.
Gallager also proposed algorithms for decoding LDPC codes. The most powerful of them is near optimum probabilistic belief propagation (BP) algorithm, also known as sumproduct algorithm (SPA) [1]. Moreover, Gallager proposed its logarithmic interpretation, which significantly easier in implementation and has higher numerical stability then common BP algorithm [3]. This algorithm is also known as logarithmic likelihood ratio BP (LLR BP) or logarithmic SPA algorithm (LSPA), and due to its advantages is more widely used then initially proposed probabilistic interpretation.

As it was mentioned above the main drawback of BP decoding algorithm is its complexity, so a lot of simplified modifications of this algorithm were proposed [4]-[6]. Also, methods of accelerating this modifications are developed, including the parallelization of decoding algorithm and transferring calculations to GPUs [7]-[12].
But for some applications it is necessary to use the most powerful near-optimal BP decoding, for example code construction, trapping sets searching [13] and error floor estimation [14]. Of course this applications have very high computational costs and this kind of simulations can take weeks and months to complete. For such applications it is very critical to have a fast implementation of the LLR BP decoding algorithm. And in this paper we propose to use the OpenCL technology for the LLR BP decoding algorithm to increase LDPC decoder throughput and reduce overall time for simulations in such a complex applications.
The commonly used approach for LDPC decoding parallelization on GPUs is a parallel implementation of whole data transmission system model [7] or running a several copies of LDPC decoder in parallel with different codewords [8]. However, in practice (for example, for different types of GPUs) there is no opportunity for a parallel implementation of all simulation model due to hardware restrictions. In such conditions parallel implementation of only one LDPC decoder block is very useful, and further in this paper we will consider this approach in details.
The paper is organized as follows. In next section, we give definitions and notations that are used in this paper, a brief review of OpenCL technology and description of the data transmission system model used for the simulations. In Section III the main steps of the LLR BP decoding algorithm and their parallel OpenCL implementations are presented. Simulation results for proposed algorithm for two codes are provided in Section IV. Finally, Section V concludes this paper.
II. PRELIMINARIES
A. Definitions and notations
Any LDPC code can be represented as a check matrix, which contains mostly zeros and only a small number of indices of ones. A regular (N, K) LDPC code is represented as the matrix H which includes M ≥ N − K rows and N columns. Each column contains exactly dc indices, and each row contains exactly dr indices. B. OpenCL technology
The OpenCL technology is an open framework for a software development for many types of heterogenous systems. It allows to implement the GPGPU (General-Purpose Computing for Graphics Processing Units) conception [15].
Modern GPUs contain huge amount of processing elements compared to a CPU’s (for example, nVidia Tesla K40 contains 2880 processing elements grouped in compute units). This option makes GPUs very useful for parallel computations.

For all steps of LLR BP algorithm parallel OpenCL modifications were developed. Lets consider the differences between common sequential steps and their parallel OpenCL modifications.
Step 1. Decoder initialization. In the OpenCL implementation of this step, in addition to initializing the initial values of the variable nodes Li, processing data is loaded into OpenCL device memory. Compilation consists in converting the original OpenCL C code into an assembler code of a selected OpenCL device.
Step 2-3. Check and variable nodes updating (figure 3). Parallel implementation of steps 2 and 3 implies simultaneous updating of several nodes. Thus, U messages are calculated at the same time. In other words, each compute unit calculates its message Lj→i during check nodes updating, and Li→j during variable nodes updating step.
Step 4. Posteriori LLR calculation (figure 4). The main idea of the OpenCL implementation of this step is similar to step 2

and 3: U posteriori LLRs are calculated simultaneously - one LLR per one compute unit, instead sequential calculation of N a posteriori LLRs.
Step 5. Hard decision making (figure 4). After LLR calculation on step 4, each compute unit compare the resulted LLR with zero and make hard decision about corresponding bit in received codeword. At this step, similar to the previous steps, U hard decisions are made at once.
Step 6. Decoding termination check. The hard decision is divided among the compute units. Then each unit compares its part of syndrome with zero vector. If at least one part of the syndrome is not equal to zero vector, the hard decision is considered as incorrect.
IV. SIMULATION RESULTS
During the simulations the test platform with the following parameters was used: CPU - Intel Xeon E5-2620 2 GHz (6 Cores), RAM - DIMM 64 GB RAM DDR-4 PC4-17000, GPUs - NVIDIA Tesla K40 (Kepler) 2880 CUDA Cores, 12 GB GDDR-5 RAM (288GB/s) and AMD Radeon R7 370 2 GB GDDR-5.
Decoder verification was made by comparison results of the developed simulation model with results of the MATLAB LDPC decoder [16] for the several LDPC codes, here results for only two codes are presented, for regular PEG (1008, 504) [17] and DVB-T2/S2 (64800, 32400) codes. For the decoder performance measurements, the following model parameters were set: modulation type - BPSK, max decoding iterations - 20. All calculations for decoding steps were performed with double precision. The simulation results for the sequential implementation of the decoder and for the parallel implementation with several OpenCL devices are presented in the table 1.
Average time titer [ms] of one decoding iteration was measured for two considered codes: PEG (1008, 504) and DVB-T2/S2 (64800, 32400).
As shown in table 1, the parallel OpenCL implementation with NVIDIA Tesla K40 GPU allows to speed up the decoding process more than 53 times for the DVB-T2/S2 (64800, 32400) code.
However, the resulting speed up depends on the LDPC code parameters (on the size of the check matrix H). For example, the decoding speed for the code (1008, 504) was increased only in 4.6 times. This is due to the fact that the process of decoding PEG (1008, 504) takes less computing time than the decoding of DVB-T2/S2 (64800, 32400), while the overhead costs associated with organizing parallel computations are approximately equal.

V. CONCLUSION AND FUTURE WORK
Developed implementation allow to simulate LLR BP LDPC decoder with more than 53 times increased throughput for NVIDIA Tesla K40 and DVB-T2/S2 (64800, 32400) code. This gain is achieved by parallel implementation of the LLR BP decoding algorithm and using graphical processing units (GPUs) for computations.
