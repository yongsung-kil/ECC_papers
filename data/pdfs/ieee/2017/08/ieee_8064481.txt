Enhancing SSD Performance with LDPC-aware Garbage Collection

Yajuan Du∗†, Yunpei Jia∗, Meng Zhang‡, Jun Zeng∗, and Chun Jason Xue† ∗School of Computer Science and Technology, Huazhong University of Science and Technology, Wuhan, China

Abstract—With the technology scaling of solid state drives (SSDs), data reliability has to be ensured by advanced LowDensity Parity-Check (LDPC) codes. As LDPC reads often take high latency, read performance involved in traditional garbage collection, called external GC, is largely decreased, which affects system performance. GC can choose not to correct data with LDPC codes but just copy valid pages into registers of NAND flash, which we call internal GC. Although this method avoids long latency in LDPC reads, the data integrity is sacrificed because of error aggregation in valid pages. To balance performance and reliability, this paper proposes a new garbage collection method, named as LDPC-aware Garbage Collection (LaGC), to selectively choose blocks with high error rates to perform external GC. These blocks are often close to uncorrectable error rates, which take a small portion during a long flash life period. Thus, most of blocks perform internal GC and keep advantage of low read latency. Experimental results show that LaGC can reduce 78% of read latency in external GC and achieve about 2% system performance improvement.
I. INTRODUCTION
Solid State Drives (SSDs), as the main storage device of phones and personal computers, provide higher [1] access performance than Hard Disk Drives (HDDs) by the physical nature that no mechanical moving happens. However, there are several shortcomings in SSDs, which degrade its potential performance advantage [2]. Garbage Collection (GC) often takes long time to perform page migrations and block erases. Furthermore, as SSDs have much higher RBERs than HDDs because of flash errors [3], Low-Density Parity-Check (LDPC) codes [4], [5] have to be applied to guarantee data reliability. As high latency is involved in LDPC reads [6], more page migration costs have to be paid in GC. Thus, SSD performance is further decreased.
In order to achieve less GC effects on write amplification, lazy GC schemes are popularly applied in SSDs, which often happen when free storage space is not enough for upcoming requests. Latency caused by this kind of GC largely affects on system performance. Existing methods improve SSD performance by reducing GC frequencies or page migration numbers [7]. This paper considers to solve this problem in view of optimizing LDPC read performance involved in page migrations. According to different read mechanisms, GC can choose to perform GC reads externally in controller or internally in flash registers. External GC decodes, corrects and re-encodes data

in valid pages by LDPC codes in controller while internal GC only reads data into registers and then programs to new pages. The first mechanism can lower RBERs and avoid read failure but induces worse read performance, which further decreases GC performance as well as SSD performance while the second mechanism has better GC performance without long-latency LDPC reads but cannot ensure data integrity.
This paper proposes LaGC, a LDPC-aware garbage collection method, to improve GC performance of SSDs. LaGC selectively performs LDPC error correction in GC process, i.e. external GC, on blocks according to their retention time. LaGC only chooses long-kept blocks for external GC, that are prone to read failure because of high RBERs. As most of blocks are far from read failures, internal GC happens and brings high read performance. All in all, LaGC not only develops latency advantage of internal GC but also avoids read failures by external GC. By testing performance of six real workloads in Disksim platform [8] with SSD extensions [9], LaGC can reduce 78% of read latency and improve about 5% of GC performance compared with external GC.
The contributions of this paper are as follows:
• Studies the tradeoff between performance and reliability of two read mechanisms in GC with real workloads.
• Proposes LaGC to selectively choose page read methods in GC by exploiting the above tradeoff. LaGC can achieve a good balance between performance and reliability.
• Evaluates LaGC with six real workloads in simulation platform Disksim and verifies its effectiveness on read latency involved in GC, GC performance, and SSD performance.
The rest of the paper is organized as follows. The background and motivation are presented in Section II. Section III presents the proposed LaGC. The evaluation is shown in Section IV. Related works are described in Section V. We conclude this paper in Section VI.
II. BACKGROUND AND MOTIVATION
This section first introduces the basic operations of SSDs. Then, two read mechanisms in garbage collection are illustrated and analyzed.
A. Basics of SSDs
SSDs, usually adopting NAND flash as storage medium, are mainly made up by two components: the controller and the

flash packages. The controller is the core of SSDs, in which there are many software modules to respond to host requests and manage flash packages, such as the Flash Translation Layer (FTL) module, the LDPC module and the GC module.
There are three main operations on SSDs: read, write and erase. Read operation, the basic unit of which is page, mainly goes through four steps: address translation with FTL, sensing flash data to register, transferring data to controller, and LDPC decoding in controller. The LDPC read process is performed iteratively with read-rety steps until the decoding succeeds. If data kept time becomes longer, read latency will be higher. In write operation, data is written to corresponding physical pages with an incremental-step-pulse programming method [10] .
Different from HDDs, SSDs have block erase operation because of the out-of-space update nature. Write amplification is induced by GC because of migration of valid pages in victim blocks. In order to reduce the write amplification, garbage collector often chooses the block with the most invalid pages as the victim block. After migrating data of valid pages, the victim block will be erased to recycle.

A GC scheme to perform LDPC error correction in page migrations is called external GC, which can ensure data reliability. As shown in Figure 1(a), when the controller sends a GC command to flash package, the external GC begins to perform five steps as follows: Step a1, The controller reads a page from the flash into the register; Step a2, The data stored in the register are transferred to the controller by controllerto-flash bus; Step a3, LDPC codes are used to decode the data and encode the error-free data again within the controller; Step a4, The controller returns re-encoded data back to the register; Step a5, The data are written to new places from register. At the expense of latency caused by LDPC decoding, external GC can correct errors in data and ensure data integrity.
Compared with external GC, internal GC has less steps because no LDPC decoding happens in page migrations. As shown in Figure 1(b), valid pages only communicate with register by two steps: b1 and b2. In Step b1, valid pages in victim blocks are read into the register while in Step b2, the

data in register are written back to new blocks. As both steps take much less time than the that in LDPC reading, the internal GC induces less read latency than external GC.

III. LDPC-AWARE GARBAGE COLLECTION
This section presents the design of LaGC in details. Figure 3 shows the architecture of SSDs integrated with LaGC. One storage module and one functional module are developed in LaGC: the Block Retention Information (BRI) and the Garbage Collection Assistant (GCA). The BRI module provides the information of block data kept time, that helps GCA to determine which GC read methods to use and the GCA module assists garbage collector when performing GC.
A. Maintaining of Block Retention Information
The BRI module contains two sub-modules, one in flash and the other in controller. The sub-module in flash stores all BRI, i.e. the time of first page written in spare area of each block. The other sub-module in controller, called Block Retention Table (BRT) always stores BRI of part of blocks with least valid pages. Each BRT entry contains a block number and first page time of the block. When invoking GC operations, the BRT is first read to provide information for GCA module. Every time during GC, the garbage collector looks up the BRT to obtain the corresponding data kept time, which assists the GCA module to make decisions. The data kept time can be computed by subtracting the stored time for each block with current time. All values in the BRT are initialized as 0 when SSDs begin to be used. There are two management scenarios to update the values of BRT. The first scenario is that when receiving a write request to a new block, first page time of the corresponding block is updated to be the current time. The second scenarios is that when GC happens, BRI of the corresponding block as well as its entry in BRT are reset to be zero after block erasing. The two sub-modules in BRI is dynamically maintained to ensure the timeliness of information for block data kept time.
B. Garbage Collection Assistant
As shown in Figure 4, the GCA module contains two submodules: the valid page detector and the LDPC selector. The valid page detector is used to determine whether the victim block contains valid pages. If there are valid pages in the victim block, the read operation should migrate valid pages to other blocks. The LDPC selector is activated to determine whether to perform LDPC decoding by judging the data kept time of the victim block. Given a fixed RBER threshold that is close to read failures, data kept time threshold are adjusted according to P/E cycles. We obtain the different data kept time from the error model in [11]. The LDPC selector identifies the dangerous blocks and performs LDPC reading on the valid pages of them. Conversely, for blocks with data kept time smaller than the threshold, the LDPC selector performs internal GC.
C. The Process of Garbage Collection with LaGC
With the support of above two modules, LaGC can make effective GC, details of which are listed as follows: Step 1: GC request is received by the garbage collector, and
the victim block detector selects a block with the most invalid pages as the victim block. Step 2: The garbage collector accesses the BRT to obtain the data kept time information of the victim block.

Step 3: The data kept time information is sent to the GCA. And the valid page detector determines whether there are valid pages in the victim block. If no, go to Step 9. Otherwise, go to Step 4.
Step 4: The valid page detector sends block information to the LDPC selector for further operations.
Step 5: The block kept time is beyond the threshold, the GCA module decides to decode the data in the LDPC decoder.
Step 6: The block kept time is less than the threshold, the GCA module decides to read the data of valid pages to register in garbage collector and then perform a write operation.
Step 7: After decoding the data successfully, the LDPC Decoder performs the write operation.
Step 8: The erase operation is launched by garbage collector in order to recycle the victim block.
Step 9: Garbage collector updates the BRT entry.
D. Analysis of Overhead Involved in LaGC
The storage overhead in LaGC comes from two submodules in the BRI. In all-BRI sub-module in flash spare areas, 2 Bytes are used to store data kept information of each block which lies in the spare area of SSDs. Assuming there are 8 packages in the SSD system with size of 32G, each package contains 8 planes and each plane is made up of 1024 blocks. The storage overhead of the BRT with all block entries is 128M, which is 4% of the SSD. We assume the BRT in flash controller contains 100 entries, this overhead is negligible.
IV. EVALUATION
1) Experiment Setup: In order to evaluate the effectiveness of LaGC, six workloads have been run to simulate a SSD system in Disksim [8] with SSD extensions [9]. The total size of the flash system is 32GB and it is configured by eight packages, which contains eight planes. Each plane is made up of 1024 blocks and each block contains 64 pages. The read, write and erase latency is set to 25µs, 200µs and 2ms respectively.
According to the error model proposed in [11], we set the BER0 as 0.006 and P/E cycle as 8000. In order to correctly simulate the situation of SSDs in the late lifetime stage, we set the value of each block’s retention time as one year.

V. RELATED WORK

Related work about optimizing the GC performance can be categorized into two methods. The first method is to reduce the number of GC operations. Hahn et al. [12] proposed a justin-time garbage collection method to avoid conflicts between

aggressive GC and lazy GC. Background garbage collection [13] is proposed to perform garbage collection in background when system is idle, which can largely reduce GC effects on performance. The second method is to select the victim block with the least number of valid pages i.e, to reduce write amplification. Analytical models [14], [15] are proposed to analyze the factors to cause amplification and tries to optimize these factors in theory. Lu et al. [16] proposed to reduce write amplification by considering applications in file system level. Our paper proposes a new view to improve GC performance.
VI. CONCLUSION

In order to improve GC performance of SSDs, this paper proposes LaGC, which can intelligently decide page read method during GC according to block data kept time. Experimental results show that LaGC can balance the tradeoff between reliability and performance.
