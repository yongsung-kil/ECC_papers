Abstract—It is well-known that the Sum-Product Algorithm (SPA) is an efficient algorithm for decoding Low Density Parity Check (LDPC) codes. In spite of this, for LDPC codes with medium length which are now attracting interests for the application to fifth generation systems, it is known that decoding error rate is much inferior to that of Maximum Likelihood Decoding (MLD). In this paper, we improve bit error rate of SPA with so called Ordered Statistic Decoding (OSD) method by Fossorier [1]D Especially, we improve stopping condition of combinatorial search in OSD method and demonstrate by simulation that the proposal stopping condition is more efficient compared to Fossorier (IEEE J. Sel. Areas Commun., 19, 2001). For that purpose, we introduce a conception “coincidence rate” which plays an important role.
Keywords—LDPC code; OSD method
I. INTRODUCTION
It is well-known that the Sum-Product Algorithm (SPA) is an efficient algorithm for decoding LDPC codes which were originally proposed by Gallager[2]. It is also known that LDPC codes with SPA decoding, Bit Error Rate (BER) tends to zero as code length n goes to infinity under the condition that Signal to Noise Ratio (SNR) does not exceed Shannon-limit;see for example [4], [5], [6].
In spite of this fact, for codes with medium length (code length some hundreds to thousands bits) which are now attracting interests for the application to fifth generation systems, SPA decoding error rate is inferior to that of maximum likelihood decoding (MLD) to a considerable extent;see Fig. 1. Integer programming approach by Feldman, etc. [7] is a novel approach for achieving MLD, but is limited to the situation that codes length are very short, even if we apply current IP solvers.
For middle codes length cases, reliability based approach is one of the method which retrieves the gap between decoding error rate results between SPA and MLD. Among these, there is a method which uses k-most reliable independent positions (k-MRIPs for short) which is called Ordered Statistic Decoding (OSD);see Fossorier [1] and Section 10.8 of [3]. In general, OSD method tries to find a code word by evaluating suitable cost function, through the search of certain combination of variables of k-MRIPs. If we can determine current combination is satisfactory at that moment, without searching all combination of variables to be considered, decoding time will be

IV. STOPPING CONDITION
OSD-method cannot suspend instantaneously even if it finds the codeword which maximizes C(x) during the search process of (4). Therefore if we can determine x is a codeword which maximize C(x) or not instantaneously, searching time will be reduced drastically. First, we introduce the algorithm by Fossorier [1] and improves its search stopping condition.

Search Stopping Condition by Fossorier.

(F-1) (F-2) (F-3) (F-4)

Fix ψ (usually ψ = 2). Set F (ψ) = 0, Cmax = −∞, l = 0 and c be an output obtained by OSD algorithm.
if F(ψ) = ψ − 1 then c is a decoding result. stop. else find x˜(l) which maximize C(x) at l-th loop of (4) of OSD algorithm.
if C(x˜(l)) > Cmax then F (ψ) = 0, Cmax = C(x˜(l)) , c = x˜(l) else F(ψ) = F(ψ) + 1

V. NUMERICAL EXPERIMENT
We demonstrate numerical experiment to confirm the effectiveness of our search stopping condition. In each experiment sp-LS means the OSD method with proposed search stopping condition and Fossorier means the OSD method with Fossorier’s search stopping condition. In the case (n, k) = (256, 129), bit error rate of proposal method (sp-LS, l0 = 3) is slightly inferior to that of ([1], l0 = 4);see Figure 5, while cpu time is quite improved (proposal method is about 32 to 69 times faster than [1]);see Figure 6. Here we note CPU time of sp-LS and Fossorier are CPU time that took for “search stopping algorithm” part. So the total amount of CPU time must be sum-product CPU time plus “search stopping algorithm” CPU time. In the case (n, k) = (512, 257), bit error rate of proposal method (sp-LS, l0 = 2) is almost the same as that of ([1], l0 = 3);see Figure 7, while CPU time is quite improved (proposal method is about 78 to 198 times faster than [1]);see Figure 8. We note all experiment was done on DELL Precision T7500 (Intel Xeon X5680 3.33GHz).
