ABSTRACT
In this paper we propose to show how signal processing algorithm designers can understand the nuances of multicore computing engines in order to conveniently exploit these powerful devices. This is illustrated by presenting source and channel coding, two fundamental operations in multimedia signal processing. We describe methods and principles to develop parallel signal processing algorithms to compute motion estimation for Advanced Video Coding, and Low-Density Parity-Check code decoding for forward error correction in the channel coding context. The paper will consider general purpose multi-core architectures and accelerators such as the Cell/B.E. and graphics processing units. Experimental evaluation of the multi-core systems allows their performance for signal processing applications to be compared side by side with previous hardware dedicated solutions.
Index Terms— Multi-core, signal processing, coding
1. INTRODUCTION
Multimedia signal processing applications are usually demanding, largely due to the amount of computation required and to time restrictions. Many solutions therefore make use of hardware-oriented specialized processing architectures like Digital Signal Processors (DSPs), or one that are fully dedicated through the design of Application Specific Integrated Circuits (ASICs). The architecture paradigm has shifted to multiple cores on a single chip providing augmented computational power and achieving improved performance. To take advantage of this opportunity, the signal processing community has to design signal processing algorithms that expose independent computations at task and data levels. These can be executed in parallel according to the number and characteristics of the multiple cores available.
The main goals of this paper are to show how to implement changes into signal processing algorithms, making them suitable for processing on platforms with multiple cores, and to describe the techniques used to achieve efficient implementations. In a pragmatic approach, we identify two classes of

representative signal processing algorithms, well known in the signal processing community, with quite different computational characteristics and requirements but both hard to compute. We use them to illustrate the principles and techniques that can be exploited for their efficient implementation in multi-core platforms. Block matching Motion Estimation (ME) in Advanced Video Coding (AVC) and Low-Density Parity-Check (LDPC) code decoders are typical classes of algorithms that require a huge amount of computation and irregular access to data.
We will also describe the application of the proposed techniques and methods to program different multi-core platforms, namely the Cell Broadband Engine (Cell/B.E.), Graphical Processing Units (GPUs) and those based on general purpose multi-cores from INTEL and AMD. Experiments on those platforms show the scalability of the proposed solutions, which is quite an important aspect when future perspectives point out to platforms with hundreds and thousands of homogeneous and heterogeneous multi-cores.
2. SIGNAL PROCESSING FOR CODING
AVC is a typical example of a computationally demanding signal processing application. A careful and thorough analysis of recent AVC standards, such as the H.264/AVC [1], reveals that the major processing bulk is in block matching ME, which usually requires more than 60% of all the operations necessary for video encoding. ME in AVC standards exploits the temporal redundancy between frames in a block-based approach: a current frame (CF) is divided into Macroblocks (MBs), and the best matching candidate is found for each of them, considering search areas (SAs) in several reference frames (RFs) and a distortion measure (typically the Sum of Absolute Differences (SAD)). Both the optimal search and the sub-optimal fast and adaptive (UMHS) search algorithms are used [2]. Binary LDPCs are linear block codes represented by bipartite Tanner graphs that require computationally intensive iterative message passing processing associated with a huge number of arithmetic operations. For the Min-Sum algorithm used here [3], each iteration is computed in two main kernels, for horizontal and vertical processing, associated with the update of all nodes in the Tanner graph. The scheduling mechanism imposes important restrictions on the attempt to parallelize the algorithm. Furthermore, irregular memory access constraints imposed by quasi-random graph connections, and usually associated with large code dimensions that provide acceptable coding gains (low BER), create challenges which are difficult to overcome.
3. MULTI-CORE ARCHITECTURES AND PROGRAMMING MODELS
There are challenges associated with modeling and programming the different multi-core architectures for signal processing applications, in order to efficiently use the computing capabilities that arise from the orchestrated use of all individual cores. However, signal processing applications have in common the fact that multiple data has to be processed, usually in a stream based mode. Therefore, the Single Program Multiple Data (SPMD) and Single Instruction Multiple Data (SIMD) programming models can be adopted to exploit data parallelism, fostering parallel processing activity. The SPMD coarse grained data parallelism is usually supported on the multi-threading approach, while the SIMD principle is implemented by the fully synchronous execution of instructions in parallel, or by executing a single instruction operating in parallel in sub-words of a wider word kept in a data register. These two types of parallel models and architectural approaches coexist in the modern multi-cores considered in this paper, such as the general-purpose multi-cores, and the more

Tesla architecture is based on a shared-memory scalable processor array, with more than a hundred of programmable cores. The array consists of a set of independent clusters of cores that support floating-point operations. Tesla uses a new processor architecture called Single Instruction Multiple Threads (SIMT), for which each instruction unit creates, manages, schedules, and executes threads in groups of 32 parallel threads. The GPU computation is performed under the SPMD model by defining blocks of independent multiple-threads that can be mapped and executed in parallel in the clusters of cores, according to the SIMT architecture.
The heterogeneous Cell/B.E. distributed memory architecture consists of one Power Processor Element (PPE) and eight Synergistic Processor Elements (SPEs). Each SPE has 256 kB of Local Storage (LS) memory for both program and data, and a 128-bit wide vector oriented architecture. Cores are connected through the Element Interconnect Bus (EIB), and data transfers between them are explicitly performed by using Direct Memory Access (DMA). In order to achieve high processing efficiency in the Cell/B.E., data locality has to be exploited in each SPE and data communication and processing must be overlapped, namely by using the double buffering technique. Therefore, a higher effort is required to efficiently program the Cell/B.E.

4.1. Video ME
For parallel ME, as shown in figure 1, the video coder is running on PPE while ME is offloaded in parallel to the SPEs. All RFs and CF are placed in the PPE’s main memory, and divided into vertical chunks. This way, each SPE accesses a single CF chunk and related RFs’ chunks using DMA transfers. Due to the limited LS size, these chunks are further divided into horizontal portions that are processed in multiple independent iterations (dashed lines in figure 1). Parallelization is performed at the level of few-MBs-per-thread. Although it would be more efficient in terms of communication time and data reuse to process larger areas in each individual core, the size of the LS only supports data corresponding to a small number of MBs. A fast and efficient set of SIMD instructions is applied to speed up the SAD computation on the SPE, although the programmer has to perform the required data alignment. As soon as the ME process is finished, each ME thread sends the results back to the main memory [2].
The finest parallelization on best-matching-candidate perthread level was applied on the GPU/CUDA architecture with a huge number of cores and threads. Regarding the CUDA programming model where threads are organized in independent blocks, we decided that each thread block accesses a single MB and the required SAs from the multiples RFs, previously cached on the GPU shared memory (see figure 3). 4.2. LDPC Decoding
In the case of LDPC decoding on the Cell/B.E. shown in figure 2, the adopted parallelizing strategies are implemented following two different approaches: (i) decoding a complete codeword (applying both horizontal and vertical kernels per iteration) for small to medium LDPC codes and exploiting SIMD intrinsic instructions on each SPE [3]; or (ii) due to the limitation imposed by the reduced size of the LS, large codes have to be partially decoded on each SPE. This latter strategy demands heavier data traffic and congestion occurs between the PPE’s main memory and the SPE’s LS. One complete iteration involves several data transfers which causes performance in this case to be limited by bandwidth. The parallelization is achieved by using the 6 SPEs available on the platform under test to simultaneously decode 16 × 6 = 96 codewords. The PPE manages the orchestration of I/O, data transfers performed by DMA and kernels execution. Memory accesses are overlapped with computation thanks to the use of a double buffering technique.
The parallelization strategy adopted for the GPU using CUDA is based on a thread-per-node approach [6], as shown in figure 4 for both horizontal and vertical processing steps. The GPU does not have memory size limitations, but it is constrained by Video RAM memory access latency. To overcome this problem, the coalescence mechanism was fully exploited, producing more efficient parallel accesses to memory performed by groups of 16 threads (a half-warp). To achieve this, an automatic data mapping procedure was developed to perform run-time data realignment on the host, allowing coalesced memory accesses on the GPU [6].

All frames except the first one are coded as Pframes, with ME search range of 32×32, and a quantization step QP=28. Common Intermediate Format (CIF) (352×288 pixels) and High Definition (HD) (720×576 and 1280×720 pixels) video formats are considered. The choice of the sequences is not important, since the regular Full-Search BlockMatching with SAD reusing (FSBMr) was adopted. The results show that real-time ME is obtained for both GPU and Cell/B.E. for the CIF format. These results are much better than for Intel Quad Core implementation using Open MP. It should be noted that the speedup increases slightly with resolution, which shows the efficiency of the parallelization for HD video formats. Our experiments also show that in the case of Cell/B.E. the ME time decreases linearly with the number of cores, which shows the scalability of the solution, regarding to the number of cores, and the practical interest of the proposed parallel algorithms for the next generation of multicores.

6. CONCLUSIONS
In this article we have shown that recent multi-core platforms already support real-time computation of two known classes of intensive signal processing algorithms. This represents an important step towards the generalization of multi-core based solutions for many of today’s signal processing needs, and thus represents a real alternative to the dedicated hardware/ASIC solutions that impose a high level of non-recurring engineering. Moreover, this novel approach benefits the development of new signal applications and more complex and effective signal processing algorithms, by using both the advantages of programmability and massive processing power.
