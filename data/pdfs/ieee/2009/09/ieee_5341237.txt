Abstract—Recently, wireless local area network (WLAN) requires high-speed and high-reliability for multimedia access at home, office and etc. The WLAN standard: IEEE802.11n, which enhanced them compared with the existing IEEE802.11a/b/g, can achieve a maximum throughput of at least 100Mbps at the medium access control (MAC) layer. In this paper we report RTL design of a high-throughput LDPC decoder using the min-sum algorithm.
I. INTRODUCTION
Recently, wireless local area network (WLAN) requires high-speed and high-reliability for multimedia access at home, offices and etc. The WLAN standard: IEEE802.11n, which enhanced them compared with the existing IEEE802.11a/b/g, can achieve a maximum throughput of at least 100Mbps at the medium access control (MAC) layer.
Low density parity check (LDPC) codes had been standardized as optional high-performance error correcting codes (ECC) technique for high-throughput (HT) system on IEEE 802.11n-based WLAN system. LDPC codes which are among the most powerful ECC, can achieve performance closeto the Shannon limit.
In many cases, the min-sum algorithm is used for decoding LDPC codes [10], [11], [12], [13]. This algorithm achieves a performance level very close to, or sometimes evens out performing that of belief propagation (BP) decoding algorithms [14], while offering significant hardware advantages. Moreover, the architectural solution can also be applied with other decoding algorithms. A LDPC decoder using the minsum algorithm is simpler than LDPC decoder using other algorithms. Nevertheless, the LDPC decoder is very large because its bit width is very long.
To solve this problem, we consider reducing area of the LDPC decoder using the min-sum algorithm. In the minsum algorithm, row operations and column operations are alternately executed. Only row (column) operations in one iteration can be executed in parallel. However, row operation and column operation in one iteration cannot be executed in parallel. In this paper, we report RTL design of the LDPC decoder using the min-sum algorithm that can overlap execution of row operations and column operations. IV. LDPC DECODER ARCHITECTURE
In this section, we describe architecture of our LDPC decoder. Our LDPC decoder is based on the min-sum algorithm. The top level block diagram of our LDPC decoder is shown in Fig. 1. Our LDPC decoder is composed by a multiplier (MUL) unit, an input buffer (IBUF) unit, an output buffer (OBUF) unit, λ−, α− and β− memory units, horizontal processing units (HPUs), vertical processing units (VPUs) and a parity check unit (PCU). Brief explanation of these units are as follow.
A. MUL unit
The MUL unit is a multiplier which calculates the log likelihood ratio (LLR) λn = 2yi/σ2, where yi and σ2 represent the received word and the variance of the channel noise, respectively.
To reduce the delay time of the MUL unit, we employed a four-stage pipelined structure which is composed of a threestage carry save adder (CSA) and a carry propagation adder (CPA).
B. Memory units
The IBUF unit buffers LLRs for the next received word and the λ-Memory unit stores LLRs which are used for LDPC decoding. The OBUF unit buffers the tentative LDPC code word.
Besides these memory units, our LDPC decoder has the αMemory and β-Memory units store the extrinsic LLR αmn and the posteriori LLR βmn, respectively. The α-Memory and βMemory units store the extrinsic LLR αmn and the posteriori LLR βmn, respectively.
C. HPU
Our LDPC decoder has HPUs to execute the row operation of the min-sum algorithm. The HPU calculates the extrinsic

LLR αmn and each extrinsic LLR is stored into the α-Memory unit. The HPU can execute one row operation in one cycle. The block diagram of HPU is shown in Fig. 2.
Since the row operation is done using the sign-magnitude arithmetic, each HPU has 2’s complement to sign-magnitude converters at its input. As shown in Fig. 2, each HPU has eight 3-to-2 and seven 4-to-2 comparators to find the two least input magnitude values. Moreover, HPU has an XOR tree to decide the sign bits.
D. VPU
Our LDPC decoder has VPUs to execute the column operation of the min-sum algorithm. The VPU calculates the posteriori LLR βmn and each posteriori LLR is stored into the β-Memory unit. The VPU can execute one column operation in one cycle. The block diagram of VPU is shown in Fig. 3.
Since the column operation is done using the 2’s complement arithmetic and extrinsic LLRs stored into the α-memory are sign-magnitude binary, each VPU has sign-magnitude to 2’s complement converters at the inputs of it. As shown in Fig. 3, each VPU has a CSA tree and a two stage CPA to accumulate input values.

E. PCU
The PCU executes parity check of the min-sum algorithm. However, result of the PCU in our LDPC decoder is not used for loop condition evaluation. The PCU is simply to output the success and failure of a decoding process.
V. IMPLEMENTATION OF LDPC DECODER
In this section, we describe implementation of our LDPC decoder and consider supporting only the LDPC code for the codeword length 648 bit and the code rate 1/2 based on the IEEE 802.11n standard. Concerning execution cycles for decoding, a constraint that has to satisfy to support this LDPC code is stricter than the other LDPC codes for IEEE 802.11n. Therefore, our implementation can easily apply the other LDPC codes based on the IEEE 802.11n standard. The parity check matrix H of the LDPC code, codeword length is 648 bit and code rate is 1/2, is shown in Fig.4. Note that a value i of each sub-matrix in H represents a shift amount of the matrix Pi.

as shown in Table II, our LDPC decoder receives 2 symbols and outputs 2 bits in one cycle. Therefore, our LDPC decoder must read one LDPC code in 324 (= 648/2) cycles. A number of cycles for one iteration under these constraints is 16 (= 324/20). To represent internal data in our LDPC decoder, we used 6-bit fixed point representation.
As we said earlier, our LDPC decoder must execute one iteration within 16 cycles. HPU and VPU can execute one row operation and one column operation in one cycle respectively. When each HPU (VPU) in our LDPC decoder executes only one row (column) operation in one iteration, our LDPC decoder can execute one LDPC iteration in two cycles. In this case, our LDPC decoder is exaggerated performance and requires large area. To balance the trade-off between area and time, we consider that one HPU (VPU) executes some row (column) operations in one iteration. The size of the submatrix in the parity check matrix H as shown in Fig.4 is 27. When one HPU (VPU) executes 3 or 9 row (column) operations in one iteration since 27 is divisible by 3 or 9, our LDPC decoder becomes simple architecture and its control is

The H9-V9 LDPC decoder must execute all row operations for rows including black boxes in these gray columns before the 8th cycle in one iteration. The number of rows including these black boxes is 18. One HPU can execute 7 row operations in 7 cycles. Therefore, to start column operations for 1st, 10th and 19th column at the 8th cycle, at least 3 HPUs are required since 18/7 = 3.
Similarly, we consider for 2nd, 11th and 20th columns shown in Fig.10. The number of rows including black boxes except rows that are executed up to 7th cycle is 9. One HPU can execute 8 row operations in 8 cycles. Consequently, to start column operations for 2nd, 11th and 20th column at the 9th cycle, at least 4 HPUs are required since (18+9)/8 = 4. Here, since 18+9 = 27, 4 HPUs can execute all row operations for the 6th column sub-matrices of H up to 8th cycle in one iteration. As a result, our overlapped H9-V9 LDPC decoder that overlaps row and column operations must have 4 HPUs for the 6th column sub-matrices of H.
In the same way, we consider for whole H. Finally, our overlapped H9-V9 LDPC decoder must have 40 HPUs for whole H to overlap row operations and column operations. Our overlapped H9-V9 LDPC decoder requires the same 40 VPUs as Hp-V9 LDPC decoder. 217 VPUs are required in Hp-V3 LDPC decoders. VII. CONCLUSIONS
We have developed the LDPC Decoder using verilog HDL and synthesized it on RTL design. We presented an implementation of a LDPC decoder that can overlap row and column operations for the min-sum algorithm. Our overlapped H9V9 LDPC decoder can reduce the number of cycles for one iteration. Moreover, it can reduce the numbers of implemented HPUs and VPUs. As a result, our overlapped H9-V9 LDPC decoder can reduce its area. As a future work, we will design a LDPC decoder for all LDPC codes based on the IEEE 802.11n standard.
