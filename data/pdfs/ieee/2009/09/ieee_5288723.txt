LDPC Iteration Control by Partial Parity Check

William STOYE Staccato Communications
Cambridge, England william.stoye@staccatocommunications.com
Abstract—LDPC codes are increasingly popular in communication standards due to their excellent coding performance and resulting spectral efficiency. Decoding an LDPC is an iterative process, with more iterations using more power but allowing greater coding gain. An effective test for early termination allows control of the iteration count to give a low average number of iterations, resulting in both low power and high coding gain. The method described gives close to optimal iteration control for a useful class of LDPC codes.
Keywords: LDPC, low power, iteration control, WiMedia, UWB
I. INTRODUCTION
LDPC codes [1] are considered and used increasingly in new communication standards due to their excellent performance. Examples include IEEE 802.11n, IEEE 802.16e, UWB [2] and IEEE 802.15.3c. Many of these standards are important in mobile devices so reducing power consumption is of great importance.
The power consumed in LDPC decoding is proportional to the number of decode iterations so controlling the number of iterations can help to reduce power consumption. This subject has already been discussed in [3] and [7]. In [7] a metric is introduced which reduces iterations in low SNR conditions. In [3] an initial pre-check is introduced which removes the single unnecessary iteration in high SNR conditions. In the ‘waterfall’ region of the codeword they both rely on the parity checks inherent to the code, and this does not perform well compared to an optimal iteration control.
This paper takes a new approach based on the study of bit error distribution in codewords where decode has failed. The bit error distribution for decodes in the waterfall region is not uniform. In many practical codes the parity bits are more likely to be in error than the user data bits. In the parity bits there is a predictable error distribution. This suggests the use of a subset of parity bits as a checksum on the user data.
Simulations are presented based on one codeword being proposed for use in UWB communication. The proposed method gives very close to optimal iteration control in the waterfall region.
An implementation architecture is also presented which reduces the cost of the parity recalculation, so that the impact on circuit size is negligible.

II. LDPC CODES
LDPC codes are linear block codes defined by a sparse binary matrix H of size n×m. A column in H corresponds to a bit in the codeword, and a row in H corresponds to a parity check. The code can be represented as a Tanner graph [4], where a column in H corresponds to a Variable Node (VN) and a row in H corresponds to a Check Node (CN) in the Tanner graph.
In normal use the n VN are partitioned into n–m Data Nodes (DN) and m Parity Nodes (PN); the matrix H is organized as a number of z×z sub-matrices where each submatrix is either all-zeros, or is a cyclically right-shifted rotation of the identity matrix; and the parity-generation part of H is in lower triangular form, so that the encoding process is linear [5].
For instance, in a proposed code of rate ½ for ultrawideband communication at 640Mb/s we have n=1200, z=30 and the part of H corresponding to the parity nodes has the structure shown in Figure 1. Each digit shown there corresponds to a z×z submatrix of H, with 0 indicating an allzeros submatrix and any other value n representing the z×z identity matrix rotated right by n–1 places.
The order of a check node is the number of connected variable nodes, and in this codeword it varies between 5 and 6. The order of a variable node is the number of connected check nodes. The order of the DN in this codeword varies between 3 and 4 and the order of the PN varies between 2 and 3. We refer to the maximum order of CN, DN and PN in a codeword as Cmax, Dmax, and Pmax respectively. When the order of nodes varies the codeword is termed irregular.

The variable nodes are divided into groups of size z, anticipating that an implementation will execute z check nodes in parallel. In the diagram, z variable node values are loaded in parallel in each cycle of operation, z updated variable node values are being stored in each cycle, z check nodes are being calculated simultaneously, and we start to calculate another z check nodes every Cmax cycles of operation. The order in which nodes are loaded into the CNP is carefully chosen to allow layered decode to be undisturbed by this implementation pipeline.
In implementation various trade-offs are possible between hardware cycles, pipeline stages, and physical replication of the design elements in order to achieve an implementation of the desired characteristics but the overall logical structure portrayed here remains valid.
IV. LDPC BIT ERROR DISTRIBUTION
When an LDPC codeword is decoded by this method, the bit error distribution after decode is not uniform. As an exampole the code described above achieves a codeword error rate of about 1% at Eb/N0=2.0 in a Gaussian channel. Figure 3 shows the distribution of error bit positions after 5 iterations. The X axis is a bit position in the codeword and the Y axis is the number of bits in error for 1000000 codewords.

Figure 3. Distribution of errors after 5 iterations
The errors are not evenly distributed. As a general rule the variable nodes with lower order are more likely to be wrong. Intuitively this occurs because each check node applies a correction to the variable nodes to which it is attached, and so a VN attached to fewer CN will have less total correction applied overall.
The PN are mostly of order 2, and so are most likely to be incorrect at this stage of the decoding process even when the data is all correct.
The specifics are shown here for one code but the general lesson is applicable to many codes. If the error distribution is not even and if parity nodes are more likely to be in error than data nodes then early termination based on the check node parity checks is likely to be affected.
At higher iteration counts the wild imbalances in Figure 3 balance out. Figure 4 shows the matching error distribution after 20 iterations.

V. PARTIAL PARITY CHECK
A termination check can be performed by using the decoded DN to recalculate a subset of the parity values and comparing these with the matching subset of the decoded PN. In effect this treats a subset of the parity nodes as a checksum on the data nodes. The subset should be chosen for a particular codeword by selecting the most reliable parity nodes, that is the ones which are most likely to be correct based on the bit error distribution.
In this example codeword the first group of 30 parity bits (600 to 629 in Figure 3) is the most reliable. This is because those parity bits have order 3 where most other parity bits have order 2. (The visible slope on the error distribution of the remaining parity bits in Figure 3 is caused by the order of decode operations chosen for the layered decoding. In this example, the check node groups were executed sequentially.)
To check the effectiveness of this technique the example codeword is decoded with various iteration control methods:
• the inherent method, terminate when all check nodes have even parity
• the unimplementable genie method, terminate when the entire codeword is correct given foreknowledge of the transmitted data

• the unimplementable partial genie method, terminate when only the DN are correct given foreknowledge of the transmitted data
• the partial parity method as described above, terminating when the first 30 decoded PN for this codeword match the recalculated values from the decoded DN
• the VNR method described in [7] with various values of the MB parameter
The results are shown in Figure 5. The partial parity method is effectively identical to the unimplementable ideal of the partial genie. In normal operation the codeword will be used in the waterfall area and in this part of the curve the partial parity method shows a substantial reduction in iteration count when compared to inherent termination.
The parity check bits are not designed as a checksum but simulation shows that this does not lead to any measurable reduction in sensitivity.
Different iteration control methods can be combined in a practical implementation, giving excellent iteration control in high, waterfall and low SNR decode cases.

VI. IMPLEMENTING THE PARTIAL PARITY CHECK
In the case of this particular codeword the selected partial parity bit range is the first 30 parity bits. This is very like to be the case also with other codewords whose parity checks are designed in a similar way.
Much of the work to calculate the first group of parity values from the user data bits is available as a free byproduct of the operation of the decoder. Calculation of the parity values can be done by using XOR operations on groups of z bits as they emerge from the Check Node Processor (see Figure 2) because at that point each group is rotated by exactly the amount required by the encoding process. In the lowest cost implementation it will take one iteration time to calculate the parity bits in this way. This must be weighed against the objectives of early termination.
VII. CONCLUSION AND FUTURE WORK
In this paper we have presented a method of iteration control for LDPC decoders which can give results very close to a ‘perfect’ iteration controller for a wide range of useful codes. This can be used to reduce the average power consumption of an LDPC decoder in many communication protocols.
As well as reducing power consumption, iteration control can be used to improve performance in a real-time system by borrowing useful iteration time for one codeword to be used on another. As controlled using this method the average iteration requirement may be very much less than the limit. Improving performance in this way will typically require increased buffering of demodulated LLR values before the LDPC decoder and will increase the average and worst-case latency of the decode process.
Early termination improves the average latency of the decode operation but does not improve the absolute worst case latency. This can only be done by limiting the maximum number of iterations at which the decoder will run.

If the termination check fails then the codeword decode has failed with very high probability. If the codeword is part of a frame of user data which will be retransmitted in the presence of a checksum failure then it may be appropriate to stop decoding the entire frame in order to save power. Alternatively it may be appropriate to tell the next layer in the receiving protocol stack that this block is very likely to be incorrect, so that other recovery actions can be taken.
