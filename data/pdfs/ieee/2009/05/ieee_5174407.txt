LDPC Decoder Design for High Rate Wireless Personal Area Networks

Jin Sha, Jun Lin, Zhongfeng Wang Senior Member, IEEE, Li Li, Minglun Gao

Abstract — This paper presents two efficient decoder designs for the Low-density parity-check codes in IEEE 802.15.3 standard proposal. These decoders feature by efficient hardware usage, low message memory requirement and code rate flexibility. The highly parallel level design can reach a throughput of 3.6 Gbps, which fulfills the standard requirement by processing 72 columns and 72 rows in parallel. The low cost design offers another tradeoff which significantly reduces the area and power consumption while maintaining necessary data throughput required by specific applications. Furthermore, both decoders support three different code rates by employing flexible check node processing units1.
Index Terms — Low Density Parity Check (LDPC), decoder, VLSI, Error Correction Codes, Wireless Personal Area Network (WPAN)
I. INTRODUCTION
Low-density parity-check (LDPC) codes, one kind of linear block codes defined by very sparse parity-check matrices, were first introduced by Gallager [1] in 1960’s. These codes have been adopted by several industrial standards such as IEEE 802.16e, IEEE 802.11n and DVB-S2 due to their remarkable error correcting capabilities [2]. Some newly proposed standards, e.g., IEEE 802.15 wireless personal area networks (WPAN), are also considering using LDPC codes as the channel coding scheme [3].
WPAN standard enables short-range ad hoc connectivity among portable consumer electronics and communications devices [4]. Its target applications are mainly for in-house consumer electronics and portable communications devices. Bluetooth radio system, which features 1Mbps data rate at most, is the first technology addressing WPAN applications. Since 2001, 802.15.3 High-Rate WPAN Task Group has been chartered with creating much higher rate WPAN standard to meet the explosively growing communication requirements. These applications can be mainly grouped into two general categories. The first involves ultra high speed transmissions which require several giga bits per second data rate, and the other includes the distributions with data rates at a relatively low level, normally below 55 Mbps [3].

In the first category, one such application is uncompressed video transmission using a millimeter waveband. For example, the data rate of the uncompressed video of (1920×1080) pixels × 24 bits per pixel × 60Hz is as high as 3 Gbps. Besides this, file transfers are always the faster, the better. Consumer studies show that users are generally dissatisfied with a transfer duration exceeding 7-15s.
The second category includes some audio and video streams transmission. The real-time transfer of computer graphics with 1024 × 768 progressive scan at 60 Hz rate requires a data rate of 38 Mbps [4], while high-definition MPEG2 stream and DVD stream need data rates of 19.2 Mbps and 9.8 Mbps. For highquality audio applications, distribution of CD audio requires 1.5Mbps, while AC3 Dolby and MP3 audio streams require data rates of 384 kbps and 128 kbps, respectively.
In this paper, according to these two category requirements, we present two hardware efficient LDPC decoders specially designed for WPAN standard.
The WPAN standard proposal [3] defines three LDPC codes with the same codeword length 576 and different code rates from 1/2 to 7/8. Three Giga bits per second throughput requirement is quite high for such a short codeword length. Implementation of such a high throughput LDPC decoder with compact hardware while keeping the decoding flexibility is a challenging task. The fully parallel decoder architecture [5] could be a candidate to meet this requirement. However, it will be problematic to add flexibility to support three different code rates specified by the standard. In this paper, a highly parallel level decoder architecture based on [6] is chosen for this implementation. By performing the decoding process in four times parallelism as the basic architecture, only 8 clock cycles are needed to finish each iteration (without considering the pipeline stages). Message memory requirement is another challenge in implementing the message passing circuits of LDPC decoders. In this paper, we manage to store the row process results in a compressed way and do not save the variable node to check node messages at all. On the other hand, the design of a flexible LDPC decoder means another challenging task and needs significant case of study. An efficient flexible architecture for check node processing unit (CNU) is proposed in this work to support three different code rates defined in the standard. It can process 4 messages for 1, 2 or 4 rows per cycle. In addition, two routing networks are introduced to support various sub-matrix shifts and row split patterns.
Besides the high-speed decoder design, a low-cost lowpower design is given for the second category applications. By slowing down the decoding schedule and removing pipeline stages, the consumption of silicon area and power can be significantly reduced.

II. LDPC CODES IN IEEE 802.15.3 STANDARD AND DECODING ALGORITHMS
The LDPC codes defined in IEEE 802.15.3 proposal is a kind of block-based QC-LDPC codes [7], with each sub-matrix size being 18. Three code rates are supported in total: 7/8, 3/4 and 1/2. The 7/8 rate code has the most compact parity check matrix which has 4 × 32 sub-matrices. If each row is split into two rows to double the number of rows, the code rate changes to 3/4. A (N, K) (j, k) regular LDPC code has a Tanner graph with N variable nodes and M check nodes in which the degree of all the variable nodes is j and the degree of all the check nodes is k. The typical LDPC decoding algorithm is the Sum-Product (or belief propagation) algorithm (SPA) [2] [8]. After variable nodes are initialized with the channel information, the decoding messages are iteratively computed by all the variable nodes and check nodes and exchanged through the edges between the neighboring nodes.

III. HIGH SPEED DECODER ARCHITECTURE In this section, we present a high speed architecture for IEEE 802.15.3 LDPC code decoder, specifically designed for ultra high speed (multi-Gb/s) applications.
A. Overall Decoder Architecture First, let us discuss about the decoding schedule. In order to satisfy the throughput requirement of the standard, 4 block columns (72 columns in total) are processed concurrently. The leftmost 72 columns are processed first, then the second leftmost 72 columns, and so on. The column process and the row

process are interleaved. In each clock cycle, 72 Variable Node Processing Units (VNU) receive 72 × 4 check-to-variable messages and compute the 72 × 4 variable-to-check messages, so that 72 Check Node Processing Units (CNU) receive four messages each, then each CNU can deal with one step of the check node process. Therefore, 72 VNUs and 72 CNUs are instantiated in this design. The whole check node process is divided into 8 steps. With this decoding schedule, we can finish a single iteration in 8 clock cycles. This proposed design schedule is much faster than the traditional partially parallel decoder architecture [10] [11].
The overall decoder block diagram is shown in Fig. 3. There are four groups of VNUs with each group having 18 VNUs and four groups of CNUs with each group having 18 CNUs. The Shifter-18 module is a barrel shifter which can transfer the 18 messages with any shift values. Before and after each CNU module, there is a 4 × 4 switch which is responsible for feeding the message to the correct CNU input. This switch is required because of the random row split pattern. For example, for rate 1/2 code, CNU-1 process the rows from 1 to 4, this switch is used to indicate the correct row each message corresponds to.

B. Architecture of Check Node Processing Unit
For the three supported code rates, the number of rows is different. From hardware perspective, the number of CNU is fixed. Therefore, the number of rows each CNU in charge of is different under different code rate condition. For rate 7/8 code, CNU-i performs row-i’s processing. For rate 3/4 code, CNU-i performs row processing of row-(2i-1) and row-(2i). For rate 1/2 code, CNU-i performs row processing of row-(4i3), row-(4i-2), row-(4i-1) and row-(4i). So the CNU should be flexible to accommodate these variations. However, there is one thing in common. The number of messages each CNU processes per cycle is always four for these three different code rates.
The row process results are stored in a compressed way that for each row, only the minimum magnitude, 2nd-minimum magnitude, the location of the minimum (index) and the sign bits are saved. This message storage method greatly reduces the memory requirement [6].

Next we explain the decoding process starting from VNU. In each clock cycle, each VNU processes 4 check-to-variable messages and generates 4 variable-to-check messages. These 4 messages are corresponding to four different CNU groups and they are distributed to 4 shifter-18 modules. After proper shifting, the messages are fed into CNUs for row processing. Each CNU processes 4 messages concurrently and stores the row process results in a compressed way.
Each CNU also generates 4 check-to-variable messages based on the row process results of prior iteration. In the following subsection, we will introduce the architecture of a flexible CNU.

After that, the updated row process intermediate results are registered. There are four register groups storing row processing results with each containing 16 bits (2×5 bits for magnitudes, 5bits for index and 1 bit for sign).
In each cycle, four messages are processed. In the following we describe the computation process for three code rates.
1) rate 7/8 code: The four input messages belong to the same row, so only one register group is used. The two c1 modules and two c3 modules are utilized to find the two minimum values. The data path is c1-c3-c3.
2) rate 3/4 code: The four input messages belong to two rows, and two register groups should be used. The two c1 modules and two c3 modules are utilized. Data go through two c1-c3 paths in parallel.
3) rate 1/2 code: The four input messages belong to four different rows and four register groups are used. The two c3 modules and two c2 modules are processed in parallel to find the four row processing results.
This CNU design may not be the fastest or the smallest architecture. However, it offers a good tradeoff between hardware efficiency, computation delay and control logic complexity.
The flags of the comparison modules are used to generate the index which indicates where the minimum message magnitude comes from. Each index is saved with 5 bits: 3 bits for the cycle number and 2 bits for the exact column location. These index registers are included in the row result register.
In addition, for each CNU, 32 registers are instantiated to save the signs of the messages. They perform like a FIFO: push in and pop out 4 sign bits per cycle. For simplicity, we omit the particular explanation of index and sign parts here.
Another function of CNU is to generate the check-tovariable message. It selects the proper message magnitude according to the index value, and computes the sign of the message. It feeds 4 messages to VNU each cycle.

D. Critical Path and Pipeline
Fig.6 shows the critical path. It starts from the row result register in CNU which stores the row process result of last iteration and ends at CNU row result register of current iteration. CNU generates the check-to-variable message, which then goes through the switch and shifter module to enter corresponding VNU. Then VNU computes the variable-to-check message. Again, through one shifter and one switch, the message gets row processing in CNU and finally the row process result is registered. To speed up the clock frequency, we add two pipeline stages in the path, as shown in Fig.6. After pipelining, each iteration takes 10 clock cycles and we can synthesize the design at 500MHz clock speed.

IV. LOW COST DECODER ARCHITECTURE
For some portable consumer electronics or low speed application devices, another kind of low-power, low-cost LDPC decoder design is required instead of the high-speed decoder described in the previous section. We present a low cost design below.
Firstly, we slow down the decoding schedule. In contrast with the previous high-speed decoding schedule, in each clock cycle, one block column is processed instead of 4. Therefore, only 18 VNUs are instantiated instead of 72. In each clock cycle, 18 VNUs receive 18 × 4 check-to-variable messages

VI. CONCLUSIONS
In this paper, we have presented two LDPC decoder implementations for IEEE 802.15.3 WPAN standard. By employing a highly parallel decoding schedule and a novel flexible CNU architecture, the high speed decoder can reach 3.6 Gbps data throughput and support three different kinds of code rates as well. The low cost design gives another tradeoff with significantly smaller area and lower power consumption. The implementation results show that these proposed designs can be very competitive candidates in real applications.
