Fully Programmable Decoder Architecture for Structured and Unstructured LDPC Codes
Christiane Beuschel and Hans-Jo¨rg Pfleiderer Institute of Microelectronics, University of Ulm Albert-Einstein-Allee 43, 89081 Ulm, Germany Email: {christiane.beuschel, hans-joerg.pfleiderer}@uni-ulm.de

Abstract—In this article we present a fully programmable and scalable partly-parallel LDPC decoder architecture together with an optimum mapping and scheduling algorithm. The proposed algorithm exploits the full parallelism of the architecture at any time for any code, which means that the mapping algorithm achieves 100% utilization of the architecture. The proposed design is fully programmable and can be reconfigured for a different LDPC code by changing the initialization of the control memory. Thus the architecture can be used for a multistandard decoder which supports decoding of any structured or unstructured LDPC code. Furthermore, the parallelism of the architecture is unconstrained and fully scalable which allows to exchange hardware cost and throughput with fine granularity. In contrast to previously proposed programmable designs our approach uses parallel variable and check node processing and thus doubles the data throughput.
I. INTRODUCTION
Low-densitiy parity-check (LDPC) codes are known to perform very close to the Shannon limit [1]. They are proposed for error correction in many current and next generation communication standards, e.g. WiMax [2], Wifi [3], DVBS2 [4]. Iterative decoding of LDPC codes is based on message passing between check and variable nodes. Today’s state of the art LDPC decoders use partly-parallel processing schemes. Between successive updates in the nodes the messages are stored in memory banks. However, access patterns on the stored data are different during check and variable node processing. Thus a key challenge in partly-parallel decoder design is to avoid memory access collisions.
Most research is restricted to classes of structured LDPC codes, e.g. quasi-cyclic (QC) codes. For QC codes with expansion factor z, all access collisions can be solved using a decoder architecture with parallelism p = z [5]. In [6] it was shown that collision free memory access is also possible for QC codes where p is a divisor of z. Difficulties arise for a parallelism p which is not a divisor of z or if QC codes with several different expansion factors need to be decoded with the same decoder chip. Exceptions have to be treated in a special way [7] so that the utilization of these architectures drops below 100%.
It is widely unknown that a partly-parallel decoder architecture with collision-free memory mapping that achieves 100% utilization also exists for any structured or unstructured LDPC code with free choice for p. This allows to implement one LDPC decoder chip with parallelism p which can decode

any LDPC code with optimum hardware utilization. However, the drawback of the memory mapping and the architecture presented in [8] is the alternate check and variable node processing. This lowers the data throughput and is not used any more in current LDPC decoder designs.
In the following we present a decoder architecture together with an optimum mapping and scheduling algorithm. It should be noted that the mapping function we use is the same as in [8] but that the mapping algorithm described in Section III-B is novel. Different from [8], variable nodes are updated during check node processing so that the data throughput is increased by factor 2. Furthermore, the architecture can be extended to layered decoding, which allows to reduce the number of decoding iterations and increases the data throughput.
The structure of this article is as follows: Section II shortly introduces LDPC codes, in Section III we present the optimum mapping algorithm and in Section IV we propose a fully programmable LDPC decoder architecture which matches the mapping algorithm. Section V compares the proposed architecture against other programmable decoder architectures and finally Section VI gives the conclusion.

II. DECODING OF LDPC CODES

A binary (N , K) LDPC code is defined by a sparse M x N parity-check matrix H. Equivalently the LDPC code can be described by a bipartite graph with M check nodes corresponding to the rows and N variable nodes corresponding to the columns of H. Check node m is connected to variable node n if hmn = 1. M (n) is the set of check nodes that are connected to variable node n and N (m) is the set of variable nodes that are connected to check node m. E is defined as the number edges in the bipartite graph. The vector y is the transmitted codeword with yi ∈ {−1, +1} and r is the received corrupted codeword. The degree of variable node n is given by dV,n.

III. MAPPING TO MEMORY BANKS
In the following we describe an assignment of the sum values Sn in (4) to memory banks which guarantees collisionfree memory access for any LDPC code and any parallelism p. The key for parallel check node and variable node processing is to apply the mapping function on the sum values whereas [8] applies the mapping function on the extrinsic values. Furthermore a dynamic instead of a static assignment of values to memory banks is required.

A. Mapping function

Given a set

V = {v0, ..., vE−1}

(5)

of E elements vi where E can be factorized as E = L · p, we arbitrarily define two partitions P and P with

P = {V0, ..., VL−1} and P = {V0 , ..., VL−1} (6)
where each subset Vi or Vj contains p elements of V . A mapping function T is defined with

T : {v0, ..., vE−1} → {0, ..., p − 1}

(7)

such that the following two conditions are fulfilled for every k = 0, ..., L − 1 and every i, j = 0, ..., E − 1 with i = j:

vi, vj ∈ Vk ⇒ T (vi) = T (vj)
vi, vj ∈ Vk ⇒ T (vi) = T (vj).
Thus any two values vi and vj which are in the same subset are always mapped to different values.
Finding an appropriate mapping function can be seen as a graph coloring problem with E vertices vi. An edge connects two vertices if they are contained in the same subset Vk or Vk. Coloring the vertices is equivalent to finding a mapping function. The authors of [8] prove that such a mapping function always exists and furthermore give an algorithm to find it.

B. Mapping algorithm

The decoding algorithm described in Section II is used for a decoder architecture with parallelism p. Each extrinsic value Rm→n is updated exactly once during a decoding iteration and always p values are updated in parallel. Thus p extrinsic

memory banks are required, and the extrinsic values are stored

in linear order in each memory bank. Memory access is

straight forward and no memory access collisions occur. Different from that, each sum value Sn is read and written
dV,n times in each iteration. For QC decoder architectures, collision-free memory access on p sum values in parallel can be guaranteed as for each access always the same p sum

values are needed in parallel. This is different for a fully

programmable decoder architecture: No constraints are set on

the parity-check matrix so that in each clock cycle a “random”
combination of p sum values is accessed. In each of the dV,n accesses on sum value Sn, the value is accessed in a different combination with other sum values.

Therefore we have to assure that each of these “random” combinations of p sum values (defined by the parity-check

matrix of the code) can be accessed without memory access

collisions. This is only possible if the p sum values for each “random” combination are stored in p different memory banks.

To achieve this, memory access collisions for all possible

combinations of sum values for a given code have to be

resolved. In the following we present a solution in three steps,

which is valid for any structured or unstructured LDPC code:

1) Copy each sum value dV,n times. This means that E sum value copies exist, and each copy is accessed only

once for read and once for write in each iteration.

2) Determine a memory bank for each copy using the

mapping function from Section III-A.

3) Remove sum value copies: Interpret each copy of a sum

value as the same sum value at a different point in time.

This means that in the final implementation only N sum

values are stored in the memory banks and that a sum

value resides in different memory banks throughout one

iteration.

In the following these three steps are explained in more detail using an example with parallelism p = 3 and an irregular LDPC code with dimensions N = 8, M = 6 and parity-check

Step 2): In this step, the mapping function from Section III-A is applied. The set V in (5) with cardinality E is defined as the set of all sum value copies. The first partition P in (6) is defined by the access indices: Sum value copies with the same access index i form one subset Vi. The second partition P is defined as shown in Fig. 1b: The access indices in each column are permuted such that each index is replaced by the next smaller one and the smallest index is replaced by the biggest index in the column. The range of the function consists of p = 3 values which are represented by the shapes square (red), hexagon (green) and circle (blue). A possible solution for the mapping function is shown in Fig. 1c. In a decoder architecture with p = 3 exactly p memory banks for the sum values exist. Each shape (color) corresponds to one memory bank, and it can be seen that copies with the same access index are located in different memory banks. Thus no memory access collisions occur.
Step 3): So far all operations were performed on the E sum value copies. This was necessary to explain how the mapping of sum values to memory banks is performed. In the actual implementation, only N sum values and no copies exist. In the following we explain how all copies can be removed. The upper part of Fig. 1d shows again the sum value copies for partition P with access indices and shapes (colors) for memory banks. In the final system, all sum copies in one column of the matrix correspond to the same sum value. During initialization of the sum memory, each sum value Sn is written to the memory bank where the first read is performed from. This corresponds to the memory bank with the smallest index in each column as indicated by the arrows in the upper part of Fig. 1d. The first read accesses the three values S0, S1, and S4 according to V0. It can be seen that different shapes (colors) are assigned to the three values which means that they are located in different memory banks. Then the values are written back to the same memory banks according to V0 . For the next read access sum values S0, S2, and S3 are read according to V1. Also these three values are read from three different memory banks. Writing back is performed in permuted order, e.g. S0 was read from the “square” memory bank but is written back to the “hexagon” bank, S2 was read from the “hexagon” bank and is written to the “circle” and S3 was read from the “circle” bank and is written to the “square” bank according to V1 . Thus after this write operation, the sum values are stored in different banks as before and are prepared for their next read access. The same process continues until all sum values are updated dV,n times. At the end of the iteration, all sum values are again in their original memory banks and the next iteration can start.

IV. FULLY PROGRAMMABLE DECODER ARCHITECTURE
Fig. 2 shows the fully programmable decoder architecture which we propose for decoding an arbitrary LDPC code. Variable nodes are updated during check node processing by accumulating the total sum Sn in (4), a technique also known from QC decoders. The degree of parallelism p of the architecture is fully scalable, allowing an optimum trade-off between data throughput and hardware cost.
A. Decoder architecture
The architecture consists of p intrinsic memory banks IMi for the values In in (1), p extrinsic memory banks EMi for the extrinsic messages Rm→n in (3), p sum memory banks SMAi and p sum memory banks SMBi to store the previous and current total sum Sn in (3), (4). Each memory bank 0 corresponds to the shape “square” in Fig. 1d, bank 1 to “hexagon” and bank 2 to “circle”. All memories are dualport memories allowing simultaneous and independent read and write access on two data entries. The four permutation networks πi can be realized as Benesˇ networks. p check and p variable node functional units (CFUi resp. VFUi) perform the node computations, the core function for check node computation is implemented in the CP blocks. In the VFUs the sum of all incoming values is sent to the output. Each CFU and each VFU updates one data value in each clock cycle, thus 2p data values are updated in each clock cycle. A controller and a control memory are needed to configure the permutation networks and to generate the addresses for the sum memory banks. A different LDPC code can be decoded by changing the initialization of the control memory.
B. Scheduling
Decoding of a received corrupted codeword starts with initialization of all memories in Fig. 2 according to (1) and (2). The intrinsic value In corresponds to variable node n or to column n in the parity-check matrix and is written to the memory bank which has the smallest access index in the left part of Fig. 1c resp. to the memory bank indicated in the line “sum value, initialization” in Fig. 1d. This means I0 is written to the “square” banks SMA0 and IM0, I1 to the “circle” banks SMA2 and IM2. The addresses within each memory bank can be chosen arbitrarily, the simplest way is to increase the address by one for each value which is written to the same memory bank during initialization.
During the decoding iterations, linear addressing is used for the EM memory banks and “random” addressing for the SM banks. p total-sum values which correspond to the copies in Vk are read at time t = k from the SMAi. Then the sum values are written back over π2 to the same memory addresses but possibly different banks as indicated by Vk. E.g. for t = 1 according to V1, V1 the values (S0, S2, S3) are read from (SMA0, SMA1, SMA2) and written back in permuted order to (SMA1, SMA2, SMA0). Each CFU sequentially updates the extrinsic messages of one check node, writes the updated messages to the EMi banks and also sends them to the VFUs.

Permutation network π0 assigns the correct message to each CFU and π1 performs the inverse permutation of π0.
In the VFUs the variable node update is performed according to (4). However, the Ri→n values for one variable node do not arrive sequentially but in “random” order. Thus a temporary total sum is read and always one value is accumulated with Sn,k+1 = Sn,k + Ri→n until all dV,n values were added. Reading and writing the total sum values in the SMBi memory banks is performed in the same order as for the SMAi banks, only a delay in time is added.
After the update of all extrinsic messages in the first decoding iteration, the second iteration is started by swapping the two sum memories SMA and SMB. All operations are now performed in reverse order, meaning that some data paths are inverted as indicated by the dashed lines in Fig. 2 and that the subsets are processed in reverse order starting with VL−1 and ending with V0. After the second iteration all sum values are again in the original memory banks at the original addresses. Processing of the next iterations is performed equivalently.
Defining the two partitions, a mapping of sum values to memories and a scheduling as described above ensures that no memory access collisions occur and that the utilization of the architecture is 100% for any LDPC code.
V. COMPARISON
The fully programmable decoder architectures presented in literature usually use heuristic mapping algorithms which result in a lower data throughput and idle hardware during decoding as in [9], [10]. The design in [8] and the proposed programmable decoder architecture are, to the best of our knowledge, the only published architectures for fully programmable LDPC decoders which use optimum mapping and thus achieve 100% utilization of the architecture. However, our design doubles the data throughput of [8].

Table I compares the different architectures. A fair comparison for the number of permutation networks is possible only for the last two designs as the first two designs need a significant number of additional multiplexers. The number of VFUs and CFUs is the same for all designs. However, in [9], [10] and [8] VFUs or CFUs are used alternately, while in our proposed architecture VFUs and CFUs are used in parallel so that no idle functional units exist. The necessary memory size is counted in multiples of messages to be stored. For an average LDPC code with E = 3.5N , our proposed architecture needs to store 6.5N values which is more than [10] and [8] but less than half the size of [9].
The next line in the table measures the utilization of the architecture for the different designs. Heuristic mapping algorithms are applied in [9], [10] which achieve significantly lower values compared with our mapping algorithm which is optimum on the proposed architecture. For any structured or unstructured code an utilization of 100% is reached on our proposed architecture.
For the heuristic mapping algorithms in [9], [10] the number of data updates per clock strongly depends on the code and in general is below the optimum p. Mapping in [8] is optimum and p messages are updated in each clock cycle. However, for our proposed design the number of updated messages is twice or even five times as high as for the other implementations. In contrast to [8] where a static mapping function is applied on the extrinsic values, our proposed architecture applies a dynamic mapping function on the sum values. Thus we are able to process check and variable nodes in parallel and double the data throughput compared with [8].
Compared with fixed decoders which can decode only one single code, the flexibility allowing to decode any code with one decoder chip comes at the cost of higher hardware complexity. Comparing our programmable architecture with a QC decoder [5], our approach needs the same amount of extrinsic, intrinsic and sum memory. Instead of each barrel shifter two permutation networks are used. Storage of a random parity-check matrix inherently requires more memory than storage of a QC parity-check matrix in the control memory. However, using the same parallelism, the developed scheme

for collision-free memory access on the programmable architecture achieves the same data throughput as QC architectures. Furthermore, code design is no longer restricted to QC codes but any structured or unstructured LDPC code can be decoded with the same optimum utilization of the architecture.
VI. CONCLUSION AND FUTURE WORK
In this article we presented a fully programmable and scalable decoder architecture which can decode any structured or unstructured LDPC code with the identical hardware. Reconfiguration for a different code is achieved by changing the initialization of the control memory. Throughput and hardware cost are exchangeable by the fully scalable parallelism. We have introduced a mapping and scheduling algorithm which guarantees collision-free memory access and maximum parallelism and data throughput for the proposed architecture. Furthermore, we have shown that parallel variable and check node processing, a technique which is known from QC decoders, can also be applied to any unstructured LDPC code. The architecture utilization for any structured or unstructured LDPC code is 100%. In contrast to other programmable architectures where static mapping of messages to memory locations is used, our dynamic mapping increases the data throughput by a factor two to five.
Future research will be dedicated to the extension of the above results to a programmable layered decoder. First results show that the concept of dynamic memory mapping for the sum values as developed in this article enables the implementation of a fully programmable layered decoding architecture. The layered decoding schedule is known to increase the convergence speed and thus further increases the data throughput.
