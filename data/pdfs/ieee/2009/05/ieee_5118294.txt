Abstract—This paper presents an efficient decoder design for the LDPC codes in IEEE 802.15 standard. This decoder features by high parallel level, low message memory requirement and code rate flexibility. By processing 72 columns and 72 rows in parallel, it can reach a throughput of 2.8 Gbps to fulfill the standard requirement. Furthermore, the decoder supports three different code rates by employing flexible check node processor units.
I. 0BINTRODUCTION
Low-density parity-check (LDPC) codes, one kind of linear block codes defined by a very sparse parity-check matrix, were first introduced by Gallager [1] in 1960’s. These codes have been adopted by several industrial standards such as IEEE 802.16e, IEEE 802.11n and DVB-S2 due to their remarkable error correcting capabilities. Some newly proposed standards, e.g. IEEE 802.15 wireless personal area networks (WPAN), are also considering using LDPC codes as the channel coding scheme [2]. In this paper, we present a LDPC decoder design which can meet the 802.15 standard requirements with low hardware complexity.
The standard proposal [2] defines three LDPC codes with the same codeword length 576 and different code rates from 1/2 to 7/8. It requires a throughput of more than 1.5 Gbps. This requirement is quite high for such a short codeword length. The fully parallel decoder architecture [3] could be a candidate to meet this requirement. However, it will be problematic to add flexibility to support three different code rates specified by the standard. The high parallel level decoder architecture presented in [4] is chosen for this implementation. By performing the decoding process in four times parallelism as the basic architecture, only 8 clock cycles are needed to finish each iteration (without considering the pipeline stages). Message memory requirement is another challenge in implementing the message passing circuits of LDPC decoders. In this paper, we manage to store the row process results in a compressed way and do not save the variable node to check node messages at all. The design of a flexible LDPC decoder represents a challenging and significant case of study. A flexible check node processor unit (CNU) architecture is proposed in this work to support three different code rates defined in the standard. It can process 4 messages for 1, 2 or 4 rows per cycle. In addition, two routing networks are introduced to support various sub-matrix shifts and row split patterns.

The remainder of this paper is organized as follows. Section II introduces IEEE 802.15 LDPC code structure and the decoding algorithm. The proposed decoder architecture is presented in Section III. Section IV gives the implementation results and the conclusion is drawn in Section V.

The typical LDPC decoding algorithm is the Sum-Product (or belief propagation) algorithm (SPA) [6]. After variable nodes are initialized with the channel information, the decoding messages are iteratively computed by all the variable nodes and check nodes and exchanged through the edges between the neighboring nodes.
The modified Min-Sum (MS) decoding algorithm [6] is similar to the Sum-Product algorithm, with an approximation of check node process. It has some advantages in implementation over the Sum-Product algorithm, such as less computation complexity and no requirement of knowledge of SNR (signal-to-noise ratio) for AWGN channels. III. D2B ECODER ARCHITECTURE
In this section, we present the decoder architecture specifically designed for IEEE 802.15 LDPC codes.

A. Overall Decoder Architecture 5B
First, let us discuss about the decoding schedule. In order to meet the throughput requirement of the standard, 4 block columns (72 columns in total) are processed concurrently. The leftmost 72 columns are processed first, then the second leftmost 72 columns, and so on. The column process and the row process are interleaved. In each clock cycle, 72 Variable Node Processor Units (VNU) get 72×4 check-to-variable messages and compute the 72×4 variable-to-check messages, so that 72 Check Node Processor Units (CNU) get four messages each, then each CNU can deal with one step of the check node process. Therefore, 72 VNUs and 72 CNUs are instantiated in this design. The whole check node process is divided into 8 steps. With this decoding schedule, we can finish one iteration in 8 clock cycles. It is much faster than the traditional partly parallel decoder architecture [8].

The overall decoder block diagram is shown in Fig. 3. There are four groups of VNUs with each group having 18 VNUs and four groups of CNUs with each group having 18 CNUs. The Shifter-18 module is a barrel shifter which can transfer the 18 messages with any shift values. Before and after each CNU module, there is a 4×4 switch which is responsible for feeding the message to the correct CNU input. This switch is required because of the random row split pattern. For example, for rate 1/2 code, CNU-1 process the rows from 1 to 4, this switch is used to indicate the correct row each message corresponds to.

Each CNU also generate 4 check-to-variable messages based on the row process results of prior iteration. In the following, we will introduce the architecture of a flexible CNU.

In addition, for each CNU, 32 registers are instantiated to save the signs of the messages. They perform like a FIFO: push in and pop out 4 sign bits per cycle. For simplicity, we omit the particular explanation of index and sign parts here.

B. Architecture of Check Node Process Unit 6B
For the supported three code rates, the number of rows is different. Therefore, the number of rows each CNU in charge of is different. For rate 7/8 code, CNU-i performs row-i’s processing. For rate 3/4 code, CNU-i performs row processing of row-(2i-1) and row-(2i). For rate 1/2 code, CNU-i performs row processing of row-(4i-3), row-(4i-2), row-(4i-1) and row(4i). Therefore, the CNU should be flexible to accommodate these variations. However, there is one thing in common. The number of messages each CNU processes per cycle is always four for these three different code rates.

The row process results are stored in a compressed way that for each row, only the minimum magnitude, 2nd-minimum magnitude, the location of the minimum (index) and the sign bits are saved. This message storage method greatly reduces the memory requirement [4].

Fig. 4 shows the CNU magnitude computation architecture, which is the main part of CNU. The work it does is to compare the magnitude of the input messages with the current row process intermediate results, and to update the minimum and 2nd-minimum values. After that, the updated row process intermediate results are registered. There are four row result registers with each containing 16 bits (5×2 bits for magnitudes, 5bits for index and 1 bit for sign).

In each cycle, four messages are processed. In the following we describe the computation process for three code rates.

1) rate 7/8 code: The four messages belong to the same row, and only one row result register is used. The two c1 modules and two c3 modules are utilized to find the two minimum values. The data path is c1-c3-c3.

Figure 4. Architecture of CNU design

2) rate 3/4 code: The four input messages belong to two rows, and two register groups should be used. The two c1 modules and two c3 modules are utilized. Data go through two c1-c3 paths in parallel.

Another work of CNU is to generate the check-to-variable message. It selects the proper message magnitude according to the index value, and computes the sign of the message. It outputs 4 messages to VNU each cycle.

3) rate 1/2 code: The four input messages belong to four different rows and four register groups are used. The two c3 modules and two c2 modules are processed in parallel to find the four row processing results.
This CNU design may not be the fastest or the smallest architecture. However, it offers a good tradeoff between hardware efficiency, computation delay and control logic complexity.

C. Architecture of Variable Node Processor Unit 7B
The architecture of variable node processor unit is the same with [3] [4]. It can be designed with only combinational logic. Fig. 5 shows the VNU architecture.

The flags of the comparison modules are used to generate the index which indicates where the minimum message magnitude comes from. Each index is saved with 5 bits: 3 bits for the cycle number and 2 bits for the exact column location. These index registers are included in the row result register.

D. Critical Path and Pipeline 8B
Fig.6 shows the critical path. It starts from the row result register in CNU which stores the row process result of last iteration and ends at CNU row result register of current iteration. CNU generates the check-to-variable message, through the switch and shifter module to enter proper VNU. Then VNU computes the variable-to-check message. Again, through one shifter and one switch, the message gets row processing in CNU and finally the row process result is registered. To speed up the clock frequency, we add two pipeline stages in the path, as shown in Fig.6. After pipelining, each iteration takes 10 clock cycles and we can synthesize the design at 500MHz clock speed.

V. 4BCONCLUSIONS

In this paper, we have presented a LDPC decoder implementation for IEEE 802.15 WPAN standard. By introducing a high parallel decoding schedule and a novel flexible CNU design, the decoder can reach 2.8Gbps throughput and support three different code rates as well. The implementation results show this design can be a competitive candidate in real applications.
