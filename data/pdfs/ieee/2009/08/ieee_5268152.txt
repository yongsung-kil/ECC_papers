2009 ISECS International Colloquium on Computing, Communication, Control, and Management

An FPGA Implementation of LDPC Simulation Platform

Jin Sha, Chuan Zhang, Li Li, Gao Minglun
Institute of VLSI Design Nanjing University Nanjing, China shajin@nju.edu.cn

Abstract—This paper presents an FPGA implementation for LDPC codes performance simulation. The goal is for fast evaluation of LDPC code to investigate the error floor. The hardware evaluation platform features by fast simulation speed and high precision. The construction of the platform is described. The critical modules designed in the platform such as LDPC encoder, decoder, and AWGN noise generator are presented. As the result, a throughput of 120 Mbps is achieved and the BER curve can reach beyond 10-11 within 10 hours.
Keywords—Error control coding, LDPC, Decoder, FPGA
I. INTRODUCTION
Low-density parity-check (LDPC) codes, one kind of linear block codes defined by a very sparse parity-check matrix, were first introduced by Gallager [1] [2] in 1960’s. With the remarkable characteristics, LDPC codes have been recently adopted in several standards, such as Wireless Local Area Networks (WLAN) (IEEE 802.11n), Wireless Metropolitan Area Networks (WMAN) (IEEE 802.16e), China’s Digital Television Terrestrial Broadcasting standards (DTTB), China Mobile Multimedia Broadcasting (CMMB) and Digital Video Broadcasting-Satellite-Second Generation (DVB-S2) [3-7].
To get the error correcting performance of LDPC codes, normally software simulation platform is applied. Language C or Matlab can be used for the simulation. With a high-end PC, we can reach the bit error rate (BER) at 10-7 within about 10 hours. However, to get to lower error rate level, the time consuming will be intolerable. In concern with this situation, some hardware acceleration system can be developed to make the simulation much faster. In this paper, we present an FPGA implementation of LDPC simulation platform with low hardware complexity. This platform can get to a throughput of 120 Mbps and the BER curve can reach beyond 10-11 within 10 hours. The precision is basically the same as software simulations. With this hardware simulation platform, the error floor phenomenon of LDPC codes can be investigated.
The remainder of this paper is organized as follows. In Section II, a brief review of LDPC codes and the parity check matrix of array LDPC code are given. In Section III, the FPGA platform is presented. Section IV presents the detailed design of each module in the platform. Section V gives the implementation results and draws the conclusion.

II. REVIEW OF LDPC CODES
LDPC codes are a set of linear block codes corresponding to the (N-K)×N parity check matrix H, which has very low density of 1’s. With the aid of the bipartite graph called Tanner graph (TG), LDPC codes can be effectively represented. There are two kinds of nodes in TG, variable nodes (v-nodes) and check nodes (c-nodes). And check node fi is connected to variable node cj only when the element hij of H is a 1. Fig.1 shows the parity check matrix H and its corresponding TG of an (8, 4)-regular LDPC code.

Therefore, an (N, K) (j, k) LDPC code has a Tanner graph with N variable nodes and M check nodes, in which every variable node and every check node have degrees of j and degrees of k, respectively. The typical LDPC decoding algorithm is the Sum-Product (or belief propagation) algorithm [2]. After variable nodes are initialized with the channel information, the decoding messages are iteratively computed by all the variable nodes and check nodes and exchanged through the edges between the neighbor nodes.
The modified Min-Sum decoding algorithm [8] is similar to the Sum-Product algorithm, with an approximation of the check node process. It has some advantages in implementation over the Sum-Product algorithm, such as less computation complexity and no requirement of knowledge of signal-to-noise ratio (SNR) for AWGN channels. III. WHOLE PLATFORM DESIGN
Fig.2 gives the diagram of the whole FPGA simulation platform. It shows the main data path of the simulation. First, the Random Generator module generates the random information to be sent. Then the information block is encoded by the LDPC Encoder to generate the codeword. The AWGN Noise Generator assumes the transmission channel to be additive Gaussian white noise channel. After adding the generated AWGN noise, the “transmitted” block is decoded by the LDPC Decoder, which is the most important module in the platform.
The FPGA board used is Altera Cyclone II PCI development board. The board is controlled by a personal computer via its PCI interface. We can set the variance of the noise to change SNR. And the error rate information can be read any time through the PCI interface. The simulation is finished when 100 error blocks are collected.
In the platform, the PCI interface and LDPC decoder work at 33 MHz and 40 MHz, respectively. All other parts work at 120 MHz.

IV. MODULES DESIGN
A. PCI Interface The PCI interface is based on an IP provided by Altera:
PCI_compiler-v4.1.1 and configured as a 32 bits slave mode one.
B. Random Generator The Random Generator is designed with a linear
feedback shift register (LFSR). In order to ensure the test patterns do not repeat in 264-1 samples, we use the key polynomial of f(x)=x64+x63+x61+x60+1.

Each P×P sub-matrix ti,j (i=1~43, j=44~47) has a cyclic property: its (i+1)-th row is the right shift of each element in i-th row.
Applying the linear block code property GHT = 0, we get the ti,j matrix, or the generate matrix G. Then the encoding can be processed as v = u ⋅G . Only the first line of ti,j needs to be stored, which is 43×188 bits in total.
The encoder hardware is designed in a serial mode. In order to finish one frame information data (2021 bits) with 188 check bits, it needs 2021 clock cycles. Since the clock speed is set to 120 M, the throughput of the encoder is 120 Mbps. Fig.3 illustrates the encoder hardware units. E. Software
The software is in charge of controlling the FPGA working and monitoring the data output. With this simple control software, we can set the AWGN channel variance (corresponding to SNR) and read the running condition such as the decoded frame number, error frame number and bit error rate, et al at any moment.

F. LDPC Decoder

The LDPC decoder is the most sophisticated module to design. Following the shift decoder architecture we proposed in [14], a dedicated array LDPC code decoder is designed with 120 Mbps throughput

There is no communication between the four CPU groups. Except for the CPUs in Group 1 (CPU-0~CPU-46), every 47 CPUs in a group form a chain, and each CPU communicates with its two neighbors. For example, in CPU group 3, CPU-i only gets data from CPU(i-2) and delivers date to CPU-(i+2). Our design relies on the structure of the H matrix. These communications between CPUs can alleviate the complex edge pattern between check nodes and variable nodes, which is the main bottleneck in designing high parallel level LDPC code decoder.
