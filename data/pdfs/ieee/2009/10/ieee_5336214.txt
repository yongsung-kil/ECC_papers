Abstract
Low Density Parity Check (LDPC) code is an error correction code that can achieve performance close to Shannon limit and inherently suitable for parallel implementation. It has been widely adopted in various communication standards such as DVB-S2, WiMAX, and Wi-Fi. However, the irregular message exchange pattern is a major challenge in LDPC decoder implementation In addition, faced with an era that diverse applications are integrated in a single system, a flexible, scalable, efficient and cost-effective implementation of LDPC decoder is highly preferable. In this paper, we proposed a multi-processor platform based on network-on-chip (NoC) interconnect as a solution to these problems. By using a distributed and cooperative way for LDPC decoding, the memory bottleneck commonly seen in LDPC decoder design is eliminated. Simulation results from long LDPC codes with various code rates show good scalability and speedups are obtained by our approach.
1. Introduction
Low-Density Parity-Check (LDPC) codes are error correction codes (ECC) featuring very sparse parity check matrices. It was proposed by Gallager in 1962 [1] and has been proven to have a performance close to the Shannon limit. The LDPC code is essentially suitable for parallel implementation, and it is recently adopted by data communication systems such as DVBS2, WiMAX (IEEE 802.16e), Wi-Fi (IEEE 802.11n), 10Gigabit Ethernet (IEEE 802.3an) and ITU-T G.hn. It is being considered for emerging standards for communication and storage applications as well.
With the advance of semiconductor technology, it is possible to build on-chip multi-core systems to provide considerable computation capabilities for applications requiring intensive computation and large amount of storage. Thus, multi-processor system-onchip (MPSoC) and chip multi-processor (CMP) are

proposed to meet such requirements. To interconnect the various on-chip components and satisfy the communication needs among these components, network-on-chip (NoC) is regarded as a promising interconnect solution for next-generation chips because it is capable of tackling various design challenges which will be faced in the near future[16].
As to the implementation of LDPC decoders, researchers have reported many design strategies ranging from serial to fully parallel, and most of them are application specific solutions [2][3][4][5] due to the fact that LDPC decoding is very computation intensive. These solutions utilizing dedicated VLSI circuits may be more efficient in terms of performance and area cost. However, they lack the flexibility and extensibility to support multiple standards or code rates when it comes to integrating various applications in a single chip/system. With the considerable computation power from MPSoCs or CMPs, we envision a scalable, programmable, software-based approach for LDPC decoding is possible. Therefore, in this work, we propose to use a distributed CMP based on NoC interconnect for LDPC decoding. The inter-processor communication is done by the message passing mechanism. In such a system, the code length and code rate can be very flexible because of the use of fully programmable processors. In addition, a data structure partition approach is used to decode long code words cooperatively, breaking the limitation of local memory size. Various LDPC codes with different code rates and lengths are implemented and evaluated on 16/32/64processor systems, and the results show the scalability of our approach.
The rest of this paper is organized as follows: Section 2 presents the background knowledge of LDPC decoding algorithms and related works. The multiprocessor platform used for evaluation is described in Section 3. Section 4 discusses the parallel LDPC decoding algorithm. Section 5 presents the experimental results and analysis. Finally, we draw a conclusion in the last section.

2.2 Related works

Because of the intensive message exchanges among BNs and CNs during the course of decoding, it is critical to resolve the access conflicts from the irregular permutation patterns. [11][12][13] proposed to solve this problem by using network-on-chip as the interconnection fabrics for transferring messages among processing nodes and memory. But their design still lacks flexibility due to the application-specific design style. Therefore, there were attempts to use programmable techniques for designing flexible LDPC decoders [8][9][10]. Recently, with the emergence of multi-core chips, there have been software based solutions proposed. [6] implemented LDPC decoding algorithm on Cell/BE processor. The Synergistic Processor Element (SPE) is used to decode codewords independently, so the codeword length that can be handled is limited by the local memory size. Applying Graphic Processing Unit (GPU) for LDPC decoding was investigated in [7], but the performance in long codes may still be restricted by the shared memory architecture.
3. NoC-based parallel computing platform

In this section, we present the Network-based Processor Array (NePA) platform, which is a multiprocessor platform using NoC as the backbone interconnect, as depicted in Fig. 2. NePA is comprised of an array of processing elements (PE). Each PE consists of a processor core, which is an OpenRISC 1000 core in the current configuration, local instruction and data memory, network interface (NI), and a router. For simplicity and cost minimization, I/D-cache, I/DMMU (memory management unit), and debug unit are omitted from the original OpenRISC core. PEs are interconnected by a two-dimensional mesh network

which provides inter-PE communication capabilities. The router uses a wormhole flow control and adaptive routing algorithm as a method of reducing latency and improving network utilization.
The delivery of data among PEs is done by packetized data communication through the on-chip network. For instance, when PE A needs to send data to PE B, it initiates a data transfer by configuring the control registers in its NI. The configuration information includes the transfer type, data identification number, destination address, and the address of the data to be transferred in the memory. NI then performs packetization to generate packets consisting of header flit and body flit, and injects them into the network. Packets are routed and transmitted to the destination PE and arrive at the NI of PE B. An interrupt is generated by NI to notify the processor core that a packet has arrived. The processor has to execute an interrupt service routine to instruct the NI to do depacketization and store the received data into the corresponding memory location. Different from bus architectures, the on-chip network allows simultaneous communication among PEs, and this is one of the factors that make our proposed decoding algorithm scalable.

4. Parallel LDPC decoding algorithm
Since programmable processor cores are adopted for implementing LDPC decoding algorithm, we first have to decide a data structure for storing data of the algorithm. Received bits and priori information can be simply stored in a one-dimensional array while the rmn and qnm messages should be carefully treated. Using a two-dimensional array would make data access easy, but it wastes much memory space because the H matrix is sparse. Hence we propose a linked structure for storing rmn and qnm messages. This data structure is illustrated in Fig. 3, and the C code of each entry is also

In single PE implementation, the decoding process is divided into 4 steps: (1) initialization; (2) update rmn; (3) update qnm; (4) update code bit. The decoding starts with the initialization step in which the priori information, rmn and qnm messages are initialized. Then Steps (2), (3) and (4) are executed iteratively until the iteration count is reached. We implemented the MinSum algorithm[14], which is a good approximation to the SPA algorithm, because of the lack of floating point arithmetic support on the OpenRISC processor. The maximal codeword that can be decoded on a single PE is constrained by the local data memory size. In the current processor configuration, each PE is equipped with 256 KB of data memory, which allows a code length up to 2048 bits (regular LDPC code, code rate = 1/2, BN degree = 3).

4.2 Separate BN/CN implantation
The first parallel decoding approach we developed is the separate BN/CN implementation: PEs are grouped into two categories according to their function: BN_PE and CN_PE, which are dedicated to the BN nodes and the CN nodes separately. To enable this, we modify the data structure as depicted in Fig. 3(b) and Fig. 3(c). The original linked data structure is divided into two parts: one is for BNs and the other for CNs. The mapping of BNs and CNs onto PEs is done by even distribution where each PE is responsible for the same number of BNs or CNs. Assume there are total of p PEs in the NePA platform, and the number of BN_PEs and CN_PEs is b and c, respetively. To decode an (N, K) code, we map N/b BNs to each BN_PE, and (N-K)/c CNs to each CN_PE. The data structure in Fig. 3(b) and Fig. 3(c) is further partitioned in a row-by-row or column-by-column fashion and mapped in accordance with the BN/CN mapping. So each PE needs to only keep the data of the BNs or CNs that are mapped to it.
Different from the single PE implementation, there are inter-PE message exchanges during decoding process, thus we define two additional operation steps: (5) send/receive rmn and (6) send/receive qnm. Each edge in the Tanner graph results in two packet transmissions: one for sending qnm message from BNs to CNs and the other for sending rmn message from CNs to BNs. The BN number and CN number associated with each edge are designated in the header flit of the packet for identification. The header flit also has a field for denoting the message type: whether it is a qnm message or a rmn message. The actual data of qnm and rmn are attached in the body flit as payload. The execution flow of this implementation is shown in Fig. 4(b). During Step (5) and Step (6), the receiving PE (BN_PEs in (5) and CN_PEs in (6)) keeps a counter to calculate how many messages were received, and it begins the next step after it gets all required messages. So there is no need for a centralized synchronization control.

The approach developed in Sec. 4.2 is simple and easy to implement, but it has drawbacks from the viewpoint of both performance and memory usage. First, the separate BN/CN approach leads to unbalanced processor utilization since some PEs may be idle while others are busy. Moreover, BN_PEs and CN_PEs require different amount of memory space for data storage, which will hinder the implementation of long LDPC codes. Therefore, we proposed to allocate PEs for both BN and CN functions instead of dedicating them to one task. We utilized the same mapping strategy used in the previous section except that both BNs and CNs are mapped onto the same PE. With p PEs to decode a (N, K) code, each PE is assigned N/p BNs and (N-K)/p CNs in this approach.
The execution flow of this combined approach is illustrated in Fig. 4(c). Though this approach requires more complicated control, it can achieve higher performance because overall PE utilization is increased. Another advantage of this approach is that there exists potential in reducing inter-PE traffic because there is no need to generate packet transmission if the target BN_PE or CN_PE is located at the same PE.

First, we compare the time needed for each iteration to decode Code A, Code B and Code C with a codeword size of 4096 bits in a single PE, separate BN/CN and combined BN/CN approaches. The total number of PEs used for parallel LDPC decoding is 16 in this comparison, and Table 2 shows the simulated speedups. Note that the time needed for the single PE approach is by estimation because the tests of codes of such length are prohibited by the local memory size. However, it is observed that the decoding time actually grows linearly with codeword size when code rate is fixed. For example, the cycle counts per iteration for (512, 448), (1024, 896), (2048, 1792) codes are 961K, 1927K and 3850K, respectively, so we can use estimated numbers for comparison. The rest of statistics listed in Table 2 are measured from simulation. Speedup of (4096, 2048) code is smaller than that of (4096, 3584) code because the computation time required by the latter code is longer, but communication time needed by both codes is similar due to the equal number of edges in the Tanner graph. So codes with higher code rates tend to gain more performance improvement from our parallel algorithms. Table 2 also shows that combined BN/CN approach has higher speedups because it can balance and enhance PE utilization and reduce the overall time required for execution. 6. Conclusion

In this paper, we demonstrated a scalable, distributed, and cooperative approach for parallel LDPC decoding. Experimental results have shown that the speedups grow in proportion to the number of PEs in the system for various LDPC codes with different code rates. Also, the proposed approach enables decoding very long LDPC codes by distributing the computation and memory storage onto multiple PEs. The overall performance improvement is constrained by communication overhead, and this is the topic we intend to address in the future.
