Abstract
Low-Density Parity-Check (LDPC) codes are among the best error correcting codes known and have been adopted by data transmission standards, such as DVB-S2 or WiMax. They are based on binary sparse parity check matrices and usually represented by Tanner graphs. LDPC decoders require very intensive message-passing algorithms, also known as belief propagation. This paper proposes a very compact stream-based data structure to represent such a bipartite Tanner graph, which supports both regular and irregular codes. This compact data structure not only reduces the memory required to represent the graph but also puts it in an appropriate format to gather data into streams. This representation also allows to map the irregular processing behavior of the Sum Product Algorithm (SPA) used in LDPC decoding into the stream-based computing model. Stream programs were developed for LDPC decoding and the results show significant speedups obtained either using general purpose processors, or graphics processing units. The simultaneous decoding of several codewords was performed using the SIMD capabilities of modern streambased architectures available on recent processing units.
1. Introduction
Recent processing units have undergone increasing performance by exploiting the potential of stream-based architectures, not only in general purpose computers but also in more specific computing engines [6]. The stream computing model gathers data into a stream, processes it and then scatters the output data stream to memory again. Programming according to this type of model minimizes the memory accesses while exploits parallelism and locality of data. It allows the speeding up of applications, namely, by increasing the parallelism using Single Instruction Multiple Data (SIMD) type of processing.

Low-Density Parity-Check (LDPC) codes are powerful error correcting codes (ECC) proposed by Robert Gallager in 1962 [4]. Their rediscovery by Mackay and Neal in 1996 [9] led to a growing interest on the research and development of efficient LDPC coding solutions for digital communication systems. They have been adopted by the DVBS2 standard [2], Wimax Mobile (802.16e), Wifi (802.11n), 10Gbit Ethernet (802.3an) and other new emerging standards oriented to space, optical and storage applications. LDPCs are linear (n, k) block codes defined by sparse binary parity check H matrices of dimension (n − k) × n. Usually, they are represented by bipartite graphs formed by Bit Nodes (BNs) and Check Nodes (CNs), linked by bidirectional edges also called Tanner graphs [13]. The Sum Product Algorithm (SPA) is a very efficient algorithm used for LDPC decoding [8]. It is based in the belief propagation between connected nodes as indicated by the Tanner graph edges (see figure 1).
LDPC decoders demand very intensive computation and, therefore, a lot of research has been performed in the last few years to implement dedicated VLSI decoders [15, 3] based on fixed-point arithmetic. These decoders, in particular the ones based on the SPA, belong to the class of applications for which stream-based computing architectures can be adopted and applied, but new data representations have to be developed. Additionally, the implementation in these architectures allows floating-point arithmetic operations, so better accuracy can be expected as well as lower Bit Error Rates (BER).
This paper proposes a SPA algorithm oriented to LDPC stream-based decoding and new data structures to efficiently represent the H matrix. The compact data representation and the development of a new stream-based approach to calculate the SPA under a specific type of recursive computing model are the main contributions of this paper. To the best of the authors’ knowledge this is also the first time LDPC decoders are investigated under a stream-based computing perspective.

3 Stream-based Computing
The stream programming style decouples computation and data locality, which encourages compilers and programmers to explore more efficiently the parallelism intrinsic to a program. It is based on a gather, compute and scatter model [5].
A stream program gathers a large amount of data from memory into input streams and applies them to one or more kernels. A kernel consists in a group of operations performed to one or more input streams, with a desired functionality implicit. A kernel can compute hundreds of operations and data parallelism may be exploited in a kernel, for example, by using SIMD processing. After computation, data feeds the input of other kernels or it is scattered to memory again. By reducing and concentrating memory accesses, this programming model converts a latency problem into a bandwidth restriction.
A Synchronous Data Flow (SDF) graph [7] such as the one depicted in figure 2, can be used to explicitly express data dependencies and the amount of data consumed and produced at each input and output data flow node. Operations are performed by kernels that can have multiple input streams and may perform complex calculations per input data item. The input and output streams are the only external data a kernel can access.
Some applications in the signal, image and graphics processing fields can gather and scatter data using sequential load/store operations, while others exhibit random, or at least irregular accesses to memory, which makes streaming them more challenging. For these types of memory accesses the addresses first need to be collected into indexed

arrays before gathering/scattering the data. Moreover, some of them are iterative and require dynamic adjustment to be programmed in a stream-based style. It happens to be the case of the LDPC decoder based in the SPA algorithm described in the next sections.

4 Stream-based LDPC decoding

The H matrix defines the Tanner graph and, thus, the flow of messages being exchanged between CNs and BNs. In medium to large sized LDPC codes a huge amount of memory is required to store this matrix. Moreover, in a stream computing model this matrix has to be itself gathered into a data stream. Therefore, we propose to separately code the information about H in two independent data streams adequate to the SPA. These particular auxiliary data streams establish the order of the data in the input stream used in the flow of messages to update the BNs and the CNs. The example of figure 1 can be used to describe the transformation performed in H in order to produce the compact data stream structures HBN and HCN. HBN codes information about edge connections used in each parity check equation (horizontal processing). This data structure is generated by scanning the H matrix in a row major order and by mapping sequentially only the BN edges associated to a single CN equation (non-null elements in a single row of H). Each element of the data structure records the address of the next entry, as a pointer, and the corresponding rmn value. The element corresponding to the last non-null element of each row in H points to the first element of this row, implementing a circular list that is used to update all the BNs connected to a single CN. Thus, a CN always knows from

which BNs it should be reading information and where to update, as well. Circular addressing allows to explore a high level of parallelism. In the limit, for a multi-processor platform, a different processor can be allocated to each single edge, since the starting point becomes no longer important, due to the circular addressing feature. Although in figure 3 the elements of the structures are represented by their row and column addresses, the structures can be easily vectorized by considering the rows of the structure in sequence, using convenient 1D or 2D reshaping according to the target stream-based architecture they apply to.
The same principle can be applied to code the information about CNs. HCN can be defined as a sequential representation of every BN connecting to all its neighboring CNs (non-null elements in a single column of H). Once again, the access between adjacent elements is circular, as illustrated in figure 4 for the H matrix given in figure 1. In this case, a careful construction of the addresses in HCN is required because every (n,m) edge must be represented exactly in the same position as it is in HBN. This meticulous positioning of the elements in HCN allows the iterative processing of consecutive inputs to be alternately performed for rm(i)n and qn(im) , guarantying that data being processed for every (n,m) edge is stored in the correct position for both situations. In fact, the in place computation here performed for horizontal and vertical processing is supported by an efficient mechanism that swaps data pointers [14] at the end of each iteration, according to the input data streams necessary for the next processing.

4.2 SPA Kernels Optimization and Parallelism
The flooding schedule [8] algorithm used controls the order in which BNs and CNs are updated. It guarantees that no BN is updated before all other BNs conclude their updating procedure. This principle is adopted in this work because it allows true parallel execution of the SPA decoder.
The circular addressing property permits that parallel processors can simultaneously access the necessary information stored in both HBN and HCN. Consequently, the overall processing time required to decode a codeword can be significantly reduced by applying parallel processing. Figure 5 represents a detail of HBN illustrating such efficient properties for the update of rm(i)n values.
An alternative representation of the SDF in figure 2 is presented in figure 6. This compact SDF is obtained by folding the pair of kernels (Kernel 1 an Kernel 2) and by introducing an additional multiplexer. Kernel 1 initially processes the input stream ps0 according to the BN indexes defined in the input stream HBN. It produces the output stream rs0 by performing the message update from CNm to BNn. The stream rs0 directly feeds the input of Kernel 2, which in turn produces the qs0 that represents messages updated from BNn to CNm. The other input of the Kernel 2 is HCN, which defines the CN indexes from H. Only data streams communicate between kernels. Figure 6 shows that by using the multiplexer at the input of the SDF, the output stream qsi becomes the input stream of Kernel 1 for all remaining iterations after the first one. At the end, the qsi output stream conveys the codeword.
This compact representation can be exploited to implement efficient LDPC decoders, not only because of the new compact stream oriented representations of the H matrix but also regarding the program memory and the data communication. The required program memory is reduced by the folding mechanism and the data streams can be multiplexed through a simple and efficient mechanism for swapping pointers.

5.2 Stream-based SPA on the GPU
The pixel processors of the GPU were used with data inputted as textures and the output data stream assuming the usual place of the pixel color components’ output in a graphical application. The basic idea underlying the initial approach adopted to compute the SPA on the GPU places the input H matrix into a texture and processes Kernel 1 and Kernel 2 (see figure 6) in the pixel processors. The proposed compact representation also allows to reduce data transfer between the host memory (RAM) and the video memory (VRAM), which is a very important aspect to achieve high performance with GPUs.
Figure 9 depicts input and output data streams for the program developed for GPU and figure 10 shows a source code segment of an efficient program that makes use of SIMD multiply (∗) and subtract (−) instructions that operate in four floating-point data elements in the same texture (Red, Green, Blue, Alpha). The shader program was developed in OpenGL (GLSL) an runs on the pixel processors. We use the Caravela programming interface recently developed for stream-based computing on GPUs [14]. The Caravela library is used in the CPU side to manage the execution of the program in the GPU and to write and read data to and from the VRAM, respectively.
In this subsection, a different implementation explores GPUs with dedicated architectures for stream-based computing [10]. The GPU used in this work is a 8800 GeForce from NVIDIA with programmable pixel processors and 768MB of RAM [1].

Results show that the speedup increases as the number of edges being processed also increase. Moreover, beyond a certain number of edges, a major percentage of time is wasted in the memory data transfer between the host RAM and VRAM. For matrices with 2997 and 4080 edges, the GPU processing power is not able to compensate the delay introduced by data transfer between the host and VRAM. As the size of matrices increases, the processing power gains more importance in the overall execution time. This is the main reason that explains the obtention of a better speedup for the largest matrix (16000 edges), where the GPU executes 100 iterations in less than 109ms against almost 300ms in the CPU with SSE instructions.
Comparing the two SPA streaming implementations, CPU with SSE versus GPU, it is possible to conclude that the GPU performs better for huge quantities of data, due to its parallelism features and tremendous processing power.
6. Conclusions
This paper proposes a LDPC decoder using streambased architectures to perform the very intensive computation that such codes demand. Pursuing this goal, dynamic and vectorized compact data structures were created to represent all exchanged messages between the Tanner graph nodes. These data structures allow a significant reduction of the memory space and the processing time necessary for LDPC decoding. The Sum Product Algorithm was adapted to stream computing suiting different hardware platforms,

which improved its efficiency. The proposed data structures and algorithms pave the way for future work in the area of stream-based applications dedicated to the intensive decoding of error correcting codes, by exploiting low cost and highly flexible alternatives against dominating hardware dedicated VLSI LDPC decoders.
