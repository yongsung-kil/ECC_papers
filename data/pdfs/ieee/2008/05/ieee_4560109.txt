I. INTRODUCTION
Low-density parity-check (LDPC) [1] code, which is a linear block code described with a binary sparse M × N parity-check matrix H, can achieve a performance that is within 0.0045dB of Shannon limit [2]. The high performance of LDPC codes led to their recent inclusion in several standards such as IEEE 802.11n [3], IEEE 802.16e [4], DVBS2 [5] and CDMA [6].
Since the applications need adaptation capabilities in system parameters such as block size and code rate imposing flexibility requirements on hardware implementations, the design of flexible LDPC decoders satisfying the need for decoders that are able to adapt to different transmission conditions in practical communication systems is necessary [7], [8]. Casado et al. [9] partly solved this problem by introducing suitable constraints in the code design and

The connection between the check nodes and the variable nodes is specified by the parity-check matrix, H which varies according to code rates and code lengths. Even if the nodes require computational capabilities, most of the implementation challenges come from the interconnection structure to support message exchange among the variable nodes and the check nodes.

LOW-DENSITY PARITY-CHECK CODES A (n, γ, ρ) LDPC code means that its block length is n and the column and row weights of its parity-check matrix are γ and ρ, which represent the number of 1’s in a column and a row, respectively. The column and row weights are much smaller than n to achieve a sparse matrix. Rows and columns of a parity-check matrix represent the connection between check nodes and variable nodes. For a (n, γ, ρ) LDPC code, the number of parity-check equations is as many as the
number of rows of the parity-check matrix: i.e., J = (n × γ / ρ)
parity-check bits are attached to the message. For a regular LDPC code where the total number of 1’s in the parity-check matrix is the same whether it is counted in the row direction
or in the column direction, the equation, n × γ = J × ρ, holds
for the code parameters and the code rate can be calculated as R = 1 − γ / ρ.

It is common to describe an LDPC code by a bipartite graph, called Tanner graph [11] containing two sets of nodes: variable nodes and check nodes. The variable nodes represent the bits of a codeword and the check nodes implement the parity-check equations. An edge between the ith variable node and the jth check node in the graph represents that ith bit is involved in the jth parity-check equation.
Fig. 1 shows an example of a (6, 2, 3) LDPC code. The Tanner graph in Fig. 1 (b) consists of variable nodes, {vj}, and check nodes, {ci}. An edge between a variable node vj and a check node ci is drawn if there is 1 at (i, j) in the paritycheck matrix. The variable nodes that are connected to a check node are called the neighbor variable nodes of the check node. For a variable node, the neighbor check nodes can be defined similarly.
The message-passing algorithm can be expressed easily based on a Tanner graph. For all the bits in a received vector, a posteriori probabilities are calculated in the variable nodes and delivered to the neighbor check nodes through the edges. Parity-check operations are executed with these probabilities in all the check nodes and the results are passed again to the neighbor variable nodes. To avoid multiplications and divisions in the variable and check nodes, these operations are transformed into summations by using log-likelihood ratio [11] and the probability ratio is expressed in terms of tanh and tanh-1 functions by applying

III. PROPOSED LDPC DECODING ARCHITECTURE
In this section we describe the proposed flexible LDPC decoder architecture whose interconnection is implemented by using Benes network. In order to reduce the interconnection complexity, the proposed decoder employs partially parallel architecture with broadcasting. The VPUs and CPUs are pipelined to reduce the delay caused by using Benes network since on Benes network the data transmissions from VPUs to CPUs and from CPUs to VPUs must be performed sequentially.
A. Interconnection Network
The interconnection network between VPUs and CPUs should be configurable to implement a flexible LDPC decoder supporting multiple code rates and code lengths. Crossbar switch, Benes network and Banyan switch can be used to implement configurable interconnection network. Examples of the networks are shown in Fig. 2. The numbers of switching elements (SEs) in these networks as a function of the number of inputs, N, are shown in Fig. 3 where we can find that the complexity of crossbar switch grows faster than those of Benes network and Banyan switch. D. Scheduling
Fig. 8 shows the processing cycles of a LDPC decoder. On Benes network only half-duplex data transmission is possible, which means that data can be transmitted in both directions, but not at the same time. As can be seen in Fig. 8, one iteration of the decoder consists of two data transmissions on Benes network, VPU operations and CPU operations. VPUs and CPUs wait until data transmission completes and start their operations. The proposed LDPC decoder divides a code block into subblocks and process a subblock at a time. The number of subblocks, k, is determined by dividing the size of a code block with the number of VPUs. In partially parallel architecture, the number of VPUs is smaller than the size of a code block and the VPUs must be reused k times to complete one iteration. Since the decoder processes a code block on a subblock basis and it takes one cycle for the decoder to

In order to reduce the cycle count, VPUs and CPUs are pipelined as shown in Fig. 9. The transmitted messages are calculated in stage 3. In stages 4 and 5 the intermediate values, tij and sji are calculated and stored in local FIFO for use in later iterations. The processing cycles after pipelining are shown in Fig. 10. The first iteration begins by transmitting initial values from VPUs to CPUs. CPUs begin operation with the first subblock and store the result values in CPU registers. After the data transmission from VPU to CPU is completed, the values in CPU registers are transferred to VPUs. VPUs begin operation with the first subblock and store the result values in VPU registers. This process is repeated until the maximum number of iterations is reached. In the final iteration the syndrome is calculated overlapped with VPU operations. The cycle count after pipelining is 2ki + 8 when the decoder iterates i times for a code block.
By applying pipelining, the cycle count is reduced by 2k(i+1)−8 cycles. The cycle count reduction ratio with respect to the cycle count of the nonpipelined decoder is plotted in Fig. 11 (a) and (b) where the values of k and i are assumed to be 16 and 6, respectively. The cycle count reduction due to pipelining decreases approaching to 50% when the number of iterations increases. However, if the number of iterations is fixed, the pipelining becomes more effective and the cycle count is reduced to about 54% of that of the nonpipelined decoder as the number of subblocks increases. This paper has presented a flexible LDPC decoding architecture proposed to support multiple code rates and code lengths achieving high throughput. In the proposed architecture, Benes network is used to implement the interconnection network that can be configured according to a parity-check matrix. In order to reduce the number of switching elements and delay of the interconnection network, we employ broadcasting technique, which reduce the number

of messages between VPUs and CPUs by transmitting only the row and column summation values. The number of VPUs and CPUs in the proposed partially parallel architecture is determined by investigating the relationship between hardware complexity and throughput. The processing units are pipelined to compensate the delay caused by using the Benes network and the operations of processing units are overlapped with that of the Benes network.
To verify the proposed architecture, a flexible LDPC decoder is implemented using a 0.18 μm CMOS process. The decoder occupies an area of 16.261mm2 and runs correctly at the frequency of 212 MHz resulting in 1 Gbps decoding throughput. Compared to the previous flexible architecture, the proposed decoder shows the highest throughput-to-area ratio. Since the proposed architecture can be easily configured to virtually any kind of parity-check matrix, it can be used for both regular and irregular LDPC codes.
