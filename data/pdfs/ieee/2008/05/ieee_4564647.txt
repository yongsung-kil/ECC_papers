A New Criterion To Compare Different LDPC Codes For Hardware Implementation

Abstract- In this paper we propose a new combinational criterion as a mean for comparison between different LDPC codes in terms of complexity and efficiency. The main idea behind the proposed criterion is to merge different aspects of a code to a unique number. We will show that the new criterion, named “Goodness Number” or GN, is applicable, and in some circumstances it is preferred to other criteria such as complexity and error probability criteria and at the same time it maintains computational simplicity1 .
Keywords: LDPC codes, Hardware Implementation, Decoder Complexity, Comparing Different Codes, Efficiency Compression
I. INTRODUCTION
LDPC codes were introduced by Gallager [1] in 1962, but they were almost forgotten for more than 30 years because of their relatively high hardware complexity. Mackay [2] rediscovered these codes in 1996 after the invention of Turbo codes. As Gallager and Mackay [1,3] showed these codes approach Shannon limits as their lengths, N, increases while their complexity for decoding increases linearly with N. With respect to the ongoing demands for implementation of LDPC codes, for example in IEEE 802.16 standards [4], it is expected to develop some criteria to compare competing codes in terms of encoder/decoder complexity, power consumption, error probability performance, latency and so on. It’s well known [5] that there is a trade off between different demands, for example codes with higher complexity usually have a better performance in terms of error probability, so a fair comparison between different codes should include different aspects of the codes, which are usually inconsistent together, in a unique criterion while criteria such as complexity and error performance give us a limited understanding.

In this paper we propose a new criterion to compare different LDPC codes together. Our criterion, which is named “Goodness Number” or GN, is capable of comparing two, or more, different codes while it incorporates two most important and competing features of a code, which are error probability and decoder complexity. We will show with some examples that GN is an useful and in some cases is the preferred criterion. The main reason that makes GN a good criterion is its performance in the actual scenarios while it is really simple at the same time. This paper is organized in five sections. In section II we introduce GN formally. Through some examples, in section III , we show that GN is useful. Finally in section IV we will give some concluding remarks.

II. GOODNESS NUMBER CRITERION

It is known that, the complexity of decoding LDPC codes is proportional to code’s length and the number of iterations. In other words, we can say that if the complexity is shown with C, then C is proportional to the products of N and the number of iterations (NItr). Equation (1) is showing this relativity

C ≡ N × NItr

(1)

The above relationship is proper to compare codes in terms of decoder complexity but it lacks information about the error probability performance of the code. To fix this drawback we introduce a new criterion by multiplying C with the corresponding error probability for a specific Eb / N 0 . The new criterion is called “Goodness Number” for a code and is defined by equation (2)

GN (Eb / N 0 )  Pe (Eb / N 0 ) ×C = Pe (Eb / N 0 ) × N × NItr

III. EXPERIMENTAL RESULTS
We show in this section that GN criterion is usable in comparison to other criteria; toward this goal we study 4 scenarios
A. Constant decoder complexity scenario The first scenario is obtained by assuming constant complexity at the receiver for all competing codes. This means that we have a constant N i × NItri for i-th code, so that each code has a different maximum iterations number associated with it. For example consider codes with lengths 96,504 and 816, obtained from [6], with a constant complexity proportional to 5×816 = 4080 . This means that the number of maximum iterations for the code with length 816 is 5. The results for 4 different values of Eb / N 0 are shown in Tables 1,2,3 and 4. It should be noticed that in computing GN, C is equal to product of N and the average iterations. Considering the two last columns of this table, the reason of profitability of GN will reveal. While C is only involved with the complexity of the decoder, GN gives a more generic view about the code, it shows the gain can be obtained by increasing the complexity of decoder through choosing a longer code or increasing the maximum number of iterations. It is obvious that a better code has a smaller GN. In the above example the code with N=96 is the best code among the set for Eb / N 0 = 2 dB, while the code with N=504 is preferable for Eb / N 0 = 3 dB and Eb / N 0 = 4 dB, and code with N=816 is surpassing the other two in Eb / N 0 = 4.5 dB. In this example, the hardware can support a limited complexity, which is fixed for all codes, but we prefer to work at the lowest possible complexity, for reasons like decreasing power consumption, without performance degradation. This aim is easily obtainable through GN criterion because it indicates the near optimum code, in the sense of compromising between complexity and performance, among a given set for a specific Eb / N 0 . If we have more restrictions, we can apply them to the code set before computing GN for them. For example, we can eliminate all codes, which do not satisfy certain specification such as reaching to a certain error probability at a specific Eb / N 0 ,exceeding a certain complexity value and so on, from the set and then choose the best code by GN criterion. D- Determining the stopping criterion In every Implementation of LDPC decoder we encounter the problem of determining the stopping criterion or, in other words, the maximum iterations number. Usually this number is set to 10 iterations [7]. In this example we will show that this choice is adequate, through two different approaches. The first is to plot obtainable gain in dB versus the maximum number of iterations for various Eb / N 0 , and the second is to use GN criterion to determine the optimum iterations number. Consider codes with N=96 and N=816. The obtainable gain versus maximum number of iterations curves, are shown in Fig. 1 and 2, respectively. As the plots suggest the maximum iterations number equal or near to 10 is optimum because with maximum iterations smaller than 10 there would be a great scarifying of performance and, at the same time, exceeding iterations would not give much better performance. Now turn back to the GN criterion, consider Tables 1,2,3 and 9,10,11 again. For code with N=96, GN is lower in Tables 9,10,11 suggesting that the maximum iterations equal to 10 is better than the maximum iterations equal to 42. There is a similar argument for code with N=816 but in this case the GNs suggest that the maximum iterations number equal to 10 is preferred to 5. Though, as this example shows, the results are consistent with Fig. 1 and 2 .

IV. CONCLUSION It was shown in the last section that in order to achieve a good result, we should choose the proper criterion according to the problem conditions. When the maximum complexity is limited, C criterion is preferred. When lowering of error probability is the main dilemma, Pe will be our main concern. If we concern about the cost of increasing the complexity of system or when the minimum requirements, according to the other two criteria, are satisfied, GN will be our primary criterion. In addition, as it is apparent from examples, GN is relatively easy to compute and use while for gathering the same information through other criteria, more effort is needed. As a general manner, the first stage in a hardware implementation is discarding codes with complexity beyond the available hardware. The second stage is to put on the shelf all codes which are not capable of reaching to a certain minimum Pe for specific Eb / N 0 and the last step is to choose the best code among the remaining, through GN criterion. It is apparent from the above examples that the selection of code is closely related to the value of Eb / N 0 therefore an essential enhancement which can applied to the system is gained through using a set of codes instead of a single code and select the best one according to Eb / N 0 of the channel. If the ensemble of our codes is large enough and we set the conditions of the problem carefully, then a great improvement will be expected compared to the single code case.
