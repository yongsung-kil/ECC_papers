Complex Rotary Codes Revisited: A Low-Complexity High-Performance Decoding Approach
Zheng Ma, Member, IEEE, Pingzhi Fan, Senior Member, IEEE, Qingchun Chen, Member, IEEE, Li Hao, Member, IEEE, and Michael Darnell, Senior Member, IEEE

Abstract—In this paper, based on a belief-propagation decoding strategy, a class of generalized parity-check codes called complex rotary codes is investigated. It is shown that, by using iterative sum-product decoding, the complex rotary codes have a much lower decoding complexity than Turbo codes, but have almost the same performance for the high code rate and short frame case (frame length< 500 bits). It is also shown that the prime block size of complex rotary codes is essential to achieve better performance because of its uniform checking characteristic.
Index Terms—Complex rotary (CR) code, sum-product algorithm, Turbo code, iterative decoding.
I. Introduction
D URING the past decade, it has been recognized that two classes of codes, namely, turbo codes and lowdensity parity-check (LDPC) codes, can achieve excellent performance very close to the Shannon limit, especially when the frame length is relatively large, e.g. 1024 bits or larger. However, in third-generation wireless communication systems (3G), beyond third-generation wireless communication systems (B3G) [1] and wireless local-area networks (WLANs) [2], there are also short frames (less than 500 information bits) which need to be transmitted. It is the purpose of this paper to investigate high rate codes for short frame applications.
Introduced in 1993 by C.Berrou et al, turbo codes have received very wide attention, but the high decoding complexity of turbo codes has hampered their practical application. Introduced by R.G.Gallager [3] and developed by MacKay, et al [4], LDPC codes are another family of extremely good codes, providing good performance, potentially at lower decoding complexity; however, the encoding complexity of

Mackay’s LDPC codes can be as high as O(K2) (K is the frame length). Structured LDPC codes, such as balanced incomplete block design (BIBD) LDPC codes [5], reduce the encoding complexity to O(K); however, good high-rate LDPC codes are difficult to construct for short frame lengths. Turbo product codes (TPC) and single parity check codes (SPC) are families of modified turbo codes having a considerably lower decoding cost at the expense of a small performance degradation. Pyndiah introduced TPC in 1998 [6]; David M. Rankin proposed a multidimensional single parity check codes on a multidimensional polyhedron with random interleaver in 2001 [7]. At almost the same time, James S. K. Tee proposed serially and parallel concatenated single parity check codes [8][9], and Ling Ping introduced the turbo-SPC, which combines the convolutional code and the SPC code [10]. Due to the poor interleaving gain, these codes have unsatisfactory performance for high rate and short frame length applications.
In 1985, by generalizing classical one-dimensional paritycheck codes to multidimensional parity-check codes, so-called complex rotary (CR) codes were proposed by Jin Fan [11]; these appear to have been largely forgotten due to their relatively low code rate. In contrast to SPC codes [8][9][10], CR codes require that the block size (the number of array columns and rows) be prime. Originally, due to the uniform parity check structure of CR codes, a majority voting decoding algorithm was adopted, exhibiting very low implementation complexity but unsatisfactory error performance when compared with other optimal multiple-error-correcting codes. It will be shown in this paper that, by employing soft input and soft output (SISO) and an iterative belief-propagation decoding strategy, the CR codes perform better than SPC codes with the same decoding complexity. For high code rate and frame length smaller than 500 bits, they perform as well as turbo codes, but with much lower decoding complexity. Moreover, although the same sum-product decoding method is adopted as with LDPC, the encoding complexity of CR codes is only O(K); also, they are more easily constructed than LDPC codes in high code rate and short data frame length cases.
II. Complex Rotary (CR) Codes
Given a prime number P and a positive integer Q which is no larger than P, the so-called [P, Q]-CR code is equivalent to a (P · (P + Q), P2) linear block code, i.e. there are P · P information bits and P · Q check bits in CR block codes [11]. It is clear that the code rate of CR codes is R = P/(P + Q).

It is well known that the larger the minimum Hamming distance of linear codes, the better their performance.

IV. Simulation Results and Analysis
In our simulation, an AWGN channel and BPSK modulation are assumed. For fair comparison, a 3-dimensional SPC(8,7) code [7], which can be expressed as a (512,343) linear block code, and a [17, 5]-CR code, which can be expressed as a (374, 289) linear block code are used; the CTC and BTC parameters employed are taken from [2]. In addition, the iteration number of the CR codes is set to 30, and the iteration number of the CTC and SPC is set to 6 [2] and 8 [7] respectively. As is shown in Fig.3, at BER = 10−5, the error performance of CR code is about 0.5dB better than that of SPC code [7], even for the higher code rate and shorter frame length with similar implementation complexity; the CR code performs as well as the CTC proposed in [2] at a BER above 10−6 , but with much smaller decoding complexity. As can be seen from Fig. 3, at a BER below 10−6, the CR code even outperforms CTC by about 0.2dB, which means that the ”error floor” of CR codes occurs later than that of CTC. In addition, at medium to high SNRs, the CR code outperforms the BTC [2] by 0.5dB at the same code rate and frame length. A further [13, 4]-CR code, which can be expressed a as (221, 169) linear block code, is also simulated and is still shown to perform as well as the BTC, even with a shorter frame length.

Further simulation results in Fig.4 show that, for different block size (different P) and almost the same code rate, the BER performance with prime block size P is about 0.5dB better than that of the nonprime block size cases. For example, for block sizes P = 16 and 18, the corresponding BER performance is about 0.5 − 1.0dB worse than the case when P is the prime number 17, although the code rates and frame lengths are almost the same.
Since we aim to use CR codes for high-rate short frame applications, values of P and Q are not likely to be large. It is therefore reasonable to assume that the parameter Q in CR codes and the constraint length J in turbo codes will be of the same order. For the practical frame lengths and code rates which are employed in the simulations, Q is no more than 5. For CTC, J is no less than 4 in order to guarantee satisfactory performance. It is shown that the (17, 5) CR code, which can be expressed as a (374,289) block code, performs as well as a CTC code of frame length 392 bits with almost the same code rate in Fig.3; here J = 5 and Q = 5. It is clear that the number of addition operations for the CR code is only about 1/425 of those for the CTC and the number of multiplication/division operations for the CR code is only about 1/72 of those for the CTC per iteration. Taking account of the fact that CR code may need more decoding iterations, the actual complexity reduction ratio may be reduced, but the addition and multiplication reduction ratios of CR codes relative to the CTC are still very high - say about 1/85 and 1/15 respectively. When compared with (8, 7) SPC code, they share almost the same decoding complexity. For the number of multiplications/divisions, the [17, 5]-CR code has slightly higher value than the 3-dimensional (8, 7) SPC code, namely, 25 for the former vs 16 for the latter. In terms of number of additions, there are 88 for the 3-dimensional (8, 7) SPC code but only 2 for the CR code per bit per iteration.
