FPGA IMPLEMENTATION OF A FLEXIBLE DECODER FOR LONG LDPC CODES
Christiane Beuschel, Hans-Jo¨rg Pfleiderer
Institute of Microelectronics, University of Ulm Albert-Einstein-Allee 43, 89081 Ulm, Germany email: christiane.beuschel@uni-ulm.de, hans-joerg.pfleiderer@uni-ulm.de

ABSTRACT
Over the last years LDPC codes became more and more popular because of their near Shannon limit error correcting performance. Structured code classes which ease decoder design have already been standardized for DVB-S2, IEEE WiMax 802.16e or WiFi. In this paper we introduce a flexible decoder architecture which can decode any structured or unstructured LDPC code using the identical hardware. Furthermore we present a mapping algorithm which ”compiles” the parity-check matrix of the desired LDPC code. This concept allows adaption of the decoder controller to different LDPC codes without requiring a new synthesis run.
We implemented the proposed decoder on a XILINX XC4LX160 FPGA and give bit error rates to verify design and mapping algorithm. In contrast to previously presented flexible implementations our design is able to decode LDPC codes of 30 times longer codeword lengths up to N = 65, 000.
1. INTRODUCTION
In the last years a lot of research was done to design structured LDPC codes that fit to certain hardware decoder architectures. This so-called decoder-aware code design is also used for the LDPC codes in recent digital communication standards, e.g. DVB-S2 [1], WiMax [2], WiFi, or CDVB-T.
Examples for decoder-aware code design are quasi-cyclic codes as in [2] or codes based on protographs as discussed in [3]. For both code classes there exist partly-parallel decoder structures which allow several codes of the same structure to be decoded with the same hardware decoder. However, optimum hardware decoders are only possible for codes with the same constituent matrix size or, in case of quasi-cyclic codes, for codes which have a common divisor of the constituent matrix size. In [4] a decoder approach for quasicyclic codes is presented which needs some extra clock cycles but can decode also codes which have no common divisor of their constituent matrix size. However, all these approaches have few degrees of freedom in choosing the degree of parallelism and furthermore code design is severely limited through the constraints imposed on the parity-check matrix during decoder-aware code design.

All these drawbacks can be circumvented by the approach to develop a decoder architecture which can decode any structured or unstructured LDPC code as in [5]. The price for the full flexibility is a more complex hardware architecture. Furthermore memory access conflicts are likely to occur when the extrinsic messages are assigned linearly to memory addresses as it is done for structured codes. Thus a more sophisticated mapping algorithm is needed.
In this paper we present a new fully programmable LDPC decoder architecture together with an appropriate mapping algorithm and an FPGA implementation which is capable of decoding codes with 30 times longer codewords as existing flexible implementations.
The structure of this paper is as follows: Section 2 shortly introduces decoding of LDPC codes, Section 3 presents our flexible decoder architecture and a mapping algorithm, Section 4 discusses the FPGA implementation of the decoder and finally Section 5 gives the conclusion.

2. LDPC CODES

A binary (N , K) LDPC code [6] is defined by a sparse M x N parity-check matrix H. Equivalently the LDPC code can be described by a bipartite graph with M check nodes corresponding to the rows and N variable nodes corresponding to the columns of H. Check node m is connected to variable node n if hmn = 1. M (n) is the set of check nodes that are connected to variable node n and N (m) is the set of variable nodes that are connected to check node m. The vector y is the transmitted codeword with yi ∈ {−1, +1} and r is the received codeword.
The iterative belief propagation (BP) decoding algorithm can be described by passing the messages Rm→n and Qn→m in two half-iterations between the nodes in the bipartite graph.

3. FLEXIBLE LDPC DECODER ARCHITECTURE
In this section we introduce a flexible hardware architecture that can decode any LDPC code and a mapping algorithm to determine the memory addresses and locations of the extrinsic messages. The mapping algorithm corresponds to a ”compilation” of the parity-check matrix before decoding starts.

3.1. Architecture
In Fig. 1 the architecture of the proposed flexible decoder is shown for a degree of parallelism p = 3. The architecture consists of p extrinsic memory banks EMi, each of these banks has an input and output wordlength of p extrinsic messages. In the intrinsic memory IM always p input values and in the decoded memory DM always p symbols of the decoded codeword are stored at the same address. All memories are dual-port memories allowing simultaneous and independent read and write access on two data entries.

The p variable node functional units VFUi and p check node functional units CFUi perform the node computations alternately in two half-iterations. The two permutation networks π and π−1 as well as the 2p multiplexers p : 1 are needed to route the required extrinsic messages to the corresponding node computation units. The central control unit determines the configuration of the multiplexers, the permutation networks, the addressing of the memories and the processing of the variable and check functional units. The control signals depend on the compiled LDPC code and are read linearly from the memory HM. The compilation can be performed offline before decoding starts using the mapping algorithm as described in Section 3.3.
During check node processing p extrinsic messages are read from each memory bank EMi. The p subsequent multiplexers p : 1 choose p extrinsic input messages which are sent over the fully programmable permutation network π to the CFUs. Each of the CFUs processes several check nodes sequentially and receives one extrinsic input message in each clock cycle. After a delay of dC + 2 cycles the updated corresponding extrinsic messages appear sequentially at the outputs of the check nodes. They are permuted using the inverse permutation network π−1 and then written back into the extrinsic memory banks to the same addresses and positions as they were read from. Altogether each CFUi processes sequentially approximately M /p check nodes of the code.
In the other half-iteration the variable nodes are processed. In each clock cycle t, p messages are read from the extrinsic memory bank EMt mod p. The subsequent multiplexers are configured in a way that these p messages are sent to the p VFUs. Together with the first extrinsic message of each variable node the intrinsic values are read from the intrinsic memory and sent to the VFUs. After a delay of dV + 2 cycles the corresponding updated extrinsic messages appear sequentially at the output of each VFU. The messages are then written back to the same extrinsic memory bank and to the same address as they were read from. Altogether each VFUi processes sequentially approximately N /p variable nodes of the code.
Decoding of a codeword is started by writing the received values sequentially to the intrinsic memory IM. Then the extrinsic memory banks EMi are initialized with the intrinsic values using the same data flow as during variable node processing except that all extrinsic input messages to the variable nodes are set to zero. Decoding of the current codeword is stopped as soon as a maximum number of iterations is reached or a valid codeword is detected.

3.3. Mapping and Scheduling
For each LDPC code the mapping of the extrinsic messages Qn→m resp. Rm→n in (4) and (6) to the extrinsic memories EMi is determined. The goal is a mapping algorithm which yields conflict-free memory access and an optimum of p processed messages in each clock cycle. However, the average degree of parallelism pa, which is in general smaller than p, depends on how good the current LDPC code can be mapped onto the decoder architecture.
During the variable node half-iteration the extrinsic messages are mapped linearly to the extrinsic memory banks. In each clock cycle the multiplexers before the VFUs select the next extrinsic memory bank. After p clock cycles the address is increased by one. At each time instant all p VFUs process nodes of the same degree to ease controlling. Therefore the average degree of parallelism during variable node processing is equal to or almost p.

During the check node half-iteration the addresses of the p extrinsic memory banks are accessed in a rather stochastic order which is determined by the biparite graph and the mapping algorithm. In each clock cycle one message is chosen from each extrinsic memory bank and sent to the CFUs. Depending on the current code the average degree of parallelism during check node processing can be smaller than p.
The mapping algorithm assigns a control vector mi,k = (addr, pos, mem) to each edge in the bipartite graph where mem is the memory bank number, addr the address and pos the position within the p values at the same address. The algorithm and can be formulated as follows:
1. Generate variable node groups: Assign always p variable nodes of the same degree to one variable node group.
2. Generate preliminary variable node subgroups: Combine always p edges from p different variable nodes of one group to one variable node subgroup and assign preliminary values to mi,k = (addr, pos, mem):
mem: All entries in a variable node subgroup get the same value for mem. The values are assigned in ascending order modulo p.
addr: All entries in a variable node subgroup get the same value for addr. The value is increased by one every pth subgroup.
pos: All entries in a variable node subgroup get different values for pos.

4. FPGA IMPLEMENTATION
We described our flexible decoder architecture with VHDL and implemented it on a XILINX Virtex4 FPGA. Special attention was payed on a scalable design where parameters can be easily changed before synthesis. These parameters are for example the wordlength of the intrinsic and extrinsic messages and the depth of the extrinsic, intrinsic and decoded memories. Furthermore the degree of parallelism p, which is not limited to powers of two but can be any natural number. The compiled parity-check matrix is stored in the external SSRAM memory HM. After synthesis the parameter α in (6) can be adjusted to the values 0.71875, 0.75, 0l78125, 0.8125, 0.84375, 0.875 or 0.90625 where each value can be represented using 4 binary digits.
Table 2 shows the synthesis results for different parameter configurations for XC4LX160-11. The degree of parallelism of the design is given by p, the wordlength of the messages by W , the depth of the intrinsic and extrinsic memories by dintr and dextr. The overall number of slices on this FPGA is 67,584 and the overall number of block RAMs is 288 with a size of 18,432 bit each.

4.1. Simulation Results
In Fig. 2 the simulation results for the bit error rate are displayed for codes C2, C5 and C7. Simulations on the PC (dashed lines) are compared with finite-wordlength simulations on the FPGA (solid lines). While the software simulations were performed using floating-point and the optimum BP algorithm, the FPGA implementation uses the normalized BP-based approximation as described in Section 3.2. The total wordlength for all intrinsic and extrinsic messages is 8 bit consisting of 1 sign bit, 5 bits for the integer part and 2 bits for the fractional part. For C2 a maximum number of 50 iterations and for C5 and C7 a maximum number of 100 iterations is used for all simulations.
As discussed in [8] the performance loss for the floatingpoint normalized BP-based approximation compared to the optimum BP algorithm is 0.05 to 0.1 dB. In Fig. 2 it can be seen that the approximation of the decoding algorithm, truncation and finite wordlengths of our FPGA implementation yield 0.1-0.15 dB performance loss. To show that our decoder is not restricted to structured codes like C5 or C7 we also simulated the bit error rate for C2 which was decoded with the identical hardware as the other two codes. With these simulations we are able to show the principle correctness of the architecture implementation and the mapping algorithm.

As the wordlength for the intrinsic and extrinsic messages is flexible and can be set to an arbitrary value, in our design we compare only the number of intrinsic and extrinsic messages that are stored in a memory. While the number of messages stored in the intrinsic memory is the same for all designs, [5] needs a memory for 4E messages to store the extrinsic and sum values whereas our design and [11] only need one fourth of this memory.
For the interconnection between the memories and the node computation units our architecture uses two Benesˇ networks with p = 16 in/out values and additional multiplexers, which accumulate to an overall number of 720 W x(2 : 1) multiplexers where W is the wordlength of the messages. In [5] one Benesˇ network with 32 in/out values is used which leads to a total of 288 W x(2 : 1) multiplexers and in [11] a 360 input/output shifter network is used which yields 1,620 W x(2 : 1) multiplexers.
While we implemented the decoding algorithm using the normalized BP-based approximation (6), our design is not limited to this implementation and the node computation units can be easily exchanged against any other approximation. Also in [11] the architecture does not restrict the node computations to a certain approximation. In [5] the decoder architecture always requires a total-sum first implementation for both check and variable node computations.

Looking at codes with maximum size that can be decoded with the architecture our limits for the maximum codeword length Nmax and maximum number of ones Emax in H are similar to [11]. Compared to the flexible decoder implementation in [5] we are able to decode any LDPC with 30 times longer codeword length and 30 time more ones in the parity-check matrix.
5. CONCLUSION
In this paper we presented a flexible LDPC decoder architecture which can decode any structured or unstructured LDPC code. Furthermore we implemented the proposed decoder architecture on an FPGA and showed simulation results for several codes. Compared to existing flexible decoder implementations our architecture has only half as many idle node functional units, uses 75% less memory for the extrinsic messages and can decode LDPC codes with 30 times longer codewords.
