ERROR FLOOR ANALYSIS FOR AN ENSEMBLE OF EASILY IMPLEMENTABLE IRREGULAR (2048, 1024) LDPC CODES
Chad A. Cole ccole@syrres.com
Syracuse Research Corp Chantilly, VA 20151

Abstract-The following paper describes a design process for constructing semirandom LDPC codes with characteristics that are suitable for a relatively simple implementation for both the encoding and decoding operation. The paper will focus on two particular code ensembles - both rate-I/2 (2048, 1024) designs with a specified irregular degree distribution. These code parameters were chosen simply because they satisfied a project design constraint, but the process described can be extended to most other low-density designs. Some new insights into the code's performance curve behavior in the low error region under message passing decoding are presented.
Index Terms-LDPC, error floors, irregular designs.
I. INTRODUCTION
The LDPC code literature has a plethora of code design algorithms. From the earliest random LDPC code designs presented in Gallager's original work [1] to the more theoretically satisfying algebraic constructions of later researchers [2], [3], [4], there is no shortage of code design options. Some random code ensembles are easy to construct and analyze. They also offer a wider array of degree distributions, block lengths, and code rates. The more structured algebraic methods typically use graph theory or algebraic arguments to ensure a larger girth for the code. Increasing the girth, or length of the shortest cycle in the code's Tanner graph, will allow the LDPC decoding algorithm to perform better.
The original focus of LDPC code design researchers was to optimize the threshold of a code, or the SNR where for large block-length codes the BER suddenly drops precipitously. This subject was broached in Gallager's dissertation for regular codes and then more thoroughly addressed in work by Richardson et al. [5] where the concept of density evolution is conceived. Near capacity-achieving codes have been observed by using these methods to construct very large block-length irregular codes. Irregular codes [6], meaning a variable number of '1 's in each column of the parity check matrix H describing the code, have been shown to have near capacity thresholds.
A more recent goal of LDPC code designers is to lower the error floor of a code - the SNR at which the error rate

of the code begins to drop at a much slower rate as the SNR increases. The iterative decoding algorithm typically used to decode LDPC codes, known as the sum product algorithm (SPA), or generically as a message passing algorithm, can induce a new non-codeword type of error in the decoder output called a trapping set (TS) [7]. The error floor problem is different for every code and not just an ensemble of codes. Although statements can be made about the average error floor behavior for an ensemble of codes, no definitive statements can be made about the individual code until it has been thoroughly analyzed by itself. The error floor behavior of short block-length codes has more variance than for long block-length codes. Since most real error control coding applications have a limited amount of memory and processing power to store and encode/decode the LDPC code, it becomes imperative for code designers to keep a watchful eye on error floor behavior in their code designs. These considerations are especially important for applications which do require a very low bit error rate at a specified signal-to-noise ratio (SNR).
This paper focuses on the application of the low bit error rate analysis tools developed in [8] to a family of practical irregular codes. All three steps of the error floor analysis tool are performed for a number of codes from the ensemble and some useful observations are documented.
This paper is organized as follows: Section II gives a brief overview of LDPC codes and trapping sets. Section III describes the two code ensembles analyzed in this paper. Section IV provides some analysis and observations of the two code ensembles. It also illustrates some simulation results. The final section summarizes the contribution offered in this paper.

Regular codes have an equal number of edges coming from each variable node, dv, and check node, de, in the Tanner graph. Regular codes are typically easier to analyze and as shown by the MPA equations above, will have an evenly distributed computational load among the nodes, thus making them more amenable to parallel processing. Irregular codes, where the number of edges coming from each variable node can differ, will be more difficult to implement because there is more variation between the maximum and minimum edge degrees for both variable and check nodes. The two irregular designs presented in this paper try to alleviate this disadvantage by keeping the spread between the max and min dv small.
As with all linear codes, it is convenient to assume the all-zeros codeword as a reference vector for study. Small subsets of non-zero bits which do not form a valid codeword but still cause the MPA decoder problems are well-documented in the literature and typically referred to as trapping sets (TS) [7]. A TS, x, is a length-n bit vector
described by a pair (a, b), where a is the Hamming weight
of the bit vector and b is the number of unsatisfied checks, i.e. the Hamming weight of the syndrome xHT . The reason why TS are problematic for the decoder is seen when the a bits of a TS have sufficient noise to cause them to individually appear as strong l's, while the others in the subgraph are soft 0 'So This causes the check nodes connected to the a TS variable nodes to appear to be satisfied, which tends to reinforce the wrong state in the graph. Increasing the graph girth will lead to a more desirable TS spectrum across a code ensemble. Increasing the average weight of dv (and thus de also by conservation of edges) will also decrease the number of low-weight TS [10].
The type of codes analyzed in this paper can also be viewed as irregular repeat accumulate (IRA) codes and can be decoded with a 'turbo-like' algorithm that uses the BCJR [11] algorithm at the accumulator. This method is shown in [12] to provide a slight performance increase over the typical LDPe MPA as described above. This performance increase could potentially become large in the high SNR region where TS dominate. This is a subject for continued research and will not be addressed in this paper.

IV. CODE ANALYSIS
The low-weight (x, 1) and (x,2) TS arising from the staircase fast-encoding constraint exhibit strange behavior in the SNR region where the error floor begins to appear. The d~ for these TS is larger than would be expected from their low Hamming weight and small number of unsatisfied checks.
Theorem 1. The staircase code construction constraint ensures the existence of (x, 1) TS, x E [1,2, ... ,n - k].
Proof· An (x,l) TS is formed by taking the last x columns of H in the format of Figure 1 and adding them modulo two. The resulting syndrome vector will have zeros in the bottom x-I positions, a '1' in the x th position from the bottom, and zeros in the first n - k - x positions. •

Proof· Take any run of x consecutive columns of H occurring among the parity bits and excluding the final
column where dv == 1. There are n - k - x of these
runs and when these columns are added modulo two, the syndrome will always have a weight of two due to the two , 1's corresponding to the first and last columns of the run.
•
Adding additional '1 's to the parity bits in H underneath the diagonal can reduce the number of these problematic low-weight (x,l) and (x,2) TS, but will increase the average dv of the parity nodes, which will negatively affect the decoding threshold. Some interesting work on this topic has been performed in [17].
A representative code from each of the two ensembles was chosen based on its superior error floor characteristics from among 10 semi-randomly generated codes from each of the two ensembles. There are three types of dominant error events in these codes.
• Type 1 - The first type of errors would arise for any type of decoder, even the ML decoder, and they are the
low-weight code words (i.e. (x, 0) TS).
• Type 2 - The next type of dominant error event is the (a,b) TS with a small and b equal to 1 or 2 and all of the TS bits are entirely within the staircase (parity) bits of the graph.

• Type 3 - The last type of dominant error event is TS with similar low-weight a and b as Type 2 errors, but the TS bits are a combination of runs of consecutive staircase (parity) bits and a small number (usually one
or two) of the dv == 4 or dv == 5 bits in the 4-8 and
5-9 code ensembles respectively.
The representative 4-8 code has a dmin == 15 and the chosen 5-9 code has dmin == 22. Even though a dmin of
15 is not particularly good for a rate-l/2 code of block length 2048, surprisingly it is still not the cause of the error floor for this code. The non-codeword TS, both error Types 2 and 3, will ultimately cause the error floor when any sort of suboptimal message passing algorithm is used for decoding. Both codes have the undesirable property of a low-weight (x,l) and (x,2) TS spectrum induced by the staircase pattern governed by the two theorems listed in this section above. Thus Type 2 errors are unavoidable with any staircase H construction such as the 4-8 and 59 ensembles defined here. The Type 3 error events differ markedly between the two code ensembles and are the chief cause of the onset of the error floor at Eb/No ~ 2 - 2.5 dB. It is also interesting to observe that as the SNR increases,
the Type 2 error events of a given (a, b) TS become just
as dominant as the error events of Type 3 with an equal (a, b) weight. It should be noted that this is only shown for the full belief propagation [18] version of the message passing algorithm. Such observations should not hold true for a min-sum [19] implementation of the message passing algorithm, as it is invariant to SNR. This convergence in behavior for Type 2 and Type 3 errors is interesting because it implies that the two code ensembles should begin to have virtually indistinguishable error performance at higher SNRs due to their equal share of Type 2 error events. This is useful for the code designer to know, because no amount of code tweaking, such as proposed in [13] for example, can improve the highest-SNR behavior of the code. The staircase structure puts unavoidable weaknesses in the error floor which don't become evident until frame error rates of
typically < 10-6 or so.
The increasing dominance of Type 2 errors will cause the performance curves of the 4-8 and 5-9 codes to converge at higher SNRs. Also, the trade of a 0.25 dB loss incurred in the performance curve in the lower SNR region for the 5-9 ensemble may not be worth the small improvement in performance over a 1-2 dB interval where the error floor first begins. V. CONCLUSIONS
The work presented here is helpful in the analysis and design of efficiently implementable irregular LDPC codes with low error floors. Once a desired short-to-moderatelength ensemble of LOPC codes has been selected, the tools

from [16] are used to facilitate the process of designing
and determining the low bit error rate performance for a
specific code from the ensemble. The ability to accurately
analyze and iteratively design a code with a low error floor
is useful for applications that must have a guaranteed low
error rate at a given SNR, and the work presented here is
the first application of the methods from [16] to irregular
code designs.
