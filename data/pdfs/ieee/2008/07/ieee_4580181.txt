Vodafone Chair Mobile Communications Systems, Technische Universita¨t Dresden, D-01062 Dresden, Germany Emails: {tavares, steffen.kunze, matus, fettweis}@ifn.et.tu-dresden.de

Abstract
In this paper, we present a novel high-speed dual-core programmable decoder architecture for LDPC convolutional codes and their tail-biting versions. This architecture uses a modified Min-Sum algorithm and enables the decoding of a multitude of codes with different node degree distributions, rates and block lengths. We show how the parallelization concepts are derived using the properties of the bipartite graphs underlying the codes. Moreover, the hardware elements composing the architecture will be presented and analyzed in detail. The programmability of the decoder is also considered. Finally, we present the synthesis results for a prototype ASIC which is capable of achieving high decoding throughput still with very high flexibility, relatively low power consumption and small area.
1. Introduction
Communications systems of the next generation impose extreme requirements on the error control coding apparatus. On one hand, there is the need for high decoding throughput and capacity approaching performance. On the other hand, high flexibility and low power consumption must be provided.
The class of low-density parity-check (LDPC) codes introduced by Gallager in [4] is able to satisfy the requirements from above [10]. However, in order to achieve highspeed decoding architectures with low complexity, some constraints are generally imposed on the construction of the LDPC codes [8]. In this context, the convolutional counterparts of Gallager’s codes – the LDPC convolutional codes (LDPCCCs) [1] – represent an elegant alternative to cope with the requirements of future communication systems.
For instance, it can be observed that the main bottleneck in any LDPC decoding architecture is represented by the data transfer operations required by the belief-propagation (BP) algorithm. Fortunately, the bipartite graphs underlying the LDPCCCs have special properties that can be exploited

to reduce the complexity of the data transfers. Furthermore, a system using LDPCCCs might benefit from additional flexibility and lower implementation complexity because the same encoder1 (decoder) can be used to encode (decode) code sequences with different lengths [1, 9, 11].
Although LDPCCCs are very well suited for coding streaming data or packetized data of different lengths, the introduction of the zero-tail that is needed for bringing the encoder back into the zero-state results in the so-called rateloss. This effect is specially noticeable for small block lengths. In order to circumvent the rate-loss problem, the tail-biting LDPCCCs (TB-LDPCCCs) have been introduced [6, 7]. On one hand, the graphs representing the TBLDPCCCs inherit the locality and regularity properties of the mother LDPCCCs from which they were derived. On the other hand, it was observed that these codes show a dual behavior with respect to their minimum distances. Namely, their minimum distances depend on the free distances dfree of the mother LDPCCCs and on their block lengths. Additionally, simulation results have shown that TB-LDPCCCs provide similar performances to semi-random LDPC codes but at lower implementation complexity [6, 7].
Differently from the architectures in, e.g., [8, 13] which are limited by the block lengths or a sub-block size, we propose in this paper the concepts and realization of a fully programmable architecture that is able to decode any LDPCCCs and TB-LDPCCCs showing quasi-cyclic symmetries.

3. Decoder Parallelization

The bipartite graphs underlying LDPCCCs and TBLDPCCCs have two important properties that allows us to derive efficient parallelization strategies. The first property is the locality, which corresponds to the relatively small separation between the variable nodes (VNs) (i.e., symbols of the codewords) that are being checked by an arbitrary check node (CN). The second property is the regularity, which concerns the invariance of the graph connections through the time.

3.1. Definitions and Observations
In Fig. 1, a section of the abstract bipartite graph G = (C, V, E) of an LDPCCC is depicted. This graph does not contain any concrete connection between nodes. However, the spatial organization of the nodes that permits the derivation of efficient decoding algorithms is shown. Here, C is the set of all CNs, V is the set of all VNs and E is the set of all edges. The main facts associated with such graphs are:
• VNs within a time slot tk are produced at the same time instant by the LDPC convolutional encoder.
• An arbitrary CN (VN) within a time slot tk will be connected only with VNs (CNs) from time slots t , where tk−ms ≤ t ≤ tk (tk+ms ≥ t ≥ tk). In other words, this is the locality property.
• CNs (VNs) can be grouped in the sets Cj = {· · · , cj,k−1, cj,k, cj,k+1, · · · } ⊂ C (Vi = {· · · , vi,k−1, vi,k, vi,k+1, · · · } ⊂ V) according to their connection profiles. We say that CNs (VNs) cj,k (vi,k) with the same index j (i) are of the same type j (i). Actually, the CNs (VNs) cj,k and cj,k+l (vi,k and vi,k+l) are connected to VNs (CNs) of the same type, but l time slots apart from each other. In other words, this is the regularity property.
• We have |Cj1 | = |Cj2 |, ∀j1, j2 (|Vi1 | = |Vi2 |, ∀i1, i2) and we also have |Cj| = |Vi|, ∀i, j, where | · | is the cardinality of a set. Moreover, |Cj| = |Vi| is the total number of time slots of the code sequence.

3.2 Parallelization Concept
The parallelization techniques used in our architecture can be separated into horizontal parallelization and vertical parallelization in analogy with the horizontal and vertical directions of the graph in Fig. 1, respectively.
The horizontal4 parallelization consists in processing nodes of the same type in parallel. For instance, if we have Γp (p = 1, · · · , NΓ) parallel processing units (PUs), the elements in each of the subsets of CNs (VNs) Ckj ,NΓ = {cj,k, · · · , cj,k+NΓ−1} ⊂ Cj (Vki ,NΓ = {vi,k, · · · , vi,k+NΓ−1} ⊂ Vi), with k = nNΓ and n = 0, · · · , |Cj|/NΓ − 1 (n = 0, · · · , |Vi|/NΓ − 1)5, can be processed in parallel.
On the other hand, the vertical parallelization consists in processing nodes of different types simultaneously. It is evident that both parallelization concepts (horizontal and vertical) are not mutually exclusive and can complement each other. Thus, they can be combined to obtain the horizontal-vertical parallelization. In this case, Λp (p = 1, · · · , NΛ) horizontal-parallelized computation cores, each with NΓ PUs, will be operating in parallel on NΛ sets of CNs (VNs) Ckj ,NΓ (Vik,NΓ ), with j (i) equal to {mNΛ, · · · , (m+1)NΛ−1} and m = 0, · · · , |NC|/NΛ − 1 (m = 0, · · · , |NV|/NΛ − 1)6.

4.2. Address Generation Units (AGUs)
In the AGUs, the data vector addresses are calculated from the code parameters stored in the ROMs. The outputs of the AGUs are used immediately to load data from the data memories but also kept in FIFOs until the loaded data is processed and needs to be stored again. The AGUs (AGU1 and AGU2) work independently for both processor cores. They provide address generation for terminated as well as for tail-biting LDPCCCs.
4.3. Routing Network
The routing network of the proposed architecture is one of the components where the regular structure of the LDPCCCs becomes most visible in terms of complexity reduction. This network consist only of two barrel shifters (Shuffles in Fig. 2), a 2 × 2 crossbar switch and their inverses. If only a single LDPCCC and its TB-LDPCCCs versions are considered for implementation, the barrel shifter becomes even more simple since not all shifts and cyclic shifts need to be implemented. In our implementation, the barrel shifters are partitioned into three pipeline stages in order to enable high-speed decoding.

4.4. Arithmetic Logic Unit (ALU)
The vector ALUs represent the main component of the computing cores. For our particular implementation, the vector ALUs (see Fig. 2) consist of NΓ = 64 functional units (FUs) that operate in a SIMD fashion, and thus, exploit the regularity of the code. The ALU FUs (Fig. 4) combine the functionalities to compute the CN updates and VN updates using the Offset Min-Sum (OMS) algorithm [5], which represents a good tradeoff between implementation complexity and performance. 6. Results
6.1. BER Performance
Fig. 5 shows simulation results that compare the BER performance of the BP algorithm in a floating point implementation against the 5-bit fixed-point OMS algorithm with β = 0.11. The code used was a regular (ms = 127, J = 3, K = 5)-TB-LDPCCC, where J is the VN degree and K is the CN degree. Furthermore, two different block lengths (i.e., L = 3200 and L = 6400) and I = 10 and I = 50 iterations were used. It can be observed that there is a very small performance degradation (≈ 0.2 dB) when using the fixed-point OMS algorithm, whereas hardware complexity is greatly reduced in comparison with the floating point implementation of the BP algorithm.
6.2. Synthesis
In order to enable cost (i.e., area and power consumptions) estimation of the proposed decoder architecture, the design was synthesized towards 0.13 µm standard cell CMOS technology for a target operating frequency of 200 MHz using the SYNOPSYS tool chain. The synthesized architecture comprises two computation cores (NΛ = 2) with horizontal parallelism NΓ = 64 and word width Nbit = 5 bits. The complete functional core was combined with 32 KByte instruction and 40 KByte data memory macros generated by Faraday’s memory generator. Moreover, the instruction and data word widths in our implementation are 256 bits and 320 bits, respectively. The post-synthesis area estimation of the principal components is summarized in Table 1. As we can observe, the complete functional core occupies an area of ≈ 350 KGates.

7. Conclusions
This paper presented the concepts and realization of a high-speed, full-programmable decoder architecture for LDPCCCs and their tail-biting versions. We showed how the parallelization strategies used in this architecture could be derived from properties of the bipartite graphs underlying the LDPCCCs. The first parallelization concept (i.e., horizontal parallelization) is based on the processing of graph nodes of the same type and results in the SIMD parallelism. The second parallelization strategy (i.e., vertical) consists in processing nodes of different types in parallel and results in a distributed computing system.
