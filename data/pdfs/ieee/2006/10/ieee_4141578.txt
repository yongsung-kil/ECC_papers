Fig.3 shows a block diagram of the decoder architecture for LDPC codes whose parity check matrices have the structure mentioned in 3.1. It provides p check node processors (CNOD), p variable node processors (VNOD), and memories which can store in or read out p data corresponding to a component matrix together. Barrel shifters (SFT1,SFT2) permute p data corresponding to component matrices. In this architecture, all memories are implemented by single port RAMs.
Memories for received LLR (RMEM) and for messages from VNOD (DMEM) are composed of two bank of memories, for high throughput decoding. That is, while one bank of RMEM is storing input data, the other bank is provided for decoding using the data already stored. And, while one bank of DMEM is provided for decoding, the other bank is reading out the already decoded data (Fig.4). The size of each bank of memory is bit width of p x bits/data, and word size of code length divided by p.

4. Quantization Method

4.1. Quantization outside Check Node Processors

We adopt 2's complement uniform quantization outside the CNOD, because it is suitable for implementing adders. We assigned 6 bits for the received LLR and messages from CNOD, while 7 bits for messages from VNOD in order to preserve dynamic range of 6 bits after subtraction at the entrance of CNOD.

The message memory (MMEM) is divided into two banks, so that one is reading out messages to check node processors, and the other is storing messages from check node processors. The size of each bank of memory is bit width of p x bits/message, and word size of number of total messages divided by 2.
A decoding iteration is divided into two phases, such as check node processing phase and variable node processing phase (Fig.5). In the check node processing phase, DMEM is randomly accessed, and MMEM is sequentially accessed. In the variable node processing phase, MMEM is randomly accessed, and RMEM and DMEM is sequentially accessed. We can decode various code rate and code length of LDPC codes by programming the memory addressing sequence.

4.2. Architecture of Check Node Processors The architecture, which is provided with p processors and
memories for storing in and reading out p data together, faces a problem that chip area increases proportional to the number of the memory macros because of the large band width when p is large. Therefore, processors are strongly required to have serial input/output interface.
Since we want a high performance decoder in a wide range of code rates, the UMP algorithm, which yields severe performance degradation especially for low rate code, was rejected. BCJR algorithm, which requires a large amount of LIFO memory especially for high rate code, was also rejected.

4.3.5. Complexity ofCheck Node Processors

LDPC codes of the DVB-S2 standard [9] are defined by structured parity check matrices. After applying row and column permutation to the matrices, they can be converted to be composed of p = 360 component matrices, such as zero matrix, circulant permutation matrices, and summation of multiple circulant permutation matrices, which can be applicable to our partly parallel decoder architecture. The received data should be reordered according to a column permutation of the parity check matrix, by controlling the write address of the received LLR memory.
5.2. Outline of the Decoder VLSI
We developed a LDPC decoder VLSI according to the architecture mentioned above, which is fully compatible for all 21 codes as defined in DVB-S2. The features of the decoder VLSI are listed in Table 3. The maximum throughput of 150 Mbps means that this decoder supports 32APSK constellation at 30 Msym/s.
