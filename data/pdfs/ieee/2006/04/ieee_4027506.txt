High-Throughput LDPC Decoder for Long Code-Length

Tatsuyuki ISHIKAWA, Kazunori SHIMIZU, Takeshi IKENAGA and Satoshi GOTO
Graduate School of Information, Production and Systems, Waseda University 2-7 Hibikino, Wakamatsuku, Kitakyushushi, Fukuoka 808-0135, Japan

Abstract We have designed and implemented the LDPC decoder with memory-reduction method to achieve high-throughput and practical hardware size for long code-length. The decoder decodes (3,6)11520-bit regular LDPC codes using modified minsum algorithm. The decoder achieves a throughput of 312 Mb/s at an operating frequency of 69 MHz with 20 iterative decoding. The gate count is 2M gates.

I. INTRODUCTION

Low-Density Parity-Check (LDPC) codes achieve good performance of error correcting with long code-length. Some works have been done on designing LDPC de-
coder with long code-length[3] [4]. However, the through-
puts of these decoders are less than 200 Mb/s. For future application requires high-throughput with long codelength, a new decoder architecture that achieves both high-throughput and small hardware size is required.
The decoding algorithm of LDPC codes exchanges messages between check-nodes and bit-nodes on tanner graph by performing row operations and column operations iteratively. To achieve high-throughput, it is necessary that row operations and column operations are operated concurrently. However, large amount of memory is required to operate these concurrently. We have proposed the method to improve this problem. In this method, a row operation outputs minimum absolute value, second minimum absolute value, these locations and signs instead of actual output value. We designed high-throughput LDPC decoder for long code-length using this method.

111. PARALLEL DECODER ARCHITECTURE
Fig.5 shows partially-parallel LDPC decoder architectures. CFU (Check-node Function Unit) indicates row operation module. BFU (Bit-node Function Unit) indicates column operation module. Partially-parallel LDPC decoder has some memory array to store temporary data of iterative decoding.
A. Shared memory architecture
Fig.5(a) shows shared memory architecture of partially-
parallel LDPC decoder. In this architecture, temporary data memory array of CFUs and BFUs are shared so that row operations and column operations can not be executed concurrently. Fig.6(a) shows decoding time of this architecture. To achieve high-throughput in this architecture, it is necessary that number of CFUs and BFUs is increased. However, number of CFUs and BFUs affects hardware size.

Individual memory architecture achieve twice throughput compared to shared memory architecture. However, individual memory architecture requires large amount of memory requirements. We proposed two methods to reduce memory requirements of individual memory architecture. In case of our designed decoder, it can be reduced 55% memory requirements using these method.
A. Reducing row operation outputs
To reduce memory requirements, we focus attention on the row-operations. It is clear that these output only 2 absolute value at each row. It is only need to store 2 absolute value, these locations and signs. The locations indicate which absolute value must be used. This infor-
mation requires only roumdup(log2 Wr) bit. Roumdup(x)
is a function that rounds up a decimal number to an in-
teger. T/Vr indicates row-weight of matrix,

B. Shifting-out used data
Fig.7 shows 10 x 10 sub-block of matrix and Fig.8 shows timing diagram of decoding this sub-block. Row operation start at m=1 and column operation starts n=1. Row operations and column operations output one cycle latency at each row or column operations. For example, alpha at m=1 and n=3 (a13) is calculated at t=2, and
column operation uses this at t=3. a13 never used in
this iteration. Therefore, it is only one a must be stored.
Analogously, beta at m=1 and n=3 (313) is calculated at t=4, and row operation uses at t =11. /13 never use
in this iteration. So it is only seven /3 must be stored.
Number of ag and /3 to be stored depends on shift value
of sub-block of matrix. Table.I shows memory requirements using this method. Apq and Bpq indicates number
of ag and /3 to be stored at each sub-block. Q indicates quantized bit of temporary data. T/Vr and Wc indicate
row-weight and column-weight of matrix respectively.
