Error Correction Using a Message-Passing Decoder to Process Cyclic Redundancy Checks
Quentin H. Spencer Distribution Control Systems, Inc. 945 Hornet Dr., Hazelwood, Missouri, 63042, USA
qspencer@ieee.org

AbstractŌĆö This paper proposes a method for correcting errors in messages encoded using cyclic redundancy checks (CRCs), which are typically only used for error detection. This is accomplished for messages of a known length by deriving a paritycheck matrix representation of the CRC. The parity-check matrix can then be used to correct errors using the message-passing decoder (MPD) commonly used to decode LDPC codes. The CRC parity-check matrix is not sparse, which inhibits performance of the MPD, but this effect can be reduced by modifying the matrix using recently proposed sparsification techniques to eliminate short cycles. The technique is practical mainly for codes with short block sizes, and can also be applied to coding schemes that concatenate a CRC with an outer error correcting code. Simulation results demonstrate some performance gains, although less than what can be gained using custom-designed LDPC codes for the same rate and message length.
I. INTRODUCTION
The impressive performance of low-density parity-check (LDPC) and related codes using iterative decoding techniques has raised the question of whether similar decoding methods can be used to improve the performance of existing codes. While the message-passing decoder (MPD) commonly used to decode LDPC codes can theoretically be applied to any linear block code, there is little advantage in this for common algebraic codes such as Hamming, BCH, and Reed-Solomon codes because their parity-check matrices are relatively dense. This high density leads to a very large number of short cycles in the Tanner graph of the code, which significantly degrades performance relative more sparse LDPC designs. However, sparsification methods have been proposed recently [1, 2] that can remove all cycles of length 4 from any block code, resulting in performance improvements over what can be achieved using algebraic decoders.
For some communications applications with short message lengths, it is important to keep coding overhead to a minimum, but it is desirable to correct as many errors as possible. The use of the MPD with extended parity-check matrices for these cases can improve performance. However, many short block high-rate codes also have a low minimum distance, making the probability of undetected error relatively high. It is therefore common in this situation to add a secondary error checking mechanism, such as a cyclic redundancy check (CRC), to each message.
While the most common reference implementation of the CRC is a linear feedback shift register, various alternate

methods have been proposed with the goal of speeding up the computation of a CRC in customized hardware. Some of the propose parallelization methods [3, 4] can be used to generate an equivalent parity-check matrix representation of an arbitrary CRC for messages of an arbitrary length. When the CRC is used with an outer error correcting code, the paritycheck matrices of the CRC and outer code can be combined into a single parity-check matrix, which can then be extended using sparsification methods to eliminate short cycles. This can improve the overall packet error rate, but this potentially comes at the cost of increasing the number of undetected errors.
In this paper, we describe the method of deriving a paritycheck matrix representation from a CRC, and illustrate the trade-off between error detection and error correction that results when using this parity-check matrix with the message passing decoder. We then present simulation results that compare MPD performance of codes consisting of various concatenations of CRC with Hamming and BCH codes, as well as customized LDPC codes of the same dimensions. We also propose methods of reducing undetected errors in order to achieve comparable undetected error rates to the traditional hard-decision CRC decoder.
II. CREATING A CRC PARITY CHECK MATRIX
The most typical implementation of the CRC is using a linear feedback shift register (LFSR). Hardware CRC implementations using LFSRs are very simple, and they are able to process messages of any length. However, for very long messages over high-speed communication links, a serial process of encoding and decoding is often too slow. For this reason, there has been interest in designing hardware that performs this operation either partially or completely in parallel [4]. The ideas behind parallelizing CRC computation can be applied to the derivation of parity-check matrix representations of CRC.

As noted earlier, this process for creating a parity-check matrix can also be viewed as the computation of an impulse response of a filter. The LFSR implementation of CRC computation consists of a shift register with taps on some of the elements of the register which are fed back and summed with the input. This is structurally very similar to an all-pole infinite impulse response (IIR) filter, except that instead of operating on real-valued inputs, all arithmetic is performed in GF(2). In both cases, the impulse response is infinite, but it can be calculated up to an arbitrary finite length. While an IIR filter typically only has one output, with a CRC we want the entire contents of the shift register, or M outputs for one input vector. One of the advantages of using CRC for error checking is that they can easily be computed for packets of any arbitrary length. The creation of a parity-check matrix compromises this flexibility somewhat because it imposes a maximum length. However, for communication protocols where the maximum packet length is already known, it is possible to compute the parity-check matrix for the maximum possible length and use the matrix in the same way as a shortened block code, by assuming that packets shorter than the maximum length are padded by zeros. When a CRC parity-check matrix is combined with an outer block code, this poses a potential problem. Suppose in the example above, the outer code is the (255, 247) Hamming code and the CRC is a CRC-16. The longest possible unencoded packet is 231 bits. This would result in the need to store a separate parity-check matrix in the receiver for every possible packet length, although this may be feasible if the number of unique lengths that must be supported is limited.

This can be solved to some extent by applying algorithms designed to remove cycles of length 4 [1, 2], although they are limited in that the auxiliary variables that are added create additional unknowns that must be solved in order to converge to a valid codeword.
The second problem posed by this method of error correction is that it reduces the number of errors that are detected. Avoiding undetected errors is a particular challenge for almost all high-rate codes with short message length, including LDPC codes [5].
Figure 2 illustrates the trade-off between error detection and correction for BPSK signals with additive white Gaussian noise, by comparing the overall packet error rates with undetected error rates for 100-bit packets using the CRC-16. Four decoding approaches are included. The first is simple error detection with no correction attempted. The second case uses the derived parity-check matrix (extended using the 4cycle removal algorithm from [2]) to correct errors, with the maximum number of iterations in the decoder set at 20 (increasing it beyond 20 had little effect). The third and fourth cases use error correction with limitations. One case imposes a limit of one bit per packet to be corrected and assumes an error has occurred if more than one bit is changed during decoding. The other case imposes an artificially low limit of 5 iterations on the decoder. There are two curves for each of these cases, the packet error rate and the undetected error rate. Using the CRC-16 in the traditional way results in the lowest undetected error rate, which reaches its maximum level at 5 dB. As the SNR approaches 0, the undetected error rate will eventually approach 2-16, the probability that a random pattern will satisfy the 16 parity checks. Using the CRC bits for error correction instead of detection results in a substantial reduction in error rate, with a gain of more than 4 dB at a packet error probability of 10-2. The cost of this error correction capability (in addition

to more computation) is an increase of more than two orders of magnitude in the probability of undetected error, such that the majority of errors are not detected at high SNR. The two cases where limits are imposed on the error correction achieve a compromise between these two extremes. Limiting the number of iterations appear to be a better approach than limiting the number of corrections, since it has better undetected and overall error rates. In this case the increase in undetected errors is only about one order of magnitude, while the gain in the overall error rate is still over 2 dB at 10-2. This idea was pursued further by limiting the number of iterations to less than 5 (these results are not included in Figure 2). For 3-4 iterations, there is a slight increase in the overall error rate and corresponding slight decrease in undetected error rate. For 2 iterations, only a few errors are corrected, but the undetected error rate is slightly below that of the traditional CRC error detection method. When only one iteration of the MPD was performed, no errors are corrected, but the undetected error rate is reduced to less than 10ŌłÆ6 (with a simulation size of 5├Ś10ŌłÆ6). This suggests the use of a single-iteration MPD as a soft-decision error detection method to improve the error detection ability of CRCs.
One cause of undetected errors is the presence of lowweight codewords, which can cause the MPD to erroneously converge to a valid codeword that is ŌĆ£closerŌĆØ to the received signal than the transmitted codeword [6]. This is a particularly challenging problem for codes of high rate and short block size, such as those we are considering here. Most CRCs are designed so that they will detect all error patterns of odd weight plus patterns of weight 2. When viewed as linear block codes, this means that they have a weight spectrum consisting of all even numbered weights beginning at a minimum distance of 4. These codewords of weight 4 likely make a significant contribution to the undetected errors observed when using the MPD to decode a CRC-16 in Figure 2. This problem is reduced somewhat when used in conjunction with other error correcting codes. If two block codes are concatenated to produce a combined parity check matrix as in equation (5), the set of valid codewords will be the intersection of the set of valid codewords for H(N, K) and 0 HC(NRCŌłÆ-M K) , respectively. This implies that the minimum distance of the combined code will be greater than or equal to the maximum of the minimum distances of the combined codes. For example, if a CRC is combined with a Hamming code, which has minimum distance 3, the minimum distance of the combined code will be 4, and only those codewords of weight 4 that satisfy both the Hamming and CRC parity checks will be valid codewords for the combined code. This results in a lower rate of undetected errors, as will be demonstrated by the simulation results in the next section.

V. CONCLUSION
CRCs were originally designed for error detection, but by creating parity-check matrix representations of the CRC calculation, the message passing decoder can be used to correct errors. To reduce the effect of the relatively dense parity-check matrices that result from this process, sparsification techniques that add auxiliary variables to the code can be used to remove

short cycles. This process results in a trade-off between error detection and error correction, where adjusting the number of iterations in the decoder or the number of bit corrections can increase the number of undetected errors while reducing the overall number of errors. Reducing the number of iterations to just one results in no error correction, but a substantial reduction in undetected errors.
Undetected errors are also reduced when the technique is applied to error control protocols that combine a CRC with a linear block code. An example of this is the power-line communication system in [5]. In such cases, the parity-check matrices can be combined into a single virtual code. Based on the simulation results presented here for high-rate codes such as Hamming and BCH codes combined with CRCs, this appears a promising method of achieving 1-2 dB gain over traditional decoding. However, custom-designed LDPC codes of the same rate can produce even better performance. Furthermore, it is important to note that a message passing decoder requires a substantial increase in computational complexity relative to traditional algebraic decoders and shiftregister CRC implementations.
For these reasons, LDPC codes are still the preferred choice for communication systems that require high code rates and short message lengths. However, there are many existing communication systems that use CRC and algebraic block codes, which could benefit from the use of iterative decoding. While the computational cost of iterative decoding is higher, it can still be practical in some cases. An example of this is the case (such as the communication system in [5]) of a network configuration where a single device is used to communicate with a large number of other devices.
