ANALYSIS AND DESIGN OF MODERATE LENGTH REGULAR LDPC CODES WITH LOW ERROR FLOORS

Chad A. Cole Stephen G. Wilson

Eric. K. Hall Thomas R. Giallorenzi

Univ. of Virginia Charlottesville, VA 22904

L-3 Communications Salt Lake City, UT 84116

February 23, 2006

Abstract
The traditional method to estimate code performance in the higher SNR region is to use a sum of the contributions of the most dominant error events to the probability of error. If an ML decoder is used, these events will be minimum distance codewords; the traditional decoder used in LDPC codes, some variant of the message passing algorithm, will introduce non-codeword error events known as trapping sets. For long LDPC codes it is difficult to enumerate all of these dominant error events. A procedure to efficiently find dominant error events by using the regular low-density structure of an LDPC code is presented here. The search method can be adapted to work with LDPC codes of various regular and irregular degree distributions, but is especially suited to a very practical subset of LDPC known as regular {3, 6} codes of moderate block length. We also show how codes with very low error floors can be created by utilizing this search method.
Keywords: LDPC, error floors, trapping sets.
1 Introduction
The most difficult task of determining the error ŌĆśfloorŌĆÖ of a code (and decoder) in the presence of additive white Gaussian noise (AWGN), is locating the dominant error events that contribute most of the error probability at high SNR. Looked at from a Euclidean n-dimensional signal space viewpoint, this means finding the error events whose decision boundary is nearest the transmitted vector.
When using the message passing algorithm (MPA) decoder, these events could be a type of non-codeword error called trapping sets. Once these events are found, an importance sampling (IS) mean-shifting method [15, 12] can be used to efficiently determine the exact probability of error contributed by each error event.
For general codes, there is no known method, other than

a brute-force search, to find all of these events. We know of two published methods for finding dominant trapping sets, the first of which [10] uses a fast hardware simulator to decode noisy messages with the nominal noise density in the high SNR region and keep track of errors, most of which will be the dominate error events. The other method [3] requires the graph cycle structure to have a small number of shortest cycles; when the graph has many cycles of the shortest length, such as for codes designed to maximize the girth, the method does not work well. The method presented here relies on a deterministic noise pointed only in the direction of n-dimensional Euclidean space where a dominant trapping set or codeword could occur. For the case of LDPC codes, it is possible to limit the search space to a more practical size by examining the local Tanner graph [13] structure around each variable node.
Regular LDPC codes are especially amenable to parallel implementation [17]. {3, 6} regular codes, meaning 3 ŌĆś1ŌĆÖs per column and 6 ŌĆś1ŌĆÖs per row in the parity check matrix, are a very important class of regular codes because they require a very small number of computations at each node of the graph while offering a lower error floor than the slightly less dense {2, 4} ensemble of regular codes. The decoder search procedure developed here can be modified to work for a large class of low-density codes, both regular and irregular and of varying code rates, but it is especially well-suited to regular {3, 6} codes, so this paper will focus on this family of codes.
This paper is organized as follows: Section 2 explains the graph arguments used in the decoder search and gives a formal definition of trapping sets. Section 3 details the efficient dominant error event decoder search procedure. Section 4 gives some results and offers an application of the tool. The final section summarizes the contribution offered in this paper.

2 Local Graph Structure and Trapping Sets

LDPC codes have sparse parity check matrices, denoted by H, and can be conveniently represented by a Tanner graph [13], where each row of H is associated with a check node ci, each column of H is associated with a variable node vj, and each position where Hij = 1 defines an edge connecting vi to ci in the graph. If a subset of variable nodes and check nodes is considered, members of this subset will be called the active nodes of a subgraph. By studying the local subgraph structure around each variable node, arguments can be made about the global graph structure. For example, bounds can be placed on the minimum distance of a code by only considering the local structure of a code [13].
As with all linear codes, it is convenient to assume the all-zeros codeword as a reference vector for study. Small subsets of non-zero bits which do not form a valid codeword but still cause the MPA decoder problems are welldocumented in the literature and typically referred to as trapping sets (TS) [10]. A TS, x, is a length-n bit vector described by a pair (a, b), where a is the Hamming weight of the bit vector and b is the number of unsatisfied checks, i.e. the Hamming weight of the syndrome xHT. Two examples of TS, both extracted from a (96,48) code on MacKayŌĆÖs website [6], are shown in Figure 1. The shaded check nodes, #4 for the (5, 1) TS and #2 and #11 for the (4, 2) TS, are the unsatisfied check nodes. There are no 4-cycles in this code, but there are many 6-cycles within the TS bits of Figure 1. The essential intuition pointing to these as problematic for the decoder is that if a bits have sufficient noise to cause them to individually appear as soft 1ŌĆÖs, while the others in the subgraph are soft 0ŌĆÖs, the check nodes with which the a variable nodes connect will be satisfied, tending to reinforce the wrong state to the rest of the graph. Only the unsatisfied check(s) is a route through which messages come to reverse the apparent state.

If a true ML decoder were available, a simple technique can be utilized to find the minimum distance of a code [2]. The idea is to introduce a very unnatural noise, called an ŌĆśerror impulse,ŌĆÖ in a single bit position as input to the ML decoder. Unfortunately, the single-bit error impulse method cannot be used with the MPA to find dominant error events [5] mainly because the MPAŌĆÖs objective is to perform a bit ML decision rule and not the vector ML rule. So we consider multiple bits for error impulses.
The choice of the impulse bits is very important - they should point in the direction of the minimum distance codewords and TS. A good candidate set of impulse bits is given by S2. A multi-bit error impulse should appear as a more ŌĆśnaturalŌĆÖ noise to the MPA and the 4-bit combinations of S2 will be likely to get error impulses into multiple bits of a dominant trapping set or codeword. For example, suppose a minimum distance codeword has a ŌĆś1ŌĆÖ in its first four bits. If an error impulse ╬Ą were applied in all four of these positions, leaving the other n ŌłÆ 4 bits alone (i.e. the noise is ŌłÆ╬Ą[1, 1, 1, 1, 0, . . . , 0]), would the message passing algorithm decode to this minimum distance codeword? It cannot be guaranteed, but if the code block length is not too long, based on extensive empirical evidence, the decoder will decode to this dmin codeword for sufficiently large ╬Ą.

Proof. First notice that in a dominant TS, the TS variable nodes should not be connected to more than one unsatisfied check (USC). If a dv = 3 variable node were, then the 2 or 3 ŌĆśgoodŌĆÖ messages coming to that variable node should be enough to flip the bit, thus creating a variable node that is connected to 0 or 1 USCŌĆÖs. Thus, for a dominant TS, we can assume that all variable nodes in the TS connected to USCŌĆÖs are connected to only one USC. Now, take any variable node in the TS that is NOT connected to any USCŌĆÖs (there will be aŌłÆb of these) and use it as a root node to unroll the graph to the first layer of variable nodes and notice that one of these (dc ŌłÆ 1)dv possible combinations of 4-bits (i.e. an element of S2) will all be within the a TS bits.
4 Results and an Application
The (96, 48) code labeled 96.3.963 on MacKayŌĆÖs website has been analyzed extensively in the literature [16, 4, 3].

It has a known low-weight codeword spectrum, with 3 wH = 6 and 24 wH = 8 codewords. This is a regular {3, 6} code, so we immediately have some information about the codeword and trapping set structure [14]: all codewords have an even Hamming weight, and all (a, 1) trapping sets have an odd a. There are 74 (4,2) TS, but their contribution to the error performance is small compared to that of the valid codewords.
When employing the multi-bit impulse error search method, there are a number of parameters to specify - the error impulse magnitude, maximum number of decoder iterations, scale factor for the n ŌłÆ 4 bits where the error impulse is not applied and the decoder input SNR are four very important specifications. For this (96,48) example code, an error impulse ╬Ą = 3 is applied to each of the 4-bit combinations in S2. No random noise is added and the other n ŌłÆ 4 bits are not scaled. The maximum number of iterations is set to 30. As stated above there are (dc ŌłÆ 1)dv = 125 combinations for each of the n variable nodes as root. Thus a total of 125n = 12000 decodings is required. The SNR parameter (used by the MPA to scale the incoming channel data y) is set to Eb/N0 = 4 dB. All wH = 6 and wH = 8 codewords were found, as well as a number of wH = 10 codewords. Also, all 74 of the (4,2) TS were located. Only one minute on a modern AMD Athlon machine is needed for this search on this code.
The Margulis [8] (2640,1320) code was conjectured to have 1320 (12,4) and (14,4) trapping sets in [10] and this was verified by enumerating them all with the decoder search. There were also 1320 (9,5) TS discovered which will become the dominant error events at slightly higher SNR. The method has been applied to numerous {3, 6} codes with block length up to 5000 bits. Because it is difficult to verify whether the search has found all of the dominant error events, and there is not much on this subject in the literature for comparison, it is not possible to say with certainty that all codewords or trapping sets of a given type have been enumerated. However, one way to increase our confidence in this method is to simulate these codes with the nominal noise density in the high SNR region where the error floor appears and keep track of all error events and make sure that none occur which were not found by the decoder search. After many thousands of (equivalent) hours on a high-performance cluster computer, all of the empirical evidence gathered to date serves to further confirm that our decoder search does in fact find all dominant error events. For longer codes, the 4-bit error impulse is not sufficient to locate all dominant error events, and the tree must be grown one level deeper, which for a {3, 6} code would give sets of 10 bits to apply the error impulse to. Our decoder search can also be used as a tool in an iterative design process for designing LDPC codes with low error floors. For moderate code lengths, say up to a few thousand bits, code behavior varies greatly across an ensemble of codes. The ability to quickly locate dominant error events allows one to check whether a given code from an ensemble has a desirable error floor behavior much faster than the traditional means of evaluation such as a Monte Carlo simulation. One previous method proposed to quickly ascertain relative performance from an ensemble of codes without resorting to Monte Carlo simulation looked at the girth distribution of the code [7]. This search method is very quick but the metric it provides does not allow one to make any claims about the error floor of a code. Our decoder search procedure can similarly give a metric to gauge relative performance among codes without Monte Carlo simulation, and though it takes longer to compute than the girth distribution, it provides much more information about the error floor of the code.
Using our decoder search, it is simple to set up a procedure to randomly-construct parity check matrices of a given block length and rate and then quickly determine if the code satisfies a predetermined set of specifications and if it does not, then build a new parity check matrix and try again. The speed of our decoder search algorithm allows millions of parity check matrices to be constructed and searched in a reasonable period of time and if the code with the best error floor performance among the candidates is chosen, the code has the potential to offer orders of magnitude improvement in Pf at a given Eb/N0 in the error floor region. The method has been applied to {3, 6} codes of block lengths between 200 and 2000 bits, and the results show our codes have lower error floors than all other known codes with comparable parameters.
For example, Figure 4 shows error probability results for a new (204,102) rate 1/2 design, compared with a design posted on the MacKay website with the same parameters. Nearly two orders of magnitude reduction in error probability is found in the error floor region by using a code with a dominant (7,1) TS as opposed to the dominant (5,1) TS found in the MacKay code. The floor probability is calculated as the sum of error probabilities of the dominant trapping sets. The MacKay code has a single (5,1) dominating TS, while the new code has (7,1), (6,2), (9,1) and (8,2) dominating TS with multiplicity 4, 32, 7, and 41 respectively.

5 Conclusions
The work presented here is very helpful in the analysis of LDPC code error floors. The concept of using a multibit deterministic error impulse to enumerate dominant error events within a graph-restricted class is a novel approach to determining code performance at high SNR. The ability to accurately analyze and iteratively design a code with a very low error floor is useful for applications that must have a guaranteed very low error rate at a given SNR.
