A HIGH SPEED, LOW MEMORY FPGA BASED LDPC DECODER ARCHITECTURE FOR QUASI-CYCLIC LDPC CODES
Paul Saunders, Anthony D. Fagan
DSP Research Lab, School of Electrical, Electronic and Mechanical Engineering, University College Dublin, Dublin Ireland
email: paul.saunders@ee.ucd.ie, tony.fagan@ucd.ie

ABSTRACT
We propose a novel, high speed, low memory fully programable FPGA decoder architecture to decode quasi-cyclic LDPC codes. By performing optimizations at the code construction, algorithmic and architecture levels we are able to achieve significant throughput and memory storage advantages over current FPGA decoder implementations. Our decoder employs the modified turbo decoding algorithm, to achieve a decoding throughput of 223Mbps for a frame length of 3200 bits whilst only consuming 71Kb of memory using a Xilinx Virtex-4 architecture.
1. INTRODUCTION
Low-Density Parity-Check (LDPC) codes were introduced to the coding community by Gallager in 1962 [1] but it was not until 1995, when MacKay and Neal [2] rediscovered them, for their full potential to be understood. The code construction of most LDPC codes is typically random. which results in highly irregular hardware interconnectivity. This goes against efficient hardware implementation paradigms which require regular and structured design features. The class of quasi-cyclic LDPC codes targeted to the FPGA architecture in this paper, exhibit these design features. At the algorithmic level, the modified turbo decoding algorithm is used to address the memory overhead problem seen in current FPGA decoder implementations. Moreover, a parallel soft-input soft-output (SISO) message update mechanism, in the form of a message decoding unit (MDU), based on the Log-MAP algorithm [3] is used. Finally at the architecture level, a scalable architecture is implemented that can decode both regular and irregular LDPC codes of varying code rates and lengths each exhibiting the quasi-cyclic structure.
The remainder of this paper is organized as follows. We begin by discussing the theory behind LDPC codes and their graphical representation in Section II. In Section III we discuss the methods used to construct the quasi-cyclic codes that are targeted to the FPGA architecture in this paper. The modified turbo decoding algorithm which is employed in our FPGA architecture is discussed in Section IV. In Section V

we present all the modules that constitute our FPGA architecture. The error correction ability of our codes as well as the final architecture statistics are given in Section VI. Section VII provides some concluding remarks.
2. LOW-DENSITY PARITY-CHECK CODES
2.1. Tanner Graph Representation
An LDPC code is a linear block code defined by a m by n sparse parity check matrix H. For block codes a vector x is a codeword if and only if it satisfies: H·x = 0. The paritycheck matrix of a LDPC code can be represented graphically by a bipartite graph called a Tanner graph (TG) [4]. The graph has on one side m check nodes [c1, c2, . . . cm] corresponding to the m rows of H, and on the other, n bit nodes [b1, b2, . . . bn] corresponding to the n columns of H. A bit node bj is connected to a check node ci if the entry hij of H is 1. The rate of the code is at least 1 − m/n. In a regular (c, r)-LDPC code, bit nodes have degree c and check nodes have degree r in the graph. In an irregular (C, R)-LDPC code, the node degrees are drawn from the sets C and R respectively. Another important property of a TG is the girth or minimum cycle length of the graph. Fig.1 shows a length 9 rate 1/3 (2,3)-LDPC code with a girth of 4.

5.2. A High Level Description Of The Architecture
We chose to design the partly parallel architecture to decode the length 3200 rate 0.46 code. We also chose to use 5 bits of resolution in our design as this offered the best compromise between performance and logic utilization (See Section VI). The architecture is composed of six components namely: (1) A set of S MDUs for message computation. Each MDU decodes a row in block row i, i = 1, . . . , D. (2) A quad port memory module (γ memory) of size {B}-by-{S ×b}, where b is the number of resolution bits used. This memory stores the posterior LLRs of all code bits.(3) Two dual port memory modules (λleft and λright) of size {D× r/2 }-by-{S × b}. Where r is the maximum trellis size of the code. These memories store the updated reliability messages pertaining to all the constituent codes. (4) A single port memory module (H memory) of size {D × r/2 }-by-{addrL + addrR}. Where addrL and addrR are the number of bits needed to index γ memory. This memory stores the memory locations of the permutation sub-matrices in H. (5) A single port memory module (π memory) of size {D× r/2 }-by-{2log2 (S)}. This memory stores the shift values of each of the permutation sub-matrices in H. These values are used by the cyclic interleavers and deinterleavers.(6) A bi-directional communication network containing two cyclic interleavers and two cyclic deinterleavers used for routing the messages between γ memory and the MDUs. 5.3. The Message Decoding Units
The structure of a MDU is shown in Fig.6. Each MDU contains 4 inputs and 4 outputs. The 4 inputs are the left and right LLRs read from the γ and λ memories. The 4 outputs are the left and right updated LLRs to be written back to the γ and λ memories. The reason for providing both left and right inputs during a single clock cycle is that the α and β recursions performed by the MDUs are independent and can be calculated simultaneously. Therefore data corresponding to the left permutation sub-matrix of a block row, appears on the left input ports. Similarly, the data corresponding to the right permutation sub-matrix of a block row, appears on the right input ports. The data appearing on the left input port moves from left to right along the trellis. Similarly, the data appearing on the right input port moves from right to left along the trellis. The point in the trellis where the α and β recursions meet is known as the crossover point. The corresponding updated LLRs for each left and right permutation sub-matrix appears on the left and right output ports. Last In First Out (LIFO) buffers are provided to store the intermediate α and β values before crossover occurs. LIFO buffers are also provided to store the left and right ρ values which are summed with the updated Λ LLRs to produce the updated Γ LLRs.

5.4. The Memory Modules
The γ memory is a quad-port memory module containing 100 storage locations, each 160 bits wide. Two ports are dedicated to reading the left and right LLRs and two ports are dedicated to writing the updated left and right LLRs. The memory is configured as a dual-port memory, but is clocked at twice the rate of the surrounding logic. The reason that a quad-port memory is implemented is to increase the throughput of the decoder. A dual-port memory prevents simultaneous reads and writes of two values (γL and γR) being performed, thus lowering data throughput. Each λ memory is a dual port memory module containing 162 storage locations each 160 bits wide. One port is configured for reading, the other for writing. The H and π memories are combined into a single port read only memory (ROM).

There are also three control bits that are appended to each storage location. These control bits give the trellis size of the block row being decoded. They are used to control the timing of the MDUs. This results in a memory module containing 162 storage locations each 27 bits wide. The total memory utilization of our decoder is 71Kb or 72214 bits.

5.5. The Bi-Directional Transport Network
The transport network contains two 160 bit wide data paths in the forward and return directions. The two cyclic interleavers in the forward direction, cyclicly shift the data on their inputs to the left by the value stored in π memory. The two cyclic deinterleavers in the return direction, cyclicly shift the data on their inputs to the right by the value stored in π memory. Traditionally shifting a single data bit one field at a time can be a slow process. However, a well known logic circuit known as a Barrel Shifter is able to shift or rotate a data word by any number of bits in a single clock cycle thus increasing the decoding throughput of the decoder. It achieves this by shifting data in incremental stages which avoids extra clocks to the registers and reduces time spent rotating/shifting data. Our interleavers and deinterleavers are implemented as Barrel Shifters.

6.3. Comparison with other FPGA architectures
In this section we attempt to compare our FPGA architecture with other FPGA architectures presented in literature on the basis of decoding throughput, memory utilization and logic size. The results presented in Table 2 assume the targeting of a Xilinx Virtex-4 device. This is achieved by scaling the decoding throughput of the various architectures by a factor, σ, which is determined by taking our design and targeting it to a Virtex-E, Virtex-II and finally a Virtex-4 device. The performance gained by taking our design from a Virtex-E device to a Virtex-4 device is given by σ = 3 and by taking our design from a Virtex-II device to a Virtex-4 device is given by σ = 1.5. This approach is considered reasonable as the Virtex-4 data sheet [8] states that a Virtex-4 device can offer up to a 40% speed improvement over previous generation devices whilst our scaling factors correspond to a 200% improvement and a 50% improvement. Secondly Virtex-II and Virtex-4 devices contain 18Kb BRAMs whilst VirtexE devices contain 4Kb BRAMs thus all Virtex-E BRAM statistics are divided by a factor of 4.5.
It appears from Table 2 that [9] followed by [10] offer the highest decoding throughput. However on closer examination, both of these architectures decode very high rate codes, 8/9 and 0.88 respectively. Since our architecture’s timing cycle is dependant on the number of block rows D, (see Eq.11), a higher rate code would result in fewer block rows and therefore a higher decoding throughput. The architecture presented in [11] provides an excellent comparison to ours as it is clocked at a similar rate. 7. CONCLUSION
Our architecture achieves a maximum decoding throughput of 223Mbps at 180Mhz for a rate 0.46, length 3200 quasicyclic LDPC code whilst only consuming 71Kb of memory. This high decoding throughput and low memory utilization is achieved by implementing various high speed optimizations such as the inclusion of quad port memories and Barrel Shifters as well as employing the modified turbo decoding algorithm. Our FPGA decoder is superior both in terms of data throughput and memory utilization when compared to other FPGA architectures presented in literature.
