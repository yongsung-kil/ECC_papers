Concatenating a Structured LDPC Code and An RLL Code to Preserve Soft-Decoding, Structure, and Burst Correction.
Y. Han1 and W. E. Ryan1 1. Electrical and Computer Engineering, University of Arizona, Tucson, AZ, USA
1.Overview Error-control codes and runlength limited codes each provide critical roles in maintain data integrity on storage devices,with ECCs doing directly by inserting redundancy via some encoding algorithm,and RLL codes doing indirectly by facilitating timing recovery.Their simultaneous presence in a storage system introduces conflicts in the receiver design when soft ECC decoding is a goal.On one hand,if RLL encoding were to follow ECC encoding,soft ECC decoding would be precluded at the receiver since for complexity reasons the RLL decoder must produce hard decisions.On the other hand,if RLL encoding were to precede ECC encoding,the RLL constraint would be lost with the addition of ECC parity bits unless provisions were made to accommodate. Several recent papers have addressed this dilemma when a LDPC code is used for the ECC code[1]-[3].[1]considers several such techniques,including a parity-insertion technique which is used in[4][5].We extend the idea as below.References[2][3] consider“deliberate-error”techniques whereby selected LDPC code bits are inverted in order to ensure RLL constraint compliance. In this paper,we add another two requirements to the RLL-LDPC concatenation.First,we require the LDPC code possess structure to facilitate encoder and decoder.Second,the nominal H matrix of the LDPC code must be modified to accommodate both the RLL constraint and the ability to correct long bursts[6].One way to accomplish this is via the deliberate-error[2][3],but we believe these techniques result in substantial performance losses,especially in the floor region.We propose a simple reverse-concatenation scheme combined with a structured permutation of H to satisfy additional requirements,at the expense of essentially no complexity increase and only the RLL code rate loss. 2.The Reverse Concatenation The components of our design are a rate-50/51 RLL code with a (0,14) constraint[7] and a rate0.911(4592,4182) SeIRA LDPC code[8],with a combined rate of 0.89.The nominal H matrix for the SeIRA code has the form H=[H1 H2],where H1 consists of an array of square circulant permutation matrices and H2 is an mxm“dual-diagonal”matrix(m is the no. of parities). A block of 4100 bits,comprising 4096 sector data bits plus four“dummy”bits,is RLL encoded to produce 4182 LDPC encoder input bits,which yields a 4592-bit LDPC codeword.The 410 parity bits can be inserted among the 4182 systematic bits which relaxes the RLL constraint to (0,16).This parityinsertion can be realized with no complexity increase by inserting the parity columns of H among its systematic columns,from which an alternative H matrix is obtained for encoding and decoding. Such a modification of H is not sufficient to obtain a code with a large value of Lmax,where Lmax is the guaranteed burst-erasure length correction capability corresponding to the code’s H matrix.When H possesses the SeIRA structure,the Lmax is only 161,an efficiency of 161/410=0.39.We may modify H as H*=[H1Πa H2]Πb to simultaneously obtain Lmax near the maximum possible for this code and accommodate RLL constraint.In this expression,Πa and Πb are deterministic permutations of the columns of H.Since Πa and Πb are structured,H* is struc-

tured.H* corresponds to a code with Lmax=260 (efficiency 0.63) compared to a maximum of 272(random permutations).The RLL constraints corresponding to H* and the above RLL code is (0,16). 3.Performance Results The performance of our scheme on a AWGN channel is shown in Fig.1.We see the performance gap between the LDPC code alone and the reverse concatenation is very small,and can be largely attributed to RLL rate loss:10log10(51/50)=0.086dB.Fig.1 can be compared to Fig.5 in[2] where the code length(4096),rate(0.83),RLL constraint [(0,15)] and performance are comparable.However,our scheme does not add errors to LDPC codewords,we expect our floor to be lower.(Our simulation at 5.25dB is unreliable to date,but indicates no floor.)
