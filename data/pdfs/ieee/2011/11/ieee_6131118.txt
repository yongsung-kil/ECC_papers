Flexible LDPC Decoder using Stream Data Processing for 802.11n and 802.16e

Honey Durga Tiwari, Non-Member, Huynh Ngoc Bao, Non-Member, and Yong Beom Cho, Member, IEEE

Abstract — Wireless data transmission standards like 802.16e, 802.11n, employ Low Density parity Check (LDPC) codes for error control coding. The bit flipping decoding algorithms presents a tradeoff between the error correcting capability, decoding resources and the decoding time. Software based LDPC decoders provide adaptation capabilities in system parameters such as block size and code rate. In a real-time, low-power mobile environments, the Single-Instruction Multiple-Data (SIMD) processor currently used for video processing, could also be used for the LDPC decoding. In this paper, the implementationefficient, reliability ratio-based, weighted bit flipping (IRRWBF) algorithm is presented using a flexible softwarebased LDPC decoder. Compact data structures are proposed for performing the decoding using SIMD architecture. Based on the implementation on two commonly used SIMD architecture for mobile platform, it was found that the decoding speed can be increased by more than 2000% (using 64 bit SIMD registers with vector integer calculation) and 1800% (using 128 bit SIMD registers with vector floating point calculation). Experimental results for different code lengths of 802.16e and 802.11n show that decoding time in order of 1×10-3~10×10-3 seconds is achievable. Due to significantly high throughput and flexibility, the proposed design algorithm and data structure can easily be adapted to any energy-sensitive mobile devices employing SIMD processors1.
Index Terms — LDPC, IRRWBF decoding, SIMD, Parallel processing.
I. INTRODUCTION
The near Shannon limit performance of Low Density Parity Check (LDPC) codes [1] decoding algorithms like sumproduct (SP) [2], min-sum, and modified min-sum [3], has helped LDPC codes to be adopted in various digital communication standards such as DVB-S2 standard, WiMAX (802.16e), Wifi (802.11n), 10Gbit Ethernet (802.3an) and others. However, due to the logarithmic and exponential functions involved in their decoding algorithms, LDPC decoders are often criticized for their large memory requirement and long convergence time. Various VLSI hardware architectures have been proposed for real time processing of Belief Propagation (BP) based decoding

methods. However, fully parallel architectures usually result in excessive implementation cost, while serial architectures are too slow for most applications. Apart from memory related requirements, flexibility with respect to code rate and length is also important. Due to these design and implementation problems, LDPC is still considered as optional feature in wireless standards for mobile network like 802.11n and WiBro.
The bit-flipping based decoding methods [4] [5], mainly Implementation-Efficient Reliability Ratio-Based Weighted Bit-Flipping (IRRWBF) algorithm [6] can be considered as an efficient trade-off between error-correcting performance and decoding complexity. Reduced number of computations and absence of complex mathematical functions like logarithm and exponential calculations in IRRWBF reduces the memory requirement as compared to BP based decoding algorithms. However, sequential bit-flipping strategy often results in slow convergence.
With the advancements in VLSI design methodologies, various code length and rate flexible LDPC decoders have been proposed [7] [8]. However, flexibility in such designs is limited to a set of pre-defined codes, and it tends to be inadequate when the support to generally defined codes is required. To overcome the design problems related to parameter adaptability, software LDPC decoders using Digital Signal Processors or Software Defined Radio (SDR) programmable hardware platforms [9] have been proposed. The performance of such software decoders is measured by their decoding speed.
In order to achieve smaller decoding time, the inherent algorithmic parallelism of standard WBF must be considered. This data level parallelism can then be utilized for representing the data in vector form and performing the parallel calculation using Single Instruction Multiple Data (SIMD) processors [10]. In [11], [12] software LDPC decoder implementation based on graphics processing units (GPUs) and multi core architecture has been presented. However, due to their extensive resource utilization, these implementations can be considered for non-mobile platforms.
In this paper, we present the implementation of software IRRWBF LDPC decoder using stream based processing in mobile environment. Firstly, the approach for designing parallel LDPC decoder using SIMD technology is presented. The resulting architecture is tailored to decode both IEEE 802.11n and IEEE 802.16e LDPC codes, as well as any other code of comparable complexity. II. LDPC CODES SPECIFICATION IN WIRELESS COMMUNICATION STANDARDS
A. WiBro or IEEE 802.16e
WiBro is the first mobile WiMax based commercial service based on IEEE 802.16e standard and provides the specification for high-speed portable Internet service. IEEE 802.16e has adopted structured LDPC as an option of channel coding scheme [13]. Each LDPC code in the set of LDPC codes is defined by a matrix H of size M-by-N, where N is the length of the code and M is the number of parity check bits in the code. The number of systematic bits is K = N - M. In IEEE 802.16e matrix H is defined as

The matrix H is expanded from a binary base matrix Hb of size mb-by-nb, where N = a × nb and M = z × mb, with a as an integer ≥ 1. The base matrix is expanded by replacing each 1 in the base matrix with a a-by-a permutation matrix, and each 0 with a a-by-a zero matrix. The base matrix size nb is an integer equal to 24. The LDPC code flexibly supports different block sizes for each code rate through the use of an expansion factor. Each base model matrix has nb = 24 columns, and the expansion factor (a factor) is equal to N/24 for code length N. In each case, the number of information bits is equal to the code rate times the coded length N. The code length in case of IEEE 802.16e ranges from 576 to 2304, while the code rate specified in the standard are 1/2, 2/3, 3/4 and 5/6.

B. IEEE 802.11n
IEEE 802.11n-2009 is an amendment to the IEEE 802.112007 wireless networking standard to improve network throughput over the two previous standards - 802.11a and 802.11g [14]. The specification of LDPC codes in IEEE 802.11n is same as that in IEEE 802.16e with the change in the permutation matrices for different code rates and code length. In total, 12 LDPC codes are defined in IEEE 802.11n using a combination of different code length (648, 1296, 1944) and code rate (1/2, 2/3, 3/4, 5/6).

IV. PARALLELISM AND USE OF SIMD EXTENSION IN IRRWBF
SIMD technology is a way of packing D (usually a power of 2) like operations (e.g. 8 adds) into a single instruction. The data for the instruction operands is packed into registers capable of holding the extra data. The advantage of this format is that for the cost of doing a single instruction, D instructions worth of work are performed. This can translate into very large speedups for parallelizable algorithms. To effectively use SIMD-architecture-based decoding, effective memory access is essential. To achieve parallel computation, we use the stream-based data structure proposed in [11] for storing the parity check matrix H. indices of the non-zero elements of H for each row. Similarly,

another data structure, Hc, was obtained by row-wise packing

of the indices of the non-zero elements of H for each column.

V. EXPERIMENTAL ENVIRONMENT
To evaluate the performance of the proposed parallel stream based IRRWBF decoding, LDPC decoders were programmed on SIMD Processor with: (i) 64 bit vector integer calculation capability working approximately at 800 MHz with 8 MB of L2 cache memory; (ii) 128 bit vector floating point calculation capability working approximately at 800 MHz with 8 MB of L2 cache memory.
A. SIMD Processor with 64 bit vector integer calculation
Many SIMD processors provide stream based instructions only for integer type data. In such a case all the floating point calculations are approximated to their nearest integer. A direct approximation often results in calculation error. This may increase the number of iterations required for successful LDPC decoding or even lead to non-convergence. Hence, the

IEEE Transactions on Consumer Electronics, Vol. 57, No. 4, November 2011
floating point data must first be scaled and then approximated to the nearest integer. The details of such approximations are beyond the scope of this paper. The time incurred in the scaling process is considered as a part of decoding process and is included in processing time discussed in next section. The SIMD processors that provides 64-bit-wide SIMD instructions, often allows concurrent processing of up to eight data elements (8 Bit each) in a single cycle.
B. SIMD Processor with 128 bit vector floating point calculation
In case of IRRWBF decoding, only the variable node step needs vector floating point calculation. For large i, conversion floating point to integer conversion time can be saved by using vector floating point calculations. The SIMD processors that provides 128-bit-wide SIMD instructions, often allows concurrent processing of up to sixteen data elements (8 Bit each) in a single cycle.

VI. IMPLEMENTATION RESULTS
In Section III, it was shown that the decoding time in IRRWBF is a function of N, i and Wc. In [6], it was concluded that the IRRWBF decoding spends a considerable percentage of time in variable node step. The higher the iteration number, the more processing time is spent on the iteration part, i.e. check node step and variable node step.
In this paper, we define LDPC codes by (N, cr) specified in 802.11n and 802.16e standard. Amongst the 12 and 19 LDPC codes defined in 802.11n and 802.16e, respectively, the decoding time for various code lengths and code rate were calculated. Firstly, to study the effect of code length, parallel implementation of IRRWBF is analyzed using different code lengths for a given code rate of 1/2. To study the dependence of decoding time on the number of iterations, parallel implementation of IRRWBF is analyzed using the maximum code length and code rate for different decoding iterations. VII. COMPARISON WITH EXISTING SYSTEMS
A. Parallel Weighted Bit Flipping
Parallel WBF (PWBF) decoding was introduced in standard WBF by flipping multiple bits [16]-[18]. This was done to reduce the large time and computational resources consumption of standard BF algorithms. The PWBF decoding is based on calculation of a threshold value which defines the number of bits to be flipped. This however requires extra computation to calculate “flip signals” for each unsatisfied check node and a threshold to determine the number of bits flipped in any iteration. Although, the number of decoding iterations in PWBF is significantly reduced, the serial WBF outperforms PWBF in terms of bit error rate (BER) performance. The target of the paper is to reduce the decoding time of the WBF coding without sacrificing the BER performance. Hence, we used the data level parallelism in serial IRRWBF scheme to implement fast decoder.
B. BP and standard WBF based decoders
The sum-product algorithm for LDPC decoding has better BER performance than the IRRWBF LDPC decoding algorithm; however, it requires more decoding resources and takes longer decoding time. BER performance of the IRRWBF LDPC decoder was established previously in [4]-[6]. The algorithm implemented in [11] uses SIMD processing for sum-

C. Memory requirement
In the proposed scheme, the processing core is shared by other operations, and therefore, we consider the resource essential for LDPC decoding exclusively. Since, all the calculations are done by the processor, the memory requirement for decoding is the only resource used exclusively for the decoding. As compared to the BP decoding algorithm, the check node variable is a single bit data. Hence the amount of memory for variables can be given as
 The memory for initialization variable: li×M (=li×N(1-cr))  The memory for hard decision bits: N  The memory for variable node message: lv×N  The memory for check node message: N  Totally required: N [li(1-cr) + lv + 2]
Here, li and lv is the word length of the Initialization node and Variable node variables respectively.
In case of 802.16e the maximum code length is 2304, while the smallest code rate is 5/6. Let the word length of the Initialization node and Variable node variables be 8. Hence, the maximum processing memory required for 802.16e is approximately equal to 3264 Bytes. Similarly, in case of 802.11n the maximum code length is 1944, while the smallest code rate is 5/6. Hence, the maximum processing memory required for 802.11n is equal to 2754 Bytes.
In partial-min algorithm [19] the total amount of memory required is given by (Li +1) × N + Le× N + Σj[(Le − 1) × (km +1) + |V (j) |] where, the meaning of various terms are Li and Le are the word lengths of the intrinsic and the extrinsic information respectively. The number of check nodes is given by j. Each check node has km + 1 different LLR magnitudes. Each check node produces |V(j)| sign bits [19]. As an approximation, if we consider that each check node produces |V(j)|av sign bits, the total amount of memory required can be approximated by (Li+1)× N + Le× N + M × [(Le - 1) × (km+1) + |V(j)|av]. As an example, assume that the word lengths of the intrinsic and extrinsic information are 6 and 8 respectively and km=4 as assumed in [19] and let |V(j)|av ≈ 20. In such a case, total amount of memory required is 6960 Bytes for 802.16e and is approximately equal to 5873 Bytes for 802.11n.
In both the cases presented above, the maximum memory requirement calculation does not include the memory required for storing the parity check matrix. In 802.16e and 802.11n based systems, the parity check matrix is represented in form of permutation matrix shift patterns. The memory requirement for storing these patterns is very small as compared to the processing memory. From the above discussion, it can be seen that the memory required for processing is reduced by more than 50 percent without the sacrifice of flexibility in terms of code rate and length.
