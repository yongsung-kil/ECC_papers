LDPC codes can be represented by a Tanner graph which is a bipartite graph where the sides represent check nodes and bit nodes respectively. Thus, check nodes correspond to the rows of the H-matrix and bit nodes correspond to the columns of the H-matrix. For example, (i,j) element of H matrix is вҖҳ1вҖҷ , the ith check node is connected to the jth bit node of the equivalent Tanner graph. B. Overlapped Message Passing and Matrix Scheduling of LDPC decoding
As explained in subsection II.A, LDPC decoding is carried out using a message passing algorithm. Therefore, as the length of the code increases, both the amount of data to exchange and the time for message passing increase significantly. In a recently proposed standard for CMMB LDPC codes, the code length is 9216. To meet the performance requirement of the CMMB standard, either a fully-parallel or a highly efficient partially-parallel hardware implementation should be employed[9, 10].
To solve the long decoding latency problem, OMP methods have been proposed. In OMP methods, check-to-variable (CTV) and variable-to-check (VTC) operations are processed in parallel as long as these two do not have any conflicts in memory access or data dependencies. To avoid conflicts in memory accesses as much as possible, a method to reconstruct the H-matrix is proposed [14], and other studies have also tried to parallelize CTV and VTC operations [11-13].
First, let us explain how a typical OMP method with the Hmatrix reconstruction is processed. Figure 3 shows a (10,5) Hmatrix. In a conventional LDPC decoding without an OMP, variable node operations for the ten columns are carried out after finishing check node operations for the five rows. III. PROPOSED LDPC DECODER The main challenges in implementing a CMMB LDPC decoder are to reduce the size of the memory requirement and to speed up the iterative decoding process so that the throughput requirement by the standard should be satisfied. The proposed decoder meets the challenges using three different techniques. First, the H-matrix for the CMMB is reconstructed in such a way that the OMP algorithm should achieve the best performance gain while minimizing the memory requirement. Second, during the time when only the VTC operations are processed, functional units for the CTV operations are shared so that they can be used for the VTC operations for shorter completion through increased parallelism. Third, to maximize the performance gain, an efficient method for memory grouping and access schedule is applied so that both CTV and VTC operations can be processed in parallel without any significant access conflicts. These three techniques will be explained in detail in the following sub-sections.
A. H Matrix Permutation for Efficient Memory Access
As explained in sub-section II.C, the CMMB H-matrix for the rate 1/2 code can be generated from an 18x6 sub-matrix using (10). And that for the rate 3/4 code can be generated from a 9x12 submatrix by (11). Figure 8 and 10 show the positions of 1вҖҷs in the Hmatrix for the two code rates, respectively. To maximize the effectiveness of the OMP algorithm, the H-matrix should be reconstructed. An excellent H-matrix reconstruction algorithm is presented in [14]. However, we cannot use the same algorithm because for the H-matrix for CMMB, only the small sub-matrix should be stored in the memory and the rest is generated by the AGU to reduce the memory requirement. Thus, we find the permutation of the H-matrix which is compatible with the CMMB H-matrix generation method.

C. An Efficient Partially Parallel Architecture
The proposed CMMB LDPC decoder has a partially parallel architecture to meet both the hardware size requirement and the throughput requirement. The architecture is shown in Figure 17. To store LLR values, the memory is divided into 8 memory groups. A set of functional units consists of 8 operators, and each set is allocated to VTC operations and CTV operations, separately. As aforementioned, the set for CTV operations can also be used for VTC operations for functional unit sharing. The LLR values are sequentially stored in the memory in the column-major order of the permuted matrix. Especially, to make the 8 VTC operators perform the OMP algorithm optimally, the order for the memory store is determined as shown in Figure 18. Each field in the memory slot has two numbers. The first number is the row position in the H-matrix, and the second number indicates that it is the nth number of the row. The LDPC codes for CMMB are regular, and each row has 3 LLR values. So 0,1 and 2 will be used to denote the order.
When we store the data in this order, we can simply read the data sequentially without having to get help from an AGU. Also this method will give you an additional benefit that by reading the least significant 3 bits of addresses generated by AGU, we can easily tell which memory group contains the data that we access. However, for CTV operations, we have to use the addresses generated by the AGU to access data in each memory group. Therefore, we need a controller in Figure 19. Suppose we want to find the real address for (1080, 0). The actual address generated by the AGU is (0, 0), which denotes that the data is in memory address 0 in memory group 0. Thus, 1080 should be replaced for 0. However, the substitution operation can happen concurrently with the time when the AGU computes the real address using (10) and (11). Therefore, it can be processed without any additional delay. Since the memory group number corresponds to the last three bits of the address, using the least significant 3 bits, memory selector selects the memory group that contains the data.

IV. IMPLEMENTATION RESULTS
The proposed LDPC decoder for the CMMB standard has a partially parallel architecture. The CMMB standard has the code length of 9216 bits and supports two code rates: (1) (9216, 4608) regular LDPC codes support rate 1/2 codes and (2) (9216, 2304) regular LDPC codes support rate 3/4 codes. The proposed architecture supports both code rates, and is optimized for both the circuit area and the throughput when the OMP algorithm is applied.
Table 2 shows the results when the CMMB LDPC decoder is design as a partially parallel structure with 8 sets of functional units. We compare the performance before and after we apply the set of proposed techniques. When the proposed techniques are applied, the area overhead is 5k (in NAND2), but the throughput is improved by more than 40%. If we analyze the results closely, in the proposed design, VTC operations are executed in 9216 times, and CTV operations are executed 2304 times, and the total number of operations will be 11,520. On the other hand, when the OMP algorithm is used, VTC and CTV operations are executed in parallel for 1260 times. Hence, about 10% throughput enhancement is achieved. We may compute the through improvement numerically by simply counting the number of operation cycles. The total number of VTC operation cycles is 7956. When we share functional units to parallelize the VTC processing, the number of operation cycles to execute VTC becomes 3978, which is 45.5% of throughput enhancement. Though, in reality, due to some memory access overhead, we could achieve a 43% improvement. The area information is obtained from the synthesized design by Synopsys Design Compiler using Chartered 0.18Ојm CMOS cell library. The total size of the memory requirement is (9216 пӮҙ 3 пӮҙ 6) bits to store LLR values, and 1728 bits to store a sub-matrix of the Hmatrix which will be used to generate the entire H-matrix by the AGU, and the total will be 163k bits.
The CMMB standard requires that the minimum throughput should be 10.852Mbps for the rate 1/2 code, and 16.243Mbps for the rate 3/4 code. Table 2 shows that the throughput for the rate 1/2 code is 17.37Mbps, and that for the rate 3/4 code is 31.34Mbps. Both meet the throughput requirements specified by the CMMB standard.
Table 3 compares the size of the proposed design with other existing designs. In case of [21], the gate count information is not provided in the paper. Thus, we cannot compare. However, when we compare our design with the one in [20], the proposed design is smaller, and yet the throughput is better. Especially, the proposed decoder has a merit that its memory requirement is very small. Also, it meets the throughput requirement by the CMMB standard merely at 185MHz. The operating clock speed of the design reported in [21] is 433MHz. Running at a lower clock speed will be beneficial to reduce power consumption as long as the throughput requirement is satisfied.

V. CONCLUSION
In this paper, we propose a novel LDPC decoder for the CMMB standard. We present optimization methods not only to reduce the memory usage, to reduce the circuit size, but also to improve the throughput. To improve the throughput, the OMP algorithm, the VTC throughput enhancement through functional unit sharing and an efficient memory grouping technique have been employed. Our proposed decoder shows better than the one recently published in the sense that our design uses smaller amount of memory while it meets the performance requirement suggested by the CMMB standard at a lower operating clock speed.
