Design of GPU-Based Platform for LDPC Decoder
Cheng-Chun Chang, Min-Yu Huang and Yang-Lang Chang*

Abstract—The needs for reliable and flexible downlink communications incorporated with high volume of satellite images in the ground station have inspired the demands for high performance and flexibility computing in the field of Earth remote sensing. Low-density parity-check (LDPC) codes have had a strong impact on achieving reliable communication links. Finding a fast and reconfigurable developing platform for designing high throughput LDPC decoders has become important. In this paper, a graphic processing unit (GPU) platform is proposed to realize this practical implementation. Experimental results show that the proposed GPU-based platform is compatible to serve as a high-throughput LDPC decoder.
Keywords— Graphic processing unit (GPU), Low-density parity-check (LDPC)

I. INTRODUCTION
Low-density parity-check (LDPC) code is a linear block code known to approach the Shannon limit via the sum-product iterative decoding algorithm. LDPC codes have been adopted in most recent communication systems such as DVB-S2, WiMAX, WI-FI and 10GBASE-T. The demands of LDPC decoders for reliable and flexible communication links in a wide variety of communication standards and configurations have inspired the development of high-performance and flexibility computing. Accordingly, finding a fast and reconfigurable developing platform for designing the high-throughput LDPC decoder has become important, especially for rapidly changing communication standards and configurations. In this paper, a new graphic-processing-unit (GPU) LDPC decoding platform is proposed to realize this practical implementation.
For practical consideration of computational complexity and error-correction performance, LDPC decoders are implemented with iterative message-passing (MP) decoding algorithms. In the past, various structures of FPGA implementation of LDPC decoders have been reported [1-4]. GPU, also called general purpose GPU (GPGPU), is an attractive alternative in place where large scale and high performance computing is required. GPU consists of hundreds of cores operating at a lower clock rate, compared to conventional central processing units (CPUs). Although CPU can handle more complicated computation, GPU can make it up with many cores by taking advantages of massively parallel processing [6]. In recent years NVIDIA has provided Compute Unified Device Architecture (CUDA) as an efficient parallel programming language for GPU implementation. A GPU-based LDPC decoding implementation [7] was recently introduced. It has a throughput of 109.1 Mbps for 20,000-bit codeword length. It uses 8-bit precision to cut the computational time on the GPU and has a lower bit error rate (BER) as compared to the VLSI-based LDPC decoders with 5 or 6-bit precision.
In our previous work [8], a novel implementation with 32-bit single-precision floating-point architecture of asynchronous data transfer for LDPC decoding was proposed to accelerate the speed-up on GPUs. II.BASICS OF THE LOG-SPA DECODING ALGORITHM

The notion of Tanner graph is often used to represent LDPC codes. As shown in Fig. 1, a Tanner graph is a bipartite graph with bit nodes (BNs) on one side and check nodes (CNs) on the other side. For example, a Tanner graph for a regular LDPC H matrix of M rows (related to CNs) by N columns (related to BNs) is shown in Fig. 1. In a decoding procedure, BNs are associated to a codeword while CNs are used to check the integrity of the codeword. In the decoding process, messages are exchanged along the graph edges, as illustrated in the bottom figures of Fig. 1. If an LDPC code is well designed, the messages will converge to the correct bits after a finite number of iterations.

III. GPU-BASED LDPC DECODERS
For flexible implementations of LDPC decoders among different code lengths, code rates, and code parameters (e.g., dc and dv ), we developed in this paper a parallel LDPC decoder to run on GPU platforms. First, the encoded bit stream transmitted through communication channels are transferred from the CPU host to the GPU device. The iterative procedures between CNs and BNs in GPU are next performed. The decoded bit stream is then determined by hard decision. Finally, the decoded bit stream is transferred from the GPU device back to the CPU host.
For GPU implementation, we note that a block is needed to be set as a fundamental computing unit for the operation of GPU. The number of threads assigned to a block should be carefully selected in order to obtain efficient parallel operation and maximize the speedup. We also note that all the BNs associated with a CN have to be assigned in the same block because different blocks may terminate the computations at different time. Otherwise, it will result in troubles for data accessing between different blocks due to lack of synchronization. Consequently it is important to find an appropriate number of threads within an individual block to optimize the performance of GPU-based implementation [8].

It should be noted that the shared memory in a block with threads has lower communication overhead compared to the global memory used in the devices of the GPU. Thus, the usage of the shared memory implemented in the proposed method is applied to the data-intensive computational tasks of the CN (horizontal procedure) and BN (vertical procedure) computations. The data with highly dependency, e.g., the elements of L(vmi n ) and L(qmi n ) in BNs and CNs respectively,
are assigned into the same block of the shared memory with a finer granularity of parallel implementation.
An improved memory-usage mechanism for effectively buffering data used in our proposed GPU-based LDPC decoding during a GPU kernel execution cycle is illustrated in Fig. 4. Each thread, which represents L(vmi n ) or L(qmi n ) each
element in different blocks, is data independent to each other. These threads are grouped in warps and dispatched by blocks in one of the multiprocessors of GPU according to the principal of data dependency. In the horizontal procedure, each thread updates all the BNs associated with the same CN according to the Tanner graph, as shown in Fig. 1, and vice versa for vertical procedure.
In Step 1, the codewords and the H matrix are transferred from host to the device (global) memory and the codewords are then translated into the corresponding initial LLRs. In order to avoid the slow accesses to global memory, a proper alignment of the global memory is needed before codewords are transferred into corresponding threads/blocks of the shared memory according to the H matrix by using the coalesced memory accesses [10]. computed by Eq. 2 and then is transferred into its corresponding address of global memory based on the index of block/thread assignments of H matrix as shown in Table 1. Note that these corresponding addresses of global memory also need to follow the rules of the coalesced memory accesses suggested by CUDA [10]. In Step 3, a similar method is applied to the update of L(qmi n ) by Eq. 3 of vertical procedure. Here, each thread
updates the CNs associated with a BN of a column of H matrix as shown in Table 1. In Step 4, the posterior LLRs are updated iteratively. If the iteration stops, the hard decision is got. If the loop is not finished, the BNs and CNs decoding are sequentially operated in Step 2. In Step 5, the decoded codewords are stored in global memory.

To boost the data throughput of LDPC decoding, a method of asynchronous data transfer (ADT) was implemented in the proposed method [8, 11]. The main advantage of ADT is that GPU can still execute the kernel while transferring data between the CPU host and the GPU device and waiting for CPU acknowledgement.
The speedups of the proposed GPU-based LDPC over its CPU-based single-threaded version for various H matrix sizes and numbers of iterations are also shown in Fig. 3. The larger the H matrix size, the higher the obtained speedup. For a matrix size of 4,000 by 8,000 (matrix C) with 10 iterations, the execution time of LDPC decoding on the GPU is 1.808 ms whereas its original CPU-based double-precision LDPC decoder takes 490.504 ms. This corresponds to 271x speedup.
Interesting to note, when the size of the H matrix increases, the required decoding time also increases. However, more speedups can be achieved when the GPU-based LDPC decoding is applied to a larger H matrix. Hence, the GPU approach can also compensate the increasing demands of the LDPC decoding on longer codeword length.
V. CONCLUSION
LDPC decoder can use the iterative message-passing algorithms to lower computational complexity while achieving superior error-correction performance. The LDPC decoding algorithm is inherently parallel and hence it is suitable for GPU massively parallel implementation. GPU is an economic and efficient alternative to FPGA for LDPC decoding implementation. In this paper, a proposed new GPU-based regular LDPC decoder is presented. The current version runs on 32-bit single precision with the log-sum-product iterative decoding algorithm. For a matrix size of 4,000 by 8,000 with 10 iterations, the proposed regular LDPC decoder has 1.808 ms execution time on Tesla C1060 GPU, whereas its original CPU-based LDPC decoder takes 490.504 ms.
