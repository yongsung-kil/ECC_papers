Abstract— in this paper, we theoretically derive the suitable normalized factor of normalized BP-based algorithm of LDPC decoding in 802.11n system. To be specific, we compute 1.15, 1.25, 1.3, 1.35 as the suitable normalized factors with regard to R=1/2, R=2/3, R=3/4, R=5/6 code in 802.11n system respectively. These factors can compensate for the performance loss in UMP BP-based algorithm. By analyzing the statistical distribution of variable node messages (R=1/2 code particularly), the word-length of variable nodes is reduced to 7 bits, and the simulation results show that it has a similar decoding performance compared with the theoretical 9-bit word-length decoding.
Keywords—LDPC decoding, 802.11n, normalized factor, word-length
I. INTRODUCTION
Low-density parity check (LDPC) code [1] was firstly proposed by Gallager in the 1960s, and later rediscovered by Mackay and Neal [2], [3]. It has been of great academic interest recently. Being similar to turbo codes, LDPC codes can achieve within 0.04dB of the Shannon limit error performance [4], [5], and denote a very promising prospect for error control coding. Because of its outstanding performance, LDPC code has been largely adopted by many communication systems, such as wireless local area network (802.11n).
With regard to LDPC decoding, Belief-propagation (BP), Log-likelihood ratios (LLR) are the most common algorithms, but they are too complex to be implemented. Recently, two important revised algorithms called normalized BP-based algorithm [6] and offset BP-based algorithm [7] are proposed and widely used in hardware implementation. The normalized factor or offset factor is used in the UMP BP-Based decoding algorithm to compensate the performance loss. Consequently, both of them are the so-called min-sum approximation [8].
In this paper, we use a set of formulas to determine the normalized factor theoretically. Furthermore, we show that for a specific code, we can compute a certain normalized factor and keep it for all SNR values and all subsequent iterations, while still observing a very suitable performance.
For hardware implementation of LDPC decoding, the word-length of bit node messages affects the total area and complexity. In this paper, we make use of normalized BP algorithm with the suitable normalized factor, 5-bit quantization scheme for initial input messages and 7-bit quantization scheme for variable node messages through iterations. Simulation result shows that no error floor is detected at BER of10−7 .
The paper is organized as follows. We give a brief

review of LLR BP algorithm, UMP BP-based algorithm and Normalized BP-based algorithm in section 2, where the process of computing normalized factor is also presented. In section 3, the simulation results are provided to show the suitable normalized factor with regard to certain code rate. How to determine the finite word-length of bit node messages is presented in section 4. Simulation results are analyzed in Section 5. Finally, a conclusion is given in section 6.

IV. FINITE WORD-LENGTH OF BIT NODE MESSAGES
In the process of decoding, we make use of layered decoding [10] for 802.11n LDPC codes since each row in identity-based matrix can be regarded as a layer. For instance, if the codeword length is 1944 and the code rate is 1/2, the permutation matrixes are 81x81 identical matrixes, and the code has 12 layers with 81 rows in each layer.
In 802.11n LDPC, different codes have different column weights. For the codes (R=1/2 code, R=2/3 code, R=3/4 code, R=5/6 code), the column weight is at most 11, 8, 6, 4 respectively, so that it is enough for us to analyze only R=1/2 code which has the largest column weight. From function (5), when an iteration is finished,
L (qn ) adds an extra summation of at most 11 values (R=1/2 code). It means that the word-length of L (qn ) must be 4 bits
more than that of the initial input. Since the initial input has
the word length of 5 bits, L (qn ) must has 9 bits
theoretically.

V. SIMULATION RESULTS
In the simulations, we make use of BPSK, AWGN channel and normalized BP-based algorithm to design a decoding module which supports all code rates and all codeword lengths defined in 802.11n.
As shown in figure 1, normalized factor varies with code rate or SNR changing. Figure 2-5 determine the suitable normalized factor for all codes. Table 1 shows the suitable normalized factor for R=1/2, R=2/3, R=3/4 and R=5/6 codes is 1.15, 1.25, 1.3 and 1.35 respectively. Simulation results show that the normalized factor in the normalized BP-based algorithm is used to achieve the similar performance with LLR in Figure 6.
Both of Figure 7 and Table 2 show the statistical and word-length distribution of variable node messages, almost the word-length of all of which is less than 8 bits. As a result, the bit node message with one bit fewer will almost save 10% of the implementation area.
Also, we have set the bit node messages’ word-length as 8 bits. Compared with BP, little performance differences are detected at low SNR, but no difference detected at high SNR. When we set the word-length as 6 bits, the decoding performance is undesirable. For instance, the R=1/2 code whose codeword length is 1944 has an error floor at BER of10−7 .

VI. CONCLUSION
In this paper, we determine the suitable normalized factor 1.15, 1.25, 1.3, 1.35 for R=1/2, R=2/3, R=3/4, R=5/6 code respectively in 802.11n system. Simulation result shows those normalized factors enable normalized BPbased algorithm achieve similar performance with LLR in Figure 6. Also, simulation result shows that the word-length of variable node messages is reduced to 7 bits, still maintaining outstanding performance.
