A Low-complexity LDPC Decoder Architecture for WiMAX Applications
Yu-Luen Wang, Yeong-Luh Ueng, Chian-Lien Peng, and Chung-Jay Yang Dept. of Electrical Engineering National Tsing Hua University Hsinchu, Taiwan, R.O.C.

ABSTRACT
In this paper, we present a low-complexity decoder architecture for WiMAX low-density parity-check (LDPC) codes based on a unified task processor. Memory access is accomplished through routing networks with fixed interconnections and memory address generators, which are quite simple due to the quasi-cyclic structure of the LDPC codes. In order to increase the decoding throughput, the check-node and variablenode operations are performed concurrently, and a modified layered decoding is employed. Based on this architecture, we implemented a full-mode WiMAX codec in a 90-nm process. This codec achieves an encoding (decoding) throughput of 960 Mb/s (240 Mb/s) and occupies an area of 0.679 mm2.
Keywords— LDPC codes, channel decoder, WiMAX.
I. INTRODUCTION
Designing a flexible WiMAX LDPC decoder architecture with low complexity for which most hardware resources can be reused when decoding different codes is a challenge [1][2]. An LDPC code can be decoded using message-passing decoding (MPD) through its Tanner graph, which is a bipartite graph consisting of variable nodes and check nodes. In order to increase the convergence speed in BER performance, layered MPD (LMPD) presented in [3][4] can be adopted. In LMPD, the rows of a parity-check matrix (PCM) are partitioned into several layers. Applying the partition method presented in [4] to a quasi-cyclic (QC) LDPC code, the resultant layers have properties such that the core matrices, which are non-zero columns in each layer, are column permuted versions of each other. This modified layered decoding is called LMPD using an identical core matrix (LMPD-ICM).
This paper presents a low-complexity LMPD-ICM-based decoder architecture using the following techniques. First, the operations of variable and check nodes are divided into small tasks such that different codes can be decoded through a unified architecture. Second, the memory access is accomplished through a simple routing network with fixed interconnections and low-complexity memory address generators based on the quasi-cyclic structure of the codes. To increase decoding throughput, the check-node and variable-node operations are performed concurrently, and a pipelined architecture is adopted. Based on this architecture, we implemented a fullmode codec that supports 114 WiMAX LDPC codes.

The PCM for the QC-LDPC code specified in WiMAX can be expressed as an array of z × z circulant matrices. The PCM is constructed by respectively substituting each -1 and each non-negative integer s in an Mb × Nb prototype matrix with a z × z zero matrix and a z × z weight-1 circulant matrix with a shift index s. Consequently, for the PCM, the dimensions are zMb × zNb, and there are Mb block rows and Nb block columns. Fig. 1 shows the prototype matrix in WiMAX for the rate-5/6 code when z = 24. In WiMAX, there are 114=(6×19) QC-LDPC codes which can be classified into six classes. Each class consists of 19 different codes with the same code rate specified by a different expansion factor z.

C. LMPD using identical core matrix (LMPD-ICM)

In [4], the rows of H are divided into z layers, in which the th layer, denoted as H , consists of the th, (z + )th, · · ·, [(Mb − 1)z + ]th rows in H for 0 ≤ < z. The non-zero columns of H form the core matrix H for layer which can be used as a PCM for a shorter code whose code bits come from those of C. It has been demonstrated in [4] that these core matrices H , 0 ≤ < z, are column-permutated versions of each other. Consequently, the routing networks for all layers are the same. LMPD using this kind of layer definition is called LMPD-ICM. It has also been demonstrated that for WiMAX LDPC codes belonging to the same class, the core matrices are almost the same. This property benefits the implementation of the routing networks, since the 19 codes in the same class can share the same routing networks.

III. A UNIFIED DECODER ARCHITECTURE
Since the layer size is not the same for the six code classes, in order to enable hardware reuse, the decoding parallelism adopted is the smallest layer size among the six code classes, which is 4 in this case. Consequently, for some code classes, a layer should be partitioned into several sub-layers with a size not larger than 4. For decoding rate-5/6 codes, the partition is not needed. Since the decoding parallelism is 4, there are four

check-node units (CNUs) and four variable-node units (VNUs) in the proposed decoder. Considering the row weights given in the WiMAX standard, 4-input CNUs and 4-input VNUs are adopted to enable efficient hardware reuse. Consequently, the number of data values, such as APP values and C2V messages, accessed per cycle is 16. The operations associated with each sub-layer (layer) are divided into several tasks. Each task must be in a form that can be processed using a unified architecture, i.e., task processor. For example, as shown in Fig. 2, each layer for a rate-5/6 code is divided into 5 tasks. In Sections III.A to III.C, a case based on rate-5/6 codes is used to illustrate the proposed decoder architecture shown in Fig. 3, which can process these smaller tasks in sequence.

The APP memory bank consists of 24 memory blocks, each of which is used to store the APP values corresponding to one block column in the PCM. Through the address generators and routing network Π1 (Π2), we can read (write) 16 APP values Λj(k, )−1 (Λj(k, )) from (into) the APP memory bank. A ROM module is used to store the information of the initial addresses for the 24 address generators. Since there are 80 weight-1 circulant matrices in the PCM of a rate-5/6 code, in order to process a single LMPD-ICM layer, we must access 80 APP values from the APP memory bank. Note that some of these APP values are the same, since there are several weight-2 variable nodes in a single layer. Remember that 16 APP values are required in each cycle and, hence, 16 APP values are accessed from the memory bank in each cycle. Due to the restriction that only one APP value can be read from an APP memory block during a single cycle, the 16 values accessed simultaneously should be individually distributed among the 24 APP memory blocks. As a result of this restriction, the 80 non-negative elements in the prototype matrix are divided into 5 groups (matrices) where each matrix consists of 16 non-negative elements drawn from 16 different columns. Accordingly, the check-node operations and variablenode operations are divided into 5 tasks. These five tasks are performed in sequence in order to process the associated APP values, C2V messages and V2C messages. The arrangement of these tasks is determined by a computer search that ensures that the hardware restriction can be met. Fig. 1 shows an arrangement of the 5 tasks and the associated assignment of data to the corresponding CNUs and VNUs for the prototype matrix, where TASKi denotes the ith task, 1 ≤ i ≤ 5, and s and j in s(j) of each task denote the shift index in the original prototype matrix and the index of its corresponding column of the prototype matrix (or the APP memory block), respectively. Note that s(j) is also the initial address for the jth APP memory block, and the columns of the tasks indicate the input positions of the VNUs and CNUs (see Fig. 1). It can be seen that the 16 elements in each task come from 16 distinct columns of the prototype matrix, and the non-negative elements belonging to the same column of the prototype matrix are distributed across distinct tasks. Although we can read (write) 24 values from the APP memory bank, only 16 APP values are read (written) from (back) into the memory in order to guarantee that the access restriction can be satisfied in the other code classes and thereby achieve a unified architecture.
B. CNU and VNU
The four VNUs and four CNUs operate concurrently in order to increase the decoding throughput. As shown in Fig. 3, a CNU is divided into four stages. In the first stage, the four V2C messages |Q(jki )| are sent to a 4-input compare (CMP4) unit. The CMP4 unit, which consists of six comparators, is used to choose the two smallest values among the four inputs. In the second stage, a simplified-CMP4 (S-CMP4) unit, which consists of only three comparators, is used to locate the two

smallest values among the four inputs, where two of the four inputs are fed back from the output of the S-CMP4 unit. The number of comparators in the S-CMP4 unit is smaller than that of CMP4, since the four input values of the S-CMP4 unit are partially ordered. In the third stage, the Sign unit is used to calculate the signs of the new C2V messages Ri(jk). The outputs of this stage are the compressed C2V messages, which are also sent to the compressed C2V memory blocks for the later use in the next iteration. In the last stage, the 20 C2V messages are concurrently calculated by the C2V Calculator. These 20 C2V messages will be sent to the VNU simultaneously in order to update the APP values.

Fig. 4 shows the architecture for the variable-node processor (VNP) which consists of four VNUs and one Parallel-update Unit. Each VNU is divided into ten stages, and the ith stage is denoted as STi. These stages are used to calculate the V2C messages, temporarily store the V2C messages, and update the APP values. Fig. 5 shows the timing diagram for the five tasks performed in the VNP. After the 16 APP values associated with a task are read from the memory and passed through Π1, they are fed into 16 subtractors located in ST1 of the four VNUs. The 16 C2V messages Ri(jk−1) are subtracted from their corresponding APP values Λj(k, )−1 by the 16 subtractors to obtain the associated 16 V2C messages Qj(ki ). Following that, the V2C messages are sent to the corresponding CNUs and the registers in ST3. Registers in both ST4 and ST5 are used to temporarily store the V2C messages while the CNUs are calculating the C2V messages. Since the five tasks are executed in sequence, the five tasks associated with the same layer must be in five consecutive stages. C. APP memory access and routing networks
As shown in Fig. 3, each of the 24 APP memory blocks is controlled by an address generator. Since the decoding operations are performed task by task, the 24 addresses generated concurrently by the address generators are used to access 16 APP values associated with a task at each cycle. Since it takes five cycles to process the five tasks for each layer, an address generator consists of a five-stage shift register, which stores the five addresses of five APP values to be processed in sequence. For the first five cycles, the generated addresses, which are initialized by the address ROM module, are used to access the APP values associated with the five tasks of the first layer. The address for the th layer is (s(j) + )mod(z) due to the quasicyclic property of the codes, where s(j) denotes the initial address of the jth memory block for the first layer, as shown in Fig. 1. For example, in TASK1 of the first layer, the APP value stored at the 6th address of the 1st APP memory block is accessed for Input3 of VNU0. In TASK1 of the second layer, the APP value stored at the 7th address of the same memory block is accessed for Input3 of VNU0. Note that when z varies, only the initial addresses change while the column index of j remains the same. Since the processing latency for each task is fixed, the difference between the address being read and the address being written is always the same. Consequently, we can always calculate the write address from certain stage of the shift register in the address generator, which stores the read address, as shown in Fig. 3.
Routing network Π1 consists of 16 multiplexors, respectively, for the 16 inputs of the four VNUs. The size of each multiplexor is 5-to-1, since there are five tasks and, hence, five possible APP memory blocks connected to it. For example, the entries in the first column and the first row of the five tasks shown in Fig. 1, i.e., 0(21), 0(15), 1(14), 20(12), and 21(8), are associated with the first input of VNU0. The advantage of the routing networks is that they can be shared by the 19 different codes in the same class, since the interconnections between the 24 APP memory blocks and the 16 inputs of the four VNUs do not change when z varies. IV. RESULTS AND CONCLUSIONS
We can divide the decoding operations of 114 WiMAX LDPC codes into several similar tasks. With a slight modification of the task processor presented in Section III, a full-mode LDPC code chip was implemented in a 90-nm CMOS process. In addition to the decoder, both early-termination circuit and encoder are included in this chip. Although the parallelism is low, the proposed codec can achieve throughput values of hundreds of Mbit/s. In addition, the throughput of this codec is nearly independent of or code length. Since the work is targeted at consumer and portable applications, the area and power consumption should be as small as possible. As shown in Table I, the proposed codec achieves the smallest area and the best energy efficiency.
