A Scalable LDPC Decoder on GPU
Kiran Kumar Abburi, Department of Electrical Engineering Indian Institute of Technology Madras, India kiran.abburi@ieee.org

Abstract—A flexible and scalable approach for LDPC decoding on CUDA based Graphics Processing Unit (GPU) is presented in this paper. Layered decoding is a popular method for LDPC decoding and is known for its fast convergence. However, efficient implementation of the layered decoding algorithm on GPU is challenging due to the limited amount of data-parallelism available in this algorithm. To overcome this problem, a kernel execution configuration that can decode multiple codewords simultaneously on GPU is developed. This paper proposes a compact data packing scheme to reduce the number of global memory accesses and parity-check matrix representation to reduce constant memory latency. Global memory bandwidth efficiency is improved by coalescing simultaneous memory accesses of threads in a half-warp into a single memory transaction. Asynchronous data transfers are used to hide host memory latency by overlapping kernel execution with data transfers between CPU and GPU. The proposed implementation of LDPC decoder on GPU performs two orders of magnitude faster than the LDPC decoder on a CPU and four times faster than the previously reported LDPC decoder on GPU. This implementation achieves a throughput of 160Mbps, which is comparable to dedicated hardware solutions.
I. INTRODUCTION
LDPC codes, originally introduced by Gallager in 1962 [1] and re-discovered by MacKay and Neal in 1996 [2], have recently received significant attention because of their excellent error correcting performance with fast decoding algorithms. These codes are acknowledged today as one of the most powerful error correcting codes. They have been widely adopted by most advanced communication systems such as DVB-S2, WiMAX (IEEE 802.16e) and Wifi (IEEE 802.11n).
LDPC decoding algorithms are based on iterative belief propagation techniques that require very intensive computation. Several dedicated hardware implementations of LDPC decoder have been proposed in the last few years [3][4][5][6]. However, dedicated hardware solutions are expensive and have poor time-to-market because of significant development, verification and validation intervals. Moreover, design of LDPC decoder is highly dependent on LDPC code structure. Thus developing a decoder for different code requires re-spin of the entire product life cycle. Conversely, programmable solutions are less expensive, flexible, scalable and have improved timeto-market.
Advances in circuit technology and new trends in computer architecture led to the evolution of new forms of parallel processors. Multi-core CPUs from Intel and AMD, manycore GPUs from NVIDIA and ATI/AMD and heterogeneous chip multiprocessors such as Cell Broadband Engine (cell/B.E) from Sony-Thosiba-IBM with impressive computational power

are already available in the market. It is estimated that the number of cores per chip would increase steadily over the next few years and the performance of the processors would improve significantly. However, there is a great need to extract sufficient parallelism from existing algorithms to exploit the computational power of processors. This paper is one such attempt to parallelize LDPC decoding algorithm on GPU.
Today’s GPUs are capable of executing thousands of threads simultaneously. CUDA is a general purpose parallel computing architecture developed by NVIDIA. CUDA allows programmers to develop applications on GPU in high–level languages like C and C++ with a minimal set of language extensions. This has significantly boosted the use of GPUs for nongraphical computing.
Two-phase message-passing (TPMP) and layered decoding are two popularly known algorithms for LDPC decoding. In TPMP, all the check-to-variable messages are computed in one phase and all the variable-to-check messages are computed in another phase at any iteration [1]. TPMP is a fully parallel algorithm in which all the messages of a phase can be computed in parallel. The layered decoding algorithm is a serialized version of TPMP. In the layered decoding algorithm, check nodes are divided into subgroups called layers, which are processed in sequence [7][8]. The layered decoding algorithm achieves about two times faster decoding convergence in comparison to TPMP algorithm. Though the layered decoding algorithm has limited parallelism, it is chosen for this work because of its faster convergence.
Multi-core architecture based programmable solutions for LDPC decoding using TPMP algorithm have been proposed in [9][10]. To the best of my knowledge, this is the first paper presenting LDPC decoding using layered decoding algorithm on GPU. Efficient multi-codeword implementation of LDPC decoder and optimization techniques to hide global and host memory latencies are the main contributions of this work. The execution configuration of the LDPC decoder for IEEE 802.16e LDPC codes is presented in this paper. The same approach can be applied to any block structured LDPC codes. Performance of the proposed LDPC decoder on GeForce 9800 GTX+ GPU is reported in this paper. The proposed LDPC decoder is scalable and achieves much better performance on more advanced GPUs.

III. INTRODUCTION TO GPU
In this section, a brief description to GPU micro-architecture and CUDA programming model is presented. More detailed description can be found in [14][15].
A. Micro-architecture of GeForce 9800 GTX+ GPU
GeForce 9800 GTX+ GPU consists of 16 streaming multiprocessors (SMs), each containing eight streaming processors (SPs) running at 1.836GHz. Each SP has a multiply-add (MAD) unit and an additional multiply (MUL) unit. Each SP can perform 3 floating point operations (FLOP) per clock (MAD contributes 2 FLOP and MUL contributes 1 FLOP). Peak theoretical performance of GeForce 9800 GTX+ can be computed as 3 F LOP/SP ∗ 128 SP ∗ 1.836 GHz = 705 GF LOP S. It has 70.4GB/s memory bandwidth to its off-chip global memory. Each SM processor has 8192 registers and 16KB of on-chip shared memory.
B. CUDA Programming Model
NVIDIA’s Compute unified device architecture (CUDA) is a general purpose programming model for GPUs. The compute intensive and data parallel portions of applications are offloaded onto GPU in multi-threaded fashion. Threads that are executed on GPU are organized into a two-level hierarchy. At the first level of hierarchy, a grid is organized as a two dimensional array of blocks. At the second level of hierarchy, each block of a grid is organized as a three dimensional array of threads. Threads within a block can share data among themselves, and their execution can be synchronized by calling
syncthreads() function in the kernel. Blocks in a grid can be executed independently in any order. Dimensions of the grid and block for an application are chosen to maximize the utilization of the available computing resources.
Threads executed on GPU have access to multiple memory spaces based on thread hierarchy. Registers are allocated to individual threads, and each thread can only access its own registers. All threads in a block have access to onchip shared memory through which threads within a block can communicate. Access to shared memory is as fast as a register access when there are no bank conflicts. All threads in a grid have access to the off-chip global memory. Due to its long latencies (hundreds of clock cycles), global memory accesses have to be minimized to improve performance of the application. In addition to global memory, all threads in a grid have access to read-only constant and texture memories. Constant and texture memories are cached to reduce memory latency. Access to constant memory has same latency as a register access, when all the 16 threads in a half-warp access the same address in a cycle.
Threads within a block are organized into warps of 32 threads. Each warp is executed in SIMD fashion by eight cores of SM on four consecutive clock cycles. If the block size is not a multiple of warp size, any remaining issue slots are wasted. It is advisable to choose the block size that is a multiple of warp size to maximize performance. When threads in one warp are waiting for data, the SM can quickly switch to

a ready warp resident in the SM. To hide the long latencies of global memory accesses, enough threads have to be launched on each SM.

IV. LDPC DECODING ON GPU
The source program of LDPC decoding consists of host code that is executed on CPU and kernel code that is executed on GPU. GPU is treated as a co-processor that executes the data-parallel, compute intensive kernels. Host code transfers the received codewords with noise to global memory of GPU, initiates the kernel for decoding and transfers the decoded codewords back to CPU. Kernel code implements the layered decoding algorithm. GPU executes the kernel according to execution configuration (number of blocks/grid and number of threads/block) specified at runtime.

A. Execution Configuration

In layered decoding algorithm, each layer is processed in sequence as described in section II. Size of layer for IEEE 802.16e LDPC codes is equal to its expansion factor (zf ) which ranges from 24 to 96. This limits the number of threads that can run in parallel for decoding a codeword to zf . Since zf is less than the number of cores of GeForce 9800 GTX+, decoding one codeword at a time results in few cores to be idle. To improve hardware utilization and thus to increase decoder throughput, multiple codewords are decoded simultaneously on GPU.
Number of threads per block is chosen as zf and kernel code is designed such that each block decodes a codeword. All threads in a block are synchronized by calling syncthreads() function after every layer in a kernel to ensure that each layer is processed in a sequence. The ratio of arithmetic operations to memory access operations is low in layered decoding algorithm. Long latency of global memory is the performance bottleneck for this algorithm. To improve performance, global memory latency hiding is achieved by ensuring many threads are simultaneously active on each SM. Since the number of threads per block is less, maximum possible number of blocks per SM are launched so that there are always threads available for execution. Number of simultaneous active blocks per SM is determined by the available resources (shared memory and registers) and maximum number of active threads and blocks the GPU can support. To execute the kernel of the layered decoding algorithm each thread requires 16 registers, and each block requires a 24 ∗ zf kB of shared memory. B. H Matrix Representation
LDPC decoding algorithm is based on iteratively exchanging probabilistic messages between variable and check nodes across the edges of tanner graph, which are represented by position of 1’s in H matrix. Storing information of each edge requires huge memory. Instead, block structured property of IEEE 802.16e LDPC codes is used to compute edge information. In this method, the Hb matrix is scanned in rowmajor order and the column positions of non-zero sub-matrices are stored in one array P, and the corresponding permutation values are stored in another array S. Since the number of threads per block is chosen as the expansion factor of Hb matrix, thread id (tid) which ranges from 0 to zf − 1 is used to expand the Hb matrix. For jth non-zero sub-matrix, position of 1 in H matrix required by thread tid is computed from the corresponding P and S values as

addr[j][tid] = P [j] ∗ zf + (S[j] + tid) mod zf , (9)
where j = {0, 1, . . . , Nnz − 1} and Nnz is the number of non-zero sub-matrices in Hb. The amount of constant memory required to store arrays P and S is 8 * Nnz bytes, which is much less than the cacheable constant memory available for each SM. Since all the threads in a half-warp access the same location of these arrays to compute position of 1’s in H matrix, constant memory access is as fast as a register access.

C. Data Packing Scheme
Floating point implementation of LDPC decoder demands huge memory bandwidth. Experimental results have shown that 6 bit fixed point implementation achieves bit error rate (BER) performance close to that of floating point implementation [16]. In this work, 8 bit and 6 bit precision is used to represent Lj and Ri,j messages. To reduce demand for

memory bandwidth and increase arithmetic intensity, 4 messages corresponding to 4 distinct codewords are packed into a single 32 bit word. Kernel code is updated to unpack these messages, compute new Lj and Ri,j messages corresponding to 4 codewords and pack these messages before writing back to global memory. This reduces the number of global memory accesses by a factor of four and thus increases performance of the decoder significantly. This technique increase the number of codewords decoded simultaneously on GPU by four times, and it is expressed as

E. Coalesced Memory Access
Global memory bandwidth is efficiently utilized when the simultaneous memory accesses by threads in a half-warp are coalesced into a single memory transaction. However, the data access patterns of Lj messages, which are controlled by permutations defined in the base matrix are random. This could result in non-coalesced memory access, that issues a separate transaction for each thread in a half-warp wasting memory bandwidth significantly. The write accesses of Lj messages are coalesced into a single write transaction by modifying the permutations (a technique used in ASIC solutions to remove second cyclic shifter) [17]. Still non-coalesced reads of Lj messages result in significant performance loss. To avoid this problem, Lj messages are pre-fetched into shared memory through coalesced access and accessed from shared memory for computation. Since all messages required in a half-warp are stored in 16 different banks of shared memory, random access of shared memory does not result in bank conflicts, and it is as fast as a register access.

VI. CONCLUSION
A flexible and scalable approach for LDPC decoding using the layered decoding algorithm on a Graphics Processing Unit (GPU) has been presented in this paper. The proposed, highly multi-threaded LDPC decoding on GPU performs two orders of magnitude faster than decoding on a CPU. Choice of a fastconverging decoding algorithm such as the layered decoding algorithm and an efficient multi-codeword implementation of decoding resulted in performance improvement by a factor of four compared to previously reported LDPC decoder on GPU. The proposed programmable solution achieved the throughput comparable to dedicated hardware solutions. This solution meets the throughput requirement of WiMAX and can be used as an alternative to dedicated hardware solutions.
