Gigabit Rate Achieving Low-Power LDPC Codes: Design and Architecture
Shadi Abu-Surra, Eran Pisek, Thomas Henige
Dallas Technology Laboratory Samsung Electronics
Richardson, Texas, USA

Abstract - The quality of any communication system is greatly determined by the performance excellence of the selected channel coding scheme. In this work, we propose algorithms for designing LDPC channel coding schemes. The proposed designs are suitable for achieving Gigabit communications with relatively low-power consumption. We design LDPC code family with fixed block length and no puncturing. Also, starting with an LDPC code base-family, we construct LDPC code families with longer block length but decodable using the same hardware as the base-family. We propose a unified decoder architecture, which can decode all LDPC codes in the designed base-family as well as all LDPC codes in the derived code families with longer block length.
Keywords - LDPC Decoder, Low Power Architecture, Flooding, Layered, WiGig, Gigabit Communications, Baseband Processing
I. INTRODUCTION
The rise of high definition and 3D media, the vision of hyper-connectivity, and the shift toward cloud computing necessitate the need for Gigabit wireless communication systems. However, implementing such high data-rate systems poses a number of engineering challenges. One of the main challenging problems is the design of a high throughput error control scheme. In this paper, we consider Low density parity check (LDPC) codes [1]. They are well known for their good performance. More important, their structure is suitable for parallel decoding. This property makes LDPC codes suitable for very fast decoding. This is very important to achieve communications at very high data rate. In view of that, many of the currently active telecommunication standards, such as IEEE 802.16e [2], WiGig [3], IEEE 802.15.3c [4], and DVB-S2 [5], have LDPC codes in their physical layer error control scheme.
In this paper, we propose a method for designing an LDPC code family with fixed code length, which can provide several rates without puncturing. Our focus is not only to design a code family with good performance, but

which is also compatible with low-power, low-complexity hardware implementation.
Moreover, we propose a method for leveraging an LDPC code base-family to produce a new LDPC code family with longer block length, where both code families are decodable using the same hardware. Finally, we propose the LDPC decoder architecture for decoding these code’s families.
In the rest of this section we present a quick overview of LDPC codes and protograph-based LDPC codes. In Section II, we present our strategies for designing the proposed LDPC code families. In Section III, we present the decoder architecture, and compare the proposed architecture to the traditional LDPC layered decoding architecture. In Section IV, we present some supporting results. Finally, we conclude the work in Section V.
A. LDPC codes and Protograph-based LDPC codes.
An LDPC code, with parity check matrix H, is usually described by a Tanner graph [6] which consists of variable nodes (VNs), which correspond to the codeword’s bits, and check nodes (CNs), which correspond to the single parity check constraints (SPC) defined by the rows of H.
Theoretically, there are no constraints on the locations of the ones in H, and so they can be very random. However, for practical considerations, it is preferable to have some structure in the locations of these ones. Consequently, a class of LDPC codes called protograph-based LDPC codes [7] appeared in the industry. A protograph [7][8] is a relatively small Tanner graph (e.g., see Fig. 1), from which a larger graph can be obtained by the following copy-andpermute procedure. Each edge in the protograph is assigned a different “type” and then the protograph is copied Z times, after which the edges of the same type among the replicas are permuted and reconnected to obtain a single large graph. In this paper, we call this procedure lifting with lifting factor Z. Parallel edges are allowed in the protograph, but not in the derived graph. However, for practical reasons, we don’t consider protographs with parallel edges in this work. Let us denote the protograph’s adjacency-matrix by Hp. Also here, we only use circulant permutations (identity matrices

II. PROPOSED LDPC CODE DESIGN In this section we will describe the proposed LDPC code structure. A. Design of structured LDPC code family. In this paper, we propose a method for designing a protograph-based LDPC code family for which all codes have the same block length. We start by designing the LDPC code with the highest rate among the family’s codes. This code is called the mother code. Then all the other codes in the family are derived from the mother code by splitting the rows of the mother code. Although one may add or delete circulant blocks to improve the code performance, we choose not to do so. Our derived codes do not introduce new circulant blocks; they only inherit the circulant blocks from the mother code. There are several advantages of this choice: First, it enables the use of the same hardware to decode all the code’s family members. Second, all the LDPC codes in the family can be fully described by the mother code’s H matrix and the splitting rules. Note that the deletion of a circulant is not a problem: one may add an indicator in the splitting rules to indicate that this circulant (in the mother code) will not be used in the derived code. However, the addition of a circulant block requires careful hardware design and usually requires saving the H matrix for each code in the family (it is no longer a code family). The following general algorithm is used to design our proposed protograph-based LDPC code family: 1- Given a fixed code length n, and the highest code rate needed Rmax. Select the integers np, mp, and Z such that

Rmax = 1-mp/np, and n = np×Z. The mother code Hbase (and Hp) consists of mp rows and np columns, and the circulant block size is Z.
2- Set the maximum row and column weights, Wr and Wc. Also, set the maximum element value allowed in Hp (For simple designs, it is recommended to set this to 1). Now use the protograph-based EXIT analysis [9] to design Hp which satisfies these constraints and achieves the lowest decoding threshold possible.
3- Use the Progressive Edge-Growth (PEG) algorithm (or any other suitable algorithm such as the Approximate Cycle Extrinsic degree (ACE) algorithm) to select proper circulant blocks. Consequently, Hbase for the mother code is now known.
4- Design a code with rate Rnew (Rnew<Rmax) from the mother code as follows:
a. The number of rows in the new protograph matrix,
Hp’, is mp’, and satisfies Rnew = 1-mp’/np. The number of extra rows needed to construct Hp’ is mp’-mp.
b. Derive the extra rows by splitting the rows of Hp. One row from Hp can be split into two or more rows. Use protograph EXIT analysis to select which rows to split and how to split them. The goal is to achieve the lowest threshold possible.
c. Use the splitting rules obtained in (b) to split Hbase into Hbase’. Note, Hbase’ defines the new code, which can also be defined by the base matrix of the mother code and the splitting rules.
5- Repeat (4) to derive all the codes in the family.
Our strategy for constructing this LDPC family differs from those in literature. One traditional way for constructing an LDPC code family is to start by designing the lowest-rate code in the family (mother code), and then to derive the higher rate codes by puncturing the mother code. However, this technique will introduce codes with different code lengths. Also, codes with a large number of punctured nodes converge slowly, require more hardware, and consume more power. There are also techniques which start with the lowest rate code as the mother code and then derive the higher rates code by deleting some rows in the mother code and adding circulant blocks (usually a large number of them are added) in the remaining rows. However, adding circulant blocks is not recommended for the reasons mentioned above. Some techniques start with the highest rate code as the mother code, but they design the rows in the mother code in a way such that one can select two rows and merge them into one row. The disadvantage of this method is that merging the rows can create cycles in the resultant code’s graph, which results in a poor performance. In contrast, our technique splits the rows, which guarantees that no cycles of smaller size will be created in the derived code.

B. Design of extended-length LDPC code families.
The use of long LDPC codes improves the system performance and enables a design with lower power consumption. However, the challenge is to design code families with different code lengths that can be decoded using the same hardware with minimum addition to complexity and power consumption. Here, we propose an algorithm for deriving an LDPC code family with code length Zp×n, starting with an LDPC code base-family with code length n. The algorithm is described by the following steps:
1. Find the Hp protograph corresponding to each code in the LDPC mother family code.
2. Design the new HZpp protograph for each code in the length- Zp×n code family by lifting each Hp protograph (in step 1) with lifting factor Zp. In every case, choose the lifting that minimizes the number of cycles of size 4 in the resulting protograph.
3. Derive the base matrices for these codes as follows:
a. Replace each zero in the protograph by the Z×Z all-zeros matrix.
b. Replace every Zp ones (which are in the same Zp×Zp circulant block) with the corresponding value in the base matrix of the corresponding code in the mother family.
Note that the proposed algorithm above preserves the properties of the base-family. That is, the new LDPC code family inherits its structure, threshold, row weight, column weight, and other properties from the base-family. Moreover, the number of non-zero elements in the new codes increases linearly with Zp. As the number of information bits also increases linearly with Zp, the decoding complexity per bit stays the same.
C. Design examples.
Two different LDPC code families were designed in order to support longer block size LDPC:
1. Using the method in Section II A, we designed a totally new LDPC code to support 672-bit base family matrices with Z=28 (with 24 columns). The family supports rates 1/2, 2/3, 3/4, and 5/6. All were based on the rate 5/6 H Matrix. Also, larger block sizes were derived from the 672-bit code, (i.e. 2x672=1344bit block, etc.). The method used in order to derive the longer block code is based on the method described in section II B.
2. Starting with the WiGig 672-bit LDPC codes, we designed a new set of Zpx672bit block sizes (2x672=1344bit, 3x672=2016bit, etc.) using the method in section II B.

III. LDPC DECODER ARCHITECTURE
Until recently, LDPC decoding was considered too complicated to be implemented in hardware. Advances in semiconductor processes (i.e. 65nm, 40nm) have removed this obstacle, enabling the practical design of LDPC decoders in hardware. LDPC decoding methods can be divided into two main categories:
1. Flooding or Parallel decoding (ignores row contentions to increase parallelism)
2. Layered or Serial decoding (using the info accumulated between row processing)
Layered decoding may require more cycles to process per iteration, however it requires significantly fewer iterations to converge (>33% less iterations) to the same BLER performance as Flooding. We will describe in detail both the LDPC Flooding block diagram performing MinSum decoding approach as well as the Layered LDPC decoder diagram.

In the rest of this section, we will describe the proposed LDPC decoder architecture and compare it to the currently WiGig designed LDPC decoder. We will use the WiGig v1.0 spec [3] as an example. The comparison is done between the three length 672-bit WiGig code and the proposed length 2016-bit LDPC code (3x672=2016, Zp=3).

First, we compared the performance of the designed base-family to their equivalent codes in IEEE 802.16e (equivalent in term of code block length and rates). The codes in IEEE 802.16e [2] were optimized individually (different hardware is needed to decode codes with different rates) so they will provide a good reference for the performance of our codes. The codeword error rate is shown in Fig. 10, where the codes labeled ‘S’ (dashed line) are the proposed codes and those labeled ‘E’ (solid lines) are the IEEE 802.16e [2] codes. Our rate-2/3 derived code has the same performance as that in the IEEE 802.16e. However, our rate1/2 and rate-3/4 derived codes are only about 0.1dB behind those in IEEE 802.16e at FER = 10E-2. This loss is due to the constraint that all the LDPC codes in the family can use the same hardware as the mother code.

V. CONCLUDING REMARKS
In this paper we proposed methods for designing families of protograph-based LDPC codes. Each family has a fixed code length, and has the following attractive properties: 1) All codes in the family can use a unified encoding and decoding hardware. 2) They are built using circulant block matrices for a reliable hardware implementation. 3) All codes have linear complexity encoders 4) All codes have a very simple representation based only on the base matrix and the lifted protograph, and more importantly, they enable highly power efficient decoders achieving Gigabit rates. In this paper, we described the complete design algorithms, and detailed examples with specific parity check matrices. Also, we provided more details about the decoder architecture with supporting results to show the advantages of the proposed LDPC codes and architecture compared to current design in terms of die size, throughput, FER performance and power efficiency.
