A minimal-resource LDPC encoder and decoder design for memory-constrained systems

Abstract— In this paper we present our LDPC encoder and decoder design focused on, but not limited to, the LDPC codes described in the recent CCSDS space data systems standard, known as the Blue Book. Two important code transformations: shortening and puncturing are described in detail with the goal of introducing a wider range of LDPC code parameters for Adaptive Coding and Modulation and minimizing the transceivers memory requirements. The effects of these transformations on the error performance of the system are evaluated using Monte-Carlo simulations and a provided along with the newly added LDPC code parameters set and memoryconstraints analysis. Our custom free and open-source C implementation of both encoder and decoder is provided along with supporting MATLAB mini-toolkit for LDPC code lifting and transformations. This toolkit further includes also a full implementation and transformations of the QC-LDPC codes utilized in the Wi-Fi 6 IEEE 802.11ax standard.
Keywords—CCSDS, LDPC, code shortening, code puncturing
I. INTRODUCTION (HEADING 1)
Low Density Parity Check (LDPC) codes [1] have found applications in a myriad of communication systems mainly because of their performance approaching the Shannon’s channel capacity [2]. An important development in LDPC design was introduced by [3] - the Quasi Cyclic H matrix structure defines the QC-LDPC codes which are used in many communication standards today, such as the IEEE 802.16 [4], IEEE 802.11ax (Wi-Fi 6) [5,6], DVB-S2 [7], the latest 5G 3GPP Release 17 TS 38.212 [8], and even the wired networking Ethernet IEEE 802.3 standard (10G Ethernet) [9]. Even with the latest processor technologies, the design and efficient implementation of encoder/decoder is a challenging task because of the size of the H matrices of practical LDPC codes. In addition to software-based approaches, the efficient implementation of LDPC decoding on hardware platforms, such as Application-Specific Integrated Circuits (ASICs) and Field-Programmable Gate Arrays (FPGAs), has garnered significant attention [10]. Harnessing the computational power of Graphics Processing Units (GPUs) has become an integral aspect of LDPC decoding research [11,12], while LDPC code matrices modifications research is still complemented by ReedSolomon codes design evolution [13], which turns out to be advantageous in ultra-low decoding latency scenarios. Besides the practical hardware accelerated designs, there is also a substantial need for high-throughput software implementations that can use modern commodity processors [14]. These implementations offer valuable resources for experimentation, optimization, and implementation in research scenarios [15,16]. For studying purposes one can choose from many implementations available at GitHub

portal [17-20] which unfortunately lack the H matrices present in current standards. Some of the publicly available implementations are more comprehensive [21] and can be used without any modifications.
This paper follows up on our previous research [22] focused on the LDPC codes described by the older IEEE 802.16 WiMAX, and more recent WiFi-6 IEEE 802.11ax standards. Both of these specifications share the same QCLDPC design with slightly different parameters. The most recent satellite communications CCSDS 131.0-B-5 for space systems [23] introduces a different LDPC design, which is in the focus of this paper. Our free and open source LDPC toolkit for C and MATLAB available on GitHub [24] has been expanded to accommodate the new CCSDS-defined codes, along with important code transformations, that extend the set of usable code parameter for more flexible Adaptive Coding and Modulation (ACM) purposes.
This paper is organized as follows: The next section briefly describes the LDPC codes used in the most recent CCSDS Blue Book standard. The third section then describes our modification to these codes, introducing a set of new code parameters to be used in flexible ACM schemes. The fourth section then provides some practical information regarding the C implementation memory footprint, results of the error performance obtained by simulation in the form of waterfall curves, and comparison of throughput with other toolkits. The fifth section then concludes the paper.
II. CCSDS LDPC CODES
The LDPC codes are a subclass of binary Linear Block Codes (LBC). Their basic parameters [ n, k ] and m = n – k define the number of rows and columns of their sparse parity check matrix Hm x n, as well as the systemic important code rate R = k / n. The code generator matrix Gk x n can be derived from H is several ways, but it will unlikely be sparse. The encoding using the G matrix is described simply by multiplication of the data word – the vector i of length k bits by G to obtain the codeword c of length n. For a systematic code, the systematic part s of the codeword is a subvector of c equal to i. We may describe the encoding in the following way:

III. CODE SHORTENING AND PUNCTURING
As described in the previous section, the CCSDS standard defines just 3 different LDPC codes with a total set of 9 different parameter combinations. This may be insufficient in situations where fine-grained code parameters selection is required. For comparison, Wi-Fi 6 defines four code rates R = {1/2, 2/3, 3/4, 5/6} [6] and three codeword lengths n = {648, 1296, 1944}, providing 12 code parameters sets, and the 5G specification [8] defines its own LDPC structure with 51 potential H and G matrix sizes, providing much greater flexibility of the code parameters combination. In our design, we use various levels of code shortening and puncturing to introduce a much improved flexibility of parameters even for the LDPC codes defined in the CCSDS standard. Furthermore, the transformations given below are flexible enough so that they may also be used for the Wi-Fi 6 LDPC codes.
A. Shortening and puncturing

To maintain compatibility with the block-matrix structure of H, we deliberately choose to work on blocks of M bits, but this design is actually completely general. We denote the number of shortened bits s. After the zero-filled data is encoded by an LDPC encoder using the standard-described systematic G matrix, the filler zeros are dropped and only the actual data and parity bits are transmitted. In the receiver, the LogLikelihood-Ratio (LLR) metrics are calculated as an input to a SISO LDPC decoder. This decoder uses the standarddefined H matrices. Since the filler zeros were not transmitted at all, corresponding High-Confidence-Zero LLR metric is inserted at their positions. Depending on the actual LLR definition and decoder data-type used, this usually translates to a relatively high positive value (e. g. 1023 in our fixed-point 10-bit decoder implementation [22,24]). After the iterative decoding, a hard decision is made on each LLR metric to obtain the binary data estimate. Further check may be introduced to make sure, that none of the untransmitted zero-fill bits get received as inverted. This would indicate error in decoding. B. CCSDS code transformations parameters
We can now apply the transformations described above to the base codes defined by the CCSDS standard. Fig. 2 and 3 are completely general, but we will limit our choices for the shortened or punctured bits to groups of bits in multiples of M, corresponding to columns of the HB matrix, so a value of s in table below will actually mean shortening of s.M bits. Furthermore, we limit the patterns of shortened columns to continuous indices starting at the index of the last data bit k and counting in reverse. We will always use the base code for shortening. Within these constraints we can write a general expression for shortened bits index range as (assuming indexing starting at 1): is = ( kb – s ).M + 1 for the start index and ie = kb.M for the index of the last shortened bit. For example, for a base code of R = 8/11, M = 128, and s = 5, the sizes of the block matrix HB are nb = 11, mb = 3 (Table II), which determines kb = nb – mb = 8. We will shorten 5 block-columns, which in terms of the large binary H matrix results to shortening of 5x128 = 640 bits at indices 385 to 1024. The remaining codeword will then consist of 384 data bits at indices 1 to 384 and 384 parity bits, originally at indices 1025 to1408, which gives an R=1/2 code. This results is given in Table III, which shows resulting code parameters for two base codes and all potential shortening values. Careful reviewing with Table I reveals the introduction of 3 new code rates R = {7/10, 5/8, 1/4}, (not counting the CCSDS Blue Book defined but unused base code rates R = {8/11, 4/7, 2/5}) and 16 new data word sizes k = {14336, 12288, 10240, 6144, 2048, 3584, 3072, 2560, 1536, 896, 768, 640, 512, 384, 256, 128}. The pre-existing values of k and n are then shown in bold. These greatly improve the choice of available LDPC code parameters to be used in ACM scheme both in real systems and in simulations. IV. PRACTICAL RESULTS
A. Error performance evaluation
We evaluated the error correcting capabilities of the CCSDS space communications systems [23] standarddefined LDPC codes using a MATLAB simulation with an AWGN channel model. We combined our MEX implementation of the encoder and decoder with MATLAB simulation for a selected parameters set. First we chose the middle-size k = 4096, R = 8/11 base code for shortening with Fig. 4 giving the resulting waterfall curves. The AWGN model was chosen because it is commonly used for satellite communications. The simulation could be easily extended by using the comm.rayleighChannel for a multipath fading channel evaluation and is out of scope of this short paper. It would shift the curves in Fig. 4 to the right, affecting all their inclines in the same way.

As expected, since the shortening of data bits, while keeping the amount of parity bits, decreases the resulting code rate (from R = 8/11 to 7/10, 2/3 and 5/8), the error performance is improving with larger s value.
We also tested several puncturing options. To show the generality of our puncturing mini-toolkit, we demonstrate the puncturing results using the LDPC codes defined in the WiFi 6 standard [5,6]. These have somewhat different structure, but the puncturing process works without problems. The choice of this code for puncturing comes from the finer granularity of the Wi-Fi 6 defined codes – the number of block columns of matrix H is far larger (nb = 24) than with any of the CCSDS codes (max. nb = 11), so there are more room for setting the puncturing value p, out of which 4 are presented.
As shown in Fig. 5, since the parity bits are punctured out, the code rate is increasing, so the waterfall shifts to higher –SNR(Eb/N0) area, which is again in line with expectations.
B. Memory requirements
Our implementation uses bitmaps/packed bits whenever possible, to increase efficiency of operations among other things and ultimately to decrease requirements for memory usage. The main factor determining the required amount of memory is the chosen code, i.e., information block length k and code rate R. V. CONCLUSION
In this paper we introduced transformations to the LDPC codes defined in the recent CCSDS Blue Book standard in order to extend the set of available code parameters for a more flexible ACM. Out of the original 18 code parameters set, we derived additional 31. We provide simulation results evaluating selected code parameters in the context of AWGN channel transmission. We tested an upgrade to our existing free and open source MATLAB minitoolkit available on GitHub by evaluating our platform-independent C implementations of LDPC encoder and decoder using MEX.
