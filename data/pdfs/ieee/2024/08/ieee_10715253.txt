In-Memory Bit Flipping LDPC Decoding
Oscar Ferraz, Gabriel Falcao, and Vitor Silva Instituto de Telecomunicac¸o˜es
Dept. of Electrical and Computer Engineering, University of Coimbra, Portugal {oscar.ferraz, gff, vitor}@co.it.pt

Abstract—Low-density parity-check (LDPC) codes play an important role in several communication and storage applications by providing a versatile and efficient solution for correcting errors. Modern computing systems suffer from data movement bottlenecks, which affect parallel processing systems more dramatically. Several solutions to this bottleneck have been proposed by moving compute units where the data resides, incorporating thousands of low-complexity processing units to perform bit-wise and simple integer operations. This paper presents the first known bit flipping LDPC processing-in-memory (PiM)-based decoder in the UPMEM system. The implementation is able to achieve 2.8 Gbit/s of decoding throughput, which is competitive when compared to implementations in high-end GPUs. I. INTRODUCTION
Low-density parity-check (LDPC) codes are a class of errorcorrecting codes that have revolutionized modern communication and storage systems. Characterized by their sparse paritycheck matrices and capacity-approaching performance, LDPC codes efficiently correct errors in data. These codes have applications in diverse fields, such as Wi-Fi [1] and LTE [2], digital storage devices [3], and satellite communications [4], [5]. Their sparsity and ability to approach theoretical limits make LDPC codes a great candidate for ensuring reliable and efficient information exchange.
Structured LDPC codes are well-suited for parallel processing by exploiting their Tanner graph representation, where nodes can be processed concurrently. This enables efficient hardware implementation and accelerates decoding speed, making LDPC codes attractive for high-throughput applications [6].
The data movement bottleneck [7] is a critical challenge in modern computing systems where the rate at which data can be transferred between different components, such as processors, memory, and storage, becomes a limiting factor in overall system performance [8]. As processors continue to improve in speed and capability, the rate at which data can be moved to and from these processors has not kept pace, creating a disparity known as the memory-wall problem [9]. This bottleneck can lead to inefficient utilization of computational resources, longer processing times, and reduced system responsiveness.

Processing-in-memory (PiM) is a solution that addresses the data movement bottleneck by performing computations directly within the memory subsystem [10]. PiM seeks to minimize this data movement by embedding processing units, such as accelerators or specialized logic, near the memory cells [11]. This allows for seamless and rapid data access, reducing latency and improving overall performance in various computing domains, including throughput energy consumption.
A PiM-based LDPC decoder has been proposed in [12] for the Min-Sum (MS) decoding algorithm. However, the lowcomplexity UPMEM’s arithmetic logic unit (ALU) maps better to less complex algorithms, such as the case of the bit flipping (BF) decoding algorithm.
In this paper, we present the first PiM-based BF LDPC decoder found in the literature to the best of our knowledge. We exploit the low-complexity characteristics of the decoding algorithm while leveraging the native hardware integer operation capabilities of the UPMEM’s PiM system to provide a high-throughput implementation and compare it against several graphics processing unit (GPU)-based decoders and a PiM-based MS decoder.

The nonzero elements of (1) dictate the connections between CNs (Sv(m) as the set of VNs connected to CN m) and VNs (Sc(n) as the set of CNs connected to VN n). This code has a CN degree of 4 (dc = 4), and a VN degree of 2 (dv = 2), meaning that each CN connects to four VNs and each VN connects to two CNs.

The LDPC bit-flipping algorithm (BFA) process works by computing a z vector received over a binary symmetric channel (BSC). As depicted in the left-hand side of Fig. 2, the syndrome s is computed (using a modulo 2 addition) for all M CNs, using all the bits connected to a CN m. Next, the number of unsatisfied checks (f ) in each VN is computed by adding all syndrome bits connected to VN n. The codeword bit (zn) is flipped when the number of unsatisfied checks in a bit (fn) is

higher than a threshold (t). Finally, to increase the convergence of the decoding process, an adaptive threshold is employed by decreasing t when no bits are flipped in a given iteration (t is initialized with the largest VN weight (max(dv))). This process is repeated until s = 0 or a maximum number of iterations is reached.
III. UPMEM ARCHITECTURE
The UPMEM PiM architecture is the first commercial realization of a PiM system implemented in hardware [16], [17]. UPMEM uses the more mature fabrication and design of 2D dynamic random access memory (DRAM). Furthermore, the UPMEM architecture allows up to 24 independent threads to perform operations in each DPU which can be viewed as an independent processing unit with dedicated memory.
The general UPMEM architecture is described in Fig. 3. The system is controlled by a host CPU that uses both DRAM and PiM-enabled memory through a standard DDR4-2400 Dual In-Line Memory Module (DIMM). The system features 20 PiM-enabled DIMMs, working at 350 MHz, with each DIMM containing 16 PiM chips. In turn, each PiM chip contains eight DPUs (represented in the right-hand side of Fig. 3), totaling 2560 DPUs in the whole system [16].
Each DPU has 64 MB of DRAM, or main RAM (MRAM), a faster 64 KB static random access memory (SRAM)-based scratchpad memory, or WRAM, and 24 KB of instruction RAM. VII. CONCLUSION
The present study demonstrates the feasibility of generalpurpose PiM-based acceleration in LDPC decoders. The UPMEM system provides rudimentary processing units with low clock frequency that do not produce high throughput performance using single units. However, exploiting multiple cores that are close to memory, alleviates the data movement bottleneck and greatly increases performance, allowing the UPMEM system to compete against high-end desktop GPUs.
Furthermore, the UPMEM system is modular and allows to increase performance by placing more PiM modules in the computing system, whereas to increase performance in GPUs is required to replace the device.
Additionally, the comparison to the MS decoder found in [12], shows that the low complexity of the UPMEM’s ALUs, which emulates floating-point operations by using integer units, benefits the implementation of computationally simple low-bit integer decoding algorithms.
Finally, this work provides data to researchers to understand the differences of general purpose PiM-based acceleration compared to other types of device acceleration, opening the door for new applications to be accelerated.
