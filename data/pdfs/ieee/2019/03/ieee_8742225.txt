Abstract—Software implementation of LDPC decoders has been 
an active area of development for the last 10 years.  Researchers 
have focused on implementing the computationally expensive 
algorithm on both GPPs and GPUs.  A major leap in 
performance was reported in the groundbreaking paper by 
Bertrand le Gal [2].  This paper builds on the work in [2] by 
considering the scaling of that implementation on modern 
many-core processors. We look at the performance of LDPC 
code specified in the DVB-S2 standard.  The large block size of 
the DVB-S2 code makes the memory architecture of the 
processor just as important as the clock rate and instruction set.  
We present results for two generations of Intel Xeons, an Intel 
Phi (KNL), the recently released AMD EPYC.  The key finding 
is that performance scaling is limited by the amount of available 
cache memory rather than the number of cores.  We also find 
that heavily multi-threaded, but deterministic software 
architecture benefits from explicit allocation of threads to cores 
vs. allowing the operating system to manage threading.  The 
maximum throughput of 1 Gbps was achieved on a mid-range 
AMD server – issuing a new era of all-software receivers for 
very high rate waveforms.  We also present the performance of 
the algorithm ported to a low-power ARM processor and 
compare that to a low-end Intel Core.

1. INTRODUCTION 
Computers keep getting faster, and the much-predicted 
demise of Moore’s Law keeps getting pushed further into the 
future.  The computational capacity of modern general-
purpose processors (GPPs) is so high that tasks that have been 
traditionally implemented in specialized hardware can now 
be done entirely in software1.  These new capabilities 
underlie the field of software defined radio (SDR).  
Forward error correction decoding is typically the most 
computationally demanding task in a wireless receiver, and 
the computational requirements of a receiver are typically 
much greater than of a transmitter.  Modern FEC decoders 
(e.g. Turbo, LDPC) require multiple iterations to converge on 
a solution.  Each iteration requires operations on each bit (soft 
decision) in the block.  The operations can be done with low 
precision (e.g. 8 bits) and therefore do not fit efficiently into 
the modern 64-bit processors.  This ‘poor match’ motivates 
some processor developers to include dedicated hardware 
accelerators for FEC decoding [3][5].  Software-centric 
modems implemented on GPPs often include a dedicated 
FPGA or GPU accelerator board specifically for FEC 
decoding.  The disadvantages of these hybrid approaches are 
clear: higher NRE due to the need to develop on multiple 
architectures, significant sustainment risk as the ‘unique’ 
accelerators become unavailable.   
Ground processing receivers that are not SWAP constrained 
can trade decreased efficiency for lower cost by 
implementing the FEC decoding entirely in software on a 
GPP.  This paper presents an implementation of a powerful 
FEC decoder that provides up to 1 Gbps of real-time 
throughput on a mid-range server.  This level of performance 
was unthinkable just a few years ago (the author has fielded 
many ‘there-is-no-way’ comments from colleagues).   
2. DVB-S2 ERROR CORRECTION CODE 
The details of forward error correction in general, and the 
LDPC algorithm in particular, are well beyond the scope of 
this paper.  This section provides the briefest of summaries.  
The two key attributes of modern FEC are: 
• 
Soft inputs.  The received symbols are mapped to

One of the most popular FEC algorithm is known as low 
density parity check (LDPC).  It is a systematic block code 
(i.e. a fixed-length input block is appended with parity bits).  
The DVB-S2 code [3] was developed specifically for high-
rate systems such as satellite video broadcast.  It uses a rather 
large block size of 64,800 bits.  The design of good LDPC 
codes was not fully understood at the time the code was 
developed.  While the code performs rather well (i.e. close to 
the Shannon capacity) at low SNR, it exhibits an error floor 
at higher SNR.  The error floor is due to infrequent bursts of 
errors.  A high-rate BCH code was concatenated with the 
LDPC code to ‘clean up’ these infrequent bursts of error.

The BCH code operates on hard decisions (i.e. ‘0’/’1’) and 
requires only one iteration. It requires much less computation 
than the LDPC. Furthermore, it is only needed when the 
LDPC decoder fails to converge – i.e. some bits do not match 
the LDPC code structure after the maximum number of 
allowed iterations. 
The LDPC code is defined by a very sparse matrix, where a 
‘1’ indicates that an input bit should be used to compute a 
particular parity bit.  For example, the rate ½ code matrix is 
64800×32400, and contains 226799 non-zero elements, for a 
density of 0.01%. 
 
3. MODERN CPU ARCHITECTURE 
 
Modern CPUs are extraordinarily complex devices optimized 
to be ‘jacks of all trades’ – from running large databases to 
video games to scientific computation.  The key parameters 
of a CPU are: clock frequency, number of cores, amount of 
cache, and the instruction set.  Despite continued progress in 
semiconductor technology, the clock frequency has not 
increased much in the past decade.  The main issue is high 
heat dissipation required at high clock rates.  The processors 
have instead been increasing the number of cores, the amount 
of cache, and adding new instructions.  The key to taking 
advantage of the modern CPUs is to parallelize the workload.  
Instead of processing everything on one core at a high rate, 
we must split up the workload into many subtasks that can be

The complexity of modern processors makes it very difficult 
to estimate the performance of any algorithms.  The old 
approach of counting the number of operations (i.e. MOPS) 
is almost meaningless. The interaction between the processor 
and the memory subsystem can change the instructions 
executed per clock cycle (IPC) by a factor of 100.  Operations 
on data in L1 cache take 10x less time than those that those 
that require data to be pulled from L3 cache, while accessing 
data in external RAM takes another 3-5x longer.  If data 
access pattern is predictable, the processor can pre-fetch from 
the ‘slower’ RAM, thereby hiding the access time.  
Unfortunately, the inherent ‘interleaving’ built into modern 
block codes makes memory access rather irregular.

advantage of each processor core.  Multiple instances of the 
algorithm are launched in different threads to take advantage

10×), and one BCH thread can be shared between multiple

executed sequentially in one thread.

7. AMD EPYC PROCESSOR 
Intel processors have dominated the high-end desktop and 
server markets for at least the past 10 years.  A new server 
processor from AMD was introduced in 2018.  The EPYC 
processor architecture is significantly different from the Intel 
Xeon.  The Xeon is a monolithic die, but EPYC uses multiple 
chips in the same package.  Xeon has significantly higher 
bandwidth and lower latency when transferring data between 
cores (if the EPYC cores are not on the same die).  Since all 
the threads are operating almost independently for the LDPC 
decoder, this latency/bandwidth difference is significant. The 
other major difference is that the EPYC does not have a good 
implementation of the AVX instruction set – the instructions

9. EARLY TERMINATION 
One of the reasons LDPC decoding is computationally 
expensive is that it requires many (>20) iterations.  Once the 
decoder has converged to a valid codeword, there is no reason 
to do any more iterations12.  The technique of stopping 
iterations once the code has converged is known as early 
termination.   The higher the SNR, the lower the average 
number of iterations required to converge.  Thus, a decoder 
that takes advantage of early termination can support much 
higher average throughput at higher SNR [6]13.  On average, 
only a few blocks require the maximum number of 
iterations14.   
The algorithm considered in this paper processes multiple 
blocks at once.  The iterations can be terminated only once 
all the blocks have converged.  This makes early termination 
less likely that if each block was considered separately.  In 
fact, the code in [2] does not implement early termination at 
all.  The cost of verifying convergence is very low, and it was 
added to the code15.  The results, shown in Figure 14, show 
that increasing SNR by 1.5 dB roughly doubles the 
throughput.

12. THREAD/CORE ASSIGNMENT OPTIMIZATION 
The decoder application benchmarked for this paper is rather 
simple – only one type of thread is used.  Further, the 
processor architecture is tractable and a logical thread/core 
assignment can be derived analytically.  However, a practical 
application (e.g. a complete receiver) will have many kinds 
of threads, with different CPU resource requirements (e.g. 
some may require lots of RAM while others require many 
SIMD instructions).  The optimal assignment of threads to 
cores, and even the decision of how many threads of each 
type to start becomes an important problem.  As discussed in 
[8], the optimization is NP-hard and becomes intractable for 
even a modest number of thread type on a 32-core machine.  
This problem has received a lot of attention [7][8][9], but the 
Linux kernel uses simple heuristics, leaving the detailed 
assignment up to the user (if necessary).  We applied genetic 
programming approach to the (almost) trivial scenarios in this 
paper: 
1. One thread type (LDPC + BCH) 
2. Two thread types: LDPC and BCH 
Each processor core can be assigned to a thread or left 
unused.  The output of the genetic optimizer17 match the 
manually tuned results.

13. CONCLUSION 
LDPC decoding is usually the most computationally 
expensive part of a receiver. For many software-centric 
receivers, the decoding was off-loaded to an accelerator – 
either an FPGA or a GPU.  The hybrid architecture 
significantly increased the development time and made 
porting to new systems more challenging.  In this paper we 
have shown that mid-range COTS servers can decode LDPC 
at gigabit rates, which is sufficient for most applications. 
Getting the best performance out of a processor and 
maintaining the throughput for real-time operation requires 
some attention to the processor architecture.
