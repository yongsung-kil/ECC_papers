Decoding Latency of LDPC Codes in 5G NR
Hao Wu1,2 and Huayong Wang3 1Department of Wireless Product Research and Design Institute, ZTE Corporation 2State Key Laboratory of Mobile Network and Mobile Multimedia Technology, ZTE Corporation
3Microelectronics Research Institute, ZTE Corporation Shenzhen, China
wu.hao19@zte.com.cn

Abstract—Fifth generation (5G) new radio (NR) is the next generation of mobile network beyond the fourth generation (4G) long term evolution (LTE). In order to meet high throughput and low latency requirements, low-density parity-check (LDPC) codes are adopted by the 5G NR standard for data channels. In 5G NR, the decoder provided to implement the peak throughput requirement may not be able to decode transport blocks over the same radio resources in the same time duration. In this paper, we analyze the decoding latency of LDPC codes in a typical scenario from a system point of view. The analysis can be used to guide the design of the decoder.
Index Terms—5G, new radio, decoding latency, low-density parity-check codes
I. INTRODUCTION
Channel coding is widely used in wireless communication systems to improve the system reliability. The fourth generation (4G) long term evolution (LTE) is developed by the third generation partnership project (3GPP). Turbo codes are the choice for data channels and are always decoded at the mother code rate 1/3, independently of the actual code rate [1]-[2].
Fifth generation (5G) new radio (NR) is the next generation of mobile network beyond the 4G LTE. 5G NR supports three scenarios: enhanced mobile broadband (eMBB), massive machine type communications (mMTC) and ultra-reliable and low-latency communications (uRLLC). The requirements for these three scenarios include low latency and high throughput [3]. The peak throughput requirement is 10 Gbps for uplink and 20 Gbps for downlink. The user plane latency requirement is 4ms for eMBB and 1ms for uRLLC. The control plane latency requirement is 20ms. Taking these requirements into consideration, low-density parity-check (LDPC) codes are adopted by the 5G NR for data channels [4]-[5]. Two rate-compatible base matrices, BG1 and BG2, are defined to guarantee the decoding performance for full ranges of transport block sizes and code rates. By lifting the base matrix, the parity check matrix can be obtained. The lifting sizes range from 2 to 384. For BG1, the mother code rate is 1/3. For BG2, the mother code rate is 1/5 [6]. To reduce the complexity of the decoder, the transport block of a higher code rate can be decoded by using a smaller sub-base matrix.
It is generally considered that the peak throughput requirement is achieved when the code rate is equal to 8/9 and the lifting size is equal to 384 [7]. The architectures of LDPC decoders can be roughly divided into three categories: full parallel, row parallel and block parallel [8]-[10]. The block

parallel architecture is usually used to reduce the complexity of the decoder. The decoding latency per code block is proportional to the number of circulant blocks in the subbase matrix, which in turn depends on the code rate of the transport block [10]-[11]. The decoder provided to implement the peak throughput requirement may not be able to decode transport blocks with the lower code rate over the same radio resources in the same time duration [7]. Since the throughput per code block is proportional to the lifting size, the total number of code blocks increases as the lifting size decreases when the throughput is constant. To reduce the complexity of the decoder, the decoding latencies per code block of different lifting sizes are approximately the same [10], [12]. The decoder provided to implement the peak throughput requirement may not be able to decode transport blocks with the lower lifting size over the same radio resources in the same time duration. In order to overcome these issues, the parallelism of the decoder needs to be increased or the number of iterations needs to be decreased or both. By increasing the parallelism of the decoder, the complexity of the decoder is increased. By decreasing the number of iterations, the performance of the system is degraded.
Reference [13] proposes a scheme to prevent pipeline hazards of the block parallel architecture. The decoding process is rescheduled to minimize the number of idle cycles. Reference [7] analyzes LDPC codes in terms of the system throughput and the decoder latency. The degree of parallelism required by the decoder is considered. In these papers, the decoding latency of LDPC codes is analyzed from a code block point of view. The effect of the number of users on the decoding latency of LDPC codes is not investigated.
In this paper, we give a simple scheme to construct the subbase matrix and analyze the decoding latency of LDPC codes in a typical scenario from a system point of view. The analysis can be used to guide the design of the decoder. The rest of the paper is structured as follows: Section II analyzes the decoding latency of LDPC codes. Numerical results are described in Section III. And the conclusion is given in Section IV.
II. THE DECODING LATENCY
This section analyzes the decoding latency of LDPC codes in a typical scenario. We focus on the uplink. The decoding latency of the downlink can be obtained similarly. Some approximations are made to simplify the analysis.

The front-loaded DMRS and the frequency-first mapping enable the transport block to be processed on the fly. It is not necessary to collect all the symbols in the slot before starting the decoding of the transport block. We assume that these radio resources are occupied by N users.

IV. CONCLUSION
In this paper, we give a simple scheme to construct the subbase matrix and analyze the decoding latency of LDPC codes in a typical scenario from a system point of view. The decoder provided to implement the peak throughput requirement may not be able to decode transport blocks with the lower code rate over the same radio resources in the same time duration. The effective code rate of the incremental redundancy retransmission is low. The latency constraint is difficult to meet when the incremental redundancy retransmission at peak throughput needs to be supported. The decoder provided to implemented the peak throughput requirement may not be able to decode transport blocks with the lower lifting size over the same radio resources in the same time duration. The lifting size decreases with the number of users when the throughput is constant. It is a challenge to support low latency services with a high connection density.
