ABSTRACT
The NAND flash memory has rapidly increased in storage capacity per unit area, and the rate of occurrence of errors per P/E cycle is also rapidly increasing accordingly. ECC modules such as LDPC have been added to flash controller for recovering from the errors. However, the system designs to increase the lifetime of the flash memory storage device are still in great demand. In this paper, we design the LDPC encoding and decoding scheme to get stepwise code rate according to the P/E cycle by applying rate-compatible LDPC, as well as the management scheme of excessive parity data. Through this, we can improve the error recovery rate of flash memory storage system and extend the lifetime of NAND flash storage system while reducing the system read and write overhead due to the increase in additional parity data.
KEYWORDS
ECC, RC-LDPC, pchk, parity data
1 INTRODUCTION
The NAND flash memory capacity has increased sharply, however, this development has led to a drastic increase in the rate of data errors. The NAND flash memory generates additional parity data by using ECC (Error Correction Code) in order to recover errors occurring in the operations on the page which is a unit of reading and writing. ECC generation module is included in the NAND flash controller and the generated ECC parity is stored in spare region of the page that stores corresponding data. Typically, the applied ECC schemes are Bose-Chaudhuri-Hocquenghem (BCH)[3] code or LDPC (low-density parity-check) code[4]. Recently, the LDPC is more likely to be used for ECC module of NAND flash controller [5–7]. Even if the ECCs are applied, the lifetime of the NAND flash memory storage device is restricted due to a rapidly increasing error. Specifically, as the P/E cycle increases, the error rate increases more and the P/E cycle corresponding to the error occurrence threshold gradually decreases. Secondly, parity data generated excessively cannot be stored in the existing spare region of within a page, which leads additional read and write operations, as well as space overhead.
In this paper, we propose the design of rate-compatible LDPC scheme that has several different code rates according to the P/E cycle, and their parity data management system as well. The scheme sets several, i.e., two or three, P/E cycle sections, and generates different length of ECC parity with different code rate for each P/E cycle section. Even though there are different lengths of codeword at each P/E cycle section, extended parity check matrix(pchk) is applied for steps with rate-compatible encoding, and as a result, the increasing ECC parity of higher level is just appended to existing codeword of lower level so as not to be mixed with the existing parity data. The additional ECC parity should be stored somewhere apart from original data if the codeword length is larger than region of each page, which includes spare region. The ECC parities, that cannot be written to spare region, is aggregated and stored together at other region. A page storing ECC parities covers several data pages of corresponding codeword. Also those are cached in buffer memory to reduce read overhead. At decoding stage, decoding is applied in the order of low-level codeword to high-level codeword. For each level, the exact codeword is retrieved from flash memory. If decoding is successful, parity data for high-level codeword does not need to be retrieved. We can reduce retrieval overhead of ECC parity for high-level codeword since those are aggregated and cached.
2 RATE-COMPATIBLE LDPC AND ECC PARITY DATA MANAGEMENT
LDPC is a data encoding method that generates a codeword including parity data from source data using a parity check matrix (pchk). In the case of binary pchk, most of the elements are zero and contain a small amount of one element. The structure of pchk can be determined according to the code rate, which is the ratio of data to code word, and parity data of various sizes can be generated from source data of the same size. Generally, when pchk is determined according to the code rate and the source data is encoded, the parity data is distributed in different bit order in the codeword, so it is difficult to separate parity data that is additionally generated for different coding rates.

Figure 2 also shows a structure in which flash memory stores and manages parity data generated in stages. The parity data generated in the first step is stored with the data in the spare region of each page as in the conventional manner. The parity data generated by the pchk of the second or higher level is stored to a separate flash memory region. At this time, since the data area is relatively larger than the additionally generated parity data, the parity data for a plurality of pages are collected and stored in one page. For instance, if additional parity data of 256 bytes is generated for 4KB data, additional parity for 16 data pages can be stored in one page of the area storing parity data together. The page write for parity data is also managed in the DRAM cache of the flash storage until one page is full, and then the page is written to the page of the parity data block when it is full. The cache for parity data is managed separately from the cache in the data area for reducing retrieval overhead. In addition, information on parity data associated with each data is designed to be tracked and managed together with a mapping table in the FTL (Flash Translation Layer) area.

3 PRELIMINARY RESULTS
In order to show the feasibility of the proposed method, pchk and pchke for three stages of LDPC with code rates of 0.8, 0.67, and 0.57 were constructed in the manner described above. The code rates are changed when P/E cycles are 2000 and 4000. With the configuration, error recovery rates of LDPCs and relative read overhead were simulated. Figures 3 and 4 are representations of these simulation results. As shown in the figure, if the stepwise extended LDPC scheme is applied, it can be confirmed that the extended LDPC shows a higher error frame error recovery rate than the basic LDPC scheme. As the P/E cycle increases by using one LDPC module, it can be confirmed that the error recovery rate can be increased by applying the stepwise extended LDPC coding scheme, and it is possible to increase the service lifetime. For read overhead, the proposed system would reduce additional read overhead by separating additional parities from data pages.
